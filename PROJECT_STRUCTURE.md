## Общее устройство репозитория

Cubica — это монорепозиторий (monorepo — один репозиторий, содержащий несколько сервисов, библиотек и документацию). В корне находятся общие документы и мета‑файлы, а доменные части системы вынесены в отдельные каталоги:

- `services/` — backend‑сервисы и инфраструктурные компоненты.
- `SDK/` — SDK (Software Development Kit — набор библиотек и инструментов для использования платформы Cubica) и вспомогательные пакеты.
- `docs/` — архитектура, схемы, задачи и планирование.
- `data/` — тестовые данные и моки.
- `games/` — пакеты игр (контент и манифесты) и связанные ресурсы.
- `scripts/` — служебные скрипты для разработки и индексирования.
- `draft/` — черновые материалы, эксперименты и архивные компоненты.
- `archive/` — устаревшие документы, сохранённые для истории.
- `repo-manifest.json` — машиночитаемый индекс артефактов репозитория и их статусов.

## Корень репозитория

Структура корневого уровня:

```text
Cubica/
├── .vscode/                # Настройки IDE
├── archive/                # Архив старых документов и черновиков
├── data/                   # Фикстуры и моки данных
├── docs/                   # Архитектура, задачи, планирование
├── draft/                  # Черновые материалы, эксперименты и архивные компоненты
├── games/                  # Пакеты игр и связанные ресурсы
├── scripts/                # Скрипты разработки и индексирования
├── SDK/                    # SDK и вспомогательные библиотеки
├── services/               # Сервисы и инфраструктура
├── AGENTS.md               # Правила работы ИИ‑агентов
├── package.json            # Корневой npm-workspace (workspaces: SDK/core, SDK/shared, SDK/react-sdk, services/runtime-api)
├── PROJECT_OVERVIEW.md     # Общий обзор проекта
├── PROJECT_STRUCTURE.md    # Описание структуры репозитория (этот файл)
├── README.md               # Основное описание репозитория
├── repo-manifest.json      # Машиночитаемый индекс артефактов репозитория
└── ...                     # Прочие вспомогательные файлы (например, .gitignore)
```

## Каталог `data/`

Каталог `data/` хранит тестовые и демонстрационные данные:

```text
data/
├── fixtures/               # Фикстуры (готовые наборы данных для сценариев)
│   └── games/
│       └── antarctica.json # Пример набора данных для игры
└── mocks/                  # Моки (поддельные ответы внешних систем)
    ├── llm/
    │   └── default-response.json   # Мок‑ответ от модели (LLM — Large Language Model, большая языковая модель)
    └── router/
        └── starter-session.json    # Стартовая сессия для мок‑роутера
```

## Каталог `docs/`

Каталог `docs/` содержит основную документацию проекта:

```text
docs/
├── architecture/           # Архитектурные документы, схемы и SQL
│   ├── adrs/               # ADR (Architecture Decision Record — записи архитектурных решений)
│   ├── backend/            # Архитектура backend-компонентов (session-persistence, view-adapters)
│   ├── engine/             # Архитектура Game Engine (LLM Context Pipeline)
│   ├── protocols/          # Протоколы взаимодействия (например, MVP Interaction)
│   ├── reviews/            # Ревью архитектуры и аудиты
│   ├── schemas/            # JSON‑схемы и другие формальные описания
│   ├── search/             # Документация по поиску (например, Qdrant — векторная база данных)
│   ├── sql/                # SQL‑скрипты и DDL
│   ├── PROJECT_ARCHITECTURE.md
│   └── ...                 # Прочие архитектурные материалы
├── legacy/                 # Старые / временно поддерживаемые документы
└── tasks/                  # Система задач, дорожная карта и шаблоны
    ├── content-packs/      # ExecPlans (исполняемые планы) и PLAN.md
    ├── epics/
    ├── features/
    ├── milestones/
    ├── README.md           # Правила работы с задачами
    └── ROADMAP.md          # Дорожная карта проекта
```

## Каталог `draft/`

Каталог `draft/` используется для черновиков, экспериментов и архивных материалов проекта:

```text
draft/
├── antarctica-nextjs-player/  # Архивный Next.js-плеер. Сохранен для истории и справки, не является активной частью разработки
├── processes/                 # Черновые описания процессов
├── scripts-draft/             # Черновые / экспериментальные скрипты
└── ...                        # Прочие временные материалы (например, планирование агентов)
```

**Назначение:**
- Архивирование материалов, которые сохраняют историческую ценность, но не являются активной частью разработки
- Хранение экспериментальных кода и процессов
- Разделение активных компонентов от справочных/архивных материалов

## Каталог `games/`

Каталог `games/` содержит пакеты игр (контент и манифесты) и связанные ресурсы:

```text
games/
├── antarctica/               # Пакет игры «Antarctica»: game manifest, UI manifests, сценарий (источник истины)
├── .../scripts/              # Пользовательские скрипты (Runtime Content Logic, Sandbox)
└── .../extensions/           # Локальные пакеты расширений (Build-time Capabilities, Native)
```

**Примечание:** Дублирующий Next.js-плеер архивирован в `draft/antarctica-nextjs-player/` для сохранения в истории. Это решение устраняет нарушение Single Source of Truth, выявленное в архитектурном ревью от 2026-01-13.

## Каталог `scripts/`

Каталог `scripts/` содержит служебные скрипты:

```text
scripts/
├── dev/                    # Скрипты для локальной разработки и bootstrap‑задач
├── indexing/               # Скрипты для индексирования и работы с поиском
└── manifest-tools/         # Инструменты для работы с JSON-манифестами (например, генерация текстовых проекций по якорям)
```

## Каталог `SDK/`

Каталог `SDK/` содержит SDK и вспомогательные библиотеки:

```text
SDK/
├── core/                   # Базовые компоненты SDK и общая логика (@cubica/sdk-core)
├── extensions/             # Публичные пакеты расширений (Extension Packs)
├── shared/                 # Общие UI‑компоненты и утилиты (@cubica/shared)
├── react-sdk/              # SDK для React/Next.js‑приложений (@cubica/react-sdk)
├── simulators/             # Симуляторы для разработки и тестирования сценариев
├── viewers/                # Библиотека плееров (Viewers) — см. подробности ниже
└── custom-examples/        # Пользовательские / демонстрационные примеры использования SDK
```

### Подкаталог `SDK/viewers/`

Каталог `SDK/viewers/` содержит библиотеку плееров (viewers) для отображения игр на различных платформах.
Viewer — это клиентский модуль, который читает JSON‑манифесты и рендерит UI игры.
Архитектура описана в [ADR-014](docs/architecture/adrs/014-viewers-library-architecture.md).

```text
SDK/viewers/
└── web-base/               # Базовый веб-viewer (@cubica/viewer-web-base)
    ├── package.json        # NPM пакет
    ├── viewer.json         # Метаданные viewer (id, version, supportedSchemas, capabilities)
    ├── README.md           # Документация по использованию
    └── src/
        ├── index.ts        # Точка входа, экспорты
        ├── types.ts        # Типы: GameManifest, UIManifest, ViewerConfig, GameState
        └── components/
            ├── index.ts           # Barrel export компонентов
            ├── ManifestLoader.tsx # Загрузка манифестов игры
            ├── StateManager.tsx   # Управление состоянием игры
            └── ActionRouter.tsx   # Маршрутизация действий от UI
```

**Назначение компонентов:**
- `ManifestLoader` — загружает GameManifest и UIManifest, кеширует результаты
- `StateManager` — хранит состояние игры, применяет патчи (JSON Merge Patch)
- `ActionRouter` — принимает действия от UI и маршрутизирует к обработчикам

## Каталог `services/`

Каталог `services/` содержит сервисы платформы и связанные компоненты:

```text
services/
├── runtime-api/            # Целевой единый deployable backend (модульный монолит)
├── game-catalog/           # Сервис каталога игр
├── game-editor/            # Сервис редактора игр
├── game-engine/            # Исполнительный игровой движок
├── game-repository/        # Репозиторий игр и их метаданных
├── metadata-db/            # Сервис / слой работы с БД метаданных
└── router/                 # Маршрутизатор запросов между сервисами
```

Каждый сервис обычно имеет однотипную структуру:

```text
<service-name>/
├── docs/                   # Локальная документация по сервису
├── src/                    # Исходный код
└── tests/                  # Автотесты
```

Для `runtime-api/` структура уточняется в соответствии с ADR-017:

```text
runtime-api/
├── DEV_GUIDE.md            # Назначение и внутренние модульные границы
├── package.json            # Пакет deployable backend-а
├── src/
│   ├── index.ts            # Точка входа и реестр модулей
│   └── modules/
│       ├── player-api/
│       ├── session/
│       ├── runtime/
│       ├── content/
│       ├── ai/
│       ├── telemetry/
│       └── admin/
└── tests/
```

## Как поддерживать структуру в актуальном состоянии

- При добавлении, перемещении или удалении каталогов обновляйте этот файл в том же pull request.
- Если структура сервисов в `services/` меняется (добавляются новые сервисы, меняется шаблон их структуры), описывайте изменения здесь и, при необходимости, в архитектурной документации в `docs/architecture/`.
- Не описывайте здесь временные файлы и локальные артефакты (вроде данных разработки), чтобы документ оставался компактным и полезным.
