# Руководство по разработке: Game Repository

## Назначение
- Обеспечивает авторитетное хранение игровых манифестов, ассетов и истории версий.
- Служит источником данных для Router, Engine, Editor и SDK.

## Объем MVP
- Использует файловое хранилище `data/fixtures/` с REST API для чтения и записи.
- Защищает write-эндпоинты API ключами, используемыми Game Editor.
- Реализует блокировку версий на снапшотах с неизменяемыми файлами.

## API
- REST-спецификация: `docs/architecture/repository-openapi.yaml` (черновик Фазы 0).
- TypeScript SDK для Editor; gRPC интерфейс планируется во Фазе 2.
- Во Фазе 2 публикация событий изменений через message bus (задокументировать ADR).

## Config
- Ключевые параметры: директория хранилища, политика версионирования, лимиты запросов.
- В ранних фазах конфигурация задаётся через `.env` и `appsettings.*`; при изменениях фиксировать здесь.
- Скрипт `scripts/dev/mock-repository.ps1` служит для локального запуска без полноценного сервиса.

## Testing
- Unit-тесты валидации схем, логики версионирования, контроля доступа.
- Контрактные тесты синхронизации реализации с OpenAPI.
- Интеграционные тесты с Game Editor и Router в docker-compose окружении.

## Legacy
- Все компромиссы и временные решения отражаются в `docs/legacy/repository-debt.md` и `docs/legacy/debt-log.csv`.
- План миграции в объектное хранилище и СУБД фиксируется здесь (целевые фазы/ADR).
- Статические ассеты и заглушки должны иметь записи в `docs/legacy/stubs-register.md`.
