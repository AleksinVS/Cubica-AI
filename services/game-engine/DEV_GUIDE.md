# Руководство по разработке: Game Engine

## Назначение
- Оркестрирует взаимодействие с внешними LLM для исполнения игрового сценария.
- Гарантирует детерминированные переходы состояния и сохраняет аудируемые транскрипты.

## Объем MVP
- Собирает промпты из правил игры и текущего состояния.
- Вызывает LLM API и преобразует ответы в структурированные дельты для Router.
- Сохраняет промпты и ответы для отладки с маскированием чувствительных данных.

## API
- REST/gRPC контракт оценки действий — `docs/architecture/engine-api.yaml` (черновик Фазы 0).
- Экспортирует сервисные эндпоинты `health`/`metrics`, делится схемой событий для Router по мере готовности.
- Потребляет манифесты Repository по HTTP до перехода на gRPC во Фазе 2.

## Config
- Критичные параметры: провайдер LLM, тайм-ауты, retrу/backoff, бюджеты токенов.
- Значения по умолчанию документируются здесь и в будущих helm/docker chart README.
- Ключи LLM и секреты хранятся во vault/secret manager; локально — через `.env` (не коммитить).

## Testing
- Unit-тесты сборки промптов и расчёта дельт состояния (golden samples).
- Контрактные тесты синхронизации схем запросов/ответов с `engine-api.yaml`.
- Нагрузочные/soak тесты: имитация активных сессий с переиспользованием записанных ответов LLM.

## Legacy
- Mock-адаптер LLM и статические ответы лежат в `data/mocks/llm/`; каждый сценарий зарегистрирован в `docs/legacy/debt-log.csv` (`LEGACY-0003`).
- Запланированные улучшения (кэширование, мультиплеер, офлайн-симуляция) описываются здесь с целевыми фазами.
- Изменения шаблонов промптов фиксируются в `services/game-engine/prompts/CHANGELOG.md`.

## Документация и обновления
- Обновлять гид новыми паттернами промптов, механизмами устойчивости и зависимостями после каждого релиза.
