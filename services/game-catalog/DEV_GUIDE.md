# Руководство по разработке: Game Catalog

## Назначение
- Поддерживает пользовательский портал с перечнем игр и входом в инструменты авторов.
- Отображает метаданные каталога, поступающие из Repository и Metadata Database.

## Объем MVP
- Интегрировать конвертированный шаблон квеста и направлять запросы редактирования в Game Editor.
- Показывать элементы запуска сессий, вызывающие API Router без локального кэширования состояния.
- Документировать ручное наполнение данных до появления публичного API Repository.

## API
- Использует session-эндпоинты Router (см. `docs/architecture/router-openapi.yaml`).
- На Фазе 0 считывает манифесты игр из `data/fixtures/`; начиная с Фазы 1 работает через сервис Game Repository.
- Планируется приём агрегатов из Metadata Database после выхода аналитики во Фазе 3.

## Config
- Конфигурируемые значения: базовый URL Router, флаги отображения экспериментальных игр, ключи превью Editor.
- До появления конфиг-сервиса параметры задаются через `.env`/CI, документация хранится в этом файле.
- Метрики и трекинг подключаются через общую систему аналитики; идентификаторы и каналы фиксируются здесь при изменении.

## Testing
- Unit-тесты компонентов каталога (React Testing Library или аналог).
- Интеграционные тесты запуска сессии с моками Router.
- Визуальные регрессионные тесты для ключевых страниц и карточек игр.

## Legacy
- До запуска Repository используется статический контент в `data/mocks/`; каждый временный сценарий регистрируется в `docs/legacy/debt-log.csv` и `docs/legacy/stubs-register.md`.
- UI-заглушки и компромиссы заносятся в `docs/legacy/portal-stubs.md` со ссылками на задачи.
- Все временные решения сопровождаются фазой удаления и владельцем.

## Документация и обновления
- Обновлять гид после каждого рубежа: эндпоинты, конфигурации, UX-изменения.
- Вести временный changelog в `docs/legacy/portal-changelog.md` до появления централизованной базы знаний.
- Ссылаться на UX-спеки и аудиты доступности из `docs/processes/`.
