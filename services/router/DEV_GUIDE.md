# Руководство по разработке: Router

## Назначение
- Оркестровывает игровые сессии, маршрутизирует действия игроков и предоставляет единый API для клиентов платформы.
- Обеспечивает наблюдаемость, безопасность трафика и управление жизненным циклом сессий.

## Объем MVP
- REST-эндпоинты для создания сессий, отправки действий и получения состояния через HTTP polling.
- Временное хранение состояния в памяти с возможностью снимков на диск для отладки.
- Расширения (rate limiting, мультиплеер, WebSocket) планируются на последующие фазы.

## API
- Основной REST-контракт: `docs/architecture/router-openapi.yaml`.
- Черновик WebSocket-протокола — `docs/architecture/router-ws-protocol.md` (Фаза 2).
- Экспортирует системные эндпоинты `health`, `metrics`, `readiness`.

## Config
- Конфигурация (timeouts, лимиты, маршруты backend-сервисов) хранится в `config/router.*`/переменных окружения.
- Включать в этот раздел новые фиче-флаги, настройки rate limiting и секреты интеграций.
- Логи и метрики передаются через OpenTelemetry; указать целевые коллекторы и каналы оповещений.

## Testing
- Unit-тесты контроллеров, валидаторов, middleware.
- Интеграционные тесты с Engine и Repository (docker compose) и mock LLM.
- Нагрузочные тесты (k6/Locust) перед релиз-кандидатами с фиксацией результатов в `docs/legacy/perf-baseline.md`.

## Legacy
- До готовности Engine/Repository отдаёт статические ответы из `data/mocks/`; все заглушки зарегистрированы в `docs/legacy/stubs-register.md` и `docs/legacy/debt-log.csv`.
- Незакрытые возможности (auth, rate limiting, pub/sub) фиксируются здесь с целевой фазой и владельцем.
- Проверка CI (`scripts/ci/validate-legacy.ps1`) должна проходить перед каждым PR.

## Документация и обновления
- Обновлять гид после каждого спринта: эндпоинты, конфигурации, ограничения.
- Поддерживать последовательные диаграммы в `docs/architecture/router-sequence.md`.
- Ссылаться на постмортемы и runbook из `docs/processes/incident-response.md`.
