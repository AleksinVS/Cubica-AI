# Предложение: Иерархическая Архитектура "Model-View-Presenter" для Cubica-AI

## 1. Введение
На основе анализа текущей реализации игры "Антарктика" и выявленных сложностей при миграции UI, предлагается формализовать иерархическую структуру слоев Model и View, а также механизм их связывания через Presenter.

## 2. Анализ текущего состояния (Antarctica)
- **Model (`game.manifest.json`):** В данный момент модель игры достаточно "плоская". Она содержит списки `infos`, `boards`, `cards`, `metrics`. Логическая связь между ними (например, "эти 6 карточек принадлежат этому экрану") задается через `screenId`, но не образует строгой иерархии объектов.
- **View (`.design.json` / Mockups):** Уже имеет иерархическую структуру: `regions` -> `elements`. Определены типы контейнеров и визуальных элементов.
- **Presenter:** Связывание происходит на уровне кода (вероятно, в `apps/player-web`), где логические ID из манифеста сопоставляются с визуальными областями.

**Проблема:** При смене UI (например, переходе от "плитки карточек" к "диалоговому окну") приходится переписывать значительную часть логики отображения, так как "функциональное назначение" элементов жестко завязано на их "визуальную реализацию".

## 3. Предлагаемая архитектура

### 3.1. Hierarchical Model (Functional/Logical UI)
Модель описывает **что** находится на экране с точки зрения игровых механик, не заботясь о том, как это выглядит.
- **Root (Game)**
    - **Screen (Scene)** - логический экран (например, "Главное меню", "Игровое поле").
        - **Area (Region/Zone)** - функциональная область (например, "Панель состояния", "Зона выбора", "Инвентарь").
            - **Element (Functional Unit)** - конкретный игровой объект.
                - `Variable` (Метрики, таймеры)
                - `Action` (Кнопки, интерактивные объекты)
                - `Content` (Текстовые блоки, карточки, медиа-холдеры)
                - `Audio/Video` (Невизуальные игровые элементы)

### 3.2. Hierarchical View (Visual/Aesthetic UI)
Вью описывает **как** это выглядит, используя иерархию графических примитивов.
- **Root (View Tree)**
    - **Scene/Layout** - общая композиция.
        - **Layer** - слой (Background, Middleground, UI-Layer, FX-Layer).
            - **Group (Container)** - группа элементов с общими трансформациями/эффектами.
                - **Component (Visual Primitive)**
                    - `Image` (Растр)
                    - `Vector` (SVG/Shape)
                    - `Text` (Текстовое поле)
                    - `Area` (Интерактивная зона клика)

### 3.3. Presenter (The Mapper)
Презентер становится декларативным мостом (Mapping Layer). Он связывает `Model Element ID` с `View Component ID`.

**Пример маппинга в манифесте:**
```json
{
  "presenter": {
    "mappings": [
      { "model": "stats.time", "view": "sidebar.timer_text" },
      { "model": "actions.move_forward", "view": "controls.btns.next_arrow" },
      { "model": "content.current_info", "view": "main_stage.info_box.text" }
    ]
  }
}
```

## 4. Критическая оценка и Best Practices

### 4.1. Преимущества (Pros)
1. **Независимость UI от Механики:** Можно создать Sci-Fi версию "Антарктики", просто заменив View-манифест и обновив маппинг в Presenter. Логика `game.manifest.json` (метрики, условия переходов, скрипты) остается нетронутой.
2. **Атомарность:** Позволяет использовать библиотеку `SDK/shared` как набор стандартных "Model Elements" и "View Components".
3. **Удобство для LLM:** Иерархическая структура легче описывается промптами ("Добавь в Area 'Stats' новый элемент 'Level'").
4. **Рекурсивное управление:** Команды типа `hide(Model.Area("Stats"))` автоматически скрывают все связанные View-компоненты.

### 4.2. Риски и вызовы (Cons)
1. **Over-engineering:** Для очень простых игр (где всего 1 экран и 1 кнопка) такая структура может показаться избыточной.
2. **Сложность маппинга:** При большом количестве элементов ручной маппинг может стать источником ошибок. Нужно внедрять **Convention over Configuration** (например, автоматический маппинг по совпадающим ID).

## 5. Рекомендации по реализации

1. **Единый Реестр ID:** Внедрить систему глобальных или контекстных ID, которые сохраняются при переходе от макапа к модели.
2. **Абстракция "Area" vs "Group":** 
    - `Area` должна группироваться по **смыслу** (например, "Область сброса карточек").
    - `Group` во View должна группироваться по **верстке** (например, "Все, что прижато к левому краю").
3. **Использование Path-адресации:** Вместо плоских ID использовать пути: `Model.get("sidebar/metrics/time")`.
4. **Mockup-to-Model Matching:** На этапе дизайна (в `design.json`) уже помечать элементы их функциональным типом (например, `"functional_type": "metric"`).

## 6. Варианты облегчения построения UI
1. **UI Templates:** Создать набор стандартных View-структур (например, "Classic Sidebar Layout", "Centered Dialog Layout").
2. **Auto-mapping Helper:** Скрипт, который предлагает варианты связывания Model и View на основе семантической близости названий (через LLM).
3. **Visual Linker:** Инструмент в редакторе, позволяющий "протянуть линию" от элемента модели к компоненту вью.

## 7. Заключение
Переход к иерархическому MVP позволит превратить разработку игр в Cubica из "написания кода" в "сборку из модулей и настройку связей". Это критически важно для масштабируемости платформы и возможности быстрого клонирования механик.
