## Understanding

Ввести объективный датчик «тяжести» рендерера: CI-проверку размера
first-load JS (JavaScript, загружаемый браузером при первом открытии
страницы) для `apps/player-web`. Текущий размер фиксируется как базовая
линия, превышение порога — ошибка проверки и сигнал триггера декомпозиции
рендерера по ADR-066 (ядро + жанровые UI capability packs).

## Architecture Source

- ADR-066 (`docs/architecture/adrs/066-renderer-core-and-ui-capability-packs.md`),
  пункт 7 решения — бюджет бандла как датчик триггера.
- ADR-064 — шов ядра и общий подход «сложность по триггерам».
- Долг LEGACY-0039 (`docs/legacy/debt-log.csv`) — отслеживаемая готовность
  к декомпозиции.

## Why

Интуитивное «рендерер потяжелел» не проверяемо и не блокирует CI.
Числовой бюджет делает рост бандла видимым в каждом PR и превращает один
из триггеров ADR-066 в автоматический сигнал. Ориентир базовой линии на
2026-07-07: first-load JS маршрута `/` ≈ 122 kB, shared-чанки ≈ 105 kB
(вывод `next build`).

## Target State

- Скрипт `scripts/ci/validate-player-web-bundle-budget.js`:
  - читает артефакты сборки `apps/player-web/.next` (манифесты чанков) и
    вычисляет first-load JS главного маршрута и суммарный размер
    shared-чанков; парсить текстовый вывод `next build` нельзя (формат не
    является контрактом);
  - сравнивает с бюджетом из явного файла конфигурации
    (`scripts/ci/player-web-bundle-budget.json`: базовая линия, порог);
  - при превышении завершается с ошибкой и печатает дельту по чанкам.
- Бюджет задан явными числами в закоммиченном файле; изменение порога —
  осознанный коммит с обоснованием (правило из ADR-066: базовую линию
  пересматривают, проверку не отключают).
- Скрипт подключён в корневой `package.json` как
  `verify:player-web-bundle-budget` и вызывается после сборки
  (в составе `verify:player-web` после `build` либо отдельным шагом CI —
  решает исполнитель, зафиксировав выбор в Handoff Log).

## Scope

- Новый CI-скрипт + файл бюджета.
- Подключение в корневые npm-скрипты.
- Короткая справка в комментарии скрипта: как обновлять базовую линию.

## Non-Goals

- Сама декомпозиция рендерера на пакеты (по триггерам ADR-066, LEGACY-0039).
- Реестр типов компонентов вместо статического switch (вводится с первым
  жанровым пакетом).
- Бюджеты для editor-web и runtime-api.

## Dependencies

- Нет блокирующих: сборка `player-web` зелёная (проверено 2026-07-07).
- Windows-хосты: сборку запускать POSIX-шеллом (`--script-shell=bash`) —
  скрипт `build` использует env-префикс.

## Execution Plan

1. Изучить структуру `.next` манифестов (`app-build-manifest.json`,
   `build-manifest.json`) и выбрать устойчивый способ подсчёта first-load
   JS; зафиксировать выбор в Handoff Log.
2. Реализовать скрипт и файл бюджета; выставить базовую линию по фактической
   сборке и порог (+15% к базовой линии, если при реализации не появится
   довод в пользу иного — отклонение зафиксировать).
3. Подключить `verify:player-web-bundle-budget` в корневой `package.json`.
4. Проверить срабатывание: искусственно завысить бюджет вниз и убедиться в
   ошибке с внятной диагностикой; вернуть корректный бюджет.
5. Обновить `NEXT_STEPS.md` и Handoff Log.

## Acceptance

- Скрипт зелёный на текущей сборке и красный при заниженном бюджете
  (проверено в п.4 плана).
- Бюджет и порог заданы явным закоммиченным файлом с инструкцией пересмотра.
- Проверка вызывается воспроизводимо из npm-скриптов после сборки.

## Validation

- `npm run build --workspace @cubica/player-web` (POSIX-шелл) →
  `npm run verify:player-web-bundle-budget` — зелёный.
- Негативный прогон из п.4 плана — красный с диагностикой.

## Risks

- Обновления Next.js меняют размер чанков без изменений кода платформы —
  лечится явным пересмотром базовой линии (см. ADR-066, риски).
- Подсчёт по внутренним манифестам `.next` может сломаться при мажорном
  обновлении Next.js — скрипт должен падать с понятной ошибкой «формат
  манифеста изменился», а не давать ложный зелёный.
