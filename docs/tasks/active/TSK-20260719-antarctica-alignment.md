## Why

Игра «Антарктика» — первая мигрированная на платформу игра. Между её текущей
реализацией и тремя нормативными источниками накопились неучтённые разрывы:

1. **UI-эталон** — `draft/antarctica-nextjs-player_AI` (эталоном является только
   внешний вид и поведение интерфейса, не технические решения и стек эталона);
2. **целевая архитектура** — `docs/architecture/PROJECT_ARCHITECTURE.md` и ADR;
3. **эталонная игровая логика** — исходная монолитная игра `draft/Antarctica`
   (`Game.html`, `GameFull.html`).

Незарегистрированный дрейф запрещён правилами репозитория: каждый разрыв нужно
либо устранить, либо явно задокументировать как намеренное отличие или долг.
Первоочередная цель — выровнять UI.

## Scope

Входит:

- измеренная (не «на глаз») инвентаризация UI-разрывов и их устранение;
- извлечение эталонной логики из монолита и сверка с `game.manifest.json`;
- проверка соответствия контура Antarctica целевой архитектуре;
- исправления только через источники истины: `games/antarctica/authoring/*`
  (компиляция `npm run compile:manifests`), общие стили `apps/player-web`,
  платформенные контракты — только по согласованию с PM.

Не входит:

- перенос стека или кода эталона (Next 14, PixiJS, локальный SDK эталона);
- portal-запуск и VPS — отдельная задача
  `TSK-20260518-portal-test-vps-and-antarctica-launch`;
- общепроектное блочное ревью кода — отложенная PM задача
  `TSK-20260717-code-architecture-conformance-review`; настоящая задача
  проверяет только контур Antarctica и не снимает её блокировку;
- Telegram-канал UI (`ui/telegram/`) — сверяется только web-канал.

### Этап 1 — UI-паритет (приоритет)

- **1.1 Стенд** *(оркестратор)* — поднять эталон и текущий стек
  (runtime-api + player-web, сессия Antarctica), зафиксировать порты, viewport
  и профиль сравнения. Учитывать: устаревшие dev-процессы убивать через
  `fuser`, браузеру давать жёсткое обновление.
- **1.2 Механический сбор** *(terra, medium; при сбоях повышение до high)* —
  по `C_ui-compare`: снимки всех экранов и состояний (topbar, leftsidebar,
  журнал, info-экраны, подсказка, переходы) в эталоне и реализации, DOM и
  computed styles, пиксельные диффы, инвентарь элементов. Только сбор и
  измерение, без оценки.
- **1.3 Итоговый сравнительный анализ** *(оркестратор fable либо sol high —
  обязательный минимум по `AGENTS.md`)* — классификация каждого расхождения
  (дефект / шум рендеринга / намеренное отличие), приоритизация, реестр
  `ui-gap-register.md`.
- **1.4 Исправления кластерами** *(terra high, frontend-профиль; кластеры с
  общей компоновкой или платформенными стилями — sol high)* — правки в
  `web.authoring.json` + компиляция и в общих стилях player-web; запрет
  game-specific ветвлений в ядре; после каждого кластера — повторное измеренное
  сравнение затронутых регионов.
- **1.5 Приёмка этапа** *(оркестратор; прогоны тестов — luna)* — пороги
  `C_ui-compare` по всем экранам, `npm run verify:player-web`, фиксация
  намеренных отличий в реестре.

### Этап 2 — архитектура и логика

- **2.1 Извлечение эталонной логики** *(terra high)* — из монолита
  `draft/Antarctica` в структурированную спецификацию: экраны, карточки,
  метрики и их эффекты, переходы, случайность, условия окончания
  (`reference-logic-spec.md` + машиночитаемая проекция).
- **2.2 Проекция текущей логики** *(terra medium)* — та же структура из
  `game.manifest.json` скриптовой проекцией (скрипт одноразовый, в `.tmp/`).
- **2.3 Дифф логики** *(sol high)* — сверка 2.1 и 2.2, реестр
  `logic-gap-register.md` с классификацией: дефект миграции / намеренное
  платформенное отличие / кандидат в долг.
- **2.4 Архитектурная сверка контура Antarctica** *(sol high)* — соответствие
  целевой архитектуре: чистота платформы (правило 10), декларативные контракты
  (правило 12), asset-канал, схемы манифестов, границы `player-web`/
  `runtime-api`; реестр `architecture-gap-register.md`. Найденные существенные
  решения — на согласование PM до реализации.
- **2.5 Исправления по реестрам** *(логика — terra high через authoring +
  компиляция; архитектурные — sol high или оркестратор после согласования)* —
  неустранимые сейчас разрывы регистрируются в `docs/legacy/debt-log.csv`.
- **2.6 Приёмка этапа** *(оркестратор; прогоны — luna)* — verify-цепочки
  манифестов и платформенной чистоты, повторный прогон UI-порогов (регресс),
  обновление документации и журналов.

Блоки 2.1, 2.2 и 2.4 независимы от этапа 1 (только чтение) и могут выполняться
фоном параллельно с UI-работами; исправления 2.5 начинаются только после
приёмки этапа 1.

### Проверка на упрощение

План уже сокращён: отдельная инфраструктура сравнения не создаётся
(используются готовые навыки `C_ui-compare`/`C_ui-from-mockup` и их пороги);
проекции логики — одноразовые скрипты в `.tmp/`, а не новые постоянные
инструменты; реестры разрывов — три файла артефактов без новой системы
планирования; дочерние `TSK-*` не создаются, пока отдельный блок не даёт
самостоятельного результата. Дальнейшее упрощение (пропуск измеренного
сравнения или сверка «на глаз») запрещено правилами навыков и снизило бы
достоверность результата.

## Acceptance

1. Реестры разрывов по трём осям созданы, каждое расхождение классифицировано
   (дефект / намеренное / долг) и не осталось неклассифицированных.
2. UI: каждый экран web-канала Antarctica проходит пороги `C_ui-compare`
   против эталона либо расхождение зафиксировано как намеренное с причиной.
3. Логика: поведение партии соответствует эталонной спецификации либо отличие
   зафиксировано как намеренное/долг; исправления сделаны в authoring-источнике
   и перекомпилированы без ручных правок сгенерированных манифестов.
4. Архитектура: незарегистрированный дрейф контура Antarctica устранён или
   внесён в `docs/legacy/debt-log.csv`; в ядре платформы не появилось
   game-specific ветвлений.
5. Verify-цепочки из раздела Validation зелёные; документация и
   `PROJECT_STRUCTURE.yaml` актуальны.
