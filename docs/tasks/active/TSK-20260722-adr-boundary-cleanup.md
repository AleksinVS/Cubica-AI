## Why

Аудит 85 ADR, не затронутых предыдущей очисткой, обнаружил нарушения границ
в 45 документах. Архитектурные решения смешаны с данными конкретных игр,
экземплярами карточек и метрик, снимками текущей реализации, пофайловыми
миграциями, очередями работ и критериями приёмки.

Цель работы — вернуть ADR роль устойчивого источника архитектурных решений,
не изменяя уже принятые границы платформы. Игровые сведения должны оставаться
в пакетах игр и Gameplay Slice Records, публичные структуры — в JSON Schema и
документации контрактов, а исполнение — в задачах и журналах долга.

## Scope

В границах:

- очистка ADR `008`, `012`, `013`, `016`–`019`, `026`–`028`, `030`,
  `032`–`037`, `039`–`041`, `048`–`058`, `060`, `062`, `063`, `065`,
  `070`, `072`, `074`, `079`–`084`, `086`;
- нейтрализация игровых примеров без потери смысла публичных контрактов;
- удаление планов, отчётов реализации, списков следующих шагов и критериев
  приёмки из ADR;
- синхронизация `docs/architecture/PROJECT_ARCHITECTURE.md` с очищенной
  формулировкой действующих решений;
- проверка, что статусы ADR и принятые архитектурные границы не изменены.

Вне границ:

- изменение публичных контрактов, источников истины или runtime-поведения;
- исправление кода и игровых манифестов;
- пересмотр архитектурной состоятельности решений, кроме уже согласованной
  переклассификации ADR-094;
- механическое удаление точных нормативных ограничений, необходимых для
  безопасности, совместимости, детерминизма или однозначности контракта.

## Plan

1. Разделить 45 ADR на три непересекающихся диапазона и очистить их по единому
   правилу: контекст → решение → ограничения → альтернативы → последствия.
2. Для игровых примеров оставить нейтральную минимальную форму либо ссылку на
   существующий Gameplay Slice Record; устаревшие отчёты реализации удалить.
3. Для планов и миграционных чек-листов оставить только целевое архитектурное
   состояние и запреты постоянного переходного режима; исполнение хранить в TSK.
4. Основному агенту проверить каждое изменение по исходному аудиту, устранить
   пограничные случаи и синхронизировать общий архитектурный обзор.
5. Выполнить текстовые проверки запрещённых маркеров, проверку ссылок и
   `git diff --check`; выборочно проверить, что решения и статусы сохранены.

## Acceptance

- Каждый из 45 ADR содержит только архитектурный контекст, решение,
  ограничения, альтернативы и последствия.
- В ADR не остаются реальные карточки, игровые тексты, числовые показатели,
  идентификаторы экранов или значения конкретного среза, если они не являются
  необходимой частью самой архитектурной границы.
- В ADR не остаются пофайловые планы, очереди этапов, результаты тестов,
  снимки интерфейса, закрытые пункты реализации и следующие шаги.
- Публичные контракты, безопасность, совместимость, детерминизм и статусы ADR
  не ослаблены и не изменены.
- `PROJECT_ARCHITECTURE.md` не противоречит очищенным ADR.
- Проверки документации проходят без ошибок.

## Validation

- `git diff --check` для всех изменённых архитектурных документов.
- Поиск игровых и исполнительских маркеров в 45 ADR с ручной классификацией
  допустимых ссылок и терминов.
- Сверка статусов ADR до и после изменения.
- Проверка внутренних Markdown-ссылок изменённых документов.
- Выборочная ручная проверка наиболее рискованных ADR: модель игровых объектов,
  промт-проекция, игровые панели, граница game/UI, платформенные возможности и
  транзакционный Mechanics IR.

Полный набор тестов кода не требуется: runtime и контракты не изменяются.

Результат 2026-07-22:

- изменены ровно 45 ADR из утверждённого списка, лишних или пропущенных нет;
- `git diff --check` завершился без ошибок;
- 35 JSON-блоков в изменённых ADR успешно разобраны стандартным JSON parser;
- все изменённые ADR представлены в `PROJECT_ARCHITECTURE.md`;
- поиск имён конкретных игр, задач, миграционных чек-листов и операционных
  расписаний в очищенных ADR не нашёл нарушений;
- исходные статусы сохранены: в ADR-013 уточнена только область superseded, а
  из статуса ADR-039 удалён отчёт о реализации без изменения `Accepted`.
- независимая read-only проверка субагента повторила `git diff --check`, число
  ADR, поиск запрещённых маркеров и разбор JSON-блоков; все проверки прошли.
