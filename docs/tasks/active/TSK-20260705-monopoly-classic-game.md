## Understanding

Работа понята так: реализовать **полную классическую «Монополию»** (аукционы,
залог, дома/отели с лимитом банка, торги, тюрьма, банкротство) как игровой
пакет `games/` поверх пакета возможностей ADR-058, тремя последовательными
моделями доставки: (A) хотсит, (B) сетевой мультиплеер (реализация ADR-011 —
отдельная платформенная задача, здесь только адаптация игры), (C) ИИ-оппоненты
(`hybrid` режим ADR-046). Игровой манифест один для всех трёх моделей.

## Why

Первая настольная игра платформы: проверяет пакет ADR-058 на реальной игре,
даёт каталогу эталонный Game Template «экономическая настольная игра»
(см. `PROJECT_OVERVIEW.md` §6 Game Templates, где «монополия» заявлена как
пример) и создаёт основу для бизнес-тренинговых адаптаций.

## Target State

1. `games/estate-race/` (рабочий id; см. вопрос о названии ниже) — полный
   игровой пакет: authoring + скомпилированные game/ui манифесты, поле 40
   клеток, обе колоды, полные классические правила, методические материалы
   `meta.training`/`content.methodology`.
2. Плагин поля `games/estate-race/plugins/estate-race-player/` — отрисовка
   циклического поля, фишек, карточек собственности (по образцу
   `antarctica-player`).
3. Хотсит-партия 2–6 игроков играется в `player-web` от начала до победы.
4. Game-specific правила задокументированы как GSR-записи.
5. Модели B и C играются без изменений game-манифеста (меняются только
   канал доставки и источник действий).

## Scope

- Контент: поле, цены/ренты (классические значения), тексты карточек, локаль RU
  (EN — follow-up), собственные названия улиц/карточек (см. товарный знак ниже).
- Механики уровня игры поверх ADR-058: аукцион, торги, тюрьма, застройка с
  лимитами 32/12, залог, банкротство (дизайн-документ §5).
- UI: web ui manifest (экраны, панели статуса игроков, панель торгов), плагин поля,
  design mockups в `games/<id>/design/mockups/`.
- Обучающий слой: компетенции (финансовое планирование, переговоры, риск),
  материалы участника/ведущего.
- GSR-записи по механикам; фикстуры состояния для превью редактора.

## Non-Goals

- Платформенные примитивы (всё в `TSK-20260705-board-game-platform-capabilities`).
- Реализация ADR-011 (очередь событий, WebSocket, PostgreSQL-сессии) — отдельная
  платформенная задача модели B.
- Домашние правила (деньги на Free Parking и т.п.) — только полная классика.
- Telegram/Phaser каналы; мобильный клиент.

## Execution Plan

### Stage A. Хотсит (первая играбельная модель)

1. **A1. Контент и данные поля**: 40 клеток как объекты коллекции
   (`cellType`, `colorGroup`, цены, ренты за 0–4 дома/отель), карточки колод,
   стартовые деньги/правила в `state.playersTemplate`.
2. **A2. Ядро хода**: бросок → перемещение (`metric.set` с `%`) → «прошёл
   СТАРТ» (`when`) → диспетчер приземления (`branch` по `cellType`); дубли и
   три дубля → тюрьма. GSR-запись.
3. **A3. Собственность и рента**: покупка, отказ → аукцион (фаза `auction`,
   действия bid/pass), рента всех типов (JsonLogic), залог/выкуп. GSR-запись.
4. **A4. Застройка**: дома/отели как объекты банка (32/12), равномерность,
   продажа за полцены. GSR-запись.
5. **A5. Тюрьма, торги, банкротство, победа**: три выхода из тюрьмы; торги
   propose/accept/decline; банкротство с передачей активов кредитору/банку;
   `endConditions` «последний оставшийся». GSR-записи.
6. **A6. UI и плагин**: mockups → web ui manifest (декларативная привязка
   ADR-055, панели ADR-053) → плагин поля → e2e полная партия.
7. **A7. Методический слой и closeout**: `meta.training`,
   `content.methodology`, фикстуры состояния, обновление NEXT_STEPS.

### Stage B. Сетевой мультиплеер (после `TSK-20260705-multiplayer-runtime-realization`, ADR-059)

1. Проверка игры на реализованной очереди событий: `{{actor}}` из
   аутентифицированного игрока, персональные проекции, реконнект.
2. Привязка к порталу (ADR-033, shared binding), лобби/приглашения.

### Stage C. ИИ-оппоненты (после `TSK-20260705-agent-controlled-players`, ADR-060)

1. Объявить `config.players.agentSeats` и fallback-действие (readiness gate
   ADR-060); стратегия агента (промт, методический контекст) — контент игры.
2. Evaluation fixtures: разумность ходов (покупка выгодной собственности,
   выход из тюрьмы) + replay-транскрипты партий с агентом.

## Acceptance

- Stage A: полная классическая партия 2–6 игроков в хотсите от старта до
  победителя, без правок generic-слоёв; все GSR-записи созданы;
  `verify:canonical` зелёный.
- Stage B: та же партия по сети двумя браузерами без изменений game-манифеста.
- Stage C: партия человек + 1–3 агента; отклонённые ходы агента не меняют
  состояние; eval fixtures зелёные.


## Risks

- **Товарный знак**: «Monopoly» — торговая марка Hasbro. Классические правила
  и механики не защищаются авторским правом, но название, оформление поля и
  оригинальные тексты — да. Для каталога нужны собственные название
  (рабочее — «Магнат», id `estate-race`), названия улиц и тексты карточек.
  Решение и утверждение — за владельцем продукта до Stage A6.
- Аукцион/торги — самые сложные механики уровня манифеста; если ограниченный
  `branch` окажется недостаточным, возвращаемся в ADR-058, а не расширяем
  выразительность молча.
- Партия «Монополии» длинная (60–180 минут): нужны фикстуры средней/поздней
  партии для тестов и превью редактора, иначе e2e станет неподъёмным.
