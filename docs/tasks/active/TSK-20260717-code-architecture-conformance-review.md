## Understanding

Провести систематическое ревью кодовой базы (~100 тыс. строк TypeScript в
6 рабочих областях + игровые пакеты + схемы + CI-скрипты) на три класса
находок:

1. **Незадокументированные противоречия с принятой архитектурой** — код
   нарушает инвариант ADR, и это нарушение не зарегистрировано в
   `docs/legacy/debt-log.csv` / `docs/legacy/stubs-register.md`.
2. **Явные ошибки** — дефекты с конкретным сценарием отказа (не «запах кода»).
3. **Неоптимальные решения** — дублирование, лишние слои, обходы существующих
   платформенных механизмов, но только там, где есть конкретное более простое
   решение в рамках принятой архитектуры.

Кодовая база не помещается в одно контекстное окно, поэтому ревью разбито на
16 блоков с явными границами, списком проверяемых инвариантов и точками входа.
Каждый блок исполним одним субагентом за один заход.

## Why

ADR-083/084/085/086 объявили полное переключение платформы на Game Intent →
Mechanics IR с удалением старого effects-пути «без постоянного адаптера».
Такие широкие миграции — главный источник незамеченного дрейфа: остаточные
обходные пути, доверие клиентским полям, ручная валидация вместо схем, игровые
ветки в общих слоях. Правило проекта: незапланированный дрейф запрещён, а
исправление зарегистрированных разрывов имеет высокий приоритет — но для этого
разрывы нужно сначала найти и классифицировать.

## Правила ревью

1. **Сначала инвариант, потом код.** Субагент начинает с формулировки
   инвариантов своего блока (из выжимки ADR), затем ищет нарушения. Находка
   без ссылки на конкретный инвариант или конкретный сценарий отказа не
   принимается.
2. **Фильтр зарегистрированного долга.** Перед репортом каждой находки —
   поиск по `debt-log.csv` и `stubs-register.md` (по пути файла и по ключевым
   словам). Зарегистрированное — не находка; но если запись долга описывает
   состояние неверно (уже исправлено или хуже, чем записано), это отдельная
   находка класса «противоречие».
3. **Тесты и фикстуры — не нарушение чистоты.** Идентификаторы игр в
   `*.test.ts`, `e2e/`, `src/test/`, fixtures — допустимы. Нарушение — только
   в производственных путях общих слоёв (`services/runtime-api/src` вне
   тестов, `apps/player-web/src` вне тестов/фикстур, `packages/contracts`,
   `packages/view-protocol`, `packages/editor-engine/src`).
4. **Классификация общего/игрового.** Для каждой найденной игровой детали в
   общем слое указывать: должна ли она уйти в манифест/схему/пакет игры или
   стать нейтральной платформенной возможностью (ADR-083).
5. **Проверяемость.** Для ошибок (класс 2) — минимальный сценарий
   воспроизведения (вход → неверный выход/отказ). Где дёшево — подтвердить
   запуском существующего теста или короткого скрипта в `.tmp/`.
6. **Дедупликация на консолидации.** Один и тот же дефект, видимый из разных
   блоков (например, дырка в схеме и её следствие в runtime), сливается в одну
   находку с двумя ссылками.

## Формат находки

```markdown
### <BLOCK>-<NN>. <одно предложение сути>
- Класс: противоречие | ошибка | неоптимальность
- Серьёзность: blocker | major | minor
- Файл: <путь:строка> (все места)
- Инвариант: <ADR-NNN / правило CLAUDE.md / схема> — <цитата или пересказ>
- Свидетельство: <фрагмент кода / вывод команды>
- Сценарий отказа (для ошибок): <вход → следствие>
- В реестре долга: нет | LEGACY-XXXX (и что с записью не так)
- Предложение: <править сразу / новый TSK / зарегистрировать долг>
```

Серьёзность: **blocker** — нарушение границы доверия/безопасности или потеря
данных; **major** — нарушение инварианта ADR с наблюдаемым эффектом либо
воспроизводимая ошибка; **minor** — неоптимальность, локальный дефект без
влияния на границы.

## Блоки ревью

Размеры даны для планирования (строки TypeScript без тестов — оценка).

### R1. runtime-api: входной допуск и границы доверия (~5–6 тыс. строк)

- Область: `services/runtime-api/src/modules/player-api`, `session`,
  `bootstrap.ts`, `modules/errors.ts`.
- ADR: 059, 061, 074, 084 (допуск), 047.
- Инварианты: цепочка входного допуска до Mechanics (аутентификация →
  структура → опубликованное действие → роль principal → версия → параметры
  Ajv → живые ссылки `x-cubica-ref` → частота/квота); отказ допуска не создаёт
  квитанцию; principal только из аутентификации, `playerId` из тела не
  используется как личность; лимит 16 плоских скалярных параметров исполняется;
  `cli_`/`sys_` профили `commandId` разделены и префикс не даёт полномочий;
  receipt lookup до проверки версии; 409 только для новой команды на устаревшей
  версии; ledger индексирован стабильным principal (хотсит-retry безопасен).
- Точки входа: маршруты `POST /sessions`, `POST /actions`,
  `POST /agent-turns`; фикстуры `apps/player-web/app/api/runtime/*` как
  потребитель.

### R2. runtime-api: исполнитель Mechanics IR и модули (~6–7 тыс. строк)

- Область: `services/runtime-api/src/modules/mechanics`, `runtime`.
- ADR: 083, 084 (IR), 085, 086, 081, 076, 062.
- Инварианты: атомарность (один commit: state/RNG/аудит/квитанция/события);
  семантический checker и бюджеты (память, строки, стоимость, единый предел
  trigger+цель); разрешение модуля строго по `moduleId + moduleVersion +
  artifactHash`; доменно-разделённые потоки случайности; `deck.*` зоны
  взаимоисключающие; `core.entities.order` — детерминизм и random только при
  полном равенстве; региональный планировщик: read-only расчёт не мутирует
  RNG, подтверждение на той же версии повторяет путь; `system.schedule.*` —
  ≤64 срабатываний, `defer|skip` семантика, повторная проверка trigger под
  session lock; квотный контроллер — поведение при точном повторе квитанции.
- Особое внимание: полнота rollback при частичном отказе транзакции; пути,
  где кандидат-состояние могло бы утечь наружу до commit.

### R3. runtime-api: персистентность и журналы (~4–5 тыс. строк)

- Область: `services/runtime-api/src/modules/session` (store),
  миграции/SQL, `admin`, `telemetry`.
- ADR: 005, 011, 077 (смежно), 078, 084 (журнал/квитанции), 067.
- Инварианты: одна транзакция на одном соединении, `SELECT FOR UPDATE
  NOWAIT`; `state_version` растёт ровно на 1 включая preview restore;
  `last_event_sequence` монотонен, restore не переиспользует `eventId`;
  `session_events` — только подтверждённые факты; `command_result` ≤64 КиБ,
  версионирован, без секретов; отпечаток replay соответствует ADR-078
  (исключаются только `public.log[*].at`, `runtime.lastUpdatedAt`);
  InMemory store доступен только явным dev/test-выбором; восстановление после
  рестарта включает роль и preview content source.
- Проверить фактическую изоляцию защищённого аудита от публичной проекции
  журнала.

### R4. runtime-api: проекции, доступность, Agent Runtime, OpenAPI (~5 тыс. строк)

- Область: `services/runtime-api/src/modules/ai`, `content`, проекционный
  код в `player-api`; `docs/architecture/runtime-api-openapi.yaml`.
- ADR: 019, 046, 047, 051, 058 (проектор), 079, 084 (проекции).
- Инварианты: единый stateModel-проектор — `state.secret`, серверные,
  необъявленные и чужие actor-поля не попадают в проекцию ни на одном пути
  (REST/agent-turn/availability/preview); `projection-only` endpoints
  недоступны Mechanics; availability не раскрывает IR и server-only
  authorization; `allowedCapabilities` — исполняемый allowlist; rejected
  Agent Turn не меняет состояние; неготовый Agent Runtime — честный отказ
  readiness; загрузка контента только published/явно разрешённый preview
  source; инвентарь endpoints ↔ OpenAPI без дрейфа (обе стороны).
- Это блок с максимальным приоритетом безопасности: искать любые «широкие»
  выборки из state в ответы.

### P1. player-web: шов presenter/lib (~4 тыс. строк)

- Область: `apps/player-web/src/presenter`, `src/lib`,
  `presentation-boundary.test.ts`, `app/api/runtime/*` (прокси-слой).
- ADR: 002, 019, 054, 064, 084 (Presenter).
- Инварианты: без React/Next-импортов; Presenter не вычисляет законность

### P2. player-web: рендерер, map-first, плагины, Phaser (~6–7 тыс. строк)

- Область: `apps/player-web/src/components`, `src/plugins`, `src/hooks`.
- ADR: 026, 037, 039, 053, 055, 062, 066, 073, 080.
- Инварианты: в общем рендерере нет игровых id кнопок/CSS-классов/карт
  подписей; привязка действий только декларативная из UI-манифеста; панели —
  общий lifecycle, содержимое из манифеста игры; map-first — только
  ограниченные `workspaceSlot`, без произвольных z-index/координат от игры;
  DOM-путь критических действий работает без Phaser (отказ Phaser не отключает
  клавиатурный путь); плагины: в preview — только session-scoped bundles через
  `PlayerFacingContent.pluginBundles`, вне preview — только published
  content-hash bundles, `apiVersion` иначе fail closed; никакого импорта
  `editor-engine`.
- Вход из Фазы 0: совпадения игровых id в `src/components`.

### P3. player-web: AI-driven путь и поверхности (~2–3 тыс. строк)

### E1. editor-engine: ядро и компилятор (~8 тыс. строк)

- Область: `packages/editor-engine/src` (23 модуля).
- ADR: 034, 036, 048, 049, 050, 052, 057.
- Инварианты: проекции (graph/tree/entity) — не источник истины, запись
  только JSON Patch/`EditorChangeSet`; компилятор вычищает authoring-only

  sidecar; reverse projection не порождает второй источник истины; риск-шкала
  `change-risk.ts` соответствует ADR-057.
- Проверка: скомпилировать текущие authoring-манифесты и убедиться в
  отсутствии authoring-ключей в выходе.

### E2. editor-web: сессии, worktree, долговечная история (~7–8 тыс. строк)

- Область: `apps/editor-web/src/lib` (session/git/worktree/GC части),
  серверные маршруты сессий.
- ADR: 042, 065, 075, 077.
- Инварианты: пользовательский Save требует сессию; Save/Restore/Close/GC —

  разрушительные файловые операции — искать пути обхода root-проверки,
  симлинки, гонки аренды**); продвижение долговечной версии — точка фиксации;
  новая сессия накладывает allowlisted project paths на текущий platform

### E3. editor-web: Agent UI и ChangeSet-граница (~5 тыс. строк)

### E4. editor-web: preview-мост и workspace (~6–7 тыс. строк)

- Область: `apps/editor-web/src/components/workspace`,

- Инварианты: постовые сообщения iframe — проверка origin/источника с обеих

## Оркестрация субагентов

1. **Порядок.** Фаза 0 → волна 1 (R1, R2, R3, R4 — можно параллельно,
   у них непересекающиеся области внутри `runtime-api`) → волна 2 (P1, P2,
   P3, E1, C1 параллельно) → волна 3 (E2, E3, E4, G1, S1, D1 параллельно).
   C1 полезно завершить до G1 (G1 использует выводы C1 о качестве схем).
2. **Контекст субагента.** Каждому субагенту в задание включать: (а) цель и
   список инвариантов блока дословно из этого файла; (б) релевантные находки
   Фазы 0; (в) правила ревью и формат находки; (г) указание прочитать выжимки
   своих ADR из `PROJECT_ARCHITECTURE.md`, а полные ADR — только при
   неоднозначности. Не выдавать субагенту весь этот файл целиком.
3. **Бюджет.** Ориентир на блок: R2, R4, E2 — глубокие (читать код построчно
   в критических путях); P3, D1 — лёгкие; остальные — средние. Если блок не
   умещается, субагент обязан сузить до критических путей и явно перечислить
   непокрытое в отчёте (поле «Не покрыто»).
4. **Отчёты.** Каждый субагент пишет
   `docs/tasks/artifacts/TSK-20260717-code-architecture-conformance-review/block-reports/<block>.md`
   и возвращает краткое резюме. Оркестратор после сбора результата закрывает
   субагента (CLAUDE.md §2.7) и не оставляет висящих сессий.
5. **Консолидация.** Оркестратор: дедупликация, повторная проверка
   blocker/major находок собственным чтением кода (недоверие субагентским
   выводам без свидетельств), сортировка, `findings.md`,
   `remediation-proposal.md`, обновление Handoff Log и `NEXT_STEPS.md`.
6. **После компакции контекста** оркестратор перечитывает `AGENTS.md` и этот
   TSK (правило CLAUDE.md §2.2).
