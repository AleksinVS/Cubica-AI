## Understanding

Работа должна дать автору игры понятную историю точек «Сохранить»: когда и кем
сохранена версия, что в ней изменилось и как безопасно вернуть прежнее состояние.
В интерфейсе не используется терминология Git. Возврат создаёт новую сохранённую
версию с содержимым выбранной точки и не переписывает существующую историю.

Это реализация уже принятой архитектуры редактора, а не новое архитектурное
решение. Git остаётся источником истории авторского проекта; рабочая копия
сессии остаётся источником несохранённых изменений; отдельная база версий или
второй источник истины не вводятся.

## Why

Сейчас редактор умеет создавать изолированную рабочую сессию и сохранять её
коммитом, но автор видит только факт последнего сохранения. Ошибка автора или
неудачное изменение требуют знания внутренних Git-команд. Новый срез должен
сделать восстановление штатным и безопасным продуктовым действием.

Пользовательский результат:

- автор понимает, сохранена ли текущая работа;
- после повторного открытия не теряет несохранённые правки;
- видит ограниченный список сохранённых версий с понятными резюме;
- может изучить состав версии и вернуть её без удаления более новых версий.

## Product Scenario

Контрольный сценарий выполняется на игре `games/simple-choice`:

1. Автор открывает редактор и существующую сессию игры.
2. Вносит ручное изменение и применяет одно изменение помощника.
3. Нажимает «Сохранить», при желании уточняет предложенное резюме.
4. После повторного открытия редактора видит созданную точку в «Истории».
5. Вносит и сохраняет второе изменение.
6. Открывает сведения о первой версии и выбирает «Вернуть эту версию».
7. Редактор создаёт новую точку восстановления, перезагружает документы и
   предпросмотр; в истории остаются обе исходные версии и новая версия возврата.

Общая возможность дополнительно доказывается изолированной временной Git-фикстурой
без имён и правил `simple-choice`.

## Current Baseline

Уже готово и должно быть переиспользовано:

- `createProjectGitSession` и `saveProjectGitSession` создают изолированную
  рабочую копию и точки Save;
- `restoreSavedVersion` восстанавливает только разрешённые пути и затем создаёт
  новый коммит;
- `EditorSessionDocument` хранит `lastSavedCommit` и вычисляемый `dirtySummary`;
- API файлов уже сохраняет активный документ в рабочую копию сессии;
- журнал редактора и `EditorDiffSummaryItem` дают понятные факты об изменениях;
- операции редакторного ядра и текущая Git-база покрыты модульными тестами.

Не готово:

- ограниченное и постраничное чтение истории сессии;
- стабильный клиентский контракт версии без Git-терминов;
- сохранение пользовательского комментария и понятного резюме;
- пользовательская поверхность «История»;
- безопасный полный цикл возврата, перезагрузки редактора и сброса устаревшего
  локального undo/redo;
- плашка восстановления после повторного открытия грязной сессии.

## Scope Classification

- **Общая возможность платформы:** история версий любого авторского проекта,
  нормализация фактов изменений, безопасное восстановление и восстановление
  несохранённой сессии.
- **Игровое содержимое:** отсутствует. `simple-choice` используется только как
  реальный доказательный сценарий.
- **Запрещённая утечка:** в `editor-engine`, `editor-web` и API не добавляются
  проверки идентификатора конкретной игры или правила конкретного манифеста.

## Scope

1. **Контракты истории.** Ввести клиентские структуры `EditorVersionSummary`,
   `EditorVersionDetails`, результат страницы истории и запрос восстановления.
   Идентификатор версии считается непрозрачным; UI не показывает ветки, SHA,
   коммиты, checkout, reset или revert.
2. **Чтение Git-истории.** Добавить серверные функции, которые:
   - читают только историю текущей ветки сессии;
   - ограничивают записи разрешёнными путями текущей игры;
   - возвращают не более 50 записей на страницу и непрозрачный курсор;
   - отдают автора, время, пользовательский комментарий, структурированное
     резюме и перечень изменённых файлов;
   - не читают произвольные refs из клиентского ввода.
3. **Резюме сохранения.** Перед Save собрать факты текущего журнала и файлового
   diff. Клиент может предложить автору короткий комментарий. Сервер валидирует
   длину и сохраняет комментарий вместе с машинно-читаемыми фактами в метаданных
   коммита. Если журнал неполон или отсутствует, детерминированный fallback
   строится из diff разрешённых файлов; история никогда не остаётся без резюме.
4. **Серверный API.** Добавить динамический Node.js Route Handler:
   - `GET` — страница истории и состояние восстановления сессии;
   - `POST` — восстановление выбранной версии;
   - единая проверка `sessionId`, принадлежности игры, статуса/совместимости
     сессии, курсора, идентификатора версии и ожидаемой текущей версии;
   - статусы `400/404/409` для ошибочного ввода, отсутствующей версии и конфликта;
     внутренние подробности Git не возвращаются клиенту.
5. **Безопасность возврата.** Возврат разрешён только когда рабочая копия чиста.
   При несохранённых изменениях UI предлагает сначала сохранить их; тихое
   затирание запрещено. Перед применением сервер повторно проверяет текущую
   вершину истории по `expectedHead`, достижимость выбранной версии из ветки
   сессии и политику разрешённых путей.
6. **Поверхность «История».** Добавить открываемую панель со списком версий,
   относительным и точным временем, автором, резюме и постраничной загрузкой.
   Детали версии показывают комментарий и понятный список изменений. Возврат
   требует явного подтверждения с объяснением, что будет создана новая версия.
7. **Восстановление грязной сессии.** При повторном открытии существующей сессии
   использовать серверный `dirtySummary` и показывать плашку «Есть
   несохранённые изменения из прошлой сессии» с числом/списком изменённых файлов.
   Продолжение работы использует существующую рабочую копию; автоматический
   сброс или автоматический коммит запрещены.
8. **Синхронизация редактора после возврата.** После успешного возврата заново
   загрузить список файлов, активный документ, sibling-проекцию, проверки и
   предпросмотр. Локальные undo/redo и очередь ещё не применённых намерений
   очищаются как относящиеся к прежней вершине истории; очистка объясняется в UI.
9. **Наблюдаемость и ошибки.** Все ожидаемые отказы получают стабильные коды и
   понятные русские сообщения; необработанные команды, пути и stderr Git в UI не
   попадают.
10. **Документация.** После реализации обновить текущую архитектурную сводку,
    пользовательский UX-документ только при выявленном расхождении, `NEXT_STEPS.md`
    и Handoff Log. Новый ADR не нужен, пока сохраняются зафиксированные здесь
    источник истины и границы.

## Non-Goals

- просмотр полностью работающей исторической игры в отдельном runtime;
- создание копии игры от версии;
- сравнение произвольных двух версий;
- синхронное совместное редактирование и разрешение конфликтов нескольких
  авторов;
- слияние ветки сессии, публикация игры или откат опубликованной версии;
- durable-база метаданных облачной студии, роли и мультиарендность;
- восстановление платформенного кода, сгенерированных файлов или путей вне
  политики текущего проекта;
- новый формат хранения авторского содержимого или второй источник истины.

1. Расширить Save входными `authorComment` и нормализованными change facts с

### Этап 3. Безопасный возврат

1. Добавить optimistic concurrency: клиент передаёт ожидаемую текущую версию,
   сервер возвращает `409`, если история успела измениться.
2. Запретить возврат грязной/несовместимой/закрытой сессии.
3. Выполнить `restoreSavedVersion` только для разрешённых путей и создать новый
   Save с явным типом `restore` и ссылкой на выбранную пользовательскую версию.
4. После успеха обновить session metadata и вернуть новую текущую версию.
5. Доказать тестом, что исходная и более новая версии остаются достижимыми.

### Этап 4. Пользовательская поверхность

## Acceptance

1. История показывает только точки Save текущей игры и сессии, отсортированные
   от новой к старой; первая страница ограничена, следующая загружается курсором.
2. Каждая строка содержит понятное резюме, автора, относительное время и точное
   время в доступном описании; Git-терминология пользователю не показывается.
3. Пользовательский комментарий необязателен, ограничен по длине и сохраняется;
   при его отсутствии доступно детерминированное автоматическое резюме.
4. Сведения о версии перечисляют только разрешённые авторские файлы и не
   раскрывают абсолютные пути, команды или внутренние ошибки Git.
5. Повторное открытие грязной сессии показывает плашку восстановления и
   продолжает работу с существующими несохранёнными файлами без сброса.
6. Возврат при несохранённых изменениях заблокирован с понятным предложением
   сначала сохранить работу.
7. Возврат чистой сессии создаёт новую точку Save; ни выбранная, ни более новые
   предыдущие версии не исчезают из истории.
8. Неактуальный `expectedHead`, чужая/недостижимая версия, несовместимая или
   закрытая сессия не изменяют файлы и возвращают ожидаемый `4xx` ответ.
9. После возврата открытый документ, проектная проекция, диагностики и
   предпросмотр соответствуют новой вершине; устаревшие undo/redo и очередь
   намерений не применяются к возвращённому содержимому.
10. Функциональность не содержит условий для `simple-choice` или другой игры и
    доказана нейтральной фикстурой.
11. Серверный Git-код отсутствует в клиентской сборке; Route Handler выполняется
    в Node.js runtime и не кэширует историю между запросами.
12. Целевые тесты, typecheck, production build и редакторный e2e зелёные.

## Validation

Минимальные автоматические проверки:

```bash
npm test --workspace @cubica/editor-engine
npm test --workspace @cubica/editor-web
npm run typecheck --workspace @cubica/editor-web
npm run build --workspace @cubica/editor-web
npx playwright test apps/editor-web/e2e/editor-session-preview.spec.ts
```

Если для истории будет создан отдельный e2e-файл, запускать его вместе с
`editor-session-preview.spec.ts` и записать точную команду в Handoff Log.

Обязательные ручные проверки:

1. Два сохранения и один возврат на `simple-choice`; убедиться, что история имеет
   три соответствующие точки и возврат не удалил второе сохранение.
2. Повторно открыть грязную сессию и убедиться, что показана плашка, а содержимое
   рабочей копии сохранено.
3. Попытаться вернуть версию при грязной сессии и при устаревшем `expectedHead`;
   убедиться, что файлы не изменились.
4. Проверить панель с клавиатуры и на узкой ширине; в пользовательских строках
   не должно быть слов `Git`, `commit`, `SHA`, `branch`, `checkout`, `reset`,
   `revert`.

## Risks and Controls

| Риск | Контроль |
|---|---|
| Возврат затирает несохранённую работу | Серверный dirty-check непосредственно перед изменением; UI-блокировка не считается достаточной |
| Клиент передаёт произвольный Git ref | Непрозрачный version id разрешается только среди достижимых версий ветки текущей сессии |
| История растёт без границ | Страница не более 50 записей, непрозрачный курсор, отсутствие полного `git log` в одном ответе |
| Журнал не описывает ручные изменения | Детерминированный fallback из diff разрешённых путей; Git остаётся источником факта изменения |
| Два возврата или Save выполняются одновременно | `expectedHead` и ответ `409` до записи |
| После возврата применяется старый undo | Полная перезагрузка редакторного состояния и явная очистка локального журнала/очереди |
| В клиентскую сборку попадает Node/Git код | Разделение Route Handler/server library и клиентских DTO; production build как обязательная проверка |
| Реализация создаёт второй источник истины | Метаданные являются производными подсказками; содержимое и порядок версий определяет только Git |
