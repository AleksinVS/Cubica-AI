## Understanding

Создать проектную файловую вики Cubica с декларативно проверяемыми страницами,
явным смысловым графом, полнотекстовым поиском, сразу доступной для явных
запросов и сравнительной оценки векторной возможностью, минимальной
AST-проекцией модульных импортов и воспроизводимым контуром качества. Режимом
обычного запроса по умолчанию сначала остаётся FTS.

Система должна встроиться в существующие источники истины: ADR, JSON Schema,
код, задачи и `PROJECT_STRUCTURE.yaml`. Она не заменяет генератор структуры,
не использует Git-трейлеры акторов и не хранит разрешение предупреждений в
локальной очереди.

## Target State

- `docs/knowledge/` содержит стартовый корпус из 10–15 страниц синтеза.
- Страницы, конфигурация и поисковые случаи проверяются JSON Schema через Ajv.
- `scripts/dev/kb.mjs` полностью перестраивает локальную SQLite-проекцию в
  `.tmp/project-knowledge/`.
- Штатный `kb query` использует FTS5/BM25.
- Векторное построение, косинусный поиск и RRF реализованы и проверены, но
  начальный `search.mode` остаётся `fts`. Гибридный режим доступен явно,
  участвует в сравнительном `kb eval`, а итоговый режим по умолчанию выбирается
  при приёмке первой версии по измеримому качеству.
- `PROJECT_STRUCTURE.yaml` индексируется и связывается со страницами через
  `covers`; `generate-structure.js` не меняется.
- AST-проекция показывает прямые и обратные импорты модулей отдельно от
  смыслового wiki-графа и полностью восстанавливается из кода.
- Полный `kb sync && kb lint` после merge обнаруживает постоянные проблемы
  объединённого корпуса без предыдущей базы.
- Архитектурный обзор перенесён в вики после обновления всех входящих ссылок.

## Scope

- JSON Schema страницы, конфигурации и поискового случая.
- Node.js CLI: `sync`, `query`, `neighbors`, `impact`, `lint`, `eval`,
  `vector status`, `vector build`.
- `better-sqlite3`, FTS5 capability probe и атомарная полная пересборка.
- Хранилище векторов, интерфейс поставщика эмбеддингов, косинусное ранжирование
  и RRF на тестовых векторах, а также один одобренный рабочий адаптер для
  оценки реального корпуса.
- Постоянные lint-правила и каноническое разрешение `contradicts`.
- Индексация `PROJECT_STRUCTURE.yaml` без изменения его производителя.
- Минимальная AST-проекция пакетов, модулей, импортов и реэкспортов.
- Стартовый корпус и перенос архитектурного обзора.
- Документация рабочего процесса и обновление входных документов агентов.

## Non-Goals

- Назначение режима поиска по умолчанию без сравнительного `kb eval`.
- Передача документов или кода внешнему поставщику эмбеддингов.
- Замена `scripts/dev/generate-structure.js`.
- Постоянный демон, сервер знаний или пользовательский web-интерфейс.
- Акторная модель, `KB-Actor`, SQLite-очередь и переходный `rename-candidate`.
- Пользовательская память продукта, журнал партий и игровые данные.
- Полный AST, граф вызовов, поток данных и автоматическое создание wiki-рёбер.

## Dependencies

- Node.js `>=22.7.0`, Ajv, `js-yaml` и TypeScript уже входят в корневой стек.
- `better-sqlite3` добавляется как корневая dev-зависимость; установка должна
  быть проверена на поддерживаемом Linux-хосте и в CI.
- Реальный поставщик эмбеддингов не блокирует фазы FTS, графа и AST: механика
  векторного поиска сначала проверяется детерминированной тестовой фикстурой.
  Но выбор рабочего адаптера обязателен до сравнительной приёмки поиска;
  внешний поставщик требует отдельного решения PM о конфиденциальности и
  стоимости.
- Перед переносом архитектурного обзора необходимо получить полный список
  входящих ссылок и обновить их атомарно.

## Execution Plan

### Phase 1. Декларативные контракты и фикстуры

1. Создать раскладку `docs/knowledge/`, `.gitattributes`, конфигурацию и
   минимальные страницы-фикстуры.
2. Добавить три JSON Schema и тесты валидных/невалидных примеров.
3. Зафиксировать типы концептов, рёбер, `contradicts.review` и `distinct_from`.
4. Создать отдельные JSON-файлы эталонного проверочного набора (golden set) с
   классом `exact|semantic`, признаком `critical` и проверкой уникальности их
   id.

### Phase 2. Пересобираемая проекция и FTS

1. Добавить `better-sqlite3` и обязательный FTS5 capability probe.
2. Реализовать безопасный поиск корня репозитория и защиту путей.
3. Разобрать Markdown/frontmatter, нормализовать LF и чанки по заголовкам.
4. Построить `pages`, `chunks`, FTS5, wiki-рёбра и метаданные проекции.
5. Строить новую базу рядом с последней валидной и заменять её атомарно.
6. Реализовать `query`, `neighbors`, `impact` и проверку полного rebuild.

### Phase 3. Векторная возможность с контролируемым использованием

1. Реализовать интерфейс поставщика эмбеддингов и версионирование модели.
2. Добавить один одобренный рабочий адаптер; для внешнего поставщика до
   передачи корпуса получить отдельное решение PM о конфиденциальности и
   стоимости.
3. Сохранить векторы по хэшу чанка и поддержать полную переиндексацию.
4. Реализовать косинусное ранжирование и гибридное RRF-слияние.
5. Добавить `vector status`, `vector build` и отказ hybrid-запроса при
   неполном индексе.
6. Проверить механику на фиксированных тестовых векторах.
7. Сделать `kb query --mode hybrid` доступным при готовом индексе и добавить
   сравнение FTS/hybrid в `kb eval`.
8. Начать с `search.mode: fts`, затем при приёмке выбрать режим по принятому
   сравнительному критерию и сохранить результат в канонической конфигурации.

### Phase 4. Постоянный контур качества

1. Реализовать три уровня результатов: ошибка, предупреждение, рекомендация.
2. Добавить проверки уникальности, ссылок, достижимости, дублей,
   `contradicts`, `covers`, `source-drift` и `desc-mismatch`.
3. Доказать, что разрешённые и отложенные противоречия одинаково
   восстанавливаются после удаления SQLite.
4. Реализовать `kb eval`, hit-rate top-3 и отчёт промахов.

### Phase 5. Интеграция Cubica и стартовый корпус

1. Прочитать `PROJECT_STRUCTURE.yaml` и связать узлы с концептами через
   deepest-match для `covers`.
2. Создать 10–15 страниц по доменам, ключевым компонентам и ограничениям.
3. Перенести `PROJECT_ARCHITECTURE.md` в вики, обновить все входящие ссылки и
   временно оставить совместимый указатель только при реальной необходимости.
4. Обновить `AGENTS.md`, `PROJECT_OVERVIEW.md`, архитектурную навигацию и
   инструкции процесса после появления рабочего CLI.
5. Выполнить полный `sync`, `lint` и `eval` на реальном стартовом корпусе.

### Phase 6. Минимальная AST-проекция

1. TypeScript Compiler API извлекает пакеты, модули, `imports`, `reexports` и
   строковые динамические импорты.
2. AST-узлы имеют отдельное пространство идентификаторов и `origin: ast`.
3. Единственный мост к вики — `covers`; `imports` не создаёт `depends_on`.
4. `kb impact` показывает обратные импорты и смысловых соседей раздельно.
5. Сырые AST-узлы и код не попадают в векторный индекс.

## Acceptance

- Все схемы имеют положительные и отрицательные тесты и исполняются Ajv.
- Две полные пересборки неизменного корпуса логически эквивалентны.
- Удаление `.tmp/project-knowledge/` не теряет разрешённые предупреждения.
- FTS5 capability probe доказан положительным и отрицательным тестом.
- Выбранный режим по умолчанию имеет top-3 hit-rate не ниже 0,8, превосходит
  альтернативу либо остаётся FTS при отсутствии доказанного выигрыша hybrid и
  не теряет критические точные запросы.
- Векторное построение и hybrid-запрос проходят тесты, но обычный запрос в
  режиме `fts` не обращается к векторному пути.
- Смена модели/размерности требует полной векторной переиндексации.
- `PROJECT_STRUCTURE.yaml` читается без изменения формата и генератора.
- Полная пересборка AST-проекции детерминированно восстанавливает пакеты,
  модули и статические рёбра импортов с отдельным `origin: ast`.
- `kb impact` показывает обратные импорты и wiki-связи раздельно; `imports`
  не создаёт `depends_on`.
- После слияния двух фикстур полный lint видит межветочный дубль и открытое
  `contradicts`.
- В исполняемых контрактах и CLI отсутствуют поля или команды `KB-Actor`,
  постоянная очередь и обязательный `rename-candidate`; документация упоминает
  их только как явно отклонённые варианты.

## Validation

- Сфокусированные Node unit tests CLI, ранжирования, JSON Schema и lint.
- E2E: чистая `.tmp/` → `sync` → `query` → ожидаемая страница.
- E2E: две веточные фикстуры → объединённый корпус → полный lint.
- E2E: векторный индекс готов, `mode: fts` не использует его; `mode: hybrid`
  выдаёт ожидаемое RRF-слияние.
- E2E: изменение импорта обновляет только производный AST-граф и отражается в
  отдельной секции `kb impact`, не изменяя Markdown-граф.
- `node scripts/dev/generate-structure.js` после структурных изменений.
- `git diff --check` и проверка внутренних ссылок документации.
- Полный `verify:canonical` только перед итоговой приёмкой реализации, потому
  что изменение затрагивает общие инструменты, схемы и входной контекст агентов.

## Risks

- Нативная зависимость SQLite может не иметь подходящей готовой сборки на
  отдельном хосте; установка и FTS5 проверяются до написания основной логики.
- Реализация векторного контура до назначения режимом по умолчанию имеет
  стоимость; ранний риск ограничивается явными запросами, сравнительным
  `kb eval` и локальным линейным поиском по малому корпусу.
- Внешний embedding provider может раскрыть код; по умолчанию он не настроен.
- Посредственный стартовый корпус ухудшит доверие к системе; страницы создаются
  небольшими итерациями и принимаются через эталонный проверочный набор.
- Перенос архитектурного обзора может сломать ссылки; миграция выполняется
  атомарно и проверяется `rg`.
- Вычисляемые динамические импорты нельзя достоверно вывести статически; первая
  версия индексирует только обычные импорты и строковые динамические пути.
