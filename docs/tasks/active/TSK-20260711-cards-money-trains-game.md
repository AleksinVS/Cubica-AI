## Understanding

Работа понята так: реализовать «Карты, деньги, поезда» как эталонную фасилитируемую бизнес-игру Cubica, начиная с одного законченного демонстрационного хода и постепенно доводя один континент до полной партии, которую завершает ведущий без заранее заданного лимита ходов. Параллельно закрываются только те общие архитектурные пробелы, без которых конкретный очередной срез нельзя реализовать чисто. Учебная методика, работа ведущего, рефлексия и объяснимость последствий являются обязательной частью продукта, а не документацией «после игры».

## Why

Игра проверяет стратегические свойства платформы на реальном сценарии:

- развитие от учебной методики к исполнимым правилам;
- разные типы взаимозависимых команд в одной сессии;
- общий изменяемый рынок и транспортная сеть;
- сочетание переговоров человека и проверяемых цифровых операций;
- ведущий, ассистент и методические паузы;
- отделение универсальных механизмов от контента конкретной игры;
- один источник истины логики в манифестах и переносимая UI-проекция.

Пользовательский результат первой очереди: ведущий может провести демонстрационный ход на общей цифровой карте и объяснить участникам все изменения через журнал. Результат полной одноконтинентальной очереди: очная группа проходит полный игровой день без ручного пересчета состояния вне системы, кроме живых переговоров и методической работы ведущего.

## Confirmed Product Decisions

- Первая версия работает на одном общем экране ведущего.
- Недостающий авторский контент будет предоставлен отдельно.
- Дорога стоит 2 монеты за каждый сегмент в пределах области; граница области начинает новый сегмент.
- Штрафов за ошибочную отчетность нет.
- Новости выбираются случайно воспроизводимым runtime-механизмом.
- Заранее заданного числа ходов нет; ведущий завершает сессию явной кнопкой с подтверждением.
- Второй континент не нужен и исключен из продукта.
- Phaser принят для поля в гибридной границе ADR-073.

## Architecture

### Уже принятые зависимости, активируемые этой игрой

- ADR-012: учебные метаданные и методические материалы.
- ADR-030: редактируемый authoring-слой и компиляция runtime-манифестов.
- ADR-041: объектное состояние терминалов, дорог, техники, грузов и проектов строительства.
- ADR-053/054/055: панели, граница смысла и отображения, декларативные UI-действия.
- ADR-058, только необходимые фазы: состояние по командам, фазы хода, переводы денег, воспроизводимые колоды. Полный пакет для «Монополии» не реализуется автоматически.
- ADR-061: проверяемые параметры действий для сумм и выбора из ограниченного набора.
- ADR-063: доставка фоновой карты и игровых ассетов из пакета игры.
- ADR-062: общий лениво загружаемый Phaser-host; Draft ADR-073 отделяет пошаговое поле от realtime `simulationSurface`.

### Новые решения, требующие согласования

- Draft ADR-071: ведущий, ассистент, наблюдатель, права на действия, методические паузы и журналируемые интервенции.
- Draft ADR-072: декларативная транспортная сеть, движение, закрытия, строительство и кратчайший маршрут.
- ADR-073: Phaser для интерактивной карты при сохранении панелей, расчетов и методики в обычном web UI.

### Обязательные границы

- В generic-коде нет `if` по id игры, цветам команд, терминалам или странам.
- Схемы JSON являются источником истины; ручные TypeScript-проверки не заменяют их.
- Runtime владеет деньгами, фазами, сетью, грузами и техникой; plugin/UI владеет только взаимодействием и отображением.
- Координаты карты находятся в web UI-слое; правила пути не выводятся из изображения.
- Общая возможность считается доказанной только после нейтральной фикстуры.
- Любой временный ручной шаг фиксируется как ограничение с планом снятия, а не скрывается под видом готовой автоматизации.

## Scope

### Первая очередь: один континент

- Полная оцифровка правил и доступного контента одного континента.
- Авторский пакет `games/cards-money-trains/`: authoring, runtime/UI manifests, методика, карта, design artifacts, fixtures и project-local plugin.
- Демонстрационный ход по `GSR-030`.
- Полный цикл одного обычного хода.
- Покупка/продажа техники, загрузка, движение, доставка, расчеты, строительство, новости и результаты.
- Неограниченная заранее последовательность ходов до решения ведущего, две промежуточные и одна финальная рефлексия.
- Два независимых победителя по типам команд.
- Один локальный общий экран ведущего с Phaser-полем и web-панелями.

### Платформенные доработки

- Минимальные части принятых задач платформы, реально используемые текущим срезом.
- Реализация ADR-073 и принятых после решения PM ADR-071/072.
- Нейтральные контрактные и runtime-фикстуры.
- Editor readiness: схемы, source maps, фикстуры состояния и preview без game-specific редактора.

## Non-Goals

- Второй континент и режим до 80 участников — исключены из целевого продукта по решению PM.
- Сетевые устройства команд и удаленная сессия.
- ИИ-команды — отдельная модель после стабильной человеческой партии.
- Автоматическая психологическая оценка участников.
- Произвольное рисование железной дороги по пикселям в первом срезе.
- Штрафы за ошибочную отчетность.
- Универсальный графовый язык запросов, workflow engine или отдельный сервис Game Engine.
- Публичный портал, покупки и лицензии, если PM отдельно не включит удаленный запуск в цель первой версии.
- Переписывание всего пакета настольных возможностей ADR-058 «на будущее».

## Plan

### Фаза 0. Закрыть продуктовые и архитектурные входы

1. Получить недостающие колоды, описания стран, стартовую сеть и данные грузов.
2. Уточнить, строятся ли дороги свободной линией или выбираются из авторского каталога маршрутов.
3. Подтвердить начало новостей и обслуживания с первого игрового хода по приоритетному сценарию.
4. Рассмотреть Draft ADR-071 и Draft ADR-072; после решения синхронно обновить их статус и `PROJECT_ARCHITECTURE.md`.
5. Зафиксировать согласованную цифровую топологию, геометрию областей и provenance всех ассетов.

Критерий выхода: нет домыслов в правилах первого среза; архитектурные публичные контракты согласованы.

### Фаза 1. Минимальный платформенный фундамент демонстрационного хода

1. Реализовать состояние команд и необходимую player/team-проекцию из принятого пакета пошаговых игр.
2. Реализовать минимальную фазовую модель, которая допускает ведущего и несколько команд одного типа в общей фазе.
3. Реализовать переводы денег и атомарные экономические операции.
4. Реализовать минимальную сеть ADR-072: узлы, открытое/закрытое ребро, один шаг движения, вместимость, кратчайшее расстояние.
5. Реализовать локальные роли ведущего и ассистента по ADR-071 без сетевой аутентификации.
6. Реализовать канал карты и ассетов ADR-063.
7. Реализовать общий лениво загружаемый Phaser-host и `interactiveBoardSurface` по ADR-073.
8. Для каждой общей возможности добавить нейтральную схему, позитивные/негативные фикстуры, runtime-тест и проверку проекции секретов.

Критерий выхода: нейтральная фикстура проводит абстрактный транспорт между узлами, переводит деньги и отклоняет закрытый/переполненный путь без упоминаний игры.

### Фаза 2. Вертикальный срез GSR-030

1. Создать `games/cards-money-trains/` и authoring-манифесты по актуальной схеме.
2. Описать четыре команды, фиксированную расстановку, новость `3 ↔ π`, деньги, технику и грузы демонстрации.
3. Добавить карту через реестр ассетов и web UI-проекцию координат.
4. Реализовать Phaser-плагин поля: узлы, дороги, локомотивы, вагоны, camera/zoom, drag/select и выделение допустимых целей.
5. Реализовать обычные web-панели ведущего: этап, деньги, карточки, подтверждения, журнал и методические подсказки.
6. Провести демонстрационный ход до итоговой таблицы и закрепить replay-транскрипт.
7. Добавить состояние демонстрации в editor fixtures и проверить preview.

Критерий выхода: критерии `GSR-030` и его сквозной browser-тест зелёные.

### Фаза 3. Первый обычный ход

1. Настраиваемое создание команд двух типов и стартовый инвентарь.
2. Покупка и продажа локомотивов/вагонов.
3. Выбор двух грузов, приоритет по деньгам/вагонам, удержание груза и заявка на перегон.
4. Прицепление, флаги ограничений, пять единиц действия, остановка и продолжение состава.
5. Автоматическое отцепление, выплата банка и плата паровозной гильдии.
6. Общая таблица и явное завершение каждой фазы ведущим.

Критерий выхода: один реальный ход с несколькими вариантами решений проходит без ручного изменения состояния.

### Фаза 4. Стратегическое строительство и изменяемый рынок

1. Допустимые дороги, места полустанков и стоимость 2 монеты за каждый пересеченный сегмент области.
2. Совместное финансирование несколькими командами.
3. Двухходовое строительство, закрытие концов и последующее открытие.
4. Новости, меняющие дороги, тарифы, цены или доступность только через объявленные эффекты.
5. Истощение старых терминалов и появление новых возможностей в соответствии с авторским балансом.
6. Платное совещание и журнал договоренностей без попытки формализовать живые переговоры.

Критерий выхода: команды могут выбрать краткосрочную перевозку либо вложение в будущую сеть, а последствия наблюдаемы через несколько ходов.

### Фаза 5. Полная учебная сессия одного континента

1. Полные колоды новостей и грузов с воспроизводимым случайным порядком.
2. Неограниченный заранее цикл ходов и доступная только ведущему кнопка «Завершить игру» с подтверждением.
3. Первая, вторая и финальная рефлексия как методические контрольные точки.
4. Роли внутри команд, инструкции участника и ведущего.
5. Объявленные интервенции для сложных случаев с причиной и аудитом.
6. Финальный расчет: деньги + техника по номиналу, два победителя, деролинг и итоговые вопросы.
7. Фикстуры начала, середины, точки невозврата и завершения партии.

Критерий выхода: ведущий проводит полную одноконтинентальную сессию по сценарию без внешней таблицы состояния и завершает ее по собственному решению.

### Фаза 6. Приемка и закрытие

1. Полный прогон контрактных, runtime, player, editor и browser-проверок.
2. Нейтральные фикстуры доказывают каждую новую общую возможность.
3. Проверка отсутствия id и терминов игры в platform core.
4. Синхронизация архитектуры, GSR, методики, `NEXT_STEPS.md` и реестра долга.
5. Удаление временных извлечений материалов из `.tmp/`.

## Acceptance

### Продукт

- Полная одноконтинентальная партия проводится до явного решения ведущего о завершении.
- Все заявленные правила из согласованной матрицы источников либо реализованы, либо явно вынесены за границы версии.
- Ведущий, ассистент, паровозные гильдии и логистические компании получают разные понятные интерфейсы и права.
- Определяются два победителя; итоговый расчет объясним и воспроизводим.
- Карта, составы, грузы, строительство, новости и таблица результатов не требуют параллельного ручного учета.

### Учебная методика

- Каждая компетенция связана с игровыми ситуациями, наблюдаемыми индикаторами и вопросами рефлексии.
- Система помогает ведущему замечать последствия, но не выдает непрозрачную автоматическую оценку личности.
- Методические паузы и интервенции являются явными событиями сессии.
- Инструкции участника и ведущего версионируются вместе с игрой.

### Архитектура

- Authoring JSON — редактируемый источник; runtime/UI JSON генерируется и валидируется по схемам.
- Общие возможности не содержат данных игры и имеют нейтральные тесты.
- Runtime является единственным владельцем игрового состояния и правил; UI-плагин не рассчитывает деньги и допустимость движения.
- Фоновая карта доставляется через платформенный реестр ассетов, а не из `apps/player-web/public`.
- Нет незарегистрированного архитектурного дрейфа или временных ручных обходов.
- Phaser не владеет правилами и применяет новое положение объектов только из подтвержденного runtime snapshot.

## Validation

Точный набор команд уточняется по мере реализации, но итоговый контур обязан включать:

```text
npm run generate:contracts
npm run verify:contracts-schema-parity
npm run verify:manifest-authoring
npm run verify:canonical
npm test --workspace services/runtime-api
npm test --workspace @cubica/player-web
node scripts/dev/generate-structure.js --check
git diff --check
```

Дополнительные проверки:

- replay демонстрационного и полного контрольного сценария;
- негативные тесты закрытой дороги, переполненного терминала, недостатка денег и неверной роли;
- browser e2e демонстрационного хода и сокращенной полной партии на фикстурах;
- поиск `cards-money-trains`, названий стран и цветов команд в `services/runtime-api/src` и generic `apps/player-web/src` должен быть пустым, кроме тестов, загружающих игру как данные;
- визуальная проверка карты на широком и узком экране;
- проверка повторного монтирования/размонтирования Phaser-сцены без накопления обработчиков и памяти.

| Правила движения окажутся во frontend-плагине | Топология и проверки принадлежат runtime по ADR-072; UI хранит координаты. |
| Свободное физическое строительство невозможно точно перенести в цифру | PM согласует кандидатные дороги/места; произвольная геометрия не маскируется как готовая. |
| Полный e2e займет восемь часов | Начало, середина и конец покрываются фикстурами; полный replay выполняется без реальных пауз. |
| Сетевой режим резко увеличит первую очередь | Решение о канале принимается до реализации; хотсит остается полноценной первой моделью. |
| Phaser увеличит загрузку и потребление памяти | Лениво загружать движок только для Phaser-игр, не включать физику без необходимости, проверять bundle budget и lifecycle cleanup. |
