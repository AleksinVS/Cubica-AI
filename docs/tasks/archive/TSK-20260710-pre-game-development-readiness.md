## Why

Архитектурное ревью показало, что игровое ядро Cubica достаточно развито для реализации конкретной игры, но репозиторий пока не дает агенту надежную точку старта. Основные причины: незавершенная сверка нового workflow, красная каноническая проверка, падающий браузерный CI, устаревшие машинные индексы и очередь, а также отсутствие технической защиты основной ветки.

Эта задача закрывает только подготовительный барьер перед планированием первой игры. Она не развивает игровые возможности и не пытается заранее проектировать все производственные контуры платформы.

## Scope

В работу входят четыре результата.

1. Завершить проверку уже реализованного workflow:
   - проверить согласованность `AGENTS.md`, `CLAUDE.md`, `.codex/skills/AGENTS.md`, ADR о человеко-утверждаемом автономном процессе, формата задач и локального навыка;
   - убедиться, что старые `wf-*`, прежний общий `$cubica` и удаленный `PROJECT_WORKFLOW_CONFIG.json` доступны только как архив, а не как активные правила;
   - исправлять только фактические противоречия, не менять принятое распределение полномочий и не включать `$cubica` для этой задачи.
2. Вернуть зеленую техническую точку отсчета:
   - заменить устаревшее серверное действие `showHint` в smoke-сценарии на реальное разрешенное действие канонической игры;
   - воспроизвести падение `player-web e2e` в режиме CI, определить причину и исправить ее без произвольных задержек и маскирующих повторов;
   - получить зеленые канонические и браузерные проверки на текущем рабочем дереве.
3. Синхронизировать машинный контекст и очередь:
   - актуализировать каталоговые `.desc.json` и регенерировать `PROJECT_STRUCTURE.yaml`;
   - привести активные TSK к каноническим статусам;
   - перенести фактически завершенные или замененные TSK из `docs/tasks/active/` в `docs/tasks/archive/` с сохранением истории;
   - оставить в `NEXT_STEPS.md` только правдивые текущие состояния и зависимости;
   - добавить или расширить детерминированную проверку, которая предотвращает повторное расхождение структуры, статусов и доски.
4. Защитить ветку `main`:
   - обновить проектную документацию обязательных GitHub-проверок;
   - применить через GitHub API правило, требующее pull request, актуальную ветку и успешные пять существующих CI-проверок;
   - распространить правило на администратора, запретить force-push и удаление `main`;
   - не вводить обязательное число человеческих одобрений, так как текущая цель — автоматические проверки и запрет прямого изменения, а не новая модель ручной приемки.

Не входят в работу:

- использование навыка `$cubica`;
- развитие адаптера внешних навыков и реализация ADR-069;
- изменение архитектуры игр, runtime, редактора, публикации, доступа или платежей;
- реализация первой игры;
- коммит, push, merge, удаление пользовательских изменений или переписывание истории Git;
- изменение секретов или вывод их содержимого;
- ослабление проверок ради зеленого результата.

## Architecture Boundary

Новых архитектурных решений в этой задаче не ожидается.

- Проверка workflow подтверждает уже принятое решение: `$cubica` включается только по прямому указанию и в своих границах требует утверждения общего плана. Это требование не распространяется на обычный режим разработки через конкретную игру.
- Исправление smoke и e2e восстанавливает заявленное поведение и проверки, не меняя публичные контракты.
- Очистка индекса и очереди приводит документацию к фактическому состоянию.
- Защита `main` реализует уже задокументированный принцип trunk-based development: короткие ветки, pull request и обязательные проверки.

Если для исправления потребуется изменить публичную схему, API, распределение архитектурных полномочий или состав продуктовых возможностей, исполнение соответствующей части останавливается и возвращается владельцу как новое решение.

## Plan

### Phase 0. Стабильный снимок и защита чужой работы

1. Убедиться, что другой пишущий агент больше не работает в репозитории.
2. Сохранить перечень исходных изменений и не откатывать файлы, которых касается незавершенный адаптер внешних навыков.
3. Прочитать ближайшие `AGENTS.md` для каждой реально изменяемой подсистемы.
4. Для библиотек, GitHub API и Playwright использовать актуальную документацию; не полагаться на устаревший синтаксис по памяти.

### Phase 1. Проверка workflow

1. Выполнить структурную сверку активных правил, архива, ADR и task-формата.
2. Проверить, что `$cubica` включается только по прямому запросу и не активирован этой задачей.
3. Проверить отсутствие активных ссылок на старые роли и артефакты за пределами разрешенного архива и исторических записей.
4. Исправить только подтвержденные расхождения и зафиксировать доказательства в `Handoff Log`.

### Phase 2. Зеленая база

1. Воспроизвести smoke-дефект и зафиксировать ожидаемое поведение разрешенного manifest action.
2. Исправить smoke-сценарий так, чтобы он проверял создание сессии, отправку реального runtime-действия и сохраненное изменение состояния, не возвращая UI-only `showHint` на сервер.
3. Запустить браузерную команду в CI-режиме, получить точный failing test и диагностические артефакты Playwright.
4. Исправить корневую причину. Предпочитать web-first assertions, корректную готовность серверов и стабильный production-mode запуск; retry допустим только как диагностическая страховка CI, а не как замена исправлению.
5. Запустить точечные проверки, затем полный `verify:canonical` и точную браузерную команду, используемую обновленным CI.

### Phase 3. Машинный контекст и очередь

1. Инвентаризировать все TSK по реальному состоянию, не выводя статус только из произвольного исторического слова.
2. Перенести подтвержденно завершенные и замененные задачи в архив; сохранить исходное содержимое и историю передачи.
3. Нормализовать оставшиеся активные статусы до формата `planned | awaiting_approval | approved | in_progress | blocked | review`.
4. Не менять статус и содержимое `TSK-20260710-external-skill-adapter`, кроме необходимой корректной ссылки на доске: эта работа принадлежит другому исполнителю.
5. Обновить `.desc.json`, `NEXT_STEPS.md`, ссылки на перемещенные задачи и `PROJECT_STRUCTURE.yaml`.
6. Добавить детерминированную проверку согласованности или документировать, какой существующий gate теперь обеспечивает ее полностью.

### Phase 4. Защита `main`

1. Проверить GitHub-аутентификацию и текущие настройки без вывода токенов или секретов.
2. Синхронизировать `.github/required-checks.md` с фактически существующими job names.
3. После локального зеленого результата применить branch protection или repository ruleset со следующими инвариантами:
   - изменение `main` только через pull request;
   - strict required status checks, то есть проверка актуального состояния с целевой веткой;
   - обязательные контексты: `legacy/stub gate`, `manifest authoring gate`, `canonical verification`, `portal rule tests`, `player-web e2e`;
   - применение к администратору;
   - запрет force-push и удаления ветки;
   - разрешение merge только после закрытия обсуждений;
   - ноль обязательных человеческих approvals, если GitHub требует числовое значение.
4. Повторно прочитать настройку через API и записать очищенный от чувствительных данных результат в `Handoff Log`.

## Acceptance

### Workflow

- Активные инструкции однозначно указывают на один текущий workflow и один формат TSK.
- `$cubica` нигде не активируется по сходству запроса; эта задача не использует его.
- Устаревшие `wf-*`, прежний общий `$cubica` и старые артефакты доступны только в архиве или явно историческом контексте.
- Незавершенная работа адаптера внешних навыков не перезаписана и не принята в рамках этой задачи.

### Green baseline

- Smoke runtime-api использует разрешенное действие из манифеста и проверяет наблюдаемое изменение состояния.
- UI-only `showHint` остается запрещенным для runtime API.
- `npm run verify:canonical` завершается с кодом 0.
- Полный Playwright-набор, который запускает CI, завершается с кодом 0 в эквивалентном локальном режиме.
- Исправление e2e не основано только на увеличении timeout, sleep или безусловных retry.

### Machine context and queue

- `PROJECT_STRUCTURE.yaml` соответствует текущим `.desc.json` и повторная генерация не создает нового различия.
- Все TSK в `docs/tasks/active/` имеют канонический активный статус; завершенные и замененные задачи находятся в архиве.
- `NEXT_STEPS.md` не содержит выполненные задачи в `Now` и правдиво показывает реально выполняющиеся или проверяемые работы.
- Автоматическая проверка обнаруживает как минимум неканонический статус, завершенную задачу в `active/`, несуществующую ссылку доски и дрейф структурного индекса.

### GitHub protection

- GitHub API больше не отвечает `Branch not protected` для `main` либо активный repository ruleset эквивалентно покрывает `refs/heads/main`.
- Прямой push, force-push и удаление `main` запрещены, включая администратора.
- Pull request обязан получить все пять существующих CI-контекстов на актуальном состоянии ветки.
- Локальная документация перечисляет те же обязательные контексты, что и GitHub.
