# FEATURE: Протокол взаимодействия Model-View-Presenter

## Контекст и цели
В рамках архитектуры MVP + LLM-first необходимо чётко определить, как компоненты (Model, View, Presenter) взаимодействуют друг с другом.
Цель — стандартизировать потоки данных и команд, чтобы любой разработчик игры понимал, как передавать состояние в LLM, как получать обновления модели и как управлять отображением.
Важно учесть, что события могут инициироваться не только пользователем, но и игровым движком (таймеры, NPC, системные триггеры).

## Цели

- [x] Описать интерфейс взаимодействия Presenter и View.
- [x] Определить формат данных (JSON-схемы) для запросов и ответов в рамках этого взаимодействия.
- [x] Обеспечить поддержку системных событий (System Events) наравне с пользовательскими.

## Объём

In scope:
- Описание потока данных: User/System Input -> Presenter -> LLM -> Presenter -> View / Model.
- Формат сообщений (Message Protocol):
  - Запрос к LLM: тип события (User Action / System Event), payload, текущее состояние, манифест.
  - Ответ от LLM: измененное состояние (State Patch), инструкции для UI (UI Instructions).
- Обработка ошибок и валидация ответов LLM.

Out of scope:
- Реализация конкретного транспорта (HTTP/WebSocket) — это детали реализации, здесь важен логический протокол.
- Дизайн конкретных экранов (View) для игр.
