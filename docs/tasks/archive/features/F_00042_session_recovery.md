## Описание
Реализация механизма восстановления игровых сессий в случае сбоев: падение сервиса, таймаут ответа от LLM (большой языковой модели) или перезагрузка компонентов Backend.

## Проблема
Текущая архитектура (ADR-005) использует пессимистичную блокировку. Если процесс упадет во время удержания блокировки или LLM не ответит, сессия может "зависнуть": новые запросы будут постоянно получать ошибку блокировки или бесконечную загрузку, а состояние может стать несогласованным.

## Цели
- [ ] Сессия не должна оставаться заблокированной навсегда при сбое или зависании обработки хода.
- [ ] При сбое состояние сессии должно оставаться согласованным и восстановимым из последней сохранённой версии.
- [ ] Пользователь должен получать понятное сообщение об ошибке и/или о восстановлении сессии, а не бесконечный индикатор загрузки.

## Scope
- In scope:
  - Описание схемы "checkpointing" (сохранение устойчивого состояния перед рискованной операцией, такой как вызов LLM или внешнего сервиса).
  - Определение TTL (time-to-live — время жизни) для блокировки сессии и правил обнаружения "мертвых" блокировок.
  - Обновление ADR-005 с явной стратегией восстановления сессии после сбоя.
  - Добавление контрактов (интерфейсов) в Router для работы с блокировками и восстановлением.
- Out of scope:
  - Полная реализация распределённого lock-сервиса (Redis, отдельный сервис locking).
  - Нагрузочное тестирование и детальная оптимизация производительности.

## Задачи
1. Архитектура "checkpointing":
   - [ ] Определить, что считается "устойчивым" состоянием для сессии (последний успешно подтверждённый ход).
   - [ ] Описать в ADR-005, когда именно фиксируется checkpoint (до/после вызова LLM, при смене статуса события и т.п.).
2. Dead lock detection / TTL:
   - [ ] Определить разумный TTL для блокировки сессии и условия обработки таймаута.
   - [ ] Описать механизм обнаружения "мертвых" блокировок (по времени, по отсутствию прогресса).
   - [ ] Описать, как выполняется безопасное снятие блокировки и перевод действия в состояние ошибки.
3. Обновление ADR-005:
   - [ ] Добавить раздел про Session Recovery (checkpointing, TTL, поведение при сбоях).
   - [ ] Согласовать терминологию и схему полей `game_sessions` с ADR-011 (например, использовать `state_version` вместо абстрактного `version`).
4. Контракты в Router:
   - [ ] Описать интерфейсы для менеджера блокировок и сервиса восстановления в `services/router` (без реальной БД).
   - [ ] Добавить простую in-memory реализацию и smoke-тесты как пример использования.

## Acceptance-Criteria
- [ ] ADR-005 описывает:
  - [ ] Что такое checkpoint для сессии и когда он создаётся.
  - [ ] Как работает TTL для блокировок и как определяется "мертвая" блокировка.
  - [ ] Как пользователь информируется о проблеме и восстановлении (общий паттерн сообщений).
- [ ] В `services/router` добавлены контракты для Session Recovery (интерфейсы и пример реализации).
- [ ] ExecPlan `CP_00042_session_recovery.yaml` поддерживается в актуальном состоянии и описывает путь к рабочей реализации.
