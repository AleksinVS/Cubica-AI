## 1. Источник и методика извлечения

Источник — `draft/Antarctica/Game.html` (~160 КБ, один HTML-файл; кодировка
UTF-8 с BOM; jQuery + Bootstrap 3, без сборки, запускается прямо из
`file://`). **Монолит** здесь и далее — этот единый файл, где вёрстка,
данные сценария и код-интерпретатор перемешаны в одном `<script>`-блоке, в
отличие от целевой платформенной архитектуры, где данные (манифест) и код
(движок платформы) разделены.

Все числа в этой спецификации (счётчики карточек, метрик, окончаний)
получены не ручным подсчётом по тексту, а программным разбором: единственный
`<script type="text/javascript">`-блок файла выполнен в изолированной
песочнице Node.js (модуль `vm`) с заглушками jQuery/DOM (без реального
рендеринга страницы), после чего из песочницы забраны реальные
JS-объекты `cardsObj` (все карточки/экраны) и `game` (состояние + таймлайн).
Скрипты лежат в рабочем каталоге блока:

- `.tmp/agent-workflow/TSK-20260719-antarctica-alignment/2.1/extract.js` —
  выполняет код монолита в песочнице, считает карточки по типам, сверяет
  идентификаторы `timeline` и `cardsObj` друг с другом, сохраняет
  `extracted.json`.
- `.tmp/agent-workflow/TSK-20260719-antarctica-alignment/2.1/build-spec.js` —
  проецирует `extracted.json` в форму, заданную
  `logic-projection-schema.md`, нормализует HTML-разметку в текстах экранов,
  разворачивает условные конструкции в читаемые `when`/`then`.

Способ подсчёта карточек: `Object.keys(cardsObj).length` по реально
выполненному объекту — это исключает ошибки ручного счёта фигурных/квадратных
скобок в плотном, неформатированном тексте данных.

## 2. Соотношение Game.html и GameFull.html

В каталоге `draft/Antarctica` лежат два файла: `Game.html` и `GameFull.html`.
Прямое сравнение (`diff`) сначала показывало полное расхождение всех строк —
причина оказалась технической: `GameFull.html` хранится с переносами строк
CRLF, а `Game.html` — с LF. После нормализации переносов строк diff даёт
всего 11 блоков различий, и **все 11 — это англоязычные поясняющие
комментарии** (в стиле docstring), присутствующие только в `Game.html`
поверх идентичного кода и данных. Ни одной строки, уникальной для
`GameFull.html`, не найдено.

Вывод: файлы **байт-идентичны по игровым данным и логике** (`cardsObj`,
`timeline`, `game.operate`/`operateArray`/`next`, `setVars` — везде совпадают
дословно); различие — только в объёме служебных комментариев. Ни один из
файлов не является «более полным» по данным в смысле задания блока 2.1.
Формально `GameFull.html` старше по дате (17.02.2023), `Game.html` — новее
(18.03.2023, той же датой, что и остальные ассеты каталога — `img/`,
`jquery.flip.min.js`, `penguines.css`), и именно на `Game.html` ссылается
`draft/Antarctica/README.md` как на файл для запуска.

**Решение**: эталоном данного извлечения выбран `Game.html`. `GameFull.html`
зафиксирован как байт-идентичный по логике дубликат без дополнительных
комментариев; отдельно не разбирался, так как это означало бы дублирование
работы без дополнительной информации.

## 3. Обзор движка

Движок — самодельный интерпретатор (не универсальный, написан специально для
этой игры), целиком реализованный внутри `<script>`-блока `Game.html`
(строки ≈445–1274). Он поддерживает четыре основных функции:

| Функция | Назначение |
|---|---|
| `game.operate(data)` | Выполняет одну операцию псевдоязыка (см. раздел 4). Псевдоязык здесь — небольшой самодельный формат условий и действий на массивах, а не полноценный язык программирования. |
| `game.operateArray(dataArr)` | Применяет список операций `game.operate` последовательно, без остановки на первом сработавшем условии (см. раздел 12 про известный TODO в исходнике). |
| `game.next()` | Главный цикл шага: продвигает курсор по `game.timeline`, определяет тип текущего экрана (`cards`/`team`/`info`) по типу первого элемента блока и рендерит нужный шаблон, назначая обработчики кликов. |
| `setVars()` | Синхронизирует видимые в интерфейсе счётчики (навигационная панель) с внутренними значениями `game.<metric>`; итоговый счёт («Остаток дней») вычисляется на лету как `Math.round(60 - game.time)`, а не хранится отдельным полем. |

Состояние партии — единственный глобальный объект `game`, поля которого:

- `timeline` — трёхмерный массив: `[сюжетная линия][шаг][элемент cardsObj]`,
  готовый порядок прохождения игры (подробнее — раздел 9);
- `line` — индекс активной сюжетной линии (`0` — основная, `1` — «проигрышная» эпилог-ветка);
- `cursor` — индекс текущего шага внутри активной линии;
- `curent` (так в исходнике, с опечаткой) — идентификатор элемента `cardsObj`, который считается «текущим» для рендера (может быть переопределён функциями `swap`/`goTo`, см. раздел 5);
- `cards` — ссылка на `cardsObj` (все карточки и экраны игры);
- `log` — единый массив истории: в него пишутся **и** идентификаторы сыгранных карточек, **и** идентификаторы выбранных персонажей команды (см. раздел 7) — это один и тот же массив, что важно для операции `log in`/`out` (раздел 4);
- 7 метрик (`pro`, `rep`, `lid`, `man`, `stat`, `cont`, `constr`) + `time` + мёртвое поле `score` — раздел 6.

## 4. Псевдоязык операций (`game.operate`)

**Псевдоязык** здесь — плоский формат «кортежей» (JS-массивов фиксированной
формы), которыми в данных карточек описаны условия и эффекты, без отдельного
парсера — это буквально ветвление `if/else if` внутри `game.operate` по
второму элементу кортежа. Каждая карточка/экран хранит такие кортежи в полях
`action` (эффекты при выборе) и `alt` (условные подмены самого экрана).

| # | Форма кортежа | Условие срабатывания | Действие | Статус в данных |
|---|---|---|---|---|
| 1 | `['func', fn, arg]` | безусловно | вызывает `fn(arg)` — см. раздел 5, список именованных функций | используется широко |
| 2 | `[metric, число]` | безусловно | `game[metric] += число` — аддитивное изменение метрики (может быть отрицательным) | базовая, основная форма эффектов |
| 3 | `[metric, '>', порог, nested]` | `game[metric] > порог` | рекурсивно выполняет `game.operate(nested)` | используется (пороговые бонусы/штрафы, ветвления) |
| 4 | `[metric, '<', порог, nested]` | `game[metric] < порог` | рекурсивно `game.operate(nested)` | используется |
| 5 | `[metric, '==', порог, nested]` | `game[metric] == порог` | рекурсивно `game.operate(nested)` | определена, но ни разу не встретилась в реальных данных карточек (проверено полнотекстовым поиском `==` внутри `action`/`alt`) |
| 6 | `[metric, '=', cap]` | `game[metric] > cap` | `game[metric] = cap` — ограничение метрики сверху (четвёртый элемент кортежа, если есть, игнорируется) | определена, но ни разу не используется в данных карточек — потенциальная неиспользуемая возможность движка |
| 7 | `['log', 'in', value, nested]` | `value` **присутствует** в массиве `game.log` | рекурсивно `game.operate(nested)` | используется (карточка 66) |
| 8 | `['log', 'out', value, nested]` | `value` **отсутствует** в массиве `game.log` | рекурсивно `game.operate(nested)` | используется (карточки 43, 44, 45, 61, 66) — `value` бывает и id карточки (`'57'`, `'62'`), и id члена команды (`'leo'`), так как это один общий массив (раздел 3) |
| 9 | `[metric, 'and', ...]` | — | **тело ветки пустое** (`{}`) — операция распознаётся диспетчером, но не реализована | не используется ни в одной карточке; заглушка/TODO автора, см. раздел 13 |

Важное поведенческое свойство: `game.operateArray` выполняет **все** элементы
переданного массива подряд, не останавливаясь на первом сработавшем условии
— так, в самом исходнике оставлен комментарий-напоминание
`.добавить останавливать при нахождении подходящей альтернативы`, то есть
сам автор считал это незавершённым, но так и не реализовал остановку.
Практическое следствие: если несколько условий в одном `action`/`alt`-массиве
верны одновременно, сработают **все** они, а не только первое совпавшее —
это не поведение по типу `if/else if` между элементами массива (внутри
одного кортежа `if/elseif` есть, а между кортежами массива — нет).

Ещё одна деталь порядка: при клике по карточке (`game.next()`, обработчик
`.front`) сначала выполняется `game.operateArray(card.b.action)` (оборот —
результат), и только **затем** `game.operateArray(card.f.action)` (лицевая
сторона) — то есть, вопреки естественному ожиданию «сначала лицо, потом
оборот», реально порядок обратный. У части карточек `f.action` как раз и
используется как «пороговый бонус», который проверяется/применяется уже
после основного эффекта оборота (например, карточка 31: `b.action` даёт
`+1 rep, +1 cont, +4 time`, а следом `f.action` проверяет `cont > 10` и, если
верно, компенсирует `time -2`).

## 5. Вспомогательные функции (`func`)

Это шесть JS-функций верхнего уровня, на которые ссылаются кортежи
`['func', <имя>, <аргумент>]`. Они — не часть псевдоязыка (это обычный JS),
но вызываются исключительно из него.

| Функция | Сигнатура | Что делает | Где используется |
|---|---|---|---|
| `goTo(args)` | `args = [curent, line, cursor]` | Жёстко переустанавливает `game.curent`, `game.line`, `game.cursor` — прыжок на произвольный шаг произвольной сюжетной линии | 1 раз: экран `i19`, при `time > 59` → `goTo(['i34_2', 1, 1])` (перескакивает сразу на 2-й шаг линии 1, минуя `i34`) |
| `swap(swapID)` | `swapID` — id элемента `cardsObj` | Меняет только `game.curent` (подмена содержимого текущего слота экрана без изменения позиции в `timeline`) | 2 раза: карточка `39` (`pro > 40` → `swap(3902)`), экран `i19` (`time < 54` → `swap('i19_1')`) |
| `unlock(unlockkey)` | id карточки | `game.cards[unlockkey].f.locked = false` + снимает CSS-класс `panel-warning` — безусловная разблокировка конкретной карточки | 2 раза: карточки `62` и `63` (обе разблокируют карточку `66`) |
| `unlock39(counter)` | число (порог) | Считает, сколько карточек **текущего экрана** уже есть в `game.log`; если осталось выбрать `<= 0` (т.е. набран порог), вызывает `unlock(39)`. Несмотря на общее имя, функция жёстко привязана к id `39` (не параметризована по цели) | 5 раз: карточки `37, 38, 40, 41, 42`, все с `counter=3` (экран `['37','38','39','40','41','42']`) |
| `f25_30(counter)` | число (порог) | Тот же счётчик выбранных карточек текущего экрана, но вместо разблокировки — добавляет кнопку «Далее» на оборот **текущей кликнутой карточки**, если набран порог | 6 раз: карточки `25–30`, все с `counter=3` |
| `f34()` | без значимого аргумента (вызывается с `''`) | Жёстко переключает `game.line = 1`, `game.cursor = -1`, вызывает `game.next()` (стартует с самого начала линии 1, т.е. с экрана `i34`) и скрывает `#cards` | 1 раз: карточка `34`, условие `stat < 25` |

## 6. Метрики

Видимые подписи взяты из навигационной панели и модального окна «Журнал
ходов» (оба места используют одинаковые русские подписи для одних и тех же
ключей состояния `game.<key>`). Ни у одной метрики в движке нет жёстко
заданных `min`/`max` — единственный встроенный механизм ограничения
(операция `=`, № 6 в таблице раздела 4) в данных ни разу не используется.

| Ключ | Подпись в UI | Начальное значение | Видима в интерфейсе | Примечание |
|---|---|---|---|---|
| `pro` | ЗНАНИЯ | 0 | да | В `README.md` монолита названа «прогресс экспедиции», но фактическая подпись в самом интерфейсе — «ЗНАНИЯ» (расхождение описания и факта, см. раздел 12) |
| `rep` | ДОВЕРИЕ | 0 | да | Отношение к «проекту» всех заинтересованных сторон (по тексту экрана `i2`) |
| `lid` | ЭНЕРГИЯ | 0 | да | Вовлечённость в проект, по тексту `i2` сильно зависит от лидерских качеств участников |
| `man` | КОНТРОЛЬ | 0 | да | Возможность координировать действия и выполнять намеченное |
| `stat` | СТАТУС | 0 | да | Понимание важности и приоритетности проекта; **служит условием ветвления на «проигрышную» линию**: карточка 34 при `stat < 25` вызывает `f34()` (раздел 5, 11) |
| `cont` | КОНТАКТ | 0 | да | Лёгкость коммуникации участников со всеми остальными |
| `constr` | КОНСТРУКТИВ | 0 | да | Способность сохранять нацеленность на результат в стрессе |
| `time` | (нет отдельной подписи) | 0 | нет, косвенно | Главный ресурс игры (по `README.md`). Сам по себе не выводится — из него на лету считается видимый счётчик «ОСТАТОК ДНЕЙ» по формуле `round(60 - time)`. Верхний предел 60 нигде не проверяется как общее условие проигрыша по всей партии — единственная проверка происходит один раз, в `alt` экрана `i19` (`time > 59`); после этой точки `time` может расти и дальше без каких-либо последствий |
| `score` | ОСТАТОК ДНЕЙ | 0 | да (но не через это поле) | Поле `game.score` объявлено и инициализировано, но нигде в движке не читается и не изменяется — **мёртвое поле**. Реально отображаемое число — `round(60 - game.time)`, пересчитываемое `setVars()` напрямую в DOM, минуя `game.score` |

отрицательным эффектом (`constr -25`), при этом он же в текстах карточек
описан как скептик-противник изменений — эффект согласован с сюжетной ролью.
Полные текстовые биографии персонажей — в `reference-logic-spec.json`
(`characters[].notes`), дословно.

## 8. Карточки и экраны

Все игровые экраны хранятся в одном объекте `cardsObj` (110 записей) с тремя
типами:

| Тип (`type`) | Количество | Что это |
|---|---|---|
| `cards` | 71 | Карточка «вопрос-ответ»: лицевая сторона (`f`) — ситуация/вопрос, оборотная (`b`) — результат выбора и эффекты. 70 из них пронумерованы `1`–`70` (основной путь игры), плюс id `3902` — альтернативный вариант карточки 39, доступный только через `swap` (раздел 5), в `timeline` напрямую не входит |
| `info` | 29 | Модальное информационное окно (блокирует интерфейс до нажатия «Далее» или закрытия), без выбора — заголовок, текст, фон (`bg`), опционально `locked` |
| `team` | 10 | Персонаж команды (раздел 7) |

Структура карточки `type: cards`:

- `f` (front, лицевая сторона): `text` — текст ситуации; необязательные
  `locked` (карточка изначально недоступна, класс `panel-warning`, снимается
  только через `func:unlock`/`func:unlock39`) и `action` (условный кортеж,
  выполняется **после** `b.action`, см. раздел 4);
- `b` (back, оборот): `text` — текст результата; `action` — список эффектов;
  `go: true` — кнопка «Далее» появляется на обороте сразу после выбора этой
  карточки (без него кнопка может быть добавлена только динамически через
  `func:f25_30`/`func:unlock39`, раздел 5). Из 71 карточки `go: true` явно
  стоит у 31 карточки — то есть у остальных 40 переход к следующему шагу
  экрана без срабатывания счётчиков `f25_30`/`unlock39` штатно не открывается
  (нужно ли это ограничение — вопрос для блока 2.3, здесь фиксируется как
  факт);
- `alt` — необязательный массив альтернатив: список условных кортежей,
  проверяемых **до** отрисовки экрана и способных подменить показываемый
  элемент (см. раздел 5, `swap`/`goTo`). Используется только у карточки `39`
  и экрана `i19` — больше нигде.

Структура `type: info`: `title`, `text` (обычно с HTML-разметкой `<p>`,
`<br/>`, `<b>`, `<small>` — в JSON-проекции разметка нормализована: `</p>`
и `<br/>` превращены в переносы строк, остальные теги убраны с сохранением
текста), `bg` (имя файла фоновой картинки в `img/`), необязательные `img`
(текстовая подпись автора к сцене, не путать с `bg`) и `locked` (см.
раздел 11 про экран `i21`).

Полные тексты, все 110 записей целиком (нормализованные, без HTML) — в
`reference-logic-spec.json`, ключ `cards`.

## 9. Структура партии и переходы

`game.timeline` — трёхмерный массив `[линия][шаг][id элемента cardsObj]`, две
сюжетные линии:

- линия `0` (основная) — 39 шагов: вступительные `info`-экраны, чередование
  блоков по 6 карточек выбора с разделяющими `info`-экранами, один экран
  выбора команды (`type: team`, все 10 персонажей разом), заканчивается
  экраном `i21`;
- линия `1` («проигрышная» эпилог-ветка) — 3 шага: `i34`, `i34_2`, `i21`
  (тот же самый `i21`, что и в конце линии 0 — общая финальная точка).

Точка входа — `i0` (линия 0, шаг 0).

**Структура хода**: один шаг таймлайна = один экран. Для `cards`-экрана на
нём одновременно доступны несколько карточек (в данных — всегда 6, вёрстка
рассчитана на до 12 слотов); игрок может выбрать несколько карточек на одном
экране до перехода дальше — это не «одна карточка на экран». Клик по лицевой
стороне необратимо переворачивает карточку (bootstrap collapse), сразу
применяются оба массива эффектов (сначала `b.action`, затем `f.action`, см.
раздел 4), результат дублируется в журнал ходов (`#modal-history`). Переход
на следующий шаг — по кнопке «Далее», которая появляется на обороте карточки
либо сразу (`b.go === true`), либо динамически (`func:f25_30`/`unlock39`).
Для `team`-экрана — обязательный выбор ровно 5 персонажей (реализация:
`team.length < 5` разрешает добавление; кнопка «Далее» показывается только
у 5-й выбранной панели), эффекты применяются одним пакетом по общей кнопке.
Для `info`-экрана — блокирующее модальное окно, один клик «Далее» (или
закрытие) переводит на следующий шаг.

Переходы **вне** линейного порядка (`timeline[line][++cursor]`):

1. `func:swap` — подмена только содержимого текущего слота (карточка `39` →
   `3902` при `pro > 40`; `i19` → `i19_1` при `time < 54`), позиция в
   таймлайне не меняется;
2. `func:goTo` — произвольный прыжок на любую линию/шаг (`i19` → линия 1,
   шаг 1 = `i34_2`, при `time > 59`);
3. `f34()` — жёсткий переход на линию 1 **с самого начала** (шаг `-1`, то
   есть следующий `game.next()` покажет `i34`), в отличие от `goTo`,
   целящегося в произвольный шаг;
4. `func:unlock` / `func:unlock39` — снятие блокировки конкретной карточки
   (не смена экрана, но открывает доступ к контенту, который иначе не
   выбрать);
5. `func:f25_30` — не меняет доступность карточек, только досрочно
   показывает кнопку «Далее».

## 10. Случайность

**Случайных элементов в движке не обнаружено.** Проверено полнотекстовым
поиском по всему `Game.html`: вызовов `Math.random`, генераторов случайных
чисел, перемешивания (`shuffle`) массивов — ноль совпадений. Всё течение
партии полностью детерминировано выбором игрока и накопленным состоянием
метрик; раздел `random` в JSON-проекции — пустой массив (это не пропуск, а
подтверждённый факт источника, схема прямо требует не выдумывать
отсутствующие данные).

## 11. Окончания партии

| id | Условие | Результат |
|---|---|---|
| `i19` (без подмены) | Экран показывается как есть, если верно `NOT(time<54) AND NOT(time>59)`, т.е. `54 ≤ time ≤ 59` | «…не хватило времени, чтобы спасти всех пингвинов» — частичный успех/неудача. **Игра не останавливается**, продолжается на карточки 69/70 → `i20` → `i21` |
| `i19_1` | `func:swap` подменяет `i19`, если `time < 54` на момент показа экрана `i19` | «Времени было достаточно, удалось спасти всех пингвинов, и это — триумф!» — полный успех. Тот же путь дальше: 69/70 → `i20` → `i21` |
| `i34` (через `f34`) | Карточка `34`, `stat < 25` при её выборе → `f34()`: `game.line=1`, `game.cursor=-1`, `game.next()` | «Гибель неизбежна» — массовое неповиновение и бунт из-за нехватки авторитета команды. Далее по линии 1: `i34_2` → `i21` |
| `i34_2` (через `goTo` из `i19`) | Экран `i19`, `time > 59` → `goTo(['i34_2', 1, 1])` (минует `i34`) | «…не удалось спасти пингвинов…» — провал по нехватке времени. Далее `i21` |
| `i21` | Финальный элемент обеих линий; `locked: true` | «СПАСИБО ЗА ИГРУ!», текст пуст. Кнопка «Далее» (`#next`) скрыта — партия фактически останавливается здесь (единый экран благодарности, без различения степени успеха) |
| `i_timeout` | **Не наступает никогда** — id присутствует в `cardsObj` (учтён в счётчике 29 `info`-экранов), но не встречается ни в `timeline`, ни в одном вызове `swap`/`goTo`/`unlock` во всём файле | Текст идентичен `i34_2` («не хватило времени…»), тот же `bg`. Похоже на неподключённую заготовку отдельной концовки по прямому таймауту — раздел 13 |

Технический момент завершения партии: у `i21` скрыта только кнопка `#next`
(`display:none`), а кнопка-крестик закрытия модалки (`#close`) явно не
скрыта (соответствующая строка в коде закомментирована). Событие
`hidden.bs.modal` при любом закрытии модалки вызывает `game.next()` —
формально после `i21` игрок технически может закрыть модалку через крестик и
получить попытку `game.next()` за пределами массива `timeline` (курсор
уходит за последний элемент линии). Это низкоуровневая деталь реализации, не
поведение сценария, поэтому не выделена отдельной записью в `endings`
JSON-проекции, но фиксируется здесь как факт устройства источника.

## 12. Особенности и дефекты источника

Не связанные с UI/вёрсткой (это вне блока 2.1) особенности логики,
подтверждённые чтением кода и данных:

1. **Расхождение имени метрики `pro` в README и в UI.** `draft/Antarctica/README.md`
   называет `pro` «прогрессом экспедиции», но фактическая подпись в
   интерфейсе (навбар, журнал ходов) — «ЗНАНИЯ». Для этой спецификации в
   качестве `title` метрики взято то, что реально показывается игроку
   (раздел 6).
2. **Мёртвое поле `game.score`.** Инициализируется `0`, но никогда не
   читается и не пишется движком; реальный счётчик — производная величина
   от `time` (раздел 6).
3. **Мёртвый код выбора команды.** Помимо реального механизма выбора команды
   (панели `.panel` + динамическая кнопка `.teamButton`, раздел 9), в том же
   обработчике `game.next()` присутствует второй, никогда не срабатывающий
   блок кода на классах `.teambut`/`#teamselected` с необъявленными
   глобальными переменными `skills`, `teamcount`, «голым» `score`, и
   ссылками на персонажей `team.baddy`, `team.jain`, `team.john`,
   `team.griffin`, `team.alice`, `team.luis`, `team.johnatan`, `team.fred`,
   `team.nennet` — имена, которых нет среди 10 реальных членов команды.
   Похоже на остаток от более раннего/иного прототипа. Полный текст — в
   `unmapped` (раздел 13), так как описывать этот путь как часть игровой
   логики было бы домыслом.
4. **Дефект отображения истории для метрики ЭНЕРГИЯ (`lid`).** В обработчике
   сборки записи журнала при уменьшении `lid` в текст подписи вставляется
   разница `game.pro - prev.pro` (метрика ЗНАНИЯ) вместо `game.lid -
   prev.lid` — явная copy-paste-опечатка. Сама цифра-значение обновляется
   верно, дефект касается только текста «ЭНЕРГИЯ −N» в журнале ходов при
   снижении энергии.
5. **`i_timeout` — неиспользуемая заготовка концовки** (раздел 11, 13).
6. **Операция `'and'` объявлена, но не реализована** (пустое тело ветки в
   `game.operate`) и ни разу не используется в данных (раздел 4, 13).
7. **Комментарий с примером `type: 'jump'`**, которого нет ни в одной
   реальной записи `cardsObj` — судя по всему, черновая заметка о
   планировавшемся, но не реализованном обобщённом переходе между сюжетными
   линиями (раздел 13).
8. **`game.operateArray` не останавливается на первом сработавшем условии**
   (раздел 4) — сам автор оставил в коде TODO-комментарий об этом,
   реализация так и осталась незавершённой.
9. Статический текст «ДНЕЙ НА ВЫПОЛНЕНИЕ: 1» на обороте каждой карточки в
   шаблоне вёрстки (`#cards`) не связан с реальной стоимостью карточки по
   `time` — это фиксированная надпись шаблона, а не динамическое значение.
   Строго говоря, это вёрстка (вне блока 2.1), но фиксируется здесь, чтобы
   при чтении интерфейса эта цифра не была ошибочно принята за часть игровой
   логики.

## 13. `unmapped` — всё, что не легло в схему

Полные тексты этих фрагментов (дословно, с указанием точного места) — в
`reference-logic-spec.json`, ключ `unmapped`. Кратко, по одному пункту:

1. **Мёртвый код выбора команды** (`Game.html`, строки ≈1103–1136,
   `.teambut`/`#teamselected`/`skills.*`/`teamcount`/голый `score`) — не
   исполняется никогда (нет DOM-элементов с классом `.teambut`), ссылается
   на несуществующих персонажей. Не включён в `flow`/`characters`, так как
   описывать несуществующий игровой путь было бы домыслом.
2. **`cardsObj['i_timeout']`** (строка 863) — валидная запись экрана, но
   недостижима: не входит ни в `timeline`, ни в один вызов
   `swap`/`goTo`/`unlock`. Учтена в счётчиках (29 `info`-экранов, 110 всего)
   и в разделе `endings` с пометкой «не наступает никогда», но как часть
   игрового потока не описывается.
3. **Закомментированная более ранняя версия `timeline`** (строка 882) — без
   вступительных `info`-экранов `i0`…`i6`. Исторический артефакт правки
   сценария, не исполняемые данные.
4. **Комментарий-пример гипотетического `type: 'jump'`** (строка 910,
   дублирует пример из `README.md` монолита) — формат внутри примера не
   совпадает с реально реализованным форматом `action`/`alt` (объект вместо
   кортежа, строчная `goto` вместо реальной `goTo`, невалидный JS
   `{1,15}`). Черновая заметка о нереализованной возможности.
5. **Операция `'and'`** (строка 944 интерпретатора) — распознаётся, но не
   реализована (пустое тело), ни разу не используется в данных. Описана в
   таблице операций (раздел 4) как определённая-но-нереализованная; здесь
   отмечена для полноты, так как её нельзя классифицировать как «действующую
   часть» логики.
6. **Копипаст-опечатка в журнале ходов для `lid`** (строка 1038) — при
   уменьшении `lid` показывается дельта `pro` вместо `lid`. Дефект
   отображения истории, не механики счёта; фиксируется, чтобы не потерять
   при сверке с платформенной реализацией в блоке 2.3 (важно понять,
   копирует ли текущая логика этот дефект «как есть» или дефекта там просто
   нет из-за другой архитектуры истории ходов).

Ни один из этих шести пунктов не является «молча пропущенным» — каждый
явно описан здесь и в JSON с указанием точного места в источнике и причины,
по которой он не уложился в основную схему (`metrics`/`characters`/`cards`/
`flow`/`random`/`endings`).
