## Порядок и параллельность

- Phase 1 (схема + валидатор) и Phase 2 (runtime-api) — последовательно
  (Phase 2 использует контракты Phase 1). Резолюция `asset:` в рендерере
  (часть Phase 3) не зависит от Phaser-хоста и может идти параллельно
  Phase 2 другим субагентом.
- Инъекция `assets` в `PhaserSceneContext` (Phase 3) заблокирована до
  Phase 3 трека flow-simulation (D1). Блокер не снят — НЕ реализуй
  Phaser-хост сам; сделай остальное и жди.
- Одновременно ≤ 2 активных субагентов; двое не меняют один файл;
  `httpServer.ts` — сверка `git log` и Handoff Log перед каждым срезом.

## Жёсткие ограничения (нарушение = стоп и эскалация владельцу)

- Нет формы/имени/регэкспа/заголовка в §3 дизайна — НЕ изобретай:
  вопрос в Handoff Log, стоп среза.
- JSON Schema — SSOT (правило 12): схема меняется только вместе с
  `generate:contracts`, позитивной И негативной фикстурой и тестами в том
  же срезе; никаких ручных type guard-ов вместо Ajv.
- Никаких id конкретных игр/ассетов в платформенном коде (правило 10);
  тестовые фикстуры — синтетические.
- `PhaserSceneContext` правится только аддитивно и только после записи в
  Handoff Log обоих треков (правило §3.3 дизайна); §4.0 flow-дизайна
  обновляется в том же срезе.
- Картинки «Антарктиды» не трогать (LEGACY-0023 — отдельная задача).
- Список правил санитизации SVG (§3.5) закрыт: ни ослаблять, ни молча
  расширять; предложения — в Handoff Log.

## Делегирование субагентам

**Opus** (безопасность, недоверенные данные, архитектурная связность):

- Phase 2 — эндпоинты и чтение файлов (path traversal, заголовки, 404-ы);
- Phase 3 — инъекция в `PhaserSceneContext` и координация с flow-треком;
- ревью диффов Phase 1 (санитизация SVG — безопасность).

**Sonnet** (механика по готовой спецификации):

- Phase 1 — схема, фикстуры, валидатор по закрытому списку правил §3.5;
- Phase 3 — резолюция `asset:` в рендерере (декларативно, по ADR-055);
- везде: тесты по перечню §5, обновление документации.

Правила: каждому субагенту — точная Phase и пункты TSK, выдержка §3 по его
конструкциям, список файлов, запреты, команды проверки; в ответе — список
изменённых файлов, вывод проверок, явный список отклонений (пустой — тоже
ответ). Результат каждого субагента проверяй сам (дифф + прогон) до
следующей делегации; срезы Sonnet, трогающие runtime или контракты,
дополнительно — Opus-ревью диффа. Дважды красная валидация у Sonnet →
передать срез Opus. После сбора результата закрывай субагента; перед
отчётом о завершении фазы — проверь, что незакрытых нет (CLAUDE.md §2.7).
