## 1. Purpose

This matrix turns `docs/tasks/active/TSK-20260522-editor-engine-authoring-manifest-editor.md` into bounded execution slices.

The goal is to build the first `editor-engine` for authoring manifests without making flow-chart a second source of truth.

Current state on 2026-05-22: the first full editor-engine slice is implemented through staged Codex worker subagents and separate e2e/testing subagents. The implementation covers repository-backed authoring files, graph projection, Monaco workspace, floating property panel, JSON Patch based edits, editor-only layout sidecars, compiler/runtime validation, player preview and a safe subset of writable graph topology.

Architecture gap documented on 2026-05-22: the target editor has three JSON editing modes - flow-chart, JSON tree and text editor. This matrix covered flow-chart and text editor. JSON tree view is now tracked as `TSK-20260522-editor-engine-json-tree-view` with its own execution matrix.

## 2. Inputs And Outputs

| Area | Input | Editor output | Runtime output |
| --- | --- | --- | --- |
| Game authoring | `games/<id>/authoring/game.authoring.json` | graph projection, JSON editor, property panel, optional `editor.layout.json`; JSON tree view in follow-up TSK | `games/<id>/game.manifest.json` generated by compiler |
| UI authoring | `games/<id>/authoring/ui/<channel>.authoring.json` | graph projection, JSON editor, property panel, optional `<channel>.layout.json`; JSON tree view in follow-up TSK | `games/<id>/ui/<channel>/ui.manifest.json` generated by compiler |
| Schemas | `docs/architecture/schemas/*.schema.json` | validation, field metadata, UI hints | none |
| Compiler source maps | `*.source-map.json` | diagnostics from runtime output to authoring source | none |

## 3. Non-Negotiable Invariants

| ID | Invariant | Enforcement |
| --- | --- | --- |
| I1 | Authoring JSON is the single editable source of truth. | All visual edits become JSON Patch against authoring JSON. |
| I2 | Core editor-engine is game-agnostic. | No game IDs or concrete game entity branches in core. |
| I3 | Graph projection is derived automatically. | No hand-written graph data required for correctness. |
| I4 | Layout state is editor-only. | Layout companion files never compile into runtime manifests. |
| I5 | Runtime/player do not understand editor-only data. | Code review and runtime manifest scans. |
| I6 | JSON Schema remains structural SSOT. | Ajv validation and no manual schema guards replacing schemas. |
| I7 | Diagnostics are pointer-based. | Every error carries authoring pointer or generated pointer plus source map. |
| I8 | Small edits do not rewrite whole documents. | Patch history and diff review. |

## 5. Projection Matrix

| Projection candidate | Source pattern | Graph role | Reverse projection support |
| --- | --- | --- | --- |
| Document root | authoring `root` | `DocumentNode` | Limited metadata edits |
| Definition | `/_definitions/*` | `DefinitionNode` | Edit definition fields; create later |
| Typed object | object with `_type` | `ObjectNode` with semantic label | Safe property edits |
| Collection item | object property or array element | `CollectionItemNode` | Edit, add and remove after rule exists |
| Reference field | string field recognized by projection rule | `ReferenceNode` / edge | Connect/disconnect after rule exists |
| Primitive property | string/number/boolean/null | `PropertyNode` | Edit through property panel |
| Asset path | field recognized as path/ref | `AssetReferenceNode` | Edit path; asset picker later |
| Generated runtime error | source map mapping | diagnostic marker | No direct edit until authoring pointer resolved |

## 7. Risk Register

| Risk | Impact | Control |
| --- | --- | --- |
| Flow-chart becomes its own format. | Authoring JSON and visual state drift. | Graph is derived; only layout is stored separately. |
| Reverse projection becomes too broad without schema metadata. | Invalid authoring mutations. | Keep writable operations in guarded, validated subsets. |
| Core editor knows game entities. | Platform purity violation. | Use roles from schemas and projection rules. |
| Monaco and graph race each other. | Lost edits or stale views. | One DocumentStore and serialized patch application. |
| Source maps are misunderstood. | Wrong diagnostics route. | Separate compiler source map from text location map. |
| Poor schema annotations. | Weak property panel UX. | Add schema enrichment tasks before broad authoring rollout. |
| Large manifests hurt graph performance. | Slow canvas and poor UX. | Collapse groups and render only selected subgraphs. |
| Remote schemas change under editor. | Non-deterministic validation. | Local schema registry only by default. |
