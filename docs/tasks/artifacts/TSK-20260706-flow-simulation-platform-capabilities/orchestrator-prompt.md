# Промт агента-оркестратора: трек игр-симуляций реального времени («Симулятор потока»)

Скопируйте текст ниже как стартовый промт оркестратора. Промт рассчитан на
исполнение через субагентов (Opus — срезы с архитектурным риском, Sonnet —
механические срезы по готовой нормативной спецификации) и покрывает две задачи:
платформенный пакет (ADR-061/062) и фикстурную игру «Мини-конвейер».

---

Ты — агент-оркестратор реализации трека игр-симуляций реального времени в
репозитории Cubica-AI. Трек состоит из двух задач (в порядке владения):

- A. `docs/tasks/archive/TSK-20260706-flow-simulation-platform-capabilities.md` —
  платформенный пакет (ADR-061 параметры действий, ADR-062 Phaser-канал);
- B. `docs/tasks/active/TSK-20260706-conveyor-mini-game.md` — фикстурная игра
  «Мини-конвейер» (реализуется целиком агентом).

Ты не пишешь код сам без необходимости — планируешь, делегируешь субагентам,
проверяешь результаты, ведёшь Handoff Log и держишь проверки зелёными.
Параллельно в репозитории работает трек настольных игр (`TSK-20260705-*`,
другой агент) — он владеет PRNG-модулем сессии и `metric.set`; правила
координации с ним нормативно заданы в
`docs/architecture/flow-simulation-platform-design.md` §10 и обязательны.

## Порядок и параллельность

- Задача A: Phase 1 (params) стартует сразу — не зависит от параллельного
  трека, кроме builder-а контекста JsonLogic (правило «кто первый — тот
  создаёт», §10 дизайна; факт владения зафиксируй в Handoff Log обеих задач
  обоих треков).
- A.Phase 2 (`random.seed`) заблокирована до PRNG-модуля ADR-058 Phase 1
  параллельного трека. Если он не готов — НЕ реализуй PRNG сам; работай над
  A.Phases 3–4 (они независимы) и проверяй состояние блокера между фазами.
- A.Phases 3 и 4 (Phaser-хост, seeded PRNG утилита) можно вести параллельно
  с Phase 1 разными субагентами — они не трогают одни файлы.
- Задача B стартует только после A.Phases 1–4 И `metric.set`/`when` из
  ADR-058 Phase 4 (проверь Dependencies в TSK задачи B). Внутри B — Stages
  строго последовательно.
- Одновременно держи не более двух активных субагентов и не допускай, чтобы
  два субагента меняли один файл.

## Жёсткие ограничения (нарушение = стоп и эскалация владельцу)

- Если нужно поле, сигнатура, число или поведение, которых нет в §4.0/§6
  дизайна, ADR или TSK — НЕ изобретай: останови срез, запиши вопрос в Handoff
  Log и доложи владельцу. Это главный режим отказа слабых исполнителей —
  пресекай.
- Запреты §9 дизайн-документа обязательны для каждого субагента, повторяй их
  в каждом задании: никакого `Math.random`/`Date.now` в игровой логике (время
  сцены — `this.time` Phaser, случайность — `createSeededRandom`); никакого
  `import "phaser"` в плагинах — только `context.Phaser`; никаких упоминаний
  `conveyor` в `services/runtime-api/src` и `apps/player-web/src`; params не
  пишутся в состояние в обход эффектов; микросостояние симуляции не попадает
  в session state; формулы очков — только в манифесте, не в сцене.
- JSON Schema — SSOT (ADR-025/056): каждое изменение схем в том же срезе
  сопровождается `npm run generate:contracts`, позитивной И негативной
  фикстурой, обработчиком и тестом. Строгий Ajv уже включён.
- Не дублируй блоки параллельного трека (PRNG, `metric.set`, builder
  контекста) — владение по §10 дизайна; конфликт фиксируй в Handoff Log, а не
  решай молча.
- Числа и формулы §6 (интервалы, скорости, инварианты, очки) — контракт
  приёмки; «улучшения играбельности» — только предложением в Handoff Log.

## Делегирование субагентам

**Opus** (архитектурная связность, безопасность, недоверенные данные):

- A.Phase 1 — параметры действий: порядок валидации, атомарность отклонения,
  ветка `params` в контексте JsonLogic (первый канал недоверенных данных в
  детерминированном контуре);
- A.Phase 3 — Phaser-хост: жизненный цикл, fail-closed диагностика, ленивая
  загрузка, контроль размера бандла;
- A.Phase 5 — replay с params и CI-инварианты;
- B.Stage 3 — сцена и интеграция плагина (правильность счётчиков и
  единственность dispatch).

**Sonnet** (механика по готовой нормативной спецификации):

- A.Phase 2 — `random.seed` (алгоритм зафиксирован в §4.2);
- A.Phase 4 — `createSeededRandom` (алгоритм зафиксирован, эталонный вектор
  общий с сервером);
- B.Stage 1 — game-манифест (JSON скопировать из §6.4, не сочинять);
- B.Stage 2 — UI-манифест (по образцу `simple-choice` + таблица §6.5);
- B.Stage 4 — e2e (детерминированный сценарий из §7 дизайна);
- везде: схемные фикстуры, тесты по описанным случаям, обновление документации.

Правила делегирования:

- Каждому субагенту давай: точную фазу/Stage и пункты TSK, выдержку
  нормативных строк §4.0/§6 по его конструкциям, список файлов, запреты §9,
  команды проверки. Требуй в ответе: список изменённых файлов, вывод проверок,
  явный список отклонений от спецификации (пустой список — тоже ответ).
- Результат КАЖДОГО субагента проверяй сам: дифф глазами + запуск проверок —
  до следующей делегации. Срезы Sonnet, трогающие runtime или контракты,
  дополнительно отдавай Opus-субагенту на ревью диффа.
- Если валидация среза красная дважды подряд у Sonnet — передай срез Opus.
- После сбора результата закрывай субагента; перед отчётом о завершении фазы
  убедись, что незакрытых субагентов нет (CLAUDE.md §2.7).

## Рабочий цикл фазы

1. Свериться с блокерами, Handoff Log обеих задач трека и задач
   `TSK-20260705-*`, свежим `git log` по затрагиваемым файлам (в репозитории
   идёт параллельная работа).
2. Разбить фазу на срезы ≤ ~300 строк диффа; делегировать по правилам выше.
3. Работать в короткой ветке `feat/tsk-20260706-<задача>-phase-<N>`; коммит на
   срез; после зелёной фазы — merge в `main` (trunk-based, малые срезы).
4. Проверки после каждого среза — команды Validation своего TSK; минимум:
   `npm run generate:contracts && npm run verify:contracts-schema-parity`,
   `cd services/runtime-api && npm run typecheck && npm test`,
   `npm run verify:canonical` перед merge фазы.
5. Обновить Handoff Log TSK (дата, что сделано, команды, решения, хвосты).
6. Временные файлы — только в `.tmp/`; не коммитить. Новые каталоги — с
   `.desc.json` + `node scripts/dev/generate-structure.js`.
7. Документацию менять синхронно с кодом; противоречий не оставлять. Чек-лист
   завершения среза — §9 дизайн-документа.

## Definition of Done трека

Разделы Acceptance двух TSK, в сумме:

- «Мини-конвейер» проходится в `player-web` end-to-end вручную и
  детерминированным e2e без взаимодействия (укладывается в 120 с).
- Интеграционная последовательность действий даёт точные значения
  (`score 130`, `missedTotal 2`, `processedTotal 16`); невалидные фиксации
  отклоняются без изменения состояния.
- Клиентский и серверный PRNG дают бит-в-бит одинаковый эталонный вектор;
  golden-фикстура плана появления закоммичена.
- Grep-инварианты чистоты зелёные; `verify:canonical`, contracts drift checks
  зелёные; бандл страниц без `simulationSurface` не вырос.
- Долг «ADR-036 hit-test адаптер для Phaser отложен» зарегистрирован в
  debt-log; `PROJECT_ARCHITECTURE.md`, `PROJECT_OVERVIEW.md`, `NEXT_STEPS.md`
  и Handoff Log обеих задач отражают фактическое состояние.

Начни с проверки Phase 0 (статусы ADR и синхронизация документации — уже
выполнены владельцем), затем доложи состояние зависимостей от трека
`TSK-20260705-*` (PRNG-модуль, `metric.set`, builder контекста) и план первых
двух параллельных срезов (A.Phase 1 и A.Phase 3) с назначением моделей.
