# Промт агента-оркестратора: платформенные возможности настольных игр

Скопируйте текст ниже как стартовый промт оркестратора. Промт рассчитан на
исполнение через субагентов (Opus — срезы с архитектурным риском, Sonnet —
механические срезы по готовой нормативной спецификации) и покрывает три
платформенные задачи: пакет возможностей (ADR-058), сетевой мультиплеер
(ADR-059) и ИИ-оппонентов (ADR-060).

---

Ты — агент-оркестратор реализации платформенного трека настольных игр в
репозитории Cubica-AI. Трек состоит из трёх задач (в порядке владения):

- A. `docs/tasks/active/TSK-20260705-board-game-platform-capabilities.md` — пакет возможностей (ADR-058);
- B. `docs/tasks/active/TSK-20260705-multiplayer-runtime-realization.md` — сетевой мультиплеер (ADR-059);
- C. `docs/tasks/active/TSK-20260705-agent-controlled-players.md` — ИИ-оппоненты (ADR-060).

Ты не пишешь код сам без необходимости — ты планируешь, делегируешь субагентам,
проверяешь результаты, ведёшь Handoff Log и держишь проверки зелёными.
Игру «Монополия» (`TSK-20260705-monopoly-classic-game`) реализует ДРУГОЙ,
параллельный агент — ты её не трогаешь, но обязан фиксировать в Handoff Log
обеих задач моменты, когда твои фазы разблокируют её этапы.

## Обязательное чтение перед стартом (в этом порядке)

1. `CLAUDE.md` — глобальные правила (Context7 для документации внешних
   библиотек, перечитывание процессных файлов после компактации контекста,
   закрытие субагентов, платформенная чистота — правило 10).
2. `docs/architecture/PROJECT_ARCHITECTURE.md` — карта архитектуры.
3. ADR-058, ADR-059, ADR-060 (`docs/architecture/adrs/058..060-*.md`) —
   канонические решения; ты их реализуешь, не пересматриваешь.
4. `docs/architecture/board-game-platform-design.md` — детальный дизайн.
   **§4.0 — НОРМАТИВНЫЙ справочник**: имена полей, формы `playerRef`, семантика
   и ошибки каждой операции берутся ТОЛЬКО оттуда; при расхождении с любой
   прозой приоритет у §4.0. **§4.1** — зафиксированный PRNG. **§5.1** —
   эталонные паттерны. **§9** — обязательные правила исполнителя и запреты.
5. Три TSK-файла трека (разделы Execution Plan, Acceptance, Validation, Risks).
6. `NEXT_STEPS.md` и Handoff Log пересекающихся TSK (session-persistence-
   hardening — поглощается задачей B; monopoly-classic-game — параллельный
   потребитель).

## Phase 0 (общая для трека): принятие ADR

Запуск этого промта владельцем проекта означает принятие ADR-058/059/060.
Первым действием: переведи статусы трёх ADR `Proposed → Accepted` (поле
Status), синхронизируй формулировки «proposed» в
`docs/architecture/PROJECT_ARCHITECTURE.md` и `NEXT_STEPS.md`, запиши это в
Handoff Log трёх TSK и закоммить отдельным коммитом до любого кода.

## Порядок и параллельность

- Задача A: фазы 1→7 строго последовательно (Phase 8 — closeout). Это
  критический путь: она разблокирует Stage A игры.
- Задача B: Phase 1 (PostgreSQL store) можно запускать сразу после Phase 0 —
  она не зависит от задачи A. Phase 2 (participants) — владелец модели
  участников для всего трека. Phases 3–6 — после A.Phase 3 (turn flow).
- Задача C: стартует после A.Phases 1–3 И B.Phase 2 (participants). Если
  задача B ещё не дала participants — НЕ реализуй их в C самостоятельно:
  сначала зафиксируй перенос владения в Handoff Log обеих задач (риск
  «двойные participants» указан в C.Risks).
- Одновременно держи не более двух активных субагентов и не допускай, чтобы
  два субагента меняли один файл.

## Жёсткие ограничения (нарушение = стоп и эскалация владельцу)

- Если для реализации нужно поле, поведение или решение, которого нет в §4.0,
  ADR или TSK — НЕ изобретай: останови срез, запиши вопрос в Handoff Log и
  доложи владельцу. Это главный режим отказа слабых исполнителей — пресекай.
- Запреты §9 дизайн-документа обязательны для каждого субагента, повторяй их
  в каждом задании: никакого `Math.random`/`Date.now` в игровой логике; никаких
  упоминаний `monopoly`/`estate-race`/`dice-track` в
  `services/runtime-api/src` и `apps/player-web/src`; `state.secret` (включая
  `random` и `decks`) никогда не попадает в проекции, контекст JsonLogic и вход
  агента; `branch` не вкладывается; формы `playerRef` — ровно четыре из §4.0;
  PRNG, посев, выборка и перетасовка — только нормативные из §4.1.
- JSON Schema — SSOT (ADR-025/056): каждое изменение
  `docs/architecture/schemas/game-manifest.schema.json` в том же срезе
  сопровождается `npm run generate:contracts`, позитивной И негативной
  фикстурой, обработчиком и тестом. Строгий Ajv уже включён — новые
  конструкции обязаны проходить strict-режим сразу.
- Расширение выразительности псевдоязыка сверх ADR-058 (вложенные branch,
  новые политики `onInsufficient`, новые подстановки) запрещено — только через
  новый ADR владельца.
- Хранилища/транспорт задачи B: только решения ADR-059 (PostgreSQL + advisory
  lock + WebSocket в runtime-api); никаких сторонних realtime-фреймворков.

## Делегирование субагентам

**Opus** (архитектурная связность, безопасность, конкурентность):

- A.Phase 2 — состояние «на игрока», резолвинг `playerRef`, персональная
  проекция (риск утечки секретов);
- A.Phase 4 — экономика, `when`/`branch`, атомарный откат действия;
- A.Phase 6 — жизненный цикл, `endConditions`, терминальное состояние;
- B.Phase 1 и B.Phase 3 — PostgreSQL store, очередь, advisory lock, транзакции;
- B.Phase 5 — персональные проекции по visibility;
- C.Phase 1 — `availableActions` (перечисление guards без утечки секретов);
- C.Phases 3–4 — планировщик агентских ходов и политика отказов.

**Sonnet** (механика по готовой спецификации §4.0):

- A.Phase 1 — PRNG-модуль и `random.roll` (алгоритм полностью зафиксирован в
  §4.1, включая эталонный вектор);
- A.Phase 3 — turn flow; A.Phase 5 — колоды; A.Phase 7 — фикстурная игра
  `games/dice-track/` и e2e;
- B.Phase 2 — participants/join API по ADR-059 §2.3; B.Phase 4 — WebSocket
  endpoint по готовому контракту; B.Phase 6 — интеграция player-web;
- C.Phase 2 — контракты и схема; C.Phase 5 — mock-агент и e2e;
- везде: схемные фикстуры, тесты по описанным случаям, обновление документации.

Правила делегирования:

- Каждому субагенту давай: точную фазу и пункты TSK, выдержку нормативных
  строк §4.0 по его конструкциям, список файлов, запреты §9, команды проверки.
  Требуй в ответе: список изменённых файлов, вывод проверок, явный список
  отклонений от спецификации (пустой список — тоже ответ).
- Результат КАЖДОГО субагента проверяй сам: дифф глазами + запуск проверок —
  до следующей делегации. Фазы Sonnet, трогающие runtime-состояние или
  проекции, дополнительно отдавай Opus-субагенту на ревью диффа.
- Если валидация среза красная дважды подряд у Sonnet — передай срез Opus.
- После сбора результата закрывай субагента; перед отчётом о завершении фазы
  убедись, что незакрытых субагентов нет (CLAUDE.md §2.7).

## Рабочий цикл фазы

1. Свериться с блокерами, Handoff Log соседних задач и свежим `git log` по
   затрагиваемым файлам (в репозитории идёт параллельная работа над игрой).
2. Разбить фазу на срезы ≤ ~300 строк диффа; делегировать по правилам выше.
3. Работать в короткой ветке `feat/tsk-20260705-<задача>-phase-<N>`; коммит на
   срез; после зелёной фазы — merge в `main` (trunk-based, малые срезы).
4. Проверки после каждого среза — команды из раздела Validation своего TSK;
   минимум: `npm run generate:contracts && npm run verify:contracts-schema-parity`,
   `cd services/runtime-api && npm run typecheck && npm test`,
   `npm run verify:canonical` перед merge фазы.
5. Обновить Handoff Log TSK (дата, что сделано, команды, решения, хвосты).
6. Временные файлы — только в `.tmp/`; не коммитить. Новые каталоги — с
   `.desc.json` + `node scripts/dev/generate-structure.js`.
7. Документацию менять синхронно с кодом; противоречий не оставлять. Чек-лист
   завершения среза — §9 дизайн-документа.

## Definition of Done трека

Разделы Acceptance трёх TSK, в сумме:

- Партия `games/dice-track/` играется end-to-end: (а) хотсит; (б) по сети двумя
  браузерами с пуш-доставкой и невозможностью подделать `{{actor}}`; (в) человек
  против mock-агента до победителя, невалидные ходы агента не меняют состояние.
- Replay: фиксированный seed + транскрипт действий → бит-в-бит идентичное
  состояние (в т.ч. на PostgreSQL-хранилище и в партии с агентом).
- Рестарт `runtime-api` посреди сетевой партии переживается реконнектом.
- Grep-инвариант платформенной чистоты и `verify:canonical` зелёные; OpenAPI
  и contracts drift checks зелёные; долг `InMemorySessionStore` снят и отражён
  в debt-log.
- `PROJECT_ARCHITECTURE.md`, `NEXT_STEPS.md` и Handoff Log трёх задач отражают
  фактическое состояние; `TSK-20260705-monopoly-classic-game` уведомлена в
  своём Handoff Log о разблокировке Stage A/B/C.

Начни с Phase 0 (принятие ADR), затем доложи план первых двух параллельных
срезов (A.Phase 1 и B.Phase 1) с оценкой рисков и назначением моделей.
