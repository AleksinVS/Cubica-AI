# Проект интерфейса «карта — основа» для игры «Карты, деньги, поезда»

Документ является проектной и исполнительной спецификацией интерфейса общего
экрана ведущего. Архитектурное основание — принятое ADR-080; очередность и
фактический статус реализации остаются в корневой задаче
`TSK-20260711-cards-money-trains-game`.

Статус: **техническая реализация выполнена и проверена; продуктовая визуальная
приёмка PM остаётся открытой**.

## 1. Пользовательский результат

Ведущий открывает игру и сразу видит карту Гвиней во всю рабочую область.
Транспортная сеть, терминалы, составы и строящиеся объекты читаются как единое
поле. Текущий ход, команды, карточка новости, действия и методические вопросы
не отнимают у карты постоянные колонки: они появляются компактными слоями
поверх неё и закрываются, когда не нужны.

Интерфейс должен позволять провести партию на проекторе или ноутбуке без
обычной прокрутки страницы, перехода к внешней таблице и угадывания, где
находится следующее действие.

## 2. Исходное состояние и причина переделки

До реализации на контрольном окне `1440 × 900` Phaser-canvas занимал примерно
`650 × 464` пикселя, то есть около 23% площади окна. Рядом постоянно
размещалась колонка действий, а вся поверхность была встроена в старую
`topbar`-страницу. Это исходное состояние сохранено здесь как объяснение
причины переделки, а не как описание текущего кода.

Текущий `map-first` путь уже изменил саму иерархию экрана: canvas совпадает с
рабочей областью, web-панели открываются поверх него, а документ не
прокручивается. `LEGACY-0062` остаётся активным только до продуктовой
визуальной приёмки PM и не означает, что старый малый canvas всё ещё является
исполняемой компоновкой этой игры.

## 3. Целевая композиция

На широком экране используется следующая схема:

```text
┌──────────────────────────────────────────────────────────────────────┐
│ статус: ход · фаза · новость · состояние сохранения                 │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  ┌ команды ┐                              ┌ контекст и действия ┐    │
│  │ деньги  │                              │ выбранный объект    │    │
│  │ техника │       КАРТА — ОСНОВА         │ допустимые команды  │    │
│  │ составы │                              │ причина блокировки  │    │
│  └─────────┘                              └─────────────────────┘    │
│                                                            + − ⛶    │
│                                                                      │
│        ┌ этап · основное действие · отмена/подтверждение ┐          │
│        └──────────────────────────────────────────────────┘          │
└──────────────────────────────────────────────────────────────────────┘
```

Карта физически лежит под всеми зонами и занимает всю рабочую область. Боковые
панели не резервируют постоянную ширину: ведущий открывает их поверх карты.
Карточка новости, рынок, подробный расчёт, методическая пауза и подтверждение
завершения используют существующий слой панелей/диалогов и не заменяют карту
новой страницей без необходимости.

## 4. Декларативный контракт экрана

UI authoring-манифест игры объявляет:

```json
{
  "type": "screen",
  "layout_mode": "map-first",
  "root": {
    "type": "screenComponent",
    "children": [
      { "type": "areaComponent", "props": { "workspaceSlot": "board" } },
      { "type": "areaComponent", "props": { "workspaceSlot": "status" } },
      { "type": "areaComponent", "props": { "workspaceSlot": "primary-panel" } },
      { "type": "areaComponent", "props": { "workspaceSlot": "context-panel" } },
      { "type": "areaComponent", "props": { "workspaceSlot": "action-tray" } },
      { "type": "areaComponent", "props": { "workspaceSlot": "floating-controls" } }
    ]
  }
}
```

Фрагмент показывает форму, а не готовое содержимое. Обязательные правила:

- `board` встречается ровно один раз и содержит `interactiveBoardSurface`;
- остальные зоны необязательны и содержат обычные декларативные компоненты;
- значение зоны определяет общая оболочка, а не CSS-класс игры;
- карточки, журнал, справка и методические паузы продолжают использовать
  декларативные `panels` с режимом `overlay` или `drawer`;
- game manifest не получает полей компоновки и не хранит открытость панелей;
  это локальное UI-состояние Presenter/renderer;
- runtime не знает о `map-first` и не меняет правила из-за способа отображения.

Распределение содержимого этой игры:

| Зона | Содержимое |
|---|---|
| `board` | карта, границы, сеть, терминалы, техника, подсветка и предварительный выбор |
| `status` | номер хода, текущая фаза, краткое название новости, сохранение/связь |
| `primary-panel` | команды, деньги, техника, активные грузы и составы |
| `context-panel` | выбранный объект, допустимые действия, параметры и понятные причины недоступности |
| `action-tray` | основная команда текущей фазы, «Далее», отмена и подтверждение |
| `floating-controls` | приблизить, отдалить, показать всю карту, полноэкранный режим |
| `overlay` | новость, рынок, подробный расчёт, методическая пауза, журнал, справка, завершение |

## 5. Слои карты

| Уровень снизу вверх | Владелец | Содержимое |
|---|---|---|
| 1. Визуальная основа | ассет игры | оптимизированная авторская карта с оформлением и подписями |
| 2. Статическая геометрия | game manifest/производные данные | контуры стран и областей, канонические точки терминалов |
| 3. Подтверждённая сеть | runtime snapshot | дороги, полустанки, состояния `planned/building/open/closed` |
| 4. Подтверждённые объекты | runtime snapshot | локомотивы, вагоны, грузы и владение |
| 5. Пространственный ввод | Phaser-сцена | hover, выбор, допустимые цели, предварительный маршрут и откат |
| 6. Обычный интерфейс | DOM/web renderer | статус, панели, действия, карточки, методика и ошибки |
| 7. Модальный слой | DOM/web renderer | подтверждения и блокирующие диалоги с управлением фокусом |

Растровая карта никогда не используется для расчёта пересечений, стоимости
дороги или допустимости действия. Phaser не применяет новое положение до
подтверждённого снимка runtime.

## 6. Координаты и преобразования

Для нормативной карты принимается каноническая плоскость `5079 × 3627`
**картографических единиц** с началом в левом верхнем углу. Числа совпадают с
размером исходного PNG, чтобы контрольное наложение не требовало пересчёта, но
это не пиксели текущего браузера.

Различаются три пространства:

1. **Картографические единицы** — источник истины для узлов, контуров и
   геометрии, независимый от устройства.
2. **Мир сцены** — Phaser использует те же границы мира `5079 × 3627`; фон и
   производные геометрические слои масштабируются в эти границы.
3. **Экранные координаты** — результат камеры, масштаба и размера canvas;
   никогда не сохраняются как правило или состояние сессии.

Логический размер viewport `interactiveBoardSurface` не обязан равняться миру:
он следует размеру контейнера. Камера преобразует указатель экрана в координаты
мира до выбора ближайшего объекта. Контуры из AI/PDF один раз нормализуются в
картографические единицы; исходный AI не загружается браузером и runtime.

## 7. Использование авторской карты

В интерфейсе используется уже подготовленный ресурс:

- исходник: `draft/trains/Игровая Карта.png`, `5079 × 3627`;
- web-производная: `games/cards-money-trains/assets/images/guinea-map.webp`,
  `2560 × 1828`;
- идентификатор реестра: `board-guinea-optimized`;
- происхождение, контрольные хеши и преобразование записаны в
  `games/cards-money-trains/asset-provenance.json`;
- право использования, изменения и публикации подтверждено PM как прямое
  разрешение автора и правообладателя.

Нормативная игра обязана использовать этот ресурс как визуальную основу.
Тестовая игра `cards-money-trains-mock` также может использовать отдельную
зарегистрированную копию той же разрешённой web-производной для проверки
компоновки, но сохраняет крупную метку `MOCK`: её сеть, координаты, команды,
колоды и цены остаются вымышленными. Нейтральная платформенная фикстура режима
`map-first` использует полностью синтетическое поле и не зависит от карты
Гвиней.

### 7.1. Формат подложки и векторные данные

Подтверждена гибридная доставка карты:

- художественная основа и авторские подписи загружаются как WebP;
- AI/PDF и производные SVG используются на этапе подготовки точных контуров,
  но не заменяют весь фон одним SVG-файлом;
- нормализованные области, узлы и линии хранятся как отдельная семантическая
  геометрия и рисуются поверх фона;
- динамические дороги, техника, выбор и подсветка принадлежат Phaser.

Причина выбора: текущий Phaser при загрузке SVG превращает его в растровую
текстуру заданного размера, поэтому простая замена WebP на SVG не сохраняет
бесконечную чёткость при увеличении. Кроме того, исходный AI содержит
растровые элементы, а подготовленные SVG описывают контуры, но не полное
оформление карты. Если визуальная приёмка покажет недостаток чёткости на
максимальном согласованном масштабе, следующий шаг — дополнительный уровень
WebP большего размера или фрагментированная карта; правила и координаты при
этом не меняются.

## 8. Поведение по фазам

### Настройка

Карта видна сразу. Основное действие и форма команд находятся в
`context-panel`/`action-tray`; карта не заменяется экраном настройки.

### Новость, обслуживание, рынок и груз

Карта остаётся фоном контекста. Карточка новости или форма рынка открывается
как панель поверх поля. После подтверждения панель закрывается, а статус и
объекты карты обновляются из нового снимка.

### Движение

Ведущий выбирает локомотив на карте или из доступного DOM-списка. Сцена
подсвечивает разрешённые соседние цели и оставшийся ресурс движения.
`context-panel` показывает состав, владельца, груз и причину недоступности.

### Строительство

`action-tray` переключает режим «дорога»/«полустанок». Для дороги выбираются
два узла; для полустанка — точка существующей дороги. Предварительная геометрия
видна на карте, но стоимость и допустимость показывает только ответ runtime.
После успешной операции ведущий может начать следующую, не покидая карту.

### Методическая пауза и завершение

Вопросы открываются поверх карты как управляемая панель. Завершение требует
модального подтверждения. Закрытие или отмена возвращает фокус к элементу,
который открыл панель.

## 9. Камера и пространственный ввод

- При первом открытии камера показывает всю карту без искажения пропорций.
- Колесо мыши и кнопки `+`/`−` изменяют масштаб относительно понятной точки;
  жест масштабирования на сенсорном экране является улучшением, но не
  единственным способом управления.
- Перетаскивание пустого участка панорамирует карту. Перетаскивание объекта
  начинается только после явного попадания по интерактивной цели.
- Камера ограничена границами мира с учётом масштаба и не показывает
  бесконечное пустое пространство.
- «Показать всю карту» возвращает исходный обзор. Полноэкранный режим не
  меняет канонические координаты или состояние игры.
- Нажатие по области, закрытой DOM-панелью, принадлежит панели; прозрачная
  декоративная часть панели не должна перехватывать ввод карты.

Для реализации достаточно уже установленного Phaser: его Scale Manager,
Camera bounds/zoom и преобразование указателя в мировые координаты являются
публичными возможностями текущей версии. Новые пакеты не требуются. Если при
реализации возникнет обоснованная потребность в зависимости, её установка
сначала согласуется с PM.

## 10. Состояния запроса и ошибки

После отправки действия:

- выбранный объект и кнопка переходят в состояние ожидания;
- повторная отправка блокируется до нового snapshot;
- карта может показывать предварительную геометрию, но не менять деньги, сеть
  или окончательное положение;
- при успехе сцена анимирует переход к подтверждённому состоянию;
- при отказе предварительный объект возвращается, а `context-panel` или
  уведомление показывает понятную причину без внутренних путей состояния;
- при конфликте версии Presenter загружает свежий снимок и просит повторить
  выбор, не повторяя действие автоматически;
- при сбое Phaser DOM-действия остаются доступны, а интерфейс явно сообщает,
  что пространственное представление временно недоступно.

## 11. Адаптация под размер экрана

| Контрольный размер | Ожидаемая компоновка |
|---|---|
| `1920 × 1080`, проектор | карта во всю область; одна боковая панель может быть открыта, вторая закрыта |
| `1440 × 900`, ноутбук | карта во всю область; панели поверх неё, нижний лоток компактный |
| `1024 × 768`, служебное окно | боковые панели открываются поверх карты по одной; статус сокращён |
| `390 × 844`, узкий контрольный режим | панели становятся нижними листами; карта остаётся рабочей, действия доступны без drag/drop |

Обычная прокрутка документа в активном игровом экране запрещена. Длинный
журнал, правила или список действий прокручиваются внутри своей открытой
панели. Учитываются безопасные отступы браузера и полноэкранного режима.

## 12. Доступность

- Все критические пространственные действия представлены обычными
  фокусируемыми DOM-кнопками из независимого поставщика ADR-073.
- Серверная доступность ADR-079 определяет `available`, `unavailable` или
  `parameter-dependent`; интерфейс не повторяет правила.
- Кнопки имеют понятные русские названия; цвет не является единственным
  признаком команды, состояния дороги или выбора.
- Размер обычной интерактивной цели DOM не меньше `44 × 44` CSS-пикселей.
- Открытая панель удерживает фокус, `Escape` закрывает неблокирующий слой, а
  после закрытия фокус возвращается к вызвавшему элементу.
- Уменьшение анимации операционной системы отключает необязательные перелёты и
  плавные переходы, но не скрывает изменение состояния.

## 13. Границы реализации

### Общая платформа

- JSON Schema: `layout_mode: "map-first"`, `workspaceSlot` и ограничения формы;
- сгенерированные manifest-контракты;
- Presenter/router, умеющий сохранить новый режим без сведения его к topbar;
- общий `MapFirstWorkspace` renderer и стили смысловых зон;
- встраиваемый и map-first варианты `interactiveBoardSurface` без дублирования
  Phaser;
- нейтральная фикстура, компонентные и браузерные проверки.

### Только игра

- карта Гвиней и её происхождение;
- дерево компонентов конкретных панелей;
- визуальный стиль, цвета и подписи;
- Phaser-сцена сети, терминалов, техники и строительства;
- преобразованные контуры, координаты и данные авторской сети.

### Не входит в этот интерфейсный блок

- алгоритм автоматической дороги по минимуму областей;
- импорт и трактовка эффектов всех новостей;
- пользовательский журнал и ИИ-разбор, зарегистрированные отдельным долгом;
- сетевые устройства команд и удалённый мультиплеер.

## 14. Порядок реализации

1. Расширить UI JSON Schema и authoring-схему аддитивным режимом и зонами;
   сгенерировать/синхронизировать контракты и добавить нейтральные позитивные и
   негативные фикстуры.
2. Провести `map-first` через Presenter, router и manifest renderer без изменения
   старых `topbar`/`leftsidebar` экранов.
3. Реализовать общий контейнер: полнооконная область, стандартные слои,
   выдвижные панели, нижний лоток, управление фокусом и отсутствие прокрутки.
4. Перевести `interactiveBoardSurface` из постоянной схемы «canvas + колонка» в
   слой карты; DOM-альтернатива остаётся независимой и открывается поверх карты.
5. Добавить камеру: границы, панорамирование, масштабирование, «показать всё» и
   полноэкранный режим; проверить освобождение обработчиков.
6. Перевести UI-манифест mock-игры на `map-first`, подключить разрешённую карту
   как визуальную основу и сохранить вымышленные игровые данные/маркировку.
7. Перевести нормативный UI-манифест, подключить авторскую сеть по мере
   подтверждения координат и контуров.
8. Выполнить компонентную, браузерную, доступностную и визуальную приёмку на
   контрольных размерах; после этого закрыть `LEGACY-0062`.

## 15. Критерии приёмки

### Контракт и чистота

- схема принимает корректный нейтральный `map-first` экран и отклоняет
  неизвестную зону, отсутствие/дублирование `board`;
- generated contracts не расходятся со схемой;
- в generic player-коде нет `cards-money-trains`, названий стран и игровых
  CSS-сигналов;
- старые topbar/leftsidebar fixture и production E2E сохраняют поведение.

### Геометрия и экран

- прямоугольник canvas совпадает с рабочей областью `map-first` с допуском 2
  CSS-пикселя на `1920 × 1080`, `1440 × 900` и `1024 × 768`;
- высота документа не превышает высоту окна более чем на 2 CSS-пикселя;
- карта сохраняет пропорции `5079:3627`, не растягивается и не обрезается без
  возможности панорамирования;
- при исходном масштабе видна вся карта, а команда «показать всю карту»
  воспроизводимо возвращает этот вид;
- панорамирование и масштаб не выводят камеру за границы мира;
- в начальном состоянии панели перекрывают не более 35% площади окна на
  `1440 × 900` и не более 45% на `1024 × 768`.

### Поведение

- ведущий открывает/закрывает команды, контекст, новость, методику и расчёт, не
  покидая карту и не теряя выбранный объект без явной причины;
- выбор локомотива, цели движения, двух концов дороги и точки полустанка имеет
  пространственный и DOM-путь;
- pending, успех, атомарный отказ и конфликт версии показаны без двойной
  мутации;
- ошибка загрузки Phaser оставляет доступным DOM-путь действий;
- повторное открытие экрана не удваивает обработчики камеры или ввода.

### Визуальная и продуктовая проверка

- на широком экране первым визуальным уровнем является реальная карта, а не
  белая страница, карточная сетка или постоянная боковая колонка;
- терминалы, дороги, техника и выбранные цели различимы на проекторе;
- текст панелей читаем и не рисуется внутрь canvas;
- mock-версия заметно помечена, хотя использует разрешённую карту;
- PM может пройти представительный сценарий фаз только через видимый интерфейс.

## 16. Открытые вопросы, не блокирующие компоновку

Режим `map-first` можно реализовать независимо от оставшихся правил. Автор уже
подтвердил случайный выбор равно минимальной траектории, запрет областей концов
для полустанка, открытие обычного строительства в `N+2`, полную очередь
локомотивов, семантику закрытий, обязательных платежей, стартовую конфигурацию,
жизненный цикл колод и специализацию рынка. Общий алгоритм поиска траектории
уже реализован. Отдельно остаются:

- визуальная сверка начальной сети и сборка полигонов реальных областей;
- судьба незавершённых грузов исключённой команды; техника уже возвращается на
  рынок, а долги списываются;
- выбор конкретных единиц по новости № 19; совместный подсчёт и постоянное
  изъятие уже подтверждены;
- конечный запас рынка и максимум техники либо явное отсутствие этих
  ограничений;
- импорт и проверка подтверждённых эффектов новостей в authoring-данные.

Эти вопросы меняют правила runtime или содержимое, но не декларативную
компоновку экрана.

## 17. Фактическое состояние реализации

На 18 июля 2026 года выполнены пункты 1–7 порядка реализации и техническая
часть пункта 8:

- JSON Schema, типизированные контракты и семантический компилятор поддерживают
  `map-first`, ограниченный словарь зон и ровно одну область `board`;
- общий web-плеер размещает карту, статус, нижний лоток и закрываемые боковые
  панели без проверки идентификатора игры; одновременно открывается не более
  одной боковой панели, `Escape` закрывает её и возвращает фокус;
- `interactiveBoardSurface` занимает всю рабочую область, а обычные DOM-кнопки
  действий остаются доступны независимо от Phaser;
- обе игры переведены на новый режим; mock-пакет показывает байт-идентичную
  разрешённую WebP-копию авторской карты и отдельную крупную маркировку
  `[MOCK]`, но сохраняет вымышленные сеть и правила;
- Phaser-сцены поддерживают границы мира, перетаскивание пустого поля,
  масштабирование, полный обзор, полноэкранную команду и очистку обработчиков;
- нейтральные контрактные и компонентные проверки, тесты обоих игровых
  модулей, проверка ассетов и производственная сборка прошли;
- браузерное измерение показало точное совпадение canvas и окна без прокрутки
  на `1920 × 1080`, `1440 × 900` и `1024 × 768`; дополнительная проверка
  `390 × 844` подтвердила нижние слои без взаимного перекрытия.

Остаются две честные границы. Во-первых, `LEGACY-0062` закрывается только после
визуальной проверки PM на представительном сценарии. Во-вторых, текущая
mock-сеть всё ещё использует тестовую плоскость `1400 × 1000`; перенос
авторских узлов и контуров в каноническую плоскость `5079 × 3627` относится к
отдельному импорту нормативного содержимого и не подменяется этим UI-блоком.
