# Контракт передачи содержимого «Карты, деньги, поезда»

Документ задает минимальный формат, в котором авторские материалы должны быть переданы для реализации. Это предотвращает домыслы и позволяет преобразовать содержимое в проверяемые authoring-манифесты — редактируемые JSON-источники, из которых собирается исполнимая игра.

## 1. Правила приема

- при конфликте главным является `Копия Сценарий Карты Деньги Поезда 040225.docx`, затем явные решения PM;
- материалы можно передать таблицей, DOCX или JSON, но каждое значение должно иметь однозначный идентификатор;
- неизвестное значение помечается как `не определено`, а не заполняется предположением;
- после приема данные переводятся в JSON authoring-слоя и проверяются схемой;
- исходные документы сохраняют роль доказательства происхождения, но runtime — серверный исполняющий слой — их не читает.

## 3. Топология карты

### 3.1. Узлы

Для каждого терминала и полустанка:

| Поле | Смысл |
|---|---|
| `id` | стабильный машинный идентификатор |
| `label` | подпись на поле |
| `type` | `terminal` или `waypoint` |
| `countryId` / `regionId` | принадлежность стране и области |
| `initiallyActive` | доступен ли в начале |
| `locomotiveCapacity` | обычно 2 для терминала |
| `x`, `y` | координаты в объявленной канонической системе игрового поля |

### 3.2. Начальные дороги

Для каждой дороги: `id`, `from`, `to`, начальный статус, признак двусторонности, каноническая геометрия линии и особые ограничения. Геометрия должна позволять runtime проверить положение нового полустанка; экранные пиксели для этого недостаточны.

### 3.3. Приём сети в виде изображения

Автор может передать сеть размеченной копией канонической карты без подготовки JSON. Изображение принимается как исходный материал, а не как исполнимый контракт. Процесс преобразования обязателен:

1. Зафиксировать точную версию и размеры исходной карты; не изменять полученный файл.
2. Совместить разметку с канонической картой. Для фотографии сначала исправить перспективу по видимым углам поля.
3. Снять координаты узлов в системе координат исходной карты и присвоить им стабильные идентификаторы.
4. По подписанным линиям составить список дорог с конечными узлами, начальным состоянием и направлением движения.
5. Отдельно зафиксировать пересечения без соединения и спорные места; не считать визуальное пересечение дорог узлом без явной отметки автора.
6. Проверить полученный JSON действующей схемой манифеста.
7. Сформировать контрольное наложение цифровой сети на исходную карту и читаемую таблицу связей.
8. Получить подтверждение автора. До этого извлечённая сеть имеет статус черновика и используется только в тестовой копии игры.

Если автор присылает несколько изображений, они должны использовать одну и ту же версию карты и сохранять её пропорции. Отдельные изображения допустимы для дорог, узлов и областей.

### 3.4. Области и динамическое строительство

Для каждой области нужны:

- стабильный `regionId`;
- проверяемый замкнутый контур в канонических координатах;
- правило принадлежности линии на общей границе, если области соприкасаются;
- связь с визуальной областью карты.

Ведущий строит дорогу между двумя существующими станциями/полустанками; runtime использует прямой канонический отрезок и выводит стоимость как `2 × количество непрерывных сегментов внутри областей`. Каталог готовых дорог не передается.

Для полустанка runtime принимает внутреннюю позицию существующей дороги, создает узел и делит связь на две. Нужно передать общие ограничения допустимости: минимальное расстояние от концов, запрещенные дороги/области и момент открытия. Если ограничений нет, это указывается явно.

## 4. Грузы

Для каждой карты груза:

| Поле | Смысл |
|---|---|
| `id` | стабильный идентификатор |
| `originNodeId` | терминал погрузки |
| `destinationNodeId` | терминал назначения |
| `bankPayout` | выплата логистической компании |
| `copies` | количество экземпляров в колоде |
| `holdCost` | стоимость хранения, если отличается от 1 |
| `title` / `description` | отображаемый текст |
| `assetId` | необязательное изображение |

Нужно отдельно подтвердить четыре груза демонстрации: `9 → 7`, `9 → 2`, `5 → 9` и все карты, которые могут быть предложены при демонстрационном выборе.

## 5. Новости

Для каждой новости:

- `id`, заголовок и текст для участников;
- вес/количество карт в колоде;
- момент начала и длительность;
- затронутые дороги, узлы, цены или правила;
- точный эффект и способ отмены;
- можно ли карте повторяться;
- подсказка ведущему и методический смысл.

Эффект должен выражаться объявленными действиями. Текст «ведущий решает по ситуации» допустим только как методическая рекомендация, но не как скрытая мутация состояния.

## 6. Команды, рынок и баланс

Нужно передать:

- допустимые конфигурации числа команд каждого типа;
- цвета и названия для каждой конфигурации; базовый старт закреплен PM: 10 монет каждой команде, 2 вагона логистической компании, 1 локомотив паровозной гильдии;
- полный каталог техники, цены покупки/продажи и обслуживание;
- правила появления/истощения терминалов и грузов;
- все tie-break правила (правила разрешения равенства);
- финансовое ограничение закреплено PM: кредит и отрицательный баланс запрещены, недостаток денег отклоняет действие, продажа техники возможна только в рыночной фазе;
- значения и длительность временных изменений цен;
- точные условия доступности строительства.

## 7. Методика

Для каждой контрольной точки:

- рекомендуемый момент;
- цель обсуждения;
- вопросы ведущего;
- наблюдаемые игровые факты;
- допустимые подсказки и интервенции;
- что не следует интерпретировать как оценку личности;
- материалы деролинга — выхода участника из игровой роли — после завершения.

## 8. Ассеты и права

Использование текущей игровой карты подтверждено PM как согласованное с автором и правообладателем. Карту можно включать в Cubica, оптимизировать для браузера и использовать в цифровой версии без правового риска. Повторное подтверждение для этой версии карты не требуется.

Для каждого нового изображения, текста или шрифта, которое не входит в уже согласованный комплект:

- автор/владелец;
- источник;
- лицензия или подтвержденное право использования;
- разрешенные каналы и срок, если ограничены;
- исходный файл достаточного качества;
- необходимость указания авторства.

Карта должна иметь однозначную версию. Любая последующая правка канонической геометрии требует повторной проверки координат узлов, дорог и контуров областей.

## 9. Временные тестовые данные

PM разрешил продолжать разработку на вымышленных данных до ответа автора. Чтобы временный баланс или геометрия не стали скрытыми правилами продукта, действует следующая граница:

- тестовые данные размещаются в отдельном запускаемом пакете `games/cards-money-trains-mock/`;
- пакет имеет собственный идентификатор игры и заметную маркировку `test-only` — «только для испытаний»;
- нормативный пакет `games/cards-money-trains/` не получает вымышленные станции, дороги, цены, колоды или контрольные результаты;
- тестовый пакет использует те же публичные схемы и платформенные обработчики, поэтому доказывает техническую работоспособность без специального кода для этой игры;
- каждое предположение записывается как тестовое и заменяемое, а не как вывод из авторских материалов;
- извлечённые из будущего изображения данные сначала проходят черновую проверку и подтверждение автора, после чего переносятся в нормативный пакет;
- наличие работающей тестовой копии не закрывает пункты готовности, требующие авторского содержимого.

## 10. Проверка полноты

Набор принят, когда:

- все ссылки ведут на существующие идентификаторы;
- нет грузов с неизвестной выплатой или несуществующим терминалом;
- каждая новость имеет исполнимый эффект и завершение;
- стартовая сеть связна в предусмотренных автором местах;
- канонические координаты, начальные дороги и контуры областей позволяют runtime воспроизвести контрольный расчет стоимости;
- конфигурация команд соответствует количеству участников;
- негативная фикстура подтверждает отклонение обязательного платежа без кредита, продажи вне рынка и частичной мутации;
- контрольный проект списывает только полную распределенную стоимость одной операцией;
- право использования текущей карты зафиксировано, а права на каждый новый ассет заполнены;
- контрольное воспроизведение GSR проходит без подстановки временных значений.
