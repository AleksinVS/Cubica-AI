# Оставшиеся платформенные решения для полной партии

**Статус:** все четыре решения приняты PM, реализованы и используются игрой.
Решение D зафиксировано в ADR-088, A–B — в ADR-089, C — в ADR-090. Общие
механизмы доказаны нейтральными тестами; игра использует их для независимых
причин блокировки, трёх параметризованных действий грузовой колоды,
ограниченного названия команды и атомарной обработки группы объектов.

Этот документ сохраняет причины и границы уже реализованных общих
возможностей платформы. Он не заменяет ADR (архитектурную запись принятого
решения) и не разрешает расширять публичные контракты сверх принятых границ.

## 1. Как понят вопрос

Нужно продолжить игру на универсальном декларативном языке Cubica Mechanics
IR — типизированном наборе операций, из которых пакет игры собирает
транзакционные правила. Платформа не должна получать ветви по названию игры,
новости, дороги или роли команды.

После подключения реальных колод и ответов автора обнаружены четыре пробела:

1. новость должна добавить свою причину блокировки дороги или терминала, не
   стирая ручную, строительную или другую независимую причину;
2. одно и то же действие должно работать с одной из 23 терминальных колод,
   выбранной ограниченным параметром;
3. ведущий должен вводить произвольное название команды обычным доступным
   web-полем;
4. одна игровая команда должна обработать ограниченную группу объектов:
   присоединить выбранные вагоны за одно очко действия либо немедленно
   начислить деньги нескольким владельцам за несколько вывезенных грузов.

Каждый пробел касается публичного переиспользуемого контракта, поэтому
требует решения PM и последующей общей ADR без правил конкретной игры.

## 3. Решение A: безопасное добавление элемента в множество

### Проблема простыми словами

У дороги и станции хранится множество причин недоступности. Например, объект
может одновременно строиться, быть вручную закрыт ведущим и попасть под
новость.

Текущий язык умеет:

- заменить всё множество целиком;
- удалить из множества один конкретный элемент.

Он не умеет добавить один элемент, сохранив остальные. Перезапись массива
новостью могла бы случайно удалить строительную или ручную блокировку и
открыть объект раньше времени.

### Рекомендуемый общий контракт

Добавить в общую операцию изменения атрибутов действие `set-add`:

- цель обязана быть объявленным атрибутом типа `set`
  (множество уникальных элементов);
- значение проверяется как один элемент объявленного типа множества;
- повторное добавление того же значения ничего не меняет и не создаёт дубль;
- максимальный размер множества проверяется до записи;
- операция атомарна вместе с остальным игровым планом;
- стоимость ограничена опубликованным максимальным размером множества.

Для текущего среза отдельная новая операция удаления не нужна: общий
механизм массового удаления конкретной причины уже существует.

### Альтернативы

1. **Перезаписывать всё множество.** Отклоняется: теряются независимые причины
   блокировки.
2. **Хранить отдельный флаг для каждой новости.** Отклоняется: платформенная
   модель и интерфейс разрастаются полями конкретных карточек.
3. **Добавить узкий эффект «закрыть дорогу новостью».** Отклоняется:
   предметная семантика игры попадёт в общий движок.

## 4. Решение B: параметризованное семейство колод

### Проблема простыми словами

У игры 23 терминальные колоды. Операции колоды сейчас принимают только заранее
вписанный идентификатор. Поэтому генератор создаёт по три почти одинаковых
внешних действия для каждого терминала:

- показать два груза;
- выбрать один;
- вернуть оба.

До реализации это были 69 действий вместо трёх. Параметризованное семейство
уже устранило 66 повторений; вместе с остальными функциями текущий пакет
содержит 82 внешних действия. Это уменьшило манифест, карту исходников,
каталог кнопок и стоимость дальнейших изменений. Производительность
проверяется отдельно: сокращение каталога не заменяет оптимизацию безопасного
расчёта доступности.

### Рекомендуемый общий контракт

Разрешить операциям колоды получать её идентификатор из ограниченного
типизированного выражения, в том числе из параметра действия:

- вычисленное значение обязано быть безопасным идентификатором;
- значение обязано указывать на уже объявленную колоду текущего
  неизменяемого пакета;
- схема параметров действия задаёт закрытый список допустимых значений либо
  безопасную ссылку на объявленную коллекцию;
- клиент не может передать имя чужой или несуществующей колоды;
- видимость, скрытый порядок, вместимость и именованный поток случайности не
  меняются;
- инициализация всех 23 колод может остаться статической: параметризуется
  повторяемое внешнее действие, а не весь пакет.

После этого пакет игры публикует три грузовых намерения с параметром терминала
вместо 69 отдельных намерений.

### Альтернативы

1. **Оставить 69 действий.** Игра работает, но каждый новый терминал или
   вариант выбора умножает кодогенерацию, размер пакета и время проверки.
2. **Спрятать выбор колоды в игровом серверном обработчике.** Отклоняется:
   появляется игровая ветвь вне декларативного пакета.
3. **Создать одну общую колоду всех терминалов.** Отклоняется: меняется
   авторское правило независимых терминальных колод.

## 5. Решение C: обычное текстовое поле действия

### Проблема простыми словами

Сервер уже умеет безопасно принять строковый параметр с ограничением длины и
проверить его по JSON Schema. Но общий доступный интерфейс действий на карте
умеет показать только:

- выпадающий список;
- числовое поле.

Поэтому ведущий не может ввести произвольное название команды через обычный
интерфейс, хотя правило автора это разрешает.

### Рекомендуемый общий контракт

Добавить в общий интерфейс действий поле `text`:

- имя, подпись, обязательность и начальное значение;
- подсказки минимальной/максимальной длины и допустимого шаблона для
  своевременной обратной связи;
- стандартная клавиатурная доступность и связь подписи с полем;
- серверная `paramsSchema` остаётся единственным доверенным валидатором;
- плагин только показывает поле и отправляет строку, но не решает, допустимо
  ли название.

Цвет команды остаётся обычным выбором из подготовленной контрастной палитры и
не требует нового вида поля.

### Альтернативы

1. **Выдавать только заготовленные названия.** Не выполняет правило автора.
2. **Сделать игровую React-форму вне общего интерфейса.** Работает локально,
   но дублирует ввод параметров и создаёт узкое исключение.
3. **Автоматически строить все формы прямо из JSON Schema сейчас.** Полезное
   дальнейшее направление, но существенно шире текущего среза; для этой игры
   достаточно одного общего поля.

## 6. Принятое решение D: ограниченный цикл по выбранным объектам

### Проблема простыми словами

При формировании состава ведущий выбирает сразу группу вагонов, которую
нужно присоединить к локомотиву. По подтверждённому правилу вся такая операция
расходует одно очко действия независимо от числа выбранных вагонов. Общая
операция связи умеет присоединить один заранее указанный объект, но текущий
язык не умеет повторить её для переменного ограниченного списка из параметра
действия. Отдельное действие на каждый вагон ошибочно списало бы очко за каждый
вагон, а варианты «присоединить один/два/три» были бы узким игровым обходом.

Новости № 28 и № 29 немедленно начисляют по одной монете владельцу вагона и
владельцу локомотива за каждый груз, покинувший заданный терминал. За одно
движение локомотив может вывезти несколько вагонов, а получатели определяются
фактическим составом.

Текущий язык умеет выбрать ограниченное множество объектов и массово заменить
их собственные поля. Он также умеет один безопасный перевод денег. Но он не
может для каждого выбранного груза выполнить переводы двум связанным
командам. Создание отдельного действия для каждого возможного вагона,
локомотива и груза невозможно и противоречит платформенному подходу.

### Рекомендуемый общий контракт

Добавить `core.entities.each` — статически ограниченный цикл по результату
предыдущей типизированной выборки:

- выборка уже имеет максимальный размер из вместимости коллекции;
- тело содержит ограниченное число существующих операций;
- вложенные циклы в первой версии запрещены;
- текущий объект доступен через существующее типизированное выражение
  `value.item`;
- обход идёт в каноническом порядке идентификаторов;
- проверяющий компилятор заранее умножает максимальную стоимость тела на
  максимальный размер выборки и отклоняет превышение бюджета;
- все итерации входят в одну транзакцию: ошибка любого перевода отменяет весь
  ход;
- случайность разрешена только через уже объявленные именованные потоки и не
  нужна новостям № 28/29;
- журнал сохраняет ограниченный итог, не раскрывая серверные данные.

Игра соберёт из него:

- существующее присоединение для каждого выбранного вагона и одно списание
  очка действия вне цикла;
- два обычных перевода на каждый подходящий груз: банку не нужно хранить
  баланс, а получатель вычисляется по владельцу фактического вагона и
  локомотива.

### Альтернативы

1. **Специальная операция массовой выплаты.** Проще исполнитель, но хуже
   компонуется с налогами, сборами, изъятием и изменениями других полей.
2. **Отдельные действия на каждый груз.** Неприемлемый рост каталога и риск
   частично выполненного движения.
3. **Посчитать деньги в клиенте или игровом серверном коде.** Нарушает
   серверный источник истины и чистоту платформы.

## 7. Что не требует архитектурного решения

Текущие четыре фиксированные команды хранятся как записи, которые общий язык
не умеет создавать и случайно упорядочивать. Пакет игры может перенести
команды в уже существующую универсальную коллекцию сущностей:

- создать 4–12 команд существующей операцией;
- присвоить тип, название, цвет и 10 монет;
- создать стартовые вагоны или локомотивы;
- получить случайный порядок существующей общей операцией
  многоуровневого упорядочивания;
- разместить технику ведущим по выбранным станциям;
- проверить предел двух локомотивов на станции;
- разрешать покупку только подходящего типа техники.

Это меняет только модель и планы конкретной игры. Публичный контракт,
хранилище и граница доверия не меняются, поэтому отдельное согласование
архитектуры не требуется. До ответа `R-28` автоматическая настройка не
домысливает соотношение типов для чётного количества команд.

## 8. Общие ограничения безопасности и совместимости

Для всех четырёх решений обязательны:

- JSON Schema остаётся единственным источником истины публичной структуры;
- новые возможности получают нейтральные тестовые фикстуры без названий и
  правил этой игры;
- проверки типов, вместимости, стоимости и видимости выполняются до записи;
- ошибка не оставляет частично изменённое состояние;
- предметные макросы живут в пакете игры, а общий runtime не знает номера
  новостей, терминалы, грузы и роли;
- новые операции получают новую точную версию модуля Mechanics;
- существующие неизменяемые сессии продолжают ссылаться на прежний
  исполнитель и не мигрируют автоматически;
- ошибки имеют стабильные коды, пригодные для объяснения в интерфейсе;
- полная проверка большого блока выполняется один раз после завершения
  независимых реализаций, а не после каждого промежуточного изменения.

## 9. Рекомендуемое согласование

Все направления приняты PM:

1. `set-add` для одного типизированного элемента множества;
2. ограниченное выражение идентификатора объявленной колоды;
3. общее доступное поле `text`, при доверенной серверной проверке параметра;

Вместе со статически ограниченным `core.entities.each` без
вложенных циклов это разрешает закрыть подтверждённые правила без игровых
исключений и снижает риск дорогой миграции следующих игр.

## 10. Последовательность после согласования

1. Основной агент уже записал принятое решение D в ADR-088 и синхронизировал
   обзор архитектуры.
2. `sol high` реализовал схемы, типизацию, бюджеты, ошибки и нейтральные
   проверки для D; основной агент принял реализацию.
3. Игра доказала D групповым формированием состава за одно очко действия:
   выбор сохраняется на сервере, все вагоны прицепляются атомарно, а ошибка
   одного объекта отменяет всю группу и списание ресурса.
4. A–C реализованы: общий язык поддерживает `set-add` и ограниченный параметр
   объявленной колоды, а Player показывает схемно ограниченное текстовое поле.
   Основной агент проверил, что игровая семантика не попала в platform core.
5. `luna high` выполнил сфокусированную общую серию контрактов и runtime после
   единой пересборки точных блокировок Mechanics: нейтральный механизм,
   серверный игровой replay, интерфейс и 13 манифестов прошли.
6. Пакет игры уже использует сетевые новости, три параметризованных грузовых
   действия и настройку команд. Новости № 28/29 ждут только решения автора о
   сроке бонуса, а не новой возможности платформы.
7. После устойчивого крупного блока выполняются расширенный прогон и
   производственный браузерный replay.

## 11. Проверка на упрощение

Решения не вводят новый сервис, базу данных, транспорт, графический движок или
параллельный источник правил.

Приняты следующие упрощения предложения:

- для множества добавляется только требуемое `set-add`; существующее удаление
  переиспользуется;
- параметризуется только идентификатор объявленной колоды, а не произвольная
  динамическая схема колод;
- добавляется одно текстовое поле, а не универсальный генератор форм;
- цикл имеет один уровень и переиспользует существующие операции вместо
  отдельного языка сценариев.

Дальнейшее упрощение потребовало бы либо дублировать десятки действий, либо
перенести игровые правила в общий код, либо отказаться от подтверждённого
поведения.
