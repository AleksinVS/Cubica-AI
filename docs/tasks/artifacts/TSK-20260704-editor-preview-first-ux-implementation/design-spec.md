## 2. Контракты editor-engine

Контракты даны эскизно; точные имена согласуются с результатом модуляризации
(`TSK-20260630-editor-engine-modularization`). Все новые модули —
framework-agnostic, без React/Monaco/game-specific зависимостей (ADR-034).

### 2.1. Вхождения сущностей в TreeViewModel

```ts
interface TreeViewNode {
  nodeId: string;          // уникален в дереве (включает путь вхождения)
  entityId?: string;       // общий для всех вхождений одной сущности
  occurrenceKind: "primary" | "occurrence"; // вхождение = «тот же объект»
  // ... существующие поля (pointer, label, diagnostics, canAdd/...)
}
```

- Выбор задаётся `entityId`; все узлы с тем же `entityId` получают
  «мягкую» подсветку.
- Auto-reveal: в текущем режиме группировки раскрывается первичное вхождение
  (в «По экранам» — позиция вложенности; в «По типам» — под своим прототипом).
- Источник связей — `EditorEntityProjection` (ADR-052); дерево не строит
  собственный индекс.

### 2.2. Интерпретатор возвращённого намерения

```ts
interface ReturnedIntentInput {
  projectionYaml: string;        // исходная промт-проекция
  returnedText: string;          // возвращённое намерение
  facetSourceMap: FacetSourceMap;
  sourceHashes: Record<string, string>; // файл -> хеш на момент проекции
  entityId: string;
}
interface InterpretationLineReport {
  fragment: string;
  bucket: "applied" | "recognized-no-change" | "unrecognized";
  targetPointer?: string;
}
interface ReturnedIntentResult {
  changeSet: EditorChangeSet | null;
  report: InterpretationLineReport[];
  path: "deterministic" | "agent";
  stale?: true;                  // prompt-stale: хеши не совпали
}
```

Порядок обработки (UX-документ §6):

1. Проверка хешей → `stale` при несовпадении (дальше не идём без
   актуализации).
2. **Быстрый путь**: `returnedText` парсится как YAML И diff с проекцией —
   только значения известных ключей → детерминированный ChangeSet.
3. Иначе — агентский путь (через существующий agent-контур editor-web).
4. Семантика удалений: удалённый элемент коллекции → намерение удалить;
   удалённое скалярное свойство → «не менять» (bucket
   `recognized-no-change`), если из текста не следует обратное.
5. Каждый изменённый фрагмент обязан попасть в отчёт; пустой отчёт при
   непустом diff — ошибка интерпретатора.
6. Результат идёт в общий конвейер: политика риска → dry-run → validation →
   undo journal → apply.

### 2.3. Политика риска операций

```ts
type ChangeRisk = "safe" | "structural" | "dangerous";
function classifyChangeSet(cs: EditorChangeSet, projection: EditorEntityProjection): {
  risk: ChangeRisk;
  reasons: string[];           // для резюме и approval envelope
}
```

- `safe`: replace значений листовых полей (тексты, подписи, стили, числа).
- `structural`: add/remove элементов коллекций, переупорядочивание.
- `dangerous`: изменение `id`, перенацеливание ссылок, удаление сущности со
  входящими ссылками, операции с файлами вне authoring/assets.
- `dangerous` требует approval envelope (ADR-047) до apply; отклонение — не
  применять, записать в журнал.
- Применяется ко **всем** агентским ChangeSet (панельный чат, сессионный чат,
  регион, текстовый режим) в одной точке конвейера.

### 2.4. Очередь интентов

```ts
interface QueuedIntent {
  id: string;
  status: "pending" | "running" | "applying" | "done" | "failed" | "cancelled" | "stale";
  baseJournalSeq: number;
  readPointers: string[];   // контекст, который читал агент
  writePointers: string[];  // цели изменения
  cancel(): void;           // обязателен для running
}
```

- Конфликт при применении: журнал с `baseJournalSeq` тронул указатель из
  `readPointers ∪ writePointers` → статус `stale`, выбор пользователя.
- MVP: максимум один `running` на сессию.
- Ручные правки формы в очередь не встают — применяются немедленно.

### 2.5. Схема фикстуры состояния

Файл `games/<gameId>/authoring/fixtures/<fixtureId>.json`, JSON Schema —
новая `state-fixture.schema.json` рядом с authoring-схемами:

```json
{
  "id": "day4-route-choice",
  "_label": "День 4, выбор маршрута",
  "screenRef": "route-choice",
  "stepRef": "i12",
  "state": { "...": "полный snapshot runtime state" },
  "manifestHash": "sha256-...",
  "sourceTraceRef": ".tmp/editor-playthroughs/<traceId>#<seq>",
  "note": "..."
}
```

- Валидация схемой + семантика: `screenRef`/`stepRef` существуют,
  `manifestHash` сверяется → `fixture-stale`.
- Compiler, runtime-api, player-web файлы фикстур не читают; редактор передаёт
  `state` в preview-only restore endpoint.
- Авто-чекпоинты: `.tmp/editor-playthroughs/`, политика «последние N на
  сессию» (конфиг, по умолчанию 20), очистка вместе с GC ADR-042.

### 2.6. Кэш проекций

- Уровень 1 (в памяти): линза объявляет `readPointerPrefixes`; DocumentStore
  patch → обратный индекс → инвалидация только затронутых сущностей и
  производных. Полная пересборка только при смене схем/словаря/версии линз.
- Уровень 2 (диск, `.tmp/editor-cache/<projectId>/`):
  - по-файловые артефакты: ключ = хеш файла;
  - проектные артефакты: ключ = сводный хеш **всех входов проекции по
    ADR-052** (authoring-файлы, схемы+аннотации, словарь, source maps,
    design artifacts, метаданные плагинов) + версия editor-engine/линз +
    `cacheFormatVersion`;
  - несовпадение/повреждение → молчаливая пересборка; запись отложенная.
- Уровень 3: кэш компиляции по хешу authoring-входов; Ajv-валидаторы
  переиспользуются по хешу схемы.
- Телеметрия hit/miss и длительностей обязательна с первого среза.
- Порядок работ: сначала профилирование на модульных границах (в задаче
  модуляризации), затем уровень 1, затем уровень 2/3 по замерам.

### 2.7. Расширение renderer adapter

- Новая **необязательная** возможность `captureRegionSnapshot(rect):
  Promise<ImageData | null>`; отсутствие → регионный промт уходит без снимка.
- Снимок и контекст сущностей во внешний LLM проходят гейты redaction/audit
  ADR-044 (та же точка, что остальной agent context).

### 2.8. Операции создания/удаления/рефакторинга

Высокоуровневые операции редактора, детерминированно сводимые к
`EditorChangeSet` (ADR-034 reverse projection):

- `createEntity({ typeOrPrototype, containerPointer?, channel, label? })` —
  атомарно: game-фасет + UI-фасет (для визуальных типов) со ссылкой на `id`;
  `id` = ASCII-слаг из `_label` + суффикс уникальности.
- `createPrototype({ baseType | fromEntityId })` — по ADR-050.
- `deleteEntity({ entityId, referencePolicy: "abort" | "clean" | "retarget", retargetTo? })`.
- `renameEntityId({ entityId, newId })` — обновление всех ссылок; всегда
  `dangerous` → approval envelope.

## 3. Поведенческие спецификации

### 3.1. Дерево сущностей

- Переключатель «По экранам / По типам» — вверху сайдбара (см. макет, зона 2);
  режим запоминается на пользователя (editor layout persistence).
- «По экранам»: верхний уровень — все экраны канала; активный раскрыт и
  синхронизирован с предпросмотром; внутри экрана вложенность = отображению;
  подгруппа «Логика экрана» — невизуальные сущности экрана, свёрнута по
  умолчанию.
- «По типам»: прототипы (визуально отличимы) → экземпляры с хлебной крошкой;
  экземпляр раскрывается в свою UI-структуру; чужие вложенные экземпляры —
  `occurrenceKind: "occurrence"`.
- «+»: в «По экранам» — компонент в выбранном контейнере / экран на верхнем
  уровне; в «По типам» — новый прототип; в «Сценарии» — этап с выбором
  привязанного объекта; в «Правилах» — правило в выбранной системе.
- Поиск фильтрует по `_label`, типу, диагностикам; результат — плоский список
  с хлебными крошками; Enter выбирает сущность.
- Пустые секции: приглашение + действие («Нет этапов — добавьте первый»).

### 3.2. Панель сущности

- Заголовок: `_label` (инлайн-редактирование), тип/прототип, иконки
  «источник» (текстовый режим) и «закрепить в док».
- Чипы фасетов: «Смысл», «Содержание», «Вид · <канал>» с переключателем
  канала; отсутствующий вид у требующего отображения типа — чип-предупреждение
  «создать вид»; у невизуального типа чип «Вид» отсутствует.
- Поля: группировка по фасетам, справа мелкий бейдж источника («игра» /
  «UI · web»); изменённые агентом поля подсвечиваются до следующего действия.
- Промт-строка: однострочная, растёт в тред; кнопка эскалации «В чат сессии».
- Текстовый режим: Monaco/read-write текст промт-проекции; Return-to-intent
  кнопка «Применить как намерение» → конвейер §2.2; недоступен при
  мультивыделении.
- Позиционирование: панель не перекрывает выделение (алгоритм: свободный
  квадрант рядом с bounding rect); Esc закрывает; переключение сущности
  переиспользует окно.

### 3.3. Режимы и оси времени

- Переключатель «Дизайн/Превью» в строке меню; цветная плашка режима на
  предпросмотре (см. макет).
- «Дизайн»: селектор контекста состояния (этап + фикстура) рядом с плашкой;
  валидная компиляция по debounce → автообновление предпросмотра.
- «Превью»: правки применяются вручную (плашка «Предпросмотр отстаёт от
  правок — Применить»); лестница восстановления UX-документа §9.2 с
  сообщениями простыми словами.
- Undo (Ctrl+Z) — только ось документа; на timeline undo не влияет; подписи
  «Отменить правку» / «Вернуться к шагу …» не смешиваются.
- События трассы, записанные на старом хеше манифеста, приглушаются на
  timeline.

### 3.4. Регион и мультивыделение

- Drag → рамка с ручками; чипы захваченных сущностей (исключаемые);
  строка промта растёт в тред.
- Вход агента: промт + список сущностей (+ снимок, если адаптер умеет).
- После применения: резюме + «Отменить» у рамки; резюме дублируется в
  «Журнал».
- Мультивыделение (Shift/Ctrl+клик, рамка в дереве): панель показывает общие
  поля; промт-строка работает по контракту региона без снимка.

### 3.5. Диагностика и последний валидный предпросмотр

- Вкладка «Проверки»: группировка по серьёзности; строка = серьёзность +
  сообщение + `_label` + бейдж фасета; клик = навигация (§9.6).
- Быстрые исправления — подготовленные ChangeSet; «Исправить агентом» —
  интент с контекстом диагностики.
- Предпросмотр держит последний валидный скомпилированный снимок; плашка
  «Показана последняя рабочая версия — N правок назад; M ошибок…» с переходом
  к первой ошибке.
- Индикатор свежести в статус-баре: `актуален` / `отстаёт` / `заблокирован
  ошибками` (зелёный/жёлтый/красный маркер, см. макет, зона 7).

### 3.6. Ассеты

- Секция «Ассеты»: сетка превью, поиск, фильтры, drag-загрузка, счётчик
  использований, диагностика сирот.
- Виджет asset-reference в панели: выбрать / загрузить / сгенерировать.
- Генерация: только явное действие пользователя; предзаполнение промта из
  контекста сущности; выбор из вариантов; provenance-метаданные
  (`generation.prompt`); операционная политика стоимости.
- Все операции — через `fileCreates`/`fileDeletes` в ChangeSet; undo работает;
  Save коммитит бинарники worktree.

## 4. Реестр диагностик

Новые коды (в дополнение к существующим schema/semantic):

| Код | Серьёзность | Условие |
| --- | --- | --- |
| `entity-view-orphan` | warning | недекоративный UI-элемент без ссылки на игровую сущность |
| `entity-missing-view` | warning | тип требует отображения, вида в канале нет |
| `prompt-stale` | info/block | хеши источников проекции не совпали (ADR-049) |
| `fixture-stale` | warning | `manifestHash` фикстуры не совпадает с текущим |
| `fixture-unknown-ref` | error | `screenRef`/`stepRef` фикстуры не существует среди экранов активного канала / шагов хронологии |
| `intent-stale` | info | журнал тронул read/write указатели интента |
| `preview-stale` | status | валидные правки не применены к предпросмотру |
| `preview-blocked` | status | компиляция сломана, показан последний валидный снимок |
| `asset-orphan` | info | ассет без использований |
| `scenario-dead-end` | warning | этап без исходящих переходов и не терминальный |
| `scenario-unreachable` | warning | этап недостижим из стартового |

## 5. Телеметрия

- Интерпретатор: доля deterministic/agent путей, доля `stale`, размер отчёта
  по корзинам.
- Кэш: hit/miss по уровням, длительность холодной/тёплой загрузки, длительность
  инкрементальной инвалидации.
- Интенты: длительность run, доля cancelled/stale.
- Предпросмотр: частота `preview-blocked`, глубина «N правок назад».

## 6. Требования к тестам

По ADR-038 (policy layer поверх текущих runners):

- **Unit (Vitest, editor-engine)**: инверсия/атомарность createEntity и
  deleteEntity; классификация риска; конфликт read/write множеств; быстрый
  путь интерпретатора (позитив/негатив); корзины отчёта; fixture-stale.
- **Contract (Ajv)**: `state-fixture.schema.json` + примеры; расширение
  adapter-контракта.
- **Snapshot**: сериализация проекции (диагностический снимок) как регрессия
  кэша уровня 2.
- **E2E (Playwright)**: сценарии — создать сущность из прототипа
  перетаскиванием; правка в «Превью» с плашкой и применением; выделение
