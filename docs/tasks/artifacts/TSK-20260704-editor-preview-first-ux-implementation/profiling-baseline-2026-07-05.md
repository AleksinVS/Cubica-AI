  `TSK-20260704-editor-preview-first-ux-implementation`).
- **Дата**: 2026-07-05.
- **Назначение**: воспроизводимо измерить, куда уходит время при холодной
  загрузке editor-проекций, по модульным границам `editor-engine`, чтобы кэш
  Phase 2 (`docs/architecture/editor-preview-first-ux.md` §10; design-spec §2.6,

- **Сырые данные**: `.tmp/profile-editor-load-2026-07-05.json` (вне Git;
  перезаписывается при каждом прогоне).
- **Правило старшинства**: этот отчёт — измерение и рекомендация, а не

## 1. Что и как измерялось

### 1.1. Метрики по модульным границам

Замеры разбиты по модульным границам `editor-engine` после модуляризации и по
стадиям, которые редактор проходит при загрузке проекта. Стабильные имена метрик
(они же ключи в JSON — для сравнения «до/после»):

**Разовая инициализация** (общая для всех игр; цель кэша уровня 3):

| Метрика | Что делает | Код |
| --- | --- | --- |
| `schema.createRegistry` | `createSchemaRegistry()` редактора | `packages/editor-engine/src/schema.ts` |
| `schema.compileValidators` | регистрация 3 authoring-схем редактора + компиляция Ajv-валидаторов | `apps/editor-web/src/lib/editor-json-schema.ts` (набор схем) |
| `compiler.buildAjv` | `authoring-compiler.buildAjv()` — компиляция 8 схем компилятора | `scripts/manifest-tools/authoring-compiler.cjs:78` |

**На каждую игру** (цель кэша уровня 1 «в памяти» и уровня 2 «на диске»):

| Метрика | Что делает | Код |
| --- | --- | --- |
| `read.files` | fs-чтение всех authoring-файлов (game + ui по каналам) | — |
| `parse.jsonOnly` | только `JSON.parse` текстов (диагностика; **не** в составном) | — |
| `parse.documentStore` | `createDocumentStore`: `JSON.parse` + text location map | `document-store.ts` |
| `validate.documents` | `validateDocument` (Ajv + семантические диагностики) | `schema.ts` |
| `graph.projection` | `buildAuthoringGraphProjection` | `graph-projection.ts` |
| `projection.entity` | `buildEditorEntityProjection` (единая проекция ADR-052) | `entity-projection.ts` |
| `tree.entityTreeView` | `buildEntityTreeViewModel` | `tree-view.ts` |
| `tree.jsonTree` | `buildTreeViewModel` (технич. JSON-дерево) | `tree-view.ts` |
| `timeline.chronology` | `buildManifestChronologyTimeline` | `entity-projection.ts` |
| `yaml.entityProjection` | `buildEditorEntityYamlProjection` для представительной сущности | `entity-projection.ts` |
| `compile.game` | компиляция game authoring → runtime manifest | `authoring-compiler.cjs` |
| `compile.ui` | компиляция ui authoring → runtime manifest (сумма по каналам) | `authoring-compiler.cjs` |
| `composite.projectionLoad` | **сумма пути** редактора при каждой загрузке (все стадии выше, кроме `parse.jsonOnly`) | — |

Представительная сущность для YAML-проекции выбирается детерминированно:
максимум facet-источников, ничьи — по `entityId` лексикографически (для
antarctica — `ui-screen:S1_LEFT`, 14 источников).

### 1.2. Методика прогона

- 9 измеряемых итераций после 1 прогревочной (прогрев отбрасывается — снимает
  разовую стоимость JIT-компиляции и трансформации TS-модулей).
- На каждой итерации сначала строится разовая инициализация, затем для каждой
  игры прогоняется весь конвейер; каждая стадия замеряется отдельно
  `process.hrtime.bigint()`, а её выход переиспользуется следующей стадией — это
  честная «сумма пути», без повторной работы.
- По каждой метрике — медиана, min, max. Между итерациями вызывается `global.gc()`
  (флаг `--expose-gc`), чтобы уменьшить дрожание.
- Скрипт пишет только в `.tmp/`.

### 1.3. Характеристики хоста и условия замера

- `nproc` = 4; Node `v22.22.0`; свободная память ≈ 2.8 ГБ.
- **Load average в момент канонического прогона: 4.52 / 3.15 / 2.65** (1/5/15 мин).
- Фон: на хосте постоянно работал сторонний процесс `python manage.py test`
  (≈ 98% одного ядра) плюс сетевой шлюз (≈ 30%). Полностью «тихого» окна
  (load < 1.5) добиться не удалось — нагрузка стороннего процесса не
  контролируется этой задачей (см. [раздел 8](#8-отклонения-и-оговорки)).
- **Как это учтено**: на 4 ядрах при ≈1.3 занятого ядра однопоточному
  бенчмарку доступно свободное ядро, поэтому измерения корректны, но абсолютные
  миллисекунды несут шум ≈ ±20%. Сделаны 3 перекрёстных прогона (load 2.03, 2.63,
  4.52): **доли стадий стабильны во всех прогонах** (это и есть надёжный сигнал
  для приоритизации кэша), а `min` ≈ наименее зашумлённый прогон. В таблицах ниже
  приведены медиана и min; для приёмки «до/после» сравнивать нужно при
  сопоставимой нагрузке и опираться на доли и на min.

### 1.4. Почему такой способ запуска (ESM/TS)

`editor-engine` — ESM/TypeScript: `main = src/index.ts`, импорты с расширением
`.ts`. В репозитории **нет** `tsx` и нет собранного `dist` у пакета. Наименее
хрупкий путь — встроенная поддержка TS в Node (тот же приём, что у
`services/runtime-api` dev: `node --experimental-strip-types`). Но
`document-store.ts` использует TS «parameter property»
(`constructor(private readonly text)`), которую strip-only режим не понимает,
поэтому нужен полный трансформ: `node --experimental-transform-types` (его
проставляет npm-скрипт). Стоимость трансформации — это стоимость загрузки модуля
(однократно, до замеров), поэтому в измеряемые стадии она не входит: замеряется
исполнение уже загруженного JS, алгоритмическая сложность которого от способа
трансформации не зависит.

## 3. Горячие точки

Для тяжёлой игры (antarctica) время распределено так (доля в `composite`):

1. **`compile.game` — 49.8% (≈ 1192 мс).** Компиляция game authoring в runtime
   manifest — безусловно доминирующая стоимость холодной загрузки. Компиляция
   нужна preview-first редактору, чтобы вообще отрисовать предпросмотр.
2. **`graph.projection` — 18.1% (≈ 433 мс).**
3. **`parse.documentStore` — 9.6% (≈ 229 мс).** Из них ≈ 97.7% — построение text
   location map, а не `JSON.parse` (сравнение: `parse.jsonOnly` = 5.3 мс на те же
   файлы). Горячая точка — собственный посимвольный парсер разметки позиций
   (`document-store.ts`, `JsonTextLocationParser`), где `skipWhitespace` дергает
   Unicode-regex `/\s/u.test()` на каждый символ (`document-store.ts:258-262`).
4. **`tree.jsonTree` + `tree.entityTreeView` — 11.1% (≈ 264 мс суммарно).**
5. **`validate.documents` — 4.8% (≈ 115 мс).** Ajv быстрый и масштабируется
   сублинейно (см. [раздел 4](#4-признаки-алгоритмических-проблем-нелинейность)).
6. **Разовая инициализация — ≈ 174 мс** (в основном `schema.compileValidators`
   137 мс — компиляция Ajv-валидаторов authoring-схем редактора).

`timeline.chronology` и `yaml.entityProjection` — доли процента, не приоритет.

## 4. Признаки алгоритмических проблем (нелинейность)

Сопоставление antarctica / simple-choice: входы отличаются в **31.7×** по байтам
и в **8.9×** по числу сущностей. Если стадия линейна по размеру, её отношение
должно быть ≈ 31.7×; больше — сверхлинейность на единицу входа.

| Стадия | antarctica/simple | нормировка на байт | вывод |
| --- | ---: | ---: | --- |
| `compile.game` | **146.4×** | **4.6×** | **сильная сверхлинейность** |
| `parse.documentStore` | 48.4× | 1.5× | мягкая сверхлинейность |
| `tree.jsonTree` | 45.5× | 1.4× | мягкая сверхлинейность |
| `tree.entityTreeView` | 39.3× | 1.2× | почти линейно |
| `graph.projection` | 36.3× | 1.15× | почти линейно |
| `validate.documents` | 19.4× | 0.6× | сублинейно (Ajv) |
| `projection.entity` | 8.7× | (≈ 8.9× по сущностям) | линейно по числу сущностей |

**Главный красный флаг — `compile.game`.** Рост в 146× при росте входа в 31.7×
(и практически совпадает с ростом числа действий 141×) указывает на квадратичный
по структуре (примерно O(узлы × глубина)) проход. Подозреваемый механизм в
`scripts/manifest-tools/authoring-compiler.cjs`:

- `deriveChildSources` (строка 237) для каждого дочернего узла проверяет каждого
  кандидата через `sourceExists` (строка 217), а тот вызывает
  `readPointer(context.authoring, source.pointer)` (строка 195), **заново
  проходя весь путь указателя от корня документа** — для каждого потомка каждого
  узла. На 141 действии с вложенностью это даёт сверхлинейный рост.
- Дополнительно `clone()` (строка 118) = `JSON.parse(JSON.stringify(value))`,
  которым `mergeObjects`/`resolveDefinition` глубоко копируют поддеревья при
  каждом разрешении `_type`.

**В этом срезе не чиню** (по заданию). Фиксирую как алгоритмическую тех-долг с
указателем на код. Прямая связь с §10 UX-документа
(«кэш не должен маскировать алгоритмические проблемы»): кэш компиляции (уровень 3)
скроет эту сверхлинейность на тёплых загрузках, но не устранит её — значит,
`deriveChildSources`/`sourceExists` нужно рассматривать отдельно от кэша.

Вторичный, но некритичный сигнал — `parse.documentStore` и деревья с нормировкой
1.2–1.5× на байт: линейны с высоким константным множителем; отдельного алгофикса
не требуют, снимаются кэшем уровня 2.

## 5. Окупаемость уровней кэша по данным

Расчёт на antarctica (медианы); полное холодное открытие ≈ 2567 мс.

**Уровень 3 — кэш компиляции и валидаторов** (design-spec §2.6): кэш результатов
компилятора по хешу authoring-входов + переиспользование Ajv-валидаторов.
Покрывает: `compile.game` 1192 + `compile.ui` 80 + `validate.documents` 115 +
`schema.compileValidators` 137 + `compiler.buildAjv` 34 = **≈ 1557 мс ≈ 61%**
холодного открытия. **Наивысшая окупаемость.** Стоимость детерминирована и
безопасна (чистая функция от хеша authoring). Замечание: `buildAjv` и
компиляция схем в процессе и так выполняются один раз за сессию; кросс-сессионный
и тёплый выигрыш — это кэш результатов `compile.game/ui` и результатов валидации
на диск по хешу файла.

**Уровень 2 — постоянный кэш на диске** (тёплый старт): по-файловые артефакты
(parse + location map) по хешу файла и проектные артефакты (проекция, граф,
деревья) по сводному хешу входов. Покрывает: `parse.documentStore` 229 +
`graph.projection` 433 + `projection.entity` 66 + `tree.entityTreeView` 110 +
`tree.jsonTree` 154 + `timeline` 0.3 + `yaml` 5 = **≈ 998 мс ≈ 39%**. **Высокая
окупаемость** для повторного открытия неизменённого проекта.

**Уровень 1 — инкрементальная пересборка в памяти** (главный по ADR): **не
уменьшает первое холодное открытие** — он устраняет полную пересборку при
правках. Данные показывают, во что обходится полная пересборка производных,
которую уровень 1 заменяет точечной инвалидацией: `graph.projection` 433 +
`projection.entity` 66 + деревья 264 = **≈ 763 мс на каждую полную пересборку**
(например, по debounce на ввод) для antarctica. Ценность уровня 1 — латентность
цикла редактирования, а не холодный старт.

`read.files` (7 мс) кэшировать смысла нет.

## 6. Рекомендация порядка работ Phase 2

Данные уточняют порядок из design-spec §2.6 (там: профилирование → уровень 1 →
уровень 2/3). Для метрики приёмки «холодная загрузка» картина такая:

1. **Сначала — алгоритмический тех-долг компилятора.** `compile.game` — половина
   холодной загрузки и единственная стадия с сильной сверхлинейностью. По §10
   («кэш не должен маскировать алгоритмические проблемы») её нужно как минимум
   зафиксировать и по возможности исправить (`deriveChildSources`/`sourceExists`,
   `authoring-compiler.cjs`) до/параллельно с кэшем, иначе кэш уровня 3 её
   замаскирует. Это отдельный от данного среза боунд.
2. **Затем — уровень 3 (кэш компиляции + переиспользование валидаторов).** Даёт
   ≈ 61% сокращения холодной загрузки, детерминирован и безопасен — лучшая
   отдача на метрику «cold load» и на приёмку «до/после».
3. **Затем — уровень 2 (диск: parse/location map + проектные проекции).** Ещё
   ≈ 39% на тёплом старте неизменённого проекта. Здесь же выгодна оптимизация
   `parse.documentStore` (location map — 97% стадии).
4. **Уровень 1 (в памяти, инкрементально)** проектировать на модульных границах
   (не поверх монолита) как основу интерактивности: он не влияет на холодное
   открытие, но убирает ≈ 763 мс полной пересборки на каждую правку в тяжёлой
   игре. Его архитектурный приоритет (ADR) сохраняется; для baseline «cold load»
   он вторичен.

Телеметрия §5 (hit/miss по уровням, длительность холодной/тёплой загрузки,
длительность инкрементальной инвалидации) должна с первого среза писать те же
имена стадий, что и этот бенчмарк, чтобы «после» сравнивалось с этим baseline
напрямую.

## 7. Воспроизведение и стабильные имена метрик

```
npm run profile:editor-load                         # 5 итераций (по умолчанию)
npm run profile:editor-load -- --iterations 9       # больше итераций — устойчивее медиана
npm run profile:editor-load -- --games antarctica   # подмножество игр
```

- Скрипт: `scripts/dev/profile-editor-load.mjs`; запуск —
  `node --expose-gc --experimental-transform-types` (см.
  [раздел 1.4](#14-почему-такой-способ-запуска-esmts)).
- Вывод: таблица в stdout + JSON в `.tmp/profile-editor-load-<дата>.json`
  (структура: `host`, `options`, `shared`, `perGame`, `representativeEntity`,
  `inputStats`; у каждой метрики — `median/min/max/samples`).
- Имена метрик (раздел 1.1) — контракт для сравнения «до/после»; менять их в
  Phase 2 нельзя без обновления этого отчёта.
- Перед замером «после» проверить нагрузку (`uptime`), прогонять бенчмарк в
  одиночку, сравнивать при сопоставимом load.

## 8. Отклонения и оговорки

- **Нагрузка хоста.** Добиться `load < 1.5` не удалось: на хосте постоянно шёл
  сторонний `python manage.py test` (≈ 98% ядра), не относящийся к задаче.
  Замеры сделаны на 4-ядерном хосте со свободным ядром; проведено 3 перекрёстных
  прогона (load 2.03 / 2.63 / 4.52). Абсолютные миллисекунды несут шум ≈ ±20%;
  **доли стадий и их ранжирование стабильны во всех прогонах** и являются
  основным выводом. Для точной цифры «до/после» повторить при более низкой
  нагрузке.
- **Продуктовый код не менялся** — только скрипт, npm-скрипт и этот отчёт.
  Существующего фасада `editor-engine` хватило для честного замера всех стадий;
  СТОП-вопросов по недостающему экспорту нет.
- **Найденная алгоритмическая проблема** (`compile.game`, раздел 4) в этом срезе
  не исправлялась — только зафиксирована с указателем на код, как требует задание.
