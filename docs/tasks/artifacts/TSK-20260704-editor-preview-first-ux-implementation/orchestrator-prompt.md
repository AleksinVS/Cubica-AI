## Жёсткие ограничения (нарушение = стоп и эскалация)

- Целевая архитектура НЕ меняется: ты реализуешь уже принятое ADR-057. Если
  для реализации нужно решение, которого нет в ADR-057/UX-документе/спецификации,
  — останови срез, сформулируй вопрос как предложение ADR и передай владельцу
  проекта; не изобретай.
- UI строится строго по эталонному макету (зоны 1–8, обязательные варианты
  состояний). Адаптация к дизайн-системе editor-web разрешена; изменение
  состава, расположения и подписей — нет.
- Платформенная чистота: никаких game-specific веток и hardcoded game id в
  editor-engine/editor-web/runtime-api (ADR-026/ADR-040).
- JSON Schema — SSOT (ADR-025): новые структуры (фикстуры, schema-признаки)
  описываются схемой + Ajv, не императивными проверками.
- Runtime/player не импортируют editor-engine, не читают файлы фикстур и кэша
  (ADR-036/ADR-052). Кэш — никогда не источник истины.
- Все агентские правки — через единый конвейер
  `EditorChangeSet → риск-политика → dry-run → validation → undo journal`;
  опасные операции — через approval envelope (ADR-047).
- Порядок фаз и блокеры из раздела Dependencies TSK обязательны: Phase 2+ не
  раньше завершения модуляризации editor-engine; Phase 4 не раньше фикса
  инверсии undo (correctness TSK, пункт 1).

## Координация с параллельной работой

Работа в репозитории идёт постоянно. Перед КАЖДОЙ фазой: перечитай
`NEXT_STEPS.md` и Handoff Log соседних TSK; собери свежий `git log` по файлам,
которые собираешься менять; при пересечении с активным чужим срезом — сначала
синхронизация (запись в Handoff Log обоих TSK), потом правки. Малые срезы,
зелёные проверки между ними.

## Делегирование субагентам

- **Opus** — срезы с архитектурной связностью и риском: Phase 1 (контракты
  ядра), Phase 2 (кэш и инкрементальная инвалидация), Phase 4 (интерпретатор
  возвращённого намерения), Phase 7 (очередь интентов и конфликты).
- **Sonnet** — механические и UI-срезы по готовой спецификации: Phase 3
  (дерево/панель по макету), Phase 5–6 (оси времени, фикстуры-UI, создание
  сущностей), Phase 8–9 (диагностика, регион, ассеты), тесты по описанным
  случаям, обновление документации.
- Каждому субагенту давай: точную фазу/пункт спецификации, список файлов,
  ограничения из этого промта, команду проверки. Требуй в ответе: список
  изменённых файлов, результат проверок, отклонения от спецификации.
- Результат каждого субагента проверяй сам (диффом и запуском проверок) до
  следующей делегации. После сбора результата закрывай субагента; перед
  отчётом о завершении фазы убедись, что незакрытых субагентов нет
  (CLAUDE.md §2.7).

## Рабочий цикл фазы

1. Свериться с блокерами и соседними TSK.
2. Разбить фазу на срезы ≤ ~300 строк диффа; делегировать.
3. После каждого среза: `npm run verify:editor-engine`,
   `npm run typecheck --workspace @cubica/editor-web`, при UI — e2e из
   design-spec §6; структурные изменения → `.desc.json` +
   `node scripts/dev/generate-structure.js`.
4. Обновить Handoff Log TSK (дата, что сделано, что дальше, известные хвосты).
5. Временные файлы — только в `.tmp/`; не коммитить `.tmp/`.
6. Документацию менять синхронно с кодом; противоречий кода и документации не
   оставлять.

## Definition of Done

Раздел Acceptance TSK: ADR оформлен и PROJECT_ARCHITECTURE.md синхронизирован;
UI соответствует макету; контракты покрыты тестами; кэш подтверждён
телеметрией; правки при прохождении не теряются; сломанная компиляция не гасит
предпросмотр; game-agnostic инварианты и все проверки зелёные.

Начни с Phase 0: ADR-057 уже оформлен — сверь его с `editor-preview-first-ux.md`
и design-spec на расхождения (старшинство у ADR-057), затем проверь готовность
блокеров. Если блокеры не готовы — доложи их статус и предложи безопасный
порядок работ по не-blocked частям Phase 1.
