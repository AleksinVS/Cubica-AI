# Эпик: Предотвращение архитектурного дрейфа и нормализация структуры

## Контекст / Зачем
Анализ репозитория (ревью от 2026-05-02) выявил опасную тенденцию: логика конкретной игры («Antarctica») начала проникать в платформенные слои и архитектурную документацию (ADR). Документация (описание структуры, задачи, манифест репозитория) рассинхронизирована с реальным положением дел. Если не остановить этот дрейф сейчас, платформа Cubica превратится в монолитную игру «Антарктика», а добавление новых игр потребует переписывания ядра.

Цель этого эпика — вернуть проект в непротиворечивое состояние, четко разделить платформу и контент, и зафиксировать правила, предотвращающие подобное в будущем.

## Цели
1. **Очистить ядро платформы:** Вынести все спецификации и хардкод «Антарктики» из платформенных ADR и сервисов в изолированную зону игры.
3. **Устранить структурный мусор:** Навести порядок в черновиках (`draft/`), временных файлах и ассетах.
4. **Зафиксировать границы:** Создать жесткие правила ревью, запрещающие протекание game-specific логики в платформу.

## Разбиение работ (Phases & Tasks)

### Фаза 1: Очистка архитектурного ядра (Изоляция «Антарктики»)
- [x] **Аудит ADR:** Найти все ADR, описывающие механику «Антарктики» (например, ADR-020...024).
- [x] **Демаркация границ:** Переместить специфичные для игры решения в `games/antarctica/docs/` или `docs/architecture/gameplay-slices/`.
- [x] **Обновление ADR:** В корневых ADR оставить только универсальные платформенные контракты (например, "Платформа поддерживает стейт-машину на основе манифеста", а не "Как работает карточка 39 в Антарктике").
- [x] **Очистка кода Runtime API:** Найти и удалить хардкод `antarctica` в `services/runtime-api/` (например, `DEFAULT_GAME_ID`, специфичную валидацию в `manifestValidation.ts`).

### Фаза 2: Синхронизация источников истины (Documentation Alignment)
- [x] **Автогенерация структуры проекта:** Реализовать скрипт (например, в `scripts/dev/`) для автоматической генерации `PROJECT_STRUCTURE.yaml` (или `.json`) на основе реального состояния файловой системы. Сделать этот автосгенерированный файл единым источником истины для агентов и удалить ручной `PROJECT_STRUCTURE.md`. Обновить ссылки в `AGENTS.md` и `ROADMAP.md`.
- [x] **Исправление гайдов:** Обновить `docs/tasks/README.md` (расширение `.yaml` для ExecPlans) и `PROJECT_OVERVIEW.md` (нумерация и описание архитектуры).

### Фаза 3: Гигиена репозитория (Codebase Cleanup)
- [x] **Зачистка Git-индекса:** Добавить в `.gitignore` и убрать из трекинга мусор (`draft/**/node_modules`, `.tmp/`, `sandbox/nanobanana-output/`).
- [x] **Архивация старых отчетов:** Переместить корневые `ARCHITECT_PLAN.md` и `HANDOFF_REPORT.md` в архив (`docs/archive/`).

### Фаза 4: Механизмы предотвращения (Drift Prevention Guardrails)
- [x] **Правило Легаси-Разрывов:** Добавить в `GEMINI.md` / `AGENTS.md` правило: "Разрыв между текущим состоянием и целевой архитектурой допускается, но должен быть осмысленным, плановым и строго задокументированным как tech debt / legacy. Исправление таких разрывов имеет высокий приоритет".
- [x] **Обновление GEMINI.md / AGENTS.md:** Добавить жесткое правило: "Любая новая механика должна реализовываться через расширение схемы манифеста, а не через добавление if/else для конкретной игры в ядре".
- [x] **CI/CD заглушки:** Заменить `echo "TODO"` в скриптах `lint` и `test` для пакетов и сервисов на реальные вызовы или явные `exit 1` до реализации, чтобы предотвратить деплой неработающего кода.

## Acceptance Criteria
- Ни один платформенный ADR в `docs/architecture/adrs/` не содержит хардкода логики «Антарктики».
- Поиск по строке `PROJECT_STRUCTURE.json` в репозитории выдает 0 результатов (или только исторические упоминания).
- `services/runtime-api` может быть запущен без падения, если удалить папку `games/antarctica/` (должен просто возвращать 404 на запросы контента).
- `git status` чист, мусорные папки игнорируются.
