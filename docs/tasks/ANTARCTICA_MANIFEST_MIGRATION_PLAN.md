# План миграции манифеста "Antarctica" на Action Templates

Этот документ описывает стратегию рефакторинга манифеста `games/antarctica/game.manifest.json` для использования механизма шаблонов действий (Macros), внедренного в рамках ADR-028.

## 1. Цели
*   Сокращение объема файла манифеста с 9000+ строк до ~2000-3000 строк.
*   Повышение читаемости логики игры для ИИ-агентов и разработчиков.
*   Стандартизация повторяющихся паттернов (переходы по шагам, выбор карточек).

## 2. Этап 1: Определение базовых шаблонов

Необходимо добавить секцию `templates` в корень манифеста. Первоочередные шаблоны:

### 2.1. Шаблон `info_advance`
Для простых переходов между экранами информации (Intro/Outro).
*   **Параметры:** `stepIndex`, `nextStepIndex`, `summary`, `legacyId`.
*   **Логика:** `guard` проверяет текущий шаг, `stateUpdate` устанавливает следующий индекс.

### 2.2. Шаблон `card_resolution`
Для стандартного выбора карточки на доске.
*   **Параметры:** `stepIndex`, `cardId`, `summary`, `pro`, `rep`, `lid`, `stat`, `time`.
*   **Логика:** `guard` проверяет, что карточка не выбрана, `metricDeltas` применяет дельты, `stateUpdate` ставит флаг `selected: true`.

### 2.3. Шаблон `info_variant`
Для экранов информации с условным переходом (например, i19 vs i19_1).

## 3. Этап 2: Пошаговая миграция действий

Миграция должна проводиться итерационно, по логическим блокам игры:

1.  **Intro Flow (i0 - i13)**: Замена ~15 действий на вызовы `info_advance`. Ожидаемая экономия: ~600 строк.
2.  **Opening Boards 1-4 (cards 1-16)**: Замена стандартных действий выбора карточек на `card_resolution`. Ожидаемая экономия: ~1200 строк.
3.  **Team Selection Boundary**: Создание специализированного шаблона для найма членов команды.
4.  **Opening Tail (step 30+)**: Рефакторинг сложных детерминированных хвостов, где сейчас много дублирования JSON-структур.

## 4. Этап 3: Удаление избыточных данных

После перевода действий на шаблоны:
*   Удалить поля `displayName`, `description`, `tags` из конкретных действий, если они могут быть выведены из шаблона.
*   Удалить дублирующуюся секцию `provenance`, вынеся общие части в параметры шаблона.

## 5. Верификация

1.  **Schema Validation**: Проверка манифеста через `services/runtime-api/src/modules/content/manifestValidation.ts`.
2.  **Regression Testing**: Запуск `npm test tests/runtime-api.integration.ts` в `services/runtime-api`.
    *   *Примечание:* Результаты тестов не должны ухудшиться по сравнению с базовым состоянием.

## 6. Риски и митигация

*   **Риск:** Потеря уникальных условий (guards) при массовой замене.
*   **Митигация:** Deep Merge в движке позволяет сохранить уникальный `guard` в действии, перекрывая стандартный из шаблона.
*   **Риск:** Ошибки типизации при подстановке параметров.
*   **Митигация:** Использование строгой JSON-схемы с паттернами для плейсхолдеров.
