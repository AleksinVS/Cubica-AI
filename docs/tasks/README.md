# Система задач, планов и передачи артефактов

Документ описывает текущую систему планирования Cubica после принятия ADR-031. Старая многоуровневая модель `Milestone -> Epic -> Feature -> ExecPlan` перенесена в архив и больше не используется для новой активной работы.

## 1. Принцип

В проекте используется легковесная система:

- `docs/tasks/STRATEGY.md` — стратегические направления и правила приоритизации.
- `NEXT_STEPS.md` — текущая доска проекта.
- `docs/tasks/active/` — активные рабочие задачи.
- `docs/tasks/artifacts/` — постоянные артефакты рабочих задач.
- `docs/tasks/archive/` — архив старой системы и завершенных задач.

Главное правило: одна активная работа имеет один корневой `TSK-*` с общим планом. В обычном режиме развития через конкретную игру агент формирует и реализует этот план самостоятельно после согласования существенных архитектурных вопросов. Утверждение всего плана требуется только в явно выбранном `$cubica` или по отдельному указанию PM. Самостоятельные результаты могут получать дочерние `TSK-*`, но отдельный формат `PLN-*` и более глубокая постоянная иерархия не используются.

## 2. Текущая структура

```text
docs/tasks/
  README.md
  STRATEGY.md
  ROADMAP.md
  active/
    TSK-YYYYMMDD-short-slug.md
  artifacts/
    TSK-YYYYMMDD-short-slug/
  archive/
    README-legacy-task-system.md
    milestones/
    epics/
    features/
    content-packs/
    YYYY/
```

`ROADMAP.md` оставлен как исторический документ старой системы. Новая текущая доска ведется в `NEXT_STEPS.md`.

## 3. Горизонты планирования

Планирование разделено на три горизонта:

- `docs/tasks/STRATEGY.md` — месяцы: стратегические темы и правила приоритизации.
- `NEXT_STEPS.md` — дни или недели: текущая доска `Now / Next / Later / Blocked`.
- `docs/tasks/active/TSK-*` — корневой исполняемый план либо самостоятельная дочерняя задача с собственной приемкой, проверками и передачей.

Стратегия не должна содержать подробные команды и списки файлов. Такие детали должны жить в активной задаче.

## 4. Как выбрать следующую работу

Перед началом работы агент или разработчик читает:

1. `AGENTS.md`.
2. `PROJECT_OVERVIEW.md`.
3. `PROJECT_STRUCTURE.yaml`.
4. `docs/architecture/PROJECT_ARCHITECTURE.md`.
5. `docs/tasks/STRATEGY.md`, если нужно понять приоритет между направлениями.
6. `NEXT_STEPS.md`.
7. Активный файл из `docs/tasks/active/`, если он указан в `NEXT_STEPS.md`.

Если корневой задачи нет, создается новый `TSK-*` в `docs/tasks/active/` и ссылка добавляется в `NEXT_STEPS.md`. До реализации его раздел `Plan Approval` должен зафиксировать требуется ли отдельное утверждение плана.

## 5. Формат рабочей задачи

Имя файла:

```text
docs/tasks/active/TSK-YYYYMMDD-short-slug.md
```

Минимальная структура:

```md
# TSK-YYYYMMDD-short-slug: Название

## Status

planned | awaiting_approval | approved | in_progress | blocked | review

## Parent

none | TSK-YYYYMMDD-parent-slug

## Why

Зачем нужна работа и какой результат должен увидеть пользователь или команда.

## Scope

Что входит в работу и что явно не входит.

## Plan Approval

Ревизия плана и режим согласования. Для явно выбранного `$cubica` или отдельного требования PM указываются `approved`, кто и когда утвердил план. Для обычного режима развития через игру указывается `not_required` и ссылка на продуктовые вводные; существенные архитектурные решения при этом согласуются отдельно. Для дочерней задачи допустимо `inherited`, если она полностью укладывается в границы родителя.

## Plan

Общий план корневой задачи либо ограниченный план дочерней. План должен быть исполнимым, но не должен превращаться в старый ExecPlan на десятки страниц.

## Execution

Текущая декомпозиция и сводный статус выполнения. Оркестратор может менять этот раздел в пределах зафиксированных цели, границ и критериев приемки.

## Acceptance

Проверяемые критерии приемки.

## Validation

Команды и ручные проверки, которые должны подтвердить результат.

## Artifacts

Список постоянных артефактов с путями и назначением.

## Plan Amendments

Существенные изменения общего плана. В режиме, где план требует утверждения, здесь фиксируется и повторное согласование. В обычном режиме здесь фиксируется изменение границ и ссылки на отдельно согласованные архитектурные решения. Обычная внутренняя корректировка исполнения сюда не попадает.

## Handoff Log

Журнал передачи между исполнителями.
```

Завершенные (`done`) и замененные (`dropped`) задачи не остаются в
`docs/tasks/active/`: после фиксации итогового журнала их перемещают в
`docs/tasks/archive/`, сохраняя исходную историю.

## 6. Родительские и дочерние задачи

Корневая задача содержит цель, границы, основные результаты, архитектурные ограничения и общую приемку. Она остается единственным источником общего плана. Если выбран `$cubica`, эти границы утверждаются человеком; в обычном игровом режиме их формирует агент по вводным PM.

Дочерняя задача создается оркестратором только когда результат имеет отдельного исполнителя или приемку, выполняется параллельно, передается между сессиями либо должен существовать дольше текущего запуска. Она:

- содержит `Parent: <TSK-ID>`;
- не дублирует общий план и архитектуру родителя;
- не меняет границы родителя;
- имеет собственные `Scope`, `Acceptance`, `Validation` и `Handoff Log`;
- обновляет сводный статус родителя после завершения.

Нормативная глубина ограничена одним уровнем. Мелкие шаги и задания субагентам хранятся в `.tmp/agent-workflow/<task-id>/`.

## 7. Согласование и автономное исполнение

Проектный `$cubica` применяется только когда пользователь явно назвал навык или недвусмысленно попросил использовать автономный процесс Cubica. Сходство запроса с его описанием, сложность работы или инициатива агента не являются основанием для автоматического включения `$cubica`. Остальные навыки, включая перенесенные или адаптированные из `agent-skills` и `superpowers`, могут включаться автоматически по собственным правилам.

В обычном режиме развития через конкретную игру PM дает продуктовые вводные. Агент самостоятельно выбирает один вертикальный срез, готовит общий план и выносит на согласование только существенные архитектурные решения. После их согласования либо, если новых решений нет, сразу реализует план. Выбор внутренних имен, библиотек, декомпозиции, исполнителей и методов проверки не требует согласования.

Если пользователь явно выбрал `$cubica`, действует отдельная граница ADR-068: перед реализацией человек утверждает общий план корневой задачи. Явная команда реализовать ранее рассмотренный план считается утверждением и фиксируется в `Plan Approval`. После утверждения оркестратор работает автономно в его границах.

В обоих режимах права, секреты и разрушительные внешние операции требуют отдельного подтверждения безопасности.

При частичном блокере независимые части плана продолжают выполняться. Корневая задача закрывается только после приемки обязательных дочерних результатов и общей проверки.

## 8. Артефакты

Постоянные артефакты задачи хранятся в:

```text
docs/tasks/artifacts/<TASK-ID>/
```

Примеры постоянных артефактов:

- отчет сверки;
- миграционная таблица;
- спецификация;
- итоговый анализ;
- сгенерированный файл, который нужен для дальнейшей работы.

Временные артефакты хранятся только в `.tmp/` и очищаются после завершения работы:

- скриншоты проверки;
- временные логи;
- дампы ответов;
- промежуточные JSON-файлы;
- локальные debug scripts, если они не становятся поддерживаемым инструментом.

## 9. Передача работы

Каждая остановка работы должна оставлять запись в `Handoff Log`.

Запись передачи содержит:

- дату;
- исполнителя;
- измененные файлы;
- выполненные команды и результат;
- что уже сделано;
- что осталось;
- следующий безопасный шаг;
- известные риски.

Пример:

```md
### 2026-05-18 — AI agent

- Changed: `services/runtime-api/src/...`, `docs/...`
- Validation: `npm test --workspace services/runtime-api` failed, 54/71 passed.
- Done: isolated failing opening-flow assertions.
- Remaining: fix action advance transition after template migration.
- Next: start from `services/runtime-api/tests/runtime-api.integration.ts`.
- Risks: manifest and runtime handler may disagree on action IDs.
```

## 10. Связь с ADR и gameplay slice records

ADR используется только для архитектурных решений: долгосрочные ограничения, последствия и отвергнутые альтернативы.

Рабочая задача используется для исполнения: план, шаги, проверки, артефакты и передача.

Gameplay Slice Records используются только для delivery-specific деталей gameplay-срезов. Они не заменяют рабочую задачу и не являются общей системой планирования.

Если рабочая задача требует архитектурного решения, сначала создается или обновляется ADR, а затем задача ссылается на него.

## 11. Архив старой системы

Старые каталоги перенесены в:

- `docs/tasks/archive/milestones/`
- `docs/tasks/archive/epics/`
- `docs/tasks/archive/features/`
- `docs/tasks/archive/content-packs/`

Старый README системы задач сохранен в `docs/tasks/archive/README-legacy-task-system.md`.

Архивные файлы сохраняются как история проекта. Новую активную работу в этих каталогах не создавать.
