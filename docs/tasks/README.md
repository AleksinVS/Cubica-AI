# Система задач: структура и правила

Проект хранит иерархию задач прямо в репозитории в виде Markdown‑чек‑листа ROADMAP.md и файлов задач разных уровней. 
Изначально задача описывается в ROADMAP.md одной строкой (без создания файла задачи) и, при необходимости, расширяется (конкретизируется) в файлах (Milestone/Epic/Feature). 
Это обеспечивает прозрачность, ревью через PR и отслеживание прогресса из кода.

## Уровни и файлы

Иерархия: Веха → Эпик → Фича → План выполнения (ExecPlan).

Структура каталога:

```
docs/tasks/
  ROADMAP.md                 # Дорожная карта (сводная)
  milestones/                # Вехи релизов/фаз
    _TEMPLATE_MILESTONE.md
  epics/                     # Крупные направления работ
    _TEMPLATE_EPIC.md
  features/                  # Конкретные фичи в рамках эпиков
    _TEMPLATE_FEATURE.md
```

## Идентификаторы и имена

Префиксы: M (Milestone), E (Epic), F (Feature), CP (ExecPlan).
Рекомендуемые правила имени файла: `<ПРЕФИКС>-<ГОД ГГ>-<Счётчик>-kebab-title.md`.
ExecPlan: `CP-YY-XXXXR-<kebab-title родителя-фичи*>.yaml` 
Последняя цифра счетчика (`R`) резервируется для дополнительных задач и планов.
Номер и заголовок ExecPlan отличается от номера родительской задачи только префиксом.
Если нужен ExecPlan без родительской задачи, то используются резервная цифра счетчика.
`YY` - последние две цифры года в котором была создана задача.

- Веха: `M_010-platform-bootstrap.md`
- Эпик: `E_0010-editor-mvp.md`
- Фича: `F_00010-router-auth.md`
- ExecPlan: `CP_00010-router-auth.yaml`

## Метаданные (YAML)

Каждый файл верхнего уровня (веха/эпик/фича) начинается YAML‑блоком:

```
---
id: E_0010
title: MVP редактора игр
status: planned # planned|in_progress|blocked|review|done|dropped
owner: @username
milestone: M_010
area: game-editor
tags: [priority:P1, type:feature]
links:
  - docs/architecture/README.md
  - services/game-editor/DEV_GUIDE.md
---
```

## Теги (tags)

- Приоритет: `priority:P0|P1|P2|P3` (P0 — срочно, P3 — низкий)
- Тип: `type:feature|bug|chore|spike|doc`
- Дополнительно: `risk:high|med|low`, `effort:S|M|L`
- Кросс‑области: при необходимости используйте доп. теги `area:<name>`

## Правила декомпозиции

- Начинайте с пункта в `ROADMAP.md`. Когда область растёт, вынесите в отдельный файл эпика/фичи по шаблону и дайте стабильный `id`.
- Для прогресса используйте чек‑листы `- [ ]`/`- [x]`. Никаких «скрытых» статусов — всё в явном виде.
- Названия невыполненных задач в `ROADMAP.md` отмечайте жирным шрифтом (`**Название задачи**`), выполненные задачи пишите обычным шрифтом.
- Внутри разделов используйте уровни заголовков для иерархии и простые списки задач для читаемости.
- Старайтесь делать строки короткими, 1 строка — 1 мысль.

## Единые статусы

- draft - черновик (до момента полного утверждения)
- planned — запланировано
- in_progress — в работе
- blocked — заблокировано (указывайте причину)
- review — на ревью
- done — готово
- dropped — отменено

## Обязательные разделы и чек‑листы

Для эпиков/фич используйте типовые блоки:

- Контекст/Зачем
- Цели и метрики
- Объём (In/Out of scope)
- Разбиение работ (Stories/Tasks)
- Acceptance Criteria (проверяемые критерии)
- Definition of Done:
  - [ ] Обновлена документация (README, DEV_GUIDE, `docs/architecture/PROJECT_ARCHITECTURE.md` if needed)
  - [ ] Покрыты тестами ключевые сценарии
  - [ ] Структура в PROJECT_STRUCTURE.md актуальна
  - [ ] ExecPlans (`docs/tasks/content-packs/*`), Milestones, Epics, Features, ROADMAP.md обновлены
  - [ ] Демо/скринкаст добавлен (по необходимости)
  - [ ] Релизные заметки/миграции описаны
  - [ ] CI зелёный, линтеры/форматтеры пройдены

 Прогресс только через `- [ ]` и `- [x]`.
 Для эпиков/фич используйте разделы из шаблонов и DoD из `docs/tasks/README.md`.
 
## Связь с существующими документами

- Архитектура/ADR: `docs/architecture/*`
- Гайды сервисов: `services/*/DEV_GUIDE.md`

## ADR — когда нужен и как оформить

ADR обязателен при планировании архитектурных, кросс‑сервисных и Backward‑incompatible изменений.

- Шаблон: `docs/architecture/adrs/000-template.md`
- Размещение: `docs/architecture/adrs/NNN-your-title.md` (NNN — номер с ведущими нулями)

## Поиск и обзор прогресса

Быстрые команды:

- Открытые пункты: `rg -n "^- \\[ \\]" docs/tasks`
- Все завершённые: `rg -n "^- \\[x\\]" docs/tasks`
- По идентификатору: `rg -n "E-\d{2}-\d+|F-\d{2}-\d+|M-\d{2}-\d+" docs/tasks`

## Практики ведения

- Один PR — одна осмысленная часть списка (подраздел эпика/фичи).
- В описании PR указывайте ID задачи: `F-25-0010`.
- После мержа не забывайте проставлять галочки и статус.

## Иерархия задач


Полное дерево уровней: Веха → Эпик → Фича - ExecPlan. Рекомендуемая раскладка файлов и навигации:

```
docs/tasks/
  ROADMAP.md                      # сводка по вехам/эпикам (высокий уровень)
  milestones/
    M-YYR-NNR-<kebab>.md           # веха: цели, список эпиков (ссылками)
  epics/
    E-YY-XXXR-<kebab>.md         # эпик: контекст, фичи (ссылками), критерии
  features/
    F-YY-XXXXR-<kebab>.md         # фича: задачи/подзадачи и DoD
  content-packs/
    CP-YY-XXXXR-<kebab>.md         # ExecPlan: фичи (ссылками), план реализации
  
  # R - резервная цифра счетчика
  # YY - последние две цифры года в котором была создана задача
```

Пример чек‑листов (дерево):

```
- [ ] **Alpha-этап игрового плеера**
      [M_010](milestones/M_010_game_player_alpha.md)

      - [ ] **Архитектура JSON-манифестов игр и LLM-first плеера**

            - [ ] **Базовый формат JSON-манифеста игр и схем**
                  [F_00010](features/F_00010-game-manifest-format-and-schema.md)
                    [CP_00010](content-packs/CP_00010-antarctica-json-manifest.yaml)
            - [x] Базовая схема UI (Hybrid SDUI)
                  [F_00020](features/F_00020_ui_schema_definition.md)

      - [ ] **Antarctica на Next.js game player**
            [E_0020](epics/E_0020_antarctica_nextjs_game_player.md)

            - [ ] **JSON-манифест сценария «Antarctica» для Next.js-плеера**
            - [x] Antarctica — разделение game/ui манифестов, протокол command/payload и пакет игры в games/

- [x] MVP
```

### Раскрытие новых / Сворачивание завершённых

- Если задача запланирована, но еще не детализирована, то файл задачи (Milestone/Epic/Feature) не создается и задача описывается в ROADMAP.md.
- Если задача детализирована и ее нельзя целиком описать 1-2 небольшими абзацами текста, то создается файл задачи (Milestone/Epic/Feature) и в ROADMAP.md добавляется ссылка на этот файл. 
- Задачи старайтесь держать «инлайн» в ROADMAP.md, создавая файлы только если описание становится достаточно большим.
- Полностью выполненные вехи и эпики в `ROADMAP.md` сворачивайте до одной строки с кратким описанием, статусом `done` и ссылкой на файл вехи или эпики. Подробные списки остаются в соответствующих файлах в `milestones/`, `epics/`, `features/`.
