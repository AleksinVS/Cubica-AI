## Цель

Зафиксировать минимальные правила ревью (review — проверка изменений перед слиянием), чтобы:

- изменения были понятны новым участникам;
- архитектурные решения не расходились между сервисами/SDK.

## MVP-правила

- Один PR — одна осмысленная задача (ссылка на `docs/tasks/`).
- Для изменений архитектуры — ADR в `docs/architecture/adrs/`.
- Обязательна проверка ссылок в документации и актуальности `PROJECT_STRUCTURE.yaml` при изменениях структуры.

## Завершение агентской ветки

1. Агент завершает работу в отдельной ветке, фиксирует изменения коммитом,
   отправляет ветку и сообщает пользователю результат и выполненные проверки.
2. До слияния в `main` агент получает явное согласование пользователя. Прямая
   команда «слей в main» является достаточным согласованием и не требует
   повторного подтверждения.
3. После согласования агент сам обновляет удалённые ссылки, сливает ветку с
   последним `main`, разрешает конфликты с сохранением обеих одобренных линий,
   выполняет соразмерные итоговые проверки и отправляет обновлённый `main`.
4. Работа не считается интегрированной, пока её изменения существуют только в
   агентской ветке. Исключение возможно только по прямому указанию пользователя
   либо при техническом ограничении защищённой ветки, о котором агент сообщает
   вместе с точным оставшимся действием.
5. Если основное рабочее дерево содержит посторонние незакоммиченные изменения,
   слияние выполняется в отдельной чистой рабочей копии без изменения или
   удаления этих файлов.
