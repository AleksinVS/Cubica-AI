# Локальная диагностика браузера через Chrome DevTools MCP

Chrome DevTools MCP (сервер Model Context Protocol, который даёт ИИ-агенту
управляемый доступ к инструментам разработчика Chrome) используется в Cubica
как необязательный локальный инструмент диагностики. Он не входит в продукт,
не является зависимостью репозитория и может отсутствовать на рабочем месте
другого разработчика или агента.

## 1. Когда использовать

Chrome DevTools MCP подходит для интерактивной диагностики работающего
интерфейса:

- просмотра DOM, сетевых запросов и сообщений консоли;
- проверки вычисленных стилей и поведения страницы;
- анализа производительности и создания диагностических снимков экрана.

Для воспроизводимых автоматических браузерных проверок используйте Playwright.
Сценарий, который должен стабильно выполняться в CI или подтверждать
регрессию, оформляется как тест Playwright, а не как последовательность ручных
действий через Chrome DevTools MCP.

## 2. Как проверить доступность

Перед использованием не предполагайте, что инструмент установлен:

```bash
codex mcp get chrome-devtools
```

Рабочая конфигурация должна иметь состояние `enabled`. После первой
регистрации может потребоваться перезапуск Codex, чтобы новый сервер появился
в текущей сессии.

## 3. Рекомендуемый локальный профиль

Для Cubica применяется ограниченный профиль:

- Chrome запускается без графического окна и с временным изолированным
  профилем;
- разрешены только `http://localhost:*/*` и `http://127.0.0.1:*/*`;
- сбор статистики использования и запросы к Google CrUX отключены;
- чувствительные сетевые заголовки скрываются в результатах инструмента;
- версия `chrome-devtools-mcp` фиксируется явно и обновляется отдельно с
  повторной проверкой.

На настроенном Linux-хосте используется `chrome-devtools-mcp@1.6.0` и Chrome
for Testing `151.0.7922.47`. Эти версии описывают текущее локальное окружение,
но не являются контрактом продукта.

## 4. Установка на Linux-хосте

Установите отдельный Chrome for Testing в пользовательский каталог:

```bash
npx --yes @puppeteer/browsers@3.0.4 install \
  chrome@stable \
  --path "$HOME/.cache/chrome-for-testing"
```

Найдите полученный исполняемый файл `chrome-linux64/chrome` внутри каталога
`$HOME/.cache/chrome-for-testing`, затем зарегистрируйте MCP-сервер, заменив
`<путь-к-chrome>` на абсолютный путь:

```bash
codex mcp add chrome-devtools \
  --env CHROME_DEVTOOLS_MCP_NO_UPDATE_CHECKS=1 \
  -- npx --yes chrome-devtools-mcp@1.6.0 \
  --headless \
  --isolated \
  --no-usage-statistics \
  --no-performance-crux \
  --redact-network-headers \
  --executable-path <путь-к-chrome> \
  --allowed-url-pattern 'http://localhost:*/*' \
  --allowed-url-pattern 'http://127.0.0.1:*/*' \
  --chrome-arg=--no-sandbox \
  --chrome-arg=--disable-setuid-sandbox
```

Параметры `--no-sandbox` и `--disable-setuid-sandbox` нужны только на текущем
Linux-хосте, где AppArmor блокирует штатную песочницу Chrome. На окружении, где
Chrome успешно работает с песочницей, эти два параметра следует удалить.

## 5. Ограничения безопасности

Фильтр разрешённых адресов не является полноценной сетевой песочницей:
ограничение действует на цели Chrome DevTools и связанные ресурсы только пока
сервер подключён. Профиль без песочницы Chrome разрешено использовать
исключительно для доверенных локальных страниц Cubica.

Не используйте этот профиль для внешних или недоверенных сайтов, не вводите в
него рабочие учётные данные и не подключайте существующий пользовательский
профиль Chrome. Для такой работы сначала требуется отдельная изолированная
среда с исправной системной песочницей.

## 6. Проверка после установки

Проверьте регистрацию и реальный запуск браузера:

```bash
codex mcp get chrome-devtools

<путь-к-chrome> \
  --headless=new \
  --no-sandbox \
  --disable-setuid-sandbox \
  --disable-gpu \
  --dump-dom about:blank
```

Ожидаемый результат второй команды содержит пустой документ
`<html><head></head><body></body></html>`. После проверки убедитесь, что не
остались процессы `chrome-devtools-mcp` или тестового Chrome, запущенные именно
этой проверкой.
