# Runbook: восстановление сервисов на одном хосте

Операционная (исполнительная) инструкция по запуску и восстановлению сервисов
платформы на **одном хосте** через минимальный супервайзер (ADR-067). Это
промежуточный механизм до перехода на оркестратор (LEGACY-0026); границы и
условия замены — в [ADR-067](../architecture/adrs/067-minimal-service-recovery.md).

ИИ-агенты применяют эти правила через проектный навык
[`low-memory-host-operations`](../../.codex/skills/low-memory-host-operations/SKILL.md),
который добавляет безопасную предварительную проверку памяти, swap и
конкурирующих сборок.

## Что это и что оно восстанавливает

«Супервайзер» (`scripts/ops/service-supervisor.mjs`) — родительский процесс,
который запускает сервисы, следит за ними и **перезапускает** упавший или
переставший отвечать на health-проверку. Он восстанавливает **только процесс**,
а состояние партии восстанавливает `runtime-api` из PostgreSQL. Режим хранения
в памяти допустим только для локальной разработки и тестов; после рестарта он
состояние не сохраняет.

Профиль по умолчанию — прод-контур доставки: **runtime-api** и **player-web**.
Редактор (`editor-web`) — локальный инструмент, в дефолт не входит (LEGACY-0036).

## Предусловия

1. Установлены зависимости: `npm install` (из корня репозитория).
2. **`player-web` собран** — супервайзер запускает его в прод-режиме
   (`next start`), который требует предварительной сборки:
   ```bash
   RUNTIME_API_URL=http://127.0.0.1:3001 \
     npm run build --workspace @cubica/player-web
   ```
   Передавайте тот же `RUNTIME_API_URL`, что будет у runtime-api в проде:
   серверная загрузка игрового содержимого использует этот адрес. Браузерные
   runtime-запросы проходят только через явные защищённые обработчики
   `player-web`; общего прокси-пути для `/api/runtime/*` нет.
3. Для рабочего запуска `runtime-api` доступна PostgreSQL, заданы
   `SESSION_STORE=postgresql` и `DATABASE_URL`, а миграция сессий применена:
   ```bash
   DATABASE_URL='postgresql://...' \
     npm run migrate:sessions --workspace @cubica/runtime-api
   ```
4. `runtime-api` отдельной сборки не требует (запускается напрямую).

## Ресурсный профиль одного хоста

Хост с 8 ГиБ оперативной памяти достаточен для `runtime-api`, собранных
`player-web` и `editor-web`, если веб-приложения работают через `next start`.
Он не рассчитан на две одновременные сборки или два постоянно работающих
`next dev` рядом с браузером, графическим рабочим столом и ИИ-инструментами.

Обязательные правила для такого хоста:

1. Одновременно выполняется не более одной команды `next build`.
2. Во время сборки не запускается второй `next build` или новый `next dev`.
3. Для постоянной совместной работы игрок и редактор запускаются из готовых
   сборок через `next start`; режим разработки оставляют только одному из них.
4. Проверки и сборки разных рабочих областей запускаются последовательно.
5. Перед ресурсоемкой операцией проверяют `MemAvailable` и свободный swap:

   ```bash
   free -h
   awk '/MemAvailable|SwapTotal|SwapFree/ {print}' /proc/meminfo
   ```

   Сборку не начинают при доступной памяти менее 2 ГиБ или заполненном swap:
   сначала безопасно освобождают ресурсы по разделу ниже.
6. Для рабочего хоста с 8 ГиБ оперативной памяти держат суммарно 8 ГиБ swap.
   Это аварийный запас, а не замена последовательной сборке: длительная запись
   в swap означает, что ресурсоемкую операцию нужно остановить или перенести.

## Правило последовательной сборки

Каноническая команда для сборки обоих веб-приложений:

```bash
RUNTIME_API_URL=http://127.0.0.1:3001 npm run build:web:sequential
```

Команда использует `&&`: сборка редактора начинается только после успешного
завершения сборки игрока. Не запускайте две отдельные команды сборки в разных
терминалах и не оборачивайте их в `&`, `concurrently` или `Promise.all`. Перед
стартом проверьте, что другой терминал, агент или CI-процесс не выполняет
`next build` в том же рабочем каталоге:

```bash
pgrep -af 'next build'
```

Параллельные сборки опасны не только пиком памяти: они одновременно изменяют
общий каталог `.next` и могут завершиться ошибкой об исчезнувшем служебном файле
(`ENOENT`, `*.nft.json`). Для настоящего параллельного CI каждой сборке нужен
отдельный рабочий каталог; на этом хосте используется последовательный режим.

Обе конфигурации Next.js ограничивают сборку одним рабочим процессом и включают
режим уменьшенного потребления памяти Webpack. Проверки типов при этом не
отключаются. Если нужно исследовать новый рост памяти, запускайте только одну
сборку с диагностикой Next.js:

```bash
NEXT_IGNORE_INCORRECT_LOCKFILE=1 \
  npm exec --workspace @cubica/editor-web -- \
  next build --experimental-debug-memory-usage
```

Снимки кучи и временные трассы складывайте только в `.tmp/` и удаляйте после
анализа.

## Совместный запуск игрока и редактора

Рекомендуемые режимы, от самого устойчивого к самому ресурсоемкому:

| Игрок | Редактор | Когда применять |
|---|---|---|
| `next start` | `next start` | постоянная совместная работа и демонстрация |
| `next start` | `next dev` | разработка редактора с работающей игрой |
| `next dev` | `next start` | разработка игрока с доступным редактором |
| `next dev` | `next dev` | не использовать на хосте с 8 ГиБ |

После последовательной сборки приложения можно запустить на разных портах:

```bash
npm run start --workspace @cubica/player-web -- --port 3000
npm run start --workspace @cubica/editor-web -- --port 3002
```

Запускайте их под супервайзером или отдельными `systemd`-службами, если они
должны переживать закрытие терминала. `runtime-api` остается на порту `3001`.
Ошибка браузера `Failed to load chunk` после исчезновения процесса Next.js —
вторичный симптом: сначала проверяйте порт и журнал процесса, а не очищайте код
или данные редактора.

## Запуск

Из корня репозитория:

```bash
npm run ops:supervise
```

Супервайзер поднимет сервисы, начнёт health-опрос и будет держать их живыми,
пока сам работает. Логи мультиплексируются в stdout с префиксом сервиса —
перенаправьте их в файл/журнал по вкусу окружения:

```bash
npm run ops:supervise >> /var/log/cubica/supervisor.log 2>&1
```

Как долгоживущий процесс на хосте супервайзер сам должен быть под неубиваемым
запуском окружения (`nohup`/`setsid`, либо один systemd-юнит на сам супервайзер).
Это единственная точка, которую поднимает окружение; остальное держит супервайзер.

## Порты и переменные окружения

| Переменная | Назначение | По умолчанию |
|---|---|---|
| `RUNTIME_API_PORT` | порт runtime-api | `3001` |
| `PLAYER_WEB_PORT` | порт player-web | `3000` |
| `SUPERVISOR_CONFIG` | путь к JSON-конфигу набора сервисов | встроенный профиль |
| `SESSION_STORE` | режим хранения сессий | обязателен; `postgresql` для рабочего запуска |
| `DATABASE_URL` | строка подключения PostgreSQL | обязательна для `postgresql` |

`player-web` получает `RUNTIME_API_URL=http://127.0.0.1:<RUNTIME_API_PORT>`
автоматически. Тот же URL следует использовать при сборке `player-web` (см.
Предусловия), чтобы серверная загрузка содержимого и рабочий запуск обращались
к одному runtime-api.

## Проверка состояния

```bash
curl -fsS http://127.0.0.1:3001/health      # liveness runtime-api (200)
curl -fsS http://127.0.0.1:3001/readiness   # readiness runtime-api (200 готов / 503 нет)
curl -fsI http://127.0.0.1:3000/            # liveness player-web (200)
```

В логах супервайзера строки вида `[supervisor] [<сервис>] …` показывают старты,
падения, рестарты и деградацию; строки `[<сервис>] …` — вывод самого сервиса.

## Остановка

Пошлите супервайзеру `SIGINT` (Ctrl+C в интерактиве) или `SIGTERM`. Он корректно
остановит все дочерние сервисы (гасит группу процессов, освобождая порты) и
завершится. Отдельно гасить дочерние сервисы не нужно.

## Настройка набора сервисов

Чтобы изменить набор/команды/порты (например, добавить `editor-web` или
supervise на других портах), передайте JSON-конфиг:

```bash
SUPERVISOR_CONFIG=./ops/services.json npm run ops:supervise
```

Форма конфига:

```json
{
  "healthIntervalMs": 5000,
  "services": [
    {
      "name": "runtime-api",
      "command": "npm",
      "args": ["run", "dev", "--workspace", "services/runtime-api"],
      "env": { "PORT": "3001" },
      "port": 3001,
      "healthPath": "/health",
      "readinessPath": "/readiness",
      "startGraceMs": 60000
    }
  ]
}
```

- `readinessPath` опционален: если задан, опрашивается он (богаче сигнал), иначе
  `healthPath`.
- `startGraceMs` — окно после старта, в течение которого неудачные health-пробы
  игнорируются (сервис ещё поднимается).

## Что делает супервайзер сам

- **Рестарт при падении** дочернего процесса — с экспоненциальным backoff
  (капнут на 30с).
- **Рестарт по здоровью**: 3 подряд неуспешные health-пробы → сервис гасится и
  перезапускается.
- **Предохранитель** (circuit breaker): при частых падениях (>5 за 60с) сервис
  удерживается на максимальном backoff и логируется как `DEGRADED`; супервайзер
  не входит в быстрый цикл и не падает сам.
- **Групповое гашение**: рестарт/остановка убивают всё дерево процесса сервиса
  (`npm`→`node`/`next`), поэтому порт всегда освобождается.

## Защита от нехватки памяти

`earlyoom` — служба, которая завершает один процесс до того, как зависнет весь
хост. Она не должна предпочитать все процессы `node` или `npm`: иначе первыми
жертвами становятся Next.js и `runtime-api`, даже когда основную память занял
браузер или графический рабочий стол.

Рекомендуемый профиль для этого хоста:

```text
EARLYOOM_ARGS="-r 60 -m 10 -s 20 --prefer '(^|/)(buildkitd|knowledge_ingest)( |$)' --avoid '(^|/)(systemd|systemd-journald|sshd|gnome-shell|Xorg|Xwayland|necobox|necoray|vray)( |$)'"
```

Здесь `--prefer` содержит только заведомо перезапускаемые фоновые операции, а
`--avoid` защищает системную сессию и перечисленные сетевые службы. Отключать
`earlyoom` полностью нельзя: без него при исчерпании памяти может зависнуть весь
хост. После изменения `/etc/default/earlyoom` примените только его конфигурацию:

```bash
sudo systemctl restart earlyoom
systemctl is-active earlyoom
systemctl show earlyoom --property=Environment
```

Для постоянных экземпляров Cubica предпочтительны отдельные `systemd`-службы с
`OOMScoreAdjust` и ограничениями группы ресурсов. Это отдельная операция
развертывания; не подменяйте ее общим исключением всех процессов `node`.

## Безопасное освобождение памяти

### После тестов и сборок

Каждый тестовый сценарий обязан завершать созданное им дерево процессов в
обработчике `finally` или эквивалентном механизме очистки — в том числе после
ошибки, `SIGINT` и истечения времени ожидания. Это относится к временным
экземплярам Chromium/Playwright, Next.js, `runtime-api` и супервайзера.

После E2E-прогона или прерванной сборки проверьте:

```bash
pgrep -af 'playwright|next (build|dev|start)|run-e2e-prod' || true
ss -ltnp '( sport = :3000 or sport = :3001 or sport = :3002 )'
free -h
```

Завершайте только процессы, которые создал конкретный тестовый запуск. Для
надежной очистки тест должен запускать их в отдельной группе процессов и сначала
посылать группе `SIGTERM`; `SIGKILL` допустим лишь после ограниченного ожидания.
Нельзя завершать совпавший по имени процесс из другого терминала без проверки
его родителя, группы, времени запуска и рабочего каталога.

После проверки удаляйте временные трассы, снимки и журналы конкретного запуска
из `.tmp/`. Не удаляйте автоматически `.next` и кэши пакетов: они ускоряют
следующий запуск и не удерживают оперативную память после завершения процесса.
Не применяйте `drop_caches`, перезапуск системных служб или `swapoff -a` как
обычную «чистку»: Linux сам вытесняет файловый кэш при потребности в памяти, а
такие команды создают дополнительную нагрузку и скрывают утечки процессов.

### При ручном освобождении ресурсов

1. Найдите забытые процессы Cubica и проверьте их время запуска, рабочий каталог
   и порт. Завершайте только подтвержденный устаревший процесс вместе с его
   родительской группой.
2. Закройте неиспользуемые вкладки браузера и ненужные сеансы инструментов
   вручную: агент не может надежно определить, какие пользовательские сеансы
   допустимо завершить.
3. Старые страницы в swap не являются утечкой сами по себе. Ядро повторно
   использует swap по мере необходимости.
4. Не выполняйте `swapoff -a` на нагруженном хосте: перенос нескольких гигабайт
   обратно в оперативную память способен немедленно вызвать новое завершение
   процессов.
5. Не останавливайте несвязанные сетевые службы ради сборки Cubica. В частности,
   `necobox`, `necoray` и `vray` находятся вне контура обслуживания проекта.
6. Если после закрытия подтвержденных потребителей памяти доступной памяти все
   равно меньше 2 ГиБ, отложите сборку или выполните ее в CI/на отдельном хосте.

## Ограничения и эскалация

- Супервайзер работает на одном хосте и с одной репликой. PostgreSQL
  сохраняет состояние одиночной сесии после рестарта, но нет высокой
  доступности, rolling restart без простоя, распределённой готовности и
  межхостового переключения (LEGACY-0042).
- Если сервис в `DEGRADED` (постоянно падает) — это инцидент: смотрите вывод
  сервиса в логе, следуйте [incident-response.md](./incident-response.md).
- Условия, при которых этот механизм ОБЯЗАН быть заменён оркестратором
  (k8s/serverless), и регистрация разрыва — в
  [ADR-067](../architecture/adrs/067-minimal-service-recovery.md) и долге
  LEGACY-0042.
