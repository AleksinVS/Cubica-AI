## 1. Цели и границы

**Цель:** подготовить платформу к реализации полной классической «Монополии» тремя последовательными моделями доставки (хотсит → сетевой мультиплеер → ИИ-оппоненты), добавив только **общие** возможности (правило 10 `CLAUDE.md`): каждая возможность должна быть полезна классу игр, а не одной игре.

**Границы:**
- Никаких `if (gameId === "monopoly")` в платформенных слоях (ADR-026, ADR-040).
- JSON Schema остаётся SSOT; контракты генерируются (ADR-025, ADR-056).
- Ветвление эффектов остаётся ограниченным (ADR-024): один уровень, без циклов.
- Правила конкретной игры (торги, аукцион, тюрьма, банкротство) — это контент манифеста и Gameplay Slice Records, не платформа.

## 2. Лучшие практики и источники

Проанализирован опыт boardgame.io — зрелого открытого движка пошаговых игр (документация получена через Context7, июль 2026):

| Паттерн boardgame.io | Что берём в Cubica |
|---|---|
| `seed` генератора хранится в состоянии игры; каждое использование сдвигает счётчик — полная воспроизводимость партии | `state.secret.random = {alg, seed, counter}`; обязательное требование к replay (§4.1) |
| `ctx.playOrder`, `ctx.currentPlayer`, `ctx.playOrderPos`, событие `endTurn` | `state.public.turn = {order, activePlayerId, phase, turnNumber}` + эффекты `turn.next`/`turn.repeat` (§4.4) |
| `playerView` / `PlayerView.STRIP_SECRETS`: клиент получает состояние, отфильтрованное для конкретного игрока | Правило проекции Presenter: своя ветка `players.<self>` целиком, чужие — только публичные поля (§4.3) |
| Фазы (`phases`) и этапы (`stages`) хода как машина состояний | Упрощённо: строковое поле `turn.phase`, объявляемое манифестом; полноценная машина фаз не вводится до подтверждённой потребности |
| Ходы (`moves`) как императивные JS-функции | **Не берём**: противоречит декларативной модели манифестов (ADR-025); вместо этого — реестр действий с guard/effects |

Ключевые собственные решения, уже принятые ранее и переиспользуемые: структура `world/players/secret` и очередь событий (ADR-011), фасетные объекты и generic guards (ADR-041), CI-подмножество JsonLogic (`scripts/ci/validate-metric-jsonlogic-subset.js`).

## 3. Инвентаризация правил полной классической «Монополии»

Термины: **рента** — плата за остановку на чужой собственности; **залог (mortgage)** — временная сдача собственности банку за половину цены; **дубль** — одинаковые значения на двух костях.

| # | Правило (полная классика) | Требуемая возможность | Уровень |
|---|---|---|---|
| 1 | 2–6 игроков, порядок хода по броску | turn flow, seeded RNG | Платформа (§4.4, §4.1) |
| 2 | Бросок 2d6, ход по кругу из 40 клеток, +200 за круг | `random.roll`, `metric.set` c `%`, `when`-эффект «прошёл СТАРТ» | Платформа (§4.1, §4.5, §4.6) |
| 3 | Дубль — повторный ход; три дубля подряд — в тюрьму | `turn.repeat`, счётчик дублей в метрике игрока, `branch` | Платформа + манифест |
| 4 | 40 клеток разных типов (собственность, налог, шанс, казна, тюрьма…) | клетки = объекты с атрибутами; диспетчеризация приземления через `branch` по `cellType` | Манифест поверх §4.6 |
| 5 | Покупка собственности; отказ → **аукцион** | владение = атрибут объекта; аукцион = игровые действия bid/pass | Манифест (§5) |
| 6 | Рента: базовая, ×2 за полный цвет, по числу вокзалов, ×4/×10 от броска для коммунальных | JsonLogic-вычисление в `metric.transfer` | Платформа (§4.5) + манифест |
| 7 | Дома/отели: равномерная застройка, лимит банка 32/12 | объекты-ресурсы банка + guards `collectionCount`/JsonLogic | Манифест |
| 8 | Залог/выкуп собственности (50% / +10%) | фасет объекта `mortgaged`, `metric.transfer` | Манифест |
| 9 | Колоды «Шанс» и «Общественная казна», карта «Освобождение из тюрьмы» удерживается | `deck.shuffle`/`deck.draw`, владение картой | Платформа (§4.2) |
| 10 | Тюрьма: 3 пути выхода, максимум 3 хода | фазы хода + метрики игрока + `branch` | Манифест |
| 11 | Торги между игроками (деньги, собственность, карты) | объект-предложение + действия propose/accept/decline | Манифест (§5) |
| 12 | Банкротство: активы кредитору или банку, выбывание | `player.status.set`, `metric.transfer`, bulk-передача атрибутов владения | Платформа (§4.6) + манифест |
| 13 | Победа: последний оставшийся | `endConditions` | Платформа (§4.6) |

Вывод: платформенных возможностей нужно шесть (ADR-058 §2.1–2.6); всё остальное выражается контентом манифеста поверх них.

## 4. Дизайн возможностей

### 4.0. Нормативный справочник новых конструкций

Этот раздел — **нормативный**: имена полей, значения и ограничения ниже являются контрактом. Разделы 4.1–4.6 дают пояснения и примеры; при расхождении приоритет у справочника. Все JSON-примеры в документе валидны и самодостаточны — в них нет сокращений, которые нужно «додумывать».

**Куда что кладётся при реализации:**

| Артефакт | Место |
|---|---|
| Схемные конструкции | `docs/architecture/schemas/game-manifest.schema.json` (definitions по образцу существующих `GameManifestDeterministicEffect*`) |
| Перегенерация TS-контрактов | `npm run generate:contracts`, проверка — `npm run verify:contracts-schema-parity` |
| Обработчики эффектов/guards | `services/runtime-api/src/modules/runtime/deterministicHandlers.ts` (или новые файлы того же модуля) |
| Модуль PRNG | новый файл в `services/runtime-api/src/modules/runtime/` |
| Персональная проекция | player-facing content builder (`services/runtime-api/src/modules/content/`) |
| Фикстурная игра | `games/dice-track/` |

**Контекст данных JsonLogic (единый для guards, `when`, значений эффектов и `endConditions`):**

```json
{
  "public":       { "означает": "state.public целиком" },
  "players":      { "означает": "state.players целиком (все ветки)" },
  "actor":        { "означает": "ветка players[actorId] целиком" },
  "activePlayer": { "означает": "ветка players[activePlayerId] целиком" },
  "actorId":        "id игрока, совершающего действие",
  "activePlayerId": "id игрока, чей сейчас ход",
  "derived": { "activePlayersCount": 3 }
}
```

`actor` и `activePlayer` — материализованные копии соответствующих веток `players.*`: они избавляют выражения от динамической адресации по id (`{ "var": "actor.metrics.position" }` вместо недостижимого `players.<динамический id>`). `derived.*` вычисляет runtime перед каждой оценкой: `activePlayersCount` — число участников со статусом `active`. `state.secret` в контекст **никогда не входит**. Примеры обращения: `{ "var": "players.p1.metrics.cash" }`, `{ "var": "public.turn.phase" }`, `{ "var": "actor.metrics.cash" }`.

**Тип `playerRef`** (везде, где в таблицах указан этот тип, допустимы ровно четыре формы; всё иное — ошибка валидации):

1. литеральный id участника (например, `"p2"`);
2. `"{{actor}}"` — игрок, совершающий действие;
3. `"{{activePlayer}}"` — игрок, чей сейчас ход;
4. `{ "fromPath": "<JSON Pointer, начинается с /public/>" }` — runtime читает строку по указателю в состоянии; результат обязан быть id существующего участника, иначе действие отклоняется целиком. Это единственный механизм «динамического владельца» (например, оплата ренты владельцу клетки); отдельной подстановки вида `{{ownerOf:…}}` НЕ существует.

**Общая семантика исполнения эффектов (нормативная):**

- Эффекты действия применяются строго по порядку массива `effects`.
- `when` каждого эффекта вычисляется непосредственно перед его применением — над состоянием, уже изменённым предыдущими эффектами того же действия. Порядок эффектов внутри действия значим (см. паттерн «прошёл СТАРТ» в §5).
- Любое отклонение (guard, недостаток средств, ошибка резолвинга playerRef, ошибка выражения) отменяет **все** эффекты действия атомарно — частично применённых действий не бывает.
- Соглашение об id игроков: runtime присваивает `p1`…`pN` в порядке занятия мест; начальный `turn.order` — тот же порядок. Классическое «первым ходит выбросивший больше всех» — игровой setup-паттерн поверх `turn.setActive` (§5).
- В хотсите запрос действия несёт явное поле `actorPlayerId`; в сетевом режиме (ADR-059) это поле игнорируется, а actor берётся из аутентифицированного участника.

**Реестр новых эффектов** (`op` → поля; все эффекты дополнительно могут иметь `when`: JsonLogic-условие применения):

| `op` | Поля | Семантика | Ошибки/ограничения |
|---|---|---|---|
| `random.roll` | `dice`: строка `NdM` (N: 1–99, M: 2–1000, regex `^[1-9][0-9]?d[1-9][0-9]{0,3}$`); `storePath`: JSON Pointer, обязан начинаться с `/public/` | N бросков M-гранной кости через PRNG сессии; в `storePath` пишется объект `{values, total, isDouble}`; `counter` в `state.secret.random` увеличивается на N | Указатель вне `/public/` — ошибка схемы; сам эффект ничего не решает — решения принимают последующие `when`/guards |
| `deck.shuffle` | `deckId`: строка; `source`: строка `collection:<имя коллекции>` | Если колода не существует — создаётся из ВСЕХ id объектов коллекции; если существует — `order` и `discard` объединяются и перемешиваются, `discard` очищается | Перестановка — через PRNG сессии (Fisher–Yates); несуществующая коллекция — отклонение действия |
| `deck.draw` | `deckId`; `storePath` (JSON Pointer в `/public/`); `onEmpty`: `"reshuffle-discard"` \| `"fail"` | Снимает первый id из `order`, пишет его строкой в `storePath`, добавляет в `discard` | Пустые `order` и `discard` при `reshuffle-discard`, либо пустой `order` при `fail` — отклонение действия |
| `deck.extract` | `deckId`; `cardId`: строка \| `{"fromPath": …}` | Удаляет id из `order` И из `discard` — карта покидает ротацию колоды (игрок удерживает карту, например «Освобождение из тюрьмы») | Отсутствие id в колоде — no-op (не ошибка): операция идемпотентна |
| `deck.return` | `deckId`; `cardId`: строка \| `{"fromPath": …}` | Добавляет id в конец `discard` (карта возвращается в ротацию и вернётся в `order` при перетасовке) | Если id уже есть в `order` или `discard` — отклонение действия (защита от дублирования карт) |
| `metric.set` | `scope`: `"session"` \| `"player"`; `playerId` (playerRef, только при `scope:"player"`); `metricId`; `value`: число \| `{"jsonLogic": <выражение>}` | Присваивает значение метрике; выражение исполняет runtime полной `json-logic-js` над контекстом выше | Нечисловой результат выражения — отклонение действия |
| `metric.transfer` | `from`, `to`: объект `{scope: "player"\|"bank", playerId?}` (playerRef при `player`); `metricId`; `amount`: число \| `{"jsonLogic": …}`; `onInsufficient`: `"fail"` | Уменьшает метрику у `from`, увеличивает у `to`; сторона `bank` баланса не имеет и не проверяется | `amount < 0` или нечисловой — отклонение; у `from:player` меньше `amount` — отклонение действия целиком (единственная политика MVP) |
| `turn.next` | нет полей | Передаёт ход следующему участнику со статусом `active` по кругу `order`; `phase` сбрасывается в первую из `config.turnModel.phases`; `turnNumber` +1 при полном круге | Если `active` остался один — ход остаётся у него (завершение решают `endConditions`) |
| `turn.repeat` | нет полей | Активный игрок сохраняется; `phase` сбрасывается в первую | — |
| `turn.phase.set` | `phase`: строка | Устанавливает `public.turn.phase` | Значение не из `config.turnModel.phases` — ошибка валидации манифеста (схема этого проверить не может — проверяет загрузчик манифеста) |
| `turn.setActive` | `playerId` (playerRef) | Делает указанного игрока активным без изменения `order`; `phase` сбрасывается в первую | Игрок не `active` — отклонение действия. Использование: «первым ходит выбросивший больше всех», передача хода вне очереди |
| `player.status.set` | `playerId` (playerRef); `status`: `"active"` \| `"eliminated"` | Меняет статус участника; `eliminated` пропускается `turn.next`, его действия отклоняются платформенно | — |
| `branch` | `branches`: массив `{when: <JsonLogic>, effects: [<эффект>]}` | Вычисляет `when` по порядку; применяет эффекты ПЕРВОЙ истинной ветви; если истинных нет — не делает ничего (это не ошибка) | `branch` внутри `branches[].effects` запрещён схемой (ровно один уровень); циклов и goto нет |

| Эффекты `object.create`, `object.state.set`, `object.attribute.patch` и guard-форма `object` | Необязательное поле `owner`: `{"playerId": playerRef}` — обязательно для коллекций, чей `objectModels.<тип>.scope = "player"`; инстансы таких коллекций живут в `state.players.<id>.objects.<коллекция>` |

**Реестр новых конструкций манифеста:**

| Конструкция | Где | Форма |
|---|---|---|
| Шаблон игрока | `state.playersTemplate` | `{metrics: {<id>: число}, flags: {…}, objects: {…}, visibility?: {metrics?: "public"\|"private", flags?: …}}`; по умолчанию `metrics: "public"`, `flags: "private"` |

**Структуры состояния, которыми владеет только runtime** (в манифесте не описываются, руками не изменяются):

- `state.secret.random = {alg, seed, counter}` — создаётся при создании сессии;
- `state.secret.decks.<deckId> = {order: [id], discard: [id]}` — создаётся `deck.shuffle`;
- `state.public.turn = {order, activePlayerId, phase, turnNumber}` — создаётся при создании сессии из участников и `config.turnModel`;
- `state.players.<playerId>` — разворачивается из `playersTemplate` при создании сессии.

### 4.1. Seeded RNG и эффект random.roll

Состояние (создаётся runtime при создании сессии, в манифесте не описывается):

```json
"state": { "secret": { "random": { "alg": "xoshiro128ss-v1", "seed": "a3f1c2d94e5b60718293a4b5c6d7e8f9", "counter": 0 } } }
```

Нормативные решения (закрыто на этапе проектирования, исполнителю не выбирать):

- **Алгоритм**: `xoshiro128**` (Blackman/Vigna), идентификатор `"xoshiro128ss-v1"`. Причина выбора: состояние 4×uint32 без BigInt, реализация ~15 строк на `Math.imul`, проверенное качество. Идентификатор фиксируется в состоянии, чтобы уже созданные сессии и их replay не сломались при будущей смене алгоритма по умолчанию.
- **Зерно**: 128-битная строка из 32 hex-символов, создаётся криптографическим генератором при создании сессии; 4 слова состояния инициализируются напрямую из зерна (по 8 hex-символов на слово); если все четыре слова нулевые — состояние заменяется фиксированной ненулевой константой `[1, 2, 3, 4]`.
- **Выборка значения кости 1..M без смещения** (rejection sampling — отбрасывание значений, дающих неравномерность при взятии остатка): `limit = floor(2^32 / M) * M`; брать очередной uint32, пока он `>= limit` (каждое взятие увеличивает `counter`); результат = `uint32 % M + 1`.
- **`isDouble`**: `true` тогда и только тогда, когда бросков ровно два и значения равны; при любом другом N — всегда `false`.
- Replay-гарантия: тот же `seed` + та же последовательность действий → бит-в-бит те же результаты.

Эффект:

```json
{ "op": "random.roll", "dice": "2d6", "storePath": "/public/turn/lastRoll" }
```

Результат записи: `{ "values": [3, 5], "total": 8, "isDouble": false }`. Запись всегда дублируется платформенной записью журнала (auditable по ADR-024). Последующие эффекты и guards читают результат из состояния — сам эффект ничего не решает.

Editor preview: фикстуры состояния (ADR-057, `state-fixture.schema.json`) могут закреплять `random` ветку, что даёт детерминированные прогоны в превью редактора.

### 4.2. Колоды: deck.shuffle / deck.draw

Колода — порядок поверх коллекции объектов (карты остаются объектами ADR-041 со своими фасетами и контентом):

```json
"state": { "secret": { "decks": { "chance": { "order": ["ch-3", "ch-11", "ch-7"], "discard": [] } } } }
```

Эффекты:

```json
{ "op": "deck.shuffle", "deckId": "chance", "source": "collection:chanceCards" }
{ "op": "deck.draw",  "deckId": "chance", "storePath": "/public/turn/drawnCard", "onEmpty": "reshuffle-discard" }
{ "op": "deck.extract", "deckId": "chance", "cardId": { "fromPath": "/public/turn/drawnCard" } }
{ "op": "deck.return",  "deckId": "chance", "cardId": "ch-7" }
```

- Порядок колоды — секрет сессии; публично только уже открытое и размер колоды (проекция Presenter).
- Перетасовка — алгоритм Фишера–Йетса (Fisher–Yates: проход с конца массива с обменом со случайно выбранной позицией) на PRNG сессии — это нормативно, иначе replay между реализациями разойдётся.
- `deck.draw` перекладывает id в `discard`. Карта, удерживаемая игроком («Освобождение из тюрьмы»), изымается из ротации явным `deck.extract` (обычно в той же ветви `branch`, что распознала карту), а владение самим объектом карты выражается атрибутом `ownerPlayerId`; при использовании карта возвращается в колоду через `deck.return`. Платформа при этом ничего не знает про «тюрьму» — это игровые правила.

### 4.3. Состояние «на игрока» и адресация {{actor}}

Манифест объявляет шаблон:

```json
"state": {
  "playersTemplate": {
    "metrics": { "cash": 1500, "position": 0, "doublesStreak": 0 },
    "flags": {},
    "objects": {}
  }
}
```

Runtime разворачивает шаблон при создании сессии по числу участников: `state.players.p1 …` (структура ADR-011). `objectModels[*].scope` получает значение `player`.

Адресация в guards/effects:

- `{{actor}}` — игрок, совершивший текущее действие (в хотсите клиент передаёт его явно, в сетевой модели — из аутентифицированного `player_id` события, в hybrid — из Agent Turn);
- `{{activePlayer}}` — игрок, чей сейчас ход.

Пример: `{ "op": "metric.add", "scope": "player", "playerId": "{{actor}}", "metricId": "cash", "delta": -50 }`.

Правило проекции (Presenter): игрок получает общий мир (`public`), свою ветку `players.<self>` целиком, из чужих веток — только поля, объявленные в манифесте публичными (`playersTemplate.visibility`, по умолчанию метрики публичны, флаги приватны). В хотсите проекция строится для активного игрока; приватность между игроками за одним устройством не гарантируется по определению режима.

### 4.4. Модель хода (turn flow)

```json
"state": { "public": { "turn": {
  "order": ["p1", "p2", "p3"],
  "activePlayerId": "p1",
  "phase": "roll",
  "turnNumber": 1
} } }
```

- Манифест объявляет разрешённые фазы: `"config": { "turnModel": { "phases": ["roll", "resolve", "manage"] } }`. Полноценная машина фаз (переходы, таймеры) сознательно не вводится.
- Эффекты: `turn.next` (следующий не-выбывший по кругу, сброс фазы в первую), `turn.repeat` (активный игрок сохраняется — дубль), `turn.phase.set`, `turn.setActive` (прямое назначение активного игрока — для setup-правил вроде «первым ходит выбросивший больше всех»).
- Начальный `order` — порядок занятия мест (`p1`…`pN`); изменение порядка на лету не поддерживается, поддерживается только смена активного игрока.
- Guard-форма: `{ "turn": { "actorIsActive": true, "phase": "roll" } }` — типовая защита каждого игрового действия.

### 4.5. Экономика: metric.transfer и metric.set

Пример 1 — фиксированный налог в банк:

```json
{ "op": "metric.transfer",
  "from": { "scope": "player", "playerId": "{{actor}}" },
  "to":   { "scope": "bank" },
  "metricId": "cash",
  "amount": 100,
  "onInsufficient": "fail" }
```

Пример 2 — вычисляемая сумма динамическому получателю (рента коммунального предприятия: 4 × сумма последнего броска, владельцу клетки `cell-12`; используется внутри ветви `branch`, где клетка статически известна):

```json
{ "op": "metric.transfer",
  "from": { "scope": "player", "playerId": "{{actor}}" },
  "to":   { "scope": "player",
            "playerId": { "fromPath": "/public/objects/cells/cell-12/attributes/ownerPlayerId" } },
  "metricId": "cash",
  "amount": { "jsonLogic": { "*": [ 4, { "var": "public.turn.lastRoll.total" } ] } },
  "onInsufficient": "fail" }
```

- Динамический получатель выражается формой `{"fromPath": …}` типа `playerRef` (§4.0). Если по указателю нет строки с id существующего участника — действие отклоняется целиком.
- Стороны: `player` или `bank` (банк — неограниченный источник/сток, отдельного баланса не имеет; лимитированы только физические ресурсы-объекты вроде домов).
- `onInsufficient: "fail"` — единственная политика MVP: действие отклоняется целиком (существующая семантика отклонённого действия). Долговые сценарии (продать дома → заложить → банкротство) — это игровые действия, которые игрок совершает сам, а не автоматика платформы.
- `metric.set` — присваивание значения метрике; значение может быть числом или JsonLogic-выражением. Важная граница вычислителей: значения эффектов исполняет **runtime** полной библиотекой `json-logic-js`, где оператор `%` (остаток от деления, нужен для циклического поля) уже доступен. CI-подмножество (`validate-metric-jsonlogic-subset.js`, LEGACY-0022) ограничивает только computed-метрики, пересчитываемые в player-web, и расширяется оператором `%` лишь при реальном использовании его в `metricViews`.

### 4.6. Условные эффекты when/branch, статус игрока и endConditions

Пример (диспетчер приземления; предполагается, что предыдущий эффект записал тип клетки в `public.turn.landedCellType`):

```json
{ "op": "branch", "branches": [
  { "when": { "==": [ { "var": "public.turn.landedCellType" }, "tax" ] },
    "effects": [
      { "op": "metric.transfer",
        "from": { "scope": "player", "playerId": "{{actor}}" },
        "to": { "scope": "bank" },
        "metricId": "cash", "amount": 100, "onInsufficient": "fail" }
    ] },
  { "when": { "==": [ { "var": "public.turn.landedCellType" }, "chance" ] },
    "effects": [
      { "op": "deck.draw", "deckId": "chance",
        "storePath": "/public/turn/drawnCard", "onEmpty": "reshuffle-discard" }
    ] }
] }
```

- `when` доступно и у одиночного эффекта. Ограничения схемы: `branch` не вкладывается в `branch`; максимум один уровень; применяется первая истинная ветвь; отсутствие истинной ветви — не ошибка.
- `player.status.set`: `active` / `eliminated`; `turn.next` пропускает выбывших; действия выбывшего отклоняются платформенно.
- Завершение игры:

```json
"endConditions": [
  { "id": "last-standing",
    "when": { "==": [ { "var": "derived.activePlayersCount" }, 1 ] },
    "outcome": { "type": "last-active-player-wins" } }
]
```

`derived.activePlayersCount` — платформенная производная величина из контекста данных JsonLogic (§4.0); `outcome.type: "last-active-player-wins"` объявляет победителем единственного оставшегося `active`-участника без дополнительных подстановок.

Проверяются после каждого применённого действия; завершение фиксируется платформенной записью журнала и переводом сессии в терминальное состояние.

## 5. Механики уровня игры (не платформы)

Эти механики «Монополии» выражаются контентом манифеста поверх §4 и документируются как Gameplay Slice Records при реализации игры:

- **Аукцион** (полная классика: отказ от покупки → аукцион для всех): объект `auction` в `public` с фасетами (`open/closed`), действия `auction.bid` (guard: ставка больше текущей, игрок активен в аукционе) и `auction.pass`; завершение — `branch` по числу оставшихся участников. Аукцион приостанавливает обычный turn flow через фазу `auction`.
- **Торги (trade)**: объект-предложение `tradeOffer` (кто, кому, что отдаёт/просит — списки id объектов и суммы), действия `trade.propose` / `trade.accept` / `trade.decline`. Принятие атомарно применяет серию `metric.transfer` + `object.attribute.patch` владения.
- **Тюрьма**: метрики игрока `inJail`, `jailTurns`; три выхода — оплата, карта (владение объектом карты), дубль; лимит трёх ходов — guard + `branch`.
- **Застройка**: дома/отели — объекты банка с ограниченным количеством (32/12), правило равномерности — JsonLogic-guard по метрикам группы.
- **Банкротство**: серия действий передачи активов; финал — `player.status.set: eliminated`.

### 5.1. Типовые паттерны (нормативные примеры для авторов игр)

**Движение по циклическому полю с зарплатой за круг.** Порядок эффектов критичен: проверка «прошёл СТАРТ» должна стоять ДО обновления позиции, потому что `when` вычисляется над состоянием после предыдущих эффектов (§4.0), и после `metric.set` старая позиция уже потеряна:

```json
"effects": [
  { "op": "random.roll", "dice": "2d6", "storePath": "/public/turn/lastRoll" },
  { "op": "metric.transfer",
    "when": { ">": [ { "+": [ { "var": "actor.metrics.position" },
                              { "var": "public.turn.lastRoll.total" } ] }, 39 ] },
    "from": { "scope": "bank" },
    "to":   { "scope": "player", "playerId": "{{actor}}" },
    "metricId": "cash", "amount": 200, "onInsufficient": "fail" },
  { "op": "metric.set", "scope": "player", "playerId": "{{actor}}", "metricId": "position",
    "value": { "jsonLogic": { "%": [ { "+": [ { "var": "actor.metrics.position" },
                                              { "var": "public.turn.lastRoll.total" } ] }, 40 ] } } }
]
```

**Определение первого игрока («первым ходит выбросивший больше всех»)** — setup-фаза игры: каждый участник совершает действие «бросок на очерёдность» (результат в свою метрику), последний бросок завершается `branch`-сравнением сохранённых метрик и `turn.setActive` на победителя. При равенстве — повторный тур бросков (это игровые правила, платформа не автоматизирует).

**Удержание вытянутой карты** («Освобождение из тюрьмы»): в ветви `branch`, распознавшей карту по id из `/public/turn/drawnCard`, — `deck.extract` + `object.attribute.patch` владения; при использовании карты — обратный `object.attribute.patch` + `deck.return`.

## 6. Три модели доставки

| Модель | Что добавляется | Что переиспользуется |
|---|---|---|
| 1. Хотсит | Ничего сверх §4: одна сессия, клиент действует за активного игрока | Весь пакет §4, существующий `player-web` + плагин поля |
| 2. Сетевой мультиплеер | Реализация ADR-011 по решениям **ADR-059**: PostgreSQL-хранилище (ADR-005), модель участников с join-токенами, `session_events`/`state_version`, WebSocket-рассылка, персональные проекции; привязка к порталу ADR-033 (shared binding). | 3. ИИ-оппоненты | **ADR-060**: агент — участник сессии (`kind: "agent"`), получает ту же персональную проекцию, что человек, плюс платформенный список доступных действий, и возвращает выбранный ход через валидируемый Agent Turn (ADR-046/047). ## 7. Тестирование и воспроизводимость

По ADR-038 (policy layer):

- **Contract-тесты**: каждая новая схемная конструкция — Ajv-валидация позитивных/негативных фикстур; TS-контракты генерируются (ADR-056), дрейф — ошибка CI.
- **Unit**: обработчики эффектов в `runtime-api` (`node:test`) — включая детерминизм PRNG (фиксированный seed → фиксированная последовательность) и переполнение/опустошение колоды.
- **Фикстурная игра** `games/dice-track/` — минимальная гоночная игра на 2 игроков (бросок, движение по циклическому треку, перевод очков, выбывание, победа): доказывает generic-путь пакета до начала «Монополии», по образцу `simple-choice`/`ai-driven-choice`; входит в game-agnostic CI invariant.
- **Replay**: транскрипт сессии (последовательность действий + seed) воспроизводит идентичное конечное состояние — автоматический тест на фикстурной игре.
- **E2E (Playwright)**: хотсит-партия фикстурной игры в `player-web`.

## 8. Решённые проектные вопросы и оставшиеся решения

Все технические вопросы закрыты на этапе проектирования — исполнителю ничего не выбирать:

1. **Алгоритм PRNG — решено**: `xoshiro128**` (`alg: "xoshiro128ss-v1"`), 128-битное hex-зерно, rejection sampling для костей, Фишер–Йетс для перетасовки (§4.0, §4.1).
2. **Динамический владелец — решено**: четвёртая форма `playerRef` — `{"fromPath": "<JSON Pointer>"}` (§4.0, пример в §4.5). Отдельной подстановки `{{ownerOf:…}}` не существует и не будет.
3. **Видимость чужого состояния по умолчанию — решено**: метрики публичны, флаги приватны; переопределяется `playersTemplate.visibility` (§4.0). Для «Монополии» это совпадает с классикой (деньги игроков публичны).
4. **Удержание карты игроком — решено**: `deck.extract` / `deck.return` (§4.0, §4.2, паттерн в §5.1).
5. **Динамическая адресация в выражениях — решено**: материализованные ветки `actor`/`activePlayer` в контексте JsonLogic (§4.0) вместо динамических путей `var`.

Единственное оставшееся решение — за владельцем продукта, платформу не блокирует:

- Название и оформление игры для каталога: «Monopoly» — торговая марка Hasbro; для публикации потребуется собственное название и тексты (рабочий id — `estate-race`, рабочее название — «Магнат»). Нужно до этапа UI игры (Stage A6 задачи игры).

## 9. Указания агенту-исполнителю

Этот раздел адресован агенту, который будет реализовывать TSK-программы. Правила ниже обязательны.

**Порядок работы:**

1. Работай строго по фазам своего TSK-файла: одна фаза — один срез с собственными тестами. Не начинай следующую фазу, пока команды из раздела Validation не зелёные.
2. Перед реализацией любой конструкции сверься с §4.0 — он нормативный. Если фаза требует поля или поведения, которого нет в §4.0 и в тексте TSK, — не придумывай: остановись и зафиксируй вопрос в Handoff Log своей задачи.
3. Любое изменение `game-manifest.schema.json` сопровождай в том же срезе: перегенерацией контрактов (`npm run generate:contracts`), позитивной и негативной фикстурой схемы, обновлением обработчика в runtime и тестом обработчика.
4. После каждой фазы обновляй Handoff Log своего TSK: что сделано, какие команды прогнаны, какие решения приняты.

**Запрещено (типовые ловушки):**

- Использовать `Math.random` или `Date.now` в игровой логике — только PRNG сессии (§4.1); иначе сломается replay.
- Упоминать конкретные игры (`monopoly`, `estate-race`, `dice-track`) в коде `services/runtime-api/src` и `apps/player-web/src` — фикстурная игра подключается только как контент `games/dice-track/` (правило 10 CLAUDE.md; generic-тесты могут загружать её манифест как данные).
- Класть в контекст JsonLogic или в персональную проекцию что-либо из `state.secret` (включая `random` и `decks`).
- Реализовывать формы `playerRef` сверх четырёх нормативных (§4.0): литерал, `{{actor}}`, `{{activePlayer}}`, `{"fromPath": …}`. Подстановки `{{ownerOf:…}}` не существует.
- Выбирать другой алгоритм PRNG, способ посева, выборки или перетасовки — они нормативно зафиксированы (§4.1); менять можно только новым ADR.
- Разрешать `branch` внутри `branch` (запрещено схемой), «мягкие» политики `onInsufficient` кроме `"fail"`, автоматические долговые сценарии.
- Менять принятые ADR или расширять выразительность псевдоязыка «по ходу дела» — любое расширение только через новый ADR.

**Чек-лист завершения среза:**

```text
[ ] Схема + негативные фикстуры + generate:contracts + verify:contracts-schema-parity
[ ] Обработчик + unit-тесты (node:test) в runtime-api
[ ] Проекция не отдаёт секретов (тест при касании state.secret/players)
[ ] verify:canonical зелёный
[ ] Handoff Log обновлён
```
