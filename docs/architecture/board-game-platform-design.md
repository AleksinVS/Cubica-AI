# Дизайн платформенных возможностей пошаговых настольных игр

Документ сопровождает принятое решение о платформенной поддержке пошаговых
настольных игр и описывает её через действующий контракт **Game Intent →
Cubica Mechanics IR**. **Game Intent (игровое намерение)** — команда в терминах
игры с устойчивым `actionId`; **Mechanics IR (промежуточное представление
механик)** — типизированный JSON-план, который сервер проверяет и выполняет
атомарно. **Presenter** — слой, который превращает серверную проекцию в данные
интерфейса и маршрутизирует выбранное действие без вычисления правил.
**Principal** — подтверждённый аутентификацией пользователь или контроллер;
**actor** — участник внутри конкретной партии, которого сервер разрешил этому
principal.

Статус: нормативный дизайн, приведённый к ADR-084 и
`mechanics-plan.schema.json` 2026-07-15. Исполнительная программа —
`docs/tasks/archive/TSK-20260705-board-game-platform-capabilities.md`.

## 1. Цели и границы

Цель — поддержать класс пошаговых настольных игр в трёх моделях доставки:
хотсит, сетевой мультиплеер и участники под управлением ИИ. Первая доказательная
игра класса — экономическая игра с циклическим полем, собственностью, рентой,
торгами, колодами и выбыванием участников.

Границы:

- платформа не содержит `if` по идентификатору конкретной игры;
- JSON Schema является единственным источником истины структуры;
- клиент, Presenter и игровой ИИ выбирают только опубликованный `actionId` и
  параметры, но не внутреннюю операцию;
- правила одной игры остаются её планами, состоянием и содержимым;
- универсальные операции описывают состояние, коллекции, ресурсы, случайность,
  колоды, ход и события;
- вся изменяющая команда выполняется одной транзакцией или полностью
  откатывается;
- произвольный код допускается только как изолированное расширение одной игры,
  если общих операций объективно недостаточно.

## 2. Используемые практики

Платформа сохраняет проверенные свойства зрелых движков настольных игр, но
выражает их декларативно:

| Практика | Реализация Cubica |
|---|---|
| Воспроизводимый генератор случайности хранится вместе с состоянием | `state.secret.random`, точная версия алгоритма в `moduleLock`, именованные потоки |
| Порядок и текущий участник являются авторитетным состоянием | `state.public.turn = {order, activePlayerId, phase, turnNumber}` |
| Секреты фильтруются для каждого клиента | `audienceRef` в `stateModel` и actor-scoped проекция Presenter |
| Фазы образуют конечный набор | `config.turnModel.phases` и `turn.phase.select` |
| Игровая команда отделена от способа доставки | один `actionId` для человека, сети и игрового ИИ |
| Правила не являются серверными функциями конкретной игры | действие связывается с опубликованным типизированным планом |

Канонический порядок не зависит от порядка свойств JSON, базы данных или
локали. Массовые операции всегда работают с ограниченной выборкой и
фиксированным последним сравнением по стабильному идентификатору.

## 3. Правила полной классической игры

Термины: **рента** — плата за остановку на чужой собственности; **залог** —
временная передача собственности банку за часть стоимости; **дубль** —
одинаковые значения двух костей.

| # | Правило | Композиция действующих механизмов | Уровень |
|---|---|---|---|
| 1 | 2–6 участников, начальный порядок по броску | `random.dice.roll`, объявленное состояние setup, `core.state.patch` активного участника | Игра поверх платформы |
| 2 | Бросок 2d6, циклическое поле, выплата за полный круг | `random.dice.roll`, типизированная числовая формула, `core.state.patch`, `core.resource.transfer` | Платформа + план игры |
| 3 | Дубль повторяет ход; третий дубль отправляет в тюрьму | предикаты над объявленными roll endpoints; при повторе нет `turn.advance`, при обычном ходе он есть | План игры |
| 4 | Клетки разных типов | типизированная коллекция клеток, `core.entities.select`, взаимоисключающие `when` | План игры |
| 5 | Покупка собственности, отказ и аукцион | `core.assert`, `core.resource.transfer`, `core.entity.attributes.patch`; аукцион как объект и отдельные intents | План игры |
| 6 | Базовая и вычисляемая рента | типизированная числовая формула и `core.resource.transfer` между метриками участников | Платформа + план игры |
| 7 | Равномерная застройка, лимит домов и отелей | ограниченные collections, `core.entities.select`, `core.assert`, entity commands | План игры |
| 8 | Залог и выкуп | facet/attribute собственности, assert, resource transfer | План игры |
| 9 | Случайные колоды | `deck.shuffle`, `deck.draw`, объявленные secret decks и public/actor target | Платформа |
| 10 | Тюрьма, три способа выхода, максимум три хода | actor endpoints, predicates, resource/entity commands, phase selection | План игры |
| 11 | Торги деньгами, собственностью и картами | объект предложения; одна transaction из transfer и entity updates | План игры |
| 12 | Банкротство и выбывание | bounded select/update, resource transfer, actor status endpoint, event | Платформа + план игры |
| 13 | Победа последнего оставшегося | terminal plan после общего запроса статусов участников; для `state.players` этот нейтральный запрос остаётся реальным schema-first пробелом | Платформа + план игры |

Предметные имена «аукцион», «тюрьма» и «банкротство» не становятся названиями
платформенных операций.

## 4. Нормативный контракт

### 4.1. Game Intent и команда

Действие в опубликованном манифесте хранит только проверяемые метаданные,
закрытую схему параметров и ссылку на план:

```json
{
  "definitionHash": "sha256:aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa",
  "paramsSchema": {
    "type": "object",
    "additionalProperties": false,
    "properties": {
      "cellId": {
        "type": "string",
        "maxLength": 64,
        "x-cubica-ref": {
          "kind": "object",
          "collection": "boardCells",
          "allowedTypes": ["estate.cell"],
          "visibility": "public"
        }
      }
    },
    "required": ["cellId"]
  },
  "binding": {
    "kind": "mechanics-plan",
    "planRef": "property.buy"
  }
}
```

Хеш в примере условный; его вычисляет компилятор. Внешняя команда содержит
`sessionId`, `actionId`, `commandId`, `expectedStateVersion` и `params`.
Principal (аутентифицированный пользователь или контроллер) приходит из
защищённого канала, а actor (участник внутри партии) разрешается сервером.
Клиентский `playerId` не является доказательством личности.

Presenter показывает actor-scoped каталог доступных действий и маршрутизирует
точный `actionId`. Он не вычисляет ренту, победителя или законность хода и не
составляет Mechanics IR.

### 4.2. Корень Mechanics IR

`mechanics` содержит ровно пять обязательных частей. Сокращённый фрагмент ниже
показывает расположение полей, но пустые словари не являются готовым
schema-valid пакетом: рабочая игра обязана объявить модули, типы и хотя бы один
план.

```json
{
  "apiVersion": "cubica.dev/mechanics/v1alpha1",
  "budgetProfile": "turn-based-standard-v1",
  "moduleLock": {},
  "stateModel": {
    "types": {},
    "endpoints": {},
    "collections": {},
    "events": {}
  },
  "plans": {}
}
```

- `apiVersion` фиксирует язык;
- `budgetProfile` выбирает платформенный предел стоимости;
- `moduleLock` закрепляет точные `moduleId`, `moduleVersion`, `artifactHash` и
  версии алгоритмов;
- `stateModel` объявляет все разрешённые чтения и записи;
- `plans` хранит заранее опубликованные транзакции.

Session закрепляет неизменяемый `bundleHash`, включающий планы, действия,
схемы, compiler, budgets, modules и версии случайности. Поэтому повтор партии
не зависит от более нового пакета игры.

### 4.3. Модель состояния

`stateModel.types` поддерживает закрытые варианты boolean, string, bounded
integer, exact decimal, enum, record, list, set, option и map.

`endpoints` дают устойчивые имена значениям состояния. Конечная точка
объявляет:

- `audienceRef`: `public`, `actor` или `server`;
- точное `storage` с корнем и сегментами;
- `valueType`;
- доступ `read-only` или `read-write`.

Динамический сегмент хранения может ссылаться только на серверный actor или на
параметр, уже ограниченный схемой действия. Клиент не передаёт JSON Pointer или
JSONPath.

`collections` объявляют аудиторию, storage, capacity, стабильный ключ,
допустимые типы объектов и типизированные facet/attribute fields. Любая
выборка имеет `cardinality.min/max`; превышение откатывает команду.

`events` объявляют аудиторию и payload type. Необязательный `journalEndpoint`
создаёт удобную проекцию события в состоянии, а полная защищённая запись
сохраняется в долговечном event ledger с монотонной последовательностью.

### 4.4. Выражения и предикаты

Выражения и предикаты — закрытое JSON AST, то есть дерево операций с
проверяемыми узлами, а не строка программы.

Актор:

```json
{ "op": "value.actor" }
```

Чтение объявленного значения:

```json
{
  "op": "value.state",
  "ref": { "endpoint": "public.turn.lastRoll.total" }
}
```

Числовая формула ренты:

```json
{
  "op": "number.multiply",
  "items": [
    { "op": "value.literal", "value": 4 },
    {
      "op": "value.state",
      "ref": { "endpoint": "public.turn.lastRoll.total" }
    }
  ]
}
```

Предусловие активного участника и фазы:

```json
{
  "op": "predicate.all",
  "items": [
    { "op": "predicate.actor.active" },
    {
      "op": "predicate.turn.phase",
      "phase": { "op": "value.literal", "value": "manage" }
    }
  ]
}
```

Словарь значений: `value.literal`, `value.param`, `value.actor`,
`value.state`, `value.result`, `value.item`, ограниченная арифметика и
`value.coalesce`. Словарь условий: константа, `all`/`any`/`not`, typed compare,
exists, active actor, turn phase, entity match и bounded collection count.

Выражение чисто: оно не меняет состояние, не читает сеть, файлы, environment
или системные часы и не расходует случайность. Ссылка на результат смотрит
только на предыдущий шаг.

### 4.5. Атомарный план

План — упорядоченный массив шагов. Пример фиксированной ренты:

```json
{
  "planHash": "sha256:bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb",
  "transaction": {
    "steps": [
      {
        "id": "precondition",
        "kind": "assert",
        "op": "core.assert",
        "predicate": {
          "op": "predicate.actor.active"
        },
        "errorCode": "ACTION_PRECONDITION_FAILED"
      },
      {
        "id": "pay-rent",
        "kind": "command",
        "op": "core.resource.transfer",
        "from": {
          "kind": "player-metric",
          "playerId": { "op": "value.actor" },
          "metricId": "cash"
        },
        "to": {
          "kind": "player-metric",
          "playerId": {
            "op": "value.state",
            "ref": { "endpoint": "public.landedProperty.ownerId" }
          },
          "metricId": "cash"
        },
        "amount": { "op": "value.literal", "value": 100 },
        "onInsufficient": "fail"
      }
    ]
  }
}
```

Хеш условный и появляется только в опубликованной форме. Авторинг хранит тот
же plan без ручного `planHash`.

Шаги выполняются по порядку над **candidate state** — пробной копией состояния.
Следующий шаг видит предыдущие изменения. Ошибка предусловия, типа, бюджета,
кардинальности или ресурса откатывает состояние, счётчики случайности, события
и версию сессии. **Квитанция команды** — неизменяемая запись принятого
результата и идентичности плана. Состояние, события и квитанция фиксируются
одним атомарным сохранением.

`when` допустим у тех шагов, где его объявляет схема, и принимает только
типизированный predicate. Несколько взаимоисключающих `when` заменяют
предметную диспетчеризацию; универсальной операции `branch` в текущем языке
нет.

## 5. Общие возможности настольной игры

### 5.1. Воспроизводимая случайность

`random.dice.roll` принимает проверенную строку `NdM`, имя потока и
типизированный `target`. Для классического броска используется `2d6`.
Результат содержит форму броска, значения и сумму; endpoint объявляет
соответствующий record type.

Каждый поток использует закреплённый PRNG (детерминированный генератор
псевдослучайных чисел) `xoshiro128ss-v1`, а корневое состояние потоков имеет
версию `xoshiro128ss-streams-v1`. Seed потока доменно отделяется от общего
seed, поэтому расход одного потока не меняет другой. Случайность хранится в
server-only состоянии и расходуется только внутри транзакции. Ошибка
последующего шага откатывает соответствующий счётчик.

Проверка дубля читает объявленные endpoints значений костей и сравнивает их
через `predicate.compare`. Скрытое состояние генератора клиенту не выдаётся.

### 5.2. Колоды

`deck.shuffle` принимает `deckId`, исходную объявленную collection и
именованный random stream. `deck.draw` принимает `deckId`, typed target и
политику `reshuffle-discard` либо `fail`.

Порядок и discard находятся в `state.secret.decks`; клиент видит только
разрешённую карту. Перетасовка использует закреплённый
`fisher-yates-xoshiro128ss-streams-v1`.

Текущий публичный словарь не содержит отдельной операции извлечения карты из
ротации и возврата удерживаемой карты. Правило «Освобождение из тюрьмы» нельзя
объявлять уже решённым через несуществующий `deck.extract`. До реализации этого
среза требуется schema-first обобщение управления членством карты в колоде с
нейтральной фикстурой и сохранением запрета дублей.

### 5.3. Состояние участника

Ветка участника хранится в `state.players.<id>`. Actor-scoped endpoint задаёт
storage с корнем `players` и сегментом `{ "context": "actor" }`. Поэтому
действие читает и меняет состояние текущего участника без строковой
подстановки ID.

Для динамического получателя платежа используется `player-metric` endpoint,
где `playerId` является typed value expression. Например, owner ID читается
через заранее объявленный public endpoint; он не становится произвольным
путём клиента.

Публичность каждого endpoint и collection задаётся `audienceRef`. Presenter
не копирует server-only данные в availability и не публикует секреты других
участников.

### 5.4. Ход и фазы

Авторитетное состояние:

```json
{
  "order": ["p1", "p2", "p3"],
  "activePlayerId": "p1",
  "phase": "roll",
  "turnNumber": 1
}
```

- `turn.advance` выбирает следующего невыбывшего участника, увеличивает номер
  и возвращает первую объявленную фазу;
- `turn.phase.select` устанавливает фазу из `config.turnModel.phases`;
- повтор хода выражается отсутствием `turn.advance` и явным выбором начальной
  фазы;
- setup может записать победителя начального броска в объявленный endpoint
  `activePlayerId` через `core.state.patch`;
- сервер проверяет `predicate.actor.active` и роль, даже если UI считает
  действие доступным.

Порядок участников не меняется скрыто. Выбывшие пропускаются. Если правило
игры требует другого порядка или одновременных участников, оно должно получить
отдельный реальный срез, а не менять семантику `turn.advance` неявно.

### 5.5. Экономика

`core.resource.transfer` поддерживает три стороны:

- `{ "kind": "bank" }` — неограниченный источник или сток;
- `{ "kind": "state", "target": { "endpoint": "..." } }` — объявленный
  числовой endpoint;
- `{ "kind": "player-metric", "playerId": <value>, "metricId": "cash" }`
  — метрика участника.

`amount` является typed value expression: literal, state/result/item reference
или ограниченная арифметика. Единственная политика недостатка средств —
`onInsufficient: "fail"`; она откатывает весь план. Продажа активов, залог,
займ и банкротство — явные игровые действия, а не скрытая автоматика transfer.

Присваивание или увеличение объявленного числового значения выполняют
`core.state.patch` и `core.number.add`. Для participant metric предпочтителен
resource transfer; прямое обновление другого участника не строится из
клиентского пути.

### 5.6. Условия и завершение

Условие действия выражается `core.assert` и typed predicates. Необязательный
`when` шага проверяется над текущим candidate state непосредственно перед
шагом. Поэтому порядок значим, но все изменения всё равно атомарны.

Явное завершение ведущим уже выражается assert текущей фазы, записью terminal
endpoint и `core.event.emit`. Автоматическое правило «последний оставшийся»
пока не следует объявлять полностью выраженным. Действующий `turn.advance`
читает статусы из `state.players`, `core.entities.select` не объявляет сам
корень `state.players` коллекцией, а `value.result` не проецирует ID
единственного участника из entity selection. Нельзя создавать вторую публичную
копию статусов только ради подсчёта.

Перед реализацией нужен нейтральный state-model-aware запрос участников turn
module либо эквивалентное schema-first решение с одним источником истины,
bounded result и scalar projection. Затем terminal plan записывает outcome,
выпускает `core.event.emit` и закрывает каталог изменяющих intents. Конкретный
текст победы, причина завершения и правила ничьей принадлежат игре.

## 6. Механики уровня игры

Следующие механики сохраняют предметную семантику, но компонуются из общих
операций:

- **Аукцион:** entity `auction`, participant collection, отдельные intents
  `bid` и `pass`; каждый bid проверяет фазу, участника и сумму, затем меняет
  объект и ресурс одной транзакцией.
- **Торги:** entity предложения хранит стороны, объекты и суммы. `accept`
  повторно проверяет владение и выполняет transfers и attribute updates одним
  планом.
- **Тюрьма:** actor endpoints `inJail`/`jailTurns`, phase predicates и три
  отдельных intents выхода. Лимит проверяется `core.assert`.
- **Застройка:** дома и отели — ограниченная collection банка; равномерность
  проверяется выборками и числовыми predicates до mass update.
- **Залог:** facet собственности и resource transfer находятся в одном плане;
  заложенная собственность исключается из начисления ренты предикатом.
- **Банкротство:** план выбирает фактически принадлежащие объекты, передаёт их
  кредитору или банку, меняет actor status и выпускает событие. Partial success
  запрещён.
- **Циклическое движение:** бросок записывается первым; arithmetic AST считает
  новую позицию и факт полного круга; условный transfer зарплаты и запись
  позиции происходят в объявленном порядке над candidate state.

Предметная логика остаётся в `plans`, `stateModel` и UI-bindings пакета игры.
Общий runtime не знает названий улиц, тюрьмы или аукциона.

## 7. Три модели доставки

| Модель | Что меняется | Что остаётся общим |
|---|---|---|
| Хотсит | один principal может последовательно управлять разрешёнными actors | action catalog, Mechanics IR, server authorization, command ledger |
| Сетевой мультиплеер | principal связан с конкретным участником; обновления доставляются персонально | тот же `actionId`, plan, projection и receipt |
| Игровой ИИ | Agent Runtime выбирает один actor-scoped опубликованный Game Intent | тот же params validation, authorization, executor и transaction |

Игровой манифест один для всех моделей. ИИ не возвращает patch состояния или
внутренние operations; сеть не доверяет actor ID из тела; хотсит retry остаётся
безопасным после смены активного участника, потому что принятая квитанция хранит
разрешённого actor.

## 8. Проверки и воспроизводимость

- JSON Schema 2020-12 проверяет каждый вариант Mechanics IR; generated types
  являются производными.
- Semantic checker проверяет module locks, типы, ссылки, аудитории,
  read/write-права, порядок result refs и статическую стоимость.
- Нейтральная фикстура без предметных имён игры доказывает новую общую
  возможность до подключения игры.
- Runtime budget ограничивает операции, сканирование, результаты, записи,
  события и память.
- Фиксированные bundle, seed, params и command transcript дают одинаковый
  полный серверный отпечаток состояния.
- Точный повтор с тем же `commandId` и canonical fingerprint (каноническим
  отпечатком содержания команды) возвращает прежнюю квитанцию,
  не повторяя plan, RNG и события.
- PostgreSQL round-trip сохраняет state, random state, receipt и event sequence.
- Player projection проверяется на отсутствие server-only данных.

## 9. Решённые вопросы и известные пробелы

Решено:

1. Именованные потоки — `xoshiro128ss-streams-v1`, PRNG потока —
   `xoshiro128ss-v1`, shuffle —
   `fisher-yates-xoshiro128ss-streams-v1`; версии закреплены в bundle.
2. Actor разрешается сервером; строки-подстановки участника не нужны.
3. Динамический владелец читается только через объявленный typed endpoint.
4. Деньги переводятся общей `core.resource.transfer` без отрицательной суммы и
   скрытого кредита.
5. Условия используют typed predicates; предметного общего `branch` нет.
6. Сессия закрепляет immutable bundle, а команда — definition/plan hashes и
   durable receipt.

Известные продуктовые или будущие срезы:

- удерживаемая карта требует общей schema-first операции членства карты в
  колоде; текущие `deck.shuffle`/`deck.draw` этого не обещают;
- победа последнего участника требует общего typed query над авторитетными
  статусами `state.players`; дублировать их в другой collection запрещено;
- полная публичная игра не должна использовать торговое название и тексты без
  соответствующих прав; рабочий пакет `estate-race` остаётся нейтральным;
- realtime-мультиплеер меняет доставку, но не игровой язык;
- новые формы порядка хода вводятся только подтверждённой игрой и нейтральной
  фикстурой.

## 10. Указания исполнителю

1. Начинать с одного вертикального игрового сценария и отделять готовые
   платформенные операции от содержимого игры и реального общего пробела.
2. Менять структуру только через JSON Schema; после этого пересоздавать
   generated contracts.
3. Использовать точные `op` из §4.6. Если нужной операции нет, не придумывать её
   в манифесте или документации.
4. Любая новая общая операция получает нейтральное имя, schema, module lock,
   checker, executor, cost model и нейтральные позитивные/негативные фикстуры.
5. Не использовать `Math.random`, `Date.now`, сеть или файловую систему в
   игровой логике.
6. Не добавлять ID или термины конкретной игры в общий runtime и Presenter.
7. Проверять весь план на rollback, воспроизводимость, отсутствие секретов и
   exact retry.
8. Не редактировать опубликованный manifest вручную: источником является
   authoring, а compiler вычисляет хеши и source map.

Чек-лист среза:

```text
[ ] JSON Schema и generated contracts синхронизированы
[ ] Semantic checker и runtime executor принимают одинаковый module lock
[ ] Нейтральные позитивные и негативные фикстуры добавлены
[ ] Candidate state полностью откатывается при ошибке
[ ] Проекция не отдаёт server-only данные
[ ] Bundle/definition/plan hashes и receipt проверены
[ ] Handoff Log обновлён свежими командами проверки
```
