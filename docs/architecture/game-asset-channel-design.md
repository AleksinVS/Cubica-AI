## 1. Цели и границы

Цели: игровые статические изображения живут в бандле игры
(`games/<id>/assets/`), описываются декларативным реестром, раздаются
runtime-api контент-адресуемо и доступны Phaser-сцене (по id через резолвер)
и UI-манифесту (форма `asset:<id>`). Платформа не знает конкретных игр.

Вне рамок (не реализовывать): аудио и другие `kind` кроме `image`;
загрузка ассетов пользователями/редактором; внешние CDN; миграция картинок
«Антарктиды» (отдельная задача, долг LEGACY-0023); генерация/оптимизация
изображений.

## 2. Обзор решения

```text
games/<gameId>/assets/assets.json  ──валидирует──▶  scripts/ci/validate-game-assets.js (verify:canonical)
        │
        ▼ читает
runtime-api (modules/content):  GET /game-assets/<gameId>/index.json      (id → контент-адресуемый URL)
                                GET /game-assets/<gameId>/<id>/<sha256>.<ext>  (файл, immutable-кэш)
        │
        ▼ запрашивает один раз на игру
player-web:  GameAssetResolver ──▶ PhaserSceneContext.assets (сцена плагина)
             форма "asset:<id>" в image-props UI-манифеста ──▶ URL (рендерер, game-agnostic)
```

## 3. Нормативный справочник

**Куда что кладётся:**

| Артефакт | Место |
|---|---|
| Схема реестра | `docs/architecture/schemas/game-assets.schema.json` |
| Перегенерация TS-контрактов | `npm run generate:contracts`, проверка — `npm run verify:contracts-schema-parity` |
| Реестр и файлы ассетов игры | `games/<gameId>/assets/assets.json`, файлы — в том же каталоге (подкаталоги допустимы) |
| CI-валидатор | `scripts/ci/validate-game-assets.js`, подключается в `verify:canonical` (образец — `scripts/ci/validate-metric-jsonlogic-subset.js`) |
| Чтение реестра/файлов, хэши | `services/runtime-api/src/modules/content/` (по образцу published-бандлов в `localFileRepository.ts`) |
| HTTP-маршруты | `services/runtime-api/src/modules/player-api/httpServer.ts` (рядом с `/published-plugin-bundles/...`) |
| Резолвер и типы | `apps/player-web` (`plugin-api` — публичный тип `GameAssetResolver`) |
| Резолюция `asset:` в рендерере | generic-рендерер `apps/player-web` (декларативно, ADR-055) |

### 3.1. Реестр `assets.json` (схема — нормативна)

```json
{
  "gameId": "rail-tycoon-mini",
  "assets": [
    { "id": "train-coal", "file": "train-coal.svg", "kind": "image",
      "origin": { "type": "authored-in-repo" } },
    { "id": "ground-tile", "file": "sprites/ground-tile.png", "kind": "image",
      "origin": { "type": "third-party", "license": "CC0-1.0",
                  "source": "https://kenney.nl/assets/..." } }
  ]
}
```

Ограничения схемы (все — декларативно в `game-assets.schema.json`,
`additionalProperties: false` на всех уровнях):

- `gameId`: строка, обязана совпадать с id каталога игры (совпадение
  проверяет валидатор, схема этого не может);
- `assets`: массив 1..64 элементов;
- `id`: `^[a-z0-9][a-z0-9-]{0,63}$`, уникален в реестре (уникальность —
  валидатор);
- `file`: `^[a-z0-9][a-z0-9/_.-]{0,127}\.(png|jpg|webp|svg)$`; валидатор
  дополнительно отклоняет подстроку `..` и ведущий `/`;
- `kind`: enum, в MVP ровно `["image"]`;
- `origin`: либо `{ "type": "authored-in-repo" }`, либо
  `{ "type": "third-party", "license": <строка>, "source": <строка-URL> }`
  (оба поля обязательны при `third-party`).

### 3.2. HTTP-эндпоинты (runtime-api)

`assertGameId`/`assertAssetId` — по образцу `assertPluginId`
(отклонение не-соответствия регэкспам выше до любого обращения к файлам).

**Индекс:** `GET /game-assets/{gameId}/index.json`

```json
{
  "gameId": "rail-tycoon-mini",
  "assets": {
    "train-coal": { "url": "/game-assets/rail-tycoon-mini/train-coal/3f2a…c9.svg", "kind": "image" }
  }
}
```

- 200 с заголовком `Cache-Control: no-cache` (клиент ревалидирует индекс,
  файлы кэширует навсегда);
- игра без каталога `assets/` или без `assets.json` → 404 (наличие ассетов
  для игры необязательно);
- невалидный реестр → 500 с диагностикой в лог (CI такое не пропускает).

**Файл:** `GET /game-assets/{gameId}/{assetId}/{sha256hex}.{ext}`
(`sha256hex` — 64 hex-символа в нижнем регистре, как у published-бандлов)

- Отдаётся только если: `assetId` есть в реестре, `ext` совпадает с
  расширением `file` записи, `sha256hex` совпадает с SHA-256 текущего
  содержимого файла. Любое несовпадение → 404 (не 400 — не раскрывать
  причину).
- Заголовки успеха: `Content-Type` по фиксированной карте расширений
  (`png → image/png`, `jpg → image/jpeg`, `webp → image/webp`,
  `svg → image/svg+xml`); `Cache-Control: public, max-age=31536000,
  immutable`; `X-Content-Type-Options: nosniff`; для svg дополнительно
  `Content-Security-Policy: default-src 'none'; style-src 'unsafe-inline'`.
- Хэши вычисляются сервером и кэшируются в памяти с инвалидацией по mtime
  файла (повторное хэширование на каждый запрос запрещено).
- Файл читается строго по пути `games/<gameId>/assets/<file из реестра>`
  с проверкой, что resolved-путь остаётся внутри каталога ассетов игры
  (по образцу проверки published-бандлов).

### 3.3. Контракт резолвера (player-web, `plugin-api`)

```ts
/** Доступ к ассетам игры по id. Строится из /game-assets/<gameId>/index.json,
 * который платформа запрашивает один раз на игру (повторные обращения — из памяти). */
export interface GameAssetResolver {
  /** URL контент-адресуемого файла ассета. Неизвестный id — выброс Error
   * (fail fast: ошибка контента ловится на первом прогоне, не маскируется). */
  url(assetId: string): string;
  /** Все id реестра (для предзагрузки циклом). */
  ids(): ReadonlyArray<string>;
}
```

- `PhaserSceneContext` (ADR-062, владелец —
  `flow-simulation-platform-design.md` §4.0) расширяется **аддитивным**
  обязательным полем `assets: GameAssetResolver`. Правка §4.0 выполняется
  в том же срезе, что и код, по согласованию с треком flow-simulation
  (запись в Handoff Log обоих треков ДО правки).
- Игра без ассетов получает резолвер с пустым реестром (`ids()` → `[]`,
  `url()` — Error). Ошибка сети при загрузке индекса → тот же пустой
  резолвер + стандартная диагностика player-web (fail closed, приложение
  не падает).

### 3.4. Форма `asset:` в UI-манифесте

- В документированных image-свойствах компонентов допускается строка
  `asset:<assetId>` (регэксп значения: `^asset:[a-z0-9][a-z0-9-]{0,63}$`).
  По состоянию на TSK-20260719 R4b резолюция обобщена на все image-свойства
  UI-манифеста, а не только на первое из них:
  - `screenComponent.props.backgroundImage`;
  - `gameVariableComponent.props.backgroundImage`;
  - `imageComponent.props.src`.
  Один резолвер (`resolveGameAssetReference`, `apps/player-web/src/lib/game-asset-resolver.ts`)
  вызывается из всех трёх мест — платформа не дублирует логику резолюции
  по компонентам. Значение `GameConfigData.themeBackgroundImage`
  (config-уровня, не UI-манифест) поддерживает ту же форму: резолвится
  через `resolveThemeBackgroundStyle` в тот же индекс ассетов игры.
- Для `imageComponent.props.src` порядок обработки фиксирован: СНАЧАЛА
  подстановка `{{...}}`-шаблона (значения из gameState/localContext), ЗАТЕМ,
  если получившаяся строка начинается с `asset:` — резолюция. Это позволяет
  автору манифеста писать шаблонизированную ссылку на ассет, например
  `asset:info-{{currentInfo.id}}`, и получать конкретный id после
  подстановки (`asset:info-i5`).
- Рендерер перед применением резолвит форму через индекс ассетов текущей
  игры. Неизвестный id → свойство не применяется (компонент без
  изображения, без `url(undefined)`) + предупреждение в консоль в
  dev-режиме (fail closed). Рендерер не знает, какая игра его использует
  (ADR-055).
- Обычные URL-строки продолжают работать без изменений (обратная
  совместимость с играми, которые ещё не мигрировали на канал ассетов).

### 3.5. CI-валидатор `validate-game-assets.js`

В канал включаются каталоги `games/*/assets/`, в которых есть
`assets.json`. Каталог без реестра временно считается незарегистрированным
наследием и пропускается валидатором: это ограниченное сосуществование с
долгом LEGACY-0023, а не допустимый способ добавления новых ассетов. Любая
игра, начинающая использовать канал, обязана добавить реестр; после миграции
наследия это исключение удаляется.

Для каждого включённого каталога:

1. `assets.json` валиден по схеме (Ajv, строгий режим).
2. `gameId` реестра == имя каталога игры.
3. `id` уникальны; `file` не содержит `..` и не начинается с `/`.
4. Каждый `file` существует; размер ≤ 512 КБ; суммарно ≤ 4 МБ на игру.
5. Файлы-сироты (в `assets/`, кроме `assets.json` и `.desc.json`, но не в
   реестре) → ошибка.
6. Санитизация каждого `.svg` (регистронезависимо, по содержимому):
   - обязан начинаться с `<svg` (после опциональных XML-декларации,
     комментариев и пробелов) и содержать атрибут `viewBox`;
   - запрещены подстроки: `<script`, `javascript:`, `<foreignobject`,
     `<image`, `<iframe`, `<embed`, `<object`;
   - запрещены атрибуты-обработчики: регэксп `\son[a-z]+\s*=`;
   - `href=` и `xlink:href=` допустимы только со значением, начинающимся
     с `#` (внутренние ссылки).
   Любое срабатывание — ошибка с именем файла и правилом.

Любая ошибка — ненулевой код выхода (валидатор блокирует
`verify:canonical`).

## 4. Безопасность

1. **Path traversal исключён тремя рубежами**: регэкспы схемы на `file`,
   проверка `..`/ведущего `/` в валидаторе, resolved-путь внутри каталога
   ассетов на сервере. Клиентские id проходят `assertAssetId` до любого IO.
2. **SVG — контролируемый репозиторный контент, не пользовательские
   загрузки**: файлы коммитятся через ревью, CI-санитизация (§3.5) —
   первый рубеж; заголовки раздачи (`nosniff`, CSP `default-src 'none'`) —
   второй; контексты потребления (`<img>`/CSS background/Phaser loader) не
   исполняют скрипты — третий.
3. **Никакой интерпретации клиентских данных**: сервер сравнивает
   присланные `gameId`/`assetId`/хэш/расширение с реестром и файлом;
   несовпадение — единообразный 404.
4. **DoS-поверхность**: лимиты §3.5, кэш хэшей по mtime, immutable-кэш на
   клиенте/прокси.

## 5. Тестирование

- **Contract**: позитивные и негативные фикстуры `game-assets.schema.json`
  (минимум негативных: лишнее поле, плохой `id`, плохое расширение,
  `third-party` без `license`); перегенерация контрактов; parity зелёный.
- **Unit runtime-api (node:test)**: индекс строится и стабилен; файл
  отдаётся при верном хэше; 404 — неизвестная игра/пустой реестр,
  неизвестный id, несовпадение хэша, несовпадение расширения, невалидные
  `gameId`/`assetId`; заголовки успеха (все из §3.2, включая CSP для svg);
  path traversal (file с `..` в подставном реестре) отклоняется.
- **Unit валидатора**: по фикстуре на каждое правило §3.5 (в т.ч. каждый
  запрет санитизации) — красная; эталонный валидный каталог — зелёный.
- **Unit player-web (Vitest)**: резолвер (`url` по индексу, Error на
  неизвестный id, `ids()`); резолюция `asset:` в рендерере; неизвестный id
  в UI — без изображения, предупреждение; ошибка индекса — пустой резолвер.
- **E2E**: через первую игру-потребителя (`rail-tycoon-mini`, её TSK);
  платформенный трек e2e не добавляет.
- **Grep-инвариант**: в `apps/player-web/src` и `services/runtime-api/src`
  не появляется id конкретных игр или ассетов (существующий контур
  `verify:game-agnostic`).

## 6. Решённые проектные вопросы

1. **Реестр, а не «раздача каталога»** — декларативный контракт (правило 12),
   происхождение/лицензии, лимиты, защита от сирот.
2. **Контент-адресуемые URL** — зеркалирование проверенного механизма
   published-бандлов: иммутабельный кэш и автоматическая инвалидация.
3. **Хэш вычисляет сервер**, в реестре его нет — реестр пишет агент, хэши в
   реестре гарантированно протухали бы; кэш по mtime снимает расход.
4. **Обращение по id** (`url(assetId)`, `asset:<id>`) — пути никогда не
   покидают сервер; UI-манифест и сцена не знают раскладку файлов.
5. **SVG разрешён и является основным форматом агентных игр** — единственный
   формат изображений, который агент пишет как текст; риски закрыты §4 п.2.
6. **`kind` только `image`** — аудио/шрифты/атласы потребуют решений о
   формате и лимитах; расширение enum — отдельным решением при первом
   реальном потребителе.
7. **Индекс — отдельный endpoint, а не часть player-facing content
   projection** — ассеты нужны и до/вне сессии (интро-экран), проекция
   контента не раздувается; клиент кэширует индекс на игру.
8. **Отсутствие `assets/` у игры — норма** (404 индекса, пустой резолвер):
   существующие игры не обязаны меняться.
9. **Fallback-примитивов при недоступном ассете нет**: CI гарантирует
   целостность реестра; маскирование ошибок загрузки усложнило бы сцены и
   прятало дефекты.

## 7. Указания агенту-исполнителю

1. Нет формы/имени/значения в §3 — не изобретать: вопрос в Handoff Log,
   стоп среза.
2. JSON Schema — SSOT: изменения схемы только вместе с
   `npm run generate:contracts`, фикстурами (позитив И негатив) и тестами в
   том же срезе (ADR-056). Никаких ручных type guard-ов вместо Ajv
   (правило 12 CLAUDE.md).
3. Не переносить и не «улучшать» существующие картинки «Антарктиды» — это
   отдельная задача (LEGACY-0023).
4. Правка `PhaserSceneContext` — только по правилу координации §3.3.
5. Никаких упоминаний конкретных игр в платформенном коде; примеры в
   тестах — на синтетических фикстурах (`test-game` и т.п.), не на
   реальных играх.
6. Новые каталоги — `.desc.json` + `node scripts/dev/generate-structure.js`;
   временные файлы — только `.tmp/`; документация синхронна с кодом.

## 8. Координация с другими треками

- **flow-simulation** (`TSK-20260706-flow-simulation-platform-capabilities`)
  владеет `plugin-api` и §4.0 своего дизайна: расширение
  `PhaserSceneContext.assets` согласуется через Handoff Log обоих треков и
  вносится синхронно с кодом; Phase 3 этого трека (инъекция в сцену)
  требует готового Phaser-хоста (их Phase 3).
- **Игры-потребители**: `rail-tycoon-mini` — первый потребитель (её дизайн
  §8.0); `conveyor-mini` остаётся на примитивах (фикстура).
- **«Антарктида»**: миграция `/images/...` → `asset:` — отдельная будущая
  задача; до неё обычные URL в UI-манифестах работают (§3.4).
- Общие файлы с другими треками: `httpServer.ts` (трек ADR-059 может
  трогать его же) — сверять `git log` и Handoff Log перед срезом.
