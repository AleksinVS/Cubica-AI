## 1. Назначение

Элементный промт - это сохраненное в authoring-манифесте описание намерения автора для конкретного экземпляра. Он отвечает на вопросы:

- что это за элемент;
- какие данные он показывает или хранит;
- какое поведение появляется при взаимодействии;
- какие изменения состояния или переходы должны произойти;
- какой методический смысл должен сохранить агент при дальнейших правках.

Это поле нужно для game authoring, UI authoring и прототипов. Для прототипов хранится шаблон, который получает новый экземпляр при создании.

## 2. Что уже есть сейчас

В проекте есть несколько близких, но неполных механизмов:

- `generation.prompt` в `docs/architecture/schemas/design-artifact.schema.json` описывает визуальную генерацию изображения или региона.
- `_semantics` в `docs/architecture/schemas/manifest-authoring-common.schema.json` дает короткое смысловое описание authoring-узла.
- `EditorPatchIntent.prompt` в `packages/editor-engine/src/index.ts` хранит пользовательский запрос на одну операцию редактирования.
- `PatchJournalStep` хранит историю конкретного изменения, включая промт операции и summary.

Ни один из этих механизмов не является сохраненным, подтвержденным и актуализируемым промтом конкретного элемента.

## 3. Найденные расхождения

1. **Визуальный промт ошибочно выглядел как универсальный.** ADR-016 говорил о поэлементном `generation.prompt`, но контекст этого поля - inpainting/refinement, compositing and image reproduction. Оно не покрывает игровое или UI-поведение, методику и state effects. ADR-048 уточняет это различие.
2. **Authoring v2 не хранил промт экземпляра.** Первый срез ADR-048 добавил `_prompt` в общий `semanticEntity`.
3. **Прототипы не хранили шаблон промта.** Первый срез ADR-048 добавил `_promptTemplate` в общий `authoringDefinition`.
4. **Редакторский prompt временный.** Текущий `EditorPatchIntent.prompt` живет в edit flow и журнале, но не становится текущим описанием элемента.
5. **Нет reverse-sync правила.** Документы не описывают, что делать, если манифест изменился вручную или агентом и сохраненный промт стал устаревшим.

## 4. Целевой контракт

Целевой authoring-контракт использует два поля:

- `_prompt` - статическая часть промта конкретного экземпляра, содержащая только невосстановимое авторское намерение;
- `_promptTemplate` - шаблон промта в прототипе.

Целевой полный промт не хранится как одно поле. Редактор собирает compiled prompt (скомпилированный промт: единый текст, который видит пользователь) из двух частей:

1. `_prompt.staticText` или external Markdown ref - статическая часть, которую нельзя надежно восстановить из структурированных полей.
2. Dynamic YAML projection - временная YAML-проекция выбранного JSON-узла и связанных узлов с русскими названиями полей.

Минимальная целевая форма `_prompt` по ADR-049:

```json
{
  "_prompt": {
    "version": 3,
    "status": "confirmed",
    "staticText": "Методический смысл и авторские ограничения, которые не выражены отдельными полями манифеста.",
    "origin": "user",
    "language": "ru",
    "updatedAt": "2026-06-13T00:00:00Z"
  }
}
```

Минимальная целевая форма `_promptTemplate`:

```json
{
  "_promptTemplate": {
    "raw": "Опишите лицевую сторону, оборотную сторону, действие при выборе, изменения метрик и методический смысл.",
    "language": "ru",
    "appliesTo": "game.ChoiceCard"
  }
}
```

Форма `_promptTemplate` реализована в первом срезе ADR-048 через `docs/architecture/schemas/manifest-authoring-common.schema.json`. Форма `_prompt.staticText` является принятым целевым направлением ADR-049 и требует отдельной миграции схемы, потому первый срез пока продолжает валидировать `raw/normalized`.

## 5. Реализованный первый срез

Первый срез реализован как authoring-only контракт:

- `manifest-authoring-common.schema.json` содержит definitions `elementPrompt` и `promptTemplate`;
- `semanticEntity` допускает `_prompt`;
- `authoringDefinition` допускает `_promptTemplate`;
- `_prompt.status` принимает `draft`, `normalized` или `confirmed`;
- `_prompt.normalized` обязателен для `status: "normalized"` и `status: "confirmed"`;
- `_prompt.source` принимает `template`, `user`, `agent`, `imported` или `migration`;
- `_prompt.updatedAt` проверяется как UTC timestamp строкой вида `2026-06-12T00:00:00Z`;
- `authoring-compiler.cjs` и `validate-manifest-authoring.js` считают `_prompt` и `_promptTemplate` authoring-only ключами;
- examples `minimal-game.authoring.json` и `minimal-ui.authoring.json` проверяют `_prompt` на экземплярах и `_promptTemplate` на прототипах;
- editor-web local schema registry принимает эти поля, а JSON tree/property panel видят `_prompt` как обычный редактируемый объект.

Первый срез не реализует reverse-sync, автогенерацию промта из структуры, hash/coverage metadata и автоматическое исправление дрейфа.

ADR-049 supersedes целевую модель хранения полного текста: `raw/normalized` остается фактом первого среза, но не является долгосрочным source of truth.

## 6. Целевой жизненный цикл

1. Пользователь добавляет элемент на игровой экран или в сценарий.
2. Если `_prompt` отсутствует, редактор строит unsaved compiled prompt из `_promptTemplate` и dynamic YAML projection текущего JSON-узла.
3. Если `_prompt` есть, редактор берет из него static residue и добавляет актуальную dynamic YAML projection.
4. Пользователь редактирует compiled prompt как единый человеко-читаемый текст.
5. Агент получает compiled prompt, JSON-фрагменты, schema context, field dictionary и projection source map.
6. Агент отделяет static residue от структурных правок и возвращает новый `_prompt.staticText` или external Markdown content вместе с bounded `EditorChangeSet`.
7. Редактор применяет ChangeSet только после dry-run, JSON Schema validation, semantic validation and undo journal recording.
8. После применения dynamic YAML projection пересобирается из JSON.
9. Если структура манифеста меняется другим путем, drift diagnostics помечает `_prompt` как `stale` или предлагает repair через `EditorChangeSet`.

Автоматическое создание unsaved compiled prompt при выборе элемента остается целевым UX-поведением. Первый реализованный срез уже валидирует `_prompt`/`_promptTemplate`, но еще не реализует dynamic YAML projection и field dictionary.

## 7. Отношение к существующим полям

`_prompt` не заменяет `_semantics`.

- `_semantics` - короткое объяснение смысла узла для дерева, поиска и подсказок.
- `_prompt` - невосстановимая статическая часть авторского описания. Полный текст для пользователя собирается редактором через compiled prompt.

`_prompt` не заменяет `generation.prompt`.

- `generation.prompt` - промт изображения, макета или визуального региона.
- `_prompt` - статический авторский остаток промта элемента манифеста и его поведения.

`_prompt` не заменяет `EditorPatchIntent.prompt`.

- `EditorPatchIntent.prompt` - запрос на конкретную правку.
- `_prompt` - текущее подтвержденное невосстановимое описание элемента после правки.

## 8. Пример compiled prompt

Пользовательский промт:

```text
Это карточка с двумя сторонами, текст на лицевой:"Поговорить с Аленой. Алена входит в Совет, ориентирована на достижения, всегда добивается результата. Быстро схватывает новые идеи. Быстро и остывает, если не видит скорой и явной отдачи.". Текст на оборотной стороне:"Алена, как обычно, схватывает на лету. Убедившись, что проблема серьезная, она тут же заручается поддержкой Вожака. Леонид готов действовать. Нужны дальнейшие шаги!". При выборе карточки появляется возможность перехода на следующий экран и "Знания" + 1, "Доверие" + 2, "Энергия" + 1, "Статус" + 1, "Время" + 1. Методическая проблема: Сотрудник обнаружил важное, но у него нет каналов связи с принимающими решения и нет четких доказательств. Методическое решение: обращение не к первому лицу, а к кому-то из наиболее активных "правых рук". Выбор этого решения - статистически наиболее быстрый способ в большой организации донести до лиц, принимающих решения серьезную проблему.
```

Compiled prompt, который редактор показывает пользователю после разнесения структурированных данных и статического остатка:

```yaml
Статическое авторское намерение: |
  Методическая проблема: сотрудник обнаружил важное, но у него нет каналов связи с принимающими решения и нет четких доказательств.
  Методическое решение: обращение не к первому лицу, а к одному из наиболее активных помощников или "правых рук".
  Выбор этого решения - статистически наиболее быстрый способ в большой организации донести серьезную проблему до лиц, принимающих решения.

Карточка выбора:
  Лицевая сторона: "Поговорить с Аленой. Алена входит в Совет, ориентирована на достижения, всегда добивается результата. Быстро схватывает новые идеи. Быстро и остывает, если не видит скорой и явной отдачи."
  Оборотная сторона: "Алена, как обычно, схватывает на лету. Убедившись, что проблема серьезная, она тут же заручается поддержкой Вожака. Леонид готов действовать. Нужны дальнейшие шаги!"
  Поведение при выборе:
    Переход на следующий экран: true
    Изменения показателей:
      Знания: +1
      Доверие: +2
      Энергия: +1
      Статус: +1
      Время: +1
```

В этом примере YAML-блок "Карточка выбора" не хранится в `_prompt`: он пересобирается из authoring JSON. В `_prompt.staticText` сохраняется только методическая проблема, решение и другие авторские ограничения, которые нельзя надежно восстановить из структурированных полей.

## 9. Принятые правила синхронизации

ADR-049 принимает следующие правила:

- канонические authoring/runtime manifests остаются JSON;
- YAML используется только как временная проекция для редактора;
- русские названия полей берутся из schema annotations и field dictionary;
- reverse direction из compiled prompt в JSON идет только через агента и `EditorChangeSet`;
- автоматическое применение reverse-sync без dry-run, validation and user confirmation запрещено;
- drift diagnostics строится на covered pointers/hash и может помечать `_prompt` как `stale`;
- full compiled prompt не сохраняется, чтобы не дублировать JSON-структуру внутри промта.

Открытыми остаются только детали реализации: имя итогового поля (`staticText`, `residue`, `authorIntent` или `notes`), физическое место field dictionary overlay и миграция первого среза `raw/normalized`.
