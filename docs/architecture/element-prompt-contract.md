## 1. Назначение

Элементный промт - это сохраненное в authoring-манифесте описание намерения автора для конкретного экземпляра. Он отвечает на вопросы:

- что это за элемент;
- какие данные он показывает или хранит;
- какое поведение появляется при взаимодействии;
- какие изменения состояния или переходы должны произойти;
- какой методический смысл должен сохранить агент при дальнейших правках.

Это поле нужно для game authoring, UI authoring и прототипов. Для прототипов хранится шаблон, который получает новый экземпляр при создании.

## 2. Что уже есть сейчас

В проекте есть несколько близких, но неполных механизмов:

- `generation.prompt` в `docs/architecture/schemas/design-artifact.schema.json` описывает визуальную генерацию изображения или региона.
- `_semantics` в `docs/architecture/schemas/manifest-authoring-common.schema.json` дает короткое смысловое описание authoring-узла.
- `EditorPatchIntent.prompt` в `packages/editor-engine/src/index.ts` хранит пользовательский запрос на одну операцию редактирования.
- `PatchJournalStep` хранит историю конкретного изменения, включая промт операции и summary.

Ни один из этих механизмов не является сохраненным, подтвержденным и актуализируемым промтом конкретного элемента.

## 3. Найденные расхождения

1. **Визуальный промт ошибочно выглядел как универсальный.** ADR-016 говорил о поэлементном `generation.prompt`, но контекст этого поля - inpainting/refinement, compositing and image reproduction. Оно не покрывает игровое или UI-поведение, методику и state effects. ADR-048 уточняет это различие.
2. **Authoring v2 не хранил промт экземпляра.** Первый срез ADR-048 добавил `_prompt` в общий `semanticEntity`.
3. **Прототипы не хранили шаблон промта.** Первый срез ADR-048 добавил `_promptTemplate` в общий `authoringDefinition`.
4. **Редакторский prompt временный.** Текущий `EditorPatchIntent.prompt` живет в edit flow и журнале, но не становится текущим описанием элемента.
5. **Нет reverse-sync правила.** Документы не описывают, что делать, если манифест изменился вручную или агентом и сохраненный промт стал устаревшим.

## 4. Целевой контракт

Целевой authoring-контракт использует два поля:

- `_prompt` - подтвержденный промт конкретного экземпляра;
- `_promptTemplate` - шаблон промта в прототипе.

Минимальная целевая форма `_prompt`:

```json
{
  "_prompt": {
    "status": "confirmed",
    "raw": "Исходный текст пользователя или текст, скопированный из шаблона.",
    "normalized": "Структурированная формулировка, подтвержденная пользователем.",
    "source": "user",
    "language": "ru",
    "updatedAt": "2026-06-12T00:00:00Z"
  }
}
```

Минимальная целевая форма `_promptTemplate`:

```json
{
  "_promptTemplate": {
    "raw": "Опишите лицевую сторону, оборотную сторону, действие при выборе, изменения метрик и методический смысл.",
    "language": "ru",
    "appliesTo": "game.ChoiceCard"
  }
}
```

Эти формы реализованы в первом срезе ADR-048 через `docs/architecture/schemas/manifest-authoring-common.schema.json`.

## 5. Реализованный первый срез

Первый срез реализован как authoring-only контракт:

- `manifest-authoring-common.schema.json` содержит definitions `elementPrompt` и `promptTemplate`;
- `semanticEntity` допускает `_prompt`;
- `authoringDefinition` допускает `_promptTemplate`;
- `_prompt.status` принимает `draft`, `normalized` или `confirmed`;
- `_prompt.normalized` обязателен для `status: "normalized"` и `status: "confirmed"`;
- `_prompt.source` принимает `template`, `user`, `agent`, `imported` или `migration`;
- `_prompt.updatedAt` проверяется как UTC timestamp строкой вида `2026-06-12T00:00:00Z`;
- `authoring-compiler.cjs` и `validate-manifest-authoring.js` считают `_prompt` и `_promptTemplate` authoring-only ключами;
- examples `minimal-game.authoring.json` и `minimal-ui.authoring.json` проверяют `_prompt` на экземплярах и `_promptTemplate` на прототипах;
- editor-web local schema registry принимает эти поля, а JSON tree/property panel видят `_prompt` как обычный редактируемый объект.

Первый срез не реализует reverse-sync, автогенерацию промта из структуры, hash/coverage metadata и автоматическое исправление дрейфа.

## 6. Целевой жизненный цикл

1. Пользователь добавляет элемент на игровой экран или в сценарий.
2. Редактор копирует `_promptTemplate` из прототипа в `_prompt.raw` нового экземпляра.
3. Пользователь редактирует шаблон или вводит свободный промт.
4. Агент анализирует текст, структурирует его и переписывает строго без изменения смысла.
5. Пользователь проверяет и подтверждает нормализованный промт.
6. Редактор сохраняет подтвержденный промт в `_prompt`.
7. Агент создает bounded `EditorChangeSet`, который меняет authoring-манифест в соответствии с подтвержденным промтом.
8. Редактор применяет ChangeSet только после dry-run, JSON Schema validation, semantic validation and undo journal recording.
9. Если манифест меняется другим путем, будущий механизм синхронизации должен либо обновить `_prompt`, либо пометить его как устаревший и запросить подтверждение предложения по исправлению.

Автоматическое копирование `_promptTemplate` в `_prompt.raw` при создании элемента через UI остается следующим UX-шагом, если конкретный editor flow еще не создает элементы из прототипов.

## 7. Отношение к существующим полям

`_prompt` не заменяет `_semantics`.

- `_semantics` - короткое объяснение смысла узла для дерева, поиска и подсказок.
- `_prompt` - рабочее авторское описание экземпляра, пригодное для генерации и проверки структуры.

`_prompt` не заменяет `generation.prompt`.

- `generation.prompt` - промт изображения, макета или визуального региона.
- `_prompt` - промт элемента манифеста и его поведения.

`_prompt` не заменяет `EditorPatchIntent.prompt`.

- `EditorPatchIntent.prompt` - запрос на конкретную правку.
- `_prompt` - текущее подтвержденное описание элемента после правки.

## 8. Пример нормализации

Пользовательский промт:

```text
Это карточка с двумя сторонами, текст на лицевой:"Поговорить с Аленой. Алена входит в Совет, ориентирована на достижения, всегда добивается результата. Быстро схватывает новые идеи. Быстро и остывает, если не видит скорой и явной отдачи.". Текст на оборотной стороне:"Алена, как обычно, схватывает на лету. Убедившись, что проблема серьезная, она тут же заручается поддержкой Вожака. Леонид готов действовать. Нужны дальнейшие шаги!". При выборе карточки появляется возможность перехода на следующий экран и "Знания" + 1, "Доверие" + 2, "Энергия" + 1, "Статус" + 1, "Время" + 1. Методическая проблема: Сотрудник обнаружил важное, но у него нет каналов связи с принимающими решения и нет четких доказательств. Методическое решение: обращение не к первому лицу, а к кому-то из наиболее активных "правых рук". Выбор этого решения - статистически наиболее быстрый способ в большой организации донести до лиц, принимающих решения серьезную проблему.
```

Нормализованный промт, который агент возвращает на проверку:

```text
Создать интерактивную карточку с двумя сторонами.

Лицевая сторона:
"Поговорить с Аленой. Алена входит в Совет, ориентирована на достижения, всегда добивается результата. Быстро схватывает новые идеи. Быстро и остывает, если не видит скорой и явной отдачи."

Оборотная сторона:
"Алена, как обычно, схватывает на лету. Убедившись, что проблема серьезная, она тут же заручается поддержкой Вожака. Леонид готов действовать. Нужны дальнейшие шаги!"

Поведение при выборе:
- открыть возможность перехода на следующий экран;
- изменить метрики: "Знания" +1, "Доверие" +2, "Энергия" +1, "Статус" +1, "Время" +1.

Методическая проблема:
Сотрудник обнаружил важное, но у него нет каналов связи с принимающими решения и нет четких доказательств.

Методическое решение:
Обратиться не к первому лицу, а к одному из наиболее активных помощников или "правых рук". Такой выбор является статистически наиболее быстрым способом в большой организации донести серьезную проблему до лиц, принимающих решения.
```

После подтверждения агент должен выразить этот смысл в структурированных authoring-полях: тексты карточки, переходы, effects/metric deltas, условия доступности и методические metadata. Конкретные поля зависят от среза схем и типа прототипа.

## 9. Открытые вопросы синхронизации

Механизм поддержания актуальности `_prompt` принят как следующий этап и пока не выбран. Для следующего обсуждения нужно решить:

- является ли `_prompt.normalized` главным источником для regeneration или только пояснением к структуре;
- нужен ли hash структурных полей, которые покрывает промт;
- должна ли система автоматически переписывать `_prompt` при правке манифеста или только предлагать исправление;
- как показывать пользователю расхождение между промтом и структурой;
- какие типы сущностей обязаны иметь `_prompt`;
- нужно ли хранить историю промтов отдельно от Git history and PatchJournal.
