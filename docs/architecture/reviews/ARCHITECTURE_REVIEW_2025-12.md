## 1. Резюме

После ревью ноября 2025 архитектура Cubica заметно продвинулась в части **версии манифестов**, **безопасности JS‑песочницы**, **мультиплеера** и **стратегии восстановления сессий**. Ряд рисков, обозначенных как критические в предыдущем документе, закрыт принятием новых ADR и уточнением backend‑дизайна.

Оставшиеся пробелы смещаются в область **качества и операционности** (тестирование LLM‑игр, наблюдаемость, rate limiting), а также в расширенные возможности Editor. Эти темы уже отражены в ROADMAP через фичи F_00050/51/52/61/62 и считаются допустимым техдолгом для этапа MVP.

## 2. Что изменилось после ревью 2025-11

### 2.1. Manifest Versioning (ADR‑008)

- Принят ADR‑008 `docs/architecture/adrs/008-manifest-versioning.md`, описывающий стратегию версионирования JSON‑манифестов.
- Введены поля `schema_version` и `min_engine_version`, а также логика совместимости между версиями движка и схемы.
- Риск «отсутствие стратегии версионирования манифестов» из ревью 2025‑11 считается закрытым.

### 2.2. JS Sandbox Security (ADR‑010)

- Принят ADR‑010 `docs/architecture/adrs/010-js-sandbox-security.md`, конкретизирующий выбор `isolated-vm`, лимиты по времени и памяти, а также контракт окружения (`state`, `args`, `std`).
- Фича F_00041 отмечена как `done`, базовая модель угроз для скриптового движка зафиксирована.
- Риск «sandbox недоописан» из ревью 2025‑11 закрыт на архитектурном уровне; реализация остаётся задачей сервисов.

### 2.3. Multiplayer Architecture (ADR‑011)

- Обновлён и расширен ADR‑011 `docs/architecture/adrs/011-multiplayer-architecture.md`, описывающий free‑form модель с очередью событий, `state_version` и `last_event_sequence`.
- Добавлен ExecPlan `docs/tasks/content-packs/CP_00060_multiplayer_architecture_hardening.yaml` и фича F_00063 для уточнения очереди событий.
- Риск по мультиплееру из ревью 2025‑11 переведён в статус «проработан в ADR, частично реализован».

### 2.4. Session Persistence & Recovery (ADR‑005 + backend/session-persistence.md)

- ADR‑005 `docs/architecture/adrs/005-session-persistence.md` дополнен разделом **Session Recovery** (checkpointing, TTL, dead lock detection) и **Redis Integration (Phase 2+)**.
- Дизайн‑док `docs/architecture/backend/session-persistence.md` синхронизирован с ADR‑005 и уточняет схему таблицы `game_sessions` и алгоритм блокировки.
- ExecPlan `CP_00042_session_recovery.yaml` задаёт путь к реализации механизма восстановления на уровне Router.
- Риск «нет описания механизма отката/восстановления сессий» из ревью 2025‑11 закрыт на уровне архитектурной спецификации.

### 2.5. Redis Usage

- Добавлен дизайн‑док `docs/architecture/backend/redis-usage.md`, описывающий единые паттерны ключей и блокировок для Repository и игровых сессий.
- Файл `docs/architecture/redis-keys.md` уточнён: теперь явно указано, что он описывает только кэш манифестов.
- Это создаёт целостную картину применения Redis в контексте фичи F_00062 (Redis Integration).

## 3. Текущее состояние ключевых областей

### 3.1. Доменные сервисы и манифесты

- Логическая архитектура сервисов и формат манифестов остаются согласованными между `PROJECT_OVERVIEW.md`, `docs/architecture/PROJECT_ARCHITECTURE.md` и ADR‑001/003/004/008.
- Центральное место JSON‑манифестов и LLM‑first подхода подтверждено в задаче F_00010 и её ExecPlan.

### 3.2. Game Engine, LLM Context Pipeline и Hybrid Execution

- ADR‑004, ADR‑007, `docs/architecture/engine/llm-context-pipeline.md` и `services/game-engine/DEV_GUIDE.md` описывают полный путь обработки хода, включая разделение на LLM и Script‑ветви.
- Безопасность скриптов закреплена ADR‑010; открытым остаётся вопрос инструментов тестирования и отладки скриптов (переносится в область Testing Strategy).

### 3.3. Session Persistence, Recovery и Multiplayer

- Хранение состояния и базовый алгоритм блокировки описаны в ADR‑005 и `backend/session-persistence.md`.
- Поверх этого слоя мультиплеерная модель (ADR‑011) дополняет архитектуру очередью событий и версионностью состояния.
- Механизм Session Recovery имеет спецификацию (ADR‑005, F_00042), но требует реализации в Router и тестового покрытия.

### 3.4. Redis и масштабирование

- Для этапа MVP Redis остаётся опциональным; основной источник истины — PostgreSQL.
- Для этапа масштабирования (Phase 2) определены паттерны read‑through кэша, распределённых блокировок и возможного write‑behind, согласованные с F_00062.

## 4. Оставшиеся пробелы и задачи

Ниже перечислены ключевые области, которые **осознанно отложены** и отражены в ROADMAP как отдельные фичи. Неполное описание этих областей **не считается архитектурным дефектом**, так как есть явные задачи и владельцы.

1. **Testing Strategy for LLM Games (F_00050):**
   - Нет отдельного архитектурного документа `docs/architecture/testing-strategy.md`.
   - Требуется формализовать подходы к snapshot/semantic тестированию диалогов, replay LLM‑ответов и unit‑тестам Script‑части.
2. **Observability Framework (F_00051, E_0050):**
   - Стандарты логирования, метрик и трейсинга пока существуют только в виде требований в задачах и обзоре.
   - Необходим отдельный ADR по Observability Standards и согласование с DEV_GUIDE сервисов.
3. **Rate Limiting & Budget Control (F_00052):**
   - Архитектура rate limiting для Router/Engine (в том числе хранение счётчиков и лимитов) пока не зафиксирована в отдельном ADR.
   - Важно увязать лимиты с манифестами и стратегией Session Persistence (например, хранить агрегаты по токенам и RPS).
4. **Redis Integration (реализация F_00062):**
   - Архитектурные паттерны описаны, но нет практической реализации и измерений производительности.
   - Потребуются дополнительные ADR/документы, если будет включён режим write‑behind.
5. **Game Editor Intelligence (F_00061):**
   - Нет отдельного архитектурного документа, описывающего подсистему интеллектуального редактора (schema‑aware prompting, online‑валидация, генерация тестов).
   - Важно увязать будущий дизайн с уже принятыми схемами манифестов и Testing Strategy.
