# AI-first + Code-first архитектурное ревью Cubica

**Дата:** 19 марта 2026 г.  
**Автор:** Codex  
**Тип:** Архитектурное ревью текущего состояния + предложение целевой AI-first и Code-first архитектуры

---

## 1. Краткий вывод

У проекта сильная целевая архитектура, но фактическая реализация заметно отстаёт от неё. Сейчас Cubica ближе не к платформе сервисов, а к архитектурному каркасу вокруг одного рабочего вертикального среза:

- игровые манифесты;
- базовые SDK-пакеты;
- архивный Next.js player;
- отдельные контрактные модули Router;
- большой слой проектной документации.

Главная проблема не в том, что архитектурные идеи плохие. Проблема в том, что проект развивается как **Docs-first + Architecture-first**, тогда как для такого продукта лучше подходит **AI-first + Code-first**: машинно-читаемые контракты, исполнимые модули, тестируемый вертикальный срез и AI как ограниченный слой возможностей, а не как неформализованное ядро системы.

## 2. Что видно по текущему состоянию репозитория

### 2.1. Самый зрелый слой — контент игры и манифесты

Наиболее оформленный и полезный слой проекта:

- `games/antarctica/game.manifest.json`
- `games/antarctica/ui/web/ui.manifest.json`
- `games/antarctica/scenario.md`
- `games/antarctica/scripts/actions.js`

Здесь уже есть правильные идеи:

- разделение логического и UI-манифеста;
- action registry;
- training metadata;
- отдельные дизайн-артефакты;
- попытка строить игру как данные, а не как жёстко прошитый UI.

### 2.2. Реальный runtime сейчас сосредоточен в SDK и draft player

Фактическая исполнимая архитектура сегодня определяется в основном такими слоями:

- `SDK/core`
- `SDK/react-sdk`
- `SDK/shared`
- `draft/antarctica-nextjs-player`

При этом:

- `SDK/core/src/index.ts` содержит placeholder для session lifecycle;
- `SDK/react-sdk` уже задаёт shape client runtime, но API ещё черновой;
- архивный Next.js player по сути лучше отражает реальный исполнимый поток, чем часть “официальной” структуры.

### 2.3. Backend-сервисы в основном декларативны

Почти все каталоги `services/*` — это пока не сервисы, а намерения:

- `game-catalog`
- `game-editor`
- `game-engine`
- `game-repository`
- `metadata-db`
- `router`

Из них только `router` содержит не пустые исходники, но и там это в основном контракты и типы:

- `services/router/src/sessionEvents.ts`
- `services/router/src/sessionRecovery.ts`

То есть с инженерной точки зрения система пока не разложена на микросервисы. Она ещё не дошла даже до одного рабочего backend ядра.

### 2.4. Монорепозиторий пока организационный, а не исполнимый

В проекте есть workspaces, но единая инженерная система ещё не собрана:

- нет общего build/test/lint контура;
- во многих пакетах `build`, `test`, `lint` — заглушки;
- отсутствует единая “точка правды” о том, что реально активно, а что draft/archive/placeholder.

Это мешает и людям, и AI-агентам.

## 3. Что стоит изменить с точки зрения AI-first

### 3.1. AI не должен быть неформализованным центром runtime

Сейчас значительная часть целевой архитектуры построена вокруг идеи `LLM-first`, где LLM мыслится как основной игровой движок. Это допустимо как исследовательское направление, но как базовая архитектура платформы для неизвестных будущих игр — слишком рискованно.

Основные проблемы такого подхода:

- слабая воспроизводимость;
- недетерминированность переходов состояния;
- сложность регрессионного тестирования;
- высокая стоимость и latency;
- трудности с аудитом игровых решений;
- плохо контролируемое поведение при масштабировании и мультиплеере.

Для **AI-first** лучше такая модель:

- есть **deterministic runtime core**;
- есть **AI capability layer**;
- AI помогает генерировать контент, narration, hints, сценарные варианты, оценки и proposals;
- любое изменение состояния проходит через формальный протокол и валидацию.

То есть AI становится сильным, но ограниченным участником системы.

### 3.2. AI-first требует machine-readable репозитория

Для AI-first проекта документы в prose недостаточны. Нужен слой, который машина может читать без интерпретационных потерь.

Рекомендация:

ввести в корне репозитория единый индекс, например:

- `repo-manifest.json`
- или `docs/architecture/repo-index.yaml`

Для каждого крупного артефакта там должны быть поля:

- `id`
- `kind`
- `path`
- `status`
- `source_of_truth`
- `owner`
- `depends_on`
- `adr_refs`
- `task_refs`
- `last_verified_at`

Особенно важны статусы:

- `actual`
- `target`
- `draft`
- `archive`
- `deprecated`
- `placeholder`

Без этого AI всегда будет смешивать активный код, исторические артефакты и целевые проектные намерения.

### 3.3. AI-first лучше строить вокруг narrow interfaces

Вместо “большой AI-системы” лучше иметь набор узких, понятных интерфейсов:

- `Manifest Analyzer`
- `Scenario Compiler`
- `Prompt Builder`
- `Narration Generator`
- `Action Evaluator`
- `State Proposal Generator`
- `Content Authoring Assistant`

Каждый такой AI-компонент должен иметь:

- явный входной контракт;
- явный выходной контракт;
- версионирование;
- logging/eval hooks.

Это гораздо лучше, чем один “умный Game Engine”, внутри которого смешано всё.

## 4. Что стоит изменить с точки зрения Code-first

### 4.1. Источник истины нужно сместить из документов в код и схемы

Сейчас многие архитектурные знания живут прежде всего в ADR и обзорных документах. Для Code-first системы это вторично.

Приоритет должен быть таким:

1. исполнимый модуль;
2. тест;
3. schema / contract;
4. архитектурная документация;
5. обзорный prose.

ADR по-прежнему полезны, но они должны фиксировать уже проверяемые границы и решения, а не подменять их.

### 4.2. Нужен один минимальный рабочий vertical slice

До дальнейшего масштабирования надо собрать один путь end-to-end:

- одна игра;
- один web viewer;
- один Router API;
- один session store;
- один runtime engine;
- один integration flow;
- один CI путь.

Без этого все разговоры о platform architecture остаются слишком абстрактными.

### 4.3. Контракты надо разделить на query / command / realtime

Сейчас в клиентском runtime уже видно смешение чтения и действий. Для Code-first системы нужно жёсткое разделение:

- `createSession`
- `getSession`
- `getSessionState`
- `dispatchAction`
- `subscribeSessionEvents`
- `loadManifestBundle`

Это упростит:

- SDK;
- сервер;
- тесты;
- eventing;
- последующее выделение сервисов.

### 4.4. Нужны реальные тесты вместо присутствия каталогов `tests/`

Наличие `tests/` не равно наличию тестового барьера. Минимум, который нужен:

- unit tests для `SDK/core`;
- unit tests для `SDK/react-sdk`;
- unit tests для session/runtime модулей backend;
- schema validation tests;
- один integration smoke по вертикальному срезу.

## 5. Не помешает ли schema-first неизвестным будущим играм?

Короткий ответ: **нет, если schema-first сделать правильно**. Помешает не schema-first, а **жёсткая domain schema**, которая заранее пытается перечислить все будущие типы игр и механик.

### 5.1. Что нельзя делать

Не стоит строить схему так, будто уже известен полный набор будущих жанров и игровых динамик.

Плохой вариант:

- жёстко вшить конкретные типы механик;
- вшить фиксированные поля под текущую игру;
- считать, что все игры имеют одинаковые стейты, экраны и типы действий.

Такой schema-first действительно убьёт гибкость.

### 5.2. Что нужно делать вместо этого

Нужен **schema-first на уровне протоколов и слоёв**, а не на уровне всех возможных механик.

То есть схема должна описывать:

- обязательные границы;
- жизненный цикл;
- extension points;
- правила композиции;
- capability negotiation.

Не надо заранее фиксировать “все типы игр”. Надо фиксировать:

- как игра описывает метаданные;
- как объявляет actions;
- как хранит initial state;
- как подключает scripts/extensions;
- как UI привязывается к данным;
- как runtime принимает и возвращает эффекты.

### 5.3. Практический принцип

Лучше строить **layered schemas**:

1. **Core schema**
   Описывает общие обязательные секции:
   - `meta`
   - `runtime`
   - `state`
   - `actions`
   - `capabilities`
   - `assets`

2. **Capability schemas**
   Подключаемые схемы под опциональные возможности:
   - multiplayer
   - timers
   - inventory
   - negotiation
   - hidden roles
   - map/board
   - campaign progression

3. **Game-specific extension schema**
   Игра может объявлять собственные расширения, если они namespaced и валидируются отдельно.

Примерно так:

- core schema отвечает за универсальные границы;
- capability schema отвечает за повторяемые паттерны;
- кастомные поля живут в `extensions`, `custom`, `domain`, `x-*` секциях.

### 5.4. Что schema-first даёт именно при неизвестном будущем

Если сделать его таким образом, то schema-first не ограничивает, а помогает:

- новые игры получают стабильный минимальный контракт;
- нестандартные механики всё ещё можно расширять;
- AI-агенты могут понимать структуру артефактов;
- runtime может договариваться о поддерживаемых возможностях;
- появляется контролируемая эволюция форматов.

Правильная формула здесь такая:

**Schema-first для интерфейсов, extension-first для доменной вариативности.**

## 6. Предлагаемая AI-first + Code-first целевая схема репозитория

Ниже не “идеальная академическая схема”, а схема, которая подходит текущему состоянию проекта и оставляет путь к масштабированию.

```text
Cubica/
├── apps/
│   ├── player-web/                 # основной web player / viewer shell
│   ├── authoring-studio/           # будущий editor UI
│   └── ops-console/                # внутренние runtime/admin инструменты
│
├── packages/
│   ├── contracts/
│   │   ├── session/                # DTO, events, commands, query contracts
│   │   ├── manifest/               # core manifest contracts + validators
│   │   ├── runtime/                # engine result/effect/state contracts
│   │   └── ai/                     # AI task contracts, eval result contracts
│   │
│   ├── sdk-core/                   # transport, state sync, protocol helpers
│   ├── sdk-react/                  # React hooks/adapters
│   ├── ui-kit/                     # shared UI primitives
│   ├── viewer-web/                 # universal web viewer runtime
│   ├── manifest-tooling/           # validation, compilation, projection tools
│   ├── runtime-core/               # deterministic engine core
│   ├── runtime-capabilities/       # reusable gameplay capabilities
│   ├── ai-orchestrator/            # prompting, eval, AI adapters
│   └── observability/              # logging/metrics/tracing helpers
│
├── services/
│   └── runtime-api/                # один deployable backend на текущем этапе
│
├── games/
│   ├── antarctica/
│   │   ├── game.manifest.json
│   │   ├── ui/
│   │   ├── scripts/
│   │   ├── scenario.md
│   │   └── tests/
│   └── ...
│
├── schemas/
│   ├── core/
│   ├── capabilities/
│   ├── api/
│   └── examples/
│
├── tools/
│   ├── manifest-cli/
│   ├── replay/
│   ├── seed/
│   └── eval/
│
├── docs/
│   ├── architecture/
│   ├── decisions/
│   └── reviews/
│
├── repo-manifest.json              # машиночитаемый индекс репозитория
├── pnpm-workspace.yaml             # или npm/turbo, но лучше единая workspace-модель
├── turbo.json                      # опционально
└── package.json
```

### 6.1. Принципы этой схемы

- `apps/` — пользовательские приложения и оболочки;
- `packages/` — повторно используемые исполнимые модули;
- `services/` — deployable units;
- `games/` — content bundles;
- `schemas/` — формальные контракты;
- `tools/` — инженерные утилиты;
- `docs/` — документация, не являющаяся источником истины для runtime.

### 6.2. Что важно сохранить из текущего проекта

- `games/` как канонический доменный слой;
- split manifest;
- layered SDK;
- отдельный viewer-слой;
- ADR-практику;
- идею capability/extension model.

### 6.3. Что важно убрать или сократить

- размытость между `draft` и `actual`;
- много пустых сервисов;
- микросервисную декомпозицию до появления рабочего ядра;
- зависимость целевой архитектуры от одного “магического” LLM engine.

## 7. Предлагаемая схема слоёв системы

### 7.1. Content Layer

Содержит игровые артефакты:

- manifests;
- сценарии;
- ассеты;
- локальные scripts;
- capability declarations;
- тестовые сценарии для конкретной игры.

### 7.2. Contracts Layer

Содержит строгие машинно-читаемые границы:

- API DTO;
- session events;
- runtime effects;
- manifest schemas;
- AI task/result schemas.

Это должен быть один из самых стабильных слоёв проекта.

### 7.3. Runtime Core Layer

Здесь живёт детерминированное исполнение:

- session lifecycle;
- action dispatch;
- state transition;
- script execution;
- event persistence;
- capability execution.

Этот слой не зависит от конкретного канала доставки.

### 7.4. AI Capability Layer

Отдельный слой, в котором живут:

- prompt builders;
- model adapters;
- evaluators;
- narrative generators;
- action proposal generators;
- replay/eval tooling.

Этот слой не должен напрямую подменять runtime core.

### 7.5. Delivery Layer

Каналы доставки:

- web viewer;
- React SDK;
- позже Telegram/mobile/другие клиенты.

### 7.6. Authoring Layer

Инструменты создания и редактирования:

- editor;
- manifest tooling;
- validation;
- AI-assisted authoring.

## 8. Как свести backend к модульному монолиту

### 8.1. Что это значит в контексте Cubica

Модульный монолит — это **один deployable backend**, внутри которого есть жёстко разделённые доменные модули с явными интерфейсами и запретом на хаотические прямые зависимости.

То есть это не “большой файл со всем подряд”. Это один процесс/репозиторный сервис с внутренней архитектурой, похожей на будущие сервисы.

### 8.2. Предлагаемая модульная разбивка

Внутри одного backend-а я бы выделил такие модули:

- `player-api`
  - create session
  - get session
  - get state
  - dispatch action
  - subscribe/replay events

- `session`
  - session lifecycle
  - locks
  - versions
  - checkpoints
  - recovery

- `runtime`
  - action routing
  - deterministic transitions
  - capability execution
  - script execution
  - effect assembly

- `content`
  - load manifest bundle
  - resolve game version
  - validate content
  - cache bundles

- `ai`
  - prompt assembly
  - AI calls
  - result normalization
  - eval/logging

- `telemetry`
  - logs
  - traces
  - metrics
  - audit trail

- `admin`
  - health
  - replay
  - inspection
  - internal ops endpoints

### 8.3. Что изменится относительно текущей архитектуры

Сейчас целевая схема подразумевает:

- Router
- Game Engine
- Game Repository
- Metadata DB

как отдельные сервисные блоки.

После перехода к модульному монолиту на ближайшую фазу получится так:

- внешний `Router` как отдельный сервис исчезает;
- `Engine` перестаёт быть отдельным deployable, но остаётся модулем;
- `Repository` перестаёт быть отдельным deployable, но остаётся content/storage модулем;
- `Metadata` сначала может быть просто набором таблиц и экспортов внутри того же backend-а.

То есть меняется **операционная декомпозиция**, но не обязательно **логическая декомпозиция**.

### 8.4. Почему это упростит жизнь

Преимущества для текущей стадии:

- один деплой;
- один интеграционный контур;
- один логический runtime path;
- меньше распределённой сложности;
- проще тестировать;
- проще дебажить;
- проще стабилизировать контракты.

Для одного-двух активных игровых сценариев это гораздо рациональнее.

## 9. Как сделать будущее выделение микросервисов простым

Ключевая идея: проектировать не “сразу сервисы”, а **модули с будущими сервисными границами**.

### 9.1. Правила модульного монолита, которые сохранят путь к сервисам

1. У каждого модуля есть свой публичный интерфейс.
2. Нельзя импортировать внутренности соседнего модуля напрямую.
3. Взаимодействие между модулями идёт через команды, query и доменные события.
4. Хранилища инкапсулированы на уровне модуля.
5. Побочные эффекты отделены от бизнес-логики.

Если эти правила соблюдаются, выделение сервиса позже становится в основном операционной задачей.

### 9.2. Какие модули логично выделять первыми

Когда система вырастет, отделять стоит не абстрактные “сервисы из красивой картинки”, а реальные hot spots.

Вероятный порядок:

1. `ai`  
   Отделяется рано, если нагрузка по LLM, бюджеты, ретраи и провайдеры становятся отдельной сложностью.

2. `content`  
   Отделяется, если появляется независимый жизненный цикл публикации, версионирования и кэширования игр.

3. `session/runtime`  
   Отделяются, если действительно нужна высокая масштабируемость active sessions.

4. `telemetry/analytics`  
   Выносятся отдельно, если аналитический контур растёт независимо.

### 9.3. Какие технические приёмы помогут не зацементироваться

- использовать внутреннюю шину доменных событий даже внутри монолита;
- формализовать DTO как будто они уже межсервисные;
- держать SQL/репозитории модульно, а не одним общим data layer;
- не позволять UI/API слою знать детали runtime internals;
- писать integration tests через публичный API, а не через внутренние вызовы.

### 9.4. Практический критерий для отделения сервиса

Выделять модуль в отдельный сервис стоит, когда одновременно выполняются хотя бы несколько условий:

- у него отдельный профиль нагрузки;
- отдельный SLA;
- отдельный жизненный цикл релизов;
- отдельные команды/владельцы;
- отдельные требования к масштабированию или безопасности.

До этого момента модульный монолит обычно дешевле и надёжнее.

## 10. Рекомендуемая эволюционная траектория

### Фаза 1. Собрать рабочее ядро

- один backend `runtime-api`;
- одна игра `Antarctica`;
- один web player;
- один deterministic runtime;
- AI как optional capability;
- реальные integration tests;
- repo-manifest с маркировкой статусов.

### Фаза 2. Нормализовать платформенные границы

- стабильные contracts packages;
- capability schemas;
- viewer-web как отдельный повторно используемый runtime;
- authoring tooling;
- manifest compiler / validator;
- replay/eval инструменты.

### Фаза 3. Выносить сервисы только по фактической нагрузке

- выделить AI service при росте AI orchestration;
- выделить content service при росте publication/versioning;
- выделить session/runtime service при росте real-time нагрузки;
- выделить analytics pipeline при росте объёмов событий.

---

## 11. Основные рекомендации

1. Сместить архитектуру из `Docs-first` в `Code-first`.
2. Считать AI слоем возможностей, а не единственным ядром исполнения.
3. Оставить schema-first, но только для интерфейсов и capability-слоёв, а не для полного предсказания всех будущих игр.
4. Упростить backend до модульного монолита на ближайшую фазу.
5. Собрать один честный вертикальный срез end-to-end.
6. Ввести машиночитаемый индекс артефактов репозитория.
7. Подготовить модули так, чтобы их можно было безболезненно выделять в сервисы позже.

---

## 12. Итог

Архитектура Cubica не требует отказа от AI-first. Но ей нужна более строгая форма AI-first:

- AI как capability layer;
- code/contracts как реальный источник истины;
- schema-first как способ стабилизировать интерфейсы, а не ограничить творчество;
- модульный монолит как промежуточная, но инженерно зрелая форма backend-а.

Это не “шаг назад” относительно платформенных амбиций. Наоборот: это способ построить платформу так, чтобы она действительно выдержала неизвестные будущие игры, форматы взаимодействия и технологические развилки.
