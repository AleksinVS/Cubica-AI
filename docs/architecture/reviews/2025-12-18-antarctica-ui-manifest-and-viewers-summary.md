## Ключевые вопросы и решения

### 1) Как заставить `?fixture=antarctica` отрисоваться

Причина: текущий плеер и утилиты (`findEntryPoint`, рендер) ориентированы на “legacy” формат UI-дерева (`state.application.elements[...]`), а данные “Antarctica” уже описываются через секцию `ui` и в перспективе должны жить в UI manifest.

Решение:
- Переписать `renderer/findEntryPoint` под целевой формат UI manifest (экраны + дерево `children`).
- На первом этапе допускаем, что viewer хранит “view model” как **единый объект** (например, `{ ui, game }`) вместо раздельных хранилищ.

### 2) Протокол действий UI → Presenter

Причина: в коде/фикстурах используется `proc/props`, но целевые схемы манифестов и SDUI-концепция используют `command/payload`, и поддержка двух протоколов создаёт долг и двусмысленность контрактов.

Решение:
- Принять `command/payload` как единственный формат действий.
- Сделать **жёсткую миграцию**: без обратной совместимости с `proc/props` (одномоментно для фикстур/манифестов/SDK).

### 3) Формат патчей состояния

Причина: для MVP нужен простой и стабильный контракт обновления состояния между Router/Engine и клиентом; при этом иногда нужен точечный контроль изменений.

Решение:
- Патчи состояния: **JSON Merge Patch по умолчанию** + **опционально JSON Patch** для сложных кейсов (формат “оба”).
- Каноническое расположение метрик: только `state.public.metrics.*`.

### 4) Где хранить контент игр и что делать с `templates`

Причина: целевая структура репозитория должна быть однозначной: контент игры — в каталоге игры; “шаблоны” как отдельный слой в текущем виде мешают и устаревают.

Решение:
- Контент игры размещается в каталоге игры внутри `games/<gameId>/...`.
- Каталог `games/templates` подлежит удалению.
- Универсальные viewers выделяются в отдельную библиотеку ближе к SDK: `SDK/viewers/*`.

### 5) Что считать “сценарием” и “скриптами” в `assets`

Причина: “источник истины” сценария должен быть переносимым и пригодным для работы методистов/разработчиков; при этом “скрипты” должны соответствовать целевой Hybrid Engine модели, а не быть HTML-заглушкой.

Решение:
- `draft/Antarctica/scenario.md` переносится в `games/...` и становится **источником истины** сценария.
- `assets.scripts` подразумевается как **скрипты движка** (server-side, hybrid engine), которые будут оформлены в процессе рефакторинга “Antarctica”.

### 6) UI внутри game manifest или отдельный UI manifest

Причина: целевая архитектура требует разделения логики игры и UI, а также поддержки нескольких каналов (web/telegram) без дублирования game manifest.

Решение:
- UI выносится в отдельные UI manifests.
- Сразу закладывается структура по каналам: минимум `web` и `telegram`.

### 7) ESLint / `next lint` как сделать неинтерактивным

Причина: `npm run lint` в Next.js-плеере запускает интерактивный мастер конфигурации, что ломает воспроизводимость, CI и автоматические проверки.

Решение:
- Добавить минимальный ESLint-конфиг **локально** в `games/antarctica-nextjs-player`, чтобы `npm run lint` проходил без интерактива.

## Общий план работ (высокоуровнево)

1) Подготовить “пакет игры” `games/antarctica/`:
   - перенести `scenario.md`;
   - создать логический game manifest;
   - создать UI manifests для `web` и `telegram`.
2) Обновить протокол действий в SDK/shared и в плеере: перейти на `command/payload` без алиасов.
3) Переписать рендерер плеера под UI manifest (экраны + дерево `children`), сохранив на старте единый `view model` объект.
4) Привести dev-стаб Router к патчам `state.public.metrics.*` (Merge Patch по умолчанию; JSON Patch опционально).
5) Добавить ESLint-конфиг в `games/antarctica-nextjs-player`, добиться успешного `npm run lint`.
6) Удалить `games/templates`, обновить `PROJECT_STRUCTURE.md` и связанные task-документы, чтобы структура и задачи не противоречили фактическому состоянию.
