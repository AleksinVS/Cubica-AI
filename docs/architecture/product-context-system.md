# Черновик архитектуры системы продуктового контекста Cubica

- **Статус:** Draft — предложение для обсуждения, не принятое решение
- **Версия:** 0.1
- **Дата:** 2026-07-18
- **Связанное черновое решение:** ADR-088
- **Область:** ИИ-помощники разработчиков игр и ведущих, пользовательские
  диалоги, проекты игр, подготовка и проведение игр

Документ описывает отдельный продуктовый контур долговременного контекста для
агентов Cubica. Контур здесь — изолированная группа компонентов, данных и
правил доступа. Он не расширяет инженерную проектную вики и не меняет принятое
решение о ней. Цель — дать пользовательским агентам ситуативную
осведомлённость, то есть собрать для текущего запроса только разрешённые и
актуальные сведения о пользователе, его роли, игровом проекте, покупке,
мероприятии и текущем диалоге.

## 1. Краткий вывод

Рекомендуется не выбирать между «одной системой для всего» и «двумя полностью
разными реализациями». Для Cubica оптимальна модель:

> **общее технологическое ядро контекста + раздельные домены данных и границы
> доверия**.

На верхнем уровне остаются две системы:

1. **Инженерная система знаний Cubica** — существующая файловая вики для
   агентов, разрабатывающих саму платформу. Её канон (первичный источник
   истины) — Git/Markdown, а SQLite, граф и индексы являются локальными
   производными.
2. **Система продуктового контекста** — новый многопользовательский контур для
   пользовательских агентов. Его постоянные данные живут в PostgreSQL,
   защищаются правами арендатора (изолированной организации или клиентского
   аккаунта) и пользователя, имеют сроки хранения, происхождение, исправление
   и удаление.

Внутри продуктовой системы нужны две изолированные области:

- **авторский контекст** — разработчик игры, команда, проект, версия проекта,
  редактор и диалоги о разработке;
- **контекст проведения** — ведущий, приобретённая опубликованная игра,
  заказчик, мероприятие, игровая сессия и диалоги о подготовке/разборе.

Это не два продукта и не два независимо разработанных сервиса. Они используют
одни контракты, алгоритмы поиска и оценки, но не общий набор доступных данных.
Такое разделение сохраняет принятую физическую границу между авторским и
публичным контурами: черновики, внутренние заметки автора и код проекта не
должны становиться доступными покупателю или игровому агенту.

Первая целевая реализация не требует отдельной графовой или векторной базы и
не требует нового микросервиса. Достаточно модулей контекста в соответствующих
контурах Cubica, PostgreSQL, встроенного полнотекстового поиска и `pgvector`
(расширения PostgreSQL для хранения и поиска числовых представлений смысла
текста). Физическое выделение сервиса оправдано позже только отдельным
профилем нагрузки, требованиями изоляции или собственным уровнем доступности.

## 2. Что переносится из проектной системы знаний

Существующая инженерная система дала полезные общие принципы, но её модель
хранения не подходит пользовательским данным. FTS (full-text search,
полнотекстовый поиск) находит документы по словам и их нормализованным формам;
эмбеддинг — производный числовой вектор для поиска близкого смысла.

| Элемент | Инженерная система | Адаптация для продукта |
|---|---|---|
| Разделение канона и индекса | Markdown — канон, SQLite — производная | Диалоги и подтверждённые записи — постоянные данные; полнотекстовые и векторные индексы — производные |
| Типизированные связи | Wiki-рёбра между архитектурными концептами | Ограниченные связи между пользователем, проектом, решением, мероприятием и источником |
| Происхождение | Ссылки на ADR, код и схемы | Ссылка на сообщение, пользователя, доменное событие, версию проекта, покупку или сессию |
| Противоречия | `open/resolved/deferred` в каноне | `candidate/confirmed/disputed/superseded/retracted` с явным замещением |
| Гибридный поиск | FTS + векторы + измерение качества | FTS + векторы после обязательной проверки прав, времени и области |
| Полная пересборка | Индекс восстанавливается из Git | Индексы восстанавливаются из разрешённых постоянных записей; удаление источника удаляет и производные |
| JSON Schema | Проверка страниц и конфигурации | Проверка команд памяти, записей происхождения, результатов извлечения и контекста агента |

Не переносятся:

- Git/Markdown как хранилище персональных диалогов;
- локальная SQLite как продуктовая база;
- граф кода, AST, символы и карта репозитория;
- общий открытый граф между платформой и данными клиентов;
- предположение, что участники репозитория имеют одинаковый доступ к знаниям.

## 3. Рекомендуемая граница систем

### 3.1. Матрица разделения

| Свойство | Инженерные знания | Авторский продуктовый контекст | Контекст проведения |
|---|---|---|---|
| Пользователь | разработчик платформы | автор, рецензент, администратор команды | ведущий, его уполномоченный помощник |
| Основной объект | репозиторий Cubica | проект игры и его версия | заказчик, мероприятие, покупка и сессия |
| Каноническое содержание | ADR, схемы, код, Markdown | Git-репозиторий проекта и Game Repository | каталог, лицензия, опубликованная версия и Runtime API |
| Постоянная память агента | Markdown-синтез | подтверждённые решения, цели, ограничения и предпочтения | подготовка, договорённости, заметки и разрешённые итоги |
| Хранилище памяти | Git | PostgreSQL авторского контура | PostgreSQL портального/публичного контура |
| Поисковая проекция | локальная SQLite | PostgreSQL FTS + `pgvector` | PostgreSQL FTS + `pgvector` |
| Граница доступа | права репозитория | арендатор + пользователь + проект + роль | арендатор + пользователь + заказчик/мероприятие + роль |
| Срок жизни | история проекта | политика проекта и пользователя | политика мероприятия, лицензии и приватности |
| Автоматический обмен | отсутствует | только явно опубликованные сведения | только опубликованная версия и разрешённые проекции |

### 3.2. Общее ядро, но не общая база

Общими могут быть:

- JSON Schema контрактов контекста;
- разбиение текста на фрагменты;
- адаптер поставщика эмбеддингов;
- полнотекстовое и векторное ранжирование;
- Reciprocal Rank Fusion, или RRF — объединение списков по позициям;
- правила происхождения, времени и замещения;
- оценочные сценарии и телеметрия без сырого текста;
- пользовательские операции «показать, исправить, удалить, не запоминать».

Раздельными обязаны оставаться:

- базы/схемы, роли базы данных и ключи шифрования;
- векторные индексы и кэши;
- пространства идентификаторов;
- политики сроков хранения;
- разрешённые источники;
- служебный доступ и аудит;
- контекстные шлюзы авторского и публичного развёртываний.

Публичные знания о платформе или игре попадают продуктовым агентам только
через управляемую публикацию: публичную документацию, опубликованный пакет
игры или разрешённый контракт API. Внутренние ADR, код платформы, активные
задачи и черновики игр не индексируются в пользовательскую память.

## 4. Сценарии и области видимости

**Область видимости** — серверно вычисленный набор данных, который конкретный
помощник вправе читать или изменять в одном запуске.

### 4.1. Разработчик игры

Контекст строится слоями:

```text
арендатор/команда
  -> пользователь
  -> проект игры
  -> подтверждённая версия проекта
  -> рабочая сессия редактора
  -> текущий диалог
```

Агенту полезны:

- цель игры, аудитория и методический замысел;
- подтверждённые решения автора и их причины;
- открытые вопросы и ограничения проекта;
- пользовательские предпочтения редактора;
- ссылки на конкретные версии authoring-источников;
- результаты прошлых проверок и принятые исправления.

Фактическая структура игры, манифесты, ассеты и плагины читаются из
Game Repository/рабочей сессии. Память не копирует их как второй канон.

### 4.2. Ведущий

Контекст строится иначе:

```text
арендатор/аккаунт
  -> пользователь-ведущий
  -> приобретённая опубликованная игра
  -> проект заказчика
  -> мероприятие
  -> runtime-сессия
  -> текущий диалог
```

Агенту полезны:

- разрешённые предпочтения ведущего;
- цель и формат конкретного мероприятия;
- подтверждённые договорённости с заказчиком;
- опубликованная методика купленной версии игры;
- разрешённая сводка состояния и хода сессии;
- заметки ведущего и подтверждённые итоги разбора.

По умолчанию не создаются долговременные психологические профили участников,
оценки личности или переносимые между заказчиками выводы о людях. Персональные
данные участника не переходят из одной сессии или организации в другую.

Зарезервированная колонка `game_sessions.history` относится к скользящему окну
одной игровой сессии и другому жизненному циклу. Она не может стать
межпроектной памятью пользователя или ведущего.

### 4.3. Один человек в нескольких ролях

Один пользователь может быть и автором, и ведущим, но это не даёт агенту права
автоматически объединять контексты. Личные настройки интерфейса могут жить в
отдельной пользовательской области. Проектные решения, данные заказчика и
сведения о проведении пересекают границу только явной пользовательской
операцией и при совместимой цели обработки.

### 4.4. Игровой агент и помощник игрока

Agent Runtime игры и помощник игрока не получают долговременную память автора
или ведущего по умолчанию. Они видят только контекст, явно разрешённый
реестром помощника, опубликованным пакетом игры, ролью участника и политикой
канала. Память никогда не расширяет список инструментов или Game Intents.

## 5. Источники истины и уровни доверия

Память не должна становиться универсальным источником истины.

| Сведения | Авторитетный источник | Роль продуктового контекста |
|---|---|---|
| Права, роли и арендатор | проверенная сервером личность и политика доступа | сохраняет только ссылку/снимок основания запроса |
| Содержимое проекта игры | Git-репозиторий проекта, долговечная авторская версия | хранит решения, пояснения и ссылки на версию |
| Опубликованная игра | неизменяемый пакет Game Repository | индексирует только разрешённую опубликованную проекцию |
| Покупка и лицензия | портал/каталог | использует актуальный запрос, не запоминает право доступа |
| Игровое состояние | Runtime API и журнал сессии | использует разрешённую проекцию или производную сводку |
| История диалога | хранилище ветки разговора | источник эпизода и происхождения утверждений |
| Подтверждённая договорённость | версия записи продуктового контекста | долговременный контекст с источником и сроком |
| Сводка/эмбеддинг | производный индекс | можно удалить и пересоздать |

При сборке ответа действует порядок:

1. системные политики безопасности и реестр помощника;
2. актуальные доменные данные Cubica;
3. подтверждённые и действующие записи памяти;
4. автоматически извлечённые кандидаты и сводки;
5. фрагменты старых диалогов.

Низший уровень не может переопределить высший. Уверенность модели не
конкурирует с авторитетностью источника: высокая числовая уверенность
извлекателя не делает предположение сильнее явной поправки пользователя или
актуального состояния проекта.

## 6. Логическая архитектура

```text
Редактор / Портал / интерфейс ведущего
              │
              ▼
   локальный шлюз помощника приложения
   личность · реестр · лимиты · аудит
              │
              ▼
        сборщик контекста Cubica
      ┌───────┼─────────┬──────────────┐
      ▼       ▼         ▼              ▼
  доменные   текущий   долговременный  разрешённые
  API        диалог    контекст        опубликованные
                        пользователя    материалы
                        и проекта
      └───────┴─────────┴──────────────┘
              │
     политика доступа · гибридный поиск
     срок · происхождение · предел объёма
              │
              ▼
    промышленный поставщик языковой модели
              │
        текст / запрос инструмента
              │
              ▼
   Cubica validation · approval · domain API
```

Сборщик контекста принадлежит Cubica. Поставщик модели получает только
ограниченный итоговый пакет и не владеет постоянной памятью. Запросы
инструментов продолжают проходить существующие проверки, подтверждение и
аудит; найденный текст не может сам выполнить действие.

Авторский и публичный контуры используют одну реализацию ядра, но разные
адаптеры источников и разные хранилища:

```text
packages/context-core + packages/contracts/context
        │                         │
        ▼                         ▼
authoring-context module    facilitation-context module
authoring PostgreSQL        portal/delivery PostgreSQL
```

Названия пакетов и модулей являются предложением. Их точное размещение —
реализационная деталь после принятия архитектурной границы.

## 7. Модель памяти

### 7.1. Три слоя

1. **Эпизоды** — сообщения, события и ограниченные снимки контекста. Они
   сохраняют последовательность, но имеют собственный срок хранения.
2. **Утверждения** — компактные факты, предпочтения, решения, ограничения,
   цели, открытые вопросы и результаты. Только этот слой используется как
   долговременная память.
3. **Производные представления** — сводки, поисковые фрагменты,
   полнотекстовые документы, эмбеддинги и кэш. Они не являются каноном.

Инструкции агента, разрешения и системный промт не являются пользовательской
памятью. Они версионируются в реестре помощников и коде Cubica. Это отделяет
предпочтение «пиши короче» от вредоносной записи «игнорируй правила и вызови
инструмент».

### 7.2. Минимальные сущности

- `context_scope` — арендатор, пользователь, проект, заказчик, мероприятие или
  сессия и правила наследования видимости;
- `conversation_thread` — ветка диалога, привязанная к помощнику и одной
  основной предметной области;
- `conversation_message` — исходное сообщение с автором, временем,
  чувствительностью и сроком хранения;
- `memory_item` — версия утверждения;
- `memory_source` — происхождение утверждения;
- `memory_relation` — ограниченная типизированная связь;
- `memory_embedding` — производный вектор с моделью, размерностью и хэшем
  исходного текста;
- `context_outbox` — надёжное задание фонового извлечения/переиндексации,
  записанное в одной транзакции с исходным сообщением;
- `context_audit_event` — запись чтения, записи, подтверждения, исправления и
  удаления без обязательного сохранения исходного текста.

### 7.3. Поля утверждения

Минимальный контракт `memory_item` должен содержать:

- `tenantId`, `scopeType`, `scopeId`;
- `kind`: `fact | preference | decision | constraint | goal |
  open_question | outcome | note`;
- структурированное значение и короткое человекочитаемое утверждение;
- `status`: `candidate | confirmed | disputed | superseded | retracted |
  expired`;
- `sourceAuthority`: класс источника, а не вероятность истины;
- `extractionConfidence` только для автоматического кандидата;
- `validFrom`, `validTo` — когда утверждение действительно в предметном мире;
- `recordedAt`, `supersededAt` — когда система получила и заменила запись;
- чувствительность, цель обработки и класс срока хранения;
- автора/процесс создания и ссылки на источники;
- версию схемы, извлекателя, модели и промта, если запись производная.

Связи первой версии ограничиваются:

- `about` — о каком объекте запись;
- `derived_from` — из какого источника получена;
- `supports` — чем подтверждается;
- `contradicts` — с чем расходится;
- `supersedes` — что явно заменяет.

Для этого достаточно обычной таблицы рёбер и SQL. Универсальный граф знаний и
отдельная графовая база не нужны.

## 8. Чтение и сборка контекста

Один запрос агента проходит последовательность:

1. Шлюз устанавливает пользователя, арендатора, роль и помощника из
   аутентифицированной сессии.
2. Реестр помощника задаёт разрешённые типы контекста и инструменты.
3. Политика вычисляет допустимые области до запроса к хранилищу; модель не
   передаёт доверенные `tenantId`, `projectId` или `sessionId`.
4. Актуальные данные запрашиваются у Game Repository, портала или Runtime API.
5. SQL-запрос содержит допустимые области и срок действия, а RLS независимо
   ограничивает видимые строки. Приложение не получает общий список с
   последующей фильтрацией.
6. Полнотекстовые и векторные кандидаты объединяются; результат повторно
   ранжируется по авторитетности, актуальности, подтверждению и разнообразию.
7. Неразрешённые противоречия помечаются или исключаются из автоматического
   вывода.
8. Сборщик ограничивает объём контекста и добавляет идентификаторы источников.
9. Извлечённое содержимое передаётся модели как недоверенные данные, отдельно
   от системных инструкций.
10. Ответ и каждый запрос инструмента проходят существующие проверки Cubica.

Кэш результата обязан включать арендатора, пользователя, помощника, набор
разрешённых областей, версию политики, индекс и модель эмбеддингов. Кэш без
этих измерений создаёт риск межпользовательской утечки.

## 9. Запись, исправление и забывание

### 9.1. Два пути записи

**Немедленный путь** используется для явных команд:

- «запомни это для проекта»;
- «это решение подтверждено»;
- «исправь название заказчика»;
- «забудь эту информацию».

Команда проходит JSON Schema, проверку области, чувствительности и прав.
Подтверждённая запись доступна в следующем запросе.

**Фоновый путь** выполняется после сохранения сообщения:

```text
диалог
  -> классификация и удаление запрещённых данных
  -> транзакционный outbox
  -> извлечение компактных кандидатов
  -> дедупликация и поиск противоречий
  -> candidate
  -> подтверждение пользователем или доменным событием
  -> индексация
```

Фоновая обработка не увеличивает задержку ответа и может сопоставить несколько
сообщений. Но её результат не становится подтверждённым фактом только потому,
что модель уверена.

### 9.2. Подтверждение

Статус `confirmed` появляется по одному из оснований:

- явное подтверждение пользователя с правом на область;
- успешная доменная операция Cubica, на которую ссылается запись;
- проверенное событие авторитетного источника.

Сводка модели, вывод по косвенным признакам или повторение старого сообщения
остаются `candidate`. Для низкорисковой персонализации помощник может
использовать кандидата с явной оговоркой, но не для покупки, публикации,
изменения игры, оценки человека или доступа.

### 9.3. Исправление и удаление

Исправление создаёт новую версию с `supersedes`; старая перестаёт участвовать
в выдаче. При споре обе версии получают `disputed`, и агент просит уточнение,
если без него нельзя безопасно продолжить.

Удаление личного источника каскадно удаляет:

- исходный текст;
- извлечённые фрагменты и сводки;
- утверждения без другого законного источника;
- эмбеддинги;
- кэш.

Можно сохранить минимальный аудит факта удаления без исходного содержимого.
Срок исчезновения из резервных копий задаётся отдельной политикой и должен
быть показан пользователю.

## 10. Поиск и индексы

### 10.1. Первая версия

PostgreSQL используется одновременно для:

- постоянных записей и транзакций;
- полнотекстового поиска через `tsvector`;
- векторного поиска через `pgvector`;
- фильтрации по правам, области, статусу и времени;
- небольшого типизированного графа связей.

Векторный поиск входит в первую версию, потому что пользовательские диалоги
часто перефразируют одно и то же. Он не заменяет точный и полнотекстовый
поиск: идентификаторы проекта, названия, версии и формулировки решений должны
находиться детерминированно.

Рекомендуемый конвейер:

```text
RLS и допустимые области
  -> точные совпадения и актуальные доменные данные
  -> FTS-кандидаты + vector-кандидаты
  -> RRF
  -> статус/время/авторитетность/разнообразие
  -> ограниченный контекст с источниками
```

### 10.2. Индексация без лишней стоимости

- Не создавать долговременный эмбеддинг для каждого сообщения. Индексировать
  подтверждённые записи, полезные кандидаты и фрагменты разрешённых материалов.
- Хэш неизменившегося текста переиспользует вектор той же модели в пределах
  одной границы данных.
- Версия модели, размерность и политика нормализации входят в ключ индекса.
- Смена модели приводит к параллельной пересборке и атомарному переключению,
  а не к смешению векторов.
- Вызовы эмбеддингов проходят общий шлюз LLM с метками арендатора,
  пользователя и функции, но авторский и публичный корпусы не объединяются.

### 10.3. Точный поиск до приближённого индекса

Для малого корпуса проекта точный векторный поиск после ограничения области
проще и полнее, чем HNSW — приближённый графовый индекс ближайших соседей.
`pgvector` может физически применить обычные фильтры после прохода
приближённого индекса. RLS всё равно не отдаёт запрещённые строки приложению,
но такая форма ухудшает полноту и создаёт нежелательную зависимость времени
запроса от общего корпуса. Поэтому:

1. начать с точного поиска внутри уже разрешённой области;
2. измерить задержку и полноту;
3. включать HNSW, итеративное сканирование или разумное секционирование только
   после превышения бюджета;
4. не создавать отдельный раздел на каждого из потенциально тысяч мелких
   арендаторов без измерений.

Отдельная векторная база нужна только если PostgreSQL перестанет выполнять
измеримые требования по объёму, задержке или независимому масштабированию.

## 11. Безопасность, изоляция и приватность

### 11.1. Многоуровневая изоляция

Каждая исходная и производная запись содержит `tenant_id`, область и политику
видимости. PostgreSQL Row-Level Security, или RLS — серверные правила
доступности строк — включается с запретом по умолчанию и
`FORCE ROW LEVEL SECURITY`. Прикладная роль не владеет таблицами и не имеет
`BYPASSRLS`.

RLS дополняет, а не заменяет:

- проверку полномочий в API;
- раздельные схемы/роли авторского и публичного контуров;
- шифрование чувствительных полей;
- отдельные кэши и индексы;
- тесты попыток межарендного доступа;
- аварийный служебный доступ только по процедуре и с аудитом.

### 11.2. Память и внедрение вредоносных инструкций

Текст пользователя, вложения, найденные документы и старые ответы считаются
недоверенными данными. Они:

- не попадают в системный промт как инструкции;
- не изменяют реестр помощника;
- не добавляют инструменты и права;
- не подтверждают собственную достоверность;
- не выполняют вложенные команды;
- проходят проверку источника и чувствительности.

Внедрение инструкций в найденный документ остаётся риском даже при
retrieval-augmented generation, или RAG — дополнении запроса найденными
фрагментами. Поэтому авторизация инструмента всегда выполняется кодом Cubica,
а рискованные действия требуют уже принятого конверта подтверждения.

### 11.3. Приватность

Для каждой категории фиксируются цель и срок:

- сообщения диалога;
- подтверждённые записи;
- записи о заказчике и мероприятии;
- сводки и эмбеддинги;
- аудит;
- резервные копии.

Пользователь должен иметь операции:

- посмотреть, что помнит система;
- исправить запись;
- удалить запись или ветку диалога;
- экспортировать свои данные;
- временно запретить формирование новой долговременной памяти.

Эмбеддинг не считается обезличенным: он наследует доступ, чувствительность и
удаление исходного текста. Сырые сообщения, промты и результаты модели по
умолчанию не записываются в трассы и метрики.

## 12. Встраивание в Cubica

### 12.1. Agent UI и шлюз помощника

Система продолжает принятую цепочку:

```text
помощник
  -> ограниченная проекция контекста Cubica
  -> запрос инструмента Cubica
  -> проверка/подтверждение
  -> доменное изменение
```

Реестр помощника получает:

- разрешённые классы памяти;
- разрешённые области;
- право `none | read | propose | confirm | delete`;
- предел контекста;
- политику чувствительности и срока;
- режим использования кандидатов.

Запись и удаление памяти являются инструментами Cubica, а не прямой записью
модели в базу.

### 12.2. Авторский контур

Авторский модуль получает актуальные проекции от:

- Game Repository и долговечной линии версий проекта;
- editor session API;
- схем и диагностик редактора;
- опубликованного каталога платформенных возможностей.

Он не передаёт в delivery-контур рабочие копии, неопубликованные ассеты,
плагины, внутренние замечания рецензента или историю диалога автора.

### 12.3. Контур ведущего

Модуль проведения получает актуальные проекции от:

- портала, покупки и лицензии;
- опубликованного пакета игры;
- launch-session binding;
- разрешённой проекции Runtime API;
- заметок конкретного мероприятия.

Он не читает авторский репозиторий и не получает server-only игровое состояние.
Заметка ведущего не изменяет игровое состояние без отдельного опубликованного
Game Intent (разрешённого игрового действия) и обычной проверки среды
исполнения.

### 12.4. Контракты и хранение

После принятия решения стабильные структуры должны получить JSON Schema, а
внешние HTTP-операции — OpenAPI. TypeScript-типы создаются из схем, а не
становятся параллельным источником истины.

Логическое размещение:

- `packages/contracts/context` — созданные из схем контракты;
- `packages/context-core` — нейтральная политика сборки, ранжирование и
  происхождение без доступа к конкретной базе;
- отдельные repository-адаптеры авторского и портального контуров;
- PostgreSQL как канон продуктовой памяти;
- LLM gateway как единственный путь к моделям и эмбеддингам.

Начальное совместное развёртывание с существующими серверными процессами
соответствует политике модульного монолита — одной развёртываемой системы с
чёткими внутренними модулями. Отдельный сервис контекста выделяется только при
стабильном API и подтверждённом отдельном профиле нагрузки, безопасности или
доступности.

Текущие локальные идентификаторы портала и доверенных инициаторов игровых
сессий ещё не образуют единую платформенную личность. До промышленного
включения контекста общая идентичность и роли портала, студии и среды
исполнения должны стать авторитетным серверным контрактом; память не может
склеивать пользователей по почте, имени или данным модели.

Существующий Qdrant-конвейер относится к семантическому поиску опубликованных
игр в каталоге. Его коллекция и жизненный цикл не подходят для персональных
диалогов и не переиспользуются как продуктовая память.

## 13. Упрощения без потери качества

| Что не строить сейчас | Чем заменить | Почему функция сохраняется |
|---|---|---|
| Единая база внутренних и пользовательских знаний | Общее ядро с разными хранилищами | Повторно используется код, но не смешиваются доверие и данные |
| Два независимых продукта памяти для автора и ведущего | Два доменных адаптера одного ядра | Разные источники и права без дублирования алгоритмов |
| Отдельный микросервис с первого дня | Модули в текущих контурах со стабильным API | Сохраняется граница будущего выделения без новой эксплуатации |
| Графовая база | Таблицы утверждений и типизированных рёбер | Текущим сценариям нужны короткие связи, а не произвольные обходы графа |
| Отдельная векторная база | PostgreSQL + `pgvector` | Транзакции, права, FTS и векторы остаются в одном управляемом контуре |
| Эмбеддинг каждого сообщения | Отбор утверждений и полезных фрагментов | Смысловой поиск сохраняется, шум, стоимость и утечки уменьшаются |
| Полная копия проекта/сессии в памяти | Ссылки на версию и актуальные доменные API | Нет второго канона и устаревших копий |
| Полное event sourcing памяти | Версии утверждений + происхождение + аудит | Исправления прослеживаются без универсального журнала всех внутренних шагов |
| Автоматическое подтверждение выводов модели | Кандидат + явное основание подтверждения | Персонализация остаётся, ложные выводы не становятся фактом |
| Процедурная память из диалога | Версионируемый реестр помощника | Пользовательские данные не превращаются в полномочия |

Дальнейшее упрощение за счёт одной общей базы, отказа от происхождения,
времени, удаления или серверного вычисления областей уже приводит к потере
безопасности и качества и поэтому не рекомендуется.

## 14. Проверка качества

Оценка должна разделять извлечение, формирование памяти и итоговый ответ.

Обязательные классы сценариев:

- разработчик продолжает проект после нового диалога и получает актуальное
  подтверждённое решение;
- старая версия решения не используется после исправления;
- ведущий получает методику именно купленной опубликованной версии;
- сведения одного заказчика не появляются в контексте другого;
- авторские черновики не видны ведущему, игроку или игровому агенту;
- пользовательский текст с вредоносной инструкцией не меняет права и не
  запускает инструмент;
- отсутствие подтверждённого ответа приводит к честному уточнению, а не
  выдуманному воспоминанию;
- удаление источника очищает утверждения, сводки, векторы и кэш;
- недоступный или отозванный источник не возвращается через старый индекс.

Метрики:

- recall@k (доля запросов, где ожидаемый источник попал в первые `k`
  результатов) по каждому классу роли;
- доля ответов с корректным происхождением;
- доля использованных устаревших/спорных записей;
- точность автоматического формирования кандидатов;
- успешность исправления и забывания;
- ноль подтверждённых межарендных утечек;
- задержка чтения, задержка появления подтверждённой записи и стоимость
  эмбеддингов;
- свежесть индекса и доля запросов с безопасным отказом.

Трассы фиксируют идентификаторы источников, версии политики/индекса/модели,
решение фильтра и причины отказа. Тексты сообщений и памяти включаются только
явной защищённой диагностической политикой.

## 15. Открытые продуктовые решения

До перевода ADR из Draft требуется согласовать:

1. Какие категории данных о заказчиках и участниках ведущий вправе сохранять,
   и какие сроки применяются по умолчанию. Рекомендация: не хранить
   межсессионные профили участников и чувствительные оценки без отдельного
   сценария, основания и согласия.
2. Должна ли личная память пользователя переноситься между ролями автора и
   ведущего. Рекомендация: переносить только явные настройки интерфейса;
   предметные сведения — только явной операцией.
3. Как пользователь просматривает, исправляет, экспортирует и отключает
   память. Рекомендация: единый раздел управления памятью во всех
   пользовательских приложениях с фильтром по области.
4. Какой минимальный срок хранения нужен для исходных диалогов, чтобы
   поддерживать качество и разбор споров, не накапливая данные бессрочно.
5. Разрешён ли внешний поставщик эмбеддингов для каждого класса данных, в
   каком регионе и с какими условиями обработки. До решения должен
   существовать локальный/выключенный режим без скрытой передачи данных.
6. Нужен ли службе поддержки доступ к содержимому памяти. Рекомендация:
   запрет по умолчанию, аварийный ограниченный доступ с причиной, сроком и
   аудитом.

Эти вопросы меняют хранение, границы доверия и права пользователей, поэтому
являются архитектурными, а не деталями реализации.

## 16. Источники и лучшие практики

- [PostgreSQL Row Security Policies](https://www.postgresql.org/docs/current/ddl-rowsecurity.html)
