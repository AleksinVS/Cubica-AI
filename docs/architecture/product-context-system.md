- **Дата:** 2026-07-20
- **Связанное черновое решение:** ADR-095

## 1. Как понята задача и краткий вывод

Задача понята так:

1. проверить, не пытается ли версия 0.2 заранее решить задачи будущего
   масштаба;
2. вернуть полезную историю личных знаний, не скрывая сложность настоящего
   удаления;
3. считать знанием результат всего диалога, включая полезную формулировку
   агента, но отделять авторство текста от полномочия подтвердить его;
4. использовать OKF там, где он помогает переносимости, но не жертвовать ради
   него полезной функцией;
5. технически запретить модели переписывать неизменённые знания;
6. определить, кто, когда и каким способом записывает знания, как переживаются
   сбои и как разрешаются противоречия;
7. проверять основания и явные зависимости за пределами текущего разговора,
   не вводя постоянно работающего агента или отдельную графовую базу.

Краткий вывод:

> **Версия 0.2 действительно была преждевременно сложной. Её можно сократить
> без потери требуемой функции, качества и безопасности.**

Не нужно заранее строить четыре специальных вида пакетов, два разных
Markdown-хранилища, постоянно работающего агента синтеза и собственный
указатель активной ревизии в PostgreSQL.

Для первой версии достаточно:

- одной универсальной сущности «область знаний»;
- одного управляемого сервером Git-хранилища Markdown;
- текущего разговорного агента, который готовит точечную правку в ходе
  диалога;
- ограниченного смыслового обзора по явному запросу или риск-триггеру;
- двух небольших служебных таблиц поверх уже существующих диалогов;
- детерминированного исполнителя, который не использует языковую модель;
- `index.md`, прямых ссылок, явных зависимостей и сканирования малого текущего
  снимка без отдельного поискового индекса.

При этом нельзя убирать:

- серверное разграничение владельцев и прав;
- подтверждение знаний;
- точное происхождение;
- разделение продуктовых и инженерных данных;
- возможность исправить, восстановить и удалить знание;
- приоритет актуальных предметных данных над памятью.

Эти границы защищают корректность и пользователей, а не обслуживают
гипотетический будущий масштаб.

## 2. Что именно было переусложнено

| Версия 0.2 | Уточнённая версия 0.4 | Что сохраняется |
|---|---|---|
| Четыре жёстко заданных вида пакетов | Одна универсальная область знаний с владельцем и политикой | изоляция личного, командного и клиентского содержания |
| Отдельный пакет опубликованной версии | Прямое чтение неизменяемого опубликованного пакета игры | ведущий получает правила нужной версии без второго канона |
| Проектный Git и отдельное управляемое файловое хранилище | Один закрытый управляемый Git для долговременных областей | история, различия, откат и атомарные изменения |
| Указатель активной ревизии в PostgreSQL | Текущий Git `HEAD`, то есть вершина доверенной ветки | согласованный снимок всех страниц и `index.md` |
| Реестр каждой страницы в PostgreSQL | Страницы и их пути читаются из Git | PostgreSQL не дублирует вики |
| Возможный постоянный агент синтеза | Текущий разговорный агент и ограниченный обзор по триггерам | локальные знания формируются в диалоге, широкое влияние проверяется отдельно |
| Постоянное фоновое извлечение | Явная запись и обзор, который только предлагает изменения | нет скрытого профилирования и фонового смыслового дрейфа |
| Отдельная таблица каждой ссылки на источник | Источники незавершённой операции внутри проверяемого JSON | меньше служебной схемы при малом корпусе |
| OKF как строгая обязательная граница | OKF как цель совместимости и экспорта | Markdown остаётся переносимым, расширения Cubica не блокируются |
| Свободная предложенная правка страницы | Точные операции над байтами и единственным фрагментом | неизменённый текст физически не переписывается |

Упрощение не означает один физический архив со всеми данными. Репозиторий
должен совпадать с границей доступа и удаления. Личный канон пользователя,
общий канон проекта и краткоживущие сведения мероприятия могут быть разными
репозиториями, но используют одну реализацию, один контракт и один интерфейс.

## 3. Минимальная архитектура первой версии

```text
Редактор / Портал / интерфейс ведущего
                    │
                    ▼
      координатор хода помощника Cubica
      личность · роль · область · лимиты
          │                       │
          │ чтение                │ точечная правка
          ▼                       ▼
  сборщик контекста       координатор знаний
    │       │       │        │             │
    │       │       │        ▼             ▼
    │       │       │   PostgreSQL    исполнитель Git
    │       │       │   предложения    без языковой модели
    │       │       │                       │
    ▼       ▼       ▼                       ▼
диалог   доменные   разрешённые        Markdown + история
в БД      API       области Git
```

Координатор хода — серверный модуль, который ведёт один запрос пользователя:
собирает разрешённый контекст, вызывает модель, принимает вызовы инструментов
и сохраняет сообщения.

Координатор знаний:

- не является вторым агентом;
- вычисляет допустимую область на сервере;
- проверяет источник, подтверждение и права;
- сохраняет точное предложение;
- применяет подтверждённую правку сразу, если Git доступен;
- оставляет ту же операцию для повторного выполнения после временного сбоя.

Исполнитель Git не рассуждает и не переписывает текст. Он применяет уже
подтверждённую точечную операцию, генерирует оглавления, проверяет результат и
создаёт один коммит.

Отдельный сервис первой версии не нужен. Модули входят в серверный контур,
который уже владеет текущим ходом пользовательского помощника.

## 4. Источники истины и границы систем

### 4.1. Почему инженерная и продуктовая системы остаются раздельными

На верхнем уровне по-прежнему нужны две системы:

1. инженерная система знаний помогает агентам разрабатывать саму платформу;
2. продуктовая система знаний помогает пользователям создавать и проводить
   игры.

Разделение сохраняется не из-за разных форматов, а из-за несовместимых
владельцев, прав, сроков и рисков. Пользовательские диалоги нельзя помещать в
репозиторий исходного кода, а внутренние решения платформы нельзя выдавать
пользовательскому помощнику.

Можно переиспользовать нейтральные схемы Markdown-страниц, генератор
`index.md` и проверки. Нельзя объединять данные, права, историю и поисковые
проекции.

### 4.2. Предметные источники сильнее памяти

| Сведения | Авторитетный источник | Роль вики |
|---|---|---|
| Пользователь, организация и роль | серверная система личности и доступа | не предоставляет право, а только получает вычисленную область |
| Проект игры и его файлы | Game Repository и версия проекта | хранит решения, причины и ссылки на точную версию |
| Опубликованная игра | неизменяемый опубликованный пакет | не копирует правила; может хранить личное пояснение со ссылкой на версию |
| Покупка и лицензия | Portal | не запоминает право, а проверяет его заново |
| Игровое состояние | Runtime API | хранит только разрешённые выводы, а не состояние сессии |
| Полный диалог | PostgreSQL | даёт точное происхождение синтеза |
| Подтверждённое синтезированное знание | текущая ветка Markdown-репозитория | человекочитаемый долговременный канон |
| Неподтверждённая формулировка | запись предложения в PostgreSQL | не участвует в обычном контексте |

Порядок разрешения расхождений:

1. безопасность и актуальные права;
2. актуальное предметное состояние;
3. явная поправка уполномоченного пользователя;
4. подтверждённая текущая страница;
5. спорное знание;
6. старый диалог без подтверждения.

Уверенность модели этот порядок не меняет.

## 5. Универсальная область знаний

**Область знаний** — минимальная единица владения, доступа, срока хранения и
истории. Это не новый микросервис и не специальный формат папки.

Минимальная серверная запись области содержит:

```text
space_id
owner_ref
scope_ref
trust_zone
access_policy_ref
retention_policy_ref
repository_ref
status
```

- `owner_ref` указывает на владельца: пользователя, проект или организацию;
- `scope_ref` указывает, к чему относятся знания;
- `trust_zone` отделяет авторский контур от публичного;
- `access_policy_ref` и `retention_policy_ref` ссылаются на серверные
  политики;
- `repository_ref` указывает на управляемый Git-репозиторий и доверенную
  ветку.

Модель и браузер не назначают эти значения. Сервер выводит их из
аутентифицированной сессии и актуальных предметных прав.

### 5.1. Какие области реально нужны на старте

1. **Личная область пользователя** создаётся при первом подтверждённом
   личном знании. В ней могут находиться знания обо всех играх пользователя,
   если они принадлежат ему и имеют одну политику доступа и хранения.
2. **Область проекта** создаётся, когда знания должны принадлежать команде, а
   не отдельному автору.
3. **Область мероприятия** создаётся только после первого подтверждённого
   долговременного знания. До этого достаточно текущего диалога, Portal,
   Runtime и опубликованной игры.

Это не закрытый перечень типов в контракте. Все три случая являются одной
сущностью с разными владельцами и политиками.

Отдельная область опубликованной версии не создаётся. Правила и методика
читаются прямо из опубликованного пакета. Если автор публикует часть
проектной вики, она становится обычным разрешённым Markdown-ресурсом этого
пакета, а не вторым каноном.

### 5.2. Одна пользовательская поверхность

Пользователь видит один раздел знаний, но сервер собирает его для текущей
задачи:

```text
текущий диалог
  + актуальные данные текущего проекта, игры или мероприятия
  + личная область пользователя
  + одна текущая командная или событийная область, если она разрешена
```

Система не загружает знания обо всех играх при каждом запросе. «Одна система
на уровне пользователя» означает единый интерфейс и единые правила, а не
смешение всех владельцев в одном контексте или репозитории.

## 6. Что хранится в PostgreSQL

PostgreSQL хранит сырые диалоги и надёжный процесс изменения знаний. Он не
хранит вторую каноническую копию подтверждённых страниц.

Предполагается, что таблицы разговоров и сообщений уже существуют. Для
системы знаний достаточно двух новых логических таблиц.

### 6.1. `knowledge_spaces`

Хранит только серверные границы области:

- идентификатор;
- владелец и предметная область;
- доверенная зона;
- ссылки на политики доступа и хранения;
- ссылку на Git-репозиторий;
- состояние области.

Текущий commit SHA не является каноническим полем этой таблицы. Каноническая
текущая ревизия — `HEAD` доверенной Git-ветки.

### 6.2. `knowledge_write_operations`

Название означает «операции записи знаний». Таблица не является историей
содержания: до Git-коммита строка хранит временное намерение и подтверждение,
а после коммита — короткую служебную квитанцию.

Одна запись одновременно представляет одно предложение, его подтверждение и
одну надёжно повторяемую задачу применения. Минимальные группы полей:

- идентичность и область: `operation_id`, `space_id`, создатель, состояние и
  машиночитаемая причина состояния (`status_reason`) и временные отметки;
- неизменяемое предложение: базовый Git commit, набор чтения, отпечатки
  проверок влияния, `patch_payload`, версия его JSON Schema и `patch_hash`;
- происхождение: типизированные `source_refs` на сообщения и доменные события;
- подтверждение: кто, каким способом и когда подтвердил точный `patch_hash`;
- повтор: ключ повторяемости, владелец и срок аренды, число попыток, время
  следующей попытки и закрытый код последней ошибки;
- результат: идентификатор применённого Git-коммита.

`patch_payload` содержит целевые пути, контрольные суммы, точные операции и
новый текст. Эти элементы не дублируются отдельными колонками, если по ним не
нужен серверный запрос. Подтверждение относится к паре
`operation_id + patch_hash`; после подтверждения payload неизменяем, а любая
новая формулировка получает новую операцию.

```text
pending_confirmation
    ├── rejected
    ├── expired
    └── ready
          └── applying
                ├── applied
                ├── conflict
                ├── failed
                └── ready      временная ошибка и повтор
```

Отдельная таблица исходящей очереди первой версии не нужна. Исполнитель
выбирает готовые строки самой `knowledge_write_operations`. PostgreSQL
допускает `SELECT FOR UPDATE SKIP LOCKED` для нескольких потребителей
таблицы-очереди, но такой запрос намеренно даёт не согласованный обзор всех
строк, а только независимый захват доступной работы.

Исполнитель держит блокировку только в короткой транзакции: выбирает строку,
переводит её в `applying`, устанавливает ограниченную по времени аренду и
фиксирует транзакцию. Git-операция выполняется уже без открытой
PostgreSQL-транзакции. После успеха или ошибки состояние обновляется новой
короткой транзакцией; истёкшая аренда разрешает безопасный повтор тех же
подтверждённых байтов.

После `applied` содержательный `patch_payload` удаляется по короткой служебной
политике. В PostgreSQL временно остаются хэш предложения, доказательство
подтверждения и commit SHA, а полный diff читается только из Git. После
`rejected` или `expired` содержательный payload удаляется ещё раньше.

Отдельная `knowledge_change_sources` первой версии не создаётся. Пока операция
не применена, типизированные ссылки находятся в её JSON. После применения
актуальное происхождение находится в метаданных Markdown-страницы. Удаление
источника сканирует малые разрешённые области и незавершённые операции.
Нормализованная таблица может появиться позднее только при измеримой частоте
обратных запросов, требовании внешних ключей или отдельной политике хранения.
Она останется производным индексом, а не каноном происхождения.

Хэши короткого или предсказуемого персонального текста также считаются
чувствительными: при физическом стирании они удаляются либо заменяются
разрешённой бесконтентной отметкой.

### 6.3. Что не нужно хранить в PostgreSQL

Не создаются:

- реестр каждой Markdown-страницы;
- второй текст текущей страницы;
- собственный указатель активного снимка;
- полнотекстовый или векторный индекс;
- универсальный граф знаний;
- отдельная очередь поверх `knowledge_write_operations`;
- отдельная таблица источников без измеримой необходимости;
- отдельный содержательный аудит, дублирующий Git.

RLS, или Row-Level Security — серверные правила доступности отдельных строк,
включается с запретом по умолчанию. Прикладная роль не владеет таблицами и не
имеет `BYPASSRLS`; чувствительные таблицы используют
`FORCE ROW LEVEL SECURITY`. RLS дополняет авторизацию API, а не заменяет её.

## 7. Markdown и Git-история

### 7.1. Уточнённое решение

Отказываться от Git-истории для всех личных знаний нецелесообразно. Польза
истории велика:

- пользователь видит, что именно и почему изменилось;
- ошибочную правку можно откатить;
- различия помогают обнаруживать смысловой дрейф;
- несколько страниц и сгенерированный `index.md` фиксируются одним коммитом;
- Git уже решает версионирование, поэтому Cubica не строит собственный
  механизм снимков.

Предлагаемый вариант — закрытый управляемый Git:

- сервер Cubica является единственным писателем;
- обычные пользователи и агенты не получают `clone`, `fork` и прямой доступ к
  Git-командам;
- агент в обычном контексте видит только текущий снимок;
- история открывается отдельным разрешённым инструментом по явному запросу;
- репозиторий совпадает с границей доступа и удаления либо содержит только
  области с одинаковыми политиками;
- резервные копии и ключ шифрования также соответствуют этой границе.

Личная область одного пользователя может хранить его знания обо всех играх.
Общий канон команды остаётся в проектной области и не копируется в личные
репозитории участников.

### 7.2. Сравнение альтернатив

| Вариант | Польза | Недостаток | Решение для первой версии |
|---|---|---|---|
| Закрытый управляемый Git | полная история, различия, откат, атомарный снимок | сложнее физически удалить старый факт | основной вариант для долговременных областей |
| Только текущие файлы в объектном хранилище | простое удаление отдельного файла | нет полноценного отката; для согласованного снимка нужен свой указатель | не вводить без подтверждённой особой потребности |
| Версионное объектное хранилище | восстановление отдельных объектов | версии не образуют атомарный снимок пакета; различия и манифест придётся строить | не вводить в первой версии |
| Текущий Markdown и журнал патчей в PostgreSQL | можно хранить понятную причину изменения | патчи дублируют личный текст и превращаются во вторую историю | использовать только бесконтентный аудит, не как историю |

Отключить историю всегда проще только на бумаге: для атомарных снимков,
согласования и восстановления всё равно пришлось бы создать собственный
механизм. Для малого Markdown-корпуса управляемый Git является более простым
решением в целом.

### 7.3. «Забыть» и «стереть» — разные операции

**Логически забыть** означает исправить или удалить знание из текущей ветки и
немедленно исключить его из обычного контекста. История остаётся и позволяет
отменить ошибку.

**Физически стереть** означает удалить содержание из всех контролируемых
копий: текущей ветки, старых коммитов, ссылок и журналов Git, кэшей,
временных каталогов, экспортов и резервных копий.

Обычное удаление файла из Git выполняет только первое. Поэтому полное
стирание является отдельной административной операцией:

1. доступ к знанию блокируется немедленно;
2. записи в область временно останавливаются;
3. история переписывается без удаляемого содержания либо создаётся новый
   очищенный репозиторий;
4. все Git-ссылки и журналы старой линии удаляются, затем выполняется сборка
   недостижимых объектов;
5. новый очищенный репозиторий включается как текущий;
6. старый ключ шифрования немедленно блокируется для обычного использования и
   уничтожается в согласованный срок;
7. удаляются кэши, временные копии и производные данные;
8. резервные копии переводятся в режим «не использовать даже при
   восстановлении», становятся недоступными после уничтожения отдельного
   ключа и физически истекают по опубликованному сроку;
9. сохраняется только бесконтентная квитанция операции.

Редкая операция полного стирания может переписать или обнулить историю
затронутой области. Это честный компромисс: история полезна в обычной работе,
а при специальном запросе на стирание приоритет получает приватность.

Для краткоживущего мероприятия выгодно создавать отдельную область лениво и
удалять её целиком по сроку. Это проще и надёжнее, чем вырезать сведения
одного заказчика из общего личного репозитория.

Конкретный срок уничтожения резервных копий и способ управления ключами
должны быть согласованы до обещания пользователю «полного удаления».

## 8. Роль OKF и профиль страницы

### 8.1. OKF не является блокером

Google Open Knowledge Format 0.1 Draft полезен как минимальная договорённость:
иерархические Markdown-файлы, метаданные YAML, `index.md`, обычные ссылки и
переносимый пакет.

Но OKF намеренно не определяет:

- хранение и поиск;
- права и сроки;
- подтверждение;
- конкурентную запись;
- удаление;
- типизированное происхождение.

Следовательно:

1. источником истины для записи является JSON Schema профиля Cubica;
2. совместимость с OKF проверяется и сообщается, но сама по себе не блокирует
   полезную и безопасную возможность;
3. если нужное поле отсутствует в OKF, используется расширение Cubica;
4. если будущая версия OKF вступит в конфликт с безопасностью, происхождением
   или локальностью правки, приоритет имеют правила Cubica;
5. экспорт обязан явно сообщать о поле, которое не понял или не смог
   перенести, а не молча терять его.

Это соответствует духу OKF: стандарт разрешает дополнительные поля и не
предписывает инфраструктуру хранения.

### 8.2. Минимальный профиль текущей страницы

```yaml
---
type: decision
title: Целевая аудитория игры
description: Подтверждённый диапазон участников.
timestamp: 2026-07-18T12:00:00Z
cubica_id: knw_01J...
subject_key: target-audience
source_refs:
  - ref: cubica://dialog/dlg_01J.../message/msg_user_01J...
    use: evidence
  - ref: cubica://dialog/dlg_01J.../message/msg_assistant_01J...
    use: wording
  - ref: cubica://dialog/dlg_01J.../message/msg_user_02J...
    use: confirmation
applies_to:
  - cubica://game-project/prj_01J...
---

# Знание

Игра рассчитана на группы от 12 до 30 участников.
```

Обязательны:

- стандартные `type`, `title`, `description`;
- стабильный `cubica_id`;
- непустые типизированные `source_refs`.

`timestamp` меняется только при смысловом изменении. Необязательный
`subject_key` — проверяемый сервером стабильный ключ предмета внутри одной
области знаний. Совпадение ключа и пересекающегося `applies_to` лишь создаёт
кандидата на дубликат или противоречие, но не разрешает его автоматически.
Отсутствующий `applies_to` означает всю область знаний и пересекается с любой
более узкой применимостью внутри неё.
`applies_to`, время применимости и метки добавляются только при необходимости.

Если утверждение действительно произведено из другого знания и может стать
неверным при его изменении, добавляется явная критическая зависимость:

```yaml
depends_on:
  - ref: cubica://knowledge/knw_audience_01J...
    relation: derives_from
    content_hash: sha256:...
```

В первой версии `content_hash` всегда является SHA-256 точных UTF-8-байтов
всего Markdown-тела страницы после закрывающего разделителя YAML frontmatter,
без нормализации. Это намеренно может пометить зависимость устаревшей при
несвязанном изменении того же тела, зато хэш однозначно пересчитывается без
ещё одного контракта селекторов. Хэш отдельного фрагмента допускается только
в будущей версии схемы вместе с точным машинно проверяемым селектором.
Обычная Markdown-ссылка остаётся навигацией и не создаёт критическую
зависимость. Несовпадение закреплённого хэша делает производную страницу
потенциально устаревшей: она не выдаётся как подтверждённый факт до повторного
обзора, но её байты автоматически не меняются.

Владелец, доступ, доверенная зона и срок принадлежат области знаний и не
дублируются на каждой странице. Страница не может ослабить политику области.

Текущая ветка содержит подтверждённые активные страницы. Для нерешённого
конфликта допускается единственное исключение:

```yaml
state: disputed
```

Предложения, отклонённые и заменённые формулировки не хранятся как активные
страницы: предложение находится в PostgreSQL, а прежняя редакция — в
Git-истории.

### 8.3. `index.md`

`index.md` является коротким оглавлением, а не поисковой базой. Оно:

- генерируется только программой;
- строится детерминированно из метаданных и структуры;
- обновляется только для затронутого каталога и его предков;
- не содержит скрытых недоступных названий;
- может закреплять `okf_version` и версию профиля Cubica в корне области.

Модель никогда не редактирует `index.md`.

Отдельный `log.md` первой версии не нужен: история Git уже даёт точные
различия, а интерфейс может построить понятное описание по метаданным коммита.

## 9. Реплики пользователя и агента как источники

### 9.1. Что было неверно в версии 0.2

Полный запрет использовать ответ агента как источник слишком груб. Итоговое
решение нередко рождается совместно:

1. пользователь описывает цель;
2. агент предлагает точную формулировку или вариант;
3. пользователь принимает, уточняет или исправляет его.

В таком случае ответ агента является источником формулировки, а реплика
пользователя — источником полномочия считать её подтверждённым знанием.

Ошибка возникает не от наличия ответа агента в происхождении, а когда агент
самостоятельно объявляет собственную фразу предпочтением пользователя.

### 9.2. Роли источников

Каждая ссылка указывает способ использования:

| `use` | Смысл |
|---|---|
| `evidence` | содержательное основание утверждения |
| `wording` | формулировка, созданная пользователем или агентом |
| `context` | пояснение, необходимое для правильного толкования |
| `confirmation` | явное принятие пользователем или авторитетным доменным действием |

Роль автора сообщения берётся из PostgreSQL и не доверяется метаданным,
которые предложила модель.

Допустимые случаи:

1. пользователь сам сформулировал и явно просит запомнить — его реплика
   является `evidence` и `confirmation`;
2. агент предложил формулировку, а пользователь её принял — ответ агента
   является `wording`, последующая реплика пользователя —
   `confirmation`;
3. совместный результат опирается на несколько существенных реплик обоих
   участников и завершается явным подтверждением;
4. авторитетное доменное событие подтверждает предметный факт, но не личное
   предпочтение;
5. результат инструмента может быть `evidence` предметного факта, но не
   `confirmation` мнения пользователя.

Недопустимы:

- подтверждённое пользовательское знание только из ответа агента;
- вывод о личности или предпочтениях без содержательной реплики или явного
  принятия;
- системный промт как мнение пользователя;
- неоднозначное «да», если одновременно обсуждались несколько предложений;
- ссылка на удалённое или изменившееся сообщение без повторной проверки.

Минимальный инвариант:

> У долговременного пользовательского знания есть либо содержательная
> реплика пользователя, либо более позднее явное принятие пользователем
> конкретной формулировки агента.

## 10. Когда формируется знание

### 10.1. Основной инициатор — текущий разговорный агент

Для локального знания отдельный агент синтеза не нужен. Текущий разговорный
агент лучше понимает:

- что именно сейчас решалось;
- какую формулировку видел пользователь;
- к какой игре, роли или встрече относится вывод;
- было ли явное подтверждение.

Каждый ход возвращает небольшой структурированный итог:

```text
memory_decision = none | propose | apply_confirmed
```

Это часть контракта хода, а не свободная инструкция. Если знания нет, агент
указывает `none`. Если оно возможно, агент вызывает ограниченный инструмент
подготовки правки.

Контекстное окно агента не считается первичным источником. Перед подготовкой
правки сервер повторно загружает из PostgreSQL точные исходные сообщения по
идентификаторам и контрольным суммам. Для длинного или многосессионного
диалога тот же контур может запустить ограниченный отдельный обзор выбранных
источников; это не постоянно работающий агент и не самостоятельный писатель.

### 10.2. Триггеры

| Событие | Результат |
|---|---|
| «Запомни X», где X сформулировано однозначно | та же реплика может подтвердить точную правку |
| «Исправь: не 30, а 20 участников» | однозначная правка может примениться в том же ходе |
| «Сформулируй и сохрани итог» | агент показывает итог; запись ждёт принятия |
| Агент заметил возможное устойчивое знание | только предложение с видимым текстом |
| Пользователь принял конкретный вариант агента | совместно созданное знание может записаться в этом ходе |
| Авторитетное предметное действие уже подтверждено | разрешённая детерминированная запись может примениться автоматически |
| Изменяется общее предпочтение, сводка или знание с зависимостями | перед подтверждением запускается обзор влияния |
| Пользователь просит «обновить везде» или изменить структуру | создаётся отдельное многостраничное предложение |
| Диалог просто завершился | новых знаний автоматически не создаётся |

Неоднозначное «да» считается подтверждением только если предыдущее сообщение
просило подтвердить ровно одну именованную формулировку. При нескольких
предложениях нужны отдельные действия или явное «сохранить все».

### 10.3. Операционный фоновый исполнитель

Операционный фоновый процесс первой версии:

- повторяет уже подтверждённую точную операцию после сбоя;
- проверяет сроки, ссылки, закреплённые зависимости и структурные метрики;
- создаёт бесконтентную карточку «требуется обзор»;
- не вызывает языковую модель;
- не извлекает новые знания из старых разговоров;
- не переформулирует и не перестраивает страницы.

Этот запрет относится к повтору, сборке мусора и техническому обслуживанию. Он
не запрещает отдельно запущенный смысловой обзор, результатом которого
является только предложение.

### 10.4. Ограниченный смысловой обзор

Смысловой обзор — дополнительный вызов языковой модели только по явному
запросу или риск-триггеру. Он использует тот же защищённый агентный контур и
не требует нового микросервиса или постоянно работающей личности агента.

Обязательные триггеры первой версии:

- исправление, замена или спор для знания широкой области действия;
- изменение общего или ролевого предпочтения пользователя;
- изменение `applies_to` либо наличие `depends_on`;
- создание возможного дубликата с тем же предметным ключом;
- изменение сводной, обзорной или методической страницы;
- запрос «обновить везде», `impact_unknown` от разговорного агента или
  превышение лимита обычной локальной операции;
- предложение оптимизировать структуру вики.

В обзор передаются кандидат на изменение, точные исходные сообщения, целевая
страница, явные зависимости и обратные ссылки, страницы того же предметного
ключа и результаты буквального поиска в текущей разрешённой области. Ответ
имеет ограниченный набор результатов:

```text
no_issue
duplicate
contradiction
scope_overlap
dependent_page_may_be_stale
requires_extended_review
```

`requires_extended_review` — результат обзора, а не дополнительное состояние
операции записи. Если он блокирует текущую правку, операция переходит в уже
определённое состояние `conflict` с
`status_reason: requires_extended_review`.

Обзор может подготовить одну или несколько точечных правок, но не может их
подтвердить или применить. Если число кандидатов превышает лимит, система не
объявляет неполный обзор полным: критическая операция блокируется до
расширенного обзора, а мягкая связь оформляется видимым последующим
предложением.

## 11. Точный конвейер записи

### 11.1. Сквозная последовательность

```text
1. Пользовательское сообщение фиксируется в PostgreSQL.
2. Сервер вычисляет личность, роль и текущую предметную область.
3. Агент получает текущий диалог, актуальные доменные факты и разрешённый
   текущий снимок знаний.
4. Агент отвечает и выбирает memory_decision.
5. При propose/apply_confirmed агент вызывает knowledge.prepare_patch.
6. Сервер сам выбирает space_id, повторно загружает источники и проверяет их.
7. Сервер сканирует метаданные и стабильные идентификаторы текущей разрешённой
   области: тот же `subject_key`, прямые и обратные ссылки, `source_refs`,
   `applies_to` и `depends_on`.
8. Сборщик начинает `read_set` из фактически переданных разговорному агенту
   страниц, сообщений и доменных версий, а также сохраняет отпечатки
   результатов проверок влияния.
9. Риск-триггер при необходимости запускает ограниченный смысловой обзор.
   Перед созданием предложения сервер финализирует `read_set` как объединение
   всех данных, фактически переданных разговорному агенту и дополнительному
   обзору; ни один вызов модели не может исключить элемент из этого набора.
10. Инструмент строит точечное предложение, patch_hash и человекочитаемое
    различие.
11. Политика определяет: подтверждено ли предложение исходной репликой,
   предметным действием или отдельным согласием.
12. Неподтверждённое предложение остаётся pending_confirmation.
13. Подтверждение точного patch_hash переводит предложение в ready.
14. Тот же запрос пытается применить ready-операцию немедленно.
15. Исполнитель на новом HEAD повторно проверяет read_set и запросы влияния.
16. Изменение основания или набора критических зависимостей переводит операцию
    в `conflict`; необходимость расширенного обзора записывается как
    `status_reason: requires_extended_review`. Автоматического пересинтеза нет.
17. Детерминированный исполнитель применяет точные изменения в чистом
    временном рабочем каталоге.
18. Программа пересоздаёт затронутые index.md.
19. JSON Schema и проверки ссылок, происхождения, зависимостей, секретов и
    локальности
    проверяют новый снимок.
20. Git создаёт один коммит и атомарно переключает ветку только при
    совпадении ожидаемого старого commit.
21. PostgreSQL получает applied и commit SHA, затем очищает payload по сроку.
22. Если запрос оборвался, исполнитель повторяет ту же операцию без нового
    синтеза.
```

### 11.2. Инструменты агента

Для первой версии достаточно:

- `knowledge.inspect_impact` — просканировать текущую разрешённую область,
  вернуть явные зависимости и кандидатов для обзора без записи;
- `knowledge.prepare_patch` — прочитать разрешённую цель, проверить
  структурированную точечную операцию и сохранить предложение;
- `knowledge.resolve_proposal` — подтвердить или отклонить уже показанное
  предложение;
- обычных инструментов чтения текущего оглавления и страницы.

Модель не получает:

- прямую файловую запись;
- произвольные Git-команды;
- право выбрать владельца или область;
- возможность редактировать `index.md`;
- возможность пометить собственный ответ пользовательским подтверждением.

Набор чтения формирует сборщик контекста как объединение объектов, реально
выданных всем вызовам модели в рамках одной операции, а не сама модель. Она
может отметить назначение страницы как `target`, `decision_basis`,
`constraint`, `navigation` или `semantic_candidate`, но не может добавить
объект, которого не было в разрешённом контексте, либо удалить фактически
прочитанный объект из итогового набора.

## 12. Запрет переформулирования неизменённых знаний

### 12.1. Почему одного промта недостаточно

Инструкция «не меняй лишнее» не является защитой. Модель может вернуть весь
файл с небольшими стилистическими изменениями, а проверка схемы сочтёт его
валидным. Поэтому локальность обеспечивается формой инструмента и
побайтовой проверкой.

### 12.2. Разрешённые операции

Модель не возвращает полный существующий файл. Она выбирает одну или несколько
точных операций:

- `replace_exact`;
- `insert_before_exact`;
- `insert_after_exact`;
- `delete_exact`;
- `create_file` — только для нового пути.

Каждая операция содержит:

- целевой путь;
- контрольную сумму базового файла;
- точный старый фрагмент или якорь;
- контрольную сумму фрагмента;
- ожидаемое число совпадений, всегда равное одному;
- новый текст;
- причину и ссылки на источники.

Исполнитель:

1. читает исходные байты;
2. проверяет контрольную сумму;
3. находит ровно одно точное совпадение;
4. меняет только этот диапазон;
5. доказывает, что остальные байты целевой страницы не изменились;
6. доказывает, что остальные содержательные страницы сохранили прежние
   Git-объекты.

### 12.3. Ограничения обычной правки

Обычная операция:

- меняет не более одной содержательной страницы;
- может создать одну новую страницу вместо изменения существующей;
- дополнительно меняет только программно сгенерированные оглавления;
- не переименовывает и не сортирует страницы;
- не исправляет пробелы вне целевого фрагмента;
- не меняет окончания строк или кодировку;
- не запускает форматировщик;
- не «улучшает стиль» соседних абзацев.

Разделение или объединение знаний иногда требует нескольких страниц. Такая
операция имеет отдельный вид, явный список затрагиваемых путей и отдельный
предварительный просмотр. Она не становится скрытым исключением из проверки
локальности.

Полное переформатирование или миграция профиля выполняется отдельной
согласованной операцией, а не в ходе обычного синтеза.

### 12.4. Защита от смыслового дрейфа

Запрет решает две разные задачи:

- неизменённые байты не могут получить новое значение случайно;
- пользователь видит только содержательную разницу, которую подтверждает.

Проверка смысла остаётся нужна внутри изменённого фрагмента. Агент показывает
старое и новое утверждение вместе с источниками. Никакая периодическая
обработка не нормализует весь корпус.

## 13. Конкуренция, повторы и восстановление после сбоя

### 13.1. Состояния предложения

```text
pending_confirmation
    ├── rejected
    ├── expired
    └── ready
            └── applying
                    ├── applied
                    ├── failed
                    ├── ready ──> applying
                    └── conflict
```

Повтор после временной ошибки применяет те же байты. Он не вызывает модель и
не изменяет формулировку.

### 13.2. Оптимистическая проверка

Оптимистическая проверка означает: страница не блокируется на всё время
диалога, но правка применяется только к редакции, которую видел пользователь.

Предложение содержит:

- базовый commit;
- набор чтения с назначением и ревизией каждого основания;
- отпечатки детерминированных запросов влияния;
- точные операции;
- уникальный `operation_id` и `patch_hash`.

Если текущий `HEAD` изменился:

- изменилась цель, содержательное основание или ограничение — операция требует
  повторного обзора и Git не меняется;
- изменилось только навигационное оглавление — оно пересоздаётся и само по
  себе не блокирует применение;
- изменилась другая страница — сервер повторяет поиск влияния на новом
  `HEAD`; патч применяется только если набор критических кандидатов и
  использованные основания остались прежними;
- появился новый дубликат, зависимость или конфликт — операция получает
  `conflict`; если требуется более широкий обзор, это отражается кодом
  `status_reason: requires_extended_review`;
- автоматическое смысловое слияние запрещено.

Git ref переключается операцией сравнения и замены: новый commit становится
текущим только если ветка всё ещё указывает на ожидаемый старый commit.

### 13.3. Повторяемость и разрыв PostgreSQL/Git

PostgreSQL и Git не образуют одну общую транзакцию. Дополнительный
PostgreSQL-указатель эту проблему не устраняет, а создаёт второй источник
текущей версии.

Разрыв закрывается повторяемостью:

1. `operation_id` создаётся сервером и записывается в метаданные Git-коммита;
2. один ключ повторяемости создаёт не более одного предложения;
3. перед повторным коммитом исполнитель проверяет, нет ли уже коммита этого
   `operation_id`;
4. если Git успел записать коммит, а PostgreSQL не успел отметить результат,
   согласователь только завершает состояние `applied`;
5. второй смысловой коммит не создаётся.

Управляемые репозитории не проходят squash или rebase, поэтому служебная
идентичность коммита не теряется.

## 14. Исправления и противоречия

### 14.1. Что можно определить надёжно

Детерминированно проверяются:

- повтор стабильного `cubica_id`;
- отсутствующий или изменившийся источник;
- битая ссылка;
- применение патча не к той редакции;
- две активные страницы, которые явно объявляют один и тот же предметный
  ключ;
- несовпадение закреплённого `depends_on.content_hash`;
- изменение результата проверки обратных зависимостей после подготовки
  предложения;
- страница с `state: disputed`, то есть уже зафиксированное нерешённое
  противоречие.

Полное обнаружение смысловых противоречий в свободном тексте первой версии не
обещается. Разговорный агент сравнивает кандидата с выбранными текущими
страницами и может предложить конфликт, но не получает право молча решить его.

### 14.2. Четыре результата сравнения

1. **Дубликат.** Новая страница не создаётся. При необходимости к существующей
   странице точечно добавляется новый источник.
2. **Исправление.** Уполномоченный пользователь явно заменяет значение.
   Меняется та же страница с тем же `cubica_id`; прежняя версия остаётся в
   Git-истории.
3. **Разные области применимости.** Оба утверждения верны для разных игр,
   версий, ролей или периодов. Они получают явные `applies_to`.
4. **Нерешённое противоречие.** Страница получает `state: disputed` и
   показывает обе формулировки с источниками. Она не выдаётся как факт до
   решения уполномоченного пользователя.

### 14.3. Кто разрешает противоречие

Автоматически применимы только заранее разрешённые правила:

- актуальное предметное состояние заменяет вывод, который прямо из него
  произведён;
- однозначная явная поправка владельца заменяет его прежнее утверждение;
- подтверждённое предметное действие фиксирует результат этого действия.

Во всех остальных случаях агент показывает варианты. Он может объяснить
разницу и предложить точечную правку, но не выбирает победителя по
уверенности, новизне сообщения или красоте формулировки.

### 14.4. Изменение знания с зависимостями

Изменение основания не переписывает производные страницы автоматически.
Несовпадение закреплённого содержательного хэша делает их потенциально
устаревшими уже при чтении и создаёт карточку обзора.

Далее выбирается один из двух явных вариантов:

1. если даже кратковременная несогласованность недопустима, агент готовит одно
   атомарное многостраничное предложение и пользователь подтверждает весь
   набор различий;
2. если исходное локальное изменение корректно само по себе, оно применяется,
   а каждая зависимая страница обновляется отдельным подтверждаемым
   предложением.

Связь с областью другого владельца не даёт права изменить её содержимое.
Система может отправить только бесконтентный сигнал об изменении стабильного
предметного идентификатора; обзор и подтверждение выполняются отдельно внутри
каждой разрешённой области.

## 15. Обслуживание и удаление

### 15.1. Что означает GC в этой системе

GC, или сборка мусора, здесь — детерминированное удаление временных,
просроченных и недостижимых технических данных. Это не периодическое
«улучшение» содержания языковой моделью.

После каждой записи выполняются:

- проверка JSON Schema;
- проверка источников и ссылок;
- поиск дублирующихся идентификаторов;
- генерация затронутых `index.md`;
- проверка локальности различия;
- проверка секретоподобного текста.

Периодическое обслуживание выполняет:

- повтор временно не выполненных одобренных операций;
- согласование Git-коммитов и состояний PostgreSQL;
- истечение неподтверждённых предложений;
- проверку версий и контрольных сумм источников;
- поиск обратных зависимостей, устаревших оснований, брошенных страниц и битых
  ссылок;
- измерение размеров страниц и оглавлений, глубины навигации и числа
  кандидатов на дубли;
- создание бесконтентных карточек структурного или смыслового обзора;
- истечение областей мероприятий;
- удаление временных рабочих каталогов;
- выполнение и контроль задач физического стирания;
- очистку кэшей и будущих производных индексов;
- Git GC только после эксклюзивной операции переписывания истории.

### 15.2. Что обслуживание делать не может

Без нового подтверждённого предложения запрещено:

- сокращать или переформулировать страницу;
- переносить знание между владельцами;
- объединять похожие страницы;
- выбирать победителя в споре;
- менять стиль или структуру;
- извлекать новые знания из старого диалога.

Если страница или оглавление превышает бюджет, обслуживание только создаёт
карточку. Разговорный агент либо ограниченный смысловой обзор затем готовит
отдельное видимое предложение.

### 15.3. Удаление источника

Удаление сообщения запускает точный поиск его `source_ref` в малых текущих
областях и в JSON незавершённых `knowledge_write_operations`.

Зависимая страница:

- сохраняется без изменений, если у утверждения есть другое независимое
  достаточное основание;
- получает точечную правку происхождения;
- становится спорной, если основание больше не достаточно;
- удаляется из текущей ветки, если знания без источника не остаётся.

Физическое стирание старого текста из Git-истории выполняется только при
соответствующем виде запроса и по процедуре раздела 7.3.

### 15.4. Оптимизация структуры вики

Структура оптимизируется для постепенного чтения, а не ради единой красивой
таксономии. Сигналами служат измеримые превышения бюджета:

- слишком длинный `index.md` или каталог;
- страница, не помещающаяся в обычный контекст;
- чрезмерная глубина переходов по подиндексам;
- повторяющиеся названия и описания;
- битые ссылки, сироты и устаревшие зависимости;
- устойчивые промахи на оценочных запросах к существующим знаниям.

Точные числовые пороги являются конфигурацией и уточняются по измерениям, а не
закрепляются как неизменная архитектура.

Оптимизация не смешивается с обычным сохранением знания. Она создаёт
отдельное предложение с причиной, полным списком путей и предварительным
различием. Разрешено переместить страницу без изменения её содержательного
blob, точечно обновить ссылки, создать подкаталог и программно пересоздать
`index.md`. Одновременно «улучшать» текст запрещено.

Разделение или объединение страниц является смысловым изменением: обзор
показывает происхождение каждого нового фрагмента, пользователь подтверждает
весь многостраничный набор, а исполнитель доказывает неизменность всего, что
не входило в предложение.

## 16. Чтение контекста и поиск

### 16.1. Последовательность чтения

1. Сервер определяет пользователя, организацию, роль и текущую задачу.
2. Portal, Game Repository или Runtime подтверждают актуальный доступ.
3. Реестр возвращает только личную и текущую предметную область.
4. Из каждой области читается текущий `HEAD`.
5. Агент получает корневой `index.md`, затем только нужные подиндексы и
   страницы.
6. Опубликованная игра читается прямо из закреплённого пакета.
7. Текущий диалог добавляется из PostgreSQL.
8. Предметные факты добавляются из актуальных API.
9. Страницы с устаревшим `depends_on` исключаются из подтверждённых фактов и
   передаются только как кандидаты на обзор.
10. Спорные сведения передаются только с явным предупреждением.
11. Сборщик регистрирует ревизии реально выданных модели оснований.
12. Git-история не входит в обычный контекст.

История читается только по отдельному разрешению и явной задаче, например
«почему решение изменилось». Даже тогда агент получает минимальное различие,
а не весь архив.

### 16.2. Поиск первой версии

Отдельная индексация не вводится. Используются:

- короткие иерархические `index.md`;
- названия, описания, типы и метки;
- стабильные идентификаторы, `applies_to`, `depends_on` и прямые предметные
  ссылки;
- точное чтение известных страниц;
- буквальный поиск только внутри уже разрешённых небольших областей.

Перед записью сервер независимо сканирует метаданные всего малого текущего
снимка и строит временную карту явных зависимостей и обратных ссылок. Это
проверка целостности, а не пользовательский поисковый индекс и не второй
канон. Если сканирование станет измеримо дорогим, его результат можно
кэшировать как полностью пересобираемую проекцию.

Полнотекстовый поиск добавляется после измеримых устойчивых промахов,
превышения бюджета оглавлений или задержки чтения.

Векторный поиск добавляется только если полнотекстовый поиск не покрывает
перефразированные или многоязычные запросы, а гибридный режим доказывает
улучшение на оценочном наборе.

Любая будущая проекция является пересобираемой, изолированной по доступу и
удаляемой вместе с исходной страницей.

## 17. Безопасность и приватность

### 17.1. Область определяется до чтения

Нельзя прочитать общий архив, а затем попросить модель удалить лишнее.
Пользователь, роль, владелец и разрешённые области определяются сервером до
получения названий страниц.

Репозиторий содержит только области с совместимыми:

- владельцем;
- доступом;
- доверенной зоной;
- сроком хранения;
- границей физического удаления.

Если свойства различаются, используется другой репозиторий той же реализации,
а не новая продуктовая система.

### 17.2. Знание не является инструкцией

Markdown, старый диалог и опубликованный документ передаются модели как
недоверенные данные. Они:

- не меняют системный промт;
- не добавляют инструменты;
- не расширяют права;
- не подтверждают сами себя;
- не выполняют вложенные команды.

Все действия проходят разрешённые инструменты Cubica и предметные проверки.

### 17.3. Секреты и данные третьих лиц

Пароли, токены, ключи и похожие строки не попадают даже в предложение
долговременной памяти.

По умолчанию запрещаются:

- скрытые психологические профили участников;
- перенос оценок человека между заказчиками;
- бессрочные сведения мероприятия;
- автоматическое сохранение клиентских данных без явной долговременной цели.

Пользователь должен видеть текущие знания, историю изменений, источники и
предложения, а также уметь исправить, логически забыть, физически стереть и
экспортировать доступное ему содержание.

## 18. Встраивание в Cubica

### 18.1. Логические модули

| Модуль | Ответственность |
|---|---|
| `ConversationStore` | сообщения, порядок, роли, контрольные суммы и сроки в PostgreSQL |
| `KnowledgeSpaceRegistry` | владелец, область, политики и ссылка на Git |
| `KnowledgeCoordinator` | выбор области, проверка происхождения и подтверждения |
| `KnowledgeWriteOperationStore` | временное предложение, подтверждение, аренда исполнителя, повтор и квитанция Git |
| `KnowledgeImpactInspector` | сканирование явных зависимостей, обратных ссылок и регистрация набора чтения |
| `KnowledgeSemanticReview` | ограниченный обзор по риск-триггеру, который создаёт только предложения |
| `GitKnowledgeStore` | чтение `HEAD`, применение точного патча, commit, история и стирание |
| `KnowledgeWriteExecutor` | детерминированные проверки, индекс и атомарное переключение Git ref |
| `ContextAssembler` | текущий диалог, разрешённые знания и актуальные доменные данные |
| `KnowledgeMaintenance` | сроки, ссылки, восстановление, временные данные и удаление |

Это логические модули, а не отдельные развертываемые сервисы.

### 18.2. Контракты

JSON Schema является единственным источником истины для:

- метаданных страницы;
- предложения точечной правки;
- набора чтения и отпечатков проверок влияния;
- ролей происхождения;
- состояний операции записи;
- результата ограниченного смыслового обзора;
- ответа сборщика контекста.

Языковые типы генерируются из схем. Ручные проверки не заменяют схемы, но
дополняют их семантическими правилами: авторизацией, единственностью якоря,
локальностью различия и допустимостью подтверждения.

### 18.3. Развёртывание

Первая версия встраивается в backend, который уже обслуживает ход помощника:

- интерфейс Agent UI показывает предложение и различие;
- адаптер CopilotKit/AG-UI остаётся заменяемой транспортной оболочкой;
- контракты и инструменты знаний принадлежат Cubica;
- поставщик языковой модели не получает постоянное хранилище и прямую Git
  запись;
- риск-триггер может вызвать второй ограниченный read-only обзор в том же
  агентном контуре;
- синхронная попытка применения даёт быстрый результат;
- существующий серверный исполнитель повторяет только не завершённые операции.

Отдельный сервис допускается позже только при измеримой отдельной нагрузке,
границе безопасности, владении командой или требованиях доступности.

Существующий Qdrant для каталога игр не переиспользуется для личных,
проектных или клиентских знаний.

## 19. Проверяемые инварианты и оценка качества

### 19.1. Автоматические инварианты

1. Неподтверждённое предложение не достигает Git.
2. Ответ агента без последующего подтверждения не становится знанием
   пользователя.
3. Совместно созданное знание содержит источник формулировки агента и более
   позднее подтверждение пользователя.
4. Сервер, а не модель, выбирает область и проверяет автора источника.
5. Подтверждение относится к точной паре `operation_id + patch_hash`.
6. Подтверждённый payload неизменяем; новая формулировка создаёт новую
   операцию.
7. Один ключ повторяемости создаёт не более одной операции.
8. Одна операция создаёт не более одного смыслового Git-коммита.
9. `applied` означает, что commit доступен из доверенной ветки.
10. Сборщик, а не модель, регистрирует объединённый набор данных, фактически
    выданных всем модельным вызовам операции.
11. Изменение цели, основания, ограничения или результата критической
    проверки влияния блокирует применение прежнего предложения.
12. Несовпадение `depends_on.content_hash` исключает производную страницу из
    подтверждённых фактов до обзора.
13. Конфликт или неполный критический обзор не меняет Git.
14. Обычная правка меняет одну содержательную страницу и сгенерированные
   оглавления.
15. Все остальные Git-объекты страниц остаются прежними.
16. Модель не редактирует `index.md`.
17. Одинаковое дерево даёт побайтово одинаковые `index.md`.
18. Повтор применённой операции не меняет дерево.
19. Операционный фон не вызывает модель и не создаёт новый смысл.
20. Смысловой обзор может использовать модель, но только создаёт предложение.
21. Структурное обслуживание не перемещает и не переписывает страницы без
    подтверждённого многостраничного различия.
22. Обычный контекст никогда не читает Git-историю.
23. Страница не ослабляет политику области.
24. Удаление источника сканирует текущие страницы и незавершённые операции.
25. Секретоподобная строка блокируется до создания предложения.
26. Недоступная область не раскрывает даже названия страниц.
27. Отказ системы знаний не останавливает детерминированную игру.

### 19.2. Проверки сбоев

Нужны тесты с принудительной остановкой:

- после записи сообщения;
- после создания предложения;
- после подтверждения;
- после захвата задачи;
- после локального коммита;
- после переключения Git ref;
- до отметки `applied`.

Каждый повтор должен закончиться либо одним применённым коммитом, либо явным
конфликтом без частичной правки.

### 19.3. Оценка смыслового качества

Оценочный набор должен включать:

- прямое «запомни»;
- совместно сформулированное решение;
- неоднозначное согласие;
- явное исправление;
- два несовместимых утверждения разных уполномоченных участников;
- разные правила двух версий игры;
- изоляцию автора, ведущего и двух заказчиков;
- попытку сохранить пароль;
- отсутствие ложного предпочтения из текста агента;
- точечное изменение без переформулирования соседнего текста;
- синтез из точных источников длинного или многосессионного диалога;
- изменение страницы, использованной как основание, при неизменной цели;
- появление нового обратного зависимого знания между обзором и применением;
- устаревшее производное знание по `depends_on`;
- структурное перемещение без изменения содержательного blob;
- логическое забывание и физическое стирание;
- восстановление после сбоя.

Измеряются:

- доля полезных знаний, которые были предложены;
- доля ложных предложений;
- доля неподтверждённых знаний, ошибочно попавших в канон;
- число лишних изменённых строк;
- число конфликтов и потерянных обновлений;
- полнота нахождения явных зависимостей и кандидатов на смысловой обзор;
- число устаревших зависимых страниц, выданных как подтверждённый факт;
- полнота нахождения знания через `index.md`;
- число индексов и страниц, прочитанных для обычного ответа;
- время записи и чтения.

## 20. Итоговая проверка упрощения

Убираются без потери функции:

- жёсткие типы пакетов;
- отдельный канон опубликованной вики;
- второй Markdown backend;
- PostgreSQL-указатель активной ревизии;
- реестр страниц;
- отдельная очередь;
- отдельная таблица источников без измеримой необходимости;
- постоянно работающий агент синтеза;
- автоматическое фоновое смысловое извлечение;
- обязательный `log.md`;
- строгая OKF-совместимость как блокирующая проверка;
- FTS, векторы, графовая база и микросервис.

Остаются только границы, потеря которых непосредственно ухудшит качество или
безопасность:

- владелец, доступ и срок на область;
- подтверждение и происхождение;
- управляемая история и процедура стирания;
- точечные патчи;
- набор чтения, явные зависимости и оптимистическая проверка оснований;
- ограниченный смысловой обзор по риск-триггеру;
- детерминированные метрики структуры и подтверждаемые структурные операции;
- актуальные предметные источники;
- детерминированное обслуживание.

Дальнейшее упрощение до одной общей папки, записи полного файла моделью,
assistant-only подтверждения или обычного публично клонируемого Git уже
приведёт к смешению владельцев, смысловому дрейфу либо невозможности честно
описать удаление.

## 21. Решения, требующие согласования

PM утвердил 2026-07-20 следующие внутренние границы будущей реализации:

- одну временную `knowledge_write_operations` вместо содержательной истории в
  PostgreSQL и отдельной таблицы источников первой версии;
- привязку подтверждения к точному хэшу предложения;
- набор чтения и повторную проверку оснований и результатов поиска влияния;
- `depends_on` только для действительно производных знаний;
- разделение операционного фона без модели и ограниченного смыслового обзора,
  который может использовать модель, но только предлагает изменения;
- отдельные подтверждаемые операции оптимизации структуры без попутного
  переформулирования содержания.

Эти уточнения больше не являются открытыми вопросами. Общий статус ADR
остаётся Draft, потому что не утверждены следующие продуктовые и
эксплуатационные границы.

Для перевода ADR из Draft требуется утвердить:

1. универсальную область знаний вместо четырёх специальных видов и прямое
   чтение опубликованной игры без второго пакета;
2. закрытый управляемый Git с полной историей как основной backend личных и
   проектных знаний;
3. различие пользовательских операций «забыть» и «физически стереть», а также
   допустимость переписывания или обнуления истории при полном стирании;
4. сроки хранения областей мероприятий, резервных копий и предельный срок
   завершения физического удаления;
5. политику автоматического применения: явная команда и однозначная поправка
   применяются сразу, остальные знания ждут подтверждения;
6. допустимый перенос личных знаний между ролями разработчика и ведущего;
7. условия передачи пользовательского текста внешнему поставщику языковой
   модели;
8. единую серверную модель личности и ролей.

Выбор поисковой технологии не требуется. Он остаётся зависимым от измеримого
качества навигации.
