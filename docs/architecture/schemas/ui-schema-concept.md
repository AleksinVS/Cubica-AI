## 1. Введение

Этот документ описывает подход к построению пользовательского интерфейса (UI) в играх на платформе Cubica. Мы используем **Hybrid Server-Driven UI (SDUI)** — подход, при котором структура интерфейса полностью определяется в JSON-манифесте игры, а не зашита в код клиента.

**Ключевые особенности:**
*   **LLM-friendly:** Формат оптимизирован для генерации языковыми моделями (компактный, логичный).
*   **Гибридность:** Сочетает низкоуровневую верстку (Atomic) для гибкости и высокоуровневые блоки (Semantic) для сложной логики.
*   **Кроссплатформенность:** Один манифест может быть отрисован как в Web (React), так и в Telegram (Text/Buttons).

## 2. Основные концепции

### 2.1. Atomic Primitives (Атомарный уровень)
Используются для построения *layout* (сетки) и базового оформления. Это аналог HTML/CSS, но упрощенный. Позволяет LLM генерировать интерфейс "по картинке".

*   `container`: Блок-обертка (аналог `div`). Поддерживает отступы, границы, фон.
*   `stack` (`v-stack`, `h-stack`, `z-stack`): Компоненты для позиционирования дочерних элементов.
*   `text`: Текстовый блок с поддержкой стилей.
*   `button`: Интерактивный элемент, вызывающий `Action`.
*   `image`: Изображение (по URL или из ассетов).

### 2.2. Semantic Widgets (Семантический уровень)
"Черные ящики" для сложной игровой логики. LLM не верстает их внутренности, а просто вставляет готовый компонент и настраивает его параметры.

*   Примеры: `widget:inventory`, `widget:stats-bar`, `widget:map`, `widget:dialogue`.
*   Рендерер клиента сам решает, как лучше отобразить этот виджет на конкретной платформе.

### 2.3. Data Binding (Привязка данных)
Свойства компонентов могут быть динамическими. Мы используем синтаксис Mustache `{{...}}` для привязки к состоянию игры (`state`).

Пример:
```json
{
  "type": "text",
  "content": "Health: {{state.hp}} / {{state.maxHp}}",
  "style": { "color": "{{state.hp < 10 ? 'red' : 'green'}}" }
}
```

## 3. Структура UI в манифесте

В актуальной архитектуре UI описывается отдельным **UI‑манифестом** (см. `ui-manifest.schema.json`), который может ссылаться на логический манифест игры. Для простых сценариев (и для обратной совместимости) эта же структура может храниться в секции `ui` внутри игрового манифеста.

Независимо от того, хранится ли UI в отдельном файле или внутри игрового манифеста, структура одинакова: описываются экраны и навигация между ними.

```json
"ui": {
  "theme": { ... }, // Глобальные стили (цвета, шрифты)
  "screens": {
    "main": { // ID экрана
      "type": "screen",
      "root": { ... } // Корневой компонент дерева
    },
    "inventory": { ... }
  },
  "entry_point": "main" // Стартовый экран
}
```

## 4. Компонентная модель (JSON Schema Definition)

Каждый узел дерева UI имеет следующую структуру:

```json
{
  "type": "string",       // Тип компонента (container, text, widget:...)
  "id": "string",         // Опциональный ID для обновлений
  "props": { ... },       // Свойства, специфичные для типа
  "style": { ... },       // Визуальные стили (CSS-like subset)
  "actions": { ... },     // Обработчики событий (onClick, etc.)
  "children": [ ... ]     // Вложенные компоненты (для контейнеров)
}
```

### 4.1. Стили (`style`)
Подмножество CSS свойств, безопасное для парсинга:
*   Layout: `width`, `height`, `padding`, `margin`, `gap`.
*   Appearance: `background`, `color`, `border`, `borderRadius`, `opacity`.
*   Flex: `flex`, `alignItems`, `justifyContent`.

### 4.2. Действия (`actions`)
Связывают UI с Presenter.
*   `onClick`: Отправляет команду (Command Pattern).

```json
"actions": {
  "onClick": {
    "command": "USE_ITEM",
    "payload": { "itemId": "potion_01" }
  }
}
```

## 5. Адаптация под платформы

| Компонент | Web (React) | Telegram (Bot API) |
| :--- | :--- | :--- |
| `container` | `<div>` | Игнорируется или перенос строки |
| `v-stack` | Flex Column | Новая строка для каждого элемента |
| `h-stack` | Flex Row | Элементы через пробел / кнопки в ряд |
| `text` | `<span>` / `<p>` | Текст сообщения (Markdown) |
| `button` | `<button>` | Inline Button |
| `widget:inventory` | Красивая сетка (Grid) | Список кнопок или текстовый список |
| `image` | `<img>` | Отправка фото |

Этот подход позволяет сохранить богатый UI для веба и обеспечить приемлемый UX для Telegram без дублирования логики в манифесте.
