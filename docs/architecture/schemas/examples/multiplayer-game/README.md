## Обзор

Этот пример демонстрирует **мультиплеерную архитектуру** согласно [ADR-011: Архитектура мультиплеера (Free-form с очередью событий)](../../../adrs/011-multiplayer-architecture.md).

**Игра:** Кооперативный survival, где 2-4 игрока с уникальными ролями должны выжить в зомби-апокалипсисе.

### Ключевые концепции

1. **Free-form модель** — игроки могут действовать одновременно (не пошагово).
2. **Очередь событий** — все действия попадают в FIFO-очередь и обрабатываются последовательно.
3. **Версионность состояния** — каждое изменение увеличивает `state_version`.
4. **Broadcast-уведомления** — все игроки получают обновления в реальном времени.

## Конфигурация мультиплеера

### Количество игроков

```json
"config": {
  "players": {
    "min": 2,
    "max": 4
  }
}
```

- `min` — минимум игроков для старта игры.
- `max` — максимум игроков в сессии.

### Секция `multiplayer`

```json
"multiplayer": {
  "mode": "cooperative",
  "turn_order": "free-form",
  "visibility": {
    "world": "shared",
    "player_data": "public_stats_only",
    "secret_data": "hidden"
  },
  "roles": { ... },
  "team_resources": { ... }
}
```

#### Режимы игры (`mode`)

| Режим | Описание |
|-------|----------|
| `cooperative` | Игроки играют в одной команде против системы |
| `competitive` | Игроки соревнуются друг с другом |
| `asymmetric` | Разные цели для разных игроков/команд |

#### Порядок ходов (`turn_order`)

| Режим | Описание |
|-------|----------|
| `free-form` | Игроки действуют одновременно (асинхронно) |
| `round-robin` | Игроки ходят по очереди в фиксированном порядке |
| `initiative` | Порядок определяется характеристикой персонажа |
| `simultaneous` | Все выбирают действия, затем выполняются одновременно |

#### Видимость данных (`visibility`)

```json
"visibility": {
  "world": "shared",           // Все видят общий мир
  "player_data": "public_stats_only",  // Видны только публичные характеристики
  "secret_data": "hidden"      // Секретные данные недоступны
}
```

## Система ролей

Каждый игрок выбирает уникальную роль с особыми способностями.

### Определение роли

```json
"roles": {
  "medic": {
    "name": "Medic",
    "description": "Can heal other players and has extra medical supplies",
    "max_instances": 1,           // Только 1 медик в команде
    "starting_stats": {
      "hp": 80,
      "max_hp": 80,
      "attack": 5,
      "defense": 5
    },
    "abilities": ["heal_ally", "revive"],
    "starting_items": ["medkit", "medkit", "bandage"]
  }
}
```

### Доступные роли в примере

| Роль | Особенности | Лимит |
|------|-------------|-------|
| **Medic** | Лечит союзников, воскрешает | 1 |
| **Soldier** | Высокий урон, дополнительные патроны | 2 |
| **Engineer** | Строит баррикады, чинит | 1 |
| **Scout** | Разведка, поиск ресурсов | 1 |

### Баланс команды

Система ограничивает количество экземпляров каждой роли (`max_instances`), чтобы обеспечить баланс команды.

## Структура состояния

### Разделение состояния

Согласно ADR-011 и схеме манифеста, состояние делится на две части: `public` и `secret`. Внутри `public` размещаются подсекции `world`, `players` и `team`:

```json
"state": {
  "public": {
    "world": { ... },     // Общий мир (виден всем)
    "players": { ... },   // Данные игроков
    "team": { ... }       // Командные ресурсы
  },
  "secret": { ... }       // Скрытые данные (только движок)
}
```

### Секция `world` (общий мир)

```json
"world": {
  "location": "abandoned_mall",
  "areas": {
    "entrance": {
      "name": "Mall Entrance",
      "barricade_level": 0,
      "max_barricade": 3,
      "connected_to": ["food_court", "parking_lot"]
    }
  },
  "zombies": [],
  "current_area": "entrance"
}
```

Все игроки видят:
- Текущую локацию
- Состояние баррикад
- Позиции зомби
- Связи между областями

### Секция `players` (данные игроков)

```json
"players": {
  "player_1": {
    "name": "Alice",
    "role": "medic",
    "hp": 80,
    "max_hp": 80,
    "inventory": ["medkit", "bandage"],
    "position": "entrance",
    "status": "alive"
  },
  "player_2": {
    "name": "Bob",
    "role": "soldier",
    "hp": 100,
    "max_hp": 100,
    "inventory": ["assault_rifle", "ammo"],
    "position": "food_court",
    "status": "alive"
  }
}
```

Каждый игрок видит:
- Свои полные данные
- Публичные данные других (имя, роль, HP, позиция)
- **Не видит** содержимое чужого инвентаря (зависит от настроек)

### Секция `team` (командные ресурсы)

```json
"team": {
  "shared_resources": {
    "ammo": 20,
    "medkits": 4,
    "scrap": 5
  },
  "objectives_completed": [],
  "players_alive": 4
}
```

Командные ресурсы доступны всем игрокам.

### Секция `secret` (скрытые данные)

```json
"secret": {
  "zombie_spawn_queue": [...],
  "hidden_loot": {
    "food_court": ["ammo", "medkit"]
  },
  "boss_wave": 5,
  "escape_conditions": { ... }
}
```

Скрытые данные:
- Очередь спавна зомби
- Нераскрытый лут
- Условия победы
- Внутренние таймеры

## Очередь событий

### Как работает очередь

```
Игрок 1: "attack zombie_1"     → sequence: 1
Игрок 2: "move to food_court"  → sequence: 2
Игрок 1: "use medkit"          → sequence: 3
            ↓
      [Очередь событий]
            ↓
    Обработка по порядку:
    1. attack → применяется урон
    2. move → перемещение
    3. use → лечение
            ↓
    Broadcast обновлений
```

### Статусы событий

| Статус | Описание |
|--------|----------|
| `pending` | Ожидает обработки |
| `processing` | Обрабатывается движком |
| `completed` | Успешно выполнено |
| `failed` | Ошибка при выполнении |

### Версионность состояния

```json
{
  "state_version": 42,
  "last_event_sequence": 156
}
```

- `state_version` — увеличивается при каждом изменении состояния.
- `last_event_sequence` — номер последнего обработанного события.

Клиенты используют эти номера для:
- Определения актуальности полученных данных
- Запроса пропущенных обновлений при переподключении

## Широковещательные уведомления

### Определение событий

```json
"events": {
  "on_player_join": {
    "broadcast": true,
    "message_template": "{{player.name}} joined as {{player.role}}!"
  },
  "on_player_death": {
    "broadcast": true,
    "message_template": "{{player.name}} has fallen!",
    "check_game_over": true
  }
}
```

### Типы событий

| Событие | Действие |
|---------|----------|
| `on_player_join` | Уведомление всем + обновление списка игроков |
| `on_player_leave` | Уведомление + пауза (если критично) |
| `on_player_death` | Уведомление + проверка game over |
| `on_wave_complete` | Уведомление + передышка |
| `on_game_win` | Уведомление + завершение игры |

### Процесс broadcast

```
Событие происходит (например, игрок умер)
              ↓
     Router обрабатывает
              ↓
    Формирует ViewCommand:
    - обновленное состояние
    - state_version
    - сообщение для UI
              ↓
    Web Gateway рассылает:
    - WebSocket → всем подключенным
    - Telegram → в чат сессии (если есть)
              ↓
    Клиенты обновляют UI
```

## Разрешение конфликтов

### Стратегия "Кто первым успел"

Когда два игрока пытаются сделать конфликтующие действия:

```
Ситуация: Один medkit на земле

Игрок 1: "take medkit" → sequence: 1
Игрок 2: "take medkit" → sequence: 2

Обработка:
1. sequence: 1 → Игрок 1 получает medkit
2. sequence: 2 → Ошибка: "Item already taken"
```

### Примеры конфликтов

| Конфликт | Разрешение |
|----------|------------|
| Два игрока берут один предмет | Первый получает, второй — ошибка |
| Атака мертвого зомби | Ошибка: "Target already dead" |
| Лечение мертвого игрока (не медик) | Ошибка: "Cannot revive" |
| Движение в занятую зону | Допускается (несколько игроков в зоне) |

## Связь с ADR

Этот пример реализует следующие решения из [ADR-011](../../../adrs/011-multiplayer-architecture.md):

| Решение ADR | Реализация в примере |
|-------------|---------------------|
| Free-form модель | `turn_order: "free-form"` |
| Структура world/players/secret | Трехуровневое разделение state |
| Роли игроков | Секция `multiplayer.roles` |
| Очередь событий | События с `sequence` и статусами |
| Версионность состояния | `state_version`, `last_event_sequence` |
| Broadcast через View Adapters | Секция `events` с `broadcast: true` |
| Контекст игрока для LLM | `engine.context.include` |
