## 1. Понимание решения

Решение PM (2026-07-19) понято так: раскладка страницы (какой макет —
«верхняя панель» / topbar или «левая боковая панель» / leftsidebar — обрамляет
одни и те же игровые данные) и навигационные стрелки — это **две разные вещи**.

- Стрелки — переходы между игровыми экранами (это отдельно закрывается тем же
  срезом и описано в исполнительской задаче, а не в этом ADR).
- Раскладка — свойство **представления**, которое **разработчик игры** выбирает
  один раз при проектировании UI (design-time). Вариант «игрок сам выбирает
  раскладку» PM сейчас не рассматривает.

Следствие для архитектуры: раскладка объявляется декларативно в UI-манифесте и
не должна храниться в серверном игровом состоянии.

## 2. Контекст

Раньше выбор между двумя вариантами одного экрана (`S1` — topbar и `S1_LEFT` —
leftsidebar в «Антарктиде») делался по полю серверного состояния
`state.public.ui.activeScreen`. Экранный роутер `player-web` сравнивал условие
маршрута `layoutMode` именно с этим полем.

Это создавало два разрыва:

1. **Нарушение границы ADR-054.** Игровое серверное состояние — источник истины
   игровой сессии. Раскладка одних и тех же данных в конкретном канале — это
   presentation, владелец которого по ADR-054 — UI-манифест, а не сервер. Сервер
   не должен знать, как клиент верстает те же данные.
2. **Мёртвый переключатель.** Значение `activeScreen` всегда оставалось
   статичным `"topbar"`: ни один игровой план его не менял, переключателя
   раскладки не было ни на одном экране. Экран `S1_LEFT` был технически описан в
   манифесте, но недостижим (зафиксировано как разрыв достижимости UI-009 в
   родительской задаче выравнивания).

Действующие ADR уже задают направление: ADR-013 (логический манифест отделён от
UI-манифеста), ADR-054 (UI-манифест владеет presentation, сервер — игровым
смыслом), ADR-055 (декоративное/поведенческое оформление объявляется в
UI-манифесте, а не выводится кодом рендерера). Настоящее решение уточняет это для
выбора раскладки экрана.

## 3. Термины

- **Раскладка экрана (layout mode)** — макет канала, обрамляющий игровые данные:
  например `topbar` (метрики сверху) или `leftsidebar` (метрики в левой колонке).
  Это способ показа, а не игровой смысл.
- **Design-time** — момент проектирования игры разработчиком, в отличие от
  runtime (времени игровой сессии конкретного игрока).
- **Вариант экрана** — несколько UI-описаний одного и того же runtime-экрана
  (одного `screenId`) под разные раскладки, например `S1` (topbar) и `S1_LEFT`
  (leftsidebar).

## 4. Принятое решение

1. Раскладка экрана — свойство представления, выбираемое разработчиком игры при
   проектировании UI, и объявляется **декларативно в UI-манифесте**.
2. UI-манифест получает необязательное верхнеуровневое поле
   `default_layout_mode` (проецируется как `defaultLayoutMode`) — design-time
   выбор активной раскладки игры. Значение из того же перечня, что и `layout_mode`
   экрана: `leftsidebar` | `topbar` | `map-first` | `auto`. При отсутствии поля
   платформа считает раскладку `topbar` (историческое поведение).
3. Экранный роутер выбирает вариант экрана, сопоставляя условие маршрута
   `screen_routing[].conditions.layoutMode` с **design-time** `defaultLayoutMode`
   манифеста, а не с серверным состоянием. Итоговую раскладку по-прежнему
   определяет `layout_mode` выбранного экрана (наиболее локальный владелец своей
   раскладки).
4. Серверное игровое состояние **не хранит** раскладку. Поле вида
   `state.public.ui.activeScreen` признано аномалией границы ADR-054 и снимается
   у игр, переведённых на это решение (в рамках среза — у «Антарктиды»).
5. Форсирование альтернативной раскладки для измерения/отладки выполняется
   естественным механизмом: подменой одного значения `default_layout_mode` в
   preview-копии манифеста игры (по штатному preview-каналу), без правки кода и
   серверного состояния.

## 5. Граница ответственности

| Данные или поведение | Владелец | Причина |
| --- | --- | --- |
| Какую раскладку игра показывает по умолчанию | UI-манифест (`default_layout_mode`) | Это design-time presentation-выбор канала. |
| Какая раскладка у конкретного варианта экрана | UI-манифест (`layout_mode` экрана) | Локальное presentation-свойство. |
| Игровые данные экрана (метрики, карточки, шаги) | game-манифест / серверное состояние | Это игровой смысл (ADR-054). |
| Выбор варианта экрана под раскладку | generic renderer/presenter по объявлению манифеста | Исполнение декларации, без game-specific эвристик (ADR-055). |

## 6. Schema and runtime constraints

1. JSON Schema остаётся источником истины (ADR-025). Поле `default_layout_mode`
   добавляется в runtime-схему UI-манифеста и в authoring-схему аддитивно
   (необязательное, тот же enum, что и у `layout_mode`).
2. Authoring compiler переносит поле в опубликованный UI-манифест без ручной
   правки; сгенерированный манифест обязан валидироваться против runtime-схемы.
3. Экранный роутер и presenter не читают серверное `activeScreen` для выбора
   раскладки; при отсутствии `defaultLayoutMode` действует безопасный fallback
   `topbar`.
4. Runtime и player не выводят раскладку из игровых ключей состояния.

## 7. Архитектурные инварианты

- Раскладка экрана объявляется в UI-манифесте, а не хранится в серверном игровом
  состоянии.
- Серверное состояние не содержит поля выбора раскладки (`activeScreen`).
- Выбор варианта экрана под раскладку — декларативный (данные манифеста), а не
  ветка кода рендерера.
- Итоговую раскладку определяет `layout_mode` выбранного экрана; манифестный
  `default_layout_mode` лишь выбирает вариант экрана при маршрутизации.
- JSON Schema остаётся источником истины для нового UI-поля.

## 8. Отклонённые альтернативы

### Переключение раскладки игроком в runtime

Отклонено PM **на текущем этапе** (не закрыто навсегда). Вариант «игрок сам
выбирает раскладку» может быть рассмотрен позже; тогда потребуется отдельное
решение о том, где живёт этот выбор (клиентское view-state конкретного игрока, а
не общее авторитетное игровое состояние), с явным scope и правилами.

### Хранение раскладки в серверном игровом состоянии

Отклонено. Это нарушает границу ADR-054 (сервер начинает знать, как клиент
верстает те же данные) и мешает будущему мультиплееру, где разные игроки одной
сессии могли бы видеть разные раскладки одних и тех же авторитетных данных.

### Оставить `activeScreen` как мёртвое поле

Отклонено. Незадействованное поле состояния — это архитектурный дрейф (правило
управления legacy). Раскладка — design-time решение, поэтому поле снимается, а не
консервируется.

## 9. Последствия

- UI manifest schema и authoring-схема получают необязательное поле
  `default_layout_mode`; контракт `GamePlayerUiContent` — поле `defaultLayoutMode`.
- Экранный роутер `player-web` и Antarctica-плагин перестают читать серверное
  `activeScreen`; выбор варианта экрана идёт от design-time объявления.
- Из authoring «Антарктиды» снимается `state.public.ui.activeScreen` и связанная
  проекция; экран `S1_LEFT` остаётся в манифесте как design-вариант.
- Форсированные измерения leftsidebar выполняются подменой одного значения
  `default_layout_mode` в preview-копии.
- Игры без нового поля продолжают работать по умолчанию как `topbar`.
