## 1. Понимание решения

Решение понято так: действие должно принимать выбранный во время партии объект — например, станцию, новую дорогу или созданный ранее полустанок, — но клиентская строка не должна становиться произвольным путем к состоянию. Манифест заранее объявляет семантику такой ссылки в JSON Schema, а runtime после проверки формы параметров проверяет существование, тип, видимость и допустимую область принадлежности объекта. Только подтвержденная ссылка попадает в обработчик действия.

Решение подтверждено PM 2026-07-11. Оно разрешает реализацию динамического выбора произвольных созданных в сессии объектов. Независимые параметры, ролевые действия, экономика и статические ссылки используют ту же границу доверия, но не обязаны иметь ссылочные параметры.

## 2. Контекст

Принятый контракт параметров действий разрешает плоские строки, числа и логические значения, но намеренно запрещает интерпретировать присланные строки как пути, ключи или идентификаторы объектов. Это защищает runtime от скрытого канала чтения и записи состояния.

Динамическое транспортное поле требует более узкой безопасной возможности:

- ведущий выбирает две существующие станции, причем одна из них могла быть создана как полустанок уже во время партии;
- ведущий выбирает существующую дорогу для размещения полустанка;
- доступные цели меняются после каждого строительства, поэтому полный набор вариантов нельзя скомпилировать в статические действия;
- идентификатор должен ссылаться только на разрешенный класс ресурсов и не давать доступ к произвольному состоянию.

Та же потребность возникает в торгах, обмене, выборе карты, цели способности, единицы техники или объекта на поле. Поэтому решение не должно быть железнодорожным либо игровым исключением.

## 3. Классификация

Механика является общей. В платформу входит безопасная ссылка на объявленный класс ресурсов сессии. Конкретные коллекции, сети, типы объектов и правила допустимой цели принадлежат манифесту игры.

## 4. Предлагаемое решение

### 4.1. Расширить допустимую строковую схему семантической аннотацией

Строковое свойство внутри `paramsSchema` может получить расширение `x-cubica-ref`. Расширение является частью нормативной JSON Schema манифеста и сгенерированных контрактов, а не отдельной TypeScript-схемой.

Минимальная форма:

```json
{
  "type": "string",
  "maxLength": 128,
  "x-cubica-ref": {
    "kind": "object",
    "collection": "transportNodes",
    "network": "main",
    "allowedTypes": ["station", "waypoint"],
    "visibility": "public"
  }
}
```

Первая версия поддерживает два общих вида ссылки:

1. `object` — объект объявленной коллекции и, при необходимости, объявленной сети;
2. `action-resource` — ресурс из ограниченного реестра, который манифест явно проецирует как доступную цель действия.

Состав полей `x-cubica-ref` закрыт схемой через `additionalProperties: false`. Имена коллекций, сети и допустимых типов являются литералами манифеста и проверяются при загрузке контента. Клиент не может прислать или изменить эти ограничения.

### 4.2. Не превращать ссылку в путь

Значение параметра остается непрозрачным идентификатором. Runtime не выполняет по нему JSON Pointer, выражение JsonLogic, имя свойства, имя коллекции или выбор обработчика. Обработчик получает отдельный проверенный дескриптор ссылки либо стабильный идентификатор после завершения всех проверок.

### 4.3. Проецировать только допустимые цели

Проекция доступных действий может содержать ограниченный список допустимых целей и их безопасные отображаемые данные. Эта проекция помогает интерфейсу, но не является доказательством допустимости: runtime повторно проверяет ссылку в момент действия с учетом актуальной версии состояния.

### 4.4. Использовать одинаковый контракт во всех каналах

Обычный web-интерфейс, Phaser-поверхность, мобильный адаптер и ИИ-участник отправляют один и тот же `params` по одной схеме. Канал отображения не получает особого доверия.

## 5. Порядок проверки

Runtime выполняет проверки в фиксированном порядке:

1. проверяет запрос действия и ограничивает размер входа;
2. проверяет `params` стандартным валидатором Ajv против `paramsSchema`;
3. отклоняет опасные имена свойств `__proto__`, `constructor` и `prototype`;
4. находит действие и объявление `x-cubica-ref` из загруженного манифеста;
5. разрешает идентификатор только внутри объявленной коллекции или реестра;
6. проверяет существование, видимость, допустимый тип, принадлежность сети и состояние цели;
7. выполняет обычные ролевые и игровые условия;
8. применяет все эффекты атомарно к одной версии состояния.

Если любая проверка не прошла, состояние и журнал успешных игровых событий не изменяются. Клиент получает типизированную игровую ошибку без внутренних путей и закрытых данных.

## 6. Инварианты безопасности

1. JSON Schema остается единственным источником истины формы и семантики параметра.
2. Только манифест объявляет коллекцию, сеть, типы и видимость ссылки.
3. Клиент передает только значение идентификатора, а не путь, коллекцию, тип или правило поиска.
4. Значение ссылки никогда не исполняется как JSON Pointer, имя свойства, JsonLogic или код.
5. Ссылка разрешается заново для актуальной версии состояния непосредственно перед действием.
6. Закрытый объект нельзя обнаружить по различию подробных сообщений об ошибке.
7. Универсальный runtime не содержит идентификаторов игр и не знает термины «дорога», «станция» или «полустанок» вне общего транспортного обработчика.
8. Нейтральные положительные и отрицательные фикстуры обязательны до подключения конкретной игры.

## 7. Отклоненные альтернативы

- **Передавать JSON Pointer или путь состояния.** Отклонено: это создает управляемый клиентом канал чтения и записи.
- **Считать любую строку идентификатором объекта.** Отклонено: теряется явная семантика, область допустимых целей и возможность строгой проверки.
- **Создавать отдельное действие для каждой пары узлов и каждой дороги.** Отклонено для динамических объектов: созданный во время партии полустанок отсутствует в статическом манифесте, а число комбинаций растет квадратично.
- **Доверять списку целей, показанному интерфейсом.** Отклонено: проекция может устареть или быть подменена.
- **Добавить ручной TypeScript-валидатор только для транспортной сети.** Отклонено: нарушает декларативный источник истины и не дает общей возможности другим играм.

## 8. Последствия

### Положительные

- полное динамическое строительство может использовать узлы и дороги, созданные в текущей партии;
- один контракт покрывает выбор игровых объектов в разных жанрах и каналах;
- ограничения цели видимы в схеме, документации и сгенерированных типах;
- runtime сохраняет контроль над доступом и атомарностью.

### Отрицательные и риски

- загрузчик контента должен выполнять межссылочные проверки, которые нельзя выразить одной структурной JSON Schema;
- проекция доступных целей и runtime-проверка должны использовать одну семантику, иначе интерфейс может показывать устаревший вариант;
- изменение расширяет публичный контракт манифеста и поэтому требует контрактных, межссылочных и негативных тестов во всех каналах доставки.

## 9. Связанные решения

- ADR-025 и ADR-056: JSON Schema как источник истины и генерация контрактов.
- ADR-040: общие декларативные механики вместо обработчиков конкретной игры.
- ADR-061: плоские проверяемые параметры действий; это решение добавляет узкую семантику ссылки, не отменяя запрет произвольных путей.
- ADR-071: проверка роли ведущего.
- ADR-072: динамическая транспортная сеть — первый потребитель ссылок на создаваемые объекты.
- ADR-073: Phaser передает только намерение и не получает дополнительного доверия.
