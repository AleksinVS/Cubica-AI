## 1. Понимание решения

Решение понято так: общий язык Cubica Mechanics должен получить две
нейтральные и комбинируемые возможности.

1. План может добавить один типизированный элемент в объявленное
   множество, не заменяя остальные элементы и не создавая дубль.
2. Несколько однотипных намерений могут использовать один план колоды,
   выбирая конкретную заранее объявленную колоду через ограниченный параметр,
   а не через размножение почти одинаковых планов.

Обе возможности относятся к общему языку игрового движка. Предметный смысл
элемента множества, назначение колоды и момент вызова остаются в пакете
конкретной игры.

## 2. Контекст

Действующий язык умеет целиком заменить множество и массово удалить из него
конкретный элемент. Он не умеет безопасно добавить один элемент с сохранением
остальных. Замена всего массива неприемлема, когда несколько независимых
правил одновременно устанавливают причины одного состояния.

Операции жизненного цикла колоды используют буквальный `deckId`. Это безопасно,
но заставляет публиковать отдельное внешнее намерение для каждого экземпляра
одинакового семейства колод. Игровой серверный обработчик, который сам
выбирает колоду по непроверенной строке, нарушил бы декларативную границу и
модель неизменяемого пакета.

## 3. Решение

### 3.1. Идемпотентное добавление элемента в множество

Закрытое семейство изменений `core.entity.attributes.patch` получает вариант
`set-add`.

- Путь первой версии указывает непосредственно на хранимый, доступный для
  записи атрибут сущности типа `set`.
- Выражение значения проверяется как один элемент объявленного `itemType`.
- Если эквивалентный элемент уже присутствует, операция успешно завершается
  без изменения значения.
- Если нового элемента нет, исполнитель проверяет `maxItems` до записи и
  добавляет элемент в конец канонически сохраняемого множества.
- Сравнение использует ту же структурную эквивалентность, что проверка
  уникальности объявленного множества.
- Проверка и запись входят в общую атомарную транзакцию.

Добавление отдельной предметной операции либо нового флага для каждого
правила запрещено. Если нужно изменить несколько выбранных сущностей, план
комбинирует существующую ограниченную выборку и `core.entities.each` с тем же
нейтральным `set-add`.

### 3.2. Параметризованная ссылка на объявленную колоду

`deck.shuffle` сохраняет буквальный идентификатор и является декларацией
колоды в неизменяемом пакете. Операции, работающие с уже созданной колодой,
принимают либо прежний буквальный идентификатор, либо ограниченное выражение
`value.param`.

В первой версии параметризуются:

- `deck.draw`;
- `deck.extract`;
- `deck.return`;
- `deck.insert`.

Схема каждого связанного внешнего намерения обязана ограничить параметр
строковым `const` или конечным `enum`. Все возможные значения должны:

- удовлетворять общему формату безопасного идентификатора;
- входить в множество буквальных `deckId`, объявленных операциями
  `deck.shuffle` того же неизменяемого пакета.

Неограниченная строка, произвольный путь состояния и имя колоды, которого нет
в пакете, отклоняются при публикации. Runtime повторно вычисляет параметр,
проверяет безопасный идентификатор и работает только с существующей закрытой
колодой сессии. Скрытый порядок, зоны `order`/`discard`/`held`, вместимость,
поток случайности и правила раскрытия данных не меняются.

Ограничение первой версии параметром намерения намеренно сохраняет минимальный
контракт. Другие конечные источники идентификатора могут быть добавлены
отдельной версией после появления реального сценария; это не разрешает
неограниченные вычисляемые строки.

## 4. Безопасность, стоимость и совместимость

- JSON Schema 2020-12 остаётся единственным источником истины формы IR.
- Семантическая проверка доказывает тип элемента множества, доступность записи,
  максимальную вместимость и полный конечный набор допустимых колод до
  публикации.
- Статическая стоимость `set-add` включает верхнюю границу просмотра
  множества; runtime учитывает фактически просмотренные элементы.
- Повторное добавление и ошибка вместимости не оставляют частичного изменения.
- Параметр колоды остаётся недоверенным вводом, даже когда прошёл JSON Schema;
  он не становится путём к состоянию и не раскрывает закрытые зоны.
- Общие возможности доказываются нейтральными фикстурами без терминов
  конкретной игры.
- Изменённые исполняемые модули получают новые точные версии и контрольные
  хеши. Уже закреплённая поддерживаемая сессия не переводится на них
  автоматически и продолжает использовать точный доступный исполнитель по
  ADR-086.
- Пока активен пробел долговечного хранения замороженных корпусов
  `LEGACY-0072`, текущие предрелизные тестовые сессии не считаются
  поддерживаемыми между версиями Mechanics: прежняя сессия остаётся
  неизменяемой и переводится в архивный режим чтения, а проверка продолжается
  в новой сессии. Первую недиспозируемую партию нельзя начинать до закрытия
  этого пробела.
- Ошибки неизвестной колоды, неверного типа, переполнения и недопустимой
  ссылки имеют стабильные коды.

## 5. Отклонённые альтернативы

1. **Заменять всё множество.** Отклонено: независимые элементы могут быть
   потеряны.
2. **Создать предметную операцию изменения конкретного состояния.** Отклонено:
   смысл одной игры попадёт в общий runtime.
3. **Публиковать отдельный план для каждой колоды.** Сохраняет корректность, но
   без необходимости раздувает пакет, каталог действий и стоимость изменений.
4. **Принимать произвольную строку `deckId` только с runtime-проверкой.**
   Отклонено: пакет не доказывает конечную область обращения до запуска.
5. **Параметризовать `deck.shuffle`.** Отклонено для первой версии: буквальная
   инициализация остаётся простым источником истины множества объявленных
   колод.

## 6. Последствия

- Независимые правила могут безопасно накапливать элементы одного множества.
- Семейство однотипных колод использует один опубликованный план без
  game-specific обработчика.
- Пакеты и интерфейсы с большим числом колод становятся меньше, при этом
  границы скрытого состояния не ослабляются.
- Схема, сгенерированные типы, семантическая проверка, точные блокировки
  модулей, runtime и нейтральные тесты должны изменяться согласованно.
