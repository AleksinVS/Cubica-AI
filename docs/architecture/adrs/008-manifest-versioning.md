# ADR-008: Стратегия версионирования манифеста игры

- **Дата**: 2025-11-27
- **Статус**: Accepted
- **Авторы**: AI Agent
- **Компоненты**: Game Engine, Game Repository, Router, SDK

## Контекст
Формат игрового JSON-манифеста (manifest — структурированный файл с описанием игры) развивается: появляются новые секции (`assets`, `actions`, раздельные UI-описания и т.д.).
Нам нужен единый механизм, который:
1. Позволяет однозначно определить версию схемы манифеста.
2. Защищает от запуска контента на несовместимом движке.
3. Даёт возможность эволюции формата без “тихих” поломок.

Каноническая JSON Schema манифеста хранится в `docs/architecture/schemas/game-manifest.schema.json`.

## Альтернативы
- **A. Не версионировать** (только “актуальный формат”).
  - Плюсы: нулевая стоимость сегодня.
  - Минусы: любые изменения схемы становятся скрытым breaking change; невозможно безопасно поддерживать старый контент.
- **B. Версионировать только движок** (engine version) и игнорировать версию схемы.
  - Плюсы: меньше полей в манифесте.
  - Минусы: невозможно различать “старый формат на новом движке”; сложнее делать миграции.
- **C. Версионировать схему манифеста и минимальную версию движка** (выбранный вариант).
  - Плюсы: явная совместимость, понятные ошибки, проще планировать миграции.
  - Минусы: нужна дисциплина обновления полей и проверок при загрузке.

## Решение
Добавить в секцию `meta` манифеста два обязательных поля:
- `schema_version` — версия схемы манифеста в формате `MAJOR.MINOR` (патч-версию для схемы не используем).
- `min_engine_version` — минимальная версия Game Engine (SemVer `MAJOR.MINOR.PATCH`), необходимая для запуска игры.

Пример:
```json
{
  "meta": {
    "id": "com.cubica.antarctica",
    "version": "1.0.0",
    "schema_version": "1.0",
    "min_engine_version": "0.1.0",
    "name": "Antarctica"
  }
}
```

### Правила совместимости
При загрузке манифеста движок выполняет проверки:
1. **Проверка `min_engine_version`**:
   - если `Engine.CurrentVersion < Manifest.meta.min_engine_version` → ошибка “нужна более новая версия движка”.
2. **Проверка `schema_version`**:
   - если MAJOR не совпадает с поддерживаемым MAJOR → ошибка “неизвестный формат манифеста”.
   - если MINOR больше поддерживаемого MINOR → предупреждение/ошибка (политика зависит от фазы; на ранних этапах лучше падать явно).

## Последствия
- Положительные:
  - Явная совместимость контента и движка.
  - Понятные диагностические сообщения при несовместимости.
  - Возможность планировать миграции формата и поддержку legacy.
- Риски и долг:
  - Требуется поддерживать проверки и актуальность схем.
  - При росте числа версий потребуется отдельная стратегия миграций (отдельный документ/задача).

## План внедрения
1. Зафиксировать поля в JSON Schema (`docs/architecture/schemas/game-manifest.schema.json`) — выполнено.
2. Обновить примеры/шаблоны манифестов и инструменты валидации — по мере появления CLI/CI.
3. Добавить проверку совместимости в Game Engine при загрузке манифеста — при реализации Engine.
