## 1. Понимание решения

Решение понято так: каждому значимому экземпляру authoring/game/UI манифеста нужен отдельный авторский промт, который объясняет не только внешний вид, но и содержание, поведение, методический смысл и ожидаемые изменения структуры манифеста.

Этот промт отличается от `generation.prompt` в design artifact. `generation.prompt` описывает воспроизводимость изображения или визуального региона. Новый элементный промт описывает намерение автора для конкретного экземпляра манифеста и должен участвовать в работе редакторского ИИ-агента.

Пользователь вводит свободный текст или редактирует шаблон, агент структурирует текст строго без изменения смысла, пользователь подтверждает подготовленную формулировку, после чего промт сохраняется рядом с элементом и агент создает проверяемое изменение манифеста.

## 2. Контекст

На момент анализа документация и реализация покрывали только части этой идеи:

- ADR-016 и `design-artifact.schema.json` описывают `generation.prompt` как контекст генерации изображения и визуальных регионов.
- До этого решения ADR-030 и authoring v2 schemas вводили `_type`, `_label` и `_semantics`, но не вводили подтвержденный промт экземпляра.
- ADR-036 описывает временный editor prompt flow: `EditorPatchIntent -> EditorChangeSet -> dry-run -> apply/undo/save`.
- `PatchJournalStep` хранит промт операции редактирования, но это история изменения, а не текущее описание элемента.

Из-за этого прежняя документация противоречила исходной идее по смыслу: она называла `generation.prompt` поэлементным промтом, но ограничивала его визуальной генерацией и не связывала с поведением, правилами, методикой и поддержанием authoring-структуры.

## 3. Термины

- **Элементный промт** - сохраненное в манифесте описание намерения автора для конкретного экземпляра: что это за элемент, какие данные он содержит, как ведет себя при взаимодействии и какой методический смысл несет.
- **Шаблон промта** - заготовка промта в прототипе, которую редактор копирует в новый экземпляр при добавлении элемента.
- **Нормализованный промт** - версия пользовательского промта, переписанная агентом для лучшей машинной обработки при строгом сохранении смысла.
- **Подтвержденный промт** - нормализованный промт, который пользователь принял как актуальное описание элемента.
- **Дрейф промта** - состояние, когда сохраненный промт и структурированные поля манифеста описывают разные факты или поведение.

## 4. Решение

Ввести отдельный authoring-only контракт элементного промта:

1. Значимые экземпляры в game authoring и UI authoring получают поле `_prompt`.
2. Переиспользуемые прототипы в `_definitions` получают поле `_promptTemplate`.
3. `_prompt` хранит подтвержденное авторское намерение для конкретного экземпляра.
4. `_promptTemplate` хранит заготовку, которую редактор копирует при создании экземпляра из прототипа.
5. Агент нормализует промт до применения структурных изменений, а пользователь подтверждает нормализованную формулировку.
6. Агент меняет манифест только через bounded `EditorChangeSet` и проверки из ADR-036.
7. Runtime manifests не должны получать `_prompt` по умолчанию. Если runtime или AI-driven gameplay когда-либо потребует доступ к промту, это должно быть отдельным runtime-контрактом и отдельным решением.

Минимальная целевая форма поля:

```json
{
  "_prompt": {
    "status": "confirmed",
    "raw": "Свободный текст пользователя или текст из шаблона.",
    "normalized": "Структурированная формулировка, подтвержденная пользователем.",
    "source": "user",
    "language": "ru",
    "updatedAt": "2026-06-12T00:00:00Z"
  }
}
```

Первый срез реализации закрепил эту форму в `manifest-authoring-common.schema.json`: `_prompt` разрешен на `semanticEntity`, `_promptTemplate` разрешен на `authoringDefinition`, `status` принимает `draft`, `normalized` или `confirmed`, а `normalized` обязателен для `normalized` и `confirmed`. Служебные поля синхронизации не входят в первый срез и должны быть определены отдельным решением.

## 5. Архитектурные инварианты

- `_prompt` не заменяет структурированные поля манифеста. Runtime-поведение остается выраженным через JSON Schema, templates, guards, effects, UI actions and Presenter projections.
- `_prompt` не заменяет `_semantics`. `_semantics` кратко объясняет смысл узла, а `_prompt` хранит рабочее авторское намерение, из которого агент может строить или проверять структуру.
- `_prompt` не заменяет `generation.prompt`. `generation.prompt` остается metadata визуальной генерации для design artifacts.
- `_prompt` не является временным `EditorPatchIntent.prompt`. `EditorPatchIntent.prompt` описывает конкретную операцию редактирования, а `_prompt` описывает текущий экземпляр.
- Авторская нормализация не должна менять смысл. Если агент меняет смысл, пользовательское подтверждение недействительно.
- Любая структурная проверка `_prompt` должна быть описана JSON Schema. Императивная проверка может дополнять схему для графовых свойств, но не заменять schema source of truth.
- Runtime-api и player-web не должны читать authoring-only `_prompt` из generated runtime manifests.

## 6. Альтернативы

- Использовать `generation.prompt`. Отклонено: поле уже занято визуальной воспроизводимостью design artifacts и не описывает игровое или UI-поведение.
- Использовать `_semantics`. Отклонено: `_semantics` слишком короткое и не хранит workflow, raw/normalized/confirmed состояния и пользовательское намерение.
- Хранить промты только в `PatchJournalStep`. Отклонено: журнал описывает прошлые изменения, а не актуальное состояние элемента.
- Генерировать промт только из манифеста. Отклонено как единственный источник: это полезно для исправления дрейфа, но теряет исходную авторскую формулировку и пользовательское подтверждение.
- Добавить промт прямо в generated runtime manifests. Отклонено для базового пути: это загрязняет runtime schema и нарушает authoring/runtime boundary.

## 7. Последствия

Положительные эффекты:

- Пользователь может описывать элемент естественным языком без ручного редактирования всех структурных полей.
- Агент получает устойчивый, проверяемый источник намерения для повторных правок.
- Прототипы становятся полезнее: они задают не только форму данных, но и стартовый шаблон авторского описания.
- UI и gameplay элементы получают единый authoring-паттерн.

Риски и долг:

- Появляется новый тип дрейфа между промтом и структурированными полями.
- Нужна отдельная политика синхронизации при ручных и автоматических изменениях манифеста.
- Первый срез расширяет authoring JSON Schema, правила удаления authoring-only полей в компиляторе и schema readiness редактора. Более глубокий AI prompt flow и diagnostics синхронизации остаются следующим этапом.
- Нужно решить, какие сущности обязаны иметь `_prompt`, а для каких он опционален.

## 9. Открытый вопрос синхронизации

Этот ADR фиксирует необходимость `_prompt`/`_promptTemplate`, но не выбирает механизм поддержания актуальности промта и структурированных полей. Механизм синхронизации принят как обязательная последующая проработка, а не как часть первого среза реализации.

Допустимые направления для следующего решения:

- автогенерация нормализованного промта из структуры манифеста;
- сравнение подтвержденного промта со структурой и диагностика дрейфа;
- двусторонний sync через hash/coverage metadata;
- пользовательское подтверждение предложения по исправлению после автоматического изменения манифеста.
