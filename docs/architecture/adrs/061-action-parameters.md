# ADR-061: Параметры действий манифеста (action parameters)

- **Status**: Accepted (2026-07-06)
- **Date**: 2026-07-06
- **Author**: AI Agent (по заказу владельца проекта)
- **Context**: Runtime platform capabilities / manifest action contract

## 1. Контекст

Текущий контракт отправки действия (`POST /actions`, `DispatchActionInput` в
`packages/contracts/session`) несёт только `sessionId`, `actionId` и необязательный
`playerId` — действие не может принять аргументы от клиента
(`services/runtime-api/src/modules/player-api/requestValidation.ts`).

Аргументы действий нужны сразу нескольким независимым потребителям:

1. **Класс игр «клиентская симуляция реального времени»** (ADR-062): итоги раунда,
   посчитанные клиентской сценой (обработано/верно/пропущено), должны попасть в
   детерминированное действие манифеста как проверяемые числа.
2. **Настольные игры** (ADR-058): ставка аукциона (`auction.bid`), состав торгового
   предложения (`trade.propose`) — суммы и списки выбираются игроком, а не
   зафиксированы в манифесте.
3. **ИИ-оппоненты** (ADR-060 §2.2): проекция `availableActions` уже объявляет поле
   `params-schema` для каждого действия — сама модель параметров при этом нигде не
   определена.

По правилу 10 `CLAUDE.md` возможность классифицирована как **общая**: она нужна
целому классу игр и уже предвосхищена принятой проектной документацией. Реализация
обязана следовать анти-паттерну п.12 `CLAUDE.md`: валидация параметров — только
декларативная, через JSON Schema и Ajv, без императивных проверок под конкретные
действия.

## 2. Решение

### 2.1. Декларация: `paramsSchema` у действия

- Действие манифеста получает необязательное поле `paramsSchema` — JSON Schema
  объекта параметров. Единственный источник истины формы параметров (ADR-025).
- Схема параметров ограничена **плоским объектом**: `type: "object"`, явный
  `additionalProperties: false`, свойства только типов `integer`, `number`,
  `string` (с обязательным `maxLength` ≤ 256; допускается `enum`), `boolean`;
  не более 16 свойств. Вложенные объекты и массивы запрещены мета-схемой
  манифеста — это сознательная граница ADR-024 (сложные структуры выражаются
  объектами состояния, а не параметрами).
- Нормативная форма — `docs/architecture/flow-simulation-platform-design.md` §4.0.

### 2.2. Передача и валидация

- `POST /actions` получает необязательное поле `params` (объект).
- Если у действия нет `paramsSchema`, присутствие `params` — ошибка запроса (400):
  молчаливое проглатывание лишних данных запрещено.
- Если `paramsSchema` есть, runtime валидирует `params` компилятором Ajv по этой
  схеме **до** guards; отсутствующее `params` при объявленной схеме валидируется
  как пустой объект `{}` (обязательность задаёт `required` схемы). Ошибка
  валидации отклоняет действие целиком без изменения состояния — та же атомарная
  семантика, что у guard-отказа.

### 2.3. Потребление: только через контекст JsonLogic

- Контекст данных JsonLogic (нормативно определён в
  `docs/architecture/board-game-platform-design.md` §4.0) дополняется веткой
  `params` — провалидированным объектом параметров текущего действия (пустой
  объект, если параметров нет).
- Параметры доступны в guard-форме `jsonLogic`, в `when`-условиях эффектов и в
  значениях-выражениях эффектов (`metric.set`, `metric.transfer` из ADR-058).
- Параметры **никогда не записываются в состояние автоматически**: любое
  попадание значения параметра в состояние — явный эффект манифеста, прошедший
  guards. Прямой записи «params → state» не существует.

## 3. Отклонённые альтернативы

1. **Свободный `payload` без схемы** (как у `POST /agent-turns`). Отклонено:
   непроверяемые данные от клиента в детерминированном контуре ломают
   auditable-требование ADR-024 и открывают путь неявным game-specific разборам
   payload в runtime (запрещено ADR-040).
2. **Императивные TypeScript-валидаторы параметров.** Отклонено правилом 12
   `CLAUDE.md` и ADR-025: JSON Schema — SSOT, проверяет Ajv.
3. **Автоматическая запись параметров в объявленный путь состояния** (аналог
   `storePath`). Отклонено: скрытый канал записи в обход явных эффектов и guards;
   выбранный путь «параметры видны только выражениям» держит запись строго явной.
4. **Вложенные объекты/массивы в параметрах.** Отклонено для MVP: раздувает
   поверхность валидации и провоцирует перенос игровых структур из состояния в
   параметры; списки выражаются идентификаторами объектов состояния. Расширение —
   только новым ADR при подтверждённой потребности.

## 4. Последствия

### Положительные

- Один общий контракт параметров обслуживает симуляции (ADR-062), настольные
  торги/аукционы (ADR-058 §2.7) и проекцию `availableActions` (ADR-060) без
  повторного проектирования.
- Валидация остаётся декларативной и генерируемой (ADR-056): `paramsSchema`
  попадает в сгенерированные TS-контракты как часть манифеста.

### Отрицательные / риски

- Параметры — первый канал недоверенных пользовательских данных внутри
  детерминированных действий; обязательные меры — лимиты схемы (§2.1), запрет
  `params` без схемы (§2.2) и плоская форма. Плазибилити-проверки значений
  (диапазоны, инварианты сумм) — ответственность guards конкретной игры.
- Ajv-компиляция схем параметров при загрузке манифеста добавляет стоимость;
  компилированные валидаторы кэшируются вместе с манифестом.
