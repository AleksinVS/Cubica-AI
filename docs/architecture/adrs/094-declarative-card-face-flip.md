# ADR-094: Декларативный переворот карточки (front/back face flip)

- **Дата**: 2026-07-20
- **Статус**: Superseded (2026-07-21; самостоятельной архитектурной границы
  не вводит)
- **Авторы**: AI agent (opus), владелец продукта (PM)

- **Связанные решения**: ADR-041 (ортогональные фасеты состояния объекта), ADR-054 (разделение игрового смысла и представления), ADR-055 (декларативное поведение рендерера), ADR-091 (канал игровых стилей)

## 1. Понимание решения

Решение понято так: Cubica предоставляет общую декларативную возможность показать
лицевую и оборотную стороны одной карточки. Игровое состояние и манифест определяют,
когда результат карточки показан, а общий рендерер отображает соответствующую сторону
без знания правил или идентификатора конкретной игры.

Это решение относится к представлению уже публичного содержимого. Оно не вводит новое
игровое действие, не меняет порядок хода и не создаёт механизм раскрытия секретных
данных.

## 2. Контекст

Публичный контракт карточки уже содержит необходимые данные:

- `backText` хранит содержимое оборотной стороны;
- `visualState` передаёт подготовленное Presenter состояние представления, включая
  значение `resolved` для показанного результата.

Presenter — слой, который преобразует игровое состояние в безопасную модель для
интерфейса. Поэтому общий рендерер не должен самостоятельно толковать фасеты объекта,
правила игры или причины изменения состояния. Для двусторонней карточки достаточно
устойчиво определить визуальную семантику уже существующих полей.

## 3. Термины

- **Двусторонняя карточка** — один компонент с лицевой стороной и оборотом, между
  которыми интерфейс переключается по декларативному состоянию представления.
- **Фасет состояния** — независимая ось игрового состояния объекта, например его
  доступность, результат или явно выбранная сторона.
- **Структурные стили** — платформенные правила, необходимые для расположения сторон,
  переключения видимой стороны и безопасного поведения компонента независимо от темы
  конкретной игры.

## 4. Принятое решение

1. Общий компонент карточки поддерживает две стороны, когда задан `backText`:
   лицевая сторона показывает обычное содержимое карточки, оборотная — `backText`.
   Карточка без `backText` остаётся односторонней.
2. Значение `visualState: "resolved"` является декларативным сигналом показать
   оборотную сторону. Новое публичное поле наподобие `flipped` не вводится.
3. Presenter получает `visualState` из игрового состояния через объявленную игрой
   проекцию. Рендерер потребляет только подготовленные `backText` и `visualState` и не
   выводит поведение из внутренних фасетов или правил игры.
4. Фасеты игрового объекта, включая `resolution` и `face`, сохраняют независимую
   семантику. Если игра хранит сторону карточки явно, связь этого фасета с
   `visualState` задаётся в проекции представления, а не в общем компоненте.
5. Структура двух сторон и само переключение принадлежат платформенному компоненту.
   Внешний вид сторон — цвета, фон, типографика и декоративные детали — принадлежит
   стилям конкретной игры, подключаемым через принятый канал игровых стилей.
6. `backText` в этом контракте является публичным содержимым. Его отображение не
   использует серверный механизм раскрытия скрытых данных.
7. Переключение стороны не меняет порядок хода. Продолжение игры остаётся отдельным
   действием, определённым правилами и UI-манифестом игры.
8. Компонент обязан учитывать настройки сокращённого движения пользователя. При
   сокращённом движении смена стороны происходит без пространственной анимации.

## 5. Граница ответственности

| Ответственность | Владелец |
| --- | --- |
| Структура лицевой и оборотной сторон | Платформенный компонент карточки |
| Выбор стороны по `visualState` | Платформенный компонент карточки |
| Игровое состояние и причина его изменения | Правила и состояние игры |
| Проекция игрового состояния в `visualState` | Presenter и декларативное описание игры |
| Содержимое `backText` | Игровой контент |
| Визуальная тема обеих сторон | Игровые стили |
| Порядок хода и действие продолжения | Правила и UI-манифест игры |
| Работа с секретными данными | Серверная граница раскрытия, вне этого решения |

## 6. Архитектурные инварианты

- Общий рендерер не содержит идентификаторов игр и не толкует предметные правила.
- Единственным сигналом рендереру о показе оборота служит публичное состояние
  представления, а не чтение внутреннего игрового состояния.
- Отсутствие `backText` не меняет поведение существующих односторонних карточек.
- Пространственная анимация является необязательным украшением; смысл и управление
  карточкой сохраняются при отключённой или сокращённой анимации.
- Публичное содержимое оборота не смешивается с механизмом серверного раскрытия
  секретной информации.
- Двустороннее представление не изменяет серверное состояние и порядок хода само по
  себе.

## 7. Отклонённые альтернативы

- **Добавить публичное поле `flipped`.** Отклонено: существующие `backText` и
  `visualState` уже полностью выражают требуемое представление, а дополнительное поле
  увеличило бы поверхность совместимости.
- **Толковать фасеты игрового состояния внутри компонента.** Отклонено: это связало бы
  общий рендерер с моделью правил. Преобразование состояния принадлежит Presenter.
- **Раскрывать оборот серверной операцией.** Отклонено для публичного `backText`:
  смена стороны является представлением, а не раскрытием секрета.
- **Реализовать структуру переключения в стилях каждой игры.** Отклонено: общая
  семантика компонента должна работать одинаково во всех играх, а игровые стили
  отвечают только за оформление.
- **Связать смену стороны с переходом хода.** Отклонено: визуальное состояние карточки
  и игровой переход являются независимыми обязанностями.

## 8. Последствия

- Любая игра может использовать двустороннюю карточку через существующий контракт,
  не добавляя игровую ветку в платформенный рендерер.
- Игры сохраняют свободу оформления, не дублируя платформенную структуру компонента.
- Существующие односторонние карточки остаются совместимыми.
- Механика показа оборота остаётся переносимой между играми и отделённой от серверной
  логики, секретных данных и порядка хода.
