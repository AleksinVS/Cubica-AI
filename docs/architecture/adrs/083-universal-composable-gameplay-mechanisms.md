# ADR-083: Универсальные комбинируемые механизмы игрового движка

- **Дата:** 2026-07-14
- **Статус:** Accepted
- **Автор решения:** PM
- **Компоненты:** `docs/architecture/schemas`, `packages/contracts/manifest`, `services/runtime-api`, `packages/editor-engine`, `games/*`
- **Изменяет:** политику обобщения ADR-024 и ADR-040; уточняет ADR-041, ADR-058, ADR-071 и ADR-072
- **Конкретизировано:** ADR-084 задаёт Game Intent как предметный фасад,
  обнаруживаемые mechanics capability packs, типизированный транзакционный
  язык, command idempotency, изолированные игровые runtime-расширения и полный
  переход без постоянного адаптера старых эффектов

## 1. Понимание решения

Решение понято так: игровой движок Cubica должен по возможности предоставлять
универсальные, комбинируемые механизмы, из которых манифест собирает правила
конкретной игры. Платформенный runtime не должен получать отдельный эффект с
предметным смыслом, если то же правило можно выразить общими операциями:
выбрать объекты, проверить условие, изменить
их состояние и записать результат в журнал.

Запрет преждевременного обобщения сохраняет смысл для внутреннего технического
кода платформы — например, для нового служебного слоя, у которого нет текущего
потребителя. Этот запрет не применяется к публичному декларативному языку
игрового движка. Узкий публичный контракт быстро закрепляется в манифестах игр,
после чего его замена требует дорогой миграции каждой опубликованной игры.

## 2. Контекст

Прежняя политика требовала повторного межигрового сценария до появления общего
механизма и допускала ограниченные предметные эффекты. Она снижала стоимость
первой реализации, но переносила её в будущее:

- каждая новая игра добавляла ещё один именованный обработчик;
- одинаковые операции выбора, фильтрации, массового изменения и отложенного
  перехода реализовывались заново внутри разных предметных модулей;
- схемы манифестов начинали отражать названия игровых процессов, а не общую
  вычислительную модель;
- авторы игр не могли комбинировать уже имеющиеся возможности без изменения
  серверного кода;
- последующая универсализация требовала миграции сохранённых манифестов,
  редактора, runtime, тестов и опубликованных пакетов.

Практический сценарий показал общую потребность: выбирать динамически созданные
объекты по их текущему состоянию и массово изменять их при наступлении условия.
Узкие предметные имена скрыли бы эту общую вычислительную структуру.

## 3. Термины

- **Универсальный механизм** — операция с нейтральным контрактом, применимая к
  разным игровым сущностям и правилам без изменения runtime-кода.
- **Композиция** — сборка более сложного правила из нескольких простых операций
  с общей атомарностью, то есть либо применяется весь набор изменений, либо не
  применяется ничего.
- **Селектор** — декларативное правило выбора текущего набора объектов по
  коллекции, типу, состоянию, атрибутам или связям.
- **Ограниченное выполнение** — выполнение с проверяемыми пределами объёма
  данных, числа шагов, вложенности и времени. Ограничение защищает runtime от
  зависания, но не запрещает универсальные циклы или сценарии как класс.
- **Предметная семантика** — название или поведение, имеющее смысл только в
  конкретном игровом процессе и не обозначающее общую вычислительную операцию.

## 4. Решение

1. Публичный язык механик Cubica развивается как универсальный декларативный
   игровой движок, а не как каталог узких предметных обработчиков.
2. Один подтверждённый игровой сценарий достаточен для добавления общего
   механизма, если:
   - его контракт использует нейтральные понятия;
   - он комбинируется с другими операциями;
   - его общность доказана нейтральной фикстурой без терминов исходной игры;
   - безопасность, детерминизм и совместимость описаны явно.
3. Повторный межигровой сценарий больше не является обязательным условием для
   универсализации языка механик.
4. Игровой движок должен по возможности предоставлять общие операции для:
   - выбора и фильтрации объектов;
   - обхода заранее ограниченных коллекций;
   - преобразования, группировки и агрегирования данных;
   - упорядочивания и детерминированного либо воспроизводимо случайного выбора;
   - ветвления, последовательности и повторяемых сценариев;
   - отложенных и запланированных событий;
   - массового и атомарного изменения состояния;
   - работы с общими графами, метриками, объектами, событиями и журналом.
5. Предметные правила остаются данными конкретной игры и собираются из этих
   операций в манифесте или в компилируемых шаблонах авторского слоя.
6. JSON Schema остаётся единственным источником истины для формы языка.
   Реализация не может подменить контракт ручными проверками только на одном
   языке программирования.
7. Универсальность не отменяет безопасность: runtime исполняет только
   объявленное подмножество операций над разрешённым состоянием с лимитами,
   атомарностью, воспроизводимостью и аудитом.

## 5. Архитектурные границы

Архитектура разделяется на четыре уровня:

1. **Универсальные примитивы движка.** Селекторы, выражения, преобразования,
   транзакции, события, планирование, графовые и числовые операции. Их имена и
   контракты не содержат понятий конкретного жанра или игры.
2. **Библиотека декларативных шаблонов.** Удобные авторские конструкции могут
   называться предметно, но при компиляции раскрываются в универсальные
   примитивы и не требуют отдельного обработчика runtime.
3. **Манифест игры.** Предметные значения, фазы, типы, условия и тексты остаются
   содержимым игры.
4. **Изолированное runtime-расширение игры.** Исполняемый код разрешён, если
   требуемую предметную функциональность нельзя разумно выразить первыми тремя
   уровнями. Он остаётся внутри bundle/namespace игры, исполняется вне процесса
   платформы с минимальными полномочиями и возвращает только проверяемое по
   JSON Schema значение или ограниченный change set, который применяет
   платформа.

Специализированный алгоритм допустим в платформе, если его вход и результат
имеют общую математическую или вычислительную семантику. Поиск пути по графу,
пересечение геометрии и детерминированная сортировка являются общими алгоритмами,
а предметные сценарии компонуются из таких алгоритмов и общих изменений состояния.

## 6. Требования к универсальному языку механик

Язык может включать селекторы, ограниченные циклы, ветвления, сценарии и
агрегации. Для них обязательны следующие свойства:

- схема заранее определяет допустимые операции и форму аргументов;
- область чтения и записи явно ограничена манифестом и правами действия;
- коллекции и число шагов имеют проверяемые пределы;
- порядок обхода канонический, а случайный выбор использует состояние
  воспроизводимой случайности сессии;
- перед записью runtime проверяет весь набор целей и изменений;
- связанная последовательность применяется атомарно;
- ошибки не оставляют частично изменённое состояние;
- журнал объясняет, какое правило выполнилось и сколько объектов затронуто, не
  раскрывая секретные данные;
- версия языка и отдельных алгоритмов позволяет воспроизводить сохранённые
  партии;
- произвольный JavaScript, нативный код, сеть, файловая система, системное время
  и переменные окружения не становятся неявной частью игрового действия.

Последнее правило ограничивает обычный декларативный план, но не запрещает
изолированное игровое расширение. Для него права, вход, выход, ресурсы и область
записи объявляются отдельно; прямой доступ к сессии и процессу платформы
запрещён.

Схемы должны переиспользовать общие определения через `$defs` и `$ref`, а
варианты операций — иметь явный признак типа и закрытую форму. Это позволяет
расширять словарь языка без неявных полей и сохранять межъязыковую проверку
контрактов.

## 7. Правило проектирования новой механики

При появлении игрового правила агент обязан:

1. Разложить его на нейтральные операции над данными и состоянием.
2. Проверить, какие универсальные примитивы уже существуют.
3. Если примитива не хватает, расширить общий язык минимально необходимой, но
   универсальной операцией. «Минимально» ограничивает объём реализуемых сейчас
   функций, а не область применимости контракта.
4. Если новая операция была бы осмыслена только в одной игре, оставить правило
   в манифесте или, при недостаточной выразительности IR, использовать
   изолированное game-scoped runtime-расширение.
5. Оставить игровые значения и последовательность предметных шагов в манифесте
   или декларативном шаблоне игры.
6. Доказать новую общую возможность текущим игровым срезом и нейтральной
   фикстурой; игровое расширение вместо этого доказывает изоляцию, объявленные
   полномочия и безопасность платформы.
7. Проверить совместимость старых манифестов и определить миграцию, если форма
   существующего контракта меняется.

Узкая предметная операция в общем runtime не допускается. Если универсальная
композиция не может обеспечить обязательную корректность, безопасность или
производительность, предметная реализация размещается в изолированном модуле
игры. Документированным временным долгом считается только её выполнение без
целевой изоляции, но не сам факт существования game-scoped кода.

Выбор уже предусмотренного изолированного модуля является реализационным
решением и не требует отдельного согласования для каждой игры. Новое
архитектурное решение требуется, если модулю нужны дополнительные права,
ослабление изоляции, другой источник истины или существенно иная
эксплуатационная модель.

## 8. Конечная граница совместимости

Публичный контракт `deterministic.effects[]`, его исполнитель и предметные
точки входа не входят в целевую архитектуру. После переключения поддерживается
только Game Intent → Cubica Mechanics IR; постоянный runtime-адаптер старой
формы запрещён.

Сохранённая сессия исполняется только на закреплённой совместимой версии либо
явно архивируется или проходит проверяемую миграцию. Временные средства
сравнения и преобразования не становятся частью production runtime.

## 9. Отклонённые альтернативы

- **Ждать второго межигрового случая до проектирования общего контракта.**
  Отклонено: публичная узкая операция успевает закрепиться в играх и делает
  последующую миграцию дороже.
- **Добавлять отдельный runtime-эффект для каждого предметного глагола.**
  Отклонено: платформа превращается в несогласованный каталог обработчиков, а
  новые игры требуют серверной разработки вместо композиции манифеста.
- **Разрешить игровой код внутри процесса платформы либо без декларативной
  схемы, лимитов и области полномочий.** Отклонено: разрешённый game-scoped код
  исполняется только через изолированный контракт ADR-084.
- **Сохранить старый runtime-адаптер после миграции.** Отклонено: он удваивает
  публичный язык, тестовую матрицу и поверхность безопасности и позволяет новым
  играм продолжать зависеть от устаревшей формы.

## 10. Последствия

Положительные последствия:

- следующая игра чаще собирается из уже имеющихся возможностей без изменения
  runtime;
- правила разных жанров используют один проверяемый язык;
- редактор может предлагать общие операции и анализировать их семантику;
- снижается риск массовой миграции игр после накопления узких контрактов;
- тесты универсальных примитивов дают защиту сразу нескольким играм.

Стоимость и риски:

- язык механик, его валидатор и редактор становятся сложнее;
- нужны лимиты выполнения, строгая типизация ссылок и версионирование;
- слишком общий примитив без понятной авторской формы может ухудшить удобство,
  поэтому нужны компилируемые шаблоны и диагностика;
- несовместимый переход требует закрепить неизменяемую версию программы за
  сессией либо явно архивировать или мигрировать сессию;
- целевой runtime не несёт стоимость и риски постоянной поддержки двух языков.
