## 1. Понимание решения

Решение понято так: Cubica получает проектную файловую вики, которая хранит
смысловой синтез для агентов и связывает действующие решения, контракты, код и
карту репозитория. Markdown остаётся человекочитаемым каноном, а граф и
поисковые индексы являются полностью производными локальными проекциями.

Полнотекстовый поиск используется штатно. Векторная возможность реализуется и
доступна для явных запросов и сравнительной оценки сразу, но становится
режимом обычной выдачи по умолчанию только после измеримого подтверждения.
Система не заменяет действующий генератор структуры, не зависит от Git-трейлеров
акторов и не создаёт отдельную постоянную очередь замечаний.
Минимальная AST-проекция модульных импортов входит в первую версию как
отдельный производный слой фактов о текущем коде.

## 2. Контекст

Архитектурный контекст Cubica распределён между ADR, JSON Schema, OpenAPI,
кодом, задачами и `PROJECT_STRUCTURE.yaml`. Это разделение правильно для
источников истины, но агентам нужен короткий смысловой слой, который связывает
их без копирования больших документов.

Ранее лежавший в архитектурном каталоге общий черновик гибридной памяти
описывал другую систему, пользовательские данные и чужую нумерацию решений.
Он был зарегистрирован как LEGACY-0033 и не мог стать канонической
архитектурой Cubica без адаптации.

## 3. Принятое решение

### 3.1. Канон и источники истины

1. Канон синтеза хранится в `docs/knowledge/` как Markdown с YAML-frontmatter.
2. Вики не становится универсальным источником истины:
   - ADR владеют архитектурными решениями;
   - JSON Schema и OpenAPI владеют исполняемыми контрактами;
   - код и манифесты владеют фактическим поведением;
   - `TSK-*` и `NEXT_STEPS.md` владеют исполнением и приоритетами;
   - `PROJECT_STRUCTURE.yaml` владеет машинной картой репозитория.
3. Текущий архитектурный обзор переносится в вики отдельным этапом с
   обновлением ссылок. ADR сохраняются как самостоятельные первичные решения.
4. SQLite, полнотекстовый индекс, векторы и производные графы не являются
   каноном и должны полностью восстанавливаться из версионируемых источников.

### 3.2. Декларативные контракты

Страница, конфигурация и проверочный поисковый случай имеют JSON Schema как
единственный источник структурного контракта. YAML разбирается `js-yaml`, после
чего Ajv выполняет схему. Ручные Node.js-проверки не могут подменять JSON
Schema.

Типы концептов и рёбер заданы перечислениями схем. Отдельная машина
`proposed/approved` для словарей не вводится.

### 3.3. Поиск

1. Штатный поиск первой версии — SQLite FTS5/BM25 по разделам Markdown.
2. В первой реализации также создаются контракт поставщика эмбеддингов,
   векторное хранилище, построение индекса, косинусное ранжирование, гибридное
   RRF-слияние и тесты на фиксированных векторах.
3. Начальный режим конфигурации — `fts`; отсутствие поставщика или полного
   векторного индекса не ухудшает штатный путь.
4. Готовый векторный индекс используется в сравнительном `kb eval`, а явный
   `kb query --mode hybrid` доступен до смены режима по умолчанию.
5. Режим по умолчанию выбирается при приёмке первой версии по сравнительному
   `kb eval`. `hybrid` выбирается, если его hit-rate top-3 не ниже 0,8,
   превосходит FTS и не теряет критические точные запросы; иначе остаётся
   `fts`. Сравнение повторяется после существенного изменения корпуса, модели
   или ранжирования.
6. Внешний поставщик эмбеддингов требует отдельного решения о передаче кода и
   документации, конфиденциальности и стоимости.

### 3.4. Качество и разрешение противоречий

1. Полный `sync + lint` выводит постоянные условия из текущего канона, Git и
   файлов проекта; корректность не зависит от предыдущей SQLite-базы.
2. `contradicts` является постоянным состоянием. Ребро обязано хранить
   `open`, `resolved` или ограниченный сроком `deferred`; разрешение содержит
   ссылку на каноническое решение, отсрочка — ссылку на задачу и дату ревью.
3. Кандидат в дубль закрывается объединением через `supersedes` либо явным
   `distinct_from` с обоснованием.
4. Постоянная очередь, `attempts`, `dead-letter` и команды ручного закрытия
   позиции не создаются. Каждое предупреждение исчезает только после правки
   канона.
5. `rename-candidate` исключён из первой версии. Переименование файла со
   стабильным `id` безопасно; эвристический поиск переименований может быть
   добавлен позднее как анализ Git diff против явной базы.
6. `prose-changed` остаётся подсказкой проверки текущего diff, а не постоянной
   гарантией полного lint.

### 3.5. Параллельная работа

После слияния выполняется полный `kb sync && kb lint` против объединённого
канона. Система не навязывает merge/squash/rebase, не требует `KB-Actor` и не
классифицирует правки по автору. Отдельные поисковые случаи хранятся в
отдельных JSON-файлах, домены имеют отдельные индексы.

### 3.6. Карта репозитория

`scripts/dev/generate-structure.js` остаётся действующим генератором
`PROJECT_STRUCTURE.yaml`. Система знаний читает результат, индексирует дерево
и связывает пути со страницами через `covers`. Замена генератора не входит в
решение.

### 3.7. Техническая граница

CLI реализуется в Node.js как `scripts/dev/kb.mjs`, используя существующие
Ajv и `js-yaml` и явную dev-зависимость `better-sqlite3`. Производная база
располагается в `.tmp/project-knowledge/kb.sqlite3`. При запуске выполняется
проверка реальной доступности FTS5; беззвучная деградация запрещена.

## 4. Архитектурные инварианты

1. Удаление `.tmp/project-knowledge/` не удаляет знания или решения.
2. Полный rebuild достаточен для всех постоянных гарантий lint.
3. Векторная модель, размерность и версия входят в метаданные индекса; их
   изменение требует полной векторной переиндексации.
4. При `search.mode: fts` векторный результат не влияет на выдачу.
5. Разрешение предупреждения выражается изменением канона, а не локальной БД.
6. Инструмент не изменяет Markdown автоматически.
7. `PROJECT_STRUCTURE.yaml` продолжает генерироваться существующим
   инструментом и не получает второго производителя.
8. Резервное чтение репозитория допустимо после поиска и графа; фактический
   промах по существующей странице пополняет проверочный набор.
9. AST-узлы и рёбра имеют отдельные идентификаторы и явное происхождение;
   синтаксический импорт не становится смысловой зависимостью вики.

## 5. AST-граф кода

Минимальная AST-проекция входит в утверждённый объём первой версии. Для неё
действуют ограничения:

- AST-граф полностью производен от кода;
- идентификаторы и типы его узлов отделены от wiki-графа;
- мостом служит только `covers`;
- фактический `imports` не превращается автоматически в смысловой
  `depends_on`;
- AST не редактирует Markdown и не участвует в векторном индексе;
- функции, классы, граф вызовов и поток данных не входят в минимальную версию.

## 6. Отклонённые альтернативы

- **Общий черновик без Cubica ADR:** отклонён как конкурирующая и чужая
  архитектура.
- **SQLite или графовая БД как канон:** отклонено из-за второго источника
  истины и потери читаемых Git-диффов.
- **Только FTS с будущей реализацией векторов:** отклонено по решению PM;
  векторы, явный гибридный запрос и сравнительная оценка входят в первую версию.
- **Заранее зафиксированный режим по умолчанию без сравнения:** отклонён;
  первая версия сразу оценивает FTS и hybrid и выбирает режим по измеримому
  качеству, а при неготовом безопасном поставщике сохраняет FTS.
- **Постоянная очередь обслуживания:** отклонено в пользу канонических
  состояний разрешения.
- **Акторные Git-трейлеры:** отклонены как избыточные для процесса Cubica.
- **Инкрементальные события как обязательная гарантия:** отклонены; первая
  версия обещает только полностью воспроизводимые постоянные проверки.
- **Замена генератора структуры:** отклонена как отдельный риск без выигрыша
  для принятой функциональности.
- **Полный AST, граф вызовов и поток данных:** отклонены до появления
  измеримого потребителя; первая версия хранит только модульные импорты.

## 7. Последствия

Положительные последствия:

- архитектурный контекст становится адресно извлекаемым и измеримым;
- полный rebuild безопасен после слияний и на новой машине;
- нет скрытой очереди, зависящей от локального состояния;
- структура проекта входит в общий путь поиска без смены рабочего генератора;
- векторный путь реально испытывается с первой версии, но не ухудшает простой
  штатный поиск;
- анализ влияния показывает фактические обратные импорты отдельно от
  смысловых wiki-связей.

Стоимость и ограничения:

- страницы и связи требуют дисциплины актуализации вместе с кодом и ADR;
- первая реализация добавляет нативную dev-зависимость SQLite;
- векторный контур реализуется до назначения режимом по умолчанию и поэтому
  имеет заранее принятую стоимость разработки;
- реальный поставщик эмбеддингов остаётся отдельной границей безопасности;
- AST-проекция увеличивает время полной пересборки и не видит вычисляемые пути
  динамической загрузки;
- эвристическое распознавание переименований сознательно отсутствует.
