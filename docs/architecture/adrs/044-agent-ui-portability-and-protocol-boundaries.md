## 1. Понимание решения

Решение понято так: после принятия CopilotKit и AG-UI как основы Agent UI в ADR-043 Cubica должна явно предотвратить зависимость от поставщика или конкретного фреймворка. CopilotKit и AG-UI полезны как интеграционные слои, но они не должны становиться предметной моделью редактора, портала, помощников игрока или production LLM backend.

Этот ADR фиксирует границы заменяемости: CopilotKit является заменяемым адаптером UI/runtime, AG-UI является заменяемым адаптером внешнего событийного протокола, а контракты Cubica остаются стабильным источником истины для контекста, инструментов, валидации и долговременных изменений состояния.

## 2. Контекст

ADR-043 принял CopilotKit для React/Next.js UI помощников и AG-UI для событийного взаимодействия между приложениями и backend-сервисами агентов.

Текущий baseline редактора включает:

- `apps/editor-web/app/api/copilotkit/route.ts` как app-local шлюз CopilotKit runtime;
- `apps/editor-web/app/api/editor/agent/ag-ui/route.ts` как детерминированный локальный AG-UI backend;
- `apps/editor-web/src/components/editor-agent-ui.tsx` как React-интеграцию CopilotKit;
- `apps/editor-web/src/lib/agent-assistant-registry.ts` как реестр помощников;
- `apps/editor-web/src/lib/agent-context-projection.ts` как ограниченную проекцию контекста;
- frontend tools редактора поверх `EditorChangeSet`, dry-run, apply, undo, preview и save.

Без дополнительных ограничений будущие изменения могут случайно связать ядро Cubica с хуками CopilotKit, формой состояния AG-UI или конкретным LLM backend. Это сделает будущую замену дорогой и может ограничить возможности редактора.

## 3. Термины

- **Адаптер** - заменяемый слой перевода между двумя контрактами. В этом ADR адаптеры переводят контракты агентов Cubica в CopilotKit, AG-UI или production LLM backend.
- **Граница переносимости** - явная граница в коде и API, которая сохраняет заменяемость библиотеки или протокола.
- **Контракт агента Cubica** - стабильный внутренний контракт помощника: идентификатор помощника, разрешённый контекст, разрешённые инструменты, политика побочных эффектов, входы инструментов, результаты инструментов и метаданные аудита.
- **Production LLM backend** - продуктовый backend-сервис больших языковых моделей, который владеет вызовами моделей, промтами, рассуждением, выбором инструментов, оценками качества и наблюдаемостью.
- **Состояние Agent UI** - временное состояние для отображения сообщений помощника, хода выполнения инструментов и статуса взаимодействия. Оно не является авторитетным состоянием Cubica.

## 4. Решение

Cubica оставляет CopilotKit и AG-UI за явными адаптерами и считает стабильной архитектурой только контракты агентов Cubica.

1. **CopilotKit является заменяемым адаптером UI/runtime.**
   - CopilotKit может отображать чат, собирать контекст и публиковать frontend tools.
   - Типы CopilotKit не должны попадать в `packages/editor-engine`, `packages/contracts`, `services/runtime-api` или игровые бандлы.
   - Поведение инструментов Cubica должно описываться независимо от API хуков CopilotKit.

2. **AG-UI является заменяемым адаптером внешнего протокола.**
   - AG-UI может передавать жизненный цикл запуска, потоковый текст, вызовы инструментов, результаты инструментов, снимки состояния и дельты состояния.
   - События AG-UI переводятся в события помощника Cubica или диагностику на границе приложения.
   - AG-UI `STATE_SNAPSHOT` и `STATE_DELTA` не должны напрямую менять авторитетное состояние Cubica.

3. **Контракты агентов Cubica являются долговременным интерфейсом.**
   - Записи реестра помощников, проекции контекста, имена инструментов, формы входов/результатов инструментов, политика побочных эффектов и требования аудита принадлежат Cubica.
   - Контракт помощника редактора остаётся `EditorPatchIntent -> EditorChangeSet -> dry-run -> apply/undo/save`.
   - Будущие помощники портала, игрока, ведущего и администратора должны идти через свои предметные API Cubica, а не через записи состояния CopilotKit или AG-UI.

4. **Production LLM backend подключается за тем же шлюзом.**
   - Браузерные клиенты продолжают обращаться к app-local маршрутам, например `/api/copilotkit`.
   - Шлюз может направлять запросы в локальный детерминированный AG-UI backend, внешний production AG-UI backend или будущий backend без AG-UI через адаптер Cubica.
   - Production LLM backend владеет поведением модели, но изменяющие эффекты остаются вызовами инструментов Cubica.

5. **Замена должна выполняться через адаптеры, а не через предметный код.**
   - Замена CopilotKit должна в основном затрагивать компоненты UI помощника и app-local runtime route.
   - Замена AG-UI должна в основном затрагивать кодирование/декодирование протокола и модули адаптера событий.
   - Реестр помощников Cubica, проекция контекста, политика инструментов и шлюзы валидации должны сохраниться при любой из этих замен.

6. **Передача на production LLM backend требует явных ворот готовности.**
   - Внешний AG-UI или Cubica protocol backend подключается только через серверный шлюз приложения; браузер не получает provider keys или прямые credentials.
   - `CUBICA_EDITOR_AGENT_AG_UI_URL` для внешнего backend должен сопровождаться token/auth policy, smoke suite, audit envelope для изменяющих tool calls и replay/eval fixtures.
   - `CUBICA_EDITOR_AGENT_LOCAL_BACKEND=0` является обязательным production-решением для окружений, где локальный deterministic backend запрещён как fallback.

## 5. Архитектурные инварианты

Обязательные инварианты:

1. Импорты CopilotKit запрещены в `packages/editor-engine`, `packages/contracts`, `services/runtime-api`, игровых манифестах и игровых плагинах.
2. Объекты событий AG-UI нельзя сохранять как предметное состояние Cubica.
3. Состояние помощника остаётся производным UI-состоянием, пока оно не преобразовано в команду Cubica, change set или API-запрос и не прошло валидацию.
4. Каждый frontend- или server-side инструмент должен иметь принадлежащее Cubica имя, форму входа, форму результата и политику побочных эффектов.
5. Изменяющие инструменты требуют явного подтверждения, если запись помощника задаёт `human-approved`.
6. Production LLM backend не получает прямой доступ на запись к рабочим копиям редактора, таблицам портала или runtime-сессиям.
7. Локальный детерминированный AG-UI backend должен быть обозначен как local/dev и не должен представляться как production-поведение модели.
8. Сторонняя телеметрия остаётся выключенной по умолчанию, пока отдельное решение не разрешит её.
9. Миграция с CopilotKit или AG-UI не должна требовать изменения схем authoring-манифестов.
10. Deterministic public player runtime остаётся работоспособным, когда все UI помощников и agent backend отключены; AI-driven games могут требовать Agent Runtime только через явно объявленный manifest execution mode и readiness policy.
11. Production Agent backend не принимается без smoke suite для text response, tool call, tool result, approved mutation and disabled-backend behavior.
12. Изменяющие tool calls должны иметь audit envelope, который связывает user, assistant, run, tool name, approval state, timestamp and correlation id.
13. Собственная замена CopilotKit допускается только после parity review: streaming messages, tool progress, approvals, diagnostics, Surface renderer, disabled fallback, accessibility and bundle-size gates.

## 6. Границы заменяемости

### Граница CopilotKit

CopilotKit может владеть:

- рендером чата;
- боковой панелью помощника и inline UI;
- регистрацией frontend tools;
- передачей контекста из React-компонентов;
- клиентским поведением runtime-шлюза.

CopilotKit не должен владеть:

- моделью документа редактора;
- контрактом планирования `EditorChangeSet`;
- семантикой dry-run, apply, undo и save;
- моделью portal launch-session;
- состоянием runtime/player session;
- источником истины для аудита.

### Граница AG-UI

AG-UI может владеть:

- потоковым текстом помощника;
- событиями жизненного цикла вызова инструмента;
- временным состоянием помощника и снимками сообщений;
- транспортом внешнего agent backend.

AG-UI не должен владеть:

- каноническими манифестами;
- рабочими копиями editor session;
- изменениями runtime state;
- изменениями базы данных портала;
- решениями по лицензиям или платежам.

### Граница Production LLM Backend

Production LLM backend может владеть:

- промтами и выбором model provider;
- рассуждением и выбором инструментов;
- оценкой модели и replay harnesses;
- backend-наблюдаемостью model runs.

Production LLM backend не должен владеть:

- обходом шлюзов валидации Cubica;
- прямым изменением core data stores;
- добавлением game-specific runtime branches;
- сохранением provider-specific message state как авторитетного состояния Cubica.

## 7. Альтернативы

### A. Считать CopilotKit постоянным предметным слоем Agent UI

Отклонено. Это сделало бы будущую замену UI и non-React поверхности дорогими. CopilotKit остаётся полезным, но только как адаптер.

### B. Считать AG-UI постоянной внутренней моделью состояния

Отклонено. AG-UI подходит на границе взаимодействия с агентом, но у Cubica уже есть авторитетные предметные контракты и JSON Schema как источники истины.

### C. Отложить переносимость до момента реальной замены

Отклонено. Восстанавливать границы после распространения библиотечных типов по core-модулям значительно дороже, чем ограничить первую реализацию.

### D. Сразу построить полностью собственный Agent UI

Отклонено. Это снижает риск зависимости, но задерживает пользовательские сценарии помощника. Принятый компромисс - использовать CopilotKit сейчас, сохраняя возможность замены.

## 8. Последствия

Положительные последствия:

- Будущая замена CopilotKit или AG-UI ограничена адаптерами и кодом UI-поверхности.
- Production LLM backend может развиваться независимо от предметной логики редактора.
- Инструменты помощника остаются тестируемыми без рендера CopilotKit UI.
- Agent UI можно отключить без поломки потоков editor/player runtime.
- Помощники портала и игрока могут переиспользовать контракты помощников Cubica, даже если их UI отличается.

Цена и риски:

- Команда должна поддерживать явные контракты адаптеров и границы импортов.
- Некоторые удобные API CopilotKit могут намеренно не использоваться, если они протекают vendor-specific типами в предметный код.
- Обновления AG-UI всё равно требуют compatibility tests, потому что протокольные пакеты молодые.
- Собственная замена UI всё равно потребует заново построить chat UX, потоковый вывод и отображение вызовов инструментов.
