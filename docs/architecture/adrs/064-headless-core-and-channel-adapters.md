# ADR-064: Стратегия клиентского ядра — headless core и адаптеры каналов (supersedes ADR-014)

- Статус: Accepted (2026-07-07)
- Дата: 2026-07-07
- Заменяет: ADR-014 (Viewers Library Architecture)
- Связанные ADR: ADR-002 (Abstract View Protocol), ADR-025/056 (JSON Schema — SSOT и генерация контрактов), ADR-037/039 (plugin-архитектура), ADR-051 (API First), ADR-055 (чистота рендерера), ADR-059 (мультиплеер в монолите), ADR-060 (агентные игроки), ADR-062 (Phaser-канал)

## Как понято решение

Расхождение с ADR-014 (слоистый SDK: `core`/`shared`/`react-sdk`/`viewers`) было
осознанной отсрочкой, а не дрейфом: команда не тормозила разработку ради
преждевременной общности. Сейчас развилка проходится явно. Решается вопрос:
в какой форме платформа сохраняет универсальность клиентского слоя — как
код-библиотеку (SDK), как контракты, или как их сочетание — и когда какую
сложность наращивать.

## Контекст

Фактическое состояние к моменту решения:

- **Выжило и работает.** Пакеты контрактов `packages/contracts/*`
  (генерируются из JSON Schema, ADR-056); Abstract View Protocol
  (`IViewGateway`/`ViewCommand`/`ViewResponse`, ADR-002) и утилиты
  JSON Merge Patch / JSON Patch — жили в `SDK/core` и использовались
  `apps/player-web`; presenter-слой внутри `player-web`
  (`src/presenter/`, `src/lib/`) — фактическая слоистость без React-импортов;
  фасад плагинов `player-plugin-api` (ADR-037/039).
- **Умерло без потребителей.** `SDK/react-sdk` и `SDK/shared` (выведены из
  workspaces, LEGACY-0021), `SDK/viewers/web-base` (скелет, LEGACY-0011),
  реестр viewers с подписью клиентских скриптов из ADR-014 (не начат,
  LEGACY-0005), сессионная заготовка `createSession` в `SDK/core`.
- **Роль «viewer» заняли другие принятые решения:** декларативный рендерер
  UI-манифестов (ADR-055) и проектные плагины (ADR-037/039). «Viewer» больше
  не единица поставки — игра описывается данными.
- **Появился реальный спрос на framework-agnostic ядро** (в отличие от
  2025 года): агентные игроки ADR-060 (headless-клиент сессии без UI),
  Phaser-канал ADR-062 (рендерер не-React), редакторский preview.

Ориентиры из практики зрелых клиентских библиотек: headless core + тонкие
адаптеры фреймворков (TanStack Query, XState, Stripe.js); генерация
транспортных клиентов из OpenAPI вместо ручного кода; «правило трёх» —
извлечение общей библиотеки при втором-третьем реальном потребителе;
публичный SDK как обязательство (semver, обратная совместимость), которое
не берут без внешнего потребителя.

## Альтернативы

1. **Полное удаление SDK-направления.** Удалить `SDK/`, влить живые куски в
   `player-web`. Минус: теряется единственный оформленный framework-agnostic
   шов; при втором канале извлечение делается под давлением сроков.
2. **«Ядро по требованию»: зафиксировать шов, извлекать по триггеру.**
   Принято, детали ниже.
3. **Возродить ADR-014 как написан** (реестр viewers, `meta.viewer`, подпись
   и ручная проверка клиентских скриптов). Отклонено: роль viewer закрыта
   ADR-055, доверенные плагины — ADR-037/039, а подпись/реестр нужны только
   для недоверенного стороннего кода — этот кейс возникает с хостируемым
   редактором/маркетплейсом (LEGACY-0036) и потребует нового ADR под реальные
   требования.

## Решение

1. **Универсальная интеграционная поверхность платформы — контракты, а не
   код-библиотека.** Канал на любом языке/стеке интегрируется через OpenAPI
   `runtime-api` (ADR-051) и JSON Schema манифестов (ADR-025/056).
   TypeScript-пакеты — удобство для одного стека, не носитель универсальности.
2. **Каталог `SDK/` упраздняется.** Живой код бывшего `SDK/core`
   (Abstract View Protocol ADR-002 + утилиты JSON Merge Patch/JSON Patch)
   переносится в `packages/view-protocol` (`@cubica/view-protocol`).
   Мёртвые каталоги (`react-sdk`, `shared`, `viewers`, `custom-examples`,
   `simulators`) и сессионная заготовка `createSession` удаляются.
3. **ADR-014 помечается Superseded.** Слоистость подтверждена частично
   (headless-ядро — да, реестр viewers и подпись скриптов — нет).
4. **Шов будущего ядра фиксируется и охраняется CI.** Каталоги
   `apps/player-web/src/presenter/`, `apps/player-web/src/lib/` и
   `packages/view-protocol/` не импортируют React/Next
   (`npm run verify:player-core-seam` в составе `verify:canonical`).
   Плагины обращаются к платформе только через фасад `player-plugin-api`.
5. **Триггеры извлечения `packages/player-core`** (перенос presenter-слоя и
   `lib/` из `player-web` в framework-agnostic пакет): первый не-React
   потребитель presenter-логики. Именованные кандидаты: headless-клиент
   агентного игрока (ADR-060), Phaser-канал как самостоятельная поставка
   (ADR-062), встраивание плеера вне Next.js. Пока правило шва выполняется,
   извлечение остаётся механическим переносом файлов.
6. **Внешний публикуемый SDK — отложенная, но планируемая работа
   (LEGACY-0037).** Форма: публикация извлечённого `player-core` +
   сгенерированного из OpenAPI транспортного клиента. Активируется
   бизнес-кейсом (внешний потребитель, встраивание игр на чужие площадки;
   связано с LEGACY-0027/0028). До этого ничего не публикуется — semver и
   обратная совместимость не берутся как обязательство без потребителя.
7. **Транспортный клиент не пишется руками при росте API.** При расширении
   сессионного протокола (например, WebSocket по ADR-059) клиентская часть
   генерируется из декларативного контракта — в согласии с правилом 12
   CLAUDE.md (декларативные контракты — SSOT).

## Последствия

- Положительные: документация и код перестают противоречить друг другу
  (закрываются LEGACY-0005/0011/0021/0031); универсальность держится на
  контрактах; сложность (извлечение ядра, публикация SDK) наращивается по
  именованным триггерам, а не заранее; извлечение ядра остаётся дешёвым,
  пока CI-проверка шва зелёная.
- Риски: правило шва может мешать быстрым правкам в `presenter`/`lib`
  (лечится выносом framework-specific кода в `components/`); триггер может
  сработать позже, чем накопится скрытая связность — поэтому проверка шва
  обязана оставаться в `verify:canonical`.
- Долги: LEGACY-0037 (внешний публикуемый SDK — отложенная планируемая
  работа); LEGACY-0036 при активации потребует отдельного ADR о доставке
  недоверенного кода (наследник отклонённой части ADR-014).
