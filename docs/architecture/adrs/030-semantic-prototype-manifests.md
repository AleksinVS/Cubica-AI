## 1. Понимание решения

Решение понимается так: Cubica вводит обязательный слой authoring-манифестов для разработки и редактирования игровых и UI-манифестов. Authoring-манифесты компилируются в обычные runtime-манифесты, которые уже потребляют `runtime-api`, `player-web` и другие каналы доставки.

ADR-030 дополняет ADR-028. ADR-028 описывает action templates внутри runtime game manifest, а ADR-030 описывает более высокий слой разработки для `game.manifest` и `ui.manifest`. Runtime не должен понимать authoring-прототипы.

## 2. Контекст и проблема

Текущие `game.manifest.json` и `ui.manifest.json` являются большими плоскими JSON-документами. Это удобно для исполнения и валидации, но неудобно для разработки больших игр и экранов:

1. Агенту приходится читать много повторяющихся узлов, чтобы понять локальное изменение.
2. Смысл узла часто следует из контекста, а не из самого узла.
3. Повторяющиеся UI-компоненты, действия и фрагменты состояния приходится редактировать вручную в нескольких местах.
4. Ошибки при ручном копировании трудно отличить от намеренного отличия.

При этом существующая архитектура уже закрепляет важные ограничения:

- исполнимая логика должна оставаться машиночитаемой и валидируемой JSON Schema;
- `game-manifest.schema.json` и `ui-manifest.schema.json` остаются контрактами runtime-слоя;
- runtime не должен получать game-specific ветвления и authoring-only правила;
- ADR не должен превращаться в план исполнения.

## 3. Термины

- **Authoring-манифест** - исходный JSON-файл для разработки. В нем разрешены прототипы, локальные определения, наследование и операторы сборки.
- **Runtime-манифест** - скомпилированный JSON-файл, который потребляют runtime/player слои. Он обязан проходить runtime JSON Schema.
- **Прототип** - переиспользуемое описание узла, от которого можно наследовать поля.
- **Семантический тип** - имя, которое объясняет назначение узла для человека, редактора и ИИ-агента.
- **Сопутствующий файл** - отдельный файл рядом с основным артефактом. Например, source map хранится рядом с runtime-манифестом, но не является входом runtime.
- **Source map** - карта соответствий между скомпилированным JSON Pointer и исходным authoring-узлом. JSON Pointer - стандартный путь к узлу внутри JSON-документа, например `/actions/start`.
- **Идемпотентность** - свойство сборки, при котором повторный запуск без изменения входных файлов дает тот же результат.

## 4. Решение

Принять двухслойную модель манифестов:

1. Authoring layer становится обязательным целевым слоем для новых и изменяемых game/UI manifest-пакетов.
2. Runtime layer остается единственным слоем исполнения.
3. Агенты и разработчики редактируют authoring-манифесты, а не generated runtime-манифесты.
4. Компилятор генерирует:
   - runtime `game.manifest.json`;
   - runtime `ui.manifest.json` для каждого канала;
   - сопутствующие source map файлы для диагностики.
5. CI должен блокировать рассинхрон между authoring-входами и generated runtime-выходами.

## 5. Архитектурные инварианты

- Authoring layer не является runtime-контрактом.
- Runtime-api и player-web не резолвят authoring-прототипы.
- Runtime game manifest обязан проходить `docs/architecture/schemas/game-manifest.schema.json`.
- Runtime UI manifest обязан проходить `docs/architecture/schemas/ui-manifest.schema.json`.
- Authoring schemas также должны быть JSON Schema, а не TypeScript-only проверками.
- Generated runtime-манифесты не должны содержать authoring-only ключи.
- Любая новая runtime-семантика добавляется через JSON Schema, contracts и reusable handlers.
- Компиляция должна быть идемпотентной и детерминированной.
- Ошибки компиляции должны указывать на authoring-источник, а не только на generated output.

## 6. Семантические типы и наследование

В authoring-манифестах `_type` используется как семантический тип экземпляра. Он говорит агенту и редактору, что означает узел.

```json
{
  "_type": "ui.PrimaryActionButton",
  "label": "Continue"
}
```

Наследование прототипов описывается в definition через `_extends`, а не через рекурсивное восхождение по `_type` самого экземпляра.

```json
{
  "_definitions": {
    "ui.PrimaryActionButton": {
      "_extends": "ui.Button",
      "_semantics": "Main action that advances the current screen flow."
    }
  }
}
```

Такой подход сохраняет короткую запись для агента, но разводит две роли:

- `_type` отвечает за намерение экземпляра;
- `_extends` отвечает за техническое наследование definition;
- `_semantics` объясняет смысл definition естественным языком.

Дополнительные поля вроде `role` или `intent` допускаются только там, где одного `_type` недостаточно для поиска, аналитики или редакторского интерфейса.

## 7. Правила компиляции

Компилятор authoring-манифестов должен соблюдать следующие правила:

1. Разрешение `_type` всегда идет через явное определение в core, template или local registry.
2. Любой локальный тип обязан быть объявлен в `_definitions` текущего authoring-файла или подключенного authoring-пакета.
3. Любое definition обязано содержать `_semantics`.
4. Глубина цепочки `_extends` ограничена пятью уровнями.
5. Циклы наследования являются фатальной ошибкой сборки.
6. Объекты сливаются детерминированно: дочернее значение переопределяет родительское значение для одного и того же ключа.
7. Массивы по умолчанию заменяются целиком.
8. Операторы `+field` и `-field` допускаются только там, где authoring schema явно разрешает merge-операции.
9. Конфликт одновременного `field`, `+field` или `-field` в одном объекте является ошибкой, если конкретная schema не задает более узкое правило.
10. Порядок выходного JSON должен быть стабильным, чтобы `compile -> diff` был надежной CI-проверкой.
11. Компилятор не должен зависеть от времени, сети или непроверенных переменных окружения.

ADR не фиксирует низкоуровневый формат всех merge-операторов. Эти детали должны быть описаны в schema/tooling документации и рабочей задаче реализации.

## 8. Валидация и схемы

Для authoring layer вводятся отдельные JSON Schema:

- `game-authoring.schema.json`;
- `ui-authoring.schema.json`;
- общие reusable definitions для `_type`, `_extends`, `_definitions`, `_semantics` и разрешенных merge-операторов.

JSON Schema должна оставаться источником истины для структуры. Императивный код компилятора может проверять графовые свойства, например циклы, существование ссылок и конфликт merge-операторов, но не должен заменять schema validation структуры.

Authoring schemas могут использовать `$ref`, `allOf`, `patternProperties` и `unevaluatedProperties`, если это помогает расширять схемы без копирования общих свойств. Это согласуется с текущими рекомендациями JSON Schema по композиции и pattern-based ключам.

## 9. Трассировка источников

Трассировка источников должна храниться в сопутствующем source map файле, а не в runtime-манифесте.

Пример:

```text
game.manifest.json
game.manifest.source-map.json
ui/web/ui.manifest.json
ui/web/ui.manifest.source-map.json
```

Пример mapping:

```json
{
  "mappings": {
    "/actions/intro.advance": [
      {
        "file": "authoring/game.authoring.json",
        "pointer": "/flows/intro/actions/advance"
      },
      {
        "file": "authoring/prototypes/game.json",
        "pointer": "/_definitions/InfoAdvanceAction"
      }
    ]
  }
}
```

Runtime-манифест не должен получать `_source_trace`, потому что это загрязняет runtime schema и создает риск попадания отладочных полей в production.

## 10. Отношение к другим ADR

- **ADR-018**: сохраняется. На выходе authoring pipeline source of truth для runtime остается JSON-манифест, валидируемый схемой. Authoring files являются исходниками этой сборки.
- **ADR-025**: сохраняется. Runtime и authoring структуры описываются JSON Schema; TypeScript-only проверки не становятся источником истины.
- **ADR-028 и ADR-029**: сохраняются как историческая линия. Authoring-макросы
  раскрываются до публикации, а runtime получает типизированный Mechanics IR по
  ADR-084, а не выбирает между runtime templates, JsonLogic и scripts.
- **ADR-031**: сохраняется. Исполнительские детали compiler-а и CI остаются в
  документации задач, а не в ADR.

## 11. Отклоненные альтернативы

- Оставить только плоские runtime-манифесты. Отклонено: это сохраняет лишнее дублирование и перегружает контекст агентов.
- Сделать authoring layer опциональным для новых изменений. Отклонено: это разрушает дисциплину и оставляет два равноправных способа правки одной структуры.
- Дать runtime-api резолвить authoring-прототипы. Отклонено: runtime получил бы второй контракт и дополнительную сложность исполнения.
- Хранить `_source_trace` внутри runtime-манифеста. Отклонено: это загрязняет runtime schema и усложняет strict validation.
- Валидировать authoring layer только TypeScript-кодом. Отклонено: это нарушает JSON Schema SSOT и кроссплатформенность.
- Использовать `_type` одновременно как semantic intent и traversal edge. Отклонено: это смешивает намерение узла и техническое наследование.

## 12. Последствия

Положительные эффекты:

- ИИ-агенты редактируют компактные authoring-дельты, а не большие generated файлы.
- UI и game manifests получают одну mental model: `_type`, `_definitions`, `_extends`, compile-to-runtime.
- Runtime остается быстрым и простым: он получает уже скомпилированный JSON.
- CI может проверять рассинхрон через идемпотентную сборку.
- Source map сохраняет диагностическую связь с authoring-источниками без загрязнения runtime schema.

Trade-offs:

- Появляется обязательный build step для манифестов.
- Требуются две пары схем: authoring и runtime для game/UI manifests.
- Агентам нужно строго соблюдать правило: править authoring-входы, а не generated output.
