# ADR: Семантические прототипы манифестов с компиляцией (Build-Time Flattening)

**Статус:** Draft
**Дата:** 17 мая 2026 г.
**Контекст:** Архитектура игровых и UI манифестов, AI-Driven разработка.

## 1. Контекст и Проблема
В текущей архитектуре игровые и UI манифесты представляют собой большие, плоские JSON-структуры, валидируемые через классические JSON Schema (например, `game-manifest.schema.json`). 

Для AI-агентов такой подход создает несколько критических проблем:
1. **Слепое чтение (Token Overhead):** Чтобы понять структуру узла, агенту нужно проанализировать глобальную схему и сопоставить ее с узлом. В больших манифестах это приводит к переполнению контекста.
2. **Отсутствие семантики:** Схема описывает синтаксис (типы данных), но не бизнес-смысл. Агент видит `{"hp": 100}`, но не сразу понимает роль этого объекта без дополнительного контекста.
3. **Дублирование (Lack of DRY):** Повторяющиеся сущности (например, 10 одинаковых кнопок или врагов) описываются целиком, что раздувает файлы и усложняет глобальные изменения.

## 2. Предлагаемое Решение
Внедрить гибридный подход: **Semantic Prototype-Based Authoring + Build-Time Flattening**.

Мы разделяем жизненный цикл манифеста на два слоя:
1. **Слой разработки (Authoring Layer):** Компактные файлы, использующие прототипы (префабы), наследование и явную семантическую разметку (тегирование типов). Идеально подходят для чтения и редактирования агентами.
2. **Слой исполнения (Runtime Layer):** Плоский, денормализованный JSON, оптимизированный исключительно для быстрого парсинга движком.

### 2.1 Явная семантическая разметка (Semantic Tagging)
Каждый значимый узел манифеста в слое разработки должен явно указывать свой тип через поле (например, `_type`). 
Прототипы не только задают структуру, но и обязательно содержат мета-поле `_semantics` (или `_description`), объясняющее смысл узла.
*Преимущество:* Агент мгновенно понимает структуру и назначение узла, читая только локальный словарь типов (Registry), без необходимости анализировать глобальную JSON Schema.

### 2.2 Каскадное хранение прототипов (Tiered Prototypes)
Словари типов формируются каскадно:
1. **Core (Движок):** Базовые абстракции (например, `BaseEntity`, `UI_Container`). Хранятся в ядре проекта.
2. **Templates (Жанр):** Специфичные для жанра префабы (например, `RPG_Weapon`, `Platformer_Player`).
3. **Local (Игра):** Кастомные типы конкретной игры. Могут быть определены прямо внутри игрового манифеста в блоке `_definitions`.

### 2.3 Build-Time Flattening (Сборка)
Перед запуском или публикацией игры специальный скрипт (Compiler) берет исходные манифесты, рекурсивно поднимается по графу `_type`, выполняет глубокое слияние (Deep Merge) свойств и генерирует на выходе "плоский" манифест, с которым работает текущая архитектура игры.

## 3. Последствия (Consequences)

### Плюсы (+)
*   **Context Efficiency:** LLM-агенты читают и генерируют только дельты (изменения), экономя токены.
*   **Улучшенное понимание (Semantic AI):** ИИ оперирует бизнес-понятиями (что делает этот префаб), а не просто типами данных (строка/число).
*   **Глобальный рефакторинг:** Изменение баланса или дизайна требует правки одного прототипа вместо сотен инстансов.
*   **Производительность клиента:** Движок продолжает получать плоский JSON, не тратя CPU на разрешение зависимостей.

### Минусы (-)
*   Усложнение пайплайна сборки (появление шага компиляции манифестов).
*   Разрыв контекста при дебаге (требуется реализация source mapping).
*   Необходимость поддержки двух наборов схем (Authoring Schema и Runtime Schema).

## 4. Детали Реализации (Предложения)

Ниже приведены предлагаемые решения для ключевых технических вызовов:

1. **Стратегии слияния коллекций (Array Merging):**
   * **Решение:** Использовать семантические префиксы в ключах для управления поведением массивов при слиянии (в слое разработки):
     * `названиеПоля` (без префикса) — полная перезапись (Replace). Массив наследника полностью заменяет массив предка. Это поведение по умолчанию.
     * `+названиеПоля` — добавление в конец (Append). Элементы из наследника добавляются к массиву предка.
     * `-названиеПоля` — удаление элементов (Remove). Элементы, перечисленные в наследнике, удаляются из массива предка, если они там присутствуют.
   * *Пример:* Если предок имеет `tags: ["enemy", "melee"]`, а наследник определяет `"+tags": ["boss"], "-tags": ["melee"]`, итоговый массив в сборке: `["enemy", "boss"]`.

2. **Схемы валидации (Authoring Schemas):**
   * **Решение:** Создать отдельные схемы для этапа разработки (например, `game-authoring.schema.json`), наследующие базовые типы из рантайм-схем.
   * Внедрить глобальные определения, разрешающие системные ключи (`_type`, `_prototype`, `_vars`, `_semantics`) в любом объекте манифеста.
   * Использовать `patternProperties` (например, `^\\+`, `^\\-`) для легализации операторов слияния в массивах и объектах.

3. **Защита от циклов и лимиты:**
   * **Решение:** Внедрить проверку графа наследования на этапе компиляции (Build-Time Flattening).
   * Сборщик должен вести стек (Set) посещенных прототипов для каждого разрешаемого узла. Обнаружение дубликата (цикл `A -> B -> A`) вызывает фатальную ошибку сборки с выводом трейса.
   * Установить жесткий лимит глубины наследования (Max Nesting Depth) равным **5 уровням**. Это убережет от "комбинаторных взрывов" и заставит сохранять архитектуру префабов плоской и читаемой.

4. **Трассировка ошибок (Source Mapping):**
   * **Решение:** В режиме отладки (Development Build) сборщик добавляет служебный ключ `_source_trace` к каждому склеенному объекту.
   * *Формат:* Массив строк или объектов, описывающий цепочку: `"_source_trace": ["core:BaseEnemy", "template:RPG_Orc", "local:Level1_Boss"]`. 
   * Это позволит агентам и разработчикам мгновенно находить файл-источник при дебаге рантайма. В Production-сборке для игроков это поле вырезается.

5. **Динамические типы (Ad-Hoc Types):**
   * **Решение:** Агентам и дизайнерам разрешается создавать локальные типы прямо внутри конкретных манифестов (уровней, сцен) без изменения глобальных схем.
   * **Правило:** Любой новый кастомный тип (например, `_type: "IceTrap"`) должен быть обязательно задекларирован в блоке `_definitions` текущего файла.
   * Декларация обязана содержать поле `_semantics` (описание бизнес-смысла объекта на естественном языке). Таким образом, ИИ при последующих проходах сможет понять назначение объекта без анализа всей сцены.
