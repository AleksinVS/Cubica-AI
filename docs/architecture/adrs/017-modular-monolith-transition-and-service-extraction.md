# ADR-017: Переход к модульному монолиту и правила будущего выделения микросервисов

- **Дата**: 2026-03-19
- **Статус**: Proposed
- **Авторы**: Codex
- **Компоненты**: Backend, Router, Game Engine, Game Repository, Metadata DB, SDK contracts, deployment model

## Контекст

Текущая целевая архитектура Cubica описывает backend как набор отдельных сервисов:

- Router
- Game Engine
- Game Repository
- Game Catalog
- Game Editor
- Metadata Database

Однако фактическая реализация заметно отстаёт от этой сервисной декомпозиции:

- большинство каталогов `services/*` пока содержат только `DEV_GUIDE.md` и пустые `src/`/`tests/`;
- реальный исполнимый слой сосредоточен в `SDK/*`, `games/*` и черновом player;
- нет одного устойчивого end-to-end backend path вида `client -> session api -> runtime -> state store -> manifest bundle`;
- стоимость распределённой архитектуры уже присутствует в документации, но ещё не компенсируется инженерной зрелостью системы.

Дополнительное ограничение проекта: платформа проектируется под ещё не известные будущие типы игр, сценариев и каналов доставки. Это делает преждевременную сервисную декомпозицию особенно рискованной: слишком рано фиксируются границы, которые пока не подтверждены реальной нагрузкой и кодом.

Для ближайшей фазы проекту нужен:

- один рабочий vertical slice;
- единый deployable backend;
- строгие машинно-читаемые контракты;
- внутренняя модульность, совместимая с будущим выделением сервисов.

## Альтернативы

- **Сохранить курс на отдельные сервисы уже сейчас** — соответствует текущему описанию в документации, но повышает операционную и интеграционную сложность до появления рабочего ядра.
- **Построить модульный монолит как промежуточную форму** — даёт один deployable backend при сохранении чётких логических границ.
- **Сразу строить один “толстый backend” без внутренней модульности** — уменьшает начальную сложность, но затрудняет дальнейшее выделение сервисов и приводит к архитектурному размыванию.

## Решение

Выбирается **модульный монолит** как основная backend-модель для ближайшей фазы развития Cubica.

### 1. Что означает модульный монолит для Cubica

На ближайшей фазе backend разворачивается как **один deployable runtime backend**, внутри которого существуют отдельные логические модули с жёсткими интерфейсами:

- `player-api`
- `session`
- `runtime`
- `content`
- `ai`
- `telemetry`
- `admin`

Внешняя сервисная декомпозиция `Router / Engine / Repository / Metadata DB` временно заменяется внутренней модульной декомпозицией в одном runtime backend.

### 2. Обязательные правила внутренней модульности

Чтобы модульный монолит не превратился в неструктурированный backend, вводятся следующие правила:

1. У каждого модуля есть собственный публичный API.
2. Внутренние детали модуля не импортируются напрямую из соседних модулей.
3. Межмодульное взаимодействие строится через:
   - команды;
   - query;
   - доменные события.
4. Хранилища инкапсулируются внутри модуля.
5. DTO и event contracts оформляются как отдельные общие контракты.
6. Integration tests проходят через публичные границы, а не через внутренние реализации.

### 3. Целевая модульная карта backend-а

#### `player-api`

Внешние player-facing endpoints:

- `createSession`
- `getSession`
- `getSessionState`
- `dispatchAction`
- `subscribeSessionEvents`

#### `session`

Жизненный цикл сессий:

- создание;
- блокировки;
- versioning;
- checkpoints;
- recovery;
- sequencing.

#### `runtime`

Исполнение игровой логики:

- action routing;
- deterministic transitions;
- script execution;
- capability execution;
- effect assembly.

#### `content`

Работа с игровыми пакетами:

- загрузка manifest bundle;
- валидация;
- версионирование;
- кэширование;
- подготовка данных для runtime.

#### `ai`

AI capability layer:

- prompt building;
- вызовы моделей;
- normalization;
- eval/replay hooks;
- narrative/content generation;
- state/action proposals.

#### `telemetry`

- logs;
- traces;
- metrics;
- audit events.

#### `admin`

- health;
- inspect/replay;
- internal tools;
- ops endpoints.

### 4. Что меняется относительно текущего сервисного описания

На ближайшей фазе:

- `Router` перестаёт рассматриваться как отдельный deployable сервис и становится внешним API-модулем backend-а;
- `Game Engine` остаётся отдельным логическим модулем исполнения, но не отдельным runtime process;
- `Game Repository` становится content-модулем и storage boundary внутри того же backend-а;
- `Metadata DB` сначала реализуется как внутренний аналитический/аудитный модуль и набор таблиц/экспортов, а не как отдельный сервис.

Это изменение касается прежде всего **операционного развёртывания**, а не логической модели предметной области.

### 5. Методика будущего выделения микросервисов

Будущая декомпозиция на микросервисы разрешена только по фактическим признакам зрелости системы, а не по заранее нарисованной схеме.

Модуль допускается к выделению в отдельный сервис, если одновременно выполняются несколько условий:

- у него отдельный профиль нагрузки;
- у него отдельный SLA;
- у него свой жизненный цикл релизов;
- у него отдельные требования к безопасности или изоляции;
- у него появляется собственная команда или явный ownership;
- он взаимодействует с остальной системой через уже стабильный контракт.

Ожидаемый порядок кандидатов на выделение:

1. `ai`
2. `content`
3. `session/runtime`
4. `analytics/telemetry`

### 6. Ограничение для будущего выделения сервисов

Нельзя выделять модуль в микросервис только на основании того, что он “уже выглядит как сервис” в документации. Выделение разрешается только после подтверждения:

- метриками;
- нагрузкой;
- требованиями эксплуатации;
- устойчивостью публичного контракта.

## Последствия

- Появляется один реалистичный backend-target для ближайшей фазы.
- Снижается операционная и интеграционная сложность.
- Упрощается сборка первого честного vertical slice.
- Появляется более надёжная база для AI-first и Code-first развития.
- Требуется обновить обзорные архитектурные документы, чтобы не описывать текущую backend-модель как уже распределённую систему.
- Часть существующих документов о сервисах должна интерпретироваться как описание **будущих модульных boundaries**, а не обязательных deployable units в ближайшей фазе.

Риски:

- если не enforce-ить внутренние границы, монолит быстро деградирует в тесно связанный backend;
- потребуется дисциплина контрактов и integration tests;
- некоторые старые описания сервисов станут временно “слишком оптимистичными” относительно фактической архитектуры и должны быть скорректированы.

## План внедрения

1. Зафиксировать в архитектуре и roadmap, что ближайшая backend-фаза Cubica строится как модульный монолит.
2. Собрать один deployable `runtime-api` backend с модулями `player-api`, `session`, `runtime`, `content`, `ai`, `telemetry`, `admin`.
3. Стабилизировать контракты query/command/event между модулями и SDK.
4. Собрать один рабочий vertical slice для игры `Antarctica`.
5. После появления рабочих метрик и нагрузки пересматривать необходимость выделения отдельных сервисов модуль за модулем.
