# ADR-004: LLM Context Pipeline and Game Editor Role

## Status
Accepted

## Context
Платформа Cubica использует LLM (Large Language Model) в качестве игрового движка. Основные вызовы этого подхода:
1. **Ограниченный контекст:** LLM имеет лимит токенов, и передача всего состояния игры (State) и истории (History) невозможна или слишком дорога.
2. **Безопасность:** Состояние игры содержит скрытые данные (секреты, ловушки, внутренние счетчики), которые не должен видеть клиент (Front-end), но должен видеть Master (LLM).
3. **Забывание:** При длинных игровых сессиях LLM "забывает" начало сюжета, если просто обрезать историю (Sliding Window).
4. **Сложность промптинга:** Создание инструкций для сжатия истории (Summarization) требует квалификации, которой может не быть у автора игры.

## Decision

Мы принимаем следующую архитектуру пайплайна подготовки контекста (**LLM Context Pipeline**):

### 1. Разделение состояния (Public vs Secret)
В JSON-манифесте игры (`manifest.state`) и в Runtime-памяти состояние строго делится на две зоны:
*   **Public State:** Данные, видимые и клиенту, и LLM (например, HP, инвентарь, локация).
*   **Secret State:** Данные, видимые только LLM (например, настроение NPC, скрытые флаги квестов).

Router Service обязан фильтровать `secret` поля перед отправкой состояния на клиент.

### 2. Фильтрация контекста (State Pruning)
Для формирования промпта используется механизм **Whitelist**, конфигурируемый в манифесте (`engine.context.include`).
*   По умолчанию LLM видит всё (`public` + `secret`).
*   Для больших игр автор может указать конкретные пути (JSONPath), которые нужны LLM (экономия токенов).

### 3. Стратегия памяти (Memory Strategy)
Используется гибридная модель:
*   **Sliding Window:** В контекст попадают последние $N$ (напр. 10) сообщений в сыром виде.
*   **Smart Summarization:** Более старые сообщения сжимаются в "summary" и хранятся в `state.secret.story_summary`.

### 4. Роль Game Editor (Design-Time Analysis)
Для решения проблемы сложности промптинга, мы переносим ответственность за создание правил суммаризации на этап дизайна.

*   **Процесс:** При публикации игры Game Editor запускает процесс "Game Analysis".
*   **Действие:** Специальный Meta-Prompt анализирует правила игры и генерирует инструкцию для суммаризатора (например: *"В этой игре важно запоминать имена подозреваемых и найденные улики"*).
*   **Результат:** Инструкция сохраняется в манифест (`engine.memory.summary_instruction`) и используется движком в Runtime.

### 5. Reference Resolution (Markdown Assets)
Для улучшения редактируемости и поддержки генерации игр через LLM, правила игры и сценарии хранятся во внешних Markdown-файлах, на которые ссылается манифест.

*   **Context Sources:** В манифесте добавляется секция `assets` (или `context_sources`), где определяются пути к файлам (например, `rules.md`, `scenario.md`).
*   **Pipeline Action:** Перед формированием промпта движок загружает содержимое указанных файлов.
*   **Injection:** Текст из файлов внедряется в System Prompt через плейсхолдеры (например, `{{rules}}`).

## Consequences

### Positive
*   **Экономия токенов:** Мы контролируем, что попадает в контекст.
*   **Безопасность:** Исключена случайная утечка секретов на клиент.
*   **Качество игры:** LLM не теряет нить повествования благодаря специфичным для игры инструкциям суммаризации.
*   **Удобство автора:** Автору не нужно быть промпт-инженером, Editor делает работу за него.
*   **Source of Truth:** Правила и сценарий хранятся в читаемом формате (.md), что упрощает их редактирование людьми и LLM-генераторами.

### Negative
*   **Усложнение Game Editor:** Требуется реализация агента-аналитика в редакторе.
*   **Задержка публикации:** Публикация игры теперь требует времени на анализ (LLM call).
*   **Сложность отладки:** Если автогенерация инструкций сработает плохо, игра может "сломаться" в длинной сессии. Требуется возможность ручной правки инструкций.
*   **IO Overhead:** Движку требуется читать дополнительные файлы при инициализации сессии или кэшировать их.
