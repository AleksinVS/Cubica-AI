# ADR-005: Session State Persistence Strategy

## Status
Accepted

## Context
Игровые сессии в Cubica строятся на LLM-first подходе. Это накладывает специфические требования:
1.  **Большой размер состояния:** JSON-состояние включает `public`, `secret` части и `history` (текст чата).
2.  **Длительные транзакции:** Обработка хода (LLM inference) занимает секунды.
3.  **Атомарность:** Необходимо исключить Race Conditions (двойное нажатие кнопок, параллельные ходы).
4.  **Надежность:** Прогресс игрока не должен теряться при перезагрузке сервисов.

## Decision

### 1. Storage Technology: PostgreSQL (JSONB)
Для этапа MVP и ранних стадий (до высокой нагрузки) мы используем **PostgreSQL** как единое хранилище.
*   Используется тип данных `JSONB` для хранения всего контекста сессии (`state` + `history`).
*   **Redis** рассматривается как опциональный слой кэширования (Hot Storage) для будущей оптимизации (Phase 2+), но сейчас не внедряется для снижения сложности инфраструктуры.

### 2. Concurrency Control: Pessimistic Locking
Для защиты от гонок данных используется **Пессимистичная блокировка** на уровне приложения или БД.
*   При начале обработки хода (`ProcessAction`) ставится эксклюзивная блокировка на сессию (`SELECT FOR UPDATE` или Redis Lock/Memory Lock в Router).
*   Если сессия заблокирована, новые запросы отклоняются с кодом `429 Too Many Requests` (или `423 Locked`) с сообщением "Предыдущий ход еще обрабатывается".
*   Блокировка снимается только после сохранения обновленного состояния.

### 3. Data Schema Strategy
Данные сессии хранятся денормализованно в одной строке для минимизации IO (так как для каждого хода нам нужен *весь* контекст).

Структура таблицы `game_sessions` (MVP, согласуется с ADR-011):
*   `id` (UUID, PK)
*   `game_id` (String) — ID манифеста
*   `user_id` (UUID) — Владелец
*   `state` (JSONB) — `{ public: {...}, secret: {...} }`
*   `history` (JSONB) — Массив последних сообщений (Sliding Window)
*   `state_version` (Integer) — Версия состояния сессии (увеличивается при каждом успешно подтверждённом ходе)
*   `last_event_sequence` (Integer, опционально) — Номер последнего обработанного события (для интеграции с ADR-011)
*   `created_at`, `updated_at` (Timestamp)

### 4. Session Recovery Strategy (Checkpointing & TTL)

Механизм Session Recovery дополняет пессимистичную блокировку и определяет, что происходит при сбоях во время обработки хода.

#### 4.1. Checkpointing (контрольная точка)

Под "checkpointing" в контексте Cubica понимается сохранение **последнего устойчивого состояния** сессии в базе данных перед началом рискованной операции, такой как:

- вызов LLM (длительный и потенциально нестабильный запрос),
- обращение к внешним сервисам (оплата, интеграции и т.п.).

Фактически, роль checkpoint выполняет текущая строка в `game_sessions`:

- перед началом обработки хода Router читает состояние (`state`, `history`, `state_version`) в рамках транзакции;
- вычисление нового состояния происходит в памяти после получения ответа от LLM/движка;
- в БД записывается **только новое, целиком сформированное состояние** вместе с увеличенным `state_version`;
- если во время вызова LLM или вычисления дельты произойдёт сбой, транзакция будет откатана, а в таблице `game_sessions` останется предыдущая, согласованная версия состояния.

Таким образом, откат к последнему консистентному состоянию — это просто повторное чтение строки сессии из БД после сбоя.

#### 4.2. TTL для блокировки и dead lock detection

Пессимистичная блокировка по умолчанию снимается при завершении транзакции (успех или ошибка). Однако на уровне приложения важно:

- ограничить максимальное время обработки одного хода;
- отличать "долгий, но живой" ход от зависшего (dead) процесса;
- корректно реагировать на ситуацию, когда предыдущий запрос пропал (сетевой разрыв, падение процесса), а игрок продолжает пытаться взаимодействовать с сессией.

Основные правила:

1. **Application-level TTL:** для обработки одного хода задаётся максимальное время (например, 30–60 секунд). Если по истечении этого времени Router не получил результат от Engine/LLM, запрос считается неуспешным, транзакция должна быть завершена (rollback), а блокировка — снята.
2. **Dead lock detection:** при поступлении нового запроса к сессии Router:
   - проверяет, нет ли активной обработки хода для этой сессии (по вспомогательному состоянию в памяти или БД);
   - если "обработка" длится дольше TTL, она считается "мертвой" и инициируется процедура восстановления: текущий запрос получает понятную ошибку, внутренняя метка обработки сбрасывается.
3. **Поведение для клиента:**
   - если ход был завершён успешно в пределах TTL — клиент получает обычный ответ с обновлённым состоянием;
   - если ход "завис" и был принудительно прекращён, следующий запрос игрока получает сообщение вида: "Предыдущий ход не был завершён из-за ошибки сервера, сессия восстановлена в последнем устойчивом состоянии."

Варианты реализации TTL:

- через внутренние таймеры в Router и/или
- через отдельную таблицу/структуру "in-flight actions" с полем `started_at`, по которому периодический фоновой процесс снимает "мертвые" обработки.

Конкретный технический вариант фиксируется в документации Router, но на уровне ADR-005 важно наличие концепции TTL и правила реакции на таймаут.

### 5. Redis Integration (Phase 2+)

Интеграция Redis (документируется в F_00062) рассматривается как путь масштабирования после MVP. Основные решения:

1. **Read-through cache для сессий:**
   - Источником истины остаётся таблица `game_sessions` в PostgreSQL.
   - При чтении состояния Router сначала обращается к Redis (`session:{session_id}`); при промахе читает из БД и записывает в Redis с ограниченным TTL.
   - Запись выполняется по схеме *write-through*: успешный `UPDATE` в Postgres сопровождается обновлением кэша в Redis тем же полным JSON‑объектом.
2. **Распределённые блокировки (Distributed Lock):**
   - Вместо `SELECT ... FOR UPDATE` в одном инстансе Router используется распределённая блокировка на ключе `lock:session:{session_id}`.
   - Рекомендуемый паттерн — `SET lock:session:{session_id} <token> NX PX <ttl_ms>` с обязательной проверкой токена при освобождении блокировки (вариант Redlock).
   - TTL блокировки согласуется с параметрами Session Recovery (см. раздел 4) и не должен превышать application‑level TTL обработки хода.
3. **Стратегия согласованности:**
   - При расхождении данных (например, Redis потерян или содержит устаревшее состояние) Router всегда может восстановиться, перечитав сессию из Postgres.
   - Redis рассматривается как *ускоритель* чтения и блокировок; потеря данных в Redis не должна приводить к потере прогресса сессий.
4. **Write-behind (опционально):**
   - Для высоких нагрузок допускается режим, когда Router пишет в Redis синхронно, а в Postgres — асинхронно (write-behind worker).
   - Такой режим должен включаться только после отдельной оценки рисков потери последних секунд прогресса и описываться в отдельном ADR или разделе F_00062.

## Consequences

### Positive
*   **Простота:** Не нужно поддерживать согласованность между Redis и Postgres.
*   **Надежность:** ACID транзакции и checkpointing гарантируют сохранение прогресса и восстановление до последнего устойчивого состояния.
*   **Гибкость:** JSONB позволяет менять структуру стейта без миграций схемы БД, а поля версионности согласованы с мультиплеерной моделью (ADR-011).

### Negative
*   **Производительность:** При очень больших JSON (сотни КБ) и высоком RPS, Postgres может стать узким местом (по CPU/IO при парсинге JSONB).
*   **UX:** Блокировка интерфейса на время хода (пессимистичная блокировка) может раздражать, если LLM отвечает очень долго; требуется правильная индикация "движок думает".
*   **Сложность обработки ошибок:** Необходимо аккуратно реализовать TTL, чтобы не снимать "живые" блокировки и не приводить к двойной обработке ходов.

## Mitigation
*   Если JSON станет слишком большим, вынесем старую историю в отдельную таблицу (Cold History).
*   Если RPS вырастет, внедрим Redis как Write-Through Cache.
*   Если TTL и управление блокировками будут становиться источником ошибок, можно выделить отдельный модуль/сервис управления сессиями и блокировками, используя опыт ADR-011 и будущей интеграции Redis (F_00062).
