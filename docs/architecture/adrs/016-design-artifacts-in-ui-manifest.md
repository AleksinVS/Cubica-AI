## Контекст

Платформа Cubica ориентирована на разработку игр с помощью ИИ-агентов
(ИИ-агент — автономная программа на базе большой языковой модели, способная
выполнять сложные задачи). В архитектуре ADR-013 UI-манифест может ссылаться на
макеты, однако одной ссылки на изображение недостаточно, чтобы агент однозначно
понял назначение областей, визуальное намерение и происхождение артефакта.

Встраивание подробных описаний и истории всех макетов непосредственно в
UI-манифест раздувает runtime-контракт и связывает доставку интерфейса с
authoring-данными. Нужна отдельная, машиночитаемая модель дизайн-артефактов,
которая сохраняет связь с UI, но не становится обязательной частью runtime.

## Альтернативы

### А1. Расширить существующую секцию `layouts` в UI-манифесте

- **Плюсы**: минимальные изменения в схеме, всё в одном файле.
- **Минусы**: манифест быстро разрастается; отдельные артефакты сложно
  версионировать и переиспользовать.

### А2. Хранить описания в манифесте, изображения — внешними файлами

- **Плюсы**: изображения не дублируются; метаданные доступны из одного файла.
- **Минусы**: подробные описания всё равно загрязняют runtime-манифест.

### А3. Полностью внешнее хранение со ссылками `source_ref` (выбрано)

- **Плюсы**: UI-манифест остаётся компактным; описания версионируются и
  переиспользуются независимо; подход совместим с ADR-013.
- **Минусы**: требуется проверять целостность внешних ссылок.

### А4. Оставить произвольный `spec`

- **Плюсы**: максимальная гибкость.
- **Минусы**: агенты и инструменты не могут полагаться на стабильную структуру.

## Решение

Ввести отдельный authoring-контур дизайн-артефактов:

1. Артефакт имеет тип с устойчивой семантикой.
2. Изображение и его машиночитаемое описание хранятся вне UI-манифеста.
3. UI-манифест содержит только реестр и ссылки на нужные артефакты.
4. История версий и связи между артефактами хранятся отдельно от runtime UI.
5. Рендерер не обязан загружать дизайн-описания для показа интерфейса.

### 1. Типы дизайн-артефактов

Поддерживаются следующие семантические типы:

| Тип | Назначение |
|---|---|
| `concept` | стиль, настроение и общая визуальная идея |
| `reference` | внешний ориентир для желаемого решения |
| `wireframe` | структура и иерархия интерфейса без финального оформления |
| `flowchart` | пользовательский путь и переходы между экранами |
| `storyboard` | последовательность визуальных состояний и анимаций |
| `mockup` | детальный макет с финальным визуальным оформлением |
| `asset` | готовый графический элемент |

Тип объясняет назначение артефакта, но не задаёт обязательную линейную
последовательность дизайн-процесса.

### 2. Хранение и владение

- Дизайн-артефакты принадлежат конкретному игровому или UI-пакету.
- Бинарный файл, файл описания и история версий хранятся рядом в
  authoring-контуре пакета, но остаются отдельными артефактами.
- Runtime bundle может не включать authoring-описания, если они не нужны каналу
  доставки.

### 3. Публичный контракт описания

Описание дизайн-артефакта обязано содержать:

- устойчивый `id` и семантический `type`;
- человекочитаемые `name` и `description`;
- ссылку на изображение и его базовые технические метаданные;
- версию и происхождение артефакта;
- при наличии семантической разметки — именованные `regions` с координатами,
  назначением и необязательной связью с UI-компонентом или состоянием;
- для визуальных элементов, участвующих в генерации, — `generation.prompt` и
  необходимые параметры воспроизводимости;
- при наличии общей визуальной системы — `style_tokens`.

Точные допустимые поля, форматы координат и ограничения задаются JSON Schema, а
не примерами конкретной игры в ADR.

Уточнение ADR-048: `generation.prompt` относится только к визуальному
дизайн-артефакту или его региону. Он не является элементным authoring-промтом
`_prompt`, описывающим содержание, поведение или методический смысл игрового
экземпляра.

### 4. Связи и версионирование

- История хранит версии артефактов и направленные связи между ними.
- Ссылка `derived_from` или эквивалентная связь указывает происхождение новой
  версии без копирования всего родительского описания.
- Поддерживаемые отношения включают вдохновение, структурирование,
  детализацию, извлечение и замену.
- UI-манифест ссылается на устойчивый идентификатор; выбор конкретной версии
  должен быть однозначным и воспроизводимым.
- История дизайна является authoring-метаданными и не определяет runtime-логику
  игры.

### 5. Интеграция с UI-манифестом

UI-манифест содержит компактный реестр `design_artifacts`. Запись реестра
включает:

- идентификатор и тип;
- `source_ref` на внешнее описание;
- необязательную цель `screen` или `component`;
- при необходимости — закреплённую версию.

Экран или компонент может ссылаться на запись реестра через устойчивый
идентификатор. Эта ссылка предназначена для редактора, проверки и агентных
инструментов; сама по себе она не меняет поведение runtime-рендерера.

## Последствия

Положительные эффекты:

- агенты получают структурированное описание визуального намерения;
- UI-манифест не раздувается подробными authoring-данными;
- происхождение и версии дизайна становятся проверяемыми;
- описания и визуальные токены можно переиспользовать;
- runtime остаётся независимым от дизайн-истории.

Риски и ограничения:

- нужно проверять согласованность изображения, описания и ссылок;
- семантическая разметка регионов требует дополнительных authoring-усилий;
- внешнее хранение увеличивает число связанных артефактов;
- изменение формата описаний требует версионирования JSON Schema.
