## 1. Понимание решения

Решение понято так: продуктовой системе знаний нужны сырые диалоги в
PostgreSQL и подтверждённые синтезированные знания в человекочитаемом
Markdown. Однако предыдущий черновик преждевременно вводил четыре специальных
вида пакетов, разные хранилища, отдельный указатель ревизии и слишком жёсткую
трактовку происхождения.

Предлагается сохранить необходимые границы владения, подтверждения и
приватности, но реализовать их одной универсальной областью знаний, одним
управляемым Git backend и точечными операциями текущего разговорного агента.
Реплики агента могут быть источником формулировки совместно созданного
знания, но не могут самостоятельно подтверждать мнение пользователя.

OKF остаётся целью совместимости и экспорта, а не причиной отказаться от
важной возможности Cubica.

## 2. Контекст

Пользовательские помощники нужны как разработчикам игр, так и ведущим.
Помощник должен помнить подтверждённые решения и предпочтения между
диалогами, но не смешивать:

- личное и командное;
- авторский и публичный контуры;
- разные проекты и мероприятия;
- актуальное предметное состояние и прошлый синтез;
- слова агента и полномочие пользователя подтвердить их.

Полный текст диалога полезен как первичный материал, но не является удобным
долговременным контекстом. Markdown-вики даёт накопительный синтез,
человекочитаемость, переносимость и исправляемость.

Версия 0.2 пыталась получить атомарность и удаление через несколько
хранилищ. Git уже предоставляет согласованный снимок, историю и различия, а
настоящее физическое удаление всё равно требует отдельной процедуры для
истории, кэшей и резервных копий.

OKF 0.1 Draft стандартизует минимальную форму Markdown-пакета, разрешает
расширения и не определяет хранение, поиск, права или подтверждение.

Принятые решения Agent UI требуют, чтобы модель меняла долговременное
состояние только через инструменты и проверки Cubica. Отдельный сервис без
подтверждённой операционной границы запрещён действующей архитектурой.

## 3. Предлагаемое решение

### 3.1. Граница инженерной и продуктовой систем

Инженерная система знаний платформы и продуктовая система пользовательских
помощников остаются раздельными.

Разрешено переиспользовать нейтральные JSON Schema, Markdown-парсер,
генератор `index.md` и проверки. Пользовательские данные, права, история,
индексы и аудит не объединяются с репозиторием и индексами разработки
платформы.

### 3.2. Универсальная область знаний

Продуктовая система реализует одну сущность `KnowledgeSpace` — область
знаний с:

- стабильным идентификатором;
- одним владельцем;
- одной предметной областью;
- одной доверенной зоной;
- одной политикой доступа;
- одной политикой хранения;
- ссылкой на управляемый Git-репозиторий;
- состоянием.

Модель и клиент не назначают владельца, область или политики. Их вычисляет
сервер по актуальной личности, роли и предметному доступу.

Личная область создаётся при первом подтверждённом личном знании. Область
проекта принадлежит команде. Область мероприятия создаётся лениво, только
когда подтверждено долговременное знание. Это разные экземпляры одной
сущности, а не закрытые типы или отдельные реализации.

Пользователь видит одну логическую поверхность, составленную из текущего
диалога, актуальных предметных данных, личной области и не более необходимой
текущей проектной или событийной области.

Опубликованная игра не получает второго пакета знаний. Помощник читает
неизменяемый опубликованный пакет напрямую. Разрешённые страницы проектной
вики могут стать обычными ресурсами публикации.

### 3.3. Источники истины

- PostgreSQL каноничен для точного текста и порядка диалога.
- Текущая доверенная Git-ветка канонична для подтверждённого синтеза.
- Identity/RBAC, Game Repository, Portal и Runtime API каноничны для
  личности, проекта, лицензии и игрового состояния.
- Неподтверждённое предложение канонично только как временный процесс в
  PostgreSQL и не входит в обычный контекст.

Актуальное предметное состояние сильнее вики. Найденный текст не
предоставляет право, инструмент или системную инструкцию.

### 3.4. PostgreSQL и Git

Поверх существующих разговоров PostgreSQL хранит:

1. `knowledge_spaces` — границы областей и ссылку на Git;
2. `knowledge_changes` — точную предложенную операцию, подтверждение,
   состояние, повтор, аренду исполнителя и итоговый commit SHA;
3. `knowledge_change_sources` — нормализованные ссылки на сообщения и
   доменные события с контрольными суммами и способом использования.

`knowledge_changes` одновременно является предложением и надёжной очередью.
Отдельный outbox первой версии не вводится. Одобренные строки захватываются
через строковые блокировки PostgreSQL с `SKIP LOCKED`.
Короткая транзакция только устанавливает состояние `applying` и ограниченную
по времени аренду; Git-операция не держит открытую транзакцию, а истёкшая
аренда разрешает безопасный повтор.

PostgreSQL не хранит вторую текущую страницу, реестр всех страниц, собственный
активный указатель, поисковый индекс или универсальный граф.

Git `HEAD` является текущей ревизией. Один commit содержит целевые страницы и
сгенерированные `index.md`. Ветка переключается сравнением ожидаемого старого
commit с новым.

### 3.5. История и удаление

Закрытый управляемый Git с полной историей является основным backend
долговременных областей:

- пишет только Cubica;
- обычный агент читает только текущий снимок;
- история открывается отдельным разрешённым инструментом;
- clone, fork и прямые Git-команды пользователям и агентам не выдаются;
- репозиторий и ключ шифрования соответствуют границе доступа и удаления.

Логическое забывание меняет текущую ветку, но сохраняет историю. Физическое
стирание блокирует доступ, переписывает или заменяет репозиторий, удаляет
старые ссылки и журналы Git, очищает производные данные и уничтожает старый
ключ. Редкая операция стирания может переписать или обнулить историю
затронутой области.

Резервные копии после запроса переводятся в режим «не использовать даже при
восстановлении», ключ области немедленно блокируется и уничтожается в
согласованный срок, а физические копии истекают за опубликованный срок.
Конкретные сроки являются вопросом принятия.

### 3.6. Markdown, JSON Schema и OKF

JSON Schema профиля Cubica является источником истины для записи. Страница
обязана иметь:

- `type`;
- `title`;
- `description`;
- стабильный `cubica_id`;
- типизированные `source_refs`.

Владелец, доступ и срок наследуются от области и не дублируются на странице.
Текущая ветка содержит подтверждённые активные страницы; нерешённый конфликт
помечается `state: disputed`.

`index.md` генерируется детерминированно и не редактируется моделью. `log.md`
не вводится: Git является точной историей.

OKF применяется как цель совместимости и экспорта. Расширения Cubica
разрешены. Несовместимость с OKF сама по себе не блокирует возможность,
необходимую для безопасности, происхождения, локальной правки или продукта.
Экспорт обязан явно сообщать о неперенесённом поле.

### 3.7. Происхождение и подтверждение

Ссылки на сообщения различают:

- `evidence` — содержательное основание;
- `wording` — источник формулировки;
- `context` — поясняющий контекст;
- `confirmation` — явное принятие.

Ответ агента может быть `wording`, `context` или частью совместно созданного
результата. Он не является самостоятельным `confirmation` пользовательского
предпочтения.

Пользовательское знание допустимо, если есть содержательная реплика
пользователя либо более позднее явное принятие конкретной формулировки
агента. Роль автора проверяется по PostgreSQL.

Явная однозначная команда «запомни» или поправка может сама подтвердить
точную правку. Неявный вывод агента только создаёт предложение. Доменное
событие подтверждает только заранее разрешённый предметный факт.

### 3.8. Запись и локальность правки

Основным смысловым писателем является текущий разговорный агент. Отдельный
агент синтеза первой версии не вводится.

Каждый ход выбирает `none`, `propose` или `apply_confirmed`. Агент использует
ограниченные инструменты подготовки и разрешения предложения, но не получает
прямой файловой или Git-записи.

Существующий файл изменяется только точными операциями:

- `replace_exact`;
- `insert_before_exact`;
- `insert_after_exact`;
- `delete_exact`.

Полное содержимое допускается только для нового файла.

Операция содержит контрольную сумму файла, точный старый фрагмент, его
контрольную сумму и единственное ожидаемое совпадение. Исполнитель доказывает
неизменность остальных байтов и Git-объектов страниц.

Обычная правка затрагивает не более одной содержательной страницы и
детерминированно сгенерированные оглавления. Многостраничное разделение или
объединение имеет отдельный явный вид и предварительный просмотр. Форматтеры,
сортировка, исправление соседнего стиля и скрытая полная перезапись запрещены.

### 3.9. Конкуренция и восстановление

Предложение закрепляет базовый commit, контрольную сумму цели и `change_id`.

Если изменилась другая страница, а целевая осталась прежней, тот же патч
может примениться к новому `HEAD`. Изменение целевой страницы создаёт
конфликт; автоматическое смысловое слияние запрещено.

Git ref переключается только при совпадении ожидаемого старого commit.
`change_id` записывается в метаданные коммита. Повтор проверяет существующий
коммит и не создаёт второй.

Синхронная попытка применения является основным путём. Детерминированный
фоновый исполнитель повторяет ту же подтверждённую операцию после сбоя, не
вызывая языковую модель.

### 3.10. Противоречия

Сравнение кандидата даёт один из результатов:

1. дубликат — новая страница не создаётся;
2. явная поправка — точечно меняется та же страница и сохраняется её
   `cubica_id`;
3. разные области — утверждения разделяются по `applies_to`;
4. нерешённый спор — страница получает `state: disputed` и не выдаётся как
   факт.

Детерминированно проверяются идентификаторы, ссылки, версии источников,
явные связи и применение к ожидаемой редакции. Полное автоматическое
обнаружение смысловых противоречий свободного текста не обещается.

Актуальное предметное состояние и однозначная поправка владельца могут
разрешить конфликт по заранее заданному правилу. В остальных случаях решение
принимает уполномоченный пользователь, а агент только предлагает правку.

### 3.11. Чтение и поиск

Права вычисляются до перечисления областей и страниц. Контекст собирается от
конкретного к общему:

```text
текущий диалог
  -> текущий проект, игра или мероприятие
  -> текущая роль
  -> небольшой личный профиль
```

Опубликованная игра читается напрямую из закреплённого пакета. Обычный
контекст не включает Git-историю.

Первая версия использует `index.md`, метаданные, прямые ссылки и буквальный
поиск в уже разрешённой малой области. FTS добавляется после измеримых
промахов или превышения бюджета. Векторы добавляются только после доказанного
недостатка FTS.

### 3.12. Обслуживание и развёртывание

Детерминированное обслуживание:

- повторяет одобренные операции;
- согласует Git и PostgreSQL после сбоя;
- удаляет просроченные предложения и временные каталоги;
- проверяет ссылки, источники и сроки;
- генерирует оглавления;
- выполняет задачи физического стирания;
- очищает производные данные.

Оно не вызывает модель, не сжимает, не переформулирует, не переносит и не
разрешает смысловые споры.

Система встраивается модулями в backend текущего хода помощника. Контракты
принадлежат Cubica; CopilotKit/AG-UI остаются заменяемыми адаптерами.
Отдельный сервис не создаётся без измеримой операционной границы.

## 4. Архитектурные инварианты

1. Пользовательские данные не попадают в инженерную вики или её индексы.
2. Инженерные документы не выдаются продуктовым помощникам.
3. Одна пользовательская поверхность не означает общий физический архив или
   общий контекст всех игр.
4. Область имеет одного владельца, одну доверенную зону, одну политику доступа
   и одну политику хранения.
5. Сервер, а не модель или клиент, назначает область.
6. Командный канон принадлежит проекту, а не личной области участника.
7. Опубликованный пакет является каноном опубликованной игры и не копируется
   в отдельную вики.
8. Актуальные предметные данные сильнее памяти.
9. PostgreSQL каноничен для диалога; Git `HEAD` каноничен для подтверждённого
   синтеза.
10. PostgreSQL не хранит второй текущий текст или активный указатель Git.
11. Неподтверждённое предложение не входит в обычный контекст.
12. Ответ агента может быть источником формулировки, но не самостоятельным
    подтверждением пользовательского знания.
13. Роль и контрольная сумма источника проверяются по первичному хранилищу.
14. JSON Schema является источником истины для долговременных контрактов.
15. OKF не блокирует необходимое расширение Cubica.
16. Модель не возвращает полный существующий файл и не редактирует
    `index.md`.
17. Обычная правка меняет одну содержательную страницу и сгенерированные
    оглавления; остальные байты и страницы остаются прежними.
18. Изменение целевой страницы после просмотра создаёт конфликт.
19. Повтор подтверждённой операции не меняет её смысл и не создаёт второй
    коммит.
20. Нерешённое противоречие не выдаётся как факт.
21. Git-история не входит в обычный контекст агента.
22. Логическое забывание и физическое стирание являются разными операциями.
23. Управляемый Git не выдаёт внешние клоны личных и клиентских областей.
24. Фоновое обслуживание не использует языковую модель и не создаёт новый
    смысл.
25. Первая версия не имеет FTS, векторного или графового индекса.
26. Отказ системы знаний не останавливает детерминированную игру.

## 5. Отклоняемые альтернативы

- **Одна общая папка на пользователя со всеми командными и клиентскими
  данными.** Смешивает владельцев, права и сроки.
- **Четыре жёстких типа пакетов и отдельные реализации.** Не дают функции,
  которую нельзя выразить универсальной областью и политиками.
- **Отдельный пакет знаний опубликованной игры.** Дублирует неизменяемый
  опубликованный канон.
- **Постоянно отключить Git-историю личных знаний.** Лишает пользователя
  различий и отката, но не устраняет резервные копии и другие копии.
- **Разрешить прямые Git-клоны личных областей.** Делает сквозное удаление
  неконтролируемым.
- **Хранить только текущие объекты.** Для согласованного снимка, истории и
  отката потребует собственного манифеста и механизма ревизий.
- **Хранить историю патчей в PostgreSQL.** Создаёт вторую содержательную
  историю и те же проблемы удаления.
- **Хранить текущий commit одновременно в PostgreSQL и Git.** Создаёт два
  конкурирующих указателя; повторяемая операция и Git compare-and-swap
  достаточны.
- **Использовать отдельный агент памяти с первого дня.** Добавляет второй
  смысловой проход, стоимость и возможность расхождения с текущим диалогом.
- **Извлекать знания фоново после каждого сообщения.** Создаёт скрытые
  предложения и увеличивает риск ложного профилирования.
- **Запретить любые ссылки на ответы агента.** Теряет совместно созданные
  решения; правильная граница проходит по подтверждению.
- **Разрешить модели возвращать целый файл.** Не позволяет технически
  предотвратить переформулирование неизменённого знания.
- **Считать OKF блокирующим контрактом.** Стандарт сам оставляет хранение и
  предметные расширения производителю.
- **Сразу добавить FTS, векторы, графовую базу или микросервис.** Выигрыш не
  доказан для малого корпуса.

## 6. Последствия

Положительные последствия:

- архитектура сокращается до одной сущности и одного Markdown backend;
- история, различия и откат доступны по умолчанию;
- редкое физическое стирание не лишает всех пользователей истории;
- текущий агент формирует знание в контексте реального диалога;
- совместно созданные ответы сохраняются с честным происхождением;
- неизменённый текст технически защищён от переформулирования;
- Git уже обеспечивает согласованный снимок и сравнение ревизий;
- PostgreSQL не превращается во второй канон;
- опубликованная игра не дублируется;
- поиск и отдельный сервис остаются зависимыми от измерений.

Стоимость и риски:

- управляемый Git для персональных данных требует закрытого доступа,
  контролируемых ссылок, резервных копий и ключей;
- физическое стирание отдельного факта дорого и может переписать историю
  области;
- PostgreSQL и Git требуют идемпотентного согласования после сбоя;
- точечные операции сложнее свободной перезаписи файла, но эта сложность
  непосредственно защищает смысл;
- семантическое обнаружение всех противоречий не гарантируется;
- без начального FTS нужен оценочный набор навигации;
- необходимо определить единую платформенную личность и политику внешней
  обработки текста.

## 7. Вопросы для принятия

Для перевода решения из Draft требуется утвердить:

1. универсальную область знаний и отказ от отдельного пакета опубликованной
   версии;
2. закрытый управляемый Git с полной историей как основной backend;
3. различие «забыть» и «физически стереть», включая допустимое переписывание
   или обнуление истории;
4. сроки областей мероприятий, резервных копий и физического удаления;
5. автоматическое применение однозначной команды или поправки и подтверждение
   всех остальных предложений;
6. перенос личных знаний между ролями;
7. правила внешней обработки пользовательского текста;
8. единую серверную модель личности и ролей.
