## Как понято решение

Ревью архитектуры 2026-07-07 зафиксировало белое пятно: редактор
(`packages/editor-engine` + `apps/editor-web`) — самая проработанная
подсистема по реализации, но у него нет архитектуры доставки внешним авторам
игр как продукта. Решается вопрос: какова целевая модель «хостируемой студии»
— где живёт контент авторов, как авторы входят и изолируются друг от друга,
как игра публикуется игрокам и кто платит за LLM-операции. ADR фиксирует
целевую модель и границы; исполнительные срезы — отдельными задачами.

## Контекст

- `editor-web` спроектирован и работает как локальный инструмент: сессия =
  Git worktree (отдельная рабочая копия репозитория) платформенного
  монорепозитория в `.tmp/editor-worktrees/`, preview через allowlisted
  `contentSourceId`, Save = коммит в ветку сессии (ADR-036 Phase 9, ADR-042).
- ADR-042 §7 уже фиксирует production-модель сессий: durable-хранилище
  метаданных, изолированные worktree-хранилища, неизменяемые платформенные
  релизы (`platformReleaseId`), автоматическая сборка мусора, запрет
  публичному игроку читать код worktree. Настоящий ADR достраивает вокруг
  этой модели продуктовые слои, не меняя механику сессий.
- Продуктовые функции Game Editor из `PROJECT_OVERVIEW.md` (создание игры по
  описанию, генерация макетов, публикация в каталог) не имели архитектурного
  якоря; черновик `docs/architecture/game-ui-mockup-flow.md` ждёт оформления.
- Отраслевые образцы подтверждают выбранное направление: Git-backed CMS
  (Decap/Tina: черновик = ветка, публикация = слияние), облачные среды
  разработки (Gitpod/Coder: жизненный цикл рабочих пространств, тайм-ауты,
  автоматическая очистка — совпадает с ADR-042), игровые студии-продукты
  (Roblox/GDevelop: облачные ассеты по стабильным ID — совпадает с ADR-063;
  публикация неизменяемых версий «в одну кнопку» как продуктовое ядро).

## Термины

- **Студия (authoring studio)** — хостируемый редактор вместе с его
  preview-сервисами как продукт для авторов игр.
- **Арендатор (tenant)** — организация-клиент платформы (команда авторов).
  **Мультиарендность** — обслуживание многих арендаторов одним развёртыванием
  с изоляцией их данных.
- **Authoring-контур / delivery-контур** — два раздельных развёртывания:
  авторский (студия + preview-экземпляры runtime/player) и игроцкий
  (портал + публичный runtime + публичный player).
- **Провайдер идентификации (IdP)** — сервис, который хранит учётные записи
  и выдаёт токены входа (протокол OIDC; токен — подписанный JWT с полями
  пользователя, арендатора и ролей).
- **Опубликованная версия** — неизменяемый контент-адресуемый (URL включает
  хэш содержимого) снимок собранной игры.
- **Шлюз LLM** — промежуточный сервис между Agent Runtime и провайдерами
  LLM, который считает бюджеты, применяет лимиты и ведёт журнал аудита.

## Решение

1. **Два контура вместо одного приложения.** Студия развёртывается как
   отдельный authoring-контур, физически и сетево отделённый от
   delivery-контура. Инвариант ADR-042 §8.1 («публичный игрок никогда не
   загружает код worktree») становится границей развёртывания, а не только
   правилом кода: у публичного runtime-api нет ни маршрутов, ни файлового
   доступа к авторским хранилищам.

2. **Контент автора живёт вне платформенного монорепозитория.**
   - Целевая модель: **Git-репозиторий на проект автора**, создаваемый из
     платформенного шаблона. Содержимое — то, что сегодня лежит в
     `games/<id>/` (authoring-манифесты, ассеты по ADR-063, плагины по
     ADR-037, `published/`), плюс манифест проекта `project.json`
     (`gameId`, `platformReleaseId`, версии схем).
   - Платформенный код в репозиторий автора не попадает. Совместимость
     проекта с платформой обеспечивается привязкой к `platformReleaseId`
     и операцией upgrade — механика уже принята в ADR-042 §5.
   - Промежуточная ступень **Б0**: один «контентный репозиторий» на
     арендатора (каталог проектов без платформы). Дешевле в управлении,
     пока авторов мало; переход к репозиторию-на-проект — механическое
     выделение истории подкаталога.
   - Канонические игры платформы (`antarctica`, `simple-choice`,
     `ai-driven-choice` и др.) остаются в монорепозитории: они —
     CI-фикстуры платформы. Вынос касается авторских проектов.
   - Владелец хранилища проектов и опубликованных версий — **Game
     Repository** (scaffold из LEGACY-0004 получает содержание).
   - Механика сессий ADR-042 не меняется: worktree создаётся от репозитория
     проекта и копирует только проект автора.

3. **Единый вход и роли.** Все API студии аутентифицируются; сессия
   принадлежит `userId` (поле уже есть в ADR-042); токен несёт `tenantId` и
   роли: `author` (правит свои проекты), `reviewer` (читает и согласует),
   `tenant-admin` (состав и квоты арендатора), `platform-operator`.
   Выбор и устройство платформенного IdP (кандидат — Keycloak/OIDC)
   фиксируется **отдельным платформенным ADR о доступе, общим для портала
   (ADR-032) и студии**; настоящий ADR запрещает отдельную систему входа
   только для редактора.

4. **Мультиарендность — общий пул с логической изоляцией.** Одни экземпляры
   сервисов на всех арендаторов; изоляция: `tenant_id` в каждой таблице,
   префикс арендатора в путях хранилищ
   (`<storage>/<tenantId>/<projectId>/...`), проверка принадлежности ресурса
   арендатору из токена на каждом запросе. Квоты арендатора: число активных
   сессий, объём хранилища, бюджет LLM. Выделенные контуры на арендатора —
   только по подтверждённым операционным границам (ADR-017).

5. **Публикация — продуктовое ядро студии.** Конвейер:
   `Save → Validate/Compile (тот же verify-профиль, что в CI) → Publish →
   Catalog`. Опубликованная версия — неизменяемый снимок результатов сборки
   (скомпилированные runtime-манифесты, ассеты, бандлы плагинов),
   адресуемый по хэшу содержимого — расширение уже работающих механизмов
   published-бандлов плагинов (ADR-039) и URL ассетов (ADR-063) на игру
   целиком. Каталог хранит записи версий; портал (ADR-032/033) продаёт
   запуск конкретной опубликованной версии; публичный runtime-api читает
   **только** хранилище опубликованного — черновики для него не существуют.
   Откат — переключение указателя на предыдущую версию.

6. **Экономика LLM.** Все LLM-операции студии и Agent Runtime идут через
   шлюз LLM: бюджеты и лимиты на арендатора/пользователя, обязательные метки
   (`tenantId`, `userId`, `feature`) на каждом вызове, постоянный журнал
   аудита (закрывает направление persisted audit store из ADR-047). Шлюз
   один и тот же для помощника студии и AI-driven runtime (ADR-046) — два
   потребителя делают его платформенным слоем.

7. **Совместная работа.** MVP: одна активная сессия на автора+проект
   (правило дедупликации ADR-042). Следующий слой — асинхронный review
   поверх веток сессий (черновик → проверка → публикация, модель
   Git-backed CMS). Синхронное совместное редактирование (CRDT) — вне рамок
   этого ADR; отдельным решением при подтверждённой потребности (открытый
   вопрос ADR-034 и ADR-057 §11).

8. **Производительность открытия сессии — только принятый кэш.** Тёплый
   старт обеспечивается трёхуровневым кэшем ADR-057 §10: его уровень 2
   ключуется по содержимому и не зависит от worktree, поэтому новые сессии
   проекта стартуют тёплыми. Допустимая операционная надстройка — фоновый
   прогрев кэша после Save/Publish. Отдельные механизмы «шаблонов сессий» /
   prebuild не вводятся.

9. **Судьба scaffold `services/game-editor/`.** Целевой дом серверной части
   студии (API метаданных сессий, конвейер публикации) при выделении по
   операционным границам ADR-017. До этого серверная часть студии живёт в
   `editor-web` (как сейчас), scaffold остаётся пустой целью и не считается
   дрейфом.

## Этапность целевой модели

Ступени — архитектурные ограничения очерёдности, не план работ:

- **Ступень 0 (этот ADR):** модель зафиксирована; студия остаётся локальным
  инструментом; смежные решения (доступ, развёртывание) ссылаются сюда.
- **Ступень 1 «закрытая студия»:** один authoring-контур на VPS, единый
  вход, durable-метаданные сессий в PostgreSQL (то же предусловие, что у
  ADR-059), автоматическая GC, один арендатор (внутренняя команда).
  Не начинается раньше портального тестового запуска (тема 4.6 STRATEGY.md).
- **Ступень 2 «публикация»:** контентные репозитории (Б0 → на проект),
  конвейер публикации, связка с каталогом и порталом.
- **Ступень 3 «много авторов»:** квоты и шлюз LLM, review-workflow,
  мультиарендное усиление.

## Отклонённые альтернативы

- **Монорепозиторий как продуктовая модель для внешних авторов** —
  worktree-сессия копирует всю платформу и чужие игры (Git не разграничивает
  доступ по каталогам), все авторы пишут в одну историю, каждое обновление
  платформы происходит в том же репозитории, где живут авторские сессии.
- **База данных вместо Git для контента авторов** — обнуляет принятые
  инвестиции в worktree/Save/версионирование (ADR-036/042), лишает продукт
  Git-истории и ломает механизм контент-адресуемых published-артефактов.
- **Отдельный desktop-редактор (модель Roblox Studio)** — платформа уже
  вложилась в web-first preview-first редактор; вторая кодовая база без
  потребителя.
- **Синхронное real-time редактирование сейчас** — CRDT-инфраструктура
  дорога и не нужна без подтверждённой потребности; асинхронная Git-модель
  покрывает командную работу авторов.
- **Отдельная система входа только для студии** — два хранилища учётных
  записей и рассинхронизированные права с порталом при одном и том же
  человеке-консультанте.
- **Выделенная инфраструктура на арендатора с первого дня** — стоимость без
  потребителя; противоречит правилу операционных границ ADR-017.
- **Отдельный prebuild-механизм «шаблонов сессий»** — дублирует уровень 2
  принятого кэша ADR-057 §10: создание worktree дёшево, дорога только
  пересборка проекций, и её уже закрывает кэш.

## Последствия

Положительные:

- Белое пятно «редактор как продукт» получает якорную целевую модель; смежные
  ADR (доступ, развёртывание) знают, на что ссылаться.
- Game Repository перестаёт быть пустым scaffold — у него появляется
  содержание (хранилище проектов и опубликованных версий).
- Портал и студия получают один вход и одну модель ролей.
- Игры становятся версионируемыми: сегодня продакшн-игра меняется правкой
  файлов в main без понятия «версия»; с конвейером публикации игроки видят
  только неизменяемые проверенные снимки.
- LLM-затраты авторов становятся измеримыми и ограничиваемыми.

Стоимость и риски:

- Требуется отдельный платформенный ADR о доступе (портал + студия + runtime).
- Появляется хранилище Git-репозиториев авторов: резервное копирование,
  права, жизненный цикл.
- Конвейер публикации требует серверного исполнения verify-профиля
  (валидация/компиляция вне CI платформенного репозитория).
- Метаданные сессий расширяются (`tenantId`), схемы и таблицы растут.
- Переход Б0 → «репозиторий на проект» — миграция с выделением истории.

## Открытые вопросы

- Выбор IdP и детальная модель ролей — отдельный ADR о доступе.
- Порог перехода Б0 → репозиторий-на-проект (по запросу автора, по размеру,
  по факту публикации).
- Хостинг Git-репозиториев авторов: голые репозитории на диске платформы
  или управляемый Git-сервис — решается на ступени 2.
- Черновик `game-ui-mockup-flow.md` («мастер новой игры») оформляется
  отдельным ADR как продуктовая функция студии со ссылкой сюда.
