## 1. Понимание решения

Решение понято так: универсальный (generic) рендерер `player-web` не должен знать
смысл конкретной игры. Сейчас он его знает: рендерер сопоставляет конкретные
идентификаторы кнопок Antarctica, переписывает дерево UI, ветвится по CSS-классу игры
и по русским подписям. Это нарушает чистоту платформенного слоя. Целевое состояние:
какое управление несёт какое действие и какую роль играет блок - объявляется
декларативно в UI-манифесте, а рендерер лишь исполняет эту декларацию.

«Декларативно» здесь значит: данные в манифесте, а не ветка в коде рендерера.

## 2. Контекст

Ревью проекта (`docs/reviews/2026-06-27-full-project-review.md`) зафиксировало
недокументированный дрейф в платформенном рендерере:

- `apps/player-web/src/components/manifest/ui-component-node.tsx:27-85` хардкодит
  `FORWARD_NAV_BUTTON_ID = "nav-right"` и `ADVANCE_BUTTON_IDS = {"btn-advance",
  "btn-finish"}` и переписывает дерево компонентов (переносит `actions.onClick` с
  кнопки advance на nav-кнопку) безусловно на каждом узле (`:154`);
- `:263` - рендер фонового блока управляется проверкой `cssClass.includes(
  "info-screen-shell")`, где `info-screen-shell` - класс UI-манифеста Antarctica;
- `apps/player-web/src/lib/layout-helpers.ts:29-39` - `resolveButtonId` сопоставляет
  русские подписи («журнал», «подсказ», «назад», «вперед») с идентификаторами кнопок;
- `:17-22` - `resolveAreaCssClass` ветвится по структурным классам Antarctica.

Эти идентификаторы и классы приходят из `games/antarctica/ui/web/ui.manifest.json`.
Платформенный рендерер не должен зависеть от значений конкретной игры.

Действующие ADR уже задают направление: ADR-001 (Model/View/Presenter; View не решает
правила), ADR-013 (логический манифест отделён от UI-манифеста), ADR-054 (ui-манифест
владеет presentation, presenter владеет projection), ADR-053 (UI-only поведение
описывается в UI-манифесте). Настоящее решение уточняет это для привязки действий и
навигации.

## 3. Термины

- **Generic renderer** - универсальный рендерер `player-web`, общий для всех игр.
- **Action role** - объявленная в манифесте семантическая роль управляющего элемента,
  например «продвинуть ход вперёд», «назад», «открыть панель X». Роль не зависит от
  конкретного `id` элемента.
- **Nav slot** - объявленная позиция навигации (например forward/back), которой
  манифест может назначить действие.
- **Game-specific signal** - любое значение, осмысленное только для одной игры:
  конкретный `id` кнопки, имя CSS-класса игры, подпись на естественном языке.

## 4. Принятое решение

Принять правило **чистоты рендерера и декларативной привязки действий**:

1. Generic renderer и player lib НЕ содержат game-specific signals: ни конкретных
   `id` кнопок, ни имён CSS-классов игр, ни сопоставления подписей на естественном
   языке с поведением.
2. UI-манифест объявляет семантику управления явными полями (точная схема - предмет
   schema review, но смысловая модель такая):
   - `actionRole` на управляющем компоненте (например `advance`, `back`,
     `openPanel`, с параметром панели при необходимости);
   - `navSlot` или эквивалент для позиции навигации, которой назначается действие.
3. Рендерер связывает действие с управляющим элементом по `actionRole`/`navSlot`, а
   не по конкретному `id`. Перенос действия «advance → forward nav» становится либо
   ненужным (манифест сразу назначает действие нужному элементу), либо выражается
   декларативным правилом манифеста, а не функцией в коде рендерера.
4. Условный рендер оформления (например дополнительный фон) управляется
   декларативным prop компонента в UI-манифесте, а не проверкой CSS-класса игры.
5. Game-specific формы состояния и резолверы (например team/card/opening Antarctica)
   живут в game plugin (`games/*/plugins/*`), а не в общем `apps/player-web/src/lib`.
6. JSON Schema остаётся источником истины для новых UI-полей (ADR-025).

## 5. Граница ответственности

| Данные или поведение | Владелец |
| --- | --- |
| Какой элемент несёт действие «вперёд/назад/advance» | UI-манифест (`actionRole`/`navSlot`) |
| Какое игровое действие выполняется | game-манифест |
| Видимость декоративного блока экрана | UI-манифест (declarative prop) |
| Игровые формы состояния (team/card/opening) | game plugin |
| Подписи на естественном языке | UI-манифест / локализация, не код рендерера |
| Связывание действия с элементом по объявленной роли | generic renderer |

## 6. Schema and runtime constraints

1. Новые поля привязки действий (`actionRole`, `navSlot` или их финальные имена)
   описываются в UI manifest JSON Schema; TS-типы следуют схеме (см. ADR-056).
2. Authoring compiler может добавлять authoring-friendly sugar, но сгенерированный
   runtime UI-манифест обязан валидироваться против runtime-схемы.
3. Generic renderer обязан корректно вести себя при отсутствии объявленной роли
   (безопасный fallback), но не должен компенсировать это game-specific эвристикой.
4. `verify:game-agnostic` расширяется проверкой отсутствия game-specific signals в
   generic player layers.

## 7. Архитектурные инварианты

- Generic renderer не содержит идентификаторов кнопок конкретной игры.
- Generic renderer не ветвится по именам CSS-классов конкретной игры.
- Generic renderer не выводит поведение из подписей на естественном языке.
- Привязка действий и навигации декларируется в UI-манифесте.
- Game-specific резолверы состояния живут в game plugin, не в platform player lib.
- JSON Schema остаётся источником истины для новых UI-полей.

## 8. Отклоненные альтернативы

### Оставить эвристику сопоставления id/подписей в рендерере

Отклонено. Это и есть текущий дрейф: платформа знает конкретную игру, новые игры и
каналы будут расходиться.

### Переопределять навигацию через отдельный per-game renderer plugin

Отклонено как избыточное для проблемы привязки кнопок. Декларативная роль в манифесте
проще, переносима между каналами и не требует исполняемого кода на игру для базовой
навигации. Game plugins остаются для настоящей game-specific логики.

### Локализационная карта подписей в платформенном слое

Отклонено. Связывать поведение с текстом на естественном языке хрупко и нарушает
i18n: смена языка не должна менять привязку действий.

## 9. Последствия

- Требуется расширение UI manifest schema полями роли действия/навигации и миграция
  UI authoring-манифеста Antarctica на эти поля (исполнение - в TSK, не в ADR).
- Из `ui-component-node.tsx` и `layout-helpers.ts` удаляются game-specific ветки.
- Antarctica-резолверы состояния переносятся из player lib в Antarctica plugin.
- `verify:game-agnostic` получает дополнительную проверку чистоты рендерера.
- Новые каналы (Telegram, Phaser) переиспользуют объявленные роли, а не повторяют
  эвристику.
