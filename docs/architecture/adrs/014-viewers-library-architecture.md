## Контекст

Viewer — это клиентский модуль, который читает UI‑манифест (JSON‑описание экранов и компонентов) и отображает его пользователю. Game manifest — JSON‑описание логики и метаданных игры. Платформе нужна библиотека viewers, чтобы переиспользовать клиентские плееры между играми, задавать правила совместимости и снижать риски при расширении функционала.

Дополнительно требуется поддержка клиентских скриптов. Скрипты должны выполняться в браузере и иметь доступ к DOM (Document Object Model — дерево элементов веб‑страницы), но при этом загрузка и использование должны контролироваться платформой. На сервере исполнение скриптов уже изолируется через sandbox (изолированное окружение выполнения кода с ограничениями), однако на клиенте такой же изоляции нет. Поэтому нужен процесс проверки и подписи, а также технические ограничения через CSP (Content Security Policy — политика безопасности контента).

## Альтернативы

1. Полный запрет клиентских скриптов. Плюс: минимальный риск. Минус: отсутствие гибкой клиентской логики.
2. Разрешить скрипты только в sandboxed iframe (iframe — встроенная веб‑страница с ограничениями через атрибут `sandbox`). Плюс: лучшая изоляция. Минус: ограниченная интеграция с DOM и UI‑компонентами.
3. Разрешить скрипты в основном контексте с доступом к DOM после проверки платформой и подписи. Плюс: максимальная гибкость и совместимость с текущими UI. Минус: повышенные риски, которые нужно закрывать процессом проверки.

## Решение

Принять вариант 3 с процедурой проверки и подписи.

1. **Размещение viewers:** все viewers хранятся как SDK‑пакеты в `SDK/viewers/`. Игры в `games/*` содержат только контент и манифесты.
2. **Выбор viewer:** game manifest содержит поле `meta.viewer` с `id`, `version`, `channel`. Совместимость проверяется по метаданным viewer.
3. **Метаданные viewer:** каждый viewer содержит файл `viewer.json` с `id`, `version`, поддерживаемыми версиями схем и перечнем обязательных возможностей.
4. **Клиентские скрипты:** игра может ссылаться на клиентские скрипты, но viewer загружает их только из реестра скриптов (централизованное хранилище, доступное платформе), после проверки подписи и хеша.
5. **Процедура проверки:** автоматический анализ (поиск опасных API, проверка зависимостей) и ручной обзор. После одобрения скрипт получает цифровую подпись (криптографическая проверка целостности и авторства) и публикуется в реестре.
6. **Исполнение:** скрипты исполняются в основном контексте страницы с доступом к DOM, без iframe. Ограничение источников и сетевых доступов задаётся через CSP. Оценка рисков такого режима выполняется отдельно.

## Последствия

- Положительные эффекты: появляется управляемый путь расширения клиентской логики, сохраняется единый процесс контроля, viewers переиспользуются между играми.
- Риски: повышенный риск на стороне клиента из‑за доступа к DOM, возможная утечка данных и подмена интерфейса; риск зависит от качества проверки.
- Технический долг: требуется отдельная оценка рисков режима без iframe и возможные дополнительные ограничения.
- Процессные изменения: нужна регламентированная проверка скриптов, ведение реестра и ключей подписи.
