## 1. Контекст

Runtime уже проверяет guard — декларативное условие допустимости действия —
перед изменением состояния. Однако клиентский снимок сессии раньше не сообщал
результат этой проверки. Интерфейс мог оставить активной кнопку после
изменения состояния и получить ожидаемый серверный отказ. Это произошло в
фазе новостей «Карт, денег, поездов»: после открытия карты кнопка повторного
открытия оставалась доступной.

Копирование правил в React или Phaser создало бы второй источник истины и
неизбежное расхождение с сервером. Передача полного технического текста guard
раскрыла бы внутренние пути состояния и потенциально секретные значения.

## 2. Решение

1. Каждый публичный снимок сессии после создания, чтения, восстановления и
   принятого действия содержит `actionAvailability` — проекцию всех объявленных
   действий для текущего состояния и доверенной роли сессии.
2. Для каждого действия сервер возвращает только идентификатор, статус
   `available`, `unavailable` или `parameter-dependent` и при необходимости
   стабильный безопасный код причины. Технические выражения, пути и фактические
   значения guard в публичный ответ не входят.
3. Состояние `parameter-dependent` означает, что серверные проверки, не
   требующие пользовательского выбора, пройдены, а окончательное решение
   зависит от параметров команды. Клиент не должен блокировать такое действие
   до выбора параметров.
4. Проекция использует тот же вычислитель guard, что и выполнение действия.
   Проверка при dispatch сохраняется и всегда повторяется по актуальному
   серверному состоянию; клиентская блокировка не является границей
   безопасности.
5. Presenter и delivery-клиенты блокируют `unavailable` и не допускают вторую
   команду, пока ожидается ответ на первую. Phaser получает только признак
   временной блокировки и серверную проекцию внутри обычного снимка; он не
   вычисляет игровые правила.
6. Человекочитаемое объяснение формируется из безопасного кода причины и,
   когда игра объявила более точный текст, из содержимого манифеста. Клиент не
   показывает пользователю внутренний диагностический текст runtime.
7. Изменение является общей возможностью платформы и доказывается нейтральной
   фикстурой без идентификаторов или правил конкретной игры.

## 3. Отклонённые альтернативы

1. **Оставить только серверный отказ после нажатия.** Отклонено: безопасность
   сохраняется, но интерфейс предлагает заведомо невозможное действие и мешает
   ведущему проводить игру.
2. **Повторить guard в игровом React/Phaser-коде.** Отклонено: появляются два
   источника истины, а общая платформа получает игровые исключения.
3. **Передать клиенту guard и вычислять его локально.** Отклонено: контракт
   раскрывает внутреннюю модель состояния, усложняет все каналы доставки и
   может открыть секретные данные.
4. **Удалять недоступные действия без причины.** Отклонено как единственная
   политика: для части интерфейсов скрытие уместно, но контракт должен позволять
   объяснить ведущему, почему ожидаемое действие заблокировано.
5. **Считать параметрическое действие недоступным до ввода всех параметров.**
   Отклонено: это заблокировало бы сам переход к выбору дороги, станции, груза
   или техники.

## 4. Последствия и ограничения

- HTTP-контракт сессии получает обязательное поле; его версия повышается.
- Роль и состояние становятся согласованными между server-rendered проекцией и
  dispatch, а устаревшая вкладка всё равно безопасно получает отказ сервера.
- Для параметрических guard сервер до выбора может доказать только независимую
  от параметров часть. Полная проверка выполняется после отправки команды.
- Первая реализация не создает универсальный мастер выбора объектов. Для него
  используются объявленные схемой ссылки на ресурсы, а отдельные delivery-
  компоненты добавляются по игровым срезам.
- Новые типы guard обязаны поддержать безопасную проекцию либо закрыто
  возвращать `runtime_unsupported`, не имитируя доступность на клиенте.
