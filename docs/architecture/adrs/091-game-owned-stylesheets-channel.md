## Как понято решение

Понимаю задачу так: около 4000 строк стилей конкретной игры «Антарктида»
живут в платформенном `apps/player-web/app/globals.css` (глобальная таблица
стилей платформы). Это нарушение правила 10 CLAUDE.md (game-specific в
платформенном слое; дефект ARC-001). PM выбрал вариант (а): стили игры должны
стать ассетом канала ADR-063 и подключаться рендерером (renderer — компонент,
который превращает декларативный UI-манифест в реальные элементы страницы)
game-agnostic способом, то есть без знания платформой о конкретной игре.

Этот ADR фиксирует **контракт** такого канала: где живут CSS-файлы игры, как
они регистрируются и доставляются, как UI-манифест их подключает, как рендерер
их применяет и как автоматически запрещается возврат игровых стилей в
`globals.css`. Реализация механизма (компилятор, runtime-api, рендерер,
перенос стилей «Антарктиды») выполняется отдельными блоками задачи
`TSK-20260719-antarctica-remediation` (R3) и здесь не описывается — правило 6
CLAUDE.md (ADR — только архитектура, не план исполнения).

## Контекст

- ADR-063 уже ввёл канал игровых ассетов: файлы изображений игры живут в
  `games/<id>/assets/`, описываются декларативным реестром
  `games/<id>/assets/assets.json` (JSON Schema — единый источник истины,
  SSOT — single source of truth), а runtime-api раздаёт их **контент-адресуемо**
  (URL выводится из хэша содержимого: изменение файла автоматически меняет URL)
  по схеме `/game-assets/{gameId}/{assetId}/{sha256}.{ext}`. UI-манифест
  ссылается на такой ассет строковой формой `asset:<id>`.
- Стили — такой же game-owned статический контент, как изображения, и должны
  жить в том же пакете игры, а не в платформенном `globals.css`.
- Прецедент детерминированной публикации уже есть: бандлы плагинов собираются
  скриптом `scripts/manifest-tools/build-player-web-plugin-bundles.cjs` в
  `games/<id>/published/` с контент-адресуемыми именами и метаданными. Тот же
  подход применим к публикации CSS и к переписыванию ссылок на изображения
  внутри CSS.
- PM (2026-07-19) отдельно решил заменить жёсткие числовые лимиты ADR-063
  (512 КБ/файл, 4 МБ/игра, 64 ассета) на значимые рекомендации без конкретных
  цифр — объективную границу сейчас оценить нельзя. Это оформлено поправкой к
  ADR-063 (см. её файл) и учтено в схеме и валидаторе канала.

## Решение

### 1. SSOT: CSS живёт в пакете игры

CSS игры — источник истины — хранится в пакете игры по пути
`games/<id>/assets/styles/*.css` и регистрируется в существующем реестре
`games/<id>/assets/assets.json`. Канал ADR-063 расширяется типом ассета `css`.

Реестр получает отдельную необязательную секцию `stylesheets` (рядом с
`assets`), а не смешивает CSS с изображениями в массиве `assets`. Причины:

- у CSS другая проверка содержимого (ссылки-токены `asset:<id>` вместо
  санитизации SVG) и другой режим доставки (единый файл стилей вместо
  адресуемых по id картинок);
- существующие потребители контракта (`GameAssetEntry` в runtime-api и в
  клиентском резолвере) типизированы как `kind: "image"`; отдельная секция
  добавляет тип `StylesheetAssetEntry` **аддитивно**, не ломая эти потребители
  и не требуя правок платформенного кода в контрактном блоке;
- при этом id изображений и стилей делят одно пространство имён и одну форму
  ссылки `asset:<id>`, поэтому канал остаётся единым.

Каждая запись `stylesheets`: стабильный `id`, относительный путь `file`
(оканчивается на `.css`), `kind: "css"` и обязательное происхождение `origin`
(`authored-in-repo` | `third-party` с `license`/`source`) — как у изображений.
Нормативная форма — `docs/architecture/schemas/game-assets.schema.json`
(`StylesheetAssetEntry`).

### 2. Слой стилей в UI-манифесте

UI-манифест канала (web) получает необязательный верхнеуровневый блок
`stylesheets` — массив ссылок `asset:<id>` на зарегистрированные CSS-ассеты:

```json
{ "meta": { … }, "stylesheets": ["asset:antarctica-theme"], "screens": { … } }
```

Блок объявляется в authoring-источнике игры (редактируемый формат, из которого
компилятор порождает `ui.manifest.json`) и проходит компиляцию `compile:manifests`
без ручной правки сгенерированных файлов: компилятор переносит `stylesheets`
из authoring в runtime-манифест как есть. Нормативная форма блока —
`docs/architecture/schemas/ui-manifest.schema.json` (свойство `stylesheets`),
authoring-форма — `docs/architecture/schemas/ui-authoring-v2.schema.json`.

Сквозной проброс `stylesheets` компилятором реализуется в блоке R3; здесь
фиксируется только форма контракта.

### 3. Ссылки на изображения внутри CSS

Внутри CSS-исходника ссылки на изображения задаются **токенами** `asset:<id>`
(например `background-image: url(asset:arctic-background)`), а не путями. На этапе
публикации/компиляции токены детерминированно переписываются в
контент-адресуемые URL канала — по образцу publish-конвейера бандлов плагинов
(`build-player-web-plugin-bundles.cjs`, каталог `games/<id>/published/`). Сам
CSS раздаётся тем же контент-адресуемым каналом, что и остальные ассеты игры.

Разграничение источника и генерата: **исходный CSS в `games/<id>/assets/styles/`
— SSOT**; опубликованный (с переписанными URL, с контент-адресуемым именем) —
**генерат**, который правится только конвейером, не руками (инвариант ADR-063 о
генерируемых артефактах). Запрет на baked-in путь в исходнике (только токены
`asset:<id>`, `data:`-URI, внутренние `#`-фрагменты и `var(--…)`) проверяется CI
(см. `validateCssSource` в `scripts/ci/validate-game-assets.js`).

Механизм переписывания токенов и раздачи CSS реализуется в блоке R3.

### 4. Подключение стилей рендерером

Рендерер player-web подключает объявленные манифестом стили game-agnostic
способом: при монтировании игры инъецирует `<link>` на каждый URL из
`stylesheets`, при размонтировании снимает их. Порядок подключения — **после**
платформенных стилей, чтобы слой стилей игры мог переопределять платформенные
значения, не требуя `!important` и не завися от порядка импортов сборки.

Рендерер не знает о конкретной игре: он читает список `stylesheets` из манифеста
и резолвит `asset:<id>` через тот же индекс канала, что и изображения (ADR-055 —
чистота рендерера). Неизвестный id — стиль не подключается (fail closed), как и
для изображений. Это контракт; код инъекции/снятия — блок R3.

### 5. Охранная граница `globals.css`

Возврат игровых стилей в платформенный `globals.css` запрещён и проверяется
автоматически. Зафиксированы три автоматизируемых признака утечки (реализованы
в `findGlobalStyleGameLeaks`, `scripts/ci/validate-game-agnostic.js`):

1. `url(...)` с baked-in абсолютным путём (например `url("/images/…")`) или
   удалённым адресом — изображения игры доставляются каналом, а не абсолютной
   ссылкой из платформенного CSS;
2. декоративные эмодзи (game chrome, например пингвины «Антарктиды»);
3. `#<id>`-селекторы, чьи id объявлены компонентами какого-либо игрового
   UI-манифеста (собираются из `games/*/ui/*/ui.manifest.json`).

**Механизм включения проверки без «красного» контура до R3.** На момент
принятия ADR стили «Антарктиды» ещё в `globals.css`, поэтому жёсткое включение
проверки сразу сделало бы контур красным для дефекта, который устраняет R3.
Поэтому проверка добавлена сразу, но включается флагом готовности
`ENFORCE_GLOBAL_STYLE_BOUNDARY` (по умолчанию `false`): пока флаг `false`,
проверка печатает список оставшихся утечек как **не валящее** предупреждение
(живой чек-лист для R3); в том же изменении R3, где `globals.css` очищается от
игровых стилей, флаг переводится в `true`, и признаки становятся жёсткими
отказами, блокирующими регресс. Логика детекторов уже покрыта тестами
`scripts/ci/validate-game-agnostic.test.js` на синтетических данных.

## Отклонённые альтернативы

- **(б) Общий stylesheet со стилями всех игр** (единый файл, где собраны стили
  каждой игры) — отклонено PM: это тот же платформенный слой с game-specific
  содержимым, лишь переименованный; не масштабируется, не изолирует игры,
  снова нарушает правило 10.
- **CSS-in-JS в плагине игры** (стили как JS-объекты внутри бандла плагина) —
  смешивает представление с кодом, конфликтует с `dependenciesPolicy:
  "platform-only"` (ADR-037), закрывает стили для DOM-экранов без плагина и
  ломает контент-адресуемое кэширование статического CSS.
- **Runtime-переписывание URL изображений внутри CSS на сервере** (runtime-api
  на лету подменяет `asset:<id>` при отдаче CSS) — добавляет вычисления и
  состояние в горячий путь раздачи, ломает иммутабельный кэш и расходится с уже
  принятой моделью детерминированной публикации (генерат создаётся конвейером
  заранее, как бандлы плагинов).
- **Смешать CSS и изображения в одном массиве `assets` с дискриминатором
  `kind`** — концептуально единообразно, но меняет тип `GameAssetEntry.kind`
  на объединение `"image" | "css"` и ломает типизацию существующих
  потребителей (`services/runtime-api` и клиентский резолвер), требуя правок
  платформенного кода уже в контрактном блоке. Отдельная секция `stylesheets`
  достигает той же цели аддитивно; унификация возможна позже, когда R3
  выровняет потребителей (открытый вопрос R3).

## Последствия

- Платформенная чистота: у стилей игры появляется законное место в её пакете;
  `globals.css` перестаёт быть свалкой game-specific стилей, а охранная проверка
  не даёт им вернуться.
- Единый канал: CSS и изображения игры доставляются одним контент-адресуемым
  механизмом ADR-063, адресуются одной формой `asset:<id>`, кэшируются
  иммутабельно.
- Разделение источник/генерат: исходный CSS с токенами `asset:<id>` читаем в
  ревью и переносим; опубликованный CSS с контент-адресуемыми URL —
  воспроизводимый генерат.
- Рендерер остаётся game-agnostic: он подключает произвольный список стилей из
  манифеста, не зная об игре; порядок «после платформенных» даёт игре
  предсказуемое переопределение без `!important`.
- Аддитивность контракта: тип `StylesheetAssetEntry` и блоки `stylesheets`
  (реестр и UI-манифест) не ломают существующих потребителей; лимиты ADR-063
  переведены в рекомендации (см. поправку ADR-063).
- Ограничение MVP: канал охватывает web-канал доставки; стили для других
  каналов (например telegram) — при появлении потребности, отдельным решением.
- Долг ARC-001/LEGACY-0023 закрывается не этим ADR, а его реализацией и
  миграцией стилей/картинок «Антарктиды» в блоках R3/R4 задачи
  `TSK-20260719-antarctica-remediation`.
