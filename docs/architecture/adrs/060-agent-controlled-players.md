## 1. Контекст

ADR-046 ввёл execution modes `ai-driven` и `hybrid`: агент может быть частью
исполнения игры и возвращать структурированный результат и `CubicaSurface`.
Прямые state effects являются legacy-контрактом до миграции ADR-084; целевой
путь долговременного изменения — опубликованный Game Intent → Mechanics IR.
ADR-047 закрепил safety gates (approval envelope, отклонённый Agent Turn не
меняет состояние, `allowedCapabilities` как runtime allowlist).

Для настольных игр нужен другой, более узкий паттерн — **агент как игрок** (ИИ-оппонент): детерминированная игра исполняется обычным runtime, а агент лишь занимает одно из мест и **выбирает ходы из того же реестра действий, что и человек**. Этого паттерна нет в принятых ADR:

- нечем объявить, что место в сессии управляется агентом;
- нет проекции «какие действия сейчас легальны для этого игрока» — агенту (и, кстати, UI человека) нужен перечислимый список;
- нет правила, кто и когда инициирует ход агента;
- нет политики отказа для случая «агент вернул невалидный ход».

## 2. Решение

### 2.1. Агент — это участник, а не режим игры

- ИИ-оппонент моделируется как участник сессии с `kind: "agent"` в модели participants из ADR-059 §2.3. Привязка выполняется **на уровне сессии** (при создании/занятии места), а не в манифесте игры.
- Манифест может лишь декларировать готовность: `config.players.agentSeats` (допустимо ли агентское место и сколько) и failure policy по ADR-046. Игра с агентскими местами остаётся `deterministic` или объявляет `hybrid` — новый execution mode **не вводится**.
- Game-манифест один и тот же для партий с агентами и без них.

### 2.2. Платформенная проекция «доступные действия»

- Runtime использует actor-scoped каталог ADR-079/084. Неизменяемые action
  definitions кэшируются по `bundleHash`, а динамическая availability содержит
  `actionId`, статус/options и `basisStateVersion` только для разрешённой
  проекции конкретного actor.
- По критерию ADR-083 возможность **общая**: она же нужна UI-подсказкам
  человека («какие ходы сейчас можно»), фасилитатору и evaluation-контуру;
  агент — лишь первый потребитель.
- Перечисление уважает персональную проекцию: доступность и Mechanics-
  утверждения вычисляются на сервере, но раскрываемые параметры действий не
  содержат чужих секретов.

### 2.3. Ход агента

- Когда `turn.activePlayerId` указывает на агентское место (или агенту доступна внеочередная реакция — например, ответ на предложение торга), runtime инициирует **системный Agent Turn** (ADR-046 допускает system-initiated turns).
- Вход агента: персональная проекция состояния этого места (та же, что получил бы человек, — строится единственным строителем проекции ADR-059 §2.6), список доступных действий (2.2) и объявленный манифестом контекст (правила/методические материалы).
- Выход агента: **выбранное действие** (`actionId` + schema-validated `params`)
  в контракте `packages/contracts/ai`. Прямые state effects и Mechanics IR от
  агента-игрока запрещены.
- Runtime создаёт и сохраняет стабильный `commandId`; исполнение идёт тем же
  детерминированным путём и через ту же очередь/command receipt, что и ход
  человека. Availability и authorization перепроверяются, отклонённый ход не
  меняет состояние.

### 2.4. Политика отказов

- Невалидный выбор агента → ограниченное число повторов Agent Turn с диагностикой; исчерпание → **детерминированный fallback**, объявленный манифестом (typically «пасовать/завершить ход» — действие, обязанное быть легальным всегда, когда у места есть ход); недоступность Agent Runtime → failure policy ADR-046 (pause / retry / facilitator takeover).
- Партия не должна зависать из-за агента: наличие объявленного fallback-действия — условие готовности (readiness gate) сессии с агентскими местами.

### 2.5. Честность и аудит

- Агент получает ровно ту проекцию, что и игрок-человек этого места: никаких чужих секретов, порядков колод и `state.secret`. Это правило честности (fair-play boundary) обеспечивается тем, что вход агента строит тот же playerView-строитель.
- Каждый ход агента фиксируется в event log и replay-транскриптах (контур ADR-046); evaluation fixtures оценивают легальность и разумность ходов (контур ADR-038).
- Игровая стратегия агента (промт, эвристики) — контент игры/сессии, не платформа: платформа гарантирует только контракты, честность и валидацию.

## 3. Отклонённые альтернативы

1. **Агент возвращает state effects напрямую.** Отклонено: ломает равенство
   «агент = игрок», требует дублирования правил в промте и расширяет поверхность
   атаки. ADR-084 запрещает этот путь и для целевого AI-driven durable state.
2. **Новый execution mode `agent-opponent`.** Отклонено: комбинация «deterministic игра + агентские места» полностью выражается участниками сессии; новый режим раздул бы матрицу совместимости манифестов.
3. **Привязка агента в манифесте игры (фиксированные ИИ-персонажи).** Отклонено для платформенного слоя: состав участников — атрибут сессии; манифест лишь декларирует допустимость. (Игровые «персонажи-боты» с характерами — контент, настраиваемый поверх session binding.)
4. **Клиентская реализация ИИ (бот в браузере).** Отклонено: невозможно гарантировать честность (клиент видит только свою проекцию, но бот в клиенте хотсита видел бы всё) и воспроизводимость; место агента должно обслуживаться серверным Agent Runtime.
5. **Платформенная встроенная стратегия (minimax/эвристики в runtime-api).** Отклонено правилом 10: стратегия зависит от игры; платформа предоставляет только контракты.

## 4. Последствия

### Положительные

- ИИ-оппоненты не требуют изменений игровых манифестов и переиспользуют весь safety-контур ADR-046/047.
- Проекция «доступные действия» улучшает и человеческий UI (подсказки легальных ходов), и тестируемость (генерация случайных легальных партий для fuzz/replay).
- Модель «агент = участник» симметрична хотситу и сетевой игре: те же места, та же очередь.

### Отрицательные / риски

- Перечисление доступных действий на каждый ход — потенциально дорогая
  операция; требуется бюджет и кэш с ключом principal/actor/session/
  `basisStateVersion`/`bundleHash`.
- «Всегда легальное» fallback-действие накладывает требование на дизайн игр с агентскими местами — фиксируется как readiness-условие, о котором должен предупреждать редактор.
- Внеочередные реакции (торги, аукционы) инициируют агентские ходы вне `turn.activePlayerId` — первый срез ограничивается ходами по очереди, реакционные ходы явно помечаются как расширение.
