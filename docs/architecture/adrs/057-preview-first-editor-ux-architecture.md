## 1. Понимание решения

Решение понято так: игровая сущность (карточка, экран, метрика, действие)
физически разложена на фасеты в разных authoring-манифестах — смысл и
содержание в game authoring manifest, вид в UI authoring manifests по каналам, —
но автор должен работать с ней как с единым объектом. Редактор обязан дать эту
целостность на уровне представления и ввода, не создавая нового хранимого
источника истины рядом с authoring-манифестами. Одновременно фиксируются:
универсальный режим текстового редактирования через промт-проекцию, структура
дерева сущностей, разделение осей времени, фикстуры состояния, модель создания
сущностей, диагностический поток и архитектура кэширования редактора.

Детальный источник решений — `docs/architecture/editor-preview-first-ux.md`
(по образцу пары ADR-043 + `agent-ui-foundation.md`). Настоящий ADR фиксирует
архитектурные решения, ограничения, отклонённые альтернативы и последствия.

## 2. Контекст

ADR-034/035/036 сделали редактор projection-based и preview-first; ADR-049
ввёл динамическую YAML-проекцию промта; ADR-052 ввёл in-memory
`EditorEntityProjection` для project-level сущностей. Обсуждение UX-архитектуры
2026-07-04 закрыло оставшиеся решения уровня целостной сущности и рабочего
пространства: как автор видит и редактирует «тройственную» сущность, как
работает текстовый режим для нетехнического автора, как устроено дерево, что
происходит с правками во время прохождения, откуда берутся состояния для
режима «Дизайн», как создаются сущности и почему редактор медленно загружается
(проекции пересобираются с нуля при каждой загрузке).

## 3. Термины

- **Фасет** — грань единой сущности из одного источника: смысл/содержание
  (game manifest) или вид (UI manifest конкретного канала).
- **Промт-проекция** — YAML-представление значимых полей сущности с русскими
  подписями (ADR-049), показываемое автору как «промт элемента».
- **Возвращённое намерение** — текст, который автор вернул после свободного
  редактирования промт-проекции; не обязан быть корректным YAML.
- **Интент** — задание для агента или детерминированного интерпретатора с
  захваченным контекстом (указатели чтения/записи, хеши источников).
- **Вхождение (occurrence)** — появление одной сущности в нескольких местах
  дерева; вхождения ссылаются на одну сущность, а не создают копии.
- **Фикстура состояния** — именованный снимок игрового состояния, задающий
  контекст режима «Дизайн» и стартовую точку прохождения.
- **Невизуальная сущность** — сущность без собственного отображения в UI
  (правило, таймер, системное действие, скрытая метрика).
- **Декоративный элемент** — UI-элемент без игрового смысла, объявленный
  таковым в UI authoring манифесте.
- **Канальный просмотрщик** — облегчённый отрисовщик канальной проекции
  (например, Telegram) внутри редактора, не эмулирующий реальный клиент.

## 4. Принятое решение

1. **Целостность сущности — свойство представления, а не хранения.**
   Единый объект автора — `EditorEntityProjection` (модель чтения);
   единственная модель записи — `EditorChangeSet`, маршрутизирующий каждое
   изменение в правильный манифест. Материализованный editor-manifest как файл
   отклонён (подтверждение ADR-052); файлы допустимы только как одноразовый
   кэш, диагностический снимок или hints-sidecar.
2. **Дисциплина идентичности.** UI-элемент ссылается на `id` игровой сущности;
   направление ссылки — только UI → game. Разрывы связей — диагностики с двумя
   объявляемыми в схеме исключениями: типы, не требующие отображения
   (невизуальные сущности), и UI-элементы, объявленные декоративными.
   Переименование `id`, удаление сущности и извлечение прототипа — атомарные
   рефакторинг-операции, обновляющие все фасеты и ссылки сразу.
3. **Рабочее пространство** — preview-first: предпросмотр является основной
   поверхностью, а дерево сущностей, timeline, диагностика, формы и текстовый
   ввод остаются синхронизированными проекциями. Точная компоновка принадлежит
   UX-спецификации.
4. **Текстовое редактирование через промт-проекцию** — универсальный режим для
   логики, содержания и вида одиночной сущности. Возвращённое намерение — не
   данные для парсера, а команда интерпретатору: детерминированный быстрый
   путь (валидный YAML, изменены только значения известных ключей) → иначе
   агент. Обязательны явный отчёт распознанных и нераспознанных изменений,
   консервативная семантика удалений и `prompt-stale` по хешам источников.
5. **Шкала риска операций** для любого агентского `EditorChangeSet` независимо
   от канала ввода: безопасные → стандартный конвейер; структурные →
   акцентированное резюме; опасные (смена `id`, перенацеливание ссылок,
   удаление сущности со входящими ссылками) → approval envelope (ADR-047).
6. **Дерево сущностей** — одна структура с переключаемой группировкой.
   Несколько вхождений ссылаются на одну сущность и дают одно состояние выбора,
   а не создают копии.
7. **Регион предпросмотра**: агент получает промт, список захваченных
   сущностей и снимок области, если renderer adapter поддерживает
   необязательную возможность «снимок области»; YAML-проекция для региона не
   строится. Снимок и контекст проходят гейты redaction/audit ADR-044.
8. **Две оси времени** разделены: история документа (undo/redo правок) и
   состояние предпросмотра (timeline, runtime-authoritative restore по
   ADR-036). Молчаливая потеря правок недопустима.
9. **Фикстуры состояния** — закреплённые фикстуры становятся проверяемым
   authoring-артефактом `games/<id>/authoring/fixtures/*.json` с собственной
   JSON Schema и диагностикой `fixture-stale`; это решение открытого вопроса
   ADR-036 о долгоживущих именованных сценариях. Runtime/player файлы фикстур
   не читают; состояние передаёт редактор в preview-only restore.
10. **Создание сущностей** — атомарная кросс-манифестная операция, объединяющая
    game- и UI-фасеты одним `EditorChangeSet`. Удаление действует на уровне
    сущности и требует явной политики входящих ссылок.
11. **Параллельность правок** — очередь интентов с оптимистичной
    конкуренцией: конфликт определяется пересечением журнала с объединением
    множеств указателей чтения и записи интента; сущности не блокируются;
    отмена выполняющегося интента обязательна.
12. **Диагностический поток** связывает диагностику с сущностью и полем, а
    исправления представляет как проверяемые ChangeSet. При ошибке компиляции
    последний валидный preview не выдаётся за актуальный.
13. **Кэширование редактора** допускается для проекции, компиляции и
    валидаторов, если ключ учитывает все входы. Кэш никогда не является
    источником истины или входом runtime.

## 5. Архитектурные инварианты

- Authoring-манифесты остаются единственным редактируемым источником истины;
  проекция, кэш, фикстуры и журнал не становятся source of truth.
- Все durable изменения проходят один конвейер
  `EditorChangeSet → шкала риска → dry-run → validation → undo journal`;
  независимые агентские стеки запрещены.
- Runtime-api, player-web и manifest compiler не импортируют editor-engine и
  не читают проекцию, кэш, фикстуры, журнал или editor layout.
- Core editor-engine остаётся game-agnostic и renderer-agnostic; признаки
  «требует отображения» и «декоративный» задаются декларативно в схемах, а не
  в коде.
- Любой вход проекции, не учтённый в ключе кэша, — ошибка проектирования
  кэша; несовпадение/повреждение кэша ведёт к молчаливой пересборке.
- Текстовый режим никогда не применяет пользовательский текст механическим
  парсингом; молчаливое игнорирование фрагментов ввода запрещено.
- Undo действует только на ось документа; восстановление предпросмотра — только
  на ось прохождения; UI не смешивает их словарь.
- Снимки области и agent context, уходящие внешнему провайдеру, проходят
  гейты переносимости и аудита ADR-044.

## 6. Отклоненные альтернативы

- **Материализованный editor-manifest** (компилируемый из authoring-манифестов
  файл как рабочий документ редактора). Отклонено: третий источник истины,
  дрейф, конфликты, обязательные drift-проверки.
- **Editor-manifest как источник** с генерацией game/UI манифестов из него.
  Отклонено: ломает канальную заменяемость UI, независимое ревью логики и
  вида, укрупняет диффы; переворачивает цепочку ADR-030 → ADR-052.
- **Постоянное дублирование сущностей в дереве** (прототипы сверху, под каждым
  повторена UI-структура). Отклонено: повтор узлов конфликтует с интуицией
  «место = вещь» у нетехнических пользователей (уроки iTunes-плейлистов,
  отказа Google Drive от многородительских файлов); принята переключаемая
  группировка с маркированными вхождениями.
- **YAML-проекция для региона выделения.** Отклонено: контракт региона —
  промт + список сущностей + необязательный снимок области.
- **Механическое применение возвращённого намерения парсером.** Отклонено:
  автор не обязан знать YAML; текст — команда интерпретатору.
- **Обязательный вызов LLM для каждой текстовой правки.** Отклонено: тривиальные
  правки значений идут детерминированным быстрым путём с тем же конвейером
  безопасности.
- **Блокировка сущности на время работы агента.** Отклонено: конфликт
  решается при применении (оптимистичная конкуренция), а не запретом ввода.
- **Полный запрет смены `id` в текстовом/агентском режиме.** Отклонено в
  пользу рефакторинг-операции «переименовать с обновлением ссылок» за
  approval envelope: голый запрет толкает к копиям-сиротам.
- **Пустой экран предпросмотра при сломанной компиляции.** Отклонено:
  показывается последний валидный снимок с явной плашкой.
- **Три равноправные вкладки панели (Промт/Чат/Параметры).** Отклонено в
  пользу композиции: форма — тело панели, промт-строка приколота снизу,
  текстовый режим «источник» — за иконкой.

## 7. Последствия

Положительные:

- автор работает с целостной сущностью, не выбирая файлы; ошибки связей
  становятся диагностиками, а не сюрпризами;
- нетехнический автор получает безопасный текстовый/агентский путь правок с
  отчётностью и undo; технический — точные JSON/Monaco инструменты;
- правки при идущем прохождении и сломанная компиляция перестают быть
  источниками молчаливых потерь;
- фикстуры дают воспроизводимый контекст дизайна и основу будущих
  replay/eval-проверок (ADR-038);
- тёплая загрузка редактора перестаёт пересобирать проекции с нуля.

Trade-offs и стоимость:

- новые слои (интерпретатор, очередь интентов, риск-политика, кэш) расширяют
  editor-engine; предпосылка — модуляризация (LEGACY-0018/0019);
- схемы получают новые декларативные признаки («требует отображения»,
  «декоративный») и новую `state-fixture.schema.json`;
- закреплённые фикстуры добавляют проверяемые файлы в репозиторий игры;
- телеметрия интерпретатора и кэша становится обязательной частью качества;
- дублирующая маркировка вхождений и переключаемая группировка усложняют
  TreeViewModel (узел ≠ сущность).
