## 1. Контекст

Модель мультиплеера принята в ADR-011 (free-form с очередью событий `session_events`, версионность `state_version`, broadcast через View Adapters) и хранение сессий — в ADR-005 (PostgreSQL, JSONB, блокировки). Оба ADR писались под целевую распределённую архитектуру (Router / Game Engine / Web Gateway), которая по ADR-017 отложена: канонический backend — модульный монолит `services/runtime-api`.

Фактическое состояние на 2026-07-05:

- сессии хранятся в `InMemorySessionStore` (зарегистрированный долг, `TSK-20260518-session-persistence-hardening`);
- сессия создаётся с одним необязательным `playerId`; модели «участники сессии» нет;
- WebSocket/стриминга обновлений в `runtime-api` нет — клиент получает состояние ответом на своё действие;
- принятый пакет возможностей пошаговых игр (ADR-058, позднее уточнённый
  ADR-084) вводит состояние «на игрока» и turn flow, но сам по себе не решает,
  **откуда** берётся аутентичный actor в сетевой игре.

Нужны решения, которых нет в ADR-011: где живёт очередь и обработчик в монолите, как игроки подключаются к сессии и получают идентичность, каким транспортом доставляются обновления, как строится персональная проекция состояния.

## 2. Решение

### 2.1. Размещение: модуль монолита, без выделения сервисов

Мультиплеер реализуется внутри `services/runtime-api` (session module + новый delivery module). Router/Web Gateway не выделяются (ADR-017); внешний контракт документируется как current API по ADR-051. Логическая схема ADR-011 (очередь → последовательная обработка → транзакционное применение → рассылка) сохраняется полностью, меняется только физическое размещение.

### 2.2. Предусловие: PostgreSQL-хранилище сессий

До включения сетевого мультиплеера `InMemorySessionStore` заменяется PostgreSQL-хранилищем по ADR-005 (`game_sessions` + `state_version`, `last_event_sequence`). Таблица `session_events` хранит только подтверждённые неизменяемые игровые факты из транзакции команды. Если асинхронной доставке понадобится очередь до исполнения, она использует отдельную таблицу `session_command_queue`; ожидающая команда не называется событием и не смешивается с журналом. Однопользовательские игры и хотсит переходят на то же хранилище — одна модель персистентности для всех режимов.

### 2.3. Модель участников (participants) и подключение

- Сессия получает **места** (seats) по `config.players` манифеста: `session.participants = [{ seatId, playerId, kind: "human" | "agent", joinState }]`.
- Идентичность игрока присваивает runtime при занятии места; клиент не выбирает свой `playerId`.
- Занятие места — по **join-токену**: для портальных запусков токены выдаёт launch session по ADR-033 (multiplayer = shared binding, у каждого участника свой токен одной runtime-сессии); для локальной разработки и превью редактора токены выдаёт создание сессии.
- **Хотсит — вырожденный случай той же модели**: одно подключение владеет
  всеми местами (`joinState: "local"`) и получает стабильную идентичность
  principal контроллера сессии. Actor определяется runtime по правилам игры
  при первом принятии `commandId` и сохраняется в квитанции. В сетевой модели
  principal выводится из аутентифицированного участника соединения, после чего
  runtime отдельно проверяет, каким actor ему разрешено действовать.

### 2.4. Очередь и последовательная обработка

- Текущий HTTP-путь обрабатывает команды **строго последовательно на сессию** внутри repository transaction и блокировки сессии. Будущий асинхронный транспорт может сначала записать аутентифицированный закрытый конверт команды в `session_command_queue`, после чего воркер монолита возьмёт его под той же блокировкой. Подтверждённые события появляются в `session_events` только вместе с состоянием и квитанцией после исполнения; статусов `pending/processing` у игрового факта нет.
- Каждый публичный `POST /actions` обязан передавать `commandId` и
  `expectedStateVersion` —
  версию снимка, на основании которого клиент выбрал действие. Runtime сравнивает
  версию с текущей **внутри той же исключительной блокировки сессии и до
  вызова игрового обработчика**, но только после поиска command receipt.
  Совпавший `commandId` возвращает прежний результат без повторного исполнения;
  новый `commandId` с устаревшей версией получает HTTP 409, не создаёт принятого
  игрового события, не применяет команду и не изменяет снимок. Если очередь уже
  создала техническую запись команды, она завершается как `rejected` и не
  транслируется клиентам как изменение состояния; стабильный результат отказа
  сохраняется в command ledger.
- Конфликты — «кто успел первым»: проигравшая команда отклоняется проверкой с управляемой ошибкой (ADR-011 §2.6). Никакого слияния состояний.
- Таймауты `processing`, лимит попыток и архивация относятся только к будущей `session_command_queue`. Подтверждённый журнал `session_events` неизменяем.

### 2.5. Транспорт обновлений: WebSocket в runtime-api

- `runtime-api` получает WebSocket endpoint подписки на сессию: `session_id + join-токен → поток обновлений`.
- Каждое сообщение несёт `state_version`, `last_event_sequence` и **персональную проекцию** (см. 2.6); правила применения на клиенте — по ADR-011 §2.7 (устаревшие версии игнорируются).
- При реконнекте клиент запрашивает полную проекцию текущего состояния (доставка дельт с версии N — возможная оптимизация, не обязательство MVP).
- Контракт сообщений описывается схемой в `packages/contracts/session` и документом протокола рядом с OpenAPI (ADR-051: current contract, не future spec). Telegram и другие каналы позже подключаются через View Adapters (ADR-006 остаётся target-ориентиром).

### 2.6. Персональная проекция (playerView)

- Player-facing content boundary (ADR-019) расширяется параметром «наблюдатель» (`viewerPlayerId`): проекция включает `public`, собственную ветку `players.<viewer>` целиком, чужие ветки — только по правилам видимости `playersTemplate.visibility` (ADR-058 §2.3); `secret` и скрытые порядки колод не покидают runtime никогда.
- Один и тот же строитель проекции обслуживает REST-ответы, WebSocket-рассылку и (в ADR-060) проекцию для агента-игрока — это единственная точка фильтрации секретов.

## 3. Отклонённые альтернативы

1. **Выделение Router/Web Gateway в отдельные сервисы сейчас.** Отклонено ADR-017: операционные границы не подтверждены; модель ADR-011 полностью реализуема в монолите.
2. **Сторонний realtime-фреймворк (Colyseus, PartyKit и т.п.).** Отклонено: вводит второго владельца игрового состояния и собственную модель комнат, конфликтуя с ADR-018/ADR-019 (state ownership у runtime-api) и контрактной дисциплиной ADR-051.
3. **CRDT/OT-слияние одновременных действий.** Отклонено ADR-011 §2.6: для пошаговых и free-form игр достаточно последовательной очереди и guard-отклонений.
4. **Long-polling как основной транспорт.** Отклонено: пошаговым играм нужны своевременные уведомления «твой ход»; polling допустим только как деградация клиента, не как контракт.
5. **Клиентский выбор собственного playerId.** Отклонено: идентичность — атрибут места, выданного runtime по токену; иначе невозможна честная адресация `{{actor}}`.
6. **Отдельная модель хранения для мультиплеера (память для single, БД для multi).** Отклонено: двойная персистентность — источник расхождений; долг `InMemorySessionStore` и так подлежит снятию.
7. **Считать проверку версии достаточной защитой от повтора.** Отклонено
   уточнением ADR-084: после потерянного успешного ответа версия уже изменилась,
   поэтому без command ledger клиент не отличит выполненную команду от
   конфликта. Ledger хранится транзакционно вместе с изменением сессии.

## 4. Последствия

### Положительные

- ADR-011/ADR-005 получают реализуемый план в текущей архитектуре без ревизии самих решений.
- Единая модель участников покрывает хотсит, сетевую игру и агентские места (ADR-060) — игровые манифесты не меняются между режимами.
- Единственная точка фильтрации секретов снижает риск утечки чужого состояния.
- Потерянный ответ и повторная отправка оплаты, карты или движения больше не
  приводят к двойному применению: `commandId` возвращает исходную квитанцию, а
  версия разрешает только конкуренцию разных команд.

### Отрицательные / риски

- Монолит получает stateful-компонент (WebSocket-подключения): рестарт процесса рвёт соединения — клиенты обязаны уметь реконнект с полной ресинхронизацией.
- Появляется эксплуатационная зависимость от PostgreSQL уже в MVP-контуре (миграции, бэкапы).
- Advisory-lock-модель ограничивает пропускную способность одной сессии одним воркером — приемлемо для настольных игр, но фиксируется как осознанная граница.
- Клиент обязан хранить последнюю подтверждённую `stateVersion` и незавершённый
  `commandId`. После HTTP 409 новый игровой выбор требует ресинхронизации;
  транспортный retry той же команды безопасно использует прежний `commandId`.
