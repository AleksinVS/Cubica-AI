# ADR-062: Класс игр «клиентская симуляция реального времени» и Phaser-канал доставки

- **Status**: Accepted (2026-07-06)
- **Date**: 2026-07-06
- **Author**: AI Agent (по заказу владельца проекта)
- **Context**: Realtime simulation game class / player-web delivery channel / plugin architecture

> [!IMPORTANT]
> Поправка ADR-084: граница «клиентское реальное время — серверные раунды» и
> Phaser-канал сохраняются. Описанные ниже `random.seed`, guards и effects —
> историческая исполнительная форма: зерно выдаёт закреплённый модуль
> случайности, параметры проверяются входным допуском и типизированными
> Mechanics assertions, а состояние меняют Mechanics commands. Старый
> effects/JsonLogic executor не восстанавливается.

## 1. Контекст

Cubica готовит первый класс игр «симулятор потока» (real-time flow simulation —
производственные, складские, логистические тренажёры: игрок в реальном времени
управляет потоком объектов на 2D-сцене). Текущий канонический web-плеер
(`apps/player-web`) рендерит дискретные состояния интерфейса (экраны, карточки,
панели) через DOM/React и не имеет игрового цикла (game loop — непрерывная
перерисовка сцены десятки раз в секунду), необходимого этому классу.

Phaser.js (2D-движок canvas/WebGL-игр для браузера) давно заявлен в целевой
архитектуре как клиентский канал (`PROJECT_OVERVIEW.md` §7, ADR-003 — расширения
рендерера `x/y/texture`, ADR-036 — требование renderer adapter для превью,
`packages/contracts/ai` — проекция `projectSurfaceForPhaser`), но канал доставки не
реализован и не имеет принятой модели включения в игры.

Ограничения, которые нельзя нарушить:

1. **Runtime остаётся детерминированным и manifest-driven** (ADR-018, ADR-024,
   ADR-040): модульный монолит — не realtime-игровой сервер; серверного тика с
   частотой кадров не будет.
2. **Политика плагинов** (ADR-037): игровые плагины не имеют npm-зависимостей
   (`dependenciesPolicy: "platform-only"`) — Phaser не может быть зависимостью
   плагина игры.
3. **Чистота рендерера** (ADR-055): generic-слои `player-web` не содержат
   game-specific логики; привязки объявляются декларативно в UI-манифесте.
4. **Воспроизводимость** (ADR-038, ADR-058 §2.1): игровая случайность — только
   через PRNG с зерном; `Math.random` в игровой логике запрещён.

## 2. Решение

### 2.1. Модель класса: клиентское реальное время, серверные раунды

- **Реальное время живёт в клиенте, runtime владеет границами раундов.** Игровой
  манифест описывает смысл (этапы, раунды, метрики, правила подсчёта, методику);
  Phaser-сцена — это View внутри плагина игры; Presenter на границе раунда
  отправляет обычные детерминированные действия манифеста.
- Раунд оформляется двумя действиями манифеста (конвенция, не новый примитив):
  «старт раунда» (runtime фиксирует зерно раунда и переключает экран) и «фиксация
  итогов раунда» (клиент передаёт агрегаты через параметры действия по ADR-061;
  guards проверяют инварианты; эффекты пересчитывают метрики и двигают сценарий).
- **Микросостояние симуляции не является session state**: позиции спрайтов,
  таймеры и анимации не покидают клиента и не сохраняются. Session state — только
  результаты, зафиксированные действиями (граница ADR-054: game manifest owns
  meaning).

### 2.2. Детерминизм симуляции: зерно раунда

- Новый эффект `random.seed` (в реестре эффектов рядом с `random.roll` из
  ADR-058): генерирует 128-битное hex-зерно из PRNG сессии и записывает строкой в
  объявленный путь `state.public`. Реализуется поверх PRNG-модуля ADR-058 §2.1 и
  наследует его replay-гарантию.
- Клиентская сцена обязана выводить **всю** игровую случайность (план появления
  объектов, их типы) из зерна раунда через платформенную утилиту детерминированного
  PRNG (тот же алгоритм `xoshiro128**`, предоставляется через plugin API). Это
  делает раунд воспроизводимым: одно зерно → один план раунда.

### 2.3. Граница честности (принятое ограничение MVP)

- Итоги раунда считает клиент; сервер проверяет их **схемой параметров**
  (ADR-061) и **guard-инвариантами** (например: `обработано + пропущено =
  объявленному числу объектов раунда`), но не пересимулирует раунд.
- Для однопользовательских обучающих игр это принятая граница: подделка
  собственного результата тренинга не является угрозой платформе. Ограничение
  документируется в манифесте класса; соревновательные и многопользовательские
  применения класса требуют отдельного ADR (серверная ресимуляция по транскрипту
  ввода — зафиксированный кандидат) до их запуска.

### 2.4. Phaser — платформенная возможность, не зависимость игры

- Пакет `phaser` — зависимость **платформы** (`apps/player-web`), версия
  фиксируется в одном месте; загрузка ленивая (динамический `import()` при первом
  монтировании сцены), чтобы игры без симуляций не платили размером бандла.
- Плагин игры получает новую точку вклада `phaserSceneFactory` в `plugin.json`
  (рядом с существующей `gameConfigFactory`). Фабрика сцены получает объект
  `Phaser` **инъекцией** через контекст; прямой `import "phaser"` в коде плагина
  запрещён (сохраняет ADR-037: бандл плагина без зависимостей, версию контролирует
  платформа).
- UI-манифест получает платформенный тип компонента `simulationSurface`
  (декларативная привязка по ADR-055): `player-web` монтирует canvas и передаёт
  плагину `sceneId`, проекцию контента, снимки сессии, фабрику seeded PRNG и
  `dispatchAction`. Никакой игровой логики в generic-рендерере.
- Нормативные контракты (формы, сигнатуры, жизненный цикл) —
  `docs/architecture/flow-simulation-platform-design.md` §4.0.

### 2.5. Превью редактора: ограниченная интеграция

- Phaser-игра запускается в превью редактора как обычный player preview (iframe);
  этого достаточно для авторинга манифестов и методики.
- Полный renderer adapter по ADR-036 (hit-test игровых объектов, selection
  overlay, editor metadata на Game Objects) **сознательно откладывается** и
  регистрируется как техдолг при реализации (правило 9 `CLAUDE.md`): выделение
  объектов Phaser-сцены в редакторе недоступно до соответствующего среза.

### 2.6. Чего решение сознательно НЕ добавляет

- **Серверный realtime-контур** (тики, стриминг позиций) — противоречит ADR-017/
  ADR-040 и не нужен классу: границы раундов достаточно.
- **Мультиплеер класса симуляций** — класс стартует однопользовательским;
  сетевые режимы — после реализации ADR-059 и отдельного решения по честности
  (§2.3).
- **Phaser-проекцию Cubica Surface** (`projectSurfaceForPhaser`) — существующие
  контракты `packages/contracts/ai` не меняются; их потребитель появится при
  подключении AI-driven игр к Phaser-каналу, это отдельный срез.
- **Общий «canvas-плагин произвольного движка»** — точка вклада типизирована под
  Phaser; другие движки (PixiJS и т.п.) — отдельные точки вклада отдельными
  решениями.

## 3. Отклонённые альтернативы

1. **Рендер симуляции на React/DOM.** Отклонено: десятки одновременно движущихся
   объектов и покадровая анимация — не задача DOM-диффинга; результат неигрबельный
   по производительности и по трудоёмкости.
2. **Phaser как npm-зависимость плагина игры.** Отклонено ADR-037
   (`platform-only`): версия движка расползлась бы по играм, бандлы плагинов
   выросли бы на мегабайты, обновление безопасности стало бы обходом каждой игры.
3. **Серверная симуляция реального времени в runtime-api.** Отклонено: модульный
   монолит — не игровой сервер (ADR-017); детерминированный manifest-runtime не
   имеет и не должен иметь понятия кадра (ADR-024/040).
4. **Полная серверная ресимуляция для анти-чита в MVP.** Отклонено как
   несоразмерная стоимость для однопользовательских тренажёров; зафиксировано как
   путь для соревновательных режимов (§2.3).
5. **Отправка сырого потока событий ввода вместо агрегатов.** Отклонено для MVP:
   раздувает контракт и хранение без потребителя; агрегаты + инварианты guards
   достаточны, транскрипт ввода — часть будущего решения о ресимуляции.
6. **Ждать реализации Phaser-проекции Cubica Surface и строить канал на ней.**
   Отклонено: Surface-проекция решает задачу AI-driven поверхностей, а не
   детерминированного игрового цикла; смешение задержало бы оба направления.

## 4. Последствия

### Положительные

- Платформа получает канал доставки для целого класса игр (производственные,
  складские, логистические, тайм-менеджмент тренажёры) без изменения модели
  runtime.
- Один платформенный экземпляр Phaser: контроль версии, ленивая загрузка, единый
  контракт сцены.
- Воспроизводимость раундов (зерно → план) совместима с replay-контуром и даёт
  детерминированные e2e-тесты.

### Отрицательные / риски

- Граница честности §2.3 — осознанное ограничение; его нарушение (запуск
  соревновательного режима без ресимуляции) должно блокироваться ревью.
- Phaser-сцены не редактируются визуально в редакторе до среза ADR-036-адаптера
  (регистрируемый долг).
- Появляется двойной исполнитель игровой логики (клиентская сцена + серверные
  эффекты); дизайн-документ обязывает держать в сцене только представление и
  micro-flow, а все правила подсчёта — в манифесте.
