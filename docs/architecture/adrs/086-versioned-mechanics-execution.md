## 1. Понимание решения

Решение понято так: неизменяемый пакет сессии должен закреплять не только
описание правил, но и точную реализацию, которая продолжает исполнять эти
правила после обновления платформы. Runtime обязан выбирать исполнителя по
полной блокировке модуля, а не считать текущий код совместимым только по имени.

Это решение **не замораживает архитектуру платформы и не возвращает удалённые
контуры**. Оно действует только между версиями модулей внутри действующей
архитектуры Game Intent → Cubica Mechanics IR. Удалённый
`deterministic.effects` и другие заменённые способы исполнения не получают
исторический runtime. Платформа продолжает развиваться, новые сессии получают
новые версии, а закреплённое поведение сохраняется лишь для уже начатых
поддерживаемых партий.

До появления этой возможности уже созданные предрелизные тестовые сессии
можно переводить только в архивный режим чтения. Их нельзя удалять,
переписывать или незаметно продолжать на новом коде. Первая
недиспозируемая учебная партия не может быть начата без доступного
многоверсионного исполнения всех закреплённых модулей.

## 2. Контекст

ADR-084 закрепил неизменяемый пакет игры и `moduleLock`, содержащий
`moduleId`, точную `moduleVersion`, `artifactHash` и версии значимых
алгоритмов. Текущий реестр runtime при этом содержит только одну реализацию
каждого модуля.

Любое изменение общего исполняемого корпуса меняет контрольные хеши всех
модулей. После обновления платформы повтор уже завершённой команды можно
вернуть из долговечной квитанции, но новую команду старой сессии невозможно
безопасно исполнить. Остановка при несовпадении защищает данные, но не
обеспечивает продолжение долгой партии.

## 3. Термины

- **Точная блокировка модуля** — тройка
  `moduleId + moduleVersion + artifactHash` вместе с версиями значимых
  алгоритмов.
- **Многоверсионный диспетчер** — доверенный серверный компонент, который
  выбирает зарегистрированный неизменяемый исполнитель по точной блокировке.
- **Корпус модуля** — канонический набор схем, семантических правил,
  исполнителей, алгоритмов и значимых общих зависимостей, по которому считается
  `artifactHash`.
- **Загрузочный конверт** — минимальная стабильная внешняя оболочка
  неизменяемого пакета, из которой без применения текущей полной схемы можно
  безопасно прочитать версию формата и точные блокировки, проверить хеш байтов
  и выбрать соответствующий версионный набор проверок.
- **Недиспозируемая сессия** — партия, которую продукт обещает сохранять и
  продолжать через плановые обновления платформы; её нельзя просто заменить
  новой тестовой сессией.
- **Архивная сессия** — неизменяемая сессия, доступная для просмотра и аудита,
  но не принимающая новые игровые команды.

## 4. Решение

1. Runtime разрешает каждый требуемый модуль по точной тройке
   `moduleId + moduleVersion + artifactHash`. Совпадения только ID или версии
   недостаточно.
2. Зарегистрированная реализация неизменяема. Новое поведение получает новую
   версию или новый хеш; существующая запись не переназначается на другой код.
3. Исполнимый корпус хешируется по границам модулей. Общая зависимость,
   влияющая на поведение нескольких модулей, имеет явную версию и входит в
   хеш каждого зависимого модуля. Изменение несвязанной операции не должно
   объявлять изменившимся другой модуль.
4. Пакет сессии продолжает закреплять `bundleHash`, планы и полный
   `moduleLock`. Диспетчер собирает исполнение только из точных закреплённых
   реализаций; частичное смешивание с текущими версиями запрещено.
5. Загрузка сохранённого пакета двухступенчата. Сначала ограниченная
   schema-backed проверка загрузочного конверта читает только версию формата,
   точные канонические байты, хеш и `moduleLock`. Затем диспетчер выбирает по
   этой блокировке согласованный набор полной JSON Schema, semantic checker и
   executor. Исторический манифест нельзя сначала проверять текущей полной
   схемой: это сделало бы многоверсионный исполнитель недоступным ещё до
   выбора версии.
6. Перед созданием недиспозируемой сессии runtime проверяет доступность всего
   закреплённого набора исполнителей. Отсутствующий модуль блокирует создание,
   а не обнаруживается после начала партии.
7. Старые предрелизные тестовые сессии, для которых точный исполнитель
   недоступен, переводятся в архивный режим без изменения их состояния,
   пакета, журнала и квитанций. Для продолжения проверки создаётся новая
   сессия на новом пакете.
8. Хранилище сессий имеет явную одинаковую по смыслу границу
   `archive/read-archive` для постоянной и тестовой in-memory реализации.
   Архивная сессия не принимает команд, но её неизменяемые пакет, последнее
   состояние, полный журнал и квитанции доступны авторизованному просмотру и
   аудиту. Один только фильтр `archived_at` без безопасного чтения не считается
   завершённым архивным режимом.
9. Автоматическая миграция живой сессии запрещена. Перенос состояния на другой
   пакет является отдельной явной операцией с собственной схемой,
   предварительной проверкой, журналом происхождения, результатом и
   полномочием владельца.
10. Удаление исторического исполнителя допускается только после истечения
   объявленного срока поддержки и архивирования всех зависимых сессий либо
   завершения их явных миграций.
11. Гарантия не распространяется автоматически на заменённую архитектуру
   исполнения. Переход на иной публичный язык или иную границу доверия требует
   отдельного решения о сроке поддержки, архивировании или явной миграции; он
   не обязывает восстанавливать удалённый production-контур.

## 5. Инварианты безопасности

- Хеш проверяется по каноническим байтам зарегистрированного корпуса до
  исполнения.
- Загрузочная схема не допускает планы, произвольные пути или исполняемые
  значения: она только безопасно разрешает версию полного валидатора.
- Историческая реализация не загружается из данных сессии и не принимает
  произвольный код из игрового пакета.
- Исполнитель использует тот же admission control, бюджет, транзакцию,
  журнал, квитанции и правила конфиденциальности, что и текущая версия.
- Исправление уязвимости не подменяет молча исторический код: платформа
  блокирует уязвимую версию, архивирует зависимые сессии либо проводит явную
  проверяемую миграцию.
- Точный повтор долговечной квитанции не требует запуска исторического
  исполнителя, но целостность закреплённого пакета всё равно проверяется.
- Наблюдаемость различает «версия доступна», «версия заблокирована» и
  «версия отсутствует» без раскрытия закрытого игрового состояния.

## 6. Отклонённые альтернативы

1. **Всегда исполнять старый пакет текущим кодом.** Отклонено: одинаковые
   данные перестают гарантировать одинаковые правила и replay.
2. **Сравнивать только `moduleVersion`.** Отклонено: один номер не доказывает
   идентичность исполняемого корпуса.
3. **Автоматически переписывать состояние при обновлении.** Отклонено:
   меняются правила уже начатой партии без отдельного решения и аудита.
4. **Навсегда оставлять все старые партии только для чтения.** Допустимо как
   безопасный аварийный режим, но не удовлетворяет продуктовой гарантии
   продолжения долгих учебных сессий.
5. **Хранить один общий хеш всего Mechanics.** Отклонено: несвязанные модули
   получают ложную несовместимость и не могут версионироваться независимо.
6. **Сохранять исполнители всех исторических архитектур.** Отклонено:
   многоверсионность защищает партии внутри принятой архитектуры, но не
   превращает завершённую миграцию платформы в вечный параллельный runtime.

## 7. Последствия

- Долгие сессии смогут продолжаться на точно тех правилах, с которыми были
  созданы, пока их версии находятся в объявленном сроке поддержки.
- Поставка runtime должна сохранять несколько доверенных версий модулей и
  учитывать их в сборке, контроле уязвимостей и удалении.
- Публикация и запуск сессии получают проверку полной доступности
  закреплённого исполняемого набора.
- Версионная поставка включает согласованные JSON Schema, semantic checker и
  executor; сохранение только старой функции исполнения недостаточно.
- Постоянное и тестовое хранилища получают явные операции архивирования и
  авторизованного чтения архива.
- Предрелизная разработка может менять Mechanics без опасной подмены правил:
  старые тестовые партии остаются доказательством истории, а проверка
  продолжается в новых сессиях.
- Любая миграция состояния становится отдельным явным продуктовым действием,
  а не побочным эффектом обновления платформы.
