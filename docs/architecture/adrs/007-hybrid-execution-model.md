# ADR-007: Hybrid Execution Model (LLM + JS Script)

## Status
Accepted

## Context
Изначально архитектура Cubica предполагала обработку всех игровых действий через LLM. Однако этот подход имеет недостатки:
1.  **Высокая латентность:** Простые действия (открыть инвентарь, переключить вкладку меню) занимают секунды.
2.  **Стоимость:** Тратить токены на тривиальную логику неэффективно.
3.  **Недетерминированность:** LLM может ошибиться в арифметике или неверно интерпретировать жесткие правила.

## Decision

Мы переходим к **Гибридной Модели Исполнения (Hybrid Execution Model)**.

### 1. Logic Router & Design Workflow
Игровой движок (Engine) анализирует входящее событие и выбирает обработчик на основе конфигурации в Манифесте (`actions`).

**Процесс принятия решения (Design-Time):**
1.  При генерации игры (в Game Editor) LLM анализирует каждое событие.
2.  LLM предлагает тип обработчика:
    *   `handler_type: "llm"` — для нарратива, диалогов и сложных, нечетких решений.
    *   `handler_type: "script"` — для механик, UI-навигации, математики.
3.  Разработчик (человек) видит эти предложения и утверждает или меняет их.

### 2. Script Execution Environment (External JS Files)
Мы используем **JavaScript (V8/QuickJS)** в изолированной среде (Sandbox).
Для улучшения DX (подсветка, линтинг) и читаемости манифеста, код скриптов **выносится в отдельный файл** (например, `game.js` или `scripts.js`).

*   **Scripts File:** JS-файл экспортирует именованные функции.
*   **Manifest:** Ссылается на файл скриптов (`assets.scripts`) и имена функций в `actions`.

Пример `scripts.js`:
```javascript
export function drinkPotion(state, args, std) {
  const heal = 20;
  if (state.public.inventory.includes('potion')) {
     state.public.hp = Math.min(100, state.public.hp + heal);
     std.inventory.remove('potion');
     std.ui.toast('You feel better!');
  } else {
     std.ui.error('No potion!');
  }
}
```

Пример `manifest.json`:
```json
"assets": {
  "scripts": "scripts.js"
},
"actions": {
  "use_potion": {
    "handler_type": "script",
    "function": "drinkPotion"
  }
}
```

### 3. Безопасность и Изоляция
*   **Sandbox:** Скрипты исполняются в строгой изоляции (без доступа к fs, network). Доступ есть только к `state`, `std` и `args`.
*   **Timeout:** На выполнение скрипта накладывается жесткий лимит времени (например, 100ms).
*   **Atomic State:** Изменения применяются к копии стейта или транзакционно.

## Consequences

### Positive
*   **DX:** Разработчики пишут код в привычных IDE с автодополнением.
*   **Чистота:** Манифест не засоряется экранированным кодом в JSON-строках.
*   **Отзывчивость:** Мгновенная реакция на действия.
*   **Гибкость:** Полноценный язык программирования позволяет реализовать любую механику.

### Negative
*   **Сложность реализации:** Необходимо интегрировать безопасный JS-рантайм и загрузчик модулей в движок.
*   **Файловая структура:** Игра теперь состоит из нескольких файлов, что требует корректной упаковки/загрузки.
