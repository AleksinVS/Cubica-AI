# ADR-018: Source of Truth для логики игры находится в JSON-манифесте

- **Дата**: 2026-03-19
- **Статус**: Proposed
- **Авторы**: проектная команда Cubica / agent-architect
- **Компоненты**: `games/*`, `services/runtime-api`, `packages/contracts/*`, content pipeline, runtime architecture
- **Уточнено:** ADR-083/084 разрешают закреплённое манифестом изолированное
  runtime-расширение конкретной игры, когда декларативных и общих механизмов
  недостаточно

**Изолированное runtime-расширение** — исполняемый модуль, принадлежащий одной
игре и работающий вне процесса платформы с ограниченными правами.
**Content-addressed bundle** — неизменяемый пакет, идентификатор которого
вычислен из его содержимого.

## Контекст

В разработке игры могут одновременно существовать runtime-манифесты,
дизайн-макеты, narrative markdown, authoring-источники и legacy-прототипы.
Если исполнимая логика распределена между этими артефактами, платформа не может
однозначно определить правила, входы действий и ожидаемые переходы состояния.
Дизайн- и narrative-материалы при этом остаются полезными источниками намерения,
но не должны конкурировать с runtime-контрактом.

Ранее в архитектурных обсуждениях допускалась модель, при которой source of truth для логики игры может находиться в markdown-сценарии, а затем интерпретироваться или переноситься в исполнимую форму. Для AI-first и Code-first подхода это создаёт проблему:

- логика оказывается за пределами исполнимых контрактов;
- возникает разрыв между narrative-описанием и runtime-поведением;
- агентам и разработчикам приходится выводить структуру игры из prose-документов, а не из машиночитаемого артефакта;
- schema validation и deterministic runtime упираются в неформализованный источник данных.

## Решение

Принять следующие правила:

1. **Каноническое объявление исполнимой логики игры находится в
   JSON-манифесте**, расположенном в `games/<game-id>/game.manifest.json`.
   Манифест определяет `actionId`, схему параметров, привязку к Mechanics IR
   либо изолированному расширению, точные версии/hashes и запрошенные права;
   runner (отдельный ограниченный исполнитель) запускает закреплённый модуль.
2. **Markdown-сценарии не являются source of truth для исполнимой логики.**
   - narrative markdown может использоваться как optional authoring/reference material;
   - но проект не должен предполагать, что у каждой игры обязательно есть `scenario.md`.
3. **Legacy-прототипы и дизайн-макеты не являются источником истины для
   runtime-логики.** Они могут служить authoring/reference-материалами, но
   устройство runtime и публичных контрактов не должно выводиться из их
   внутренней структуры.
4. **Любая исполнимая игровая логика должна быть объявлена манифестом и
   проверяемыми машиночитаемыми контрактами.** Предпочтительный путь — Mechanics
   IR и общие runtime capabilities. Когда их недостаточно, исполняемый код может
   находиться в content-addressed bundle конкретной игры, но манифест обязан
   закрепить его hash, schema-defined вход/выход, область полномочий и
   изолированный способ запуска. Код не становится частью общего `runtime-api`.
5. **Вопрос о том, как именно заполняется JSON-манифест, не фиксируется этим ADR.**
   - это может быть ручное редактирование, tooling, генерация, AI-assisted authoring, импорт из draft-источников или иная pipeline-модель;
   - но на выходе авторитетное объявление обязано быть JSON-манифестом, а
     исполняемый bundle — неизменяемо связан с ним по hash.

## Последствия

Положительные эффекты:

- логика игры становится доступной для schema validation, тестов и deterministic runtime;
- AI-first архитектура получает машиночитаемый контекст вместо prose как первичного источника;
- code-first подход закрепляется на уровне content layer, а не только backend-кода;
- упрощается дальнейшее развитие `packages/contracts/manifest` и capability-based runtime;
- уникальная логика игры может использовать безопасный исполняемый модуль, не
  перенося предметную ветку в платформенное ядро и не создавая второй
  незакреплённый источник истины.

Ограничения и trade-offs:

- narrative markdown может оставаться полезным, но не считается обязательным артефактом content bundle;
- для сложных игр может понадобиться richer manifest model и tooling, чтобы JSON оставался поддерживаемым;
- публикация игры с runtime-расширением обязана хранить и проверять связку
  manifest/bundle hash, схемы, права и политику изоляции.
