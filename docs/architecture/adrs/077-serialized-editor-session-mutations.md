## 1. Понимание решения

Нужно определить единственную границу, внутри которой редактор меняет файлы,
индекс Git и метаданные рабочей сессии. Атомарная проверка вершины долговечной
истории защищает порядок версий, но не защищает общую рабочую папку от
одновременных Save, Restore, Close или GC. Одновременно старый HTTP-путь записи
без сессии позволяет успешно изменить основной checkout, не создав авторскую
версию.

Решение затрагивает публичное поведение Save и модель конкурентного доступа.
Владелец продукта подтвердил его 2026-07-12. Реализация и сквозная проверка
завершены в `TSK-20260710-editor-author-version-history`; задача перенесена в
архив.

## 2. Контекст

ADR-075 разделил временный черновик и долговечную историю. Реализация корректно
фильтрует проектные пути и атомарно продвигает вершину истории, но независимое
ревью выявило оставшийся разрыв:

- запись файла и последующий Save являются несколькими операциями над одной
  рабочей папкой;
- Restore и GC могут пересечься с такой записью;
- GC использует сохранённый ранее признак «есть несохранённые изменения»;
- `PUT /api/editor/file` без `sessionId` сохраняет прямо в основной checkout и
  обходит историю.

Такое поведение допустимо только как прежний локальный прототип, но
противоречит продуктовому смыслу команды «Сохранить» после ADR-075.

## 3. Решение

1. Любая изменяющая операция authoring-контура обязана иметь действующую
   рабочую сессию. Обычный HTTP Save без `sessionId` отклоняется; прямой
   filesystem adapter остаётся внутренним тестовым механизмом и не является
   пользовательским маршрутом.
2. Save, Restore, Close, принудительное удаление GC и изменение session
   metadata выполняются под одной **арендой сессии** (временным исключительным
   правом одного процесса менять конкретную рабочую сессию).
3. Локальная реализация использует межпроцессную блокировку в каталоге `.tmp/`
   с ограниченным ожиданием и проверяемым владельцем. Блокировка в памяти одного
   Next.js процесса недостаточна.
4. GC получает блокировку, затем повторно читает фактический Git status. Грязная
   сессия без явного `removeDirty` не удаляется, даже если сохранённые метаданные
   устарели.
5. До рекурсивного удаления сервер проверяет, что рабочая папка после
   `realpath` принадлежит разрешённому `.tmp/editor-worktrees` текущего проекта
   и зарегистрирована как editor worktree.
6. Успешное атомарное продвижение долговечной версии является точкой
   фиксации Save. Ошибка последующей записи вспомогательных метаданных не
   откатывает уже созданную версию; состояние сессии сверяется с долговечной
   вершиной при следующем чтении.
7. Хостируемая студия заменяет локальную файловую аренду транзакционной или
   распределённой арендой Game Repository и дополнительно проверяет владельца
   проекта/сессии по ADR-065.

## 4. Инварианты

1. В каждый момент одну рабочую сессию изменяет не более одной операции.
2. Успешный пользовательский Save всегда создаёт долговечную авторскую версию.
3. GC не доверяет устаревшему признаку чистоты перед разрушительным удалением.
4. Метаданные сессии не могут направить удаление за пределы разрешённого
   editor-worktree root.
5. Ошибка после точки фиксации не превращает успешную версию в повторяемый
   «неуспешный» Save.
6. Публичные ответы не содержат stderr Git и абсолютные локальные пути.

## 5. Альтернативы

### A. Оставить Save без сессии как равноправный режим

Отклоняется: создаёт второй смысл команды
«Сохранить» и позволяет обходить долговечную историю. Допустим только отдельный
явно названный служебный маршрут, если появится подтверждённый потребитель.

### B. Блокировка только в памяти процесса

Отклоняется: Next.js может выполнять запросы в разных процессах, а локальный
редактор и GC могут запускаться отдельными командами.

### C. Полагаться только на CAS долговечной ссылки

Отклоняется: CAS защищает порядок версий, но не файлы рабочей папки, Git index
и жизненный цикл сессии.

## 6. Последствия

- Обычный Save становится однозначным: без рабочей сессии он невозможен.
- Одновременный запрос получает ограниченное ожидание или стабильный конфликт,
  а не частично применённое изменение.
- Локальный слой получает небольшую служебную блокировку; её содержимое не
  входит в Git и очищается вместе с временными ресурсами.
- Перед хостируемым многопользовательским запуском локальная блокировка должна
  быть заменена Repository-owned механизмом и проверкой владельца.

## 7. Подтверждение

Владелец продукта принял раздел 3 целиком 2026-07-12. Межпроцессная
сериализация, запрет пользовательского Save без сессии и сквозная корневая
проверка завершены в тот же день; контракт действует в локальном редакторе.
