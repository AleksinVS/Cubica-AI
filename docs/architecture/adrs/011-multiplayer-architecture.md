# ADR-011: Архитектура мультиплеера (Free-form с очередью событий)

- **Status**: Accepted
- **Date**: 2025-11-27
- **Author**: AI Agent
- **Context**: Feature F_00060, Epic E_0030

## 1. Контекст

Изначальная архитектура Cubica (ADR-005) была спроектирована под **одиночные** игры. Каждая сессия (таблица `game_sessions`) принадлежит одному пользователю, а действия обрабатываются последовательно.

**Проблема:** Для поддержки мультиплеера (нескольких игроков в одной сессии) нам нужны:

1. Возможность нескольким пользователям присоединяться к одной игровой сессии.
2. Синхронизация действий и обновлений состояния между всеми участниками.
3. Разрешение конфликтов при одновременных действиях (например, два игрока пытаются взять один и тот же предмет).
4. Широковещательная отправка (broadcast) обновлений UI всем подключённым клиентам.

## 2. Решение

Мы принимаем **свободную модель мультиплеера (Free-form) с очередью событий**.

### 2.1. Модель взаимодействия: Free-form (асинхронная)

Игроки могут действовать **одновременно**. Все действия попадают в очередь событий и обрабатываются Игровым Движком по одному.

- **Очередь событий:** для каждой сессии поддерживается очередь ожидающих действий в порядке поступления (FIFO — first in, first out, «первым пришёл — первым обработан»).
- **Обработка:** Движок (Engine) обрабатывает одно событие за раз (или небольшой пакет, если это поддерживается логикой игры).
- **Оптимистичный UI:** клиент может сразу показать игроку предварительный результат, но «истинное» (авторитетное) состояние определяется только ответом сервера.

**Почему не пошаговая модель?**

- Пошаговая игра — частный случай свободной модели (может быть реализована правилами игры поверх общей схемы).
- Свободная модель более гибкая и подходит как для почти‑реального времени, так и для совместных сессий с несколькими участниками.

### 2.2. Структура состояния игры

Состояние игры расширяется для поддержки нескольких игроков. Логическая структура:

```json
{
  "world": {
    "location": "tavern",
    "items": ["sword", "potion"],
    "weather": "rainy"
  },
  "players": {
    "player_1": {
      "hp": 100,
      "inventory": ["key"],
      "role": "warrior"
    },
    "player_2": {
      "hp": 80,
      "inventory": [],
      "role": "mage"
    }
  },
  "secret": {
    "story_progress": 3,
    "hidden_trigger": true
  }
}
```

- `world` — общий мир (локации, предметы, погода и т.п.), виден всем игрокам.
- `players` — данные конкретных игроков (здоровье, инвентарь, роль). Игрок видит свои данные и публичные части других игроков.
- `secret` — скрытая часть состояния, недоступная игрокам (сюжетные флаги, заметки ведущего и т.п.).

### 2.3. Очередь событий и версионность состояния

#### 2.3.1. Расширение структуры БД

Добавляется таблица `session_events`, в которой хранятся события сессий:

| Column          | Type      | Description |
|-----------------|-----------|-------------|
| `id`            | UUID      | Уникальный идентификатор события. |
| `session_id`    | UUID      | Ссылка на сессию (`game_sessions`). Индексируется. |
| `player_id`     | UUID      | Идентификатор игрока, инициировавшего событие. |
| `sequence`      | BIGINT    | Порядковый номер события внутри сессии (монотонно растущий). |
| `action_id`     | String    | Тип действия (например, `"pick_item"`, `"move"`). |
| `payload`       | JSONB     | Аргументы действия. |
| `status`        | Enum      | `pending`, `processing`, `completed`, `failed`. |
| `attempts`      | INT       | Количество попыток обработки события. |
| `error_code`    | String?   | Код ошибки при неуспешной обработке (опционально). |
| `created_at`    | Timestamp | Время создания события. Индексируется. |
| `processed_at`  | Timestamp | Время завершения обработки (успех или окончательная ошибка). |

Таблица `game_sessions` дополняется полями версионности:

- `state_version` (BIGINT) — номер версии состояния сессии. Увеличивается на 1 при каждом успешно применённом событии.
- `last_event_sequence` (BIGINT) — `sequence` последнего обработанного события (для возможности восстановления и сверки).

#### 2.3.2. Поток обработки событий

Обработка событий должна быть **атомарной**: либо все изменения состояния и статуса события фиксируются, либо не фиксируется ничего.

Логический поток:

1. Клиент отправляет действие → Router записывает событие в `session_events` со статусом `pending`. Для события внутри данной сессии назначается следующий по порядку `sequence`.
2. Компонент Router/обработчик событий выбирает самое старое событие в статусе `pending` для конкретной сессии и переводит его в статус `processing`.
3. Router читает текущее состояние сессии (`game_sessions.state`, `state_version`), передаёт событие в Engine вместе с `player_id` и текущей версией.
4. Engine вычисляет результат (дельту состояния и/или новый вывод для LLM), возвращает Router дельту и информацию об успехе/ошибке.
5. Router внутри одной транзакции:
   - применяет дельту к `game_sessions.state`,
   - увеличивает `state_version` на 1,
   - обновляет `last_event_sequence`,
   - переводит событие в статус `completed` (или `failed`, если логика определила ошибку).
6. После фиксации транзакции Router отправляет обновление всем подписанным клиентам через View Adapters (включая номер версии и `last_event_sequence`).

Этот процесс гарантирует, что состояние сессии и журнал событий (`session_events`) остаются согласованными.

#### 2.3.3. Управление жизненным циклом событий

Чтобы очередь событий оставалась управляемой и не разрасталась бесконтрольно, вводятся базовые правила:

- Событие в статусе `processing`, которое находится в этом статусе дольше заданного таймаута, считается «подозрительным». Следующая итерация обработчика может вернуть его в статус `pending` или пометить для повторной попытки (инкремент `attempts`).
- Количество попыток обработки ограничено (например, 3–5 раз). После превышения лимита событие переводится в статус `failed`, а информация об ошибке сохраняется в `error_code` и логах.
- Старые завершённые события (`completed` и окончательно `failed`) могут либо:
  - храниться ограниченный период (например, N дней) и затем переноситься в архивную таблицу;
  - либо удаляться, если альтернативный журнал истории не требуется.

Конкретные значения таймаутов и лимитов попыток задаются конфигурацией Router/обработчика и могут различаться для разных игр.

### 2.4. Идентификация игрока в контексте LLM

При формировании запроса (prompt) к LLM необходимо явно указывать, какой игрок совершил действие, и в каком окружении:

**До мультиплеера (одиночная игра):**

```text
Ввод пользователя: "Открыть дверь"
```

**После введения мультиплеера:**

```text
Событие: Игрок "Алекс" (роль: Медик, HP: 95) выполняет действие: "Открыть дверь".
Текущие игроки в сессии: ["Алекс", "Боб"].
Текущее положение: таверна, идёт дождь.
```

Требования к формированию контекста:

- LLM получает:
  - общую информацию о мире (`world`, видимую всем),
  - публичные данные других игроков (например, имена и роли),
  - приватные данные текущего игрока (его `players[current]`),
  - **не получает** значения из секции `secret` и приватные данные других игроков.
- Формат описания должен быть достаточно простым, чтобы его можно было стабильно воспроизводить и отлаживать (желательно иметь лог или трассировку с тем же текстом).

При большом количестве игроков допустимо вместо полного списка давать краткую сводку (например, «в комнате ещё 4 игрока, двое из них — воины»).

### 2.5. Широковещательная отправка через View Adapters

Web Gateway Service (описан в ADR-006) поддерживает отображение того, какие соединения (например, WebSocket) относятся к какой сессии. Логически это отображение вида:

```text
session_id -> набор открытых соединений
```

При завершении обработки события:

1. Router формирует `ViewCommand` (команду для представления) с:
   - обновлённым состоянием (или дельтой),
   - номером версии `state_version`,
   - номером последнего события `last_event_sequence`.
2. Router отправляет эту команду во все зарегистрированные для `session_id` адаптеры.
3. Веб-шлюз (Web Gateway) рассылает обновление по всем открытым WebSocket-подключениям сессии.
4. Клиенты получают обновление, сверяют номер версии и либо применяют его, либо игнорируют, если у них уже есть более свежая версия.

Для Telegram-адаптера широковещательная отправка означает рассылку сообщения всем участникам чата, ассоциированного с сессией (если игра поддерживает такой режим).

### 2.6. Разрешение конфликтов

**Стратегия по умолчанию: «кто успел первым» (Last-Write-Wins)**

Когда два игрока пытаются выполнить противоречивые действия (например, взять один и тот же предмет):

1. Оба события записываются в очередь и получают свои `sequence`.
2. Событие с меньшим `sequence` обрабатывается первым, предмет переносится из `world.items` в инвентарь соответствующего игрока.
3. При обработке второго события логика проверки обнаруживает, что предмета больше нет, и возвращает управляемую ошибку (например, «предмет уже взят»).
4. Router фиксирует событие как `failed` с соответствующим `error_code`, а клиент получает понятное сообщение и обновление состояния.

В будущем допускается использование более сложных стратегий (например, алгоритмы слияния изменений для сложных структур данных), но это выходит за рамки MVP и текущего ADR.

### 2.7. Обновления для клиентов и порядок применения

Чтобы клиенты могли уверенно определять актуальность полученных обновлений, каждое сообщение об изменении состояния должно включать:

- идентификатор сессии;
- `state_version`;
- `last_event_sequence`;
- дельту или полное состояние (в зависимости от выбранного протокола).

Клиентская логика:

- если полученная версия больше текущей локальной — применить обновление;
- если версия равна — либо игнорировать, либо использовать как подтверждение (в зависимости от сценария);
- если версия меньше — игнорировать обновление как устаревшее.

При восстановлении соединения клиент может запросить:

- либо полное текущее состояние сессии;
- либо все изменения, начиная с известной версии (если сервер поддерживает такой режим).

## 3. Последствия

### Положительные

- **Гибкость:** поддерживаются как почти‑реальные, так и «медленные» игры с несколькими участниками.
- **Управляемость:** состояние сессии имеет явную версию, а события — последовательность и статусы, что упрощает поддержку и разбор инцидентов.
- **Подготовка к масштабированию:** очередь событий и разделение на Router/Engine/View Adapters позволяют в будущем выделить обработку на отдельные рабочие процессы.
- **Прозрачность для клиентов:** наличие версии состояния и номера события облегчает обработку рассинхронизаций на стороне клиента.

### Отрицательные

- **Сложность реализации:** требуется поддерживать очередь событий, статусы, обработку ошибок, версионность состояния и рассылку обновлений.
- **Нагрузка на базу данных:** увеличивается количество записей (таблица `session_events`) и запросов на выборку/обновление.
- **Потенциальная задержка:** при длинной очереди игрок может видеть задержку между своим действием и «подтверждённым» результатом от сервера.

## 4. Статус реализации

- Таблица `session_events` — должна быть создана с полями, описанными выше (идентификаторы, статус, последовательность, попытки, временные метки).
- Таблица `game_sessions` — должна быть дополнена полями `state_version` и `last_event_sequence`.
- Router — должен реализовать:
  - запись событий в очередь,
  - выборку и обработку событий по одной сессии,
  - обновление состояния сессии и статуса событий атомарно,
  - отправку обновлений через View Adapters с указанием версии и номера события.
- View Adapters — должны поддерживать получение и корректную обработку обновлений с версией состояния.
- Манифесты игр — поле `config.players.max` уже существует, изменений не требуется, но реализации движка и Router должны уважать это ограничение.
