# ADR-025: JSON Schema as Single Source of Truth for Manifest Validation

- **Date**: 2026-05-03
- **Status**: Accepted
- **Авторы**: AI Agent / Project Architect
- **Компоненты**: `services/runtime-api`, `packages/contracts/manifest`, `docs/architecture/schemas`

## Контекст

В ходе разработки `services/runtime-api` произошел архитектурный дрейф: валидация игровых манифестов была реализована через масштабный императивный код на TypeScript (сотни строк `if/else` и `assert` в `manifestValidation.ts`), в то время как декларативная схема `game-manifest.schema.json` перестала обновляться и перешла в статус "legacy reference".

Такой дрейф нарушает ключевые принципы платформы Cubica:
1. **Кроссплатформенность**: Манифест должен валидироваться не только на бэкенде (Node.js), но и в мобильных клиентах (Swift/Kotlin), браузерах и инструментах аналитики. Императивный код на TS привязывает валидацию к одной экосистеме.
2. **Генерация UI**: Визуальный редактор игр (Game Editor) требует JSON Schema для автоматической генерации форм ввода. Сгенерировать формы из TS-кода невозможно.
3. **Single Source of Truth (SSOT)**: Дублирование структур данных в TS-интерфейсах, коде валидации и JSON-схеме неизбежно приводит к рассинхронизации.

## Решение

1. **JSON Schema как SSOT**: Файл `docs/architecture/schemas/game-manifest.schema.json` возвращается в статус единственного источника истины для структуры игровых манифестов.
2. **Отказ от императивной валидации структуры**: В `services/runtime-api` запрещается писать ручной код для проверки типов, наличия полей и структуры манифеста.
3. **Использование AJV**: Валидация на стороне Node.js-сервисов должна выполняться исключительно через компиляцию JSON Schema с помощью библиотеки `ajv` (или аналогичного стандартного валидатора).
4. Императивный код на стороне сервисов допускается только для сложной бизнес-логики, которую невозможно или нецелесообразно выразить в JSON Schema (например, проверка целостности графа переходов, где ID действия должно существовать в реестре).

## Альтернативы

- **Оставить TS-валидацию (Zod / ручные assert)**. *Rejected*: Это усугубляет привязку платформы к TypeScript и блокирует развитие Game Editor на базе автогенерируемых форм.
- **Двойная поддержка (TS + JSON Schema)**. *Rejected*: Нарушает принцип DRY и гарантированно приведет к рассинхронизации контрактов со временем.

## Последствия

Положительные:
- Строгое разделение платформы (Engine) и контента (Manifest).
- Готовность платформы к созданию Game Editor (интеграция с `react-jsonschema-form`).
- Значительное сокращение объема кода (удаление сотен строк ручных проверок).

Trade-offs:
- Синтаксис JSON Schema более громоздкий для описания сложных полиморфных структур (например, специфичных экшенов). Потребуется аккуратное использование `$ref` и `allOf` / `oneOf`.
- Разработчикам и агентам необходимо обновлять JSON Schema при добавлении новых механик, прежде чем писать код.
