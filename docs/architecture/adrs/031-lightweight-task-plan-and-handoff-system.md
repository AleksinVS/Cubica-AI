# ADR-031: Lightweight Task, Plan, and Handoff System

- **Дата**: 2026-05-18
- **Статус**: Accepted
- **Авторы**: AI agent
- **Компоненты**: `docs/tasks`, `NEXT_STEPS.md`, `docs/architecture/adrs`, `docs/architecture/gameplay-slices`, `.tmp`

## Контекст

Я понял решение так: текущая многоуровневая система `Milestone -> Epic -> Feature -> ExecPlan` больше не соответствует фактической работе проекта. Она помогла структурировать стартовую фазу, но сейчас создает лишнюю синхронизацию и не является реальным источником истины. Агенты часто пишут планы туда, куда считают удобным, из-за чего теряется связь между задачей, планом, результатами проверки и переданными артефактами.

Проекту нужна более короткая система, где:

- текущие приоритеты видны из одного места;
- каждая активная работа имеет один рабочий файл;
- план и отчет о выполнении живут рядом;
- временные артефакты не попадают в документацию;
- долговременные артефакты имеют владельца и ссылку из задачи;
- ADR остается только для архитектурных решений, а не для трекинга исполнения.

Термины:

- **Рабочая задача** — один проверяемый кусок работы с целью, границами, критериями приемки и журналом передачи.
- **Передача** — короткий блок в задаче, который сообщает следующему исполнителю, что сделано, что проверено, что осталось и где лежат артефакты.
- **Артефакт** — результат работы, который нужен после завершения задачи: отчет, спецификация, сгенерированный файл, снимок состояния, таблица сверки.
- **Временный артефакт** — файл для отладки или проверки, который не нужен после завершения задачи и должен жить в `.tmp/`.

## Альтернативы

- Сохранить `Milestone/Epic/Feature/ExecPlan` без изменений. Это сохраняет историю, но не решает проблему фактического неиспользования и высокой стоимости синхронизации.
- Перейти полностью во внешний трекер задач. Это может быть удобно для команды, но лишает репозиторий самодостаточности для AI-агентов и ревью через код.
- Использовать только `NEXT_STEPS.md`. Это просто, но не дает надежной передачи контекста, критериев приемки и артефактов между исполнителями.
- Использовать только ADR. Это неверно, потому что ADR фиксирует архитектурное решение, а не план исполнения и не журнал работы.

## Решение

Принять упрощенную репозиторную систему из четырех типов артефактов и зафиксировать границу ответственности между ними.

1. `NEXT_STEPS.md` становится текущей доской проекта.

   В нем остаются только ближайшие приоритеты: `Now`, `Next`, `Later`, `Blocked`, ссылки на активные рабочие задачи и короткое правило выбора следующей работы. Он не должен содержать длинные исторические отчеты, полные сценарии миграции и подробные чек-листы.

2. Рабочая задача становится главным исполняемым документом.

   Новый формат хранится как один файл в `docs/tasks/active/TSK-YYYYMMDD-short-slug.md`. Подробная форма задачи, правила передачи и архивирования принадлежат `docs/tasks/README.md`, а не ADR.

3. Долговременные артефакты получают явный индекс.

   Если задача производит отчеты, сверочные таблицы, пакет миграции или другие файлы, которые должны остаться в репозитории, они хранятся рядом с задачей или в `docs/tasks/artifacts/<TASK-ID>/` и индексируются из рабочей задачи.

   Временные скриншоты, логи, дампы и промежуточные проверки продолжают храниться только в `.tmp/` и очищаются после завершения работы.

4. Durable architecture остается в ADR и slice records.

   ADR содержит только долгосрочные архитектурные решения, ограничения, последствия и отвергнутые альтернативы. Он не содержит детальный план исполнения, список файлов к правке или статус по шагам.

   Gameplay Slice Records остаются местом для delivery-specific правил конкретных gameplay-срезов. Они не заменяют рабочую задачу и не становятся общим трекером.

## Последствия

Положительные эффекты:

- меньше уровней и меньше рассинхронизации;
- следующий агент начинает с `NEXT_STEPS.md` и одного активного task-файла;
- план, проверка и передача находятся в одном месте;
- ADR перестает использоваться как трекер исполнения;
- временные файлы не засоряют репозиторий.

Риски:

- старые `milestones`, `epics`, `features` и `content-packs` нельзя удалить сразу, потому что они содержат историю и ссылки из других документов;
- потребуется миграционная маркировка старых файлов как `legacy planning archive`;
- первое время возможен смешанный режим, пока агенты не привыкнут к новому входу.

Ограничения:

- старые документы остаются доступными только как архивная история;
- новая активная работа не должна создаваться в старых `epics/features/content-packs`, кроме явного архивного обновления;
- если задача требует архитектурного решения, сначала создается или обновляется ADR, затем рабочая задача ссылается на него.
