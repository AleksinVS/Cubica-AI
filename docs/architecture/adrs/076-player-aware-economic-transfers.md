# ADR-076: Единый контракт денежных переводов между областями состояния

- **Дата:** 2026-07-11
- **Статус:** Accepted (2026-07-11)
- **Авторы:** Codex
- **Компоненты:** Game Manifest, Runtime API, Session State, Player Projection
- **Связанные решения:** ADR-011, ADR-025, ADR-058, ADR-061, ADR-071, ADR-074, ADR-084

> [!IMPORTANT]
> Поправка ADR-084: принятые здесь экономические инварианты сохранены, но имя
> эффекта `metric.transfer`, формы `scope/path/playerRef` и его отдельный
> обработчик удалены. Действующий IR использует `core.resource.transfer` с
> заранее объявленными типизированными конечными точками; actor разрешается
> сервером, а произвольный JSON Pointer или клиентский `playerId` не
> принимаются. Разделы ниже фиксируют историческое происхождение требований.

## 1. Понимание решения

Нужно устранить расхождение между принятым дизайном пошаговых игр и ранней
реализацией `metric.transfer`, выбрав один окончательный контракт для новых
игровых потребителей. Контракт должен одинаково описывать банк, баланс
участника и общий либо командный баланс, не позволяя клиенту строить путь к
данным участника.

## 2. Контекст

ADR-058 описывает перевод `bank ↔ player` и `player ↔ player` через
`scope`, `playerId`, общий `metricId` и политику `onInsufficient`.

Фактическая JSON Schema и Runtime API реализовали перевод `bank ↔ state`
через конечные точки `{ kind: "bank" }` и
`{ kind: "state", path: "/public/..." }`, а политика называется
`insufficientFunds`.

На момент согласования ранняя форма не имела опубликованных игровых
потребителей, для которых требовалась бы обратная совместимость. Поэтому две
формы контракта не поддерживаются параллельно.
Использование произвольного динамического пути для игрока при этом остаётся
недопустимым: идентификатор участника не должен становиться непроверенным
фрагментом JSON Pointer (строкового пути в JSON-состоянии).

## 3. Решение

Ранняя форма полностью заменяется единым контрактом.

1. Имя эффекта сохраняется: `metric.transfer`.
2. Каждая сторона `from`/`to` использует поле `scope` и одну из трёх форм:

   - `{ "scope": "bank" }` — неограниченный источник или сток;
   - `{ "scope": "player", "playerId": playerRef, "metricId": string }` —
     метрика конкретного участника;
   - `{ "scope": "state", "path": "/public/..." | "/secret/..." }` —
     статически объявленный общий или командный баланс.

3. Политика недостатка средств называется `onInsufficient: "fail"`.
   Поля `kind` и `insufficientFunds` удалены из схемы и Runtime API.
4. Пример стороны участника:

   ```json
   {
     "scope": "player",
     "playerId": "{{actor}}",
     "metricId": "cash"
   }
   ```

5. `playerId` принимает только четыре формы `playerRef`, уже принятые ADR-058:
   литеральный id, `{{actor}}`, `{{activePlayer}}` или безопасную ссылку
   `fromPath` из публичного состояния.
6. Runtime разрешает такую конечную точку только в
   `state.players.<playerId>.metrics.<metricId>`, проверяет существование
   участника и числового неотрицательного баланса.
7. Форма `scope: "state"` сохраняет полезную возможность общих и командных
   балансов. Её путь проходит JSON Schema и не строится из параметров действия.
8. Все формы используют одну атомарную политику недостатка денег: действие
   отклоняется целиком. Продажа имущества, залог и банкротство остаются
   игровым процессом через явное состояние обязательства, а не скрытым
   поведением перевода.

## 4. Инварианты

1. JSON Schema остаётся единственным источником истины; TypeScript-контракты
   генерируются из неё.
2. Один эффект не может частично списать или зачислить деньги.
3. Баланс источника не становится отрицательным.
4. Клиент не передаёт произвольный путь к балансу участника.
5. Серверный контекст `actor` формируется из доверенного контекста сессии;
   параметр действия не может подменить его.
6. Старые поля `kind` и `insufficientFunds` не валидируются.
7. Статический `scope: "state"` сохраняется только для общего состояния;
   баланс участника всегда адресуется типизированной формой `scope: "player"`.

## 5. Альтернативы

### A. Обратно совместимое расширение ранней формы

Добавить `player` к `kind: bank | state` и сохранить `insufficientFunds`.
Отклонено после проверки потребителей: игровых зависимостей нет, а поддержка
двух словарей усложнит схему, Runtime API, редактор и документацию.

### B. Отдельный эффект `player.metric.transfer`

Сохранить обе модели под разными именами. Отклоняется: две операции с одинаковой
атомарной семантикой будут расходиться в проверках, журналировании и ошибках.

### C. Подстановка id участника в `scope: "state".path`

Отклоняется: превращает данные действия в часть пути записи и размывает
типизированную границу состояния участников.

## 6. Последствия

- Игра получает безопасные платежи покупателю, владельцу и банку без
  специальных веток в Runtime API.
- Схема, Runtime API и редактор получают один словарь `scope` и
  `onInsufficient`, согласованный с остальными player-aware эффектами.
- Потребители удалённых полей требуют явной миграции; временная совместимость
  старого и нового словаря не входит в контракт.
- Области `bank`, `state` и `player` подчиняются одной атомарной семантике без
  игровых исключений в Runtime API.

## 7. Запись принятия

Владелец продукта 2026-07-11 принял полную замену ранней формы без постоянного
слоя совместимости двух словарей.
