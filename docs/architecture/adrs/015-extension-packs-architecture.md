# ADR 015: Архитектура пакетов расширений (Extension Packs) и Гибридная модель Engine

Status: Accepted
Date: 2025-12-31
Author: Gemini Code Assist
Deciders: @todo
Consulted: @todo

## Контекст
Платформа Cubica должна поддерживать широкий спектр игровых механик и UI-компонентов. Базовые реализации Viewers (плееров) и Engine (движка) не могут покрыть все возможные требования конкретных игр.

Существует дилемма между гибкостью и производительностью/безопасностью:
- **User Scripts (Runtime):** Обеспечивают максимальную гибкость для контент-мейкеров (быстрые правки, логика квестов), но сложны в изоляции и медленны для тяжелых вычислений.
- **Native Modules (Build-time):** Обеспечивают производительность и доступ к системным API, но требуют пересборки и CI/CD.

Необходимо решение, сочетающее безопасность пользовательского контента и мощь системных модулей.

## Решение
Внедрить концепцию **Extension Packs (Пакетов расширений)** и утвердить **Гибридную модель** для Game Engine.


**Extension Pack** — это модуль, который содержит манифест, код и ресурсы, необходимые для добавления новой функциональности в Viewer или Engine. Каждая игра представляет собой не просто набор данных (JSON), а **самостоятельный программный продукт** (артефакт сборки), который включает в себя ядро (Base Viewer/Engine) и необходимые расширения.


### Основные принципы:
1.  **Build-time Composition:** Расширения подключаются на этапе сборки приложения (игры). Код расширений становится частью единого бандла.
2.  **Hybrid Engine Execution:**
    - **Слой Контента (Runtime / Sandbox):** `game.manifest.json` + User Scripts. Отвечают за сценарий и логику конкретной игры. Загружаются динамически. Выполняются в Sandbox (`isolated-vm`).
    - **Слой Возможностей (Build-time / Extensions):** NPM-пакеты и локальные модули. Отвечают за системные возможности (БД, физика, интеграции). Компилируются с ядром. Выполняются нативно (Node.js).
    - **Слой Ядра (Engine Core):** Базовый код движка. Управляет циклом игры, загрузкой расширений и Sandbox.
3.  **Hybrid Composition:**
    - **Engine Instance** собирается из Core + Extensions на этапе сборки (Build-time). Это создает "Специализированный Движок" (например, "RPG Engine").
    - **Game Logic** загружается в этот движок как данные (Runtime).

4.  **Shared Context (для Extensions):** Расширения выполняются в том же контексте, что и ядро.
    - **Viewer Extensions:** Имеют прямой доступ к DOM, React Context, Browser API.
    - **Engine Extensions:** Имеют прямой доступ к памяти процесса и Node.js API (в рамках контейнера).
5.  **Infrastructure as Code:** Публикация игры инициирует CI/CD пайплайн, который собирает Docker-образ (для Engine) и статику (для Viewer) и разворачивает их как выделенный сервис.
6.  **Приоритет:** Поддержка глобальных расширений (`SDK/extensions`) и локальных расширений игры (`games/<id>/extensions`).

### Типы расширений:
- **Viewer Extension:** Добавляет новые UI-компоненты, визуальные эффекты, стили или логику на клиенте.
- **Engine Extension:** Добавляет новые действия (Actions), обработчики эффектов, интеграции с внешними API. Работает на сервере или в Worker'е движка.
- **NPM Packages:** Публичные расширения распространяются как стандартные npm-пакеты (`@cubica/ext-ui-kit`, `@cubica/ext-rpg-mechanics`).
- **Local Modules:** Локальные расширения — это модули внутри репозитория игры, импортируемые напрямую.


## Техническая реализация

### Схема сборки (Build Pipeline)
1.  **Configuration:** `game.manifest.json` (или `package.json` игры) определяет список зависимостей (расширений).
2.  **Provisioning:** CI-пайплайн разворачивает окружение сборки.
3.  **Build:**
    - **Viewer:** Webpack/Vite собирает SPA, импортируя код ядра и указанных расширений. Tree-shaking удаляет неиспользуемое.
    - **Engine:** Собирается Node.js приложение (или Docker-образ), включающее ядро движка и плагины логики.
4.  **Deploy:** Артефакты деплоятся в инфраструктуру (K8s Pod, Serverless Function, Static Hosting).

### Схема работы (Runtime)

#### Viewer
- Расширение экспортирует React-компоненты или функции инициализации.
- Ядро (`Viewer Core`) при старте вызывает `Extension.install(context)`.
- Расширение может рендерить свои компоненты в слоты UI, подписываться на события стора (Zustand/Redux) и вызывать методы API напрямую.

#### Engine
- **Build-time (Extensions):**
    - Расширение экспортирует функции-обработчики (Actions) и хуки жизненного цикла.
    - Ядро (`Engine Core`) регистрирует их в едином реестре.
    - Расширение может регистрировать **нативные функции** в Sandbox (например, `inventory.addItem`), делая их доступными для скриптов.
- **Runtime (User Scripts / Content):**
    - Движок загружает скрипты из манифеста игры.
    - Скрипты выполняются в `isolated-vm`.
    - Скрипт может вызывать функции, предоставленные расширениями.

### Безопасность
- **Engine Extensions:** Доверенный код. Безопасность обеспечивается на этапе **Code Review** и **CI/CD сканирования** (SAST/DAST).
- **User Scripts:** Недоверенный код. Безопасность обеспечивается **Runtime Sandbox** (`isolated-vm`), лимитами по памяти/CPU и белым списком API.
- **Изоляция сбоев:** Падение процесса игры затрагивает только эту сессию/игру.

## Последствия

### Положительные
- **Баланс:** Высокая производительность для системных модулей и гибкость для игровой логики.
- **Безопасность:** Пользовательские скрипты не могут "уронить" сервер или украсть данные, если API расширений безопасен.
- **Гибкость:** Разработчик может собрать свой "идеальный движок" из расширений и писать под него скрипты.
- **Изоляция сбоев:** Падение процесса игры затрагивает только эту сессию/игру.

### Отрицательные
- **Сложность инфраструктуры:** Требуется оркестрация множества сервисов (по одному на версию игры).
- **Время обновления:** Обновление ядра требует пересборки и передеплоя всех игр.
- **Сложность отладки:** Ошибки могут быть как в нативном коде расширения, так и в скрипте.
