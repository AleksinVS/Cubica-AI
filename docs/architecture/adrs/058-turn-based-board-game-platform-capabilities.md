# ADR-058: Платформенные возможности пошаговых настольных игр

- **Status**: Accepted (2026-07-06)
- **Date**: 2026-07-05
- **Author**: AI Agent (по заказу владельца проекта)
- **Context**: Runtime platform capabilities / board game class support

## 1. Контекст

Cubica готовит первую полноценную настольную игру класса «экономическая гонка по циклическому полю» (рабочий пример — классическая «Монополия», см. дизайн-документ `docs/architecture/board-game-platform-design.md`). Полный анализ правил показал, что текущий детерминированный runtime не покрывает целый класс **общих** механик пошаговых настольных игр:

1. **Случайность** — в реестре эффектов (`timeline.set`, `metric.add`, `state.patch`, `object.create`, `object.state.set`, `object.attribute.patch`, `log.append`) нет ни броска кубиков, ни перемешивания колоды. Использовать невоспроизводимый `Math.random` нельзя: контуры replay/eval (ADR-038, ADR-046) требуют воспроизводимости сессии.
2. **Состояние «на игрока»** — метрики и объекты сейчас сессионные (`state.public` / `state.secret`); деньги, позиция фишки и собственность должны принадлежать конкретному игроку. Структура `state.players.<playerId>` уже описана в ADR-011, но не реализована ни в схеме манифеста, ни в runtime.
3. **Очерёдность хода** — нет понятий «порядок игроков», «активный игрок», «фаза хода». ADR-011 сознательно выбрал free-form модель, где пошаговость — частный случай, реализуемый правилами игры; сейчас платформенных примитивов для этих правил нет.
4. **Экономические операции** — нет переводов значений метрик между владельцами (игрок ↔ игрок, игрок ↔ банк).
5. **Жизненный цикл игрока и завершение игры** — нет выбывания игрока (банкротство) и декларативного условия победы.
6. **Условное ветвление эффектов** — приземление на клетку требует выбора эффектов по типу клетки; сейчас каждое действие имеет один безусловный список эффектов.

По правилу 10 из `CLAUDE.md` все перечисленные механики классифицированы как **общие** (нужны любой пошаговой настольной игре: «Монополия», гуськовые игры, карточные экономические стратегии), поэтому они должны быть реализованы как расширения схемы манифеста и платформенные возможности `runtime-api`, а не как ветки под конкретную игру.

Ориентиры лучших практик: библиотека boardgame.io (seed генератора случайных чисел хранится в состоянии игры и даёт воспроизводимость; `playOrder`/`currentPlayer`/`playOrderPos` для очерёдности; `playerView` для фильтрации чужого секретного состояния), а также уже принятые ADR-011 (структура `world/players/secret`, очередь событий) и ADR-041 (фасетная модель объектов).

## 2. Решение

Cubica добавляет **пакет платформенных возможностей пошаговых настольных игр** (board game capability pack) — ограниченный набор общих расширений схемы игрового манифеста и `runtime-api`. Все расширения проходят обычный контур ADR-056: JSON Schema — единственный источник истины, TypeScript-контракты генерируются, дрейф ловится CI.

### 2.1. Детерминированная случайность с зерном (seeded RNG)

- Зерно (seed — начальное значение генератора псевдослучайных чисел, однозначно определяющее всю последовательность) создаётся при создании сессии и хранится в `state.secret.random` вместе со счётчиком использований и идентификатором алгоритма.
- Алгоритм — детерминированный PRNG (генератор псевдослучайных чисел) `xoshiro128**` с идентификатором `"xoshiro128ss-v1"`; выборка значений костей — rejection sampling (без смещения по модулю), перетасовка — Фишер–Йетс. Идентификатор алгоритма сохраняется в состоянии, чтобы replay оставался стабильным при будущей смене алгоритма. Нормативные детали — дизайн-документ §4.1.
- Новый эффект `random.roll` (нотация костей `NdM`) записывает результаты бросков в объявленный путь публичного состояния и в журнал. Внутренние выборки для колод используют тот же генератор.
- Повтор сессии с тем же зерном и той же последовательностью действий обязан давать те же результаты (требование контуров replay/eval).

### 2.2. Колоды карт как платформенная возможность

- Классификация по правилу 10 `CLAUDE.md`: возможность **общая**. Колода со скрытым порядком нужна целому классу игр (карточные, настольные с событийными колодами), и её принципиально нельзя выразить контентом манифеста: перетасовка требует доступа к платформенному RNG, а скрытый порядок обязан жить в `state.secret`, которым управляет только runtime. При этом внутри пакета это возможность **второй очереди**: ядро (случайность, игроки, ход, экономика) от неё не зависит, и она сводится к двум узким операциям поверх RNG из §2.1 — новых концепций не вводится.
- Колода — это порядок поверх существующей коллекции объектов (ADR-041): `state.secret.decks.<deckId>.order` хранит скрытый порядок идентификаторов, `discard` — стопку сброса.
- Новые эффекты: `deck.shuffle` (перемешивание через seeded RNG), `deck.draw` (взять верхнюю карту: записать id в объявленный путь, переложить в сброс) с политикой автоперетасовки сброса при опустошении, а также `deck.extract`/`deck.return` — изъятие карты из ротации и возврат в неё.
- Удержание карты игроком («Освобождение из тюрьмы») выражается `deck.extract` плюс перевод объекта карты во владение игрока через атрибут; возврат — `deck.return`.

### 2.3. Состояние «на игрока» и адресация действующего лица

- Схема манифеста получает `state.playersTemplate` — шаблон начального состояния одного игрока (метрики, объекты, флаги). При создании сессии runtime разворачивает шаблон в `state.players.<playerId>` для каждого участника — структура совпадает с принятой в ADR-011.
- `objectModels[*].scope` расширяется значением `player` (сейчас `session`).
- Guards и effects получают адресацию владельца — тип `playerRef` с ровно четырьмя формами: литеральный id, подстановки `{{actor}}` (игрок, совершивший действие) и `{{activePlayer}}` (игрок, чей сейчас ход), и `{"fromPath": <JSON Pointer>}` для динамического владельца (например, получатель ренты). Все формы резолвит runtime, а не клиент; контекст JsonLogic дополняется материализованными ветками `actor`/`activePlayer`.
- Проекция для игрока (Presenter) отдаёт: общий мир, собственную ветку `players.<self>` целиком, чужие ветки — только объявленные публичными поля (правило `playerView` из ADR-011 §2.4, аналог `PlayerView.STRIP_SECRETS` из boardgame.io).

### 2.4. Модель хода (turn flow)

- `state.public.turn` — платформенная структура: `order` (порядок игроков), `activePlayerId`, `phase` (фаза хода, объявляемая манифестом), `turnNumber`.
- Новые эффекты: `turn.next` (передать ход следующему невыбывшему игроку по кругу), `turn.repeat` (повторный ход — например, дубль на костях), `turn.phase.set`.
- Новая guard-форма `turn`: проверка «действует активный игрок» (`activePlayerId == {{actor}}`) и текущей фазы.
- Хотсит (hotseat — режим, когда все игроки играют по очереди за одним устройством) — первый режим доставки: одна runtime-сессия, клиент действует за текущего активного игрока, аутентификация игроков не требуется. Тот же turn flow без изменений работает и в сетевом мультиплеере по ADR-011 — там `{{actor}}` начинает резолвиться из аутентифицированного `player_id` события.

### 2.5. Экономические операции

- Новый эффект `metric.transfer`: перевод значения метрики между владельцами (`bank` ↔ `player`, `player` ↔ `player`) с политикой недостатка средств `fail` (действие отклоняется целиком; сложные политики долга — предмет игровых правил, не платформы).
- Новый эффект `metric.set` со значением-выражением JsonLogic (в пределах уже введённого CI-контролируемого подмножества) — нужен, в частности, для перемещения по циклическому полю (`(позиция + бросок) % длина поля`).
- Владение объектами (собственность) — не новый примитив: это атрибут объекта (`ownerPlayerId`), изменяемый существующим `object.attribute.patch` с подстановками владельца.

### 2.6. Ограниченное условное ветвление эффектов, жизненный цикл, завершение и результат

- Эффекты получают необязательное поле `when` (JsonLogic-условие применения) и ограниченный групповой эффект `branch`: один уровень списка `[{when, effects}]`, применяется первая подошедшая ветвь. Вложенные `branch`, циклы и goto запрещены схемой — это сознательная граница, удерживающая решение в рамках ADR-024 (никакого универсального workflow-движка).
- Новый эффект `player.status.set` (`active` / `eliminated`): выбывший игрок пропускается `turn.next` и теряет доступ к действиям.
- Манифест получает декларативный блок `endConditions`: список `{when: JsonLogic, outcome}`; runtime проверяет условия после каждого применённого действия и фиксирует завершение игры платформенным событием журнала.
- Эффект `ranking.compute` рассчитывает объяснимый итог по объявленным группам участников: складывает баланс и стоимость принадлежащих им объектов из разрешённых коллекций, присваивает места внутри каждой группы и сохраняет всех победителей при равенстве. Имена групп, состав участников, формула стоимости и место записи результата принадлежат манифесту; runtime не знает типы команд и не придумывает правило разрешения ничьей.

### 2.7. Чего пакет сознательно НЕ добавляет

- **Торги между игроками (trade) и аукционы** — это ограниченные игровые механики по ADR-024: моделируются объектами-предложениями (`pendingTrade`, `auction`) и явными действиями манифеста (`propose/accept/decline`, `bid/pass`) поверх возможностей 2.3–2.5, без новых платформенных примитивов.
- **Пространственная модель поля** — клетки поля являются обычными объектами коллекции с атрибутами (`cellType`, `colorGroup`, цены); позиция фишки — метрика игрока. Отдельного «графа поля» на уровне платформы не вводится, пока не подтверждён повторный use case за пределами циклического трека.
- **Оркестрация мультиплеера** — модель уже принята в ADR-011 (очередь событий, версии состояния, broadcast); её посадка на текущий модульный монолит закреплена отдельным решением ADR-059 и реализуется как отдельный этап доставки, а не в этом пакете.
- **ИИ-оппоненты** — агент действует как игрок, выбирая ходы из того же реестра действий манифеста и проходя те же guards; прямой записи состояния у агента нет. Модель «агент — участник сессии» и платформенная проекция доступных действий закреплены отдельным решением ADR-060 поверх контрактов ADR-046/047.

### 2.8. Последовательность доставки моделей игры

Принимается последовательная стратегия для настольных игр:

1. **Хотсит** — на текущем однопользовательском runtime (одна сессия, один клиент, платформенный turn flow).
2. **Сетевой мультиплеер** — реализация ADR-011 по решениям ADR-059 (PostgreSQL-хранилище, участники с join-токенами, `session_events`/`state_version`, WebSocket-рассылка, персональные проекции) плюс привязка к порталу по ADR-033.
3. **ИИ-оппоненты** — по ADR-060: агент — участник сессии, получает персональную проекцию и платформенный список доступных действий, возвращает выбранное действие через валидируемый Agent Turn (ADR-046/047); сетевой мультиплеер для этой модели не обязателен.

Пакет возможностей 2.1–2.6 проектируется так, чтобы ни одна из трёх моделей не требовала его переделки: меняется только источник `{{actor}}` и канал доставки обновлений.

## 3. Отклонённые альтернативы

1. **Принять boardgame.io (или аналог) как игровой движок.** Отклонено: движок исполняет императивные JS-функции ходов, что противоречит декларативной модели манифестов (ADR-018, ADR-025 — JSON Schema как SSOT, анти-паттерн imperative drift) и политике расширений runtime-api (ADR-040). Используем только его проверенные архитектурные паттерны (seed в состоянии, playOrder, playerView).
2. **Реализовать механики «Монополии» ветками в `runtime-api`.** Отклонено правилом 10 `CLAUDE.md` и ADR-040: game-specific код в общем runtime запрещён.
3. **Универсальный workflow/rules-движок для ветвлений и ходов.** Отклонено ADR-024; вместо него — ограниченный `branch` без вложенности и явные действия манифеста.
4. **Невоспроизводимая случайность (`Math.random`).** Отклонено: ломает replay transcripts и evaluation fixtures (ADR-038, ADR-046) и честный аудит спорных партий.
5. **Начать с сетевого мультиплеера.** Отклонено: хотсит проверяет все игровые механики без блокирующей зависимости от реализации ADR-011, что даёт более короткий путь до играбельного среза.
6. **Хранить порядок колоды в публичном состоянии.** Отклонено: порядок колоды — секрет сессии (игрок не должен предсказывать следующую карту); публичны только уже открытые карты и размер колоды.

## 4. Последствия

### Положительные

- Платформа получает переиспользуемый пакет для целого класса игр (настольные, гуськовые, карточные экономические) без game-specific дрейфа.
- Воспроизводимая случайность сразу совместима с replay/eval контурами и будущим аудитом сессий.
- Единая модель `{{actor}}`/turn flow делает переход хотсит → сетевой мультиплеер → ИИ-оппоненты изменением канала доставки, а не переделкой игр.
- Структура состояния заранее совпадает с ADR-011, что снижает стоимость его реализации.

### Отрицательные / риски

- Реестр эффектов заметно растёт; каждая новая операция, включая колоды и итоговый рейтинг, требует схемы, обработчика, генерации контрактов и нейтральных тестов.
- Ограниченный `branch` — первый шаг в сторону выразительности, которую нужно дисциплинированно удерживать от превращения в workflow-движок (граница: без вложенности, без циклов).
- Подстановки `{{actor}}`/`{{activePlayer}}` добавляют неявный контекст исполнения эффектов; требуется строгая валидация и явные ошибки при использовании вне player-scoped контекста.
- До реализации ADR-011 персональные секреты игроков в хотсите видны всем за устройством — приемлемо для хотсита по определению, но должно быть явно задокументировано в проекции Presenter.
