## Контекст

> **Актуальность для current canonical slice:** этот ADR остаётся полезным как историческое и возможное будущее направление для text-anchor/tooling workflows, но его truth-model часть не является текущей канонической моделью `Antarctica`. Для исполнимой логики текущего slice источником истины остаётся `games/<id>/game.manifest.json` по ADR-018, а `draft/Antarctica/Game.html` используется только как текущий extraction source during migration.

Архитектура платформы Cubica (см. `PROJECT_OVERVIEW.md`, `PROJECT_ARCHITECTURE.md`, ADR-001, ADR-003, ADR-007, ADR-008, ADR-009, ADR-012) строится вокруг JSON‑манифестов игр и LLM-first подхода (LLM-first — архитектурный подход, при котором большая языковая модель рассматривается как основной игровой движок), где LLM (Large Language Model, «большая языковая модель») работает поверх структурированных данных.

Текущая версия манифеста (`docs/architecture/schemas/game-manifest.schema.json`, `manifest-structure.md`) объединяет:
- описание логики и состояния игры;
- ссылки на текстовые файлы (`assets.rules`, `assets.scenario`, `assets.methodology.*`);
- описание UI (Hybrid SDUI — гибридный подход к управлению интерфейсом, сочетающий серверно управляемый UI и локальный код) напрямую в разделе `ui`.

В процессе проектирования сценариев и игры «Antarctica» стало очевидно следующее:
- смысловое текстовое содержимое («правила», «карточки», «реплики», «подсказки», методические материалы) размазано между Markdown‑описаниями и JSON‑манифестом;
- при изменении описаний (правил/сценария/методических материалов) требуется вручную синхронизировать манифест, что ведёт к расхождению текста и структуры;
- UI‑описание внутри того же манифеста усложняет эволюцию клиентских каналов и затрудняет переиспользование общей логики и состояния игры с разными наборами UI (например, Web, Telegram, мобильный клиент).

Также появилась потребность:
- в явных **якорях** внутри текстовых файлов (Markdown/HTML), по которым манифест может ссылаться на конкретные фрагменты описаний;
- в возможности отделить **логический манифест** (структура игры, состояние, действия, ссылки на тексты) от **UI‑манифестов** (экраны, компоненты, макеты, связи с UI‑библиотекой);
- в введении сущности **макета** (макет — артефакт дизайна, состоящий из изображения и структурированного описания дизайна), которая служит мостом между работой дизайнера и реализацией UI/верстки.

Без формализации этих аспектов:
- трудно гарантировать, что манифест совпадает с актуальной версией описаний;
- сложно поддерживать несколько UI для одной игры (разные каналы, разные визуальные темы);
- сложно верифицировать и автоматизировать генерацию/обновление манифестов силами Game Editor и LLM‑помощника.

## Альтернативы

- **А1. Хранить весь текст в манифесте, а ссылки на исходные описания считать опциональными**
  - Плюсы:
    - просто для ранних прототипов;
    - не требует отдельного генератора манифеста.
  - Минусы:
    - нарушает принцип «одного источника истины», часть текстов живёт только в JSON;
    - высокие риски рассинхронизации с правилами/сценарием/методикой;
    - усложняет работу редактора и LLM‑помощника (появляются две параллельные формы текста).

- **А2. Использовать только Markdown‑ID в заголовках как якоря**
  - Плюсы:
    - хорошо читаемо для людей;
    - поддерживается экосистемой Markdown‑инструментов.
  - Минусы:
    - неудобно для мелких фрагментов (карточки, отдельные реплики, подписи);
    - ведёт к искусственным заголовкам ради якорей;
    - не позволяет «приклеивать» якоря к произвольным абзацам/элементам внутри текста.

- **А3. Не выделять отдельный UI‑манифест, а продолжать описывать UI внутри общего манифеста**
  - Плюсы:
    - одна точка правды для игры (один файл/артефакт);
    - меньше связок между документами.
  - Минусы:
    - усложнённая эволюция UI при сохранении логики игры неизменной;
    - сложно поддерживать разные UI‑пакеты для одного и того же сценария (Web/Telegram/мобильный);
    - сложнее организовать переиспользование логики между играми с разным UI.

- **А4. Ничем не формализовать макеты (изображения и дизайн‑описания)**
  - Плюсы:
    - меньше требований к документации;
    - можно продолжать обмениваться макетами в виде произвольных файлов.
  - Минусы:
    - дизайн‑артефакты плохо интегрируются в пайплайн генерации манифестов;
    - невозможно связать конкретный экран/компонент с его утверждённым макетом;
    - сложно использовать LLM для генерации UI по картинкам и описаниям.

## Решение

Мы:

1. Вводим явный механизм **текстовых якорей** в исходных описаниях (Markdown/HTML) по правилу:
   - крупные разделы помечаются заголовками с опциональными Markdown‑ID (`### Раздел {#rules-intro}`);
   - все фрагменты, от которых зависят элементы манифеста (экраны, карточки, реплики, подсказки), помечаются специальными **комментариями‑якорями**:
     - `<!-- anchor: game.screen.intro.title -->`
     - `<!-- anchor: game.card.15.body -->`
     - и т.п.;
   - комментарии‑якоря могут располагаться до или непосредственно внутри связанного блока текста.

2. Объявляем **логические манифесты** (Game Logic Manifest) и **UI‑манифесты** (UI Manifest) отдельными артефактами:
   - логический манифест описывает:
     - `meta`, `assets`, `engine`, `state`, `actions`, а также логический поток экранов/сцен (flow);
     - ссылки на текстовые фрагменты через объект `source_ref` (см. пункт 3);
     - не содержит дерева `ui` и не привязан к конкретному UI‑каналу;
   - UI‑манифест описывает:
     - структуру экранов и компонентов (Hybrid SDUI, разделы `theme`, `screens`, `uiComponent`);
     - опциональные связи с UI‑библиотекой (ключи компонентов/контроллеров);
     - раздел `layouts` (макеты).

3. Стандартизируем **ссылки из манифеста на текстовые фрагменты**:
   - вводим объект `source_ref`:
     - `file`: логическое имя файла (`rules`, `scenario`, `methodology.participants`, `methodology.facilitators` и т.п.);
     - `anchor`: строковый идентификатор якоря (`game.screen.intro.title`, `game.card.15.body` и т.д.).
   - все смысловые текстовые поля в логическом манифесте (названия экранов, описания карточек и т.п.) хранят ссылку на источник и, при необходимости, кэш‑проекцию:
     - `resolved`: текст, полученный из соответствующего якоря;
     - `format`: при необходимости, формат разметки (`"markdown"`, `"html"` и др.).
   - вводим **инвариант**: любые поля `resolved` являются **перегенерируемыми**, авторы не редактируют их вручную, а изменение текстов производится только в исходных описаниях.

4. Вводим сущность **макета** в UI‑манифесте:
   - раздел `layouts` хранит набор логических макетов:
     - `layout_id` (ключ макета);
     - `target` — к какому объекту применяется макет (`kind: "screen" | "component"`, `id` объекта);
     - `image` — путь к изображению макета (скриншот/экспорт дизайна);
     - `spec` — произвольный JSON‑описатель дизайна (сетка, зоны, нотация компонентов и т.п.);
     - опциональные `variant` и `meta` (канал, автор, дата, комментарии).
   - сами экраны и компоненты могут ссылаться на макет через `layout_id`, но рендерер не обязан использовать макеты в рантайме — это в первую очередь дизайн‑артефакт для редактора и ревью.

5. Определяем **UI‑библиотеку и игровые кастомные компоненты**:
   - UI‑манифест использует устойчивые логические ключи `type`/`widget:*`/`controller_id`, которые мапятся на конкретные классы/компоненты в UI‑библиотеке;
   - общая библиотека хранится в `SDK/shared` и, при необходимости, в отдельных пакетах;
   - игра может определять собственные виджеты с неймспейсом, например `widget:antarctica:snowstorm-meter`, реализованные в каталоге `games/<game-id>/ui/*`, а рендерер объединяет общий реестр и игровой.

6. Фиксируем **однонаправленный поток обновления**:
   - источник истины для смыслового текста — только описания (`assets.rules`, `assets.scenario`, `assets.methodology.*`);
   - генератор логического манифеста:
     - читает текстовые файлы;
     - извлекает якоря;
     - обновляет все поля `source_ref`/`resolved`/`format` и хэши исходных файлов;
   - инструменты редактирования не дают изменять текст в манифесте напрямую (или помечают такие изменения как ошибку/устаревшее состояние).

## Последствия

- **Положительные эффекты**
  - Единый источник истины для текстов: все смысловые тексты живут в Markdown/HTML‑описаниях, манифест — их структурированная проекция.
  - Облегчение работы для авторов и методистов: они оперируют привычными текстовыми файлами и якорями, не правят JSON‑манифесты руками.
  - Возможность автоматизировать генерацию/обновление манифестов и проверять их согласованность с описаниями (по хэшам и наборам якорей).
  - Разделение логического манифеста и UI‑манифестов позволяет:
    - использовать один и тот же сценарий/логику с несколькими UI‑пакетами;
    - развивать UI и клиентские каналы независимо от логики игры;
    - вводить «одежду» для данных в виде различных UI‑тем и макетов.
  - Макеты становятся полноценными артефактами в репозитории: их можно связывать с экранами/компонентами, отслеживать изменения и использовать в инструментах/LLM‑помощнике.

- **Риски и технический долг**
  - Появляется отдельный слой tooling (генераторы/валидаторы), который нужно поддерживать и развивать.
  - Требуется дисциплина в использовании якорей и конвенций нейминга (`anchor:`), иначе манифесты будут содержать «битые» ссылки.
  - Разделение логики и UI увеличивает количество артефактов (минимум два манифеста на игру) и усложняет начальное обучение команды.
  - На раннем этапе макеты могут расходиться с фактическим UI, если не внедрить проверку или автоматическое обновление.

- **Необходимые изменения в процессах**
  - Обновление сценариев и методических материалов должно сопровождаться прогоном генератора манифестов.
  - В ревью изменений сценариев нужно обращать внимание на добавление/изменение `anchor`‑комментариев.
  - Команда разработки UI должна договориться о нейминге `type`/`widget:*`/`controller_id` и вести реестр соответствий в коде.

## План внедрения

1. **Формализация в схемах и архитектурной документации**
   - Обновить `docs/architecture/schemas/manifest-structure.md`, добавив:
     - раздел про текстовые источники и якоря (Markdown‑ID и `<!-- anchor: ... -->`);
     - описание `source_ref`/`resolved`/`format` и инвариантов по тексту;
     - структуру логического манифеста и отдельного UI‑манифеста, включая раздел `layouts`.
   - Добавить/обновить JSON Schema:
     - новую схему логического манифеста (например, `manifest.v1.schema.json` или обновлённую `game-manifest.schema.json`);
     - отдельную схему UI‑манифеста (`ui-manifest.schema.json`) с секциями `screens`, `components`, `layouts`.
   - Обновить `PROJECT_ARCHITECTURE.md` (раздел про манифесты и Hybrid SDUI), отразив разделение логики и UI.

2. **Инструменты генерации и проверки**
   - Реализовать утилиту/скрипт для генерации/обновления логических манифестов из текстовых описаний:
     - парсинг Markdown/HTML, поиск `anchor`‑комментариев;
     - построение индекса `file` + `anchor` → текст/формат;
     - обновление полей `source_ref`/`resolved`/`format` и хэшей в манифесте;
     - валидация по JSON Schema.
   - Добавить проверку в пайплайн (скрипты/CI), который:
     - сравнивает хэши исходных файлов в манифесте с фактическими;
     - предупреждает или блокирует сборку при рассинхронизации или битых ссылках.

3. **Применение к реальной игре (Antarctica)**
   - Вынести существующий UI сценария «Antarctica» в отдельный UI‑манифест в `games/antarctica-nextjs-player`:
     - описать экраны и основные компоненты;
     - добавить хотя бы один макет (`layouts`) как пример.
   - Скорректировать логический манифест «Antarctica»:
     - перевести текстовые поля на `source_ref` + `resolved`;
     - связать логический манифест с UI‑манифестом по идентификаторам экранов/шаблонов.
   - Обновить Next.js‑плеер для загрузки пары «логический манифест + UI‑манифест» и передачи View‑слою уже собранного JSON‑представления.

4. **Документация и обучение команды**
   - Обновить `docs/tasks/ROADMAP.md` и создать Feature‑задачу под реализацию (F‑25‑00070 или следующий свободный ID), связав её с Epic E‑25‑0010.
   - Добавить ExecPlan (`docs/tasks/content-packs/CP_00070_manifest_text_anchors_and_ui_split.yaml`) с подробным сценарием внедрения, ориентированным на нового участника команды.
   - Описать в `docs/architecture/PROJECT_ARCHITECTURE.md` референсный пример (на базе «Antarctica») с полным путём от текстовых описаний и якорей до UI‑манифеста и работающего плеера.
