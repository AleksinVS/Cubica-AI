# ADR-032: Portal Session Launch Boundary

- **Дата**: 2026-05-18
- **Статус**: Proposed
- **Авторы**: Codex
- **Компоненты**: Portal, Runtime API, Player Web, Session, Catalog, Deployment

## Контекст

Порталу нужна общая граница запуска игр, которая не переносит в него владение
игровым состоянием или правилами конкретной игры.

Термины:

- **Портал** — веб-приложение, где консультант видит каталог игр, покупки и ссылки для запуска.
- **Игровая сессия** — запись о конкретном запуске или продолжении игры с собственным сроком действия и связью с покупкой.
- **Ссылка запуска** — URL, который ведет не прямо в игру, а в механизм выбора: создать новую игровую сессию или продолжить уже начатую.
- **Архивная игра** — покупка или сессия, срок действия которой истек либо которая завершена прохождением.

Текущая архитектура уже разделяет исполнимую игровую логику, Runtime API и
клиент игрока. Портал управляет покупками, ссылками и доступом, но не становится
игровым движком и не содержит логику конкретной игры.

## Альтернативы

- **Встроить правила запуска прямо в текущий portal draft** — быстро для демонстрации, но создаст неявную платформенную логику в UI и усложнит перенос на другие игры.
- **Запускать `player-web` напрямую по статическим ссылкам** — упрощает первый запуск, но не дает контроля сроков, количества запусков, архивации и многопользовательских сессий.
- **Сразу выделить отдельный session/router service** — соответствует
  долгосрочной декомпозиции, но повышает операционную сложность до подтверждения
  отдельной эксплуатационной границы.
- **Добавить границу запуска сессии в модуль портала** — сохраняет модульные
  границы без преждевременного выделения отдельного сервиса.

## Решение

Портал является поверхностью запуска, а не владельцем runtime-состояния.

Обязательные границы:

1. Портал владеет каталогом, покупками, ссылками запуска, генерацией игровых сессий для ссылок и административным просмотром сессий.
2. Runtime API владеет игровым состоянием, обработкой действий и журналом ходов.
3. Player Web владеет отображением игры и работает через player-facing runtime contracts.
4. Поведение конкретной игры остается в её пакете, манифестах и ограниченных
   записях игровых срезов; портал хранит только ссылку на `game_id` и общие
   настройки запуска.
5. Сущности ссылок и сессий проектируются как общая платформенная механика;
   конкретные коммерческие варианты доступа задаются данными продукта, а не
   ветвлениями в платформенном коде.

Минимальная доменная модель для launch boundary:

- `purchase` — покупка игры консультантом.
- `game_link` — купленная ссылка с `package_type`, токеном и связью с покупкой;
  значения типа и их коммерческие правила принадлежат продуктовому контракту.
- `game_launch_session` — конкретная игровая сессия для запуска, включая домен,
  токен, `purchase_id`, границы срока действия, статус и счетчики запусков.
- `session_launch_event` — аудит копирования, переходов по ссылке, продолжения, завершения и архивирования.

Правила высокого уровня:

- создание и разрешение ссылки всегда проходит через backend-границу;
- срок действия и повторное использование определяются продуктовым контрактом
  для `package_type`, а не условными ветками интерфейса;
- однопользовательская и многопользовательская сессии используют разные
  стратегии привязки, уточнённые в ADR-033;
- административный маршрут требует отдельной проверки прав: знание URL не
  является полномочием;
- завершённая сессия сохраняет доступный по политике продукта журнал, не
  раскрывая порталу правила или содержимое конкретной игры.

## Последствия

- Контроль сроков и количества запусков становится частью платформенного слоя, а не UI-условием.
- PostgreSQL остаётся источником истины; ускоряющий кэш не должен менять
  семантику доступа.
- Реализация каталога и платежей может меняться, но lifecycle сессии должен
  иметь явную сервисную границу и не превращаться в набор неявных CMS-правил.
- Точная привязка портальной сессии запуска к runtime-сессии, включая
  привязку к устройству и общий многопользовательский режим, уточнена в ADR-033.
