# ADR-032: Portal Session Launch Boundary

- **Дата**: 2026-05-18
- **Статус**: Proposed
- **Авторы**: Codex
- **Компоненты**: Portal, Runtime API, Player Web, Session, Catalog, Deployment

## Контекст

Следующий стратегический шаг Cubica — доработать портал и выполнить тестовый запуск через одну игру `Antarctica`.

Термины:

- **Портал** — веб-приложение, где консультант видит каталог игр, покупки и ссылки для запуска.
- **Игровая сессия** — запись о конкретном запуске или продолжении игры с собственным сроком действия, счетчиком ссылки и связью с покупкой.
- **Ссылка запуска** — URL, который ведет не прямо в игру, а в механизм выбора: создать новую игровую сессию или продолжить уже начатую.
- **Архивная игра** — покупка или сессия, срок действия которой истек либо которая завершена прохождением.
- **Платежная заглушка** — временный backend-путь, который создает покупку без фактического обращения к платежному сервису; нужен для test VPS до включения реальной оплаты.

Текущая архитектура уже разделяет исполнимую игровую логику (`games/antarctica`, `services/runtime-api`, `apps/player-web`) и draft-артефакты портала. Для тестового запуска нужна ясная граница: портал управляет покупками, ссылками и доступом, но не становится игровым движком и не содержит game-specific logic.

## Альтернативы

- **Встроить правила запуска прямо в текущий portal draft** — быстро для демонстрации, но создаст неявную платформенную логику в UI и усложнит перенос на другие игры.
- **Запускать `player-web` напрямую по статическим ссылкам** — упрощает первый запуск, но не дает контроля сроков, количества запусков, архивации и многопользовательских сессий.
- **Сделать отдельный session/router service уже сейчас** — соответствует долгосрочной декомпозиции, но повышает операционную сложность до стабилизации первого test VPS.
- **Добавить session-launch boundary в ближайший модульный backend/portal integration path** — сохраняет модульные границы и достаточно прост для тестового VPS.

## Решение

Для ближайшего test VPS портал становится launch surface, а не runtime owner.

Обязательные границы:

1. Портал владеет каталогом, покупками, ссылками запуска, генерацией игровых сессий для ссылок и административным просмотром сессий.
2. Runtime API владеет игровым состоянием, обработкой действий и журналом ходов.
3. Player Web владеет отображением игры и работает через player-facing runtime contracts.
4. Game-specific behavior `Antarctica` остается в `games/antarctica`, manifest/contracts и bounded gameplay records; портал хранит только ссылку на `game_id` и настройки запуска.
5. Сущности ссылок и сессий проектируются как общая платформенная механика, применимая к разовым, дневным и месячным ссылкам для разных игр.

Минимальная доменная модель для launch boundary:

- `purchase` — покупка игры консультантом.
- `game_link` — купленная ссылка с типом (`single_use`, `day`, `month`), токеном и связью с покупкой.
- `game_launch_session` — конкретная игровая сессия для запуска, включая домен, токен, `purchase_id`, `valid_from`, `valid_to`, счетчик ссылки, статус и счетчики запусков.
- `session_launch_event` — аудит копирования, переходов по ссылке, продолжения, завершения и архивирования.

Уточненные параметры test VPS:

- Ссылка запуска имеет форму path внутри портала, даже если физически портал и player/runtime будут размещены на разных серверах.
- Для первого VPS источник покупок — платежная заглушка; существующая Robokassa-интеграция в `services/portal-backend` остается кандидатом для восстановления реальной оплаты.
- Основная timezone дневной ссылки — Москва.
- `launch_count` считается по созданным runtime sessions, а не по кликам или уникальным устройствам.
- Минимальная авторизация консультанта и admin window — логин/пароль.
- Для завершенной разовой `Antarctica` показывается журнал ходов самой игры; runtime/player boundary должен сохранять этот журнал после закрытия игровой сессии.

Правила высокого уровня:

- Копирование ссылки в портале должно проходить через backend boundary: для дневной и месячной ссылки оно создает новую `game_launch_session`, а для разовой ссылки переиспользует единственную сессию.
- После копирования портал показывает сообщение: `Ссылка скопирована в буфер обмена`.
- Разовая ссылка после завершения игры переводит покупку в архив; повторный переход открывает журнал ходов или другой доступный лог игры.
- Дневная ссылка действует в дату покупки с `00:00:00` до `23:59:59` по московскому времени.
- Месячная ссылка может генерировать сессии в пределах купленного периода; каждая такая сессия действует 48 часов с момента генерации.
- Однопользовательская игра продолжает состояние по device token в cookie; многопользовательская игра продолжает состояние по номеру игровой сессии.
- Для многопользовательских игр портал должен показывать список активных сессий и давать доступ к админскому окну сессии.
- Добавление `admin` в конец ссылки является административным route variant и должно требовать отдельной проверки прав, а не только знания URL.

## Последствия

- Тестовый запуск может быть построен как один понятный vertical slice: portal -> launch boundary -> runtime-api -> player-web -> `Antarctica`.
- Контроль сроков и количества запусков становится частью платформенного слоя, а не UI-условием.
- Для MVP достаточно PostgreSQL как источника истины; Redis остается будущим ускорителем и не требуется для первого VPS.
- Потребуются новые API contracts, миграции, seed-данные для `Antarctica`, тесты времени действия ссылок и e2e-проверки переходов.
- Текущий `draft/cubica-portal-nextjs` должен рассматриваться как visual/product draft и starting point для тестового портала, но не как готовый архитектурный источник истины.
- Strapi может быть использован как краткосрочный backend для каталога, пользователей, платежной заглушки и Robokassa-интеграции, но session lifecycle должен иметь явный service boundary и не должен превращаться в набор неявных CMS-правил.
