## 1. Понимание решения

Решение понято так: Cubica использует Phaser как один из каналов интерактивного
пошагового поля, но Phaser остаётся View — слоем отображения и ввода. Он
показывает пространственную проекцию игры, а все авторитетные правила и состояние
остаются в `runtime-api` и манифестах.

Дополнение, подтверждённое PM 2026-07-13, понято так: обычные браузерные
кнопки действий поля не являются частью Phaser-сцены и должны работать, даже
если движок не загрузился или сцена не создалась. Игровой модуль предоставляет
их отдельной чистой функцией (функцией без побочных эффектов), которая читает
только доступный игроку снимок состояния. Сервер по-прежнему единолично решает,
какие действия законны.

## 2. Контекст

Принятый ADR-062 вводит Phaser как платформенную зависимость для клиентских
симуляций реального времени через компонент `simulationSurface`. Интерактивные
пошаговые поля имеют другой жизненный цикл и могут требовать:

- масштабирования и перемещения камеры по карте;
- выбора пространственных объектов и связей;
- перетаскивания игровых объектов;
- визуального предварительного просмотра пространственных действий;
- анимации подтвержденных перемещений;
- полноэкранного показа на общем или индивидуальном экране.

Использовать название `simulationSurface` для пошагового поля неверно: это смешает разные жизненные циклы и границы честности. При этом повторно загружать второй экземпляр Phaser или давать плагину собственную версию библиотеки также нельзя.

## 3. Классификация

Возможность является общей для класса интерактивных 2D-полей: транспортных, тактических, маршрутных и экономических настольных игр. Конкретная сцена, карта и визуальные объекты принадлежат игре.

## 4. Принятое решение

### 4.1. Один платформенный Phaser-host

`apps/player-web` лениво загружает одну зафиксированную версию Phaser. Игровой плагин получает объект Phaser через платформенный контекст и не импортирует библиотеку напрямую. Это сохраняет политику project-local plugins и позволяет обновлять движок централизованно.

Точка вклада `phaserSceneFactory` переиспользуется как общая фабрика сцен. UI-манифест получает отдельный семантический компонент `interactiveBoardSurface`, не равный `simulationSurface`.

### 4.2. Phaser отображает проекцию runtime

Сцена получает:

- разрешенные игровые ассеты;
- player-facing snapshot с разрешёнными объектами и доступными действиями;
- UI-геометрию карты;
- функцию отправки обычного действия runtime;
- уведомления о новом подтвержденном snapshot.

Сцена не получает прямого доступа к session store, файлам игры или секретному состоянию.

### 4.3. Ввод является намерением, а не изменением состояния

Перетаскивание объекта или предварительный просмотр пространственного действия
сначала показывает неподтверждённый вариант. После завершения ввода сцена
отправляет действие с проверяемыми параметрами. Только успешный ответ runtime
меняет объект в авторитетном snapshot.

При отклонении сцена возвращается к подтверждённой проекции, а обычный
web-интерфейс показывает причину. Оптимистичное изменение авторитетного игрового
состояния запрещено.

### 4.4. Камера и масштаб

Сцена использует камеру с ограниченными границами мира, перемещением и масштабированием. Canvas сохраняет пропорции исходной карты через режим масштабирования `FIT` или `EXPAND`; для общего экрана доступен полноэкранный режим.

Игровые координаты хранятся в нормализованной UI-геометрии и не зависят от фактического разрешения телевизора или проектора.

### 4.5. Жизненный цикл

Phaser.Game и сцена создаются только при монтировании поверхности. При смене игры, закрытии preview или размонтировании компонента сцена снимает обработчики, освобождает игровые объекты и уничтожает принадлежащий поверхности экземпляр/контекст. Обработчики `shutdown` и `destroy` обязательны, чтобы повторное открытие игры не накапливало ввод, текстуры и анимации.

Физический движок Phaser не включается платформенным хостом автоматически. Его
использование требует отдельной объявленной потребности поверхности.

### 4.6. Независимые доступные действия

Публичный интерфейс игрового модуля содержит необязательную точку вклада
`registerAccessibleBoardActionsProvider(gameId, provider)`. Поставщик имеет тип
`AccessibleBoardActionsProvider`: он синхронно принимает только
`InteractiveBoardSessionSnapshot` — доступный игроку снимок сессии — и
возвращает полный список `AccessibleBoardAction` для обычных DOM-кнопок.

Нормативная форма публичного контракта:

```ts
export type AccessibleBoardActionsProvider = (
  session: InteractiveBoardSessionSnapshot
) => readonly AccessibleBoardAction[];

export interface PlayerPluginApi {
  registerAccessibleBoardActionsProvider?(
    gameId: string,
    provider: AccessibleBoardActionsProvider
  ): () => void;
}
```

Возвращаемая функция снимает только соответствующую регистрацию, как и другие
точки вклада игрового модуля.
Модуль, который вызывает этот метод, декларативно объявляет
`targets.player-web.contributes.accessibleBoardActionsProvider: true` в
`plugin.json`; общая JSON Schema проверяет эту точку вклада до
сборки и публикации.

Поставщик обязан быть чистой детерминированной проекцией: он не импортирует
Phaser, не обращается к DOM, внутреннему хранилищу сессий или сети и не изменяет
полученный снимок. Он может переводить игровые данные в понятные подписи,
описания, идентификаторы действий и проверяемые параметры, но не повторяет
серверные правила законности. Доступность команды берётся из подтверждённой
серверной проекции действий по ADR-079.

`player-web` получает поставщик по `gameId` до динамической загрузки Phaser и
повторно строит DOM-список после каждого подтверждённого снимка. Для нового или
мигрированного игрового модуля список не зависит от
`InteractiveBoardSceneHandle`; поэтому ошибка импорта движка, создания сцены или
canvas отключает только пространственное представление, а не клавиатурный путь.
Отправка кнопки проходит через тот же Presenter и Runtime API, что и ввод сцены.

Точка вклада необязательна для игр без интерактивного поля. Если игра объявляет
важные пространственные действия через `interactiveBoardSurface`, контракт
поверхности требует доступную DOM-альтернативу.

Получение доступных действий из handle уже созданной Phaser-сцены не
удовлетворяет этому контракту: оно имеет ту же точку отказа, что и canvas, и не
доказывает доступность независимой альтернативы.

## 5. Граница Phaser и обычного web-интерфейса

### Phaser

- пространственное поле и его фон;
- игровые объекты, связи и слои положения;
- прямой выбор, drag/drop и подсветка допустимых целей;
- панорамирование, zoom и анимация подтверждённого результата.

### DOM/React player-web

- этапы, команды и подтверждения действий;
- текст, формы, таблицы и карточки;
- диалоги параметров и расчётов;
- журнал, правила, методические подсказки и рефлексия;
- сообщения об ошибках и доступная клавиатурная альтернатива действиям поля.

Это разделение сохраняет читаемость текста, доступность и тестируемость бизнес-операций, одновременно используя сильные стороны Phaser для пространственного взаимодействия.

## 6. Инварианты

1. Phaser не рассчитывает правила, экономику, допустимость действий или окончание партии.
2. Случайность создается runtime через воспроизводимый генератор; визуальные частицы и анимации не влияют на результат.
3. Любое взаимодействие сцены превращается в действие Cubica и проходит серверную валидацию.
4. UI-геометрия не является источником игровых правил или авторитетной топологии.
5. Плагин игры не имеет собственной зависимости Phaser.
6. Игры без Phaser не загружают его основной bundle.
7. Все важные действия поля доступны без точного drag/drop — через выбор
   объекта и цели или доступную DOM-альтернативу.
8. Generic host не содержит названий, ассетов или правил конкретной игры.
9. DOM-альтернатива `interactiveBoardSurface` не
   зависит от импорта Phaser, создания сцены или `InteractiveBoardSceneHandle`
   и использует только доступный игроку снимок.
10. Поставщик действий не вычисляет законность команды: серверная проекция и
    повторная проверка Runtime API остаются источником истины.

## 7. Отклоненные альтернативы

- **Полностью реализовать игру внутри Phaser.** Отклонено: текстовые панели, формы, методика и доступность хуже, а canvas рискует стать вторым владельцем состояния.
- **Ограничить платформу только SVG/DOM-полями.** Отклонено: сложные поля с
  камерой, прямым манипулированием и анимацией требуют отдельного
  производительного канала. Конкретная игра по-прежнему может выбрать SVG/DOM.
- **Использовать `simulationSurface`.** Отклонено: у пошагового поля нет клиентского симуляционного результата и границы честности ADR-062.
- **Добавить Phaser в зависимости плагина игры.** Отклонено: дублирование версии и нарушение платформенной политики плагинов.
- **Применять положение объекта сразу после drag.** Отклонено: клиент не является источником истины и может нарушить правила сети.
- **Хранить путь к списку действий в UI-манифесте.** Отклонено: такой путь
  связывает общий рендерер с внутренней формой состояния каждой игры и не
  выражает без игровой логики подписи и параметры составных пространственных
  действий. Поставщик оставляет игровую проекцию внутри игрового модуля, не
  превращая UI-манифест во второй источник правил.
- **Получать DOM-действия из handle Phaser-сцены как основной контракт.**
  Отклонено: handle появляется только после успешной загрузки движка, поэтому
  canvas и клавиатурный резерв имеют общую точку отказа. Старый обработчик
  сохранён только как ограниченный резерв API 2.0 до следующей несовместимой
  версии.

## 8. Последствия

Положительные:

- поле удобно для большого общего экрана;
- появляется повторно используемый канал интерактивных пошаговых 2D-игр;
- camera/zoom, drag/drop и анимации не требуют самописного canvas-движка;
- серверная модель и методические интерфейсы остаются независимыми от Phaser.
- клавиатурные действия остаются доступны при сбое загрузки Phaser или сцены;
- сцена и DOM-кнопки используют одну игровую проекцию поверх подтверждённого
  снимка, но имеют независимые жизненные циклы.

Риски и стоимость:

- платформенный Phaser-host поддерживает две семантически разные поверхности,
  что увеличивает стоимость сопровождения его жизненного цикла;
- нужны проверки жизненного цикла, бюджета клиентского кода и визуального
  поведения;
- каждый игровой модуль с важными пространственными действиями должен
  поддерживать чистый поставщик доступных действий;
- пространственный ввод всё равно требует серверной модели правил или
  авторского каталога допустимых результатов — Phaser сам эту архитектурную
  проблему не решает.

## 9. Принятое подтверждение

PM подтвердил гибридную границу 2026-07-11: Phaser отвечает только за
пространственное поле и прямое взаимодействие, а текстовые панели, формы,
расчёты, журнал и методические материалы остаются обычным web-интерфейсом. Это
даёт визуальную выразительность без переноса бизнес-логики и методики в canvas.

PM подтвердил 2026-07-13 независимую доступную альтернативу: DOM-кнопки
получают действия через необязательный чистый поставщик публичного интерфейса
игровых модулей, а не из созданного Phaser-handle. Возможность остаётся
необязательной для модулей без интерактивного поля, но является инвариантом
доступности для важных действий такого поля.
