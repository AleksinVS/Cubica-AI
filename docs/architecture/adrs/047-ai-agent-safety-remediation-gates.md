## 1. Понимание решения

Решение понято так: миграция Cubica Surface и AI-driven runtime завершила MVP-контур, но review выявил safety gaps, которые нельзя оставлять перед подключением real provider backend или расширением gameplay catalog.

Нужно зафиксировать архитектурные ворота исправления: подтверждение человека не может быть обычным аргументом агента, failed Agent Turn не может применять effects, manifest `allowedCapabilities` должен стать runtime allowlist, а channel renderer не должен получать действия, которые он не умеет безопасно исполнять.

Этот ADR не отменяет ADR-043..ADR-046. Он уточняет обязательные safety gates поверх них.

## 2. Контекст

Текущий MVP имеет:

- CopilotKit shell and local AG-UI backend в `apps/editor-web`;
- `CubicaSurface` and Agent Turn contracts в `packages/contracts/ai`;
- `POST /agent-turns` and mock Agent Runtime в `services/runtime-api`;
- Web Surface renderers в `apps/player-web` and `apps/editor-web`;
- framework-neutral Telegram/Phaser projection helpers;
- replay/eval/audit contract shapes.

Review нашёл четыре класса риска:

- mutating editor tools доверяют `approved=true` из tool arguments;
- `CubicaAgentTurnResult.ok=false` не запрещает persistence of returned effects;
- `allowedCapabilities` передаётся агенту, но не ограничивает accepted effects/actions;
- default Surface catalog может признать валидными actions, которые конкретный channel renderer не поддерживает.

## 3. Термины

- **Approval envelope** - собственная запись Cubica о подтверждении действия человеком: кто подтвердил, что именно подтвердил, на какой run/tool/action это распространяется и когда истекает.
- **Capability gate** - runtime-проверка, которая сопоставляет manifest-declared capability с разрешёнными effect kinds, target paths, tool names and surface action kinds.
- **Accepted Agent Turn** - Agent Turn, у которого `ok: true`, schema validation passed, semantic validation passed, capability gate passed and effects persisted.
- **Rejected Agent Turn** - Agent Turn, который не прошёл один из gates; он может быть залогирован, но не может менять session state.
- **Channel action policy** - allowlist action kinds and targets для конкретного renderer/channel, например Web player, editor sidebar, Telegram or Phaser.

## 4. Решение

Cubica вводит mandatory remediation gates before production Agent Runtime or custom provider traffic.

1. **Human approval is a Cubica-owned envelope, not agent input.**
   - Agent arguments may request approval but cannot prove approval.
   - Mutating tools accept only approval generated by a Cubica UI approval flow.
   - CopilotKit `useHumanInTheLoop` may be used as the MVP adapter UI, but the durable contract is Cubica approval envelope.
   - The future custom Cubica Agent UI must produce the same approval envelope.

2. **Failed Agent Turn cannot persist effects.**
   - `ok: false` is a rejected turn.
   - Rejected turns may return narration, diagnostics and audit metadata.
   - Runtime must not apply `effects`, `availableActions` or primary gameplay `surface` from rejected turns unless a later ADR defines a read-only diagnostic surface policy.
   - Rejected event log entries must have `effectCount: 0`.

3. **Manifest capabilities become runtime enforcement.**
   - `agentRuntime.allowedCapabilities` must map to concrete allowed effect kinds, target prefixes, tool names and action kinds.
   - Runtime rejects any Agent Turn result outside the declared capability set.
   - Provider adapters are not accepted until capability gate tests cover accepted and rejected effects.

4. **Surface actions are validated against channel policy.**
   - A Surface component being in catalog is not enough.
   - Runtime or projection layer must also validate that each action is supported by the target channel and context.
   - Web player primary gameplay must fail closed for `editorTool`, `portalCommand` and unsupported `openUrl` actions.
   - Editor helper Surface must accept only editor catalog tools.

5. **Non-Web projections fail closed.**
   - Telegram/Phaser projections must expose `ok` or an equivalent validation status.
   - If validation has an error, projections must suppress interactive actions and emit safe diagnostics.
   - Provider-specific payloads remain forbidden in projections.

6. **External production backend requires auth gate.**
   - `CUBICA_EDITOR_AGENT_AG_UI_URL` or a future Cubica protocol backend requires server-side auth policy before production enablement.
   - Local fallback must be explicitly disabled in production environments where fallback is not allowed.
   - Provider SDK imports remain prohibited outside approved adapter boundaries.

## 5. Архитектурные инварианты

1. Agent output is untrusted until schema, semantic, capability and channel policy gates pass.
2. Approval cannot be inferred from prompt text, tool arguments or model output.
3. Only Cubica UI or role-authorized server flow can create approval envelopes.
4. Rejected Agent Turn results never mutate session state.
5. Manifest `allowedCapabilities` is not documentation after remediation; it is an executable allowlist.
6. Surface catalog validation and channel action validation are separate gates.
7. Player clients never execute editor, portal or provider actions from gameplay Surface.
8. Telegram and Phaser projections never expose interactive controls from invalid Surface payloads.
9. Production Agent Runtime cannot be enabled without auth, audit, replay/eval and operation policy.
10. Mock Agent Runtime remains test/dev only and cannot satisfy production provider readiness.

## 6. Альтернативы

### A. Keep `approved=true` As A Tool Argument

Rejected. It lets the agent approve its own mutating tool call and breaks human-in-the-loop.

### B. Treat `ok=false` As Advisory Only

Rejected. A failed turn with persisted effects would make audit and replay misleading.

### C. Leave `allowedCapabilities` Descriptive Until Real Provider Work

Rejected for production. It is acceptable only as documented MVP debt while the adapter is mock/test-only.

### D. Use One Global Surface Catalog For Every Channel

Rejected. Component availability and action safety depend on the renderer context.

## 7. Последствия

Positive:

- Real provider integration gets a clear safety checklist before traffic is enabled.
- Editor mutating tools cannot be called by an agent without a human-controlled approval path.
- Agent Turn replay and audit become semantically trustworthy.
- Non-Web channels can consume Surface projections without inheriting Web-only behavior.

Costs and risks:

- Contracts need additional fields or helper types for approval envelope, capability policy and projection status.
- Runtime validation becomes stricter and may reject early provider experiments until they declare capabilities correctly.
- Editor MVP must integrate CopilotKit human-in-the-loop adapter or an equivalent Cubica approval surface.
- Existing tests must be expanded beyond happy-path mock Agent Runtime.
