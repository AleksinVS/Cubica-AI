# ADR-075: Долговечная история авторских версий

## 1. Понимание решения

Нужно определить, что именно означает пользовательская команда «Сохранить» и
откуда редактор читает историю после закрытия или очистки рабочей сессии.
Текущая реализация создаёт коммит только во временной ветке сессии. Такая ветка
годится для изоляции черновика, но сама по себе не гарантирует долговечную
историю проекта: после удаления рабочей копии новая сессия начинается от
основной линии проекта и может не видеть прежние сохранения.

## 2. Контекст

ADR-042 разрешает быстро очищать чистые сессии, а ADR-057 обещает автору
историю точек «Сохранить», доступную после повторного открытия. ADR-065 задаёт
целевую модель: отдельный Git-репозиторий проекта автора, асинхронный review и
неизменяемые опубликованные версии.

Без дополнительного правила эти решения оставляют разрыв: сохранение успешно,
но пользовательская версия остаётся достижима только из временной ветки
конкретной сессии. Реализация истории поверх `git log` этой ветки закрепила бы
неверный продуктовый контракт.

## 3. Термины

- **Рабочая сессия** — изолированная рабочая копия и временная ветка, где живут
  несохранённые изменения автора.
- **Линия авторских версий** — долговечная последовательность подтверждённых
  точек «Сохранить» одного проекта, доступная новым сессиям.
- **Публикация** — создание неизменяемого проверенного снимка для каталога и
  игроков. Сохранение не является публикацией.
- **Продвижение версии** — перенос подтверждённого состояния из ветки сессии в
  долговечную линию проекта без переписывания уже существующей истории.

## 4. Решение

Принять долговечную линию авторских версий проекта.

1. Рабочая сессия остаётся источником несохранённых изменений и может быть
   удалена по правилам ADR-042.
2. «Сохранить» создаёт неизменяемую авторскую версию и продвигает её в
   долговечную линию проекта. Новая сессия создаётся от последней доступной
   авторской версии, а не от произвольной временной ветки.
3. Продвижение выполняется атомарно с оптимистической проверкой ожидаемой
   текущей версии. Конкурентное изменение возвращает конфликт и не переписывает
   историю.
4. Возврат прежней версии создаёт новую авторскую версию с содержимым выбранной
   точки. Разрушительные `reset` и удаление более новых версий запрещены.
5. История содержит только разрешённые authoring-файлы проекта и производные
   безопасные метаданные: автора, время, комментарий и нормализованное резюме.
6. Публикация остаётся отдельной операцией `Validate/Compile → Publish` по
   ADR-065. Авторская версия не становится доступной публичному runtime.
7. В локальном монорепозитории первая реализация может использовать выделенную
   проектную ссылку Git. Хостируемая студия хранит ту же логическую линию в
   репозитории конкретного проекта, которым владеет Game Repository.
8. Локальная новая сессия берёт текущую версию платформы и восстанавливает из
   долговечной линии только разрешённые пути проекта. История автора не должна
   откатывать `apps/`, `services/`, `packages/`, схемы или другой платформенный
   код к состоянию старой сессии.

Точное имя внутренней ссылки и механизм атомарного продвижения являются
реализационными деталями. Клиент получает только непрозрачный идентификатор
версии и не видит Git-термины.

## 5. Инварианты

1. Удаление или сборка мусора рабочей сессии не удаляет доступные авторские
   версии.
2. Новая совместимая сессия видит ту же историю проекта.
3. Несохранённые изменения никогда не продвигаются автоматически.
4. Сохранение, возврат и публикация — разные пользовательские операции.
5. История одного проекта не раскрывает файлы, версии или авторов другого
   проекта либо арендатора.
6. Git остаётся источником содержимого и порядка версий; отдельная база может
   хранить индекс и подсказки, но не становится вторым источником истины.
7. Открытие сохранённого проекта на обновлённой платформе сохраняет текущий
   platform checkout; совместимость проекта проверяется отдельно по ADR-042.

## 6. Альтернативы

### A. История только текущей сессии

Ограниченная альтернатива: честно назвать функцию «точками сохранения текущей
сессии» и предупредить, что после закрытия или очистки они могут быть
недоступны. Это дешевле, но не закрывает выбранный продуктовый сценарий и
противоречит ожиданию истории проекта автора.

### B. Сохранять историю только в базе данных

Отклоняется: создаёт второй источник истины для содержимого рядом с Git и
противоречит ADR-036/065. База допустима только как индекс долговечных Git-версий.

### C. Никогда не очищать ветки сессий

Отклоняется: маскирует отсутствие проектной линии неограниченным хранением
временных ресурсов и противоречит обязательному жизненному циклу ADR-042.

## 7. Последствия

- История становится настоящей функцией проекта, а не интерфейсом временной
  ветки.
- Сохранение требует атомарного продвижения и обработки конфликтов между
  сессиями.
- Жизненный цикл сессии можно сохранять коротким без потери сохранённых версий.
- Будущий review использует авторские версии как вход, а публикация создаёт из
  них отдельные неизменяемые delivery-артефакты.

## 8. Подтверждение

Владелец продукта 2026-07-11 подтвердил долговечную линию авторских версий из
раздела 4. Ограниченная session-only альтернатива отклонена как не закрывающая
пользовательский сценарий истории проекта.
