## 1. Понимание решения

Решение понято так: серверная игровая логика должна развиваться сначала через
манифест и общие возможности платформы, а не через код конкретной игры внутри
`runtime-api`. Если механику можно описать данными, типизированным планом,
правилом, проверкой или ограниченным изменением состояния, она должна быть
описана в манифесте и JSON Schema. Исполняемый код конкретной игры разрешён,
если декларативный язык и общие модули недостаточны, но он обязан остаться в
изолированной границе игры и быть безопасным для платформы.

Доверенные проектные серверные плагины, если они понадобятся, должны запускаться
отдельным процессом с JSON-протоколом и проходить общий publish/security gate.
Сам выбор уже предусмотренного изолированного расширения не требует нового
архитектурного согласования для каждой игры. Для marketplace и иных
недоверенных случаев целевой путь — изолированный исполнитель плагина:
контейнер, микро-виртуальная машина или WebAssembly/WASI для чистых вычислений.

## 2. Контекст

`runtime-api` владеет загрузкой игрового контента, созданием сессий, обработкой действий и авторитетным состоянием игры. Ошибка в серверной логике может:

- испортить состояние сессии;
- открыть игроку лишние данные;
- применить неправильный переход;
- зависнуть обработчиком действия;
- выполнить лишний внешний вызов;
- нарушить повторяемость тестов.

Поэтому runtime-плагин опаснее `player-web` плагина. Ошибка в `player-web` чаще ломает отображение. Ошибка в `runtime-api` меняет саму игру.

ADR-037 оставляет первую реализацию плагинов на стороне `player-web`. Этот ADR фиксирует серверную политику расширения: как решать, что должно попасть в манифест, что может быть доверенным проектным runtime-плагином в отдельном процессе, а что обязано идти через контейнерную или WebAssembly/WASI изоляцию.

## 3. Термины

- **Runtime-api plugin** - подключаемый серверный модуль игры, который добавляет обработчики действий, проверки, вычисления состояния или интеграции внутри границы `runtime-api`.
- **Декларативная механика** - игровое правило, описанное как данные в манифесте, а не как произвольный код.
- **Псевдоязык механик** - небольшой JSON-формат для описания игровых правил: когда действие разрешено, какие данные читать, какие изменения состояния вернуть и что записать в журнал.
- **Возможность платформы** - переиспользуемая функция платформы, например изменение метрики, переход по timeline, установка флага, проверка условия или запись события.
- **Доверенный проектный runtime-плагин** - серверный плагин, принадлежащий команде платформы или проекта. Он не является marketplace-кодом, запускается отдельным процессом и проходит отдельное ревью.
- **Песочница** - изолированная среда выполнения кода с ограничениями по файлам, сети, переменным окружения, процессам, памяти и времени.
- **Runner** - исполнитель плагина, то есть отдельный процесс или изолированная среда, которая получает JSON-вход и возвращает JSON-ответ.
- **Протокол вызова** - стабильный schema-first JSON-формат входа и выхода
  плагина. Плагин получает только разрешённую проекцию данных и возвращает
  типизированное вычисленное значение либо предложенный change set в заранее
  объявленной области записи; `runtime-api` повторно проверяет и атомарно
  применяет результат.

## 4. Практики и ограничения

1. JSON Schema остается единственным источником истины для структуры манифеста и plugin/runtime контрактов. Нельзя заменять схему ручными TypeScript guard checks.
2. `node:vm` нельзя считать защитной границей. Официальная документация Node.js прямо говорит, что этот модуль не является механизмом безопасности для недоверенного кода.
3. `worker_threads` полезен для параллельных вычислений, но остается внутри процесса Node.js. Он не подходит как защита для сторонних runtime-плагинов и не заменяет отдельный процесс с JSON-протоколом для доверенных проектных серверных плагинов.
4. Проверочные команды и вспомогательные процессы должны запускаться через `execFile`, `spawn` или `fork` с явными аргументами, таймаутом и `AbortSignal`. Shell-строки нельзя строить из пользовательского ввода.
5. Анализ и верификация плагина снижают риск, но не заменяют песочницу. Даже проверенный сторонний код не должен исполняться внутри процесса `runtime-api`.
6. WebAssembly/WASI подходит для чистых вычислительных обработчиков, если используется среда с реальной моделью прав. Node.js WASI не должен приниматься как единственная защита для недоверенного кода.
7. Внешние интеграции не должны незаметно попадать в критический путь действия игрока. Если внешний вызов нужен, он должен иметь таймаут, идемпотентность и явные права.

## 5. Решение

### 5.1. Манифест сначала

Для любой новой серверной игровой механики действует следующий порядок выбора:

1. Сначала проверить, выражается ли механика существующим Game Intent и
   типизированным Mechanics IR.
2. Если не выражается, проверить, можно ли добавить нейтральную общую
   возможность платформы и JSON Schema. Одного реального игрового среза
   достаточно, если контракт нейтрален и доказан нейтральной фикстурой.
3. Если общая возможность внесла бы в платформу предметный смысл одной игры,
   рассматривать изолированное игровое runtime-расширение.

Этот порядок удерживает предметную логику вне общего ядра, но не делает
изолированное расширение временной уступкой. Жёсткая граница после выбора
расширения — изоляция внутри игры и безопасность платформы.

### 5.2. Запрет на функционал под конкретную игру в runtime-api

В `runtime-api`, `packages/contracts` и других общих runtime-слоях запрещены:

- проверки вида `gameId === "specific-game"`;
- импорты кода из `games/<gameId>/plugins/**`;
- отдельные ветки обработчиков под конкретную игру;
- поля схемы, названные по конкретной игре или ее персонажам;
- скрытые правила, которые нельзя понять из манифеста.

Данные, привязанные к конкретной игре, могут жить в игровом манифесте, игровом
пакете или проектном плагине. Общий runtime должен видеть только типизированные
Game Intents, общие IR-контракты и ограниченный результат расширения, а не
предметную ветку конкретной игры.

### 5.3. Доверенные проектные runtime-плагины

Доверенный проектный runtime-плагин допускается после publish/security gate,
который проверяет границы данных, модель изоляции и безопасность протокола.
Причина, почему декларативного пути или общего модуля недостаточно, фиксируется
для сопровождения, но не является отдельным ручным разрешением PM. Он
запускается не внутри процесса `runtime-api`, а отдельным runner с
JSON-протоколом.

Каждый такой случай обязан иметь:

- владельца;
- причину, почему манифест и общая возможность платформы недостаточны;
- список затронутых действий и данных;
- JSON Schema для входа и выхода;
- правило «без прямой мутации состояния»: плагин возвращает только
  schema-validated значение или change set в объявленной области записи;
- таймаут, лимит размера ответа и диагностику runner-а;
- тесты;
- диагностику и журналирование;
- срок или условие пересмотра; перенос в платформу требуется только если
  механизм стал общим, а не потому, что сам игровой код всегда временный;
- точный hash/version в игровом bundle и путь к более строгой изоляции при
  изменении модели угроз.

Новое архитектурное согласование требуется только если расширению нужны права,
которых нет в принятом контракте, ослабление изоляции, новый источник истины
или существенно иная эксплуатационная модель.

Такой код не становится прецедентом для marketplace и не может использоваться сторонними авторами.

### 5.4. Целевой путь для остальных runtime-плагинов

Для сторонних, marketplace и иных недоверенных runtime-плагинов целевой вариант - изолированный исполнитель:

- отдельный процесс вне процесса `runtime-api`;
- контейнер, микро-виртуальная машина или сопоставимый механизм изоляции;
- запрет сети, файловой системы, переменных окружения и запуска процессов по умолчанию;
- права выдаются явно из `plugin.json`;
- вход и выход описаны JSON Schema;
- плагин возвращает schema-validated значение или ограниченный change set, но
  не мутирует состояние напрямую;
- `runtime-api` проверяет ответ и применяет только разрешенные изменения;
- каждый вызов имеет таймаут, лимит размера входа/выхода и запись диагностики.

Один и тот же протокол вызова должен подходить и доверенному проектному runner-у, и будущему изолированному runner-у. Тогда ранний trusted этап не заблокирует marketplace.

## 6. Декларативный язык механик

Нормативным целевым языком является типизированный транзакционный Mechanics IR
из ADR-084. Он универсален для разных жанров и позволяет авторам собирать
правила из общих операций, не добавляя новый runtime-обработчик для каждого
предметного глагола. Селекторы, агрегирование, структурное ветвление,
последовательности и отложенные действия имеют проверяемые лимиты выполнения.

Старый `deterministic.effects[]` остаётся только миграционным входом. Нормативные
правила типов, ссылок, информационных потоков, транзакций и структурных узлов
определяет ADR-084; произвольные JSON Pointer и runtime-подстановки не являются
целевыми типизированными ссылками.

Целевые архитектурные правила:

- структура описывается JSON Schema 2020-12;
- операции и ссылки типизированы, а варианты закрыты от неизвестных полей;
- алгоритмы являются чистыми вычислениями, а состояние, случайность и события
  меняют только команды;
- структурные узлы работают только с ограниченными коллекциями и лимитом
  числа шагов;
- нет доступа к сети, файлам, времени, переменным окружения и случайности без отдельной общей возможности платформы;
- результат применения должен быть проверяемым и журналируемым;
- одна операция должна быть понятной автору манифеста и редактору.

Целевое направление — полностью удалить `effects[]` и использовать Game Intent
с `actionId`, заранее скомпилированный Mechanics IR и изолированное игровое
extension только там, где декларативной выразительности действительно
недостаточно. Размер словаря определяется не искусственным пределом, а
самостоятельной проверяемой семантикой операций.

Поддерживающий документ: `docs/architecture/runtime-mechanics-language.md`.

## 7. Расширение runtime-api

Новая серверная возможность добавляется в `runtime-api` только если она проходит классификацию:

- **общая**: подходит классу игр, например квестам, стратегиям, карточным выборам, командным выборам, обучающим симуляциям;
- **привязанная к конкретной игре**: нужна только одной игре или зависит от конкретного мира, персонажей, названий сцен или локальной структуры.

Общая возможность может попасть в платформу. Логика, привязанная к конкретной
игре, должна остаться в манифесте этой игры или в изолированном игровом
runtime-расширении.

Требования к общей возможности:

- JSON Schema обновляется до кода;
- контракт документируется;
- runtime handler называется по общей возможности, а не по игре;
- тесты покрывают реальный игровой сценарий и нейтральную фикстуру;
- handler не читает `gameId`, кроме как для диагностики;
- handler не импортирует plugin/game code;
- команды применяются атомарно: либо весь типизированный план зафиксирован,
  либо действие отклонено;
- все изменения попадают в журнал с `actionId`, версией контракта и диагностикой.

Разрешенные семейства возможностей должны называться общими словами, например:

- `game.timeline`;
- `game.metric`;
- `game.flag`;
- `game.collection`;
- `game.log`;
- `game.random`, только если есть детерминированный seed;
- `integration.*`, только после отдельной политики внешних вызовов.

## 8. Варианты и статус

### 8.1. Расширять манифесты и платформенные возможности

Статус: **обязательный вариант везде, где это возможно**.

Плюсы:

- лучше проверяется схемой;
- безопаснее;
- проще для редактора и предпросмотра;
- сохраняет повторяемость тестов;
- не допускает код, привязанный к конкретной игре, в `runtime-api`.

Минусы:

- не покрывает все алгоритмы и внешние интеграции;
- при неаккуратном развитии может раздуть схему.

### 8.2. Встроенный JavaScript в процессе runtime-api

Статус: **не является архитектурой runtime-плагинов**.

Плюсы:

- самый простой путь для платформенного кода;
- минимальные накладные расходы;
- удобно для временной внутренней проверки гипотезы.

Минусы:

- ошибка может повредить весь процесс `runtime-api`;
- код получает слишком много возможностей процесса;
- это не путь для marketplace;
- легко скатиться к импортам, привязанным к конкретной игре.

Оценка: не использовать как runtime-плагин. Если код нужен как часть платформы, он проходит обычный platform review и не может быть привязан к конкретной игре. Если нужен доверенный проектный runtime-плагин, используется отдельный процесс с JSON-протоколом.

### 8.3. `node:vm` или `worker_threads`

Статус: **только как внутренний ускоренный вариант, не как защита для чужого кода**.

Эти механизмы могут быть полезны внутри доверенного runner-а для ускорения
вычислений или изоляции задачи от event loop, но не являются целевой безопасной
границей для стороннего runtime-кода. Их использование не отменяет правило:
runtime-плагин общается с `runtime-api` через schema-first JSON-протокол и
возвращает только разрешённое значение или change set.

### 8.4. Отдельный процесс с JSON-протоколом

Статус: **принятый вариант для доверенных проектных серверных плагинов**.

Он задает протокол вызова, таймауты, диагностику и защиту `runtime-api` от падения плагина. Сам по себе такой процесс не является достаточной песочницей для marketplace-кода, поэтому сторонний код должен идти через вариант 8.5 или 8.6.

### 8.5. Контейнер или микро-виртуальная машина

Статус: **целевой вариант для marketplace runtime-плагинов, которым нужен обычный процесс и внешняя изоляция**.

Плюсы:

- изоляция сильнее, чем у встроенного кода;
- можно явно применять права из `plugin.json`;
- сбой runner-а не обязан ронять `runtime-api`;
- подходит для будущего marketplace.

Минусы:

- дороже в эксплуатации;
- сложнее локальная разработка;
- нужны лимиты, доставка runner-а и управление холодным стартом.

### 8.6. WebAssembly/WASI

Статус: **целевой вариант для чистых вычислительных marketplace-обработчиков, когда задача подходит под WASI-модель**.

WASI хорошо сочетается с моделью явных возможностей, но выбор конкретной среды выполнения должен быть отдельным решением. Node.js WASI сам по себе не принимается как защита для недоверенного кода.

### 8.7. Внешний сервисный плагин

Статус: **только для интеграций, не как основной путь deterministic gameplay**.

Сетевой вызов сложнее повторять, тестировать и откатывать. Такой сервис может вернуть рекомендацию или событие, но не должен применять состояние напрямую.

## 9. Последствия

Положительные:

- `runtime-api` остается универсальным, без веток под конкретные игры;
- большая часть механик остается проверяемыми данными;
- редактор получает основу для визуальных форм и диагностики;
- доверенные проектные и будущие marketplace runtime-плагины получают один протокол вызова;
- marketplace не блокируется ранними внутренними решениями.

Стоимость:

- нужно развивать JSON Schema, compiler/checker и исполнитель Mechanics IR
  согласованно;
- новые общие возможности добавляются через JSON Schema и нейтральные runtime
  modules без веток под конкретную игру;
- доверенные проектные runtime-плагины требуют отдельного процесса, строгого ревью и учета долга;
- отдельный runner дороже прямого импорта кода.

## 10. Источники

- Node.js `vm`: `node:vm` не является механизмом безопасности для недоверенного кода: https://nodejs.org/api/vm.html
- Node.js `child_process`: `exec()` запускает shell-строку; для известных команд безопаснее использовать `execFile`/`spawn` с явными аргументами, таймаутом и отменой: https://nodejs.org/api/child_process.html
- JSON Schema object reference: осторожность с `additionalProperties` и составными схемами: https://json-schema.org/understanding-json-schema/reference/object
- JSON Patch RFC 6902: последовательные операции изменения JSON-документа: https://www.rfc-editor.org/rfc/rfc6902
- JsonLogic: JSON-формат правил с операторами и `var`: https://jsonlogic.com/
- Common Expression Language: переносимый и безопасный expression language, полезный как ориентир, но не выбранный сейчас runtime: https://cel.dev/
- Wasmtime security: WASI filesystem access follows a capability-based security model: https://docs.wasmtime.dev/security.html
- Node.js WASI: Node.js threat model не дает такой же secure sandboxing, как некоторые WASI runtimes: https://nodejs.org/api/wasi.html
- Deno permissions: пример deny-by-default модели доступа к файлам, сети, env и subprocess: https://docs.deno.com/runtime/fundamentals/security/
- Kubernetes Pod Security Standards: ориентир для restricted workload isolation в контейнерной среде: https://kubernetes.io/docs/concepts/security/pod-security-standards/
