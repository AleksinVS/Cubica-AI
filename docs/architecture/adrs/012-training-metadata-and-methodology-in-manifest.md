# ADR-012: Обучающие метаданные и методические материалы в манифесте игры

## Контекст

Платформа Cubica позиционируется как платформа бизнес‑игр и тренингов. Для методистов и ведущих важно понимать:

- какие **компетенции** (навыки, модели поведения) тренируются в конкретной игре;
- в каком **формате** проводится игра (один игрок, команда за одного персонажа, полноценный мультиплеер);
- какова типичная **продолжительность** сессии;
- какие есть **методические материалы** для участников и для ведущих (рекомендуемые вопросы, интерпретации, подсказки, чек‑листы).

Текущая структура Game Manifest (см. `docs/architecture/schemas/manifest-structure.md`, ADR‑001, ADR‑008, ADR‑009) описывает:

- технические метаданные (`meta.id`, `meta.version`, `schema_version`, `min_engine_version`);
- конфигурацию игроков (`config.players`);
- игровые правила и сценарий (`assets.rules`, `assets.scenario`, `assets.scripts`);
- состояние, UI и обработчики действий.

При этом **обучательные цели** и **методические материалы** находятся только во внешних документах (README, презентации) и не привязаны формально к манифесту. Это делает сложными:

- поиск и фильтрацию игр по тренируемым компетенциям в каталоге;
- автоматическую подготовку ведущего (Game Editor / Game Catalog не знают, какие методички нужно показать);
- единое версионирование игровых и методических материалов.

Нужно расширить манифест так, чтобы:

- обучательные атрибуты игры были частью `meta` и могли использоваться бэкендом и UI;
- ссылки на методические `.md`‑файлы (для участников и ведущих) были декларативно описаны в `assets`.

## Альтернативы

- **Хранить обучательные данные только в отдельном каталоге/БД**  
  Плюсы: не усложняем схему манифеста.  
  Минусы: требуется дополнительный источник истины, сложнее поддерживать согласованность версий игры и методических материалов; редакторам сложнее понимать, какие файлы относятся к конкретной игре.

- **Кодировать компетенции в тегах или названии игры**  
  Плюсы: не нужно менять схему.  
  Минусы: теги плохо структурированы, нет возможности хранить развёрнутые описания компетенций; нельзя формализовать формат/продолжительность; нет явных ссылок на методические материалы.

- **Расширить Game Manifest отдельными полями в `meta` и `assets`** (выбранный вариант)  
  Плюсы: единый источник истины для контента и методики; данные доступны и движку, и каталогу, и редактору; легко валидировать через JSON Schema.  
  Минусы: небольшое усложнение схемы манифеста.

## Решение

Расширить Game Manifest двумя группами полей:

1. **Обучающие метаданные в `meta.training`**.
2. **Методические материалы в `assets.methodology`**.

### 1. `meta.training`: компетенции, формат и продолжительность

В секции `meta` вводится объект `training`, описывающий обучательный контекст игры:

```json
"meta": {
  "id": "com.cubica.antarctica",
  "version": "1.0.0",
  "schema_version": "1.0",
  "min_engine_version": "0.1.0",
  "name": "Antarctica",
  "training": {
    "competencies": [
      {
        "id": "leadership",
        "name": "Лидерство",
        "description": "Отработка принятия решений и влияния в условиях неопределённости."
      }
    ],
    "format": "single_team",
    "duration": {
      "min_minutes": 60,
      "max_minutes": 90
    }
  }
}
```

Семантика полей:

- `training.competencies` — массив описаний **тренируемых компетенций**:
  - `id` — стабильный машинный идентификатор (например, `communication`, `strategy-thinking`);
  - `name` — человекочитаемое название компетенции;
  - `description` — краткое описание того, как именно эта компетенция раскрывается в данной игре.

- `training.format` — строковый код **формата игры**:
  - `"single"` — однопользовательская игра: один игрок управляет одним персонажем;
  - `"single_team"` — однопользовательская для команды: несколько участников совместно принимают решения за одного персонажа/роль;
  - `"multi"` — многопользовательская игра: несколько игроков/ролей в одной игровой сессии.

- `training.duration` — объект с оценкой **продолжительности**:
  - `min_minutes` — минимальная рекомендуемая длительность сессии в минутах;
  - `max_minutes` — максимальная рекомендуемая длительность (время, после которого игра обычно завершается или требует перерыва).

Эти поля используются:
- Game Catalog и Game Editor — для фильтрации игр, отображения целей тренинга и планирования сессий;
- методистами — для согласования программы обучения и тайминга;
- Router / Game Engine — для ориентировочной настройки TTL сессий и лимитов (в будущем, без жёсткой привязки).

### 2. `assets.methodology`: методические материалы в `.md`

В секции `assets` добавляется объект `methodology` со ссылками на Markdown‑файлы:

```json
"assets": {
  "rules": "assets/rules.md",
  "scenario": "assets/scenario.md",
  "scripts": "assets/game.js",
  "images": {
    "hero_icon": "assets/img/hero.png"
  },
  "methodology": {
    "participants": "assets/methodology/antarctica-participants.md",
    "facilitators": "assets/methodology/antarctica-facilitators.md"
  }
}
```

Семантика:

- `methodology.participants` — путь к `.md`‑файлу с материалами для **участников**:
  - правила игры в текстовой форме;
  - описания объектов, персонажей, переменных и метрик;
  - общие рекомендации/модели, на которые можно опираться при принятии решений;
  - подсказки для конкретных ситуаций (например, разбор типичных ошибок на ключевых шагах);
  - чек‑листы, если игра тренирует конкретный алгоритм (набор шагов, с которым участники сверяются по ходу игры).

- `methodology.facilitators` — путь к `.md`‑файлу с материалами для **ведущих/тренеров**:
  - интерпретации и пояснения, что означают те или иные действия игроков;
  - рекомендуемые вопросы к определённым событиям/ходам;
  - возможные подсказки и наводящие вопросы, которые ведущий может использовать;
  - критерии оценки, индикаторы паттернов мышления или поведения, которые игра помогает диагностировать.

Эти файлы:
- хранятся рядом с манифестом и версионируются вместе с ним;
- подгружаются Game Editor, Game Catalog или отдельным «методическим» интерфейсом;
- могут частично включаться в промпты LLM (например, для генерации подсказок или пояснений).

### Обновление схемы и спецификаций

- `docs/architecture/schemas/manifest-structure.md` дополнен описанием `meta.training` и `assets.methodology`, а также примером.
- `docs/architecture/schemas/game-manifest.schema.json` обновлён:
  - `meta` теперь требует поля `schema_version` и `min_engine_version` (в соответствии с ADR‑008);
  - добавлен объект `meta.training` с полями `competencies`, `format`, `duration`;
  - добавлена секция `assets` с подпунктом `methodology.participants|facilitators`.

## Последствия

Положительные:
- Манифест становится единым источником истины не только для игровой логики и UI, но и для обучательных целей.
- Каталог и редактор могут фильтровать игры по компетенциям и формату, отображать ожидаемую длительность.
- Методические материалы явно связаны с конкретной версией игры и не теряются при изменениях.

Негативные/риски:
- Требуется дисциплина при заполнении `training` и поддержании актуальности `.md`‑файлов.
- Появляется небольшое дополнительное поле для валидации схемы; старые манифесты без `schema_version`/`min_engine_version` нужно либо мигрировать, либо трактовать как `schema_version = "1.0"` (как описано в ADR‑008).

## План внедрения

1. Обновить `docs/architecture/schemas/manifest-structure.md` и `docs/architecture/schemas/game-manifest.schema.json` (выполнено).
2. Добавить использование `meta.training` и `assets.methodology` в новых манифестах (начиная с сценария «Antarctica» и последующих игр).
3. Обновить инструменты редактирования (Game Editor) и каталог игр, чтобы:
   - отображать компетенции, формат и длительность в UI;
   - при необходимости подтягивать и отдавать методические `.md`‑файлы.
4. Запланировать миграцию существующих манифестов (если появятся) на новую схему с заполнением обучательных полей.
