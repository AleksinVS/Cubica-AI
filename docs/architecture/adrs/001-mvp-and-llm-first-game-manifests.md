## Контекст

Платформа Cubica разрабатывается как система для запуска разных «игр» и обучающих сценариев, в которой:
- пользовательские интерфейсы могут отличаться по стеку (Next.js, Telegram, mobile, Phaser и т.д.);
- игровые правила и сценарии должны быть данными, а не зашитым кодом;
- в роли игрового «движка» используется LLM (Large Language Model, большая языковая модель), которой нужно передавать структурированный контекст.

В `PROJECT_OVERVIEW.md` уже зафиксирован шаблон MVP (Model–View–Presenter) как базовый подход к фронтенд‑архитектуре. Возникает необходимость зафиксировать более конкретные принципы:

- как MVP сочетается с LLM-first архитектурой;
- как описывать игры через JSON‑манифесты;
- как учитывать, что разные игры могут требовать разных структур манифеста;
- как разделять данные/логику и описание UI;
- как роль LLM влияет на формат и хранение игровых данных.

Отсутствие единых принципов ведёт к риску:
- что каждая игра будет придумывать собственный формат «манифеста» без явной схемы;
- что Model/View/Presenter для разных игр будут расходиться по контрактам;
- что LLM будет получать хаотичный контекст, затрудняющий контроль качества и развитие движка.

## Альтернативы

- **Продолжать неформально**: проектировать манифесты и MVP‑слой для каждой игры отдельно, без общего ADR.
  - Плюсы: минимум формальной работы на старте.
  - Минусы: быстрое накопление несовместимых форматов, высокий порог входа для новых разработчиков и сложность эволюции протоколов.

- **Ввести один жёсткий формат манифеста для всех игр**:
  - Плюсы: простая реализация, единая схема.
  - Минусы: разные жанры игр и сценарии потребуют разных структур; один формат приведёт либо к избыточной универсальности, либо к взрывному росту условностей внутри одного JSON.

- **Зафиксировать набор архитектурных принципов + допустить вариативные схемы для манифестов** (выбранный вариант):
  - Плюсы: общее основание (MVP, LLM-first, контракт Presenter↔View↔Model), при этом каждая игра может иметь собственную схему манифеста.
  - Минусы: требуется дисциплина в описании и хранении схем, а также валидация манифестов.

## Решение

Принять следующие архитектурные принципы для игровых манифестов и клиентской части:

1. **Проект основывается на шаблоне MVP и LLM-first архитектуре.**
   - Для фронтенда и SDK базовым паттерном является MVP (Model–View–Presenter).
   - Игровой слой строится по принципу LLM-first: LLM рассматривается как основной «движок» игр, принимающий структурированный контекст и возвращающий решения/ходы.

2. **JSON‑манифест игры и его схема.**
   - Игры описываются JSON‑манифестами: структурированными документами, включающими:
     - игровые данные и правила;
     - описание шагов/сцен, сущностей и ограничений;
     - ссылки или разделы, описывающие UI‑структуру (экраны, области, виджеты).
   - Структура JSON‑манифеста _может различаться у разных игр_: каждая игра или семейство игр может вводить собственные разделы и поля.
   - Вместе с каждым манифестом **обязательно хранится его схема** (например, JSON Schema или эквивалент), описывающая допустимую структуру, типы полей и инварианты для данной игры.

3. **Вариативные Model, View и Presenter между играми.**
   - Каждая игра может определять свои Model, View и Presenter (например, специфические типы экранов, дополнительные состояния, отдельный набор виджетов).
   - При этом сохраняются ключевые требования MVP:
     - View не знает о Model и не работает напрямую с хранилищами/LLM;
     - Model не знает о View и не содержит логики отображения;
     - Presenter выступает единственной точкой координации между Model и View.

4. **Model и View не знают друг о друге и взаимодействуют через Presenter.**
   - В рамках любой игры:
     - View обращается только к Presenter (через контекст/хуки/протоколы) за данными и отдаёт туда пользовательские действия.
     - Presenter вызывает Model (или сервисы, предоставляющие Model) и преобразует их ответы в формат, удобный View.
     - Манифесты и схемы не нарушают это разделение: они описывают данные и UI, но не делают View «умнее» в обход Presenter.

5. **Данные и логика хранятся совместно, а описание UI — отдельно.**
   - Игровые данные (факты, состояния, правила, ограничения, параметры) и логика (в том числе та, с которой работает LLM) группируются в одном логическом блоке манифеста или связанных с ним ресурсах.
   - Описание UI (расположение элементов, типы виджетов, визуальные атрибуты) либо:
     - хранится в отдельных разделах манифеста, явно отделённых от чисто игровых данных;
     - либо вынесено в отдельные файлы/ресурсы, на которые ссылается манифест.
   - Цель: минимизировать смешение неструктурированных UI‑деталей с логикой, важной для LLM.

6. **Манифест содержит системный промт для LLM.**
   - В манифест добавляется явный раздел/поле системного промта (`system_prompt` или эквивалент), который:
     - задаёт роль LLM (игровой «движок», ведущий, симулятор персонажей и т.п.);
     - описывает допустимые операции и формат ответов (например, список разрешённых действий, JSON‑формат патчей состояния).
   - Этот промт рассматривается как часть контракта между Presenter и LLM и должен версионироваться вместе с манифестом.

7. **LLM как игровой «движок».**
   - Game Engine/Router передаёт LLM на вход:
     - JSON‑манифест (или ссылку на него + необходимые фрагменты);
     - текущее состояние игры;
     - дополнительные параметры сессии (игрок, язык, сложность).
   - В ответ LLM возвращает структуру, содержащую:
     - обновлённое состояние или список патчей к Model;
     - указания Presenter по изменению View (например, какой экран/область активировать, какие элементы скрыть/показать).
   - Presenter интерпретирует ответ LLM строго по согласованной схеме и:
     - обновляет Model;
     - диспатчит изменения в View (через контекст/Redux/собственные механизмы).

## Последствия

Положительные эффекты:
- Единая концепция для всех игр: MVP на уровне клиентов и LLM-first на уровне игрового движка.
- Возможность поддерживать разные жанры игр, сохраняя общие принципы и точки расширения.
- Чёткое разделение данных/логики и описания UI облегчает эволюцию и реюз логики между разными интерфейсами.
- Наличие схем для манифестов даёт основу для валидации, статического анализа и миграций.

Риски и технический долг:
- Требуется дисциплина в сопровождении схем (любое изменение формата манифеста должно сопровождаться обновлением схемы и ADR/документации).
- Ошибки в проектировании системного промта или формата ответов LLM могут приводить к трудноотлавливаемому поведению.
- Появляется необходимость в конвейере валидации (schema validation) для манифестов и их обновлений.

Требуемые изменения:
- Дополнение `PROJECT_OVERVIEW.md` разделом про MVP + LLM-first и игровые манифесты.
- Создание описаний схем игровых манифестов в `docs/architecture/schemas/` и/или `games/*`.
- Обновление документации задач (ROADMAP, Milestone/Epic/Feature), чтобы первая итерация была посвящена проектированию формата манифеста и схем.
