# ADR-070: Единый каталог проектных навыков

## Контекст

Проектные навыки (локальные инструкции в `SKILL.md`, которые уточняют работу агента для конкретной задачи) сейчас физически находятся в `.codex/skills/`. Этот путь является механизмом автоматического обнаружения Codex, но не выражает того, что навыки принадлежат проекту и должны быть одинаково доступны Codex, встроенным субагентам и внешним агентам.

Размещение канонических файлов в каталоге конкретного агента создает два риска: другие агенты получают копии или не получают навык вовсе, а различия между копиями приводят к дрейфу инструкций. Также требуется различать навыки, созданные в Cubica, и предметные навыки, адаптированные из внешних репозиториев.

## Решение

1. Корневой каталог `skills/` является единственным каноническим местом проектных навыков.

   Каждый активный проектный навык хранится в `skills/<имя>/SKILL.md` вместе с его скриптами, справочными материалами и метаданными. Копировать эти файлы в каталоги отдельных агентов запрещено.

   `skill-candidates/` является отдельным неактивным хранилищем внешних снимков. Его `SKILL.md` служат только референсами или входом управляемой адаптации, не считаются проектными навыками и не могут входить в реестр либо мосты обнаружения до создания очищенной версии в `skills/`.

2. Внутренние навыки Cubica именуются с префиксом `C_`.

   Каноническое имя каталога и идентификатор в общем реестре имеют вид `C_<назначение>`, например `C_cubica` и `C_external-skill-adapter`. Техническое поле `name` сохраняет совместимый с Codex строчный формат и для `C_cubica` остается `cubica`, чтобы не изменить явный вызов `$cubica`. Префикс не применяется к навыкам, адаптированным из внешнего источника: они сохраняют свое предметное имя, например `debugging-and-error-recovery`.

3. Каждый агент использует мост обнаружения к общему каталогу, а не собственную копию.

   Для Codex `.codex/skills/<имя>` является символической ссылкой на `../../skills/<имя>`. Другой агент использует аналогичный поддерживаемый его средой мост либо получает путь `skills/` в стартовом контексте. Если среда агента не поддерживает автоматическое обнаружение, она обязана читать релевантный навык непосредственно из `skills/`.

4. Общий реестр описывает происхождение и способ включения.

   Реестр в `skills/registry.json` фиксирует имя, назначение, происхождение, режим включения, обязательные зависимости и путь к каноническому навыку. Зависимость выражается полем `requires` со списком канонических имён каталога. Валидатор запрещает ссылки на отсутствующие навыки, зависимость навыка от самого себя и циклы; среда агента загружает обязательный вспомогательный навык вместе с вызывающим. Для адаптированных навыков реестр внешней адаптации продолжает хранить upstream-commit, лицензию, снимок исходника и хэши; он ссылается на путь в `skills/`, а не на агентский мост.

5. Обнаружение не меняет проектный порядок полномочий.

   `$cubica` остается навыком только явного включения. Остальные допущенные специализированные навыки могут включаться автоматически по точному назначению. Все навыки, независимо от пути обнаружения, подчиняются прямому указанию пользователя, ближайшему `AGENTS.md` и действующим ADR.

6. Миграция выполняется атомарно и проверяемо.

   При реализации сначала переносятся канонические каталоги в `skills/`, затем создаются мосты всех поддерживаемых агентов, обновляются реестры, скрипты, документация и проверки. Миграция не считается завершенной, пока не доказано отсутствие второй активной копии каждого навыка и корректное обнаружение Codex.

## Отклоненные альтернативы

- **Оставить `.codex/skills/` каноническим каталогом.** Отклонено: путь закрепляет навыки за одной средой и вынуждает остальных агентов зависеть от неявной договоренности.
- **Копировать навыки в каталог каждого агента.** Отклонено: копии неизбежно расходятся и усложняют проверку происхождения.
- **Хранить только один общий файл с инструкциями для всех способностей.** Отклонено: агенту пришлось бы загружать нерелевантный контекст, а точное автоматическое включение стало бы невозможным.
- **Переименовать адаптированные внешние навыки в `C_*`.** Отклонено: исчезает видимое различие между собственной практикой проекта и материализованной внешней способностью.
- **Хранить неадаптированные кандидаты под `skills/`.** Отклонено: канонический активный путь должен оставаться однозначным и одинаково безопасным для агентов с разными механизмами обнаружения.

## Последствия

Положительные последствия:

- все агенты получают один и тот же проверенный текст навыка;
- происхождение и тип навыка видны по имени и реестру;
- обновление выполняется один раз в каноническом месте;
- Codex сохраняет автоматическое обнаружение без дублирования файлов.

Ограничения и риски:

- каждый новый тип агента требует поддерживаемого моста обнаружения или явного пути в стартовом контексте;
- символические ссылки должны проверяться в CI, чтобы сломанная ссылка не скрыла навык;
- изменение имен и путей требует одновременного обновления реестров, скриптов и документации.
- навыки с обязательными зависимостями можно переносить или отключать только вместе с проверкой всех ссылок `requires`.
- каталог кандидатов требует отдельного реестра целостности и не должен подключаться ни одним агентским мостом.
