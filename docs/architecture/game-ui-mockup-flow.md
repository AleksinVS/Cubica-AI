## 1. Термины

- **Бриф** — структурированное описание игры или экрана (жанр, список экранов,
  ключевые сущности, настроение стиля), которое агент собирает из свободного
  текста автора через короткое интервью.
- **Изображение-концепт (концепт стиля)** — сгенерированное полноэкранное
  изображение интерфейса, задающее визуальное направление. Концепт — не
  пиксельный эталон: финальный вид собирается из структуры, дизайн-токенов и
  ассетов, а не воспроизведением картинки один в один.
- **Vision-модель** — модель, понимающая изображения (мультимодальная LLM);
  извлекает из изображения-концепта разметку зон и дизайн-токены.
- **Дизайн-токены** — именованные визуальные параметры (палитра, типографика,
  скругления, тени и т.п.); формат уже определён в design artifacts (ADR-016,
  `style_tokens`).
- **Каркасный макет** (wireframe) — структурный черновик интерфейса без
  финальной стилизации: серые контейнеры, реальные компоненты, реальная
  навигация.
- **Структурный черновик** — черновик UI authoring-манифеста каркасного уровня,
  собранный агентом из брифа; сразу отрисовывается в предпросмотре редактора.
- **Разметка зон** — семантическое описание изображения в формате
  `*.design.json` (ADR-016): зоны `regions`, элементы, их типы, подписи,
  границы, промты генерации.
- **Маппинг** — сопоставление «зона разметки → прототип каталога / игровая
  сущность», из которого агент строит `EditorChangeSet`.
- **Адаптер генерации изображений** — платформенная граница вызова внешних
  провайдеров генерации изображений; провайдеры заменяемы по образцу ADR-044.
- **Операционная политика** — платформенные правила стоимости, квот и
  разрешений на вызов внешних генеративных операций (контракты в
  `packages/contracts/ai`).
- **Мастер новой игры** — пошаговая UX-обёртка потока для создания игры с нуля
  поверх галереи шаблонов (Game Templates, ADR-057 п.10).
- **Дисциплина идентичности** — правило ADR-057: UI-элемент обязан ссылаться на
  `id` игровой сущности; направление ссылки — только UI → game.

## 2. Понимание задачи

Проектируется платформенный (не привязанный к конкретной игре) инструмент и
поток: нетехнический разработчик игры описывает интерфейс текстом → платформа
помогает получить визуальные макеты → из макетов рождается работающий прототип
интерфейса. Самый сложный вопрос — надёжная цепочка «текст → изображение-макет
→ прототип в коде».

Ограничения: решения ADR-057 (preview-first редактор, целостная сущность через
проекцию, конвейер `EditorChangeSet`, шкала риска, фикстуры состояния, кэш) —
действующие рельсы; поток макетирования строится поверх них и не пересматривает
их. Платформенная чистота обязательна (CLAUDE.md п.10, ADR-040): никаких веток
под конкретную игру в общих слоях.

## 3. Ключевое решение: прототип — это манифест, а не код

В Cubica прототип интерфейса — **не сгенерированный React-код, а UI
authoring-манифест**: экраны, панели и компоненты из каталога прототипов
(ADR-050), которые платформенный рендерер `player-web` уже отрисовывает в
предпросмотре редактора.

Следствия:

1. Не нужен генератор кода — нужен генератор манифеста. Для манифеста уже
   есть вся инфраструктура надёжности: JSON Schema как источник истины
   (ADR-025), валидация Ajv, конвейер
   `EditorChangeSet → шкала риска → dry-run → validation → undo journal`
   (ADR-057).
2. Сгенерированный агентом черновик интерфейса проходит те же ворота
   безопасности, что и любая ручная или агентская правка. Отдельный
   «генеративный» путь записи запрещён (инвариант ADR-057: независимые
   агентские стеки недопустимы).
3. Прототип с первого шага — живой: кликабельный, проходимый в режиме
   «Превью», проектируемый в контексте фикстур состояния.

## 4. Уроки индустрии

Состояние инструментов «текст/картинка → интерфейс» на 2025–2026 (источники в
[разделе 14](#14-источники-и-ориентиры)):

- **Image-first инструменты** (генерация изображений интерфейса) дают быстрые
  концепты для обсуждения, но результат «слишком случайный и
  нефункциональный»: структуру приходится пересобирать вручную. Типичная
  рекомендация практиков — «сгенерируй картинку для согласования, затем
  пересобери структуру правильно в дизайн-системе или коде».
- **Structure-first инструменты** (генерация структуры поверх существующей
  библиотеки компонентов — Figma Make, v0) работают тем лучше, чем богаче и
  дисциплинированнее библиотека компонентов, на которую опирается генерация.
- У Cubica «библиотека компонентов» уже есть: каталог платформенных и игровых
  прототипов ADR-050 плюс JSON Schema UI-манифеста. Это прямо указывает на
  structure-first как основную дорожку.

## 5. Предлагаемая архитектура: две дорожки с общим мостом

Принцип (предлагается к принятию):

> **Структура — всегда источник истины прототипа; изображение — концепт стиля
> и источник ассетов, а не источник структуры напрямую.**

Мост между дорожками — существующий формат дизайн-артефактов ADR-016
(`*.design.json`): семантическая разметка зон изображения с типами, подписями,
границами, промтами генерации и дизайн-токенами. Формат был спроектирован
ровно для того, чтобы ИИ-агент понимал структуру макета-изображения; поток
макетирования делает его рабочей валютой между картинкой и манифестом.

Схема потока:

```text
        текст автора
             │
        [1. Бриф] ──────────────────────────────┐
             │                                   │
   [2. Структурный черновик]           [3. Концепты стиля]
   UI authoring-манифест (каркас)      изображения + разметка
   сразу живёт в предпросмотре         зон + дизайн-токены
             │                                   │
             └────────── [4. Слияние] ◄──────────┘
                  маппинг зон → прототипы/сущности,
                  токены → манифест, ассеты по зонам
                             │
                      [5. Итерации]
              регионный промт, панель сущности,
              промт-проекция, диагностики (ADR-057)
```

## 6. Поток из пяти шагов

### 6.1. Шаг 1 — бриф

- Автор описывает игру или экран свободным текстом в чате агента.
- Агент ведёт короткое интервью и превращает текст в структурированный бриф:
  список экранов, ключевые игровые сущности, настроение стиля, канал (для MVP —
  web).
- Механизм — обычный агентский интент по ADR-057; новых контрактов записи не
  требуется, требуется сценарий агента и схема брифа как промежуточного
  артефакта интента (не источник истины).

### 6.2. Шаг 2 — структурный черновик

- Агент из брифа генерирует черновик UI authoring-манифеста каркасного уровня:
  экраны из прототипов каталога ADR-050, контейнеры, компоненты — плюс
  заготовки игровых сущностей в game authoring-манифесте.
- Дисциплина идентичности соблюдается с первого шага: каждый содержательный
  UI-элемент ссылается на `id` игровой сущности; декоративные элементы
  объявляются декоративными (ADR-057).
- Черновик применяется одним атомарным `EditorChangeSet` (кросс-манифестное
  создание — уже принятая операция ADR-057 п.10) и **сразу отрисовывается в
  предпросмотре**: автор с первой минуты видит работающий серый интерфейс.
- Скорость — секунды; это быстрая дорожка по умолчанию.

### 6.3. Шаг 3 — концепты стиля

- По брифу адаптер генерации изображений создаёт 2–4 полноэкранных
  изображения-концепта (медленная дорожка, явное действие пользователя за
  операционной политикой — по образцу генерации ассетов из
  `editor-preview-first-ux.md` §9.4).
- Автор выбирает концепт и комментирует; доступны «перегенерировать с тем же
  промтом» и «изменить промт».
- Из выбранного концепта vision-модель извлекает: (а) дизайн-токены — палитру,
  типографику, форму элементов; (б) разметку зон в формате `design.json`.
- Изображение сохраняется как design artifact типа `concept`/`mockup` в
  `games/<id>/design/` с метаданными происхождения (`generation.prompt`) и
  записью в `design-history.json` — всё по ADR-016, формат уже существует.
- Разметка зон обязательно проходит правку человеком: оверлей поверх картинки
  для корректировки границ, типов и подписей зон (переиспользует паттерн
  preview selection overlay из ADR-036/057).

### 6.4. Шаг 4 — слияние стиля и структуры

- Агент строит маппинг «зона разметки → элемент структурного черновика /
  прототип каталога / игровая сущность» и возвращает `EditorChangeSet`:
  дизайн-токены применяются к UI-манифесту, подписи и содержимое зон
  сопоставляются с сущностями.
- Для декоративных зон (фон, иллюстрации, рамки) — генерация ассетов по
  зонным промтам (`generation.prompt` из разметки) через принятый поток ассетов
  (`editor-preview-first-ux.md` §9.4): явное действие, операционная политика,
  метаданные происхождения, `fileCreates` в `EditorChangeSet`.
- Отчётность маппинга — по образцу «трёх корзин» интерпретатора ADR-057:
  сопоставлено (с указанием цели) / распознано без изменений / не сопоставлено
  (показывается автору явно). Молчаливое выбрасывание зон запрещено.
- Честный принцип для автора, показываемый в UI: **картинка — концепт, а не
  пиксельный эталон**; соответствие финального вида концепту достигается
  токенами и ассетами.

### 6.5. Шаг 5 — итерации

Дальше работают уже принятые механизмы ADR-057 без новых изобретений:

- регионный промт на предпросмотре («сделай эти карточки крупнее»);
- промт-строка панели сущности и текстовый режим промт-проекции (ADR-049);
- диагностики целостности: «вид без смысла», «смысл без вида», зоны без
  маппинга;
- шкала риска и approval envelope для опасных операций (ADR-047/057).

### 6.6. Порядок шагов и обратная дорожка

Автор может пройти шаги в любом порядке:

- **Дорожка по умолчанию**: 1 → 2 → (3 → 4 по желанию) → 5. Сначала структура,
  стиль потом.
- **Обратная дорожка («сначала картинки»)**: 1 → 3 → 4 → 2-как-результат:
  разметка концепта порождает структурный черновик через маппинг зон на
  прототипы. И в этом случае структура рождается через `design.json` и каталог
  прототипов, а не «из пикселей в код».

## 7. Новые платформенные контракты и компоненты

1. **Адаптер генерации изображений** — платформенный контракт: запрос
   (промт, дизайн-токены, размер/формат, количество вариантов) → ответ
   (варианты + метаданные происхождения). Провайдеры заменяемы; вызовы идут
   за операционной политикой и гейтами redaction/audit — по образцу ADR-044.
2. **Контракт vision-разметки** — «изображение → `design.json` + дизайн-токены»
   с валидацией результата по схеме design artifacts и обязательной правкой
   человеком (оверлей зон). Vision-провайдер — тоже заменяемый адаптер.
3. **Контракт маппинга** — «разметка зон + структурный черновик + каталог
   прототипов → `EditorChangeSet` + отчёт трёх корзин». Вход и выход
   фиксируются как framework-neutral контракты (`packages/contracts/*`);
   применение — только через единый конвейер редактора.
4. **Мастер новой игры** — UX-обёртка шагов 1–4 поверх галереи шаблонов
   (ADR-057 п.10: новая игра — из шаблона или минимального каркаса, никогда не
   пустые файлы). Тот же поток в режиме «один экран» доступен внутри
   редактора для существующей игры (точка входа — создание экрана в дереве
   сущностей).

## 8. Классификация: общее и игровое

По правилу платформенной чистоты (CLAUDE.md п.10):

- **Общие (платформенные) механики**: адаптер генерации изображений, контракт
  vision-разметки, контракт маппинга, мастер новой игры, схема брифа,
  диагностики маппинга. Живут в платформенных контрактах, editor-engine и
  editor-web.
- **Игровые артефакты**: концепты, разметка, дизайн-токены, ассеты, манифесты —
  живут в пакете конкретной игры (`games/<id>/design/`, `games/<id>/assets/`,
  authoring-манифесты). Платформа не хранит и не знает game-specific стилей.

## 9. Отклоняемые альтернативы (предложение)

Предлагаются к отклонению при оформлении ADR:

- **Image-first как единственный источник структуры** («из пикселей в код»
  напрямую). Причина: подтверждённый индустрией разрыв между красивой
  картинкой и работающей структурой; пересборка вручную дороже, чем
  structure-first с самого начала.
- **Генерация framework-кода (React и т.п.) вместо манифеста.** Причина:
  ломает декларативную модель (ADR-025), лишает результат валидации схемами и
  конвейера `EditorChangeSet`, привязывает игру к каналу.
- **Пиксельное воспроизведение концепта рендерером.** Причина: недостижимо и
  не нужно; порождает ложные ожидания. Соответствие достигается токенами и
  ассетами.
- **Отдельный новый формат макета рядом с `design.json`.** Причина: ADR-016 уже
  покрывает разметку зон, токены и историю; третий формат — дрейф.
- **Автоматическое применение сгенерированной структуры мимо конвейера
  ChangeSet.** Причина: нарушает инвариант ADR-057 о едином конвейере записи.

## 10. Риски

- **Тонкий платформенный каталог прототипов**: structure-first настолько
  хорош, насколько богат каталог ADR-050; на старте платформенный уровень
  почти пуст (прототипы Antarctica — game-level). Смягчение: мастер начинает с
  шаблонов игр; повышение прототипов до платформенного уровня — по принятому
  процессу ADR-050.
- **Качество vision-разметки**: зоны и типы могут распознаваться неточно.
  Смягчение: обязательная правка человеком, отчёт трёх корзин, eval-фикстуры
  разметки и маппинга (контур ADR-038).
- **Ожидания автора от картинки**: концепт может восприниматься как обещание
  финального вида. Смягчение: явная подача «концепт ≠ финальный вид» в UI,
  быстрая дорожка по умолчанию.
- **Стоимость генерации**: изображения и vision-вызовы платные. Смягчение:
  операционная политика (квоты, подтверждение), генерация — только явным
  действием пользователя, телеметрия расходов.

## 11. Открытые вопросы

1. **Роль изображения-макета** (ключевой): подтвердить принцип «структура —
   источник истины, изображение — концепт стиля и ассетов». Альтернатива
   (изображение как обязательный первый артефакт, из которого всё выводится)
   предложена к отклонению — см. [раздел 9](#9-отклоняемые-альтернативы-предложение).
2. **Объём MVP**: предложение — один канал (web), поток «один экран за раз»,
   генерация изображений только для концептов и ассетов.
3. **Шаблоны игр**: считать ли галерею шаблонов частью этого потока (бриф
   может начинаться с «выбери похожий шаблон») или отдельным срезом.
4. **Хронология для новой игры**: мастер создаёт проект из шаблона/каркаса до
   или после генерации концептов (где хранятся концепты, если проекта ещё
   нет).
5. **Выбор провайдеров** генерации изображений и vision-разметки для MVP и
   форма их операционной политики.
6. **Юзабилити-валидация** потока на нетехнических авторах (по образцу
   валидации дерева из ADR-057) — состав задач и порог приемки.

## 12. Зависимости и координация

- **ADR-057 / TSK-20260704** — поток макетирования потребляет панель сущности,
  регионный промт, очередь интентов, шкалу риска и фикстуры; старт реализации
  зависит от готовности соответствующих фаз
  `TSK-20260704-editor-preview-first-ux-implementation`.
- **Модуляризация editor-engine** (`TSK-20260630-editor-engine-modularization`)
  — как и для ADR-057, новые контракты не наращиваются на монолитный
  `index.ts`.
- **ADR-016** — при необходимости расширения схемы design artifacts (например,
  формализация `style_tokens` для извлечения из концептов) — правкой схемы, а
  не новым форматом.
- **ADR-044/047** — все внешние вызовы (генерация изображений, vision-разметка)
  идут через заменяемые адаптеры, redaction/audit-гейты и операционную
  политику.
- **ADR-050** — наполнение платформенного каталога прототипов повышает
  качество структурной дорожки; повышение прототипов — только по принятому
  процессу.
