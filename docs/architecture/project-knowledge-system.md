## 1. Назначение и границы

Система решает шесть задач:

1. даёт агенту короткий и актуальный путь к архитектурному контексту;
2. хранит явные смысловые связи между решениями, компонентами и ограничениями;
3. обнаруживает устаревшие страницы, дубли, сломанные связи и неразрешённые
   противоречия;
4. измеряет качество поиска, чтобы усложнение извлечения происходило по фактам;
5. показывает фактические зависимости модулей и ссылки на программные символы
   для анализа влияния изменений, не смешивая их со смысловыми связями вики;
6. прослеживает путь декларативного контракта до созданных из него типов,
   проверяющего и исполняющего кода и тестов.

Вики предназначена для разработки Cubica. Она не является пользовательской
памятью продукта, хранилищем игровых сессий, журналом партий или заменой
исполняемых контрактов.

Страница вики — синтез: она объясняет назначение, причинность, границы и
последствия, но не копирует большие фрагменты ADR, схем или кода.

## 2. Источники истины

У проекта нет одного универсального документа, способного авторитетно хранить
все виды фактов. Полномочия разделены по типу сведений.

| Сведения | Авторитетный источник | Роль вики |
|---|---|---|
| Принятые архитектурные решения | `docs/architecture/adrs/` | Кратко объясняет действующие последствия и связывает решения |
| Текущая архитектурная картина | `docs/knowledge/architecture/project-architecture.md` после миграции | Является канонической страницей синтеза |
| Исполняемые структуры данных | JSON Schema, OpenAPI | Объясняет назначение и ограничения контрактов |
| Фактическое поведение | Код, тесты, игровые манифесты | Даёт навигацию и смысловой контекст |
| План и приоритеты | `NEXT_STEPS.md`, `docs/tasks/active/TSK-*` | Не дублирует очередь исполнения |
| Физическая карта репозитория | `PROJECT_STRUCTURE.yaml`, создаваемый существующим генератором | Индексирует узлы и связывает их с концептами |
| Локальные описания каталогов | `.desc.json` | Использует как источник, не заменяет |

До миграции `docs/architecture/PROJECT_ARCHITECTURE.md` остаётся каноническим
обзором. Перенос в `docs/knowledge/architecture/project-architecture.md`
выполняется отдельным этапом с обновлением всех входящих ссылок. ADR остаются
источником самих решений и истории их обоснования.

## 3. Общая архитектура

```text
 ADR · JSON Schema · код · TSK · PROJECT_STRUCTURE.yaml
                         │
                         │ sources / covers
                         ▼
             docs/knowledge/ (канон синтеза)
                         │
                    kb sync / lint
                         ▼
          .tmp/project-knowledge/kb.sqlite3
             ├─ страницы и чанки
             ├─ FTS5 / BM25
             ├─ векторные представления
             ├─ явный wiki-граф
             └─ производные узлы дерева, кода и контрактов
                         │
                  query / impact / eval
                         ▼
                       агент
```

**Канон** — версионируемые Markdown-страницы и декларативные схемы.
**Проекция** — локальное производное представление, которое можно удалить и
воспроизвести. Ни SQLite, ни векторный индекс, ни производный граф кода и
контрактов не имеют права переписывать канон автоматически.

## 4. Канон вики

### 4.1. Раскладка

```text
docs/knowledge/
  .gitattributes
  .desc.json
  index.md
  architecture/
    project-architecture.md
  domains/
  components/
  constraints/
  patterns/
  insights/
  eval/
    cases/
      <case-id>.json
```

Один файл описывает один концепт. Идентификатор концепта — стабильное поле
`id`, а не путь файла. Благодаря этому обычное переименование файла не рвёт
рёбра. `id` меняется только как отдельная миграция со сменой входящих ссылок.

### 4.2. Формат страницы

```yaml
---
id: component.runtime-api
type: component
title: Runtime API
description: Авторитетное исполнение правил и состояние игровых сессий.
timestamp: 2026-07-13T12:00:00Z
lifecycle: current
tags: [runtime, backend]
sources:
  - docs/architecture/adrs/051-api-first-contract-for-modular-monolith.md
covers:
  - services/runtime-api/
relations:
  - type: part_of
    target: domain.runtime
    provenance: docs/architecture/PROJECT_ARCHITECTURE.md
---

# Runtime API

## Назначение и границы

...
```

Обязательные поля: `id`, `type`, `title`, `description`, `timestamp`,
`lifecycle`, `sources`, `covers`, `relations`. Пустые массивы допустимы только
для нового `draft`; страница `current` обязана иметь источник и хотя бы одну
смысловую связь либо явное место в корневом индексе.

`sources` указывает, из чего получен синтез. `covers` указывает, какие пути
репозитория страница описывает. При вложенных зонах действует правило самого
глубокого совпадения пути.

## 5. Контракты JSON Schema

Декларативные структуры проверяются стандартным валидатором Ajv против JSON
Schema. Ручные проверки типов в Node.js не могут заменять схемы.

Первая реализация добавляет:

```text
docs/architecture/schemas/project-knowledge-page.schema.json
docs/architecture/schemas/project-knowledge-eval-case.schema.json
docs/architecture/schemas/project-knowledge-config.schema.json
```

YAML-frontmatter (машиночитаемый YAML-блок в начале Markdown) сначала
разбирается `js-yaml`, затем полученный объект
проверяется Ajv. Дополнительные правила, которые зависят от нескольких файлов
или Git, выполняет `kb lint`, но структура их входов остаётся в JSON Schema.

Словари типов не имеют отдельного процесса `proposed/approved`. Допустимые
значения заданы перечислениями JSON Schema. Изменение смысла типов или рёбер
проходит обычное архитектурное ревью; простое уточнение схемы является
реализационной правкой.

Проверочный поисковый случай содержит как минимум `id`, `query`, ожидаемые
идентификаторы страниц, класс `exact|semantic` и признак `critical`. Критический
точный случай защищает обязательную навигацию по известному названию, пути или
идентификатору от ухудшения при смене ранжирования.

## 6. Граф знаний

### 6.1. Типы концептов

- `domain` — крупная предметная или техническая область;
- `component` — приложение, сервис, пакет или значимый модуль;
- `integration` — внешний контур обмена;
- `pattern` — принятая повторяемая архитектурная практика;
- `constraint` — ограничение среды, безопасности или совместимости;
- `insight` — неочевидный факт, причина или известная ловушка;
- `source-summary` — краткий путеводитель по большому первоисточнику.

### 6.2. Типы рёбер

- `part_of` — входит в состав;
- `depends_on` — зависит от;
- `interacts_with` — взаимодействует с;
- `implements` — реализует решение или контракт;
- `constrained_by` — ограничен правилом;
- `supersedes` — замещает;
- `contradicts` — фиксирует содержательное противоречие;
- `distinct_from` — явно различает два похожих, но не дублирующих концепта;
- `relates_to` — слабая связь последней очереди.

Симметричные рёбра материализуются в обе стороны только в производной базе.
В Markdown хранится одна нормализованная запись, чтобы параллельные агенты не
правили два файла ради одной связи.

### 6.3. Постоянное состояние противоречия

`contradicts` — не одноразовое событие «ребро только что появилось», а
постоянное проверяемое состояние канона:

```yaml
- type: contradicts
  target: constraint.session-version-ownership
  provenance: docs/architecture/adrs/042-editor-session-versioning-and-lifecycle.md
  review:
    state: open             # open | resolved | deferred
    rationale: Два документа назначают разных владельцев версии.
    decision: null
    task: null
    review_after: null
```

- `open` выдаёт предупреждение при каждом `lint`;
- `resolved` требует `rationale` и ссылку `decision` на каноническое решение;
- `deferred` требует ссылку `task` и дату `review_after`;
- просроченный `deferred` снова становится предупреждением;
- отсутствие `review` является ошибкой схемы.

Разрешение хранится в каноне и поэтому не теряется при полном rebuild.

## 7. Поиск

### 7.1. Полнотекстовый режим

Штатный режим первой версии — SQLite FTS5 (полнотекстовый индекс) с BM25
(формула ранжирования по совпадениям и редкости слов). Страница делится на чанки по
заголовкам Markdown; заголовок, описание, теги и идентификатор индексируются с
повышенным весом. Поиск возвращает страницу, подходящие разделы, причины
ранжирования и ближайшие явные рёбра.

### 7.2. Векторная возможность

Векторный поиск реализуется и доступен в первой версии. Начальный режим по
умолчанию — FTS до первого сравнения качества на реальном корпусе.

Сразу реализуются:

- интерфейс поставщика эмбеддингов — числовых представлений смысла текста;
- таблицы версий модели, хэшей чанков и векторов;
- построение и полная переиндексация векторов;
- косинусное сходство — сравнение направления числовых векторов — для
  небольшого локального корпуса;
- гибридное объединение BM25 и векторной выдачи через Reciprocal Rank Fusion
  (RRF — слияние списков по их позициям, а не по несопоставимым баллам);
- команды состояния и проверки готовности;
- как минимум один реально работающий адаптер поставщика эмбеддингов;
- тесты на фиксированных векторах без сетевого вызова.

Конфигурация канона начинает с режима:

```yaml
search:
  mode: fts
  vector:
    available: true
    provider: null
    model: null
```

`available: true` означает, что механизм реализован. `mode: fts` означает, что
он не используется в обычном `kb query`. Команда гибридного поиска обязана
отказаться с понятной ошибкой, если модель не выбрана или индекс неполон.

С первой версии готовый индекс используется в сравнительном `kb eval`, а
`kb query --mode hybrid` доступен как явный диагностический режим. Поэтому
откладывается не фактическое использование векторов, а назначение `hybrid`
режимом обычного запроса по умолчанию.

Первичный выбор режима по умолчанию выполняется уже при приёмке первой версии,
а не откладывается на неопределённое будущее. `hybrid` выбирается, если на
версионированном проверочном наборе он имеет hit-rate top-3 (долю запросов с
ожидаемой страницей в первых трёх результатах) не ниже 0,8, превосходит FTS и
не теряет критические точные запросы. Если это условие не выполнено или
реальный векторный индекс не готов, по умолчанию остаётся `fts`, а явный
hybrid-режим сохраняется для диагностики. Сравнение повторяется после
существенного изменения корпуса, модели или ранжирования.

Выбор поставщика является обязательной границей Phase 3: без него можно
проверить механику на фиксированных векторах, но нельзя доказать качество
реального поиска. Внешний поставщик требует отдельного решения о
конфиденциальности и стоимости; внешний сервис не может получать код или
закрытую документацию по умолчанию.

### 7.3. Дисциплина чтения

1. `kb query`;
2. явные соседи найденных страниц;
3. чтение выбранных страниц и их первоисточников;
4. при недостатке — целевой поиск по репозиторию.

Если целевой поиск обнаружил существующую страницу вики, которую штатный
поиск не поднял после разумного переформулирования запроса, до завершения
задачи добавляется отдельный случай в `docs/knowledge/eval/cases/`.

## 8. Контур качества

### 8.1. Категории результатов

- **Ошибка** — канон не соответствует JSON Schema или содержит однозначно
  сломанную ссылку. Исправляется только правкой канона.
- **Предупреждение** — существует постоянная нерешённая проблема. Оно
  переиздаётся до явного канонического разрешения.
- **Рекомендация** — эвристика улучшения, например избыточный размер страницы.
  Она не считается скрытой незакрытой работой.

Постоянная очередь в SQLite не создаётся. `TSK-*` не заменяет разрешение
предупреждения: отсроченное противоречие хранит ссылку на задачу и срок прямо в
ребре `contradicts`.

### 8.2. Постоянные проверки

- схема страницы, конфигурации и поисковых случаев;
- уникальность `id`;
- существование `target` и нормализация симметричных рёбер;
- достижимость текущих страниц из индексов;
- дубли `title`/`description`, кроме пары с явным `distinct_from`;
- открытые и просроченные `contradicts`;
- пересечение равноправных зон `covers`;
- `source-drift`: источник или покрываемый путь изменился после `timestamp`;
- `desc-mismatch`: `.desc.json` новее содержательной сверки накрывающей страницы;
- совместимость конфигурации векторного индекса с сохранённой моделью;
- доступность FTS5 и целостность производной базы.

`rename-candidate` не входит в первую версию. Переименование страницы с
сохранением стабильного `id` безопасно; эвристика «удалено + создано похожее»
может появиться позднее как воспроизводимый анализ Git diff против явно
заданного base-коммита.

`prose-changed` является подсказкой при проверке текущего Git diff, а не
постоянным обещанием полного lint. Агент обязан пересмотреть рёбра изменённой
страницы до завершения задачи.

### 8.3. Разрешение предупреждений

- дубль: объединить через `supersedes` либо добавить `distinct_from` с
  обоснованием и провенансом;
- противоречие: `resolved` со ссылкой на решение или ограниченный `deferred`;
- `source-drift`/`desc-mismatch`: сверить страницу и обновить содержание и
  `timestamp`, либо пометить страницу `superseded`;
- сирота или битая ссылка: восстановить навигацию или удалить неверное ребро.

Общего механизма молчаливого подавления предупреждений нет.

## 9. Параллельная работа

Ветки и рабочие деревья наследуются от Git. Проекция локальна для каждого
рабочего дерева. После слияния в основную ветку выполняется полный
`kb sync && kb lint`, который проверяет объединённый канон.

Корректность не зависит от Git-трейлеров, автора коммита, squash/rebase или
наличия предыдущей SQLite-базы. Акторная модель и `KB-Actor` не используются.

Для уменьшения текстовых конфликтов:

- домены имеют отдельные индексные страницы;
- один поисковый случай хранится в одном JSON-файле;
- симметричное ребро записывается один раз в нормализованном направлении;
- общий индекс при конфликте сводит агент-интегратор.

## 10. Интеграция с картой проекта

Работающий `scripts/dev/generate-structure.js` остаётся владельцем генерации
корневого `PROJECT_STRUCTURE.yaml`. Система знаний не заменяет его и не вводит
команду `kb export structure`.

`kb sync` читает `PROJECT_STRUCTURE.yaml`, создаёт производные узлы дерева и
соединяет их со страницами через `covers`. Благодаря этому карта проекта
становится частью единого пути запроса и impact-анализа, но её источник истины
и формат не меняются.

`desc-mismatch` означает необходимость содержательной сверки, а не требование
дословного равенства: `.desc.json` описывает локальный узел кратко, страница
вики объясняет концепт глубоко.

## 11. Производный контекст кода и контрактов

AST (Abstract Syntax Tree, абстрактное синтаксическое дерево) — структурное
представление исходного кода после разбора синтаксиса. Система использует AST
как вход анализа, но не сохраняет его полную копию. В производную базу попадают
только факты, которые помогают агенту найти код, оценить влияние изменения и
выбрать проверки.

Wiki-граф отвечает на вопрос «почему и по какому правилу связаны компоненты».
Производный контекст отвечает на вопросы «где определён символ», «кто его
использует», «какой контракт приводит к этому обработчику» и «какие тесты
проверяют путь». Эти слои дополняют друг друга и не обмениваются рёбрами
автоматически.

### 11.1. Модульная AST-проекция

Первый слой полностью пересобирается из TypeScript и JavaScript уже
установленным TypeScript Compiler API и содержит:

- рабочие пакеты и исходные модули;
- рёбра `contains`, `imports`, `reexports` и `dynamic_imports_literal` для
  динамического импорта с литеральным строковым путём;
- прямые и обратные импорты в отдельной секции `kb impact`;
- единственный явный мост к wiki через `covers`.

Фактический `imports` не создаёт смысловой `depends_on`. Вычисляемые пути,
загрузка через реестры плагинов и иные динамические переходы не объявляются
доказанными зависимостями.

### 11.2. Обязательный слой символов и ссылок

После модульной проекции отдельным обязательным блоком внедряется индекс
программных символов. **Символ** — именованная сущность программы: функция,
метод, класс, интерфейс, псевдоним типа, перечисление или экспортируемое
значение. Слой хранит не тела функций и не полное дерево синтаксиса, а:

- определения именованных символов и содержащие их модули;
- статически разрешённые ссылки на символы;
- связи интерфейсов и абстракций с реализациями;
- связь символа с тестом, если тест действительно ссылается на него;
- исходное местоположение как проверяемое доказательство каждого факта.

Это даёт агенту точные команды поиска определения, ссылок, реализаций и
затронутых тестов. Идентификатор символа строится из контекста проекта
TypeScript, пакета, вида, квалифицированного имени и пути объявления. Номер строки
не входит в идентификатор и хранится только как доказательство, поэтому обычное
смещение кода не создаёт новый символ.

Cubica содержит несколько конфигураций TypeScript с разными правилами
разрешения модулей. Анализ выполняется отдельно для каждой `tsconfig`; факты
объединяются только при совпадении канонического файла, символа и результата
разрешения. Несогласованные результаты сохраняются раздельно и помечаются как
неоднозначные, а не сливаются молча.

### 11.3. Иерархия вызовов по запросу

**Иерархия вызовов** — список функций или методов, которые вызывают выбранный
символ либо вызываются им. Команды `kb callers` и `kb callees` вычисляют её для
конкретного символа через TypeScript Language Service (языковую службу
статической навигации) по текущему коду.

Общий постоянный граф всех вызовов не строится: он дорог, быстро устаревает и
даёт ложное ощущение полноты при колбэках, внедрении зависимостей, реестрах и
динамической диспетчеризации. Каждый результат содержит исходные позиции и
состояние `resolved`, `ambiguous`, `dynamic` или `unresolved`. Недоказанный
переход показывается как ограничение анализа, но не записывается как точное
ребро.

### 11.4. Происхождение контрактов

**Граф происхождения контрактов** показывает, как декларативный источник
истины доходит до исполняемого кода. Он особенно важен для Cubica, где JSON
Schema и OpenAPI определяют контракты, а реестры и манифесты могут скрывать
связь от обычного графа вызовов.

Проекция поддерживает узлы JSON Schema, OpenAPI, исходных манифестов,
сгенерированных типов, валидаторов, обработчиков и тестов, а также доказанные
рёбра:

- `generated_from` — артефакт создан из декларативного источника;
- `validated_by` — структура проверяется указанным валидатором;
- `handled_by` — операция или тип обрабатывается указанным символом;
- `tested_by` — контрактный путь проверяется тестом.

Ребро создаётся только из явного импорта, настройки генератора, регистрации,
ссылки в тесте или другого воспроизводимого доказательства. Отсутствующая связь
остаётся неизвестной; инструмент не угадывает её по похожести имён.

### 11.5. Целевой анализ потоков данных

**Анализ потока данных** отслеживает, как значение проходит от источника через
преобразования к месту использования. Он не входит в общую проекцию первой
реализации и внедряется отдельным блоком после составления списка конкретных
источников и приёмников данных на границах доверия.

Предпочтительный путь — целевые запросы CodeQL (статического анализатора кода),
а не собственный глобальный граф всех значений. Первыми кандидатами являются
путь недоверенных идентификаторов игрока и команд до изменения состояния,
проверка ожидаемой версии состояния и транзакции, а также путь секретов и
закрытого кода к публичному ответу или внешнему поставщику эмбеддингов.

Результат такого анализа является отчётом безопасности, а не wiki-связью. Для
каждого запроса фиксируются источники, приёмники, исключения и ожидаемые тестовые
примеры; универсальная полнота не обещается.

### 11.6. Инварианты расширения

- пространства идентификаторов `wiki:*`, `code:*` и `contract:*` разделены;
- каждый производный факт хранит `origin`, версию извлекателя, `tsconfig` при
  применимости, отпечаток исходного файла и местоположение доказательства;
- версия TypeScript Compiler API берётся из зафиксированной зависимости проекта
  и записывается в метаданные проекции;
- полная пересборка остаётся детерминированной, а инкрементальный режим в
  будущем обязан давать тот же логический результат;
- `kb impact` показывает wiki-, module-, symbol- и contract-факты раздельно;
- сырые AST-узлы, тела кода и производные кодовые факты не включаются в
  векторный индекс;
- производный анализ не изменяет Markdown, ADR, схемы, код и манифесты;
- существующая `EditorEntityProjection` остаётся отдельной предметной проекцией
  игровых authoring-сущностей и не поглощается контуром проектных знаний.

## 12. Техническая реализация

### 12.1. Стек и размещение

- CLI: `scripts/dev/kb.mjs`;
- язык: Node.js в поддерживаемой версии монорепозитория;
- YAML: существующий `js-yaml`;
- схемы: существующий Ajv;
- SQLite: явная dev-зависимость `better-sqlite3`;
- база: `.tmp/project-knowledge/kb.sqlite3`;
- кодировка канона: UTF-8, окончания строк LF.

При старте CLI выполняет capability probe — проверку возможностей SQLite:

```sql
SELECT sqlite_compileoption_used('ENABLE_FTS5');
CREATE VIRTUAL TABLE temp.kb_fts5_probe USING fts5(content);
DROP TABLE temp.kb_fts5_probe;
```

Отсутствие FTS5 завершает команду с понятной ошибкой. Беззвучный переход на
ухудшенный поиск запрещён.

### 12.2. Производные таблицы

- `pages`, `chunks`, `chunks_fts`;
- `wiki_edges`;
- `tree_nodes`, `tree_page_links`;
- `embedding_models`, `chunk_embeddings`;
- `derived_nodes`, `derived_edges` для модульных, символических и контрактных
  фактов с разделёнными пространствами идентификаторов;
- `projection_meta` с версией схемы и отпечатком входов.

Постоянных `queue`, `actor`, `attempts` и `dead_letter` нет.

### 12.3. Команды

```text
kb sync
kb query <text> [--mode fts|hybrid]
kb neighbors <id>
kb impact <id|path|symbol-id>
kb symbol <name>
kb references <symbol-id>
kb implementations <symbol-id>
kb callers <symbol-id>
kb callees <symbol-id>
kb provenance <path|symbol-id>
kb lint
kb eval
kb vector status
kb vector build
```

Команды не изменяют Markdown автоматически. `sync` строит новую базу рядом со
старой, проверяет её и атомарно заменяет последнюю валидную проекцию. Читатели
могут продолжать использовать предыдущую валидную базу во время построения.

## 13. Рабочий процесс агента

### 13.1. Перед изменением

1. Выполнить `kb query` по теме.
2. Просмотреть графовую окрестность и `kb impact` затрагиваемых путей или
   символов.
3. При изменении контракта проверить `kb provenance`; при изменении поведения —
   определения, ссылки, реализации и вызывающую сторону целевых символов.
4. Прочитать выбранные страницы и авторитетные источники.
5. При отсутствии достаточного ответа выполнить целевой поиск по репозиторию.

### 13.2. После изменения

1. Обновить существующую страницу или создать новый концепт только при наличии
   нового устойчивого знания.
2. Проверить `sources`, `covers`, рёбра и `timestamp`.
3. Выполнить `kb sync && kb lint`.
4. Запустить `kb eval`, если изменились поиск, чанкинг, ранжирование или
   проверочный набор.
5. Добавить поисковый случай, если штатный поиск пропустил релевантную страницу.

Система не требует отдельного коммита знаний и не навязывает merge-стратегию.

## 14. Критерии качества

- полный rebuild из чистой `.tmp/` создаёт валидную проекцию;
- повторный rebuild на неизменном каноне даёт тот же логический результат;
- все страницы проходят JSON Schema;
- все постоянные предупреждения имеют разрешение, выраженное правкой канона;
- выбранный режим по умолчанию имеет top-3 hit-rate на принятом наборе не ниже
  0,8; отчёт сохраняет отдельные показатели FTS и hybrid;
- векторный механизм строит индекс, участвует в сравнительном `kb eval` и
  доступен через явный `--mode hybrid`, но не влияет на обычную выдачу при
  `search.mode: fts`;
- `PROJECT_STRUCTURE.yaml` индексируется без изменения его генератора и формата;
- модульная и символическая проекции полностью перестраиваются из кода,
  сохраняют отдельное происхождение и показываются отдельно от wiki-связей;
- эталонные фикстуры с несколькими `tsconfig` доказывают поиск определения,
  ссылок, реализации и затронутого теста без молчаливого слияния неоднозначных
  символов;
- вычисляемая по запросу иерархия вызовов не создаёт рёбра без доказательства и
  явно показывает динамические и неразрешённые переходы;
- фикстура происхождения прослеживает путь JSON Schema или OpenAPI до
  сгенерированного типа, валидатора либо обработчика и теста;
- сырые AST-узлы и тела кода отсутствуют в проекции и векторном индексе;
- удаление SQLite не приводит к потере знаний или статусов разрешения;
- после слияния полный lint обнаруживает дубли и открытые противоречия всего
  объединённого корпуса.

## 15. Отложенные расширения

- повторный выбор режима по умолчанию после существенного изменения корпуса,
  модели или ранжирования;
- переход к инкрементальной индексации, если полный rebuild превышает
  согласованный бюджет времени;
- целевые запросы анализа потоков данных после утверждения списка источников и
  приёмников для конкретной границы безопасности;
- динамический анализ фактических вызовов, если статической иерархии окажется
  недостаточно для измеримого сценария;
- отдельный сервис или векторная БД только при росте корпуса, который SQLite
  перестанет обслуживать в установленном бюджете.

## 16. Отклонённые варианты

- **SQLite как второй канон:** отклонено; проекция должна быть удаляемой.
- **Постоянная очередь обслуживания:** отклонено; разрешение предупреждений
  хранится в самих страницах и рёбрах.
- **Git-трейлеры акторов:** отклонено как не дающее полезного различения для
  преимущественно агентной документации Cubica.
- **Переходное `contradiction`:** отклонено; открытый конфликт должен быть
  виден после любого полного rebuild.
- **Замена `generate-structure.js`:** отклонено в текущем объёме из-за нулевого
  функционального выигрыша и миграционного риска.
- **Векторный поиск только в будущем:** отклонено по решению PM; механизм,
  явный гибридный запрос и сравнительная оценка доступны сразу, откладывается
  только назначение гибридного режима значением по умолчанию.
- **AST как автоматически авторитетный граф:** отклонено; синтаксические факты
  не заменяют смысловые связи и архитектурные решения.
- **Полная сохранённая копия AST:** отклонена; она дублирует исходный код,
  увеличивает стоимость и не добавляет агенту навигации сверх отобранных
  модульных и символических фактов.
- **Постоянный универсальный граф вызовов:** отклонён в пользу вычисления по
  запросу с явной оценкой полноты.
- **Глобальный самописный граф потоков данных:** отклонён; для границ
  безопасности применяются ограниченные проверяемые запросы специализированного
  анализатора.
