# GSR-036: «Карты, деньги, поезда» — первый срез универсального языка механик

- **Дата:** 2026-07-14; реализация обновлена и проверена 2026-07-17
- **Статус:** Implemented and verified
- **Предусловие:** выполнены W0 trust boundary и W1 version/schema foundation:
  principal приходит из аутентификации, actor разрешается сервером, клиентский
  `playerId` не считается доверенным, session закрепляет `bundleHash`, а
  transaction использует точные module/artifact/algorithm locks и command
  receipt
- **Архитектура:** ADR-041, ADR-058, ADR-072, ADR-078, ADR-079, ADR-083, ADR-084
- **Связанные срезы:** GSR-032, GSR-035

Запись фиксирует первый реализованный ограниченный игровой сценарий Cubica
Mechanics IR. Он целиком переносит действие ведущего «Начать следующий ход», внутри которого
одновременно заменяет предметную активацию созревшего строительства и ошибочное
перечисление стартовых локомотивов при восстановлении ресурса.

**Cubica Mechanics IR** — канонический типизированный JSON-план, который runtime
исполняет атомарно. **Candidate state** — пробная копия состояния, которая
становится авторитетной только после успеха всего действия.

## Граница среза

Срез начинается перед системным действием начала хода `N+2` и заканчивается
после того, как:

1. сервер проверил фазу `debrief` и право ведущего выполнить действие;
2. номер хода увеличился ровно на один;
3. все фактически созревшие узлы и связи потеряли только причину блокировки
   `construction-pending` и открылись, если иных причин нет;
4. фаза, строительная доступность, методологическая пауза, текущая новость и
   предложение грузов сброшены в состояние начала нового хода;
5. все фактически активные локомотивы получили по 5 единиц действия;
6. купленный ранее `mock-market-locomotive-green-2` может выполнить движение;
7. запись о начале хода добавлена в журнал;
8. всё перечисленное зафиксировано одним commit либо не изменилось вовсе;
9. старые effects действия, включая
   `transport.construction.activateDue`, больше не участвуют в этом сценарии
   даже как runtime fallback.

Срез доказывает минимальную полную транзакцию
`assert/add/patch/select/update/event`, а не изолированные фрагменты
query/update. Первоначально он не включал полную миграцию остального каталога;
последующий общий переход выполнил и эту работу, не расширяя предметную границу
приёмки GSR-036.

## Исходное состояние

- текущий ход меньше либо равен сроку части объектов и больше либо равен сроку
  другой части;
- текущая фаза равна `debrief`, а состояние методологической паузы, текущей
  новости и предложения грузов требует сброса;
- коллекции nodes и edges содержат открытые, строящиеся и уже созревшие объекты;
- минимум один созревший объект имеет дополнительную независимую причину
  блокировки;
- коллекция `locomotives` содержит два стартовых active объекта;
- `mock-market-locomotive-green-2` куплен, принадлежит команде, переведён из
  `reserve` в `active` и присутствует в фактическом state;
- у локомотивов перед новым ходом разные значения `actionPoints`, включая ноль;
- состояние сессии и программа закреплены за неизменяемым hash.

## Полная транзакция следующего хода

Опубликованный `actionId = mock.debrief.next-turn` связывается с одним заранее
скомпилированным планом. План последовательно:

1. проверяет фазу и server-side authorization роли ведущего;
2. увеличивает `turnNumber` на один;
3. выполняет сценарий открытия созревших объектов;
4. типизированно устанавливает `phase = news`, закрывает доступность
   строительства и сбрасывает поля методологической паузы;
5. очищает текущую карту новостей и составное предложение грузов;
6. выполняет сценарий восстановления ресурса по фактической коллекции;
7. добавляет объявленную запись в журнал;
8. публикует изменения одной candidate-state транзакцией.

Ни один из этих шагов не остаётся старым effect внутри той же команды. Это
необходимо, чтобы срез доказал атомарность нового исполнителя: ошибка типа,
лимита или одной цели после увеличения счётчика обязана откатить также счётчик,
фазу, объекты и журнал.

## Сценарий открытия объектов

Authoring-макрос игры раскрывается в одинаковую bounded-последовательность для
nodes и edges:

1. выбрать объекты, у которых `activationTurn <= currentTurn` и набор причин
   блокировки содержит `construction-pending`;
2. удалить только это значение из набора причин;
3. на уже обновлённом candidate state выбрать затронутые объекты с пустым
   набором причин;
4. установить объявленный facet жизненного цикла в открытое состояние;
5. оставить объект закрытым, если после удаления строительной причины осталась
   иная причина.

Порядок объектов канонический и не зависит от порядка свойств JSON или базы
данных. Пустая выборка разрешена `cardinality.min = 0`; превышение capacity,
budget или ошибки типа откатывают действие целиком.

## Сценарий восстановления ресурса

Системное действие начала хода:

1. выбирает фактическую коллекцию `locomotives`;
2. оставляет объекты типа `transport.locomotive` с facet
   `availability = active`;
3. сортирует их по стабильному ID;
4. устанавливает typed attribute `actionPoints = 5` всем выбранным объектам.

В программе нет массива ID стартовой техники. Поэтому созданный или купленный
во время партии active-локомотив автоматически попадает в действие, а reserve,
проданный или заблокированный объект — нет.

## Общая платформенная возможность

Срез использует только нейтральные элементы принятого языка:

- state model коллекции с item type, visibility, capacity и stable key;
- чистые typed item/context/value expressions;
- `core.assert` для проверяемого предусловия;
- `core.number.add` и `core.state.patch` для скалярных и составных значений;
- `core.entities.select` с predicate, canonical order и cardinality;
- typed `EntitySet<T>` result и ссылка на предыдущий шаг;
- `core.entities.update` для facet/attribute/set changes без partial success;
- `core.event.emit` для типизированного события и проекции игрового журнала;
- структурный `sequence`, не являющийся предметной операцией;
- candidate-state transaction с одним commit или полным rollback;
- static cost и runtime scan/operator/memory/output budget;
- аудит числа выбранных и изменённых объектов без публикации закрытых значений.

Платформенный код не содержит ID игры, названий команд, локомотивов, дорог или
строительства.

## Содержимое только этой игры

- коллекции и object types транспортных объектов;
- имена `activationTurn`, `construction-pending`, `availability` и
  `actionPoints`;
- число 5;
- момент вызова макроса в начале хода `N+2`;
- значения `debrief`, `news`, правила сброса строительства, методологии,
  новостей и предложения грузов;
- тексты журнала и интерфейса.

Эти значения находятся в authoring-макросе и state model игры; они не создают
новые platform ops.

## Нейтральное доказательство

Отдельная фикстура без транспортных терминов содержит фазу, счётчик, составной
record, журнал, две bounded collections, несколько типов, срок, набор причин и
числовой ресурс. Один объект добавляется после старта сессии.

Она проверяет:

- выбор фактического, а не заранее перечисленного набора;
- bytewise canonical order;
- `cardinality` для пустого, допустимого и переполненного результата;
- отдельный scan-budget, даже если совпадений нет;
- сохранение независимой причины;
- типизированные add/set над скаляром и record;
- строгий порядок: assert до изменений, event после успешных изменений;
- откат всех targets при одной ошибке;
- откат state, random counters, events и session version;
- запрет secret/control-flow → public;
- одинаковый результат replay после PostgreSQL round-trip.

## Доказательства реализации

- `games/cards-money-trains-mock/authoring/mechanics.source.json` хранит
  воспроизводимый исходник Mechanics IR для генератора mock-пакета.
- Действие `mock.debrief.next-turn` связано с одноимённым планом через
  `binding.planRef`; план содержит assert фазы, увеличение хода, сбросы,
  выборки созревших nodes/edges, снятие только
  `construction-pending`, открытие объектов, выбор фактически активных
  локомотивов, массовое восстановление `actionPoints` и событие журнала.
- `mechanics-plan.schema.json`, реестр модулей, строгий валидатор и
  семантический checker проверяют структуру, ссылки, типы, locks и статический
  budget до публикации.
- Общий executor в `services/runtime-api/src/modules/mechanics/` выполняет план
  над candidate state и передаёт state, квитанцию, объявленные события и
  ограниченный защищённый аудит шагов/стоимости в одну транзакцию session
  store; публичная квитанция этот аудит не раскрывает.
- Старые production-обработчики deterministic effects, колод, рейтинга и
  предметного transport switch удалены; mock builder больше не создаёт старые
  effects.
- Контрактные фикстуры mock-пакета и нейтральный тест Mechanics executor
  переведены на новую форму. Сфокусированные проверки компилятора и блокировок,
  полный серверный сценарий, браузерные срезы и канонический межсистемный контур
  прошли; точные команды и результаты зафиксированы в активной задаче.

## Критерии приёмки

- полное действие следующего хода использует один опубликованный IR-план без
  предметного runtime handler и без смешивания со старыми effects;
- guard, увеличение хода, все сбросы, открытие объектов, восстановление ресурса
  и журнал входят в один atomic commit;
- купленный локомотив получает 5 единиц действия и проходит реальное действие
  движения;
- due nodes и edges открываются, но дополнительная причина блокировки не
  снимается;
- no-match является успешным только через объявленную cardinality, а не через
  отдельную скрытую политику;
- одна невалидная цель не оставляет частично изменённого состояния;
- neutral fixture доказывает типы, порядок, budgets, labels и replay;
- authoring source является источником истины; mock builder и compiler
  воспроизводимо пересобирают runtime manifest и карту связи generated-шагов с
  authoring-исходником;
- старое определение effects для `mock.debrief.next-turn`,
  `transport.construction.activateDue` и его handler branch удалено и не
  осталось production fallback;
- production dispatch получает principal из аутентификации, разрешает actor на
  сервере, отклоняет доверенный `playerId` из тела, сохраняет внешний
  `commandId` до квитанции и атомарно записывает state с внутренней command
  receipt и защищённым Mechanics-аудитом.

## Вне среза

Перечисленные ниже возможности не входили в критерии GSR-036. Последующая
полная миграция закрыла их отдельными пакетами и проверками, но это не
расширяет задним числом границу первого доказательного сценария. Список
сохранён как история декомпозиции:

- полная миграция Antarctica, Estate Race и остальных CMT effects;
- ranking/group/aggregate;
- road/waypoint planning, cargo и relation workflows;
- deck/random trusted disclosures;
- event scheduling;
- изолированные игровые runtime-расширения: этому срезу они не нужны, потому
  что всё правило выражается общими операциями;
- editor forms для всего IR;
- финальное удаление общего legacy executor до миграции всех пакетов.
