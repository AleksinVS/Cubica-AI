## 1. Назначение документа

Этот документ описывает, **как платформа Cubica использует Redis** — высокопроизводительное хранилище ключ–значение в памяти — для кэширования и распределённых блокировок.

Цели:
- Уточнить общую стратегию использования Redis.
- Согласовать схемы ключей для разных подсистем (Repository, сессии).
- Связать решения с ADR‑005 (Session Persistence) и фичей F_00062 (Redis Integration).

## 2. Области применения Redis

В архитектуре Cubica Redis используется в двух основных контекстах:

1. **Кэш манифестов игры (Game Repository):**
   - Описан в `docs/architecture/redis-keys.md`.
   - Используется для ускорения чтения манифестов и работы HTTP‑клиентов (ETag, версии каналов).
2. **Горячее хранилище и блокировки для игровых сессий:**
   - Описано в общих чертах в ADR‑005 и конкретизировано в этом документе.
   - Цель — снизить нагрузку на PostgreSQL и обеспечить распределённые блокировки между несколькими инстансами Router.

## 3. Паттерны ключей и кэширование

### 3.1. Репозиторий манифестов

Ключи для Repository фиксируются в `docs/architecture/redis-keys.md` и включают, например:

- `game:{game_id}:version:{version_id}:manifest` — неизменяемый JSON‑манифест.
- `game:{game_id}:channel:{name}:version` — указатель на актуальную версию манифеста.

Эти ключи **не зависят** от состояния сессий и могут кэшироваться с длительным TTL.

### 3.2. Игровые сессии (Session Cache)

Для сессий используется отдельное пространство ключей:

- `session:{session_id}` — сериализованное состояние сессии (эквивалент строки `game_sessions.state` + `history`).
  - TTL: соответствует типичному времени жизни *активной* сессии (например, 10–30 минут без активности).
  - При промахе по ключу Router читает состояние из PostgreSQL и восстанавливает запись.

Паттерн кэширования — **read-through**:
- Router сначала обращается к Redis; при промахе загружает состояние из БД и записывает его в Redis.
- При изменении состояния сначала фиксируется транзакция в Postgres, затем обновляется кэш.

## 4. Блокировки и согласованность

### 4.1. Ключи блокировок

Для распределённой блокировки сессий используются отдельные ключи:

- `lock:session:{session_id}` — логическая блокировка обработки хода для конкретной сессии.
  - Значение: случайный токен блокировки (строка UUID).
  - TTL: немного больше максимального времени обработки одного хода (см. ADR‑005, раздел Session Recovery).

Атомарная установка блокировки выполняется через команду `SET key value NX PX ttl_ms`. Снятие блокировки должно проверять, что токен совпадает (см. паттерн Redlock), чтобы избежать удаления чужой блокировки.

### 4.2. Согласованность с PostgreSQL

Основные правила согласованности:

- **Источник истины** — PostgreSQL (`game_sessions`). Redis рассматривается как ускоритель чтения и блокировок.
- Потеря данных в Redis (например, перезапуск сервера) не должна приводить к потере прогресса:
  - Router может всегда перечитать состояние из БД.
  - Блокировки при потере ключей автоматически восстанавливаются при новых запросах.
- В случае расхождений между Redis и БД приоритет имеет состояние, сохранённое в PostgreSQL.

Детали работы кэша и блокировок могут уточняться в рамках реализации фичи F_00062, но любое изменение должно сохранять инвариант: *PostgreSQL остаётся единственным источником истины для состояния сессий*.
