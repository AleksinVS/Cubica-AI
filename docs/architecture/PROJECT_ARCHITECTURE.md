# Архитектура платформы Cubica (MVP-стадия)

Документ описывает целевую и фактическую архитектуру платформы Cubica на уровне сервисов, SDK и данных. Он дополняет `PROJECT_OVERVIEW.md` (бизнес‑и логический обзор) и ADR‑решения в `docs/architecture/adrs/` и помогает быстро понять, как концепции отражены в структуре репозитория.

Для общего понимания архитектуры этот файл является обязательной краткой выжимкой ADR: сами ADR нужно читать только для дополнительного контекста, альтернатив или глубокого разбора решения.

> Термины:
> - **LLM** — Large Language Model, «большая языковая модель», используемая как игровой движок.
> - **SDK** — Software Development Kit, набор библиотек и утилит для интеграции с платформой.
> - **MVP‑паттерн** — Model–View–Presenter, шаблон разделения модели данных, представления и прослойки‑«презентера».
> - **ADR** — Architecture Decision Record, документ с зафиксированным архитектурным решением.
> - **CopilotKit** — React/Next.js-фреймворк для встраивания ИИ-помощников в приложения; в Cubica это текущий адаптер UI для Agent UI.
> - **AG-UI** — Agent-User Interaction protocol, событийный протокол между пользовательскими приложениями и backend-сервисами ИИ-агентов.
> - **Generative UI** — подход, при котором ИИ-агент выбирает, описывает или обновляет часть интерфейса во время работы.
> - **Элементный промт** — сохраненный authoring-промт конкретного элемента манифеста: описание намерения, содержимого и поведения экземпляра, из которого агент может строить или проверять структурированные поля.
> - **A2UI** — декларативная JSONL-спецификация UI-поверхностей, которую Cubica рассматривает как внешний compatibility reference, а не как источник истины.
> - **Cubica Surface Protocol** — собственный протокол Cubica для описания ограниченных UI-поверхностей помощника: словаря компонентов, модели данных, действий и проверок.
> - **AI-driven game** — игра, где ИИ-агент является обязательной частью runtime и управляет ходом, состоянием шага и UI-поверхностью через валидируемые контракты Cubica.
> - **Agent Runtime** — backend-граница, которая выполняет шаг агента, вызывает модель или локального агента и возвращает структурированный результат для runtime validation.
> - **Продуктовая система знаний** — предлагаемый многопользовательский контур, который хранит подтверждённый синтез в изолированных пакетах Markdown и собирает для агента только разрешённые актуальные сведения о пользователе, игровом проекте или мероприятии; это не инженерная проектная вики и не источник игрового состояния.
> - **Game-level prototype** — authoring-прототип, принадлежащий конкретной игре и допускающий локальные предметные детали.
> - **Platform-level prototype** — authoring-прототип платформы Cubica, предназначенный для класса игр или UI-паттернов и не содержащий game-specific деталей.
> - **API First** — подход, при котором внешний контракт API фиксируется и проверяется до или вместе с реализацией endpoint'а.
> - **OpenAPI** — машинно-читаемая спецификация HTTP API: пути, операции, параметры, тела запросов, ответы, ошибки и переиспользуемые схемы.
> - **Map-first workspace** — декларативный режим web-интерфейса, в котором карта или другое большое пространственное поле заполняет рабочую область, а статус, действия и панели располагаются управляемыми слоями поверх него.
> - **Game Intent (игровое намерение)** — стабильная изменяющая команда в терминах конкретной игры; клиент передаёт существующий `actionId` и проверяемые параметры, а сервер связывает их с опубликованным правилом.
> - **Cubica Mechanics IR** — внутреннее типизированное промежуточное представление игровой механики, которое исполняется атомарно и не отправляется обычным клиентом.
> - **Mechanics capability pack** — пакет времени сборки со схемами, макросами, шаблонами действий и правилами преобразования для общей области механик; он не является исполняемым расширением или клиентским UI-пакетом.
> - **Идемпотентность команды** — безопасный повтор одной логической команды с тем же `commandId` без повторного применения уже принятого изменения; внешний клиент использует случайный профиль `cli_`, планировщик — отдельный детерминированный профиль `sys_`, а `requestId` и идентификаторы трассировки относятся только к доставке и наблюдаемости.

## Текущий канонический срез

Текущий канонический срез уже реализован и должен читаться первым:

- `games/antarctica/` - canonical content bundle.
- `games/antarctica/game.manifest.json` - source of truth для исполнимой логики игры.
- `games/simple-choice/` - minimal second game fixture that proves the game-agnostic runtime/player path without a custom web plugin.
- `games/ai-driven-choice/` - minimal AI-driven fixture that proves manifest-declared Agent Runtime readiness, выбор агентом только опубликованного и доступного Game Intent, его исполнение общей Mechanics IR-транзакцией, Web `CubicaSurface` rendering и replay/eval/audit contract gates без runtime-веток по конкретной игре.
- `games/antarctica/design/mockups/` - source of truth для UI intent.
- `games/antarctica/game.manifest.json` уже покрывает bounded gameplay slice records до terminal `i21`; архитектурное правило для этих механик закреплено в ADR-024, а step-specific delivery details вынесены в `docs/architecture/gameplay-slices/`.
- `draft/Antarctica/GameFull.html` - текущий factual extraction source для Antarctica mechanics migration; это состояние миграции, а не новое архитектурное решение, и это не canonical runtime source of truth.
- `services/runtime-api/` - канонический backend runtime в формате модульного монолита и owner загрузки игрового контента для runtime/player delivery (ADR-019).
- ADR-051 закрепляет API First для текущего монолита: current `runtime-api` имеет собственный OpenAPI-контракт `docs/architecture/runtime-api-openapi.yaml`, а старые Router/Engine/Repository specs остаются future extraction references до физического выделения сервисов.
- `apps/player-web/` - канонический web delivery layer, который потребляет player-facing content API/DTO и рендерит игры из session snapshot + manifest content projection, а не из repo files напрямую (ADR-019). Для простых игр используется default config builder из `PlayerFacingContent.ui`; для сложных игр сохраняется plugin layer; для AI-driven игр `player-web` проверяет game readiness, показывает pause/retry/unavailable state при недоступном Agent Runtime и рендерит validated `CubicaSurface` из Agent Turn. ADR-053 уточняет UI panels: player-web owns generic panel lifecycle and manifest rendering, but game-specific panel UI such as Antarctica move history must be declared in the game UI manifest, not in platform React components. В editor preview mode player-web может отдавать generic runtime pointer metadata через `postMessage`, не импортирует `editor-engine` и не хранит authoring/editor state. ADR-037 меняет целевой plugin home: пользовательские плагины должны жить в `games/<gameId>/plugins/<pluginId>/`; Antarctica уже перенесена в `games/antarctica/plugins/antarctica-player`. Local preview loads session-scoped plugin bundles through `PlayerFacingContent.pluginBundles`; non-preview mode loads only published bundle references generated under `games/<gameId>/published/`.
- ADR-054 фиксирует границу game/UI manifests: game manifest owns game meaning, UI manifest owns channel presentation, а Presenter owns player-facing projection. ADR-084 уточняет запись: Presenter маршрутизирует опубликованное `actionId + params + commandId`; он не вычисляет правило, не компилирует Mechanics IR и не заявляет доверенный `playerId` из клиентского тела. В `Antarctica` `time` означает прошедшие игровые дни, `remainingDays` является вычисляемой метрикой "осталось дней", а `score` не должен оставаться скрытым alias for remaining days.
- ADR-079 фиксирует серверную проекцию доступности действий. ADR-084 разделяет целевой транспорт на неизменяемые action definitions по `bundleHash`, динамическую principal/actor/session/state-scoped availability с `basisStateVersion` и server-only authorization. Наличие, порядок, варианты и preview также считаются информационным потоком. Клиенты блокируют недоступные действия и сохраняют `commandId`, но runtime всегда повторно проверяет новую команду; внутренний IR, пути и закрытые значения клиенту не раскрываются.
- ADR-080 фиксирует общий декларативный режим `map-first`: пространственное поле заполняет рабочую область, а статус, боковые панели, нижние действия, плавающие органы камеры и модальные слои объявляются через ограниченные смысловые зоны UI-манифеста. Режим включается явно, не меняет старые компоновки, не допускает игровых CSS-сигналов или произвольных уровней слоёв и сохраняет независимый от Phaser доступный путь критических действий. Контракт не навязывает формат визуальной подложки, но требует отделять её жизненный цикл от канонической геометрии правил. Логическая координатная плоскость карты может быть больше окна браузера, тогда как общий клиент независимо ограничивает физический canvas и показывает плоскость через камеру.
- Accepted ADR-081 добавляет явно включаемую стратегию транспортных сетей: runtime строит дорогу по версионированному навигационному графу опубликованных областей, минимизирует число непрерывных региональных сегментов и сохраняет ломаную, сегменты, стоимость и версию алгоритма. Политика разрешения равных вариантов входит в версию планировщика; первая версия использует воспроизводимый `session-random`. Поддерживаются простые неперекрывающиеся полигоны без отверстий и точные общие границы положительной длины; неподдерживаемая или непроверенная геометрия отклоняется без автоматической починки. Клиент передаёт только концы и параметры объявленного действия, но не доверенную линию, число сегментов или цену.
- Accepted ADR-082 вводит проектную файловую вики Cubica как канон смыслового синтеза поверх действующих ADR, JSON Schema, кода, задач и карты репозитория. Markdown проверяется декларативными схемами; SQLite, явный граф, FTS5 и векторы являются полностью пересобираемыми локальными проекциями. Векторная возможность реализуется сразу, используется в явных hybrid-запросах и сравнительном `kb eval`; режим по умолчанию выбирается при приёмке первой версии по измеримому качеству, с FTS как безопасным начальным и резервным вариантом. Производный контекст кода включает модули и импорты, обязательные определения символов, ссылки, реализации, затронутые тесты и происхождение контрактов; иерархия вызовов (список вызывающих и вызываемых функций) вычисляется по запросу. Эти факты отделены от смыслового wiki-графа и связываются с ним только через `covers`; полная копия AST (абстрактного синтаксического дерева кода), постоянный универсальный граф вызовов и общий граф движения значений по коду не создаются. Открытые противоречия и их разрешения живут в каноне; постоянная очередь, акторные Git-трейлеры и обязательные переходные детекторы не используются. Существующий генератор `PROJECT_STRUCTURE.yaml` сохраняется, а система знаний только индексирует его результат и связывает пути через `covers`.
- Accepted ADR-083 меняет политику развития игрового движка: один реальный игровой сценарий может обосновать общий примитив без ожидания второго потребителя. Публичный язык механик использует нейтральные селекторы, bounded control nodes, вычисления, планирование и атомарные массовые изменения, а не предметные effects. Не выразимая общими механизмами уникальная функциональность остаётся в изолированном game-scoped extension, а не попадает веткой в core.
- Accepted ADR-084 задаёт два уровня единого контракта. Клиент и игровой ИИ видят actor-scoped Game Intents и отправляют `actionId + params + commandId`; `actionId` сохраняется без нового `intentId`, principal и actor разделены, авторитетная роль берётся только из аутентифицированного principal, а клиентский `playerId` не является источником доверенной личности. До канонизации текущий конверт ограничен 16 плоскими скалярными параметрами. Presenter использует опубликованные bindings, но не владеет правилами. Входной Intent Agent Turn проходит общий допуск и начальные предусловия до обращения к модели. Authoring-агент и редактор обнаруживают широкий каталог через mechanics packs; platform modules, изолированные game extensions и UI packs остаются отдельными границами. Mechanics IR состоит из чистых query/assert/algorithm, изменяющих command и structural control nodes, проверяется JSON Schema 2020-12 и semantic checker и исполняется атомарно. Внешний `commandId` создаётся случайно в профиле `cli_`; системное срабатывание получает детерминированный `sys_` из schedule/occurrence, не принимает клиентскую version precondition и повторно проверяет trigger на текущем состоянии под session lock. Префикс не даёт полномочий. Command receipt lookup предшествует version check; точный повтор проверяет целостность закреплённого bundle, но не требует активного исторического исполнителя, тогда как новая команда по недоступной версии безопасно останавливается. State/RNG, защищённый ограниченный аудит шагов и стоимости, внутренняя receipt и полные серверные события фиксируются одним commit; публичная receipt аудит не раскрывает. Полные события получают монотонные номера в защищённом журнале сессии; объявленный `journalEndpoint` хранит только предназначенную игре публичную проекцию и не заменяет этот журнал. Editor preview может восстановить старое игровое состояние, но создаёт новую версию и не откатывает последовательность защищённых событий. `moduleLock` раздельно закрепляет identity, точную версию, hash содержимого и версии алгоритмов; сессия закрепляет весь `bundleHash`. `planHash` также включает `stateModel`, `objectModels` и `networkModels`, поэтому смена интерпретируемой графовой или инвентарной модели меняет идентичность плана. Раскрытие `planHash` для учебных данных не считается критичным и не блокирует выпуск. Старые effects, executor/templates, client command maps, direct AI effects, доверенный клиентский `playerId`, `params/payload` duality и постоянные adapters удалены при полном переключении.
- Accepted ADR-085 завершает три недостающих нейтральных блока Mechanics. Колода имеет взаимоисключающие скрытые зоны `order`, `discard` и `held`; `deck.extract`, `deck.return` и `deck.insert` изымают, возвращают и включают объекты, не меняя прежнюю семантику `deck.draw`. Транзакционная `core.entities.order` в отдельном нейтральном модуле строит лексикографический порядок по последовательным типизированным ключам текущего или связанного объекта и ограниченным агрегатам; случайность разрешает только полное равенство через именованный поток и поэтому делает операцию `command`, даже когда конкретный вызов использует каноническое равенство без изменения генератора. `graph.edge.position.inspect` выдаёт серверное доказательство координаты, областей и отпечатка геометрии, `predicate.set.disjoint` сравнивает ограниченные множества, а `graph.edge.split` изменяет только проверенную версию ребра. Точка на границе принадлежит всем касающимся областям. Все операции schema-first, ограничены по стоимости и доказываются нейтральными фикстурами; предметные правила остаются в игровых планах.
- Accepted ADR-086 превращает точную блокировку Mechanics из проверки несовпадения в гарантию продолжения поддерживаемых сессий внутри действующей архитектуры Game Intent → Mechanics IR; решение не замораживает развитие платформы и не восстанавливает удалённый `deterministic.effects` или иной заменённый runtime. Загрузка сначала проверяет только стабильный ограниченный конверт и точные байты, затем по `moduleId + moduleVersion + artifactHash` выбирает согласованный набор полной JSON Schema, semantic checker и executor; проверка исторического пакета текущей полной схемой до выбора версии запрещена. Значимые общие зависимости имеют явную версию, а несвязанные модули не получают общий ложный хеш. Перед недиспозируемой партией runtime проверяет доступность полного закреплённого набора. Старые предрелизные тестовые сессии без доступного исполнителя архивируются без перезаписи через явную одинаковую для PostgreSQL и in-memory границу архивирования и авторизованного чтения; автоматическая миграция живого состояния запрещена.
- Accepted ADR-087 добавляет ограниченный `finite-number` для точных конечных binary64-значений без десятичного округления и вычисляемое поле коллекции только для чтения. Такое поле извлекает значение по пути из одного непосредственно сохранённого поля той же коллекции, не имеет отдельного storage, не допускает цепочек, записи, JSONPath или индексов массивов и наследует аудиторию коллекции. Record-пути проверяются статически, bounded JSON уточняется на входной и итоговой границе состояния. Единый логический reader используется выражениями, выборками и упорядочиванием, а стоимость пути входит в бюджет. Упорядочивание допускает точные сравнения, `min` и `max`, но запрещает неассоциативный `sum` по binary64; геометрические координаты результата типизированы как `finite-number`.
- Accepted ADR-088 добавляет `core.entities.each`: ограниченный атомарный обход результата предшествующего `core.entities.select` либо его проверенной производной `core.entities.order`. Производная сохраняет исходную коллекцию и статическую границу выборки, а сам обход повторно устанавливает канонический порядок идентификаторов. Тело содержит не более шестнадцати существующих шагов, вложенный `each` запрещён, а максимальная стоимость заранее умножается на объявленную границу выборки. `value.item` получает безопасное чтение идентификатора текущего объекта без дублирующего игрового `selfId`; результаты тела локальны одной итерации и не протекают между итерациями или наружу. Ошибка любой итерации отменяет всю транзакцию. Защищённый аудит сохраняет ограниченный итог `each` и безопасные записи реально выполненных шагов тела с областью итерации; оба объёма входят в бюджеты. Операция не знает о сцеплении, деньгах или других предметных правилах; списание общего ресурса остаётся отдельным шагом игрового плана.
- Accepted ADR-089 расширяет общий Mechanics двумя ограниченными возможностями. Вариант `set-add` у `core.entity.attributes.patch` идемпотентно добавляет один типизированный элемент непосредственно в записываемый атрибут-множество, учитывает просмотр до `maxItems` и не заменяет независимые элементы. `deck.shuffle` с буквальным `deckId` остаётся декларацией колоды, а `deck.draw`, `deck.extract`, `deck.return` и `deck.insert` могут получить `deckId` из строкового `value.param`, если `const`/`enum` связанного Game Intent целиком состоит из колод, объявленных в том же неизменяемом пакете. Неограниченные строки и неизвестные колоды отклоняются до публикации; runtime повторно проверяет безопасный идентификатор и существование закрытой колоды. Оба изменения schema-first, атомарны, ограничены по стоимости, получают новые точные версии модулей и нейтральные доказательства. Поддерживаемая сессия остаётся на точном прежнем исполнителе; пока активен `LEGACY-0072`, это не обещается предрелизным тестовым партиям — они архивируются без изменения, а проверка продолжается в новой сессии. Недиспозируемую партию до сохранения замороженного корпуса начинать нельзя.
- Accepted ADR-090 добавляет общий доступный вариант `kind: "text"` в проекцию полей действия интерактивного поля. Host отображает управляемый стандартный текстовый ввод с подписью, обязательностью, начальным значением и необязательными подсказками длины/шаблона, не изменяет строку и отправляет её через обычный Game Intent. Клиентские ограничения служат только ранней обратной связью; доверенным источником допустимости остаётся серверная `paramsSchema`. Поле живёт в DOM-пути, поэтому остаётся доступным с клавиатуры и при отказе canvas, а общий renderer не получает правил конкретной игры и не превращается этим решением в полный генератор форм из JSON Schema.
- Draft ADR-095 предлагает отделить продуктовые знания пользовательских помощников от принятой инженерной вики. Полные диалоги и процесс изменения каноничны в PostgreSQL, а подтверждённый синтез — в текущей ветке закрытого управляемого Git-репозитория Markdown. Вместо четырёх специальных пакетов используется одна универсальная область с владельцем, доверенной зоной, доступом и сроком; пользователь видит динамическую композицию личной и текущей предметной области, а опубликованная игра читается прямо из неизменяемого пакета. Текущий разговорный агент готовит точную операцию над одним фрагментом, но сервер назначает область и проверяет подтверждение; детерминированный исполнитель сохраняет остальные байты, генерирует `index.md` и атомарно переключает Git ref. Ответ агента допустим как источник формулировки совместно созданного знания только вместе с более поздним подтверждением пользователя. Обычный контекст читает только текущий снимок; Git-история доступна отдельно, логическое забывание отделено от физического стирания истории и резервных копий. OKF остаётся целью совместимости, а не блокером расширений Cubica. FTS добавляется по измеримому сигналу, векторы — после доказанного недостатка FTS. Решение остаётся черновиком до согласования Git-границы и удаления, сроков данных, автоматического применения, переноса между ролями, внешней обработки и единой платформенной личности.
- `packages/contracts/*` - общий contracts layer. `packages/contracts/ai` owns framework-neutral AI contracts: Cubica Surface, Agent Turn, A2UI-like adapter mapping, channel projections for Telegram/Phaser adapters, plugin Surface component contribution metadata, operation policy and replay/eval/audit fixtures.
- `packages/editor-engine/` - framework-agnostic core первого полного authoring editor slice: DocumentStore, JSON Pointer/JSON Patch helpers, text location map, schema registry, graph/tree projections, `TreeViewModelBuilder`, semantic entity tree projection, renderer-neutral preview descriptors, manifest chronology timeline, preview playthrough trace model, diagnostics и reverse projection (ADR-034/ADR-036). Phase 1A добавила authoring v2 schemas, `_label` как поле "Синоним" and minimal compiler fixtures; Phase 1B migrated current authoring manifests to v2; preview hit-test/highlight contracts stay renderer-neutral.
- `apps/editor-web/` - текущий Next.js authoring editor surface: preview-first shell with central preview workspace, iframe preview message bridge, preview selection overlay/object picker/region prompt baseline, collapsible manifest entity tree/graph rail, advanced JSON tree toggle, Monaco JSON editor rail, floating property panel, session-backed authoring file workflow, layout persistence и validate/compile/preview actions; это editor surface, а не runtime/player delivery layer. Web preview поддерживает desktop/tablet/mobile и landscape/portrait; структурный Telegram viewer читает channel-specific UI authoring manifest, а не AI Surface, и даёт message/action/unknown-component projection. Telegram DOM подключён к общему renderer adapter: source pointer selection, локальные bounds, point/rect hit-test, highlight и переход из channel diagnostics работают тем же контуром, что и Web; небезопасный quick-fix для неизвестной схемной цели намеренно скрыт. Phase 8 baseline added automatic active-file JSON ChangeSet apply from preview AI prompts, dry-run validation, undo/redo journal and diff summary. The MVP Agent UI keeps CopilotKit as shell, while editor helper `CubicaSurface` renders tool progress, diagnostics, diff summary and approved editor-tool actions from framework-neutral data. Phase 9 now opens Git worktree sessions for file/layout reads and Save commits; validate/compile run from the session worktree when `sessionId` is present, local preview registers an allowlisted session worktree as runtime-api/player-web `contentSourceId`, and Playwright covers the three-service session preview path. ADR-042 baseline is implemented locally: editor sessions carry platform/plugin API version metadata, reuse compatible active sessions by default, expose upgrade dry-run diagnostics, and provide garbage-collected lifecycle management for expired/orphaned worktrees. Save также продвигает разрешённое authoring-содержимое в долговечную проектную линию ADR-075; новая сессия накладывает его на актуальную версию платформы, а возврат создаёт новую версию без переписывания истории. Пользовательский Save требует сессию, Save/Restore/Close/GC сериализованы межпроцессной арендой, а GC повторно проверяет Git status и безопасный worktree root согласно ADR-077. JSON Tree mode реализован в `TSK-20260522-editor-engine-json-tree-view`; structural tree operations пока отложены.
- ADR-057 и `docs/architecture/editor-preview-first-ux.md` - accepted UX-архитектура preview-first редактора: единая игровая сущность как проекция фасетов (смысл/содержание/вид) поверх `EditorEntityProjection`, текстовое редактирование через промт-проекцию с интерпретатором возвращённого намерения и шкалой риска, дерево с переключаемой группировкой «По экранам/По типам», разделение осей времени, фикстуры состояния как authoring-артефакт и трёхуровневое кэширование редактора. Реализовано в `TSK-20260704-editor-preview-first-ux-implementation` (Phases 0–9 core, 2026-07-07): проектная game+ui проекция, дерево с occurrences и панель сущности по эталонному макету, интерпретатор возвращённого намерения и текстовый режим, оси времени с лестницей восстановления, фикстуры состояния (`games/<id>/authoring/fixtures/`), атомарные кросс-манифестные операции создания/удаления/рефакторинга через мультидок-конвейер с approval envelope, очередь интентов, диагностический поток с last-valid снимком, регион-снимок как optional adapter capability и секция ассетов; кэш L1/L2/L3 поверх алгофикса компилятора. Концепты §9.7–9.8 разделены на `TSK-20260710-editor-author-version-history` (долговечная история ADR-075) и `TSK-20260707-editor-concept-surfaces` (Telegram/viewport).
- `docs/architecture/agent-ui-foundation.md` и ADR-043 - принятая и baseline-реализованная основа пользовательских ИИ-помощников. CopilotKit владеет React/Next.js UI помощника, AG-UI владеет внешним событийным протоколом, а контракты Cubica продолжают владеть манифестами, сессиями, launch-данными и валидацией. Первый кодовый срез находится в `apps/editor-web`: выключенный по умолчанию provider, app-local `/api/copilotkit`, реестр помощников, ограниченная проекция контекста, frontend tools поверх `EditorChangeSet` и нормализация событий AG-UI.
- `docs/architecture/agent-ui-portability-and-risk-controls.md` и ADR-044 - слой переносимости Agent UI: CopilotKit и AG-UI считаются заменяемыми адаптерами, production LLM backend не получает прямую запись в состояние Cubica, а новые помощники проходят через собственные Cubica-контракты, контроль рисков и планы миграции.
- `docs/architecture/generative-ui-surface-protocol.md` и ADR-045 - целевая архитектура Cubica-owned Generative UI: CopilotKit является MVP-адаптером первого этапа, будущий самописный совместимый Cubica Agent UI потребляет те же Cubica-контракты, а декларативные поверхности UI описываются как Cubica Surface specs вместо сохранения CopilotKit/AG-UI/A2UI state в предметных данных.
- `docs/architecture/ai-agent-safety-remediation.md` и ADR-047 - accepted remediation layer после review миграции: approval envelope, rejected Agent Turn semantics, manifest capability gates, channel action policies and production backend auth readiness.
- `docs/architecture/element-prompt-contract.md` и ADR-048 - accepted контракт элементного промта: первый срез реализовал `_prompt` для authoring-экземпляров, `_promptTemplate` для прототипов, compiler stripping и editor schema readiness; `generation.prompt` остается только metadata визуальной генерации для design artifacts. Направление синхронизации уже принято ADR-049; YAML-проекция, словарь полей и `prompt-stale` baseline реализованы, а миграция persisted `_prompt` к static residue/Markdown refs остаётся зарегистрированным implementation gap.
- ADR-046 - целевой AI-driven game runtime mode: манифест может объявить `ai-driven` или `hybrid` execution mode, Agent Runtime становится обязательной runtime-зависимостью этой игры, а агент возвращает валидируемые narration/Surface и выбирает разрешённый Game Intent. Прямые agent-produced state effects удалены; действующий контракт мутации проходит только через Game Intent → Mechanics IR.
- `docs/architecture/runtime-mechanics-language.md` - проектное описание единой цепочки Game Intent → Cubica Mechanics IR: action definitions/availability, mechanics packs, изолированные игровые extensions, типизированные транзакции, безопасность, budgets, command receipts и полное удаление старого effects-пути.
- `draft/cubica-portal-nextjs/` - current portal draft for test launch analysis; портал должен стать launch surface для покупок, ссылок и игровых сессий, но не runtime source of truth (ADR-032).
- `draft/*` и импортированные portal/player drafts - reference only, а не canonical runtime/architecture sources.

## 1. Контекст и цели

### Бизнес‑контекст
- Платформа Cubica предназначена для создания и проведения обучающих и бизнес‑игр.
- Игры должны поддерживать разные клиентские каналы (браузер, мобильные клиенты, мессенджеры и др.).
- Основная ценность — возможность быстро разрабатывать сценарии (игровые «манифесты») и переиспользовать инфраструктуру запуска игр.

### Техническая стратегия
- Исторический target дизайна платформы опирался на LLM‑first подход; текущий canonical slice использует deterministic runtime и JSON‑manifest source of truth, а ADR-046 закрепляет AI-driven runtime как целевой first-class mode для игр, которые явно требуют Agent Runtime.
- LLM остаётся future/current capability layer в зависимости от execution mode: для deterministic games это authoring/assistance layer, для AI-driven games это обязательная runtime dependency через Agent Runtime.
- Пользовательские ИИ-помощники используют CopilotKit/AG-UI как MVP-адаптер Agent UI, но не владеют каноническим состоянием игры, редактора или платформы; целевой слой по ADR-045 - собственный совместимый Cubica Agent UI and Cubica Surface Protocol.
- Review findings по AI agent safety закрыты remediation slice по ADR-047 for MVP provider-safety: approval is not model input, rejected turns do not persist state changes, capabilities are executable gates, and channels fail closed for unsupported Surface actions. ADR-084 дополнительно требует, чтобы целевая долговременная мутация Agent Turn сходилась в Game Intent → Mechanics IR.
- Формат данных: **JSON‑манифесты игр** и версионируемый каталог контента (`data/fixtures/`, `games/`, схемы в `docs/architecture/schemas/`).
- Архитектурные принципы MVP и формат манифестов формализованы в ADR‑001 (`docs/architecture/adrs/001-mvp-and-llm-first-game-manifests.md`).
- Протоколы взаимодействия слоев зафиксированы в ADR-002 (`docs/architecture/adrs/002-abstract-view-protocol.md`) и описывают взаимодействие Presenter и View через абстрактный шлюз (Command Pattern + Promises).

## 2. Логическая архитектура платформы

### 2.1. Backend-сервисы

Target backend layer still describes a set of independent services (each in `services/`):

API First applies during the modular-monolith phase. The implemented HTTP API
of `services/runtime-api` is documented in
`docs/architecture/runtime-api-openapi.yaml`; future Router, Game Engine and
Game Repository OpenAPI files describe extraction targets, not current
production coverage, until those services exist as deployable boundaries.

- **Game Editor** (`services/game-editor/`) — редактор игр и сценариев.
  - Позволяет редактировать сценарии и манифесты (JSON) и публиковать их в репозиторий.
  - Первый полный `editor-engine` slice реализован вне service folder: `packages/editor-engine` содержит framework-agnostic core, а `apps/editor-web` содержит Next.js authoring editor surface.
  - Архитектурное правило ADR-034 сохраняется: flow-chart и JSON tree view являются проекциями authoring JSON, Monaco/JSON editor и property panel работают через единый DocumentStore, а visual edits возвращаются в authoring JSON через JSON Patch или в отдельный layout target для editor-only операций.
- **Game Repository** (`services/game-repository/`) — авторитетное хранилище игровых манифестов и ассетов.
  - На ранних этапах использует файловое хранилище (`data/fixtures/`), в будущем — полноценный сервис с API по спецификации `docs/architecture/repository-openapi.yaml`.
- **Game Catalog** (`services/game-catalog/`) — каталог опубликованных игр.
  - Обеспечивает поиск и фильтрацию игр, интегрируется с системой индексации (см. `docs/architecture/search/qdrant.md`).
- **Router** (`services/router/`) — фронтовой API‑шлюз (маршрутизатор игровых действий).
  - Принимает запросы от клиентов, управляет сессиями и пересылает действия в Game Engine, а также читает данные из Game Repository.
  - **Мультиплеер (ADR-011):** Поддерживает несколько игроков в одной сессии через Event Queue и бродкастинг обновлений.
- **Game Engine** (`services/game-engine/`) — слой интеграции с LLM и выполнения скриптов.
  - **Историческая гибридная композиция (ADR-015):**
    - **Engine Extensions (Build-time):** компилируются с ядром, отвечают за системные возможности (БД, физика).
    - **User Scripts (Runtime):** были ранней идеей для динамической контентной логики.
  - **Текущая граница (ADR-040):** новая серверная механика сначала выражается через манифест, JSON Schema и общие platform capabilities. `isolated-vm`, `node:vm` и `worker_threads` не считаются защитой для чужого кода; доверенные runtime-плагины требуют отдельного процесса с JSON-протоколом и отдельного ревью.
  - Возвращает структурированные дельты состояния в Router.
- **Metadata Database** (`services/metadata-db/`) — аналитический слой.
  - Сводит события из Router, Engine и Repository для аналитики и отчетности (детально будет развиваться во Фазе 2+).

Outside the current canonical `runtime-api` slice, most service folders remain scaffolds with `DEV_GUIDE.md` and mostly empty `src/`/`tests/`. Это target architecture scaffolding, а не текущая runtime reality.

### 2.2. Клиентские компоненты и стратегия клиентского ядра

Клиентская часть построена вокруг игро-агностичного плеера `apps/player-web`
(декларативный рендерер UI-манифестов по ADR-055 + проектные плагины по
ADR-037/039), общих контрактов и стратегии «headless core + адаптеры каналов»
(ADR-064). Headless core — ядро без привязки к UI-фреймворку; адаптер канала —
тонкий слой, связывающий ядро с конкретным способом доставки (React/DOM,
Phaser, будущие каналы).

**Опорные слои клиентской части:**
- `packages/contracts/*` — генерируемые из JSON Schema контракты (ADR-056);
  вместе с OpenAPI `runtime-api` (ADR-051) это универсальная интеграционная
  поверхность платформы для любого канала на любом стеке.
- `packages/view-protocol/` — Abstract View Protocol (ADR-002: команды
  Presenter → View) и утилиты JSON Merge Patch / JSON Patch; framework-agnostic
  пакет, семя будущего `player-core`.
- `apps/player-web/src/presenter/` и `src/lib/` — presenter-слой и клиент
  runtime-api; по ADR-064 эти каталоги не импортируют React/Next (шов охраняет
  `npm run verify:player-core-seam`), поэтому извлечение `packages/player-core`
  по триггерам ADR-064 (агентный headless-клиент ADR-060, Phaser-канал ADR-062,
  встраивание вне Next.js) останется механическим переносом.

Историческая слоистая SDK-стратегия (`SDK/core`/`shared`/`react-sdk`/`viewers`,
ADR-014) заменена ADR-064: каталог `SDK/` упразднён, живой код перенесён в
`packages/view-protocol`, внешний публикуемый SDK отложен как планируемая
работа (LEGACY-0037).

Рост самого рендерера управляется ADR-066: зафиксирована ось декомпозиции
«ядро + жанровые UI capability packs» с ленивыми канальными поверхностями,
именованными триггерами и CI-бюджетом бандла как датчиком; до срабатывания
триггеров рендерер остаётся единым (LEGACY-0039).

Для пространственных игр ADR-080 добавляет в ядро компоновки опциональный режим
`map-first`. UI-манифест объявляет смысловые зоны `board`, `status`,
`primary-panel`, `context-panel`, `action-tray`, `floating-controls` и
`overlay`; общий renderer владеет адаптацией и слоями, а конкретная игра —
содержимым этих зон. Это не новый renderer на игру и само по себе не активирует
жанровую декомпозицию ADR-066.

**Клиентские приложения:**
- `draft/antarctica-nextjs-player/` — archived UI prototype and visual reference only.
- `apps/player-web/` — current canonical web-player scaffold.
  - Должен опираться на `runtime-api` как на session/action boundary и player-facing content boundary.
- `draft/cubica-portal-nextjs/` — current portal draft from upstream `aproskur/cubica-portal-nextjs`; используется для анализа и подготовки test VPS launch with `Antarctica`.
- `apps/portal-nextjs/` и `services/portal-backend/` — imported portal drafts for later analysis and redesign.
- `apps/editor-web/` — current authoring editor surface for ADR-034/ADR-036. Он содержит preview-first workspace, manifest entity tree, advanced JSON tree, JSON editing через Monaco, floating property panel, preview selection overlay/object picker/region prompt baseline, repository file workflow и validate/compile/preview actions поверх `@cubica/editor-engine`.
- `draft/Antarctica/` — legacy mechanics reference. На текущем migration этапе `draft/Antarctica/GameFull.html` остаётся фактическим extraction source для ещё не перенесённой gameplay-логики, но не считается canonical runtime truth.

### 2.3. Данные и игровые манифесты

Игровые манифесты описывают игру как данные: метаданные, состояние, действия, design references и runtime-facing content. Для current canonical slice `Antarctica` источником истины для исполнимой логики является `games/antarctica/game.manifest.json`, а не narrative markdown.

Практически это означает:

- **game manifest** содержит метаданные игры, конфигурацию, `content` references, конфигурацию движка, начальное состояние (`state`) и реестр действий (`actions`);
- **UI manifests** продолжают существовать как отдельный delivery layer для каналов (`games/<id>/ui/<channel>/ui.manifest.json`), но не являются источником истины для runtime-логики;
- `games/antarctica/design/mockups/` остаётся источником UI intent;
- `draft/Antarctica/GameFull.html` на текущем этапе является только фактическим extraction source для ещё не перенесённой механики. Это текущее состояние миграции, а не новое архитектурное решение, и не canonical runtime truth.

**Приём геометрии карты до authoring-манифеста (ADR-025/030):**

Изображение или векторный макет может быть доказательством координат, но не
становится исполнимым правилом напрямую. Общая JSON Schema
`map-annotation.schema.json` задаёт промежуточную аннотацию узлов, связей и
областей; статус `review-draft` допускает неизвестные состояния и неполные
области только для контрольного наложения. Общий конвертер обязан отклонить
публикацию такого черновика. Игровой адаптер добавляет стоимость, типы объектов
и допустимые состояния только после авторского подтверждения, а нейтральная
фикстура доказывает, что этот конвейер не зависит от конкретной игры.

**Game-defined UI panels (ADR-053, Accepted):**

UI-манифесты должны различать primary screens and panels. `screens` описывает основные визуальные состояния игры, связанные с текущим scenario/timeline state. `panels` описывает временные UI-слои поверх текущего экрана: журнал ходов, подсказки, справку, инвентарь и похожие элементы. Платформа владеет только общим lifecycle панели и декларативным manifest rendering. Конкретный UI журнала ходов и подсказки `Antarctica` живет в `games/antarctica/ui/web/ui.manifest.json` как `panels.history` and `panels.hint`, а не в платформенных React-компонентах.

Game manifest не должен содержать UI-only actions для локального открытия таких панелей. Runtime log entries остаются игровыми событиями и могут быть источником данных журнала, но команды "открыть журнал" и "открыть подсказку" принадлежат UI layer and Presenter state. В `Antarctica` старые platform-specific `JournalRenderer`, `HintRenderer`, `showHistory`, `showHint`, `showTopBar` and `showScreenWithLeftSideBar` удалены; layout-only переключатели должны задаваться UI-манифестом, routing state or future schema-backed Presenter commands, not runtime actions.

**Граница game/UI manifest и метрики (ADR-054, Accepted):**

Cubica использует правило смыслового владения: game manifest owns meaning, UI manifest owns presentation, Presenter owns player-facing projection. `game`-манифест хранит игровые сущности и их смысловые тексты: карточки, этапы, персонажей, игровые метрики, названия и описания этих метрик, правила and actions. UI-манифест хранит channel-specific отображение: screen variants, panels, layout, CSS classes, web icons, backgrounds, button labels and local UI-only copy.

Если значение должно быть одинаково понятно в Web, Telegram, Mobile and facilitator reports, оно принадлежит game manifest или player-facing projection из game manifest. Если значение можно заменить при смене канала без изменения игры, оно принадлежит UI manifest.

Для `Antarctica` `state.public.metrics.time` является авторитетной метрикой прошедших игровых дней. `remainingDays` является вычисляемой player-facing метрикой "осталось дней", которую Presenter строит из `time` and declared game limit. `remainingDays` не должен храниться как независимое изменяемое session state, а `score` не должен использоваться как скрытый alias for remaining days. UI topbar/sidebar должны ссылаться на metric identifiers or `metricViews`, а не быть единственным владельцем подписей "Знания", "Доверие", "Энергия" and similar gameplay concepts.

**Дизайн-артефакты для ИИ-агентов (ADR-016):**

Для эффективной работы ИИ-агентов с дизайном UI-манифест поддерживает секцию `design_artifacts`, содержащую:
- реестр дизайн-артефактов (reference, concept, flowchart, wireframe, storyboard, mockup, asset);
- ссылки на внешние JSON-описания изображений с семантической разметкой зон (`regions`) и дизайн-токенами (`style_tokens`);
- граф связей и историю версий артефактов (`design-history.json`).

Это позволяет ИИ-агентам понимать структуру макетов, генерировать UI-код по описаниям и отслеживать эволюцию дизайна от концепта до финального asset.

**Элементные промты (ADR-048, Accepted):**

Authoring-экземпляры получают отдельное поле `_prompt`, а прототипы - `_promptTemplate`. Эти поля описывают авторское намерение конкретного элемента: содержимое, поведение, связи с состоянием, переходы и методический смысл. Они не заменяют `generation.prompt` из design artifacts, потому что `generation.prompt` описывает визуальную генерацию изображения или региона. Первый срез реализован в `manifest-authoring-common.schema.json`, authoring v2 examples, compiler stripping rules и editor schema readiness. Reverse-sync и механизм синхронизации промта со структурой остаются отдельной последующей проработкой.

**Dynamic element prompts, хранение и drift diagnostics (ADR-049, Accepted):**

Принятое направление объединяет storage и A + B lite sync вокруг динамического compiled prompt. `_prompt` должен хранить только static residue - невосстановимое авторское намерение, которое агент не смог надежно сопоставить со структурированными полями. Динамическая часть prompt пересобирается из выбранного game/UI JSON-узла и связанных узлов как YAML-проекция с русскими названиями только значимых игровых, методических или визуальных свойств; технические поля остаются в скрытом source map/context и не показываются пользователю. Русские названия берутся из JSON Schema annotations и authoring/editor field dictionary, а не записываются в каждый элемент. Reverse direction идет через ИИ-агента, который получает compiled prompt, schema context, dictionary и projection source map и возвращает только `EditorChangeSet` с dry-run, validation и подтверждением пользователя. Канонические manifests остаются JSON; YAML используется как presentation format для prompt projection.

**Извлечение и повышение authoring-прототипов (ADR-050, Accepted):**

Cubica принимает три направления для прототипов: локальное извлечение повторяющихся game/UI authoring-элементов в `_definitions` конкретной игры; двухуровневую модель, где локальный game-level prototype может вручную повышаться до platform-level prototype; и AI-assisted prototype designer, который предлагает имя, параметры, `_semantics`, `_promptTemplate` и `EditorChangeSet`, но не применяет изменения напрямую. Platform-level prototype является authoring-only артефактом, не попадает в runtime-api/player-web и не может содержать game-specific id, тексты, метрики или правила. Полностью автоматическое повышение в платформенный каталог отклонено.

Регулярный поиск кандидатов делится на быстрый детерминированный аудит и недельный LLM-семантический аудит. PR-аудит проверяет только измененные authoring-файлы и сначала работает как advisory report; недельный цикл выполняется через CI scheduled workflow на default branch и включает полный deterministic scan, semantic LLM review и promotion backlog review. LLM-аудит может находить смысловые повторы, которые не совпали по JSON-форме, но он создает только candidate records и не может применять `EditorChangeSet`, писать `_definitions` или повышать прототипы без deterministic gates и ручного решения. Editor surface должен показывать неблокирующее уведомление, если weekly audit отсутствует, просрочен, упал или прошел без LLM-семантической части.

Текстовые источники и якоря из более ранней manifest-lineage остаются историческим направлением (см. ADR-013), но не считаются текущей канонической truth model для `Antarctica`. В текущем срезе narrative/extraction artifacts могут существовать как reference material или migration input, но не как runtime source of truth.

- `games/antarctica/` — эталонный пакет игры (манифесты + ассеты).
- `games/simple-choice/` — минимальный второй пакет игры для проверки generic runtime/player boundary без custom plugin.
- `games/ai-driven-choice/` — минимальный AI-driven пакет игры для проверки Agent Runtime readiness, local mock adapter and Agent Turn execution.
- **Схемы и форматы:**
  - `docs/architecture/schemas/manifest-structure.md` — описание концепции манифестов.
  - **Важно:** Согласно ADR-025, единственным источником истины (SSOT) для структуры манифеста является `docs/architecture/schemas/game-manifest.schema.json`. Императивные TypeScript-проверки структуры запрещены; бэкенд использует `ajv` для декларативной валидации.
  - `docs/architecture/schemas/ui-manifest.schema.json` — формальная JSON Schema UI‑манифеста (экраны, компоненты, макеты).
  - `docs/architecture/schemas/examples/*.json` — примеры валидных манифестов.
  
  **Версионирование манифестов:**

- Метаданные манифеста содержат поля `schema_version` и `min_engine_version`, описанные в ADR‑008.
  - `schema_version` фиксирует версию схемы данных, а `min_engine_version` — минимальную версию движка, на которой игра может выполняться.
  - Game Engine при загрузке игры проверяет совместимость версий и отклоняет манифесты с неподдерживаемой схемой.

  **Обучающие метаданные и методические материалы:**
  
  - В секции `meta.training` манифеста описываются тренируемые компетенции (список компетенций с идентификаторами и описаниями), формат игры (`single`, `single_team`, `multi`) и рекомендуемая продолжительность сессии (минимум/максимум в минутах).
  - В текущем manifest shape методические материалы живут в `content.methodology`:
    - `participants` — материалы для участников (правила, описания сущностей, рекомендации, подсказки, чек‑листы);
    - `facilitators` — материалы для ведущих (интерпретации, вопросы, подсказки, критерии оценки и индикаторы паттернов поведения).
  - Эти поля формализуют обучение поверх игрового сценария и позволяют каталогу/редактору и LLM‑движку учитывать обучательные цели при работе с играми.

**Хранение и доставка манифестов:**

- Канонические манифесты хранятся в Game Repository (файловая система или БД в зависимости от фазы).
- ADR-005 допускает отложенный Redis-кэш манифестов (Phase 2+, см. `docs/architecture/redis-keys.md` и `docs/architecture/backend/redis-usage.md`); сейчас кэш не внедрён, источником истины остаётся Repository.

### 2.4. Execution Model и LLM Context Pipeline

Execution Model определяет, как платформа обрабатывает ход игрока: какая часть логики выполняется LLM, а какая — детерминированными скриптами, и как формируется контекст запроса к модели.

**LLM Context Pipeline** (`docs/architecture/engine/llm-context-pipeline.md`, ADR‑004):
- Загружает текущее состояние сессии и разделяет его на `public` и `secret` ветки.
- Формирует историю сообщений (sliding window) и, при необходимости, добавляет краткое резюме долгих сессий.
- Подгружает markdown‑ассеты (правила, лор, описания) по ссылкам из манифеста и подставляет их в системный промпт.
- Собирает финальный запрос к LLM с учётом настроек контекста (`include`, размер окна истории и т.п.).
- Статус: справочная историческая модель контекста. Для Agent Runtime действует принятый stateModel-профиль персональной проекции (ADR-058/084, см. «AI-driven runtime mode» ниже); новый профиль контекста требует schema-first контракта, role-scoped проектора и тестов отсутствия утечек.

**Hybrid Execution Model** (ADR-007, ADR-015, updated by ADR-040 and ADR-084):
- Исторически действия в манифесте могли обрабатываться LLM (`llm`) или скриптом (`script`).
- Действующий путь — actor-scoped Game Intent → Cubica Mechanics IR: клиент передаёт только `actionId + params + commandId`, а runtime выполняет раздельные `availability`/`authorization` и типизированную атомарную транзакцию через явно закреплённые алгоритмические модули. Старые `guard + deterministic.effects[]`, runtime templates и подстановки удалены без постоянного адаптера.
- **Игровой исполняемый код** разрешён как крайний путь, когда декларативного языка и общих платформенных модулей недостаточно. Такой код работает только в изолированном процессе или песочнице конкретной игры, не получает прямого доступа к памяти и хранилищу платформы и возвращает проверяемый типизированный результат; произвольный JSON patch, старый effect или обход Game Intent/Mechanics IR не допускаются.
- **Engine Extensions** для тяжёлых вычислений или внешних систем используют ту же ограниченную границу результата и не подменяют доступные manifest/platform capabilities.

**AI-driven runtime mode** (ADR-046):
- Cubica supports `deterministic`, `ai-driven` and `hybrid` execution modes as manifest-declared platform capabilities.
- In `ai-driven` mode, an Agent Runtime is part of gameplay execution: a player action or system event starts an agent turn, and the agent returns structured output.
- Agent output may include narration, diagnostics, a `CubicaSurface` and selection of an actor-scoped Game Intent. Прямого поля state effects больше нет; агент не отправляет произвольные Mechanics IR или patches.
- Runtime validates all agent output through JSON Schema and semantic rules. Durable deterministic state changes converge on the same Game Intent → Mechanics IR transaction path as human commands.
- Текущий проверенный профиль контекста использует единый stateModel-driven строитель персональной проекции для человека и игрового ИИ: `stateScope.public` содержит объявленные общедоступные символы, `stateScope.actor` — только символы actor, подтверждённого principal, а `server`, `state.secret`, необъявленные поля и данные других actor исключаются. Широкие экранные ветви допускаются только как ограниченные `projection-only` endpoints: они видимы проектору, но недоступны Mechanics IR, поэтому не обходят узкие типизированные рабочие endpoints. Фиксированная проекция `/meta` + `/actions` сохраняется; новый профиль нельзя добавить без schema-first контракта, role-scoped projector и тестов отсутствия утечек.
- Deterministic games and deterministic paths remain playable without Agent Runtime. AI-driven games must declare readiness and failure policy before launch.
- Telegram and Phaser integrations consume validated Surface projections, not provider messages or React component instances.
- Production AI-driven rollout requires accepted/rejected event log entries, replay transcripts, evaluation fixtures, audit metadata and operation policy before real provider adapters.

**Manifest-driven gameplay mechanics** (ADR-024, amended by ADR-083 and ADR-084):
- Antarctica gameplay migration uses explicit manifest actions and deterministic state, while reusable selectors, conditions, branching and transitions belong to the universal mechanics language rather than separate subject-specific runtime effects.
- Threshold progression, metric-gated outcomes, line switching, locked-card unlock and entry-time swaps remain rules of the game manifest, but missing computation primitives are designed as neutral schema-first platform mechanisms from the first proven scenario.
- Player-visible availability and progress must stay auditable through explicit deterministic state, typically in `state.public`.
- Delivery-specific step-, board-, and card-level rules live in Gameplay Slice Records under `docs/architecture/gameplay-slices/`, not in ADRs.

**Runtime mechanics language** (ADR-029, ADR-040, superseded in concrete form by ADR-084):
- Клиентский контракт выражается Game Intent в терминах игры; существующий `actionId` становится его каноническим идентификатором без переименования и второго каталога. Новая серверная механика выражается через типизированный Cubica Mechanics IR, а не через добавление нового `deterministic.effects` op, JsonLogic-строки или публичного JSON Pointer.
- Каноническая программа состоит из чистых запросов, утверждений, изменяющих команд и версионированных алгоритмов; `objects.updateMany` является одной командой, а не отдельным языком.
- Любая доступная запросам коллекция имеет тип, видимость, стабильный ключ и максимальную ёмкость. До публикации semantic checker проверяет наличие и тип начальных значений endpoints, а также сверяет начальные `objectType`, facet/attribute-поля, значения и capacity с закрытой моделью коллекции; runtime повторяет проверки при чтении и на границе мутации. Профиль бюджета ограничивает статическую стоимость, runtime-работу, память и выход; манифест не может повышать платформенный предел.
- Для графового перемещения модель вместимости узла перечисляет типы объектов и может сузить их фасетом жизненного цикла: резервные или рыночные объекты тогда хранятся в узле, но не занимают транспортный слот. Без такой пары полей все перечисленные типы консервативно считаются занимающими место.
- `availability` читает только открытый участнику контекст, `authorization` остаётся серверным, а транзакция исполняется на candidate state и делает один commit либо полный rollback вместе со случайностью, событиями и версией сессии.
- Presenter строит проекцию для чтения и маршрутизирует запись; он не вычисляет законность, не выполняет преобразование в IR и не заявляет доверенный `playerId`. Повтор одной внешней логической команды сохраняет случайный `cli_ commandId`, сначала ищет прежнюю квитанцию и не исполняет IR второй раз; `expectedStateVersion` независимо разрешает конкуренцию разных внешних команд. Внутреннее срабатывание использует детерминированный `sys_ commandId` и вместо неизвестной заранее версии проверяет schedule/trigger под session lock; префикс не даёт полномочий.
- Предметные authoring-макросы раскрываются до публикации. Mechanics capability packs служат build-time discovery/composition; runtime algorithm modules и UI capability packs имеют отдельные пространства имён и границы доверия. Граф, геометрия, колоды и случайность остаются нейтральными runtime modules с JSON Schema, типизированным результатом, моделью стоимости и тестами соответствия; `moduleLock` отдельно закрепляет `moduleId`, точную `moduleVersion`, `artifactHash` и версии значимых алгоритмов.
- Если механика требует расширения `runtime-api`, расширение добавляется как общая platform capability with JSON Schema, not as `gameId` branch.
- Cleanup манифеста `Antarctica`, описанный в `docs/tasks/artifacts/TSK-20260527-editor-engine-preview-timeline-editor/antarctica-manifest-cleanup.md`, следует этому правилу: привязанные к игре family names, старые script markers и ссылка на runtime-script заглушку закрыты через manifest/platform capabilities, без runtime plugin и без веток под конкретную игру.

**Протокол взаимодействия с View** (ADR‑002, `docs/architecture/protocols/mvp-interaction.md`):
- Presenter общается с клиентом через абстрактный шлюз команд (`ViewCommand` / `ViewResponse`), не завися от конкретного UI‑фреймворка.
- Это позволяет подключать разные каналы (Web, Telegram и др.) через View Adapters без изменения игровой логики.

### 2.5. Модель сессий и мультиплеера

Модель сессий описывает, как хранится и изменяется состояние игры между ходами и как несколько игроков могут участвовать в одной игровой сессии.

**Сессии и хранение состояния** (ADR‑005, `docs/architecture/backend/session-persistence.md`):
- Реализован основной постоянный контур: состояние сессии хранится в таблице `game_sessions` в PostgreSQL как JSONB (`state` + резерв `history`), с полями `state_version` и `last_event_sequence`; после перезапуска восстанавливаются также доверенная роль и источник контента предпросмотра.
- Все текущие пути изменения сессии используют одну транзакцию на одном соединении и пессимистичную блокировку `SELECT FOR UPDATE NOWAIT`; условная запись по ожидаемой версии дополнительно защищает от потери обновления. `state_version` всегда увеличивается ровно на единицу, включая restore предпросмотра; при restore откатывается только игровое состояние, а `last_event_sequence` защищённого журнала остаётся монотонным, поэтому новый факт не может повторно занять существующий `eventId`. Хранилище в памяти разрешено только как явно выбранный dev/test-адаптер.
- Механизм Session Recovery основан на концепции checkpoint: в БД всегда хранится последнее устойчивое состояние, а при сбоях обработка откатывается к нему с учётом TTL блокировки.
- Прикладной TTL длительного хода остаётся открытой частью Session Recovery; очередь событий и WebSocket-доставка относятся к следующим фазам мультиплеера.

**Мультиплеерная модель** (ADR‑011):
- Для многопользовательских игр `session_events` является неизменяемым журналом только уже подтверждённых игровых фактов; `state_version` и `last_event_sequence` задают явную версионность состояния и журнала.
- Текущий монолит последовательно обрабатывает команды сразу под блокировкой сессии. Если будущему асинхронному транспорту понадобится предварительная очередь, закрытые конверты команд хранятся отдельно в `session_command_queue`, а не смешиваются с фактами `session_events`.
- Целевая доставка (ещё не реализована, `TSK-20260705-multiplayer-runtime-realization`): обновления состояния рассылаются всем участникам сессии через WebSocket-endpoint `runtime-api` по ADR-059; сегодня клиенты получают состояние по HTTP-запросам.

### 2.6. Agent UI Foundation

ADR-043 принимает CopilotKit и AG-UI как текущую основу пользовательских ИИ-помощников. ADR-044 уточняет переносимость: CopilotKit и AG-UI являются заменяемыми адаптерами, а стабильным внутренним интерфейсом остаются контракты Cubica. ADR-045 задаёт целевую траекторию: CopilotKit остаётся MVP-адаптером первого этапа, а долгосрочный слой должен стать собственным совместимым Cubica Agent UI с Cubica Surface Protocol для декларативных UI-поверхностей. ADR-046 расширяет применение Surface: в AI-driven games агент может возвращать основную игровую UI-поверхность, если манифест объявляет такой execution mode.

Разделение ролей:

- **CopilotKit** владеет UI помощника в React/Next.js-приложениях: чатом, боковыми панелями, потоковыми ответами, рендером frontend tools и human-in-the-loop controls.
- **AG-UI** владеет внешним событийным протоколом между пользовательскими приложениями и backend-сервисами агентов: run lifecycle, потоковым текстом, tool calls, state snapshots, state deltas и custom events.
- **Контракты Cubica** владеют долговременным предметным состоянием: манифестами, рабочими копиями editor session, runtime sessions, portal launch sessions, лицензиями и аудитом.
- **Cubica Surface Protocol** владеет будущим внутренним форматом декларативных UI-поверхностей помощника: словарём компонентов, data model, actions, validation diagnostics and side-effect policy. A2UI-подобные события могут переводиться в этот формат только через адаптер.

Принятое правило:

```text
assistant suggestion -> Cubica command/change set -> Cubica validation -> Cubica state change
```

Для редактора это означает, что вывод помощника должен стать `EditorChangeSet` и пройти dry-run, JSON Schema validation, semantic validation, plugin boundary checks и запись в undo journal до apply. Для помощников портала, игрока, ведущего и администратора тот же паттерн применяется через их собственные domain APIs и RBAC.

Состояние CopilotKit/AG-UI является производным состоянием UI помощника. Оно не записывается в generated runtime manifests и не меняет gameplay state напрямую. Deterministic public player runtime остаётся работоспособным без agent backend; AI-driven games могут требовать Agent Runtime только через явно объявленный execution mode и readiness policy.

Текущий baseline редактора защищён feature flags:

- `NEXT_PUBLIC_CUBICA_EDITOR_AGENT_UI=1` включает браузерную оболочку помощника.
- `CUBICA_EDITOR_AGENT_RUNTIME=1` включает app-local маршрут CopilotKit runtime.
- `CUBICA_EDITOR_AGENT_AG_UI_URL` и необязательный `CUBICA_EDITOR_AGENT_AG_UI_TOKEN` подключают внешний AG-UI backend.
- Если `CUBICA_EDITOR_AGENT_AG_UI_URL` отсутствует, editor-web использует встроенный локальный AG-UI backend по `/api/editor/agent/ag-ui`, если не задано `CUBICA_EDITOR_AGENT_LOCAL_BACKEND=0`.

Без этих флагов `apps/editor-web` сохраняет прежнее поведение редактора. Если локальный backend явно отключён и внешний backend URL не задан, редактор показывает статус подключения помощника и не монтирует CopilotKit hooks, привязанные к агенту.

### Архитектурные решения (ADR)

Ключевые решения зафиксированы в документах `docs/architecture/adrs/`:

- **ADR-001 (MVP & LLM-first):** Отделение данных (манифеста) от движка.
- **ADR-002 (View Protocol):** Абстрактный протокол UI команд.
- **ADR-003 (Hybrid SDUI):** Гибридная схема серверно-управляемого UI (атомарные примитивы + семантические виджеты) — основа текущего UI-манифеста; развита ADR-055 (чистота рендерера, декларативные привязки) и ADR-066 (ядро + UI capability packs); агентные поверхности Cubica Surface (ADR-045) — отдельный контур, не замена UI-манифеста каналов.
- **ADR-004 (LLM Context Pipeline):** Формирование контекста для LLM с разделением public/secret состояния и историей.
- **ADR-005 (Session Persistence):** Хранение состояния сессий, блокировки и стратегия восстановления.
- **ADR-006 (View Adapters):** Архитектура развёртывания и взаимодействия View Adapters.
- **ADR-007 (Hybrid Engine):** Совмещение LLM и JS‑скриптов.
- **ADR-008 (Versioning):** Стратегия версионирования манифестов (`schema_version`, `min_engine_version`).
- **ADR-009 (Assets):** Централизованное управление медиа-ассетами в манифесте.
- **ADR-010 (JS Security):** Историческое решение про `isolated-vm`; для текущего runtime-plugin направления действует уточнение ADR-040: `isolated-vm`, `node:vm` и `worker_threads` не являются защитной границей для чужого кода.
- **ADR-011 (Multiplayer):** Free-form модель с очередью событий для поддержки нескольких игроков в сессии, явной версионностью состояния (`state_version`) и последовательностью событий (`sequence`), а также правилами обработки зависших и ошибочных событий.
- **ADR-012 (Training Metadata):** Обучающие метаданные и методические материалы в манифесте игры.
- **ADR-013 (Text Anchors & Manifest Split):** Текстовые якоря для синхронизации с источниками и разделение логического и UI-манифестов.
- **ADR-014 (Viewers Library Architecture):** superseded by ADR-064. Исторический proposed путь для библиотеки viewers в `SDK/viewers/` и проверенных клиентских скриптов; роль viewer закрыта декларативным рендерером (ADR-055) и плагинами (ADR-037/039), а подпись/реестр стороннего кода отложены до появления недоверенного кода (LEGACY-0036).
- **ADR-015 (Extension Packs):** Историческая архитектура пакетов расширений и гибридная модель движка; текущая runtime-api политика уточнена ADR-040.
- **ADR-016 (Design Artifacts):** Дизайн-артефакты для ИИ-агентов в UI-манифесте — JSON-описания изображений с семантической разметкой и дизайн-токенами.
- **ADR-017 (Modular Monolith Transition):** Ближайшая backend‑фаза строится как модульный монолит с жёсткими внутренними границами; выделение микросервисов откладывается до появления подтверждённых operational boundaries.
- **ADR-018 (JSON Manifest Truth Model; amended by ADR-083/084):** авторитетное объявление исполнимой логики закрепляется в `games/<id>/game.manifest.json`, а narrative и draft-артефакты не считаются runtime source of truth. Предпочтительный путь хранит Mechanics IR и общие capabilities как данные; если их недостаточно, код изолированного расширения лежит в content-addressed bundle игры, а манифест закрепляет его hash, schema-defined вход/выход, права и runner. Такой bundle не становится вторым незакреплённым источником истины и не загружается в процесс общего `runtime-api`.
- **ADR-019 (Runtime-Owned Player Content Boundary):** `runtime-api` владеет загрузкой game content и проекцией player-facing content DTO/API; `player-web` не должен читать `games/*` напрямую.
- **ADR-024 (Manifest-Driven Gameplay Mechanics, amended by ADR-083):** Cubica моделирует игровые правила через explicit manifest actions и проверяемое детерминированное состояние; общие селекторы, правила, циклы и сценарная композиция могут вводиться с первого доказанного игрового сценария, а delivery-specific slice specs выносятся в Gameplay Slice Records.
- **ADR-025 (JSON Schema SSOT):** Runtime manifest structures валидируются через JSON Schema как cross-platform source of truth; TypeScript-only guards не должны заменять декларативные схемы.
- **ADR-026 (Game-Agnostic Plugin Architecture):** Game-specific presentation logic остается в plugin layer, а generic player/runtime слои не получают hardcoded game branches.
- **ADR-027 (Platform Universality Improvements):** `player-web` поддерживает data-driven routing, metric specs и plugin-optional default path для простых игр.
- **ADR-028 (Action Templates, Superseded by ADR-084):** историческая runtime-шаблонизация и `{{...}}`-подстановки удалены. Компактность authoring может обеспечиваться только build-time макросами, полностью раскрытыми и проверенными до публикации.
- **ADR-029 (Three-Tier Logic Model, Superseded by ADR-084):** runtime больше не выбирает templates, JsonLogic или scripts. Действующий путь — опубликованный Game Intent → типизированный Mechanics IR; необходимый игровой код допускается только как изолированное расширение игры.
- **ADR-030 (Semantic Prototype Manifests):** Draft-решение вводит обязательный authoring layer для game/UI manifests, который компилируется в runtime JSON без authoring-only ключей.
- **ADR-031 (Lightweight Task System, уточнен ADR-068):** Текущая работа планируется через `NEXT_STEPS.md`, корневые и необязательные дочерние `TSK-*` и артефакты задач, а ADR не используется как execution tracker. Общий план всегда хранится в корневой задаче. В явно выбранном `$cubica` его утверждает человек; в обычном игровом режиме агент сам формирует и реализует план по вводным PM, согласуя только существенные архитектурные решения.
- **ADR-032 (Portal Session Launch Boundary):** Портал управляет покупками, ссылками запуска и launch sessions, а runtime/player сохраняют владение игровым состоянием и отображением.
- **ADR-033 (Portal Runtime Session Binding):** Портальная launch session должна явно связываться с runtime session; single-player day/month используют per-device binding, multiplayer использует shared binding, а one-time всегда ведет в одну runtime session.
- **ADR-034 (Editor Engine For Authoring Manifest Editing):** `editor-engine` редактирует authoring-манифесты через schema-first graph projection, JSON tree view, Monaco/JSON editor, floating property panel и reverse projection в JSON Patch, не создавая второго source of truth. Текущая реализация закрывает graph/tree/text/panel; у Tree mode реализован scalar `set value`, а structural operations вынесены в follow-up.
- **ADR-035 (Progressive Semantic Graph UX For Editor Engine):** следующий UX-срез `editor-engine` заменяет полный JSON-tree canvas на progressive semantic graph: текущая ветка раскрывается поэтапно, соседние ветки сворачиваются, узлы получают semantic roles/titles, а JSON/property panels становятся сворачиваемыми.
- **ADR-036 (Semantic Authoring Structure And Preview-Timeline Editor):** следующий архитектурный поворот editor-engine: реальные game/UI сущности должны жить в `root` authoring-манифестов, `_definitions` остаются прототипами, `_label` хранит отображаемое имя сущности, а основным editor surface становится preview-first workspace с timeline, playthrough traces, entity tree and AI prompt overlay. Phase 1A реализовала v2 schema/fixture/compiler baseline; Phase 1B migrated current authoring manifests to v2; Phase 2/3 added renderer-neutral preview adapter contracts, a thin DOM adapter baseline and the preview-first editor shell; Phase 4 added default semantic entity tree with `_label` diagnostics; Phase 5 added manifest chronology timeline and preview trace restore planning; Phase 6/7 baseline added iframe preview descriptors, source-map pointer mapping, click/region selection overlay, object picker and AI intent queue. Phase 8 baseline added automatic active-file JSON ChangeSet apply, dry-run validation, undo/redo journal and diff summary. Phase 9 added Git worktree sessions for editor file/layout reads, Save commits, restore-commit rollback helpers, plugin boundary checks, session-aware validate/compile, allowlisted local session-aware runtime preview through `contentSourceId` and browser e2e for that local path.
- **ADR-037 (Project-Local Plugins And Marketplace-Safe Evolution):** user-editable plugins move to project-local `games/<gameId>/plugins/<pluginId>/`; first implementation supports trusted local player-web plugins with schema/typecheck/build/test validation, no npm dependencies, hot preview reload and Antarctica migration to `games/antarctica/plugins/antarctica-player`. Marketplace and first-class runtime-api plugins are reserved for later sandboxed evolution. Runtime-api plugin-like code may exist before that ADR only as documented legacy/technical debt when server-side game logic cannot be expressed otherwise.
- **ADR-038 (Testing Architecture And Policy):** тестовая архитектура строится как policy layer (слой правил, который определяет обязательные проверки) поверх текущих runners (запускателей тестов): `node:test` для backend, Vitest для TypeScript/UI packages, Playwright для browser E2E, Ajv/JSON Schema для contract validation и replay/eval contour для gameplay/LLM behavior. Версионированный канонический SHA-256-отпечаток повтора сортирует ключи полного серверного состояния и исключает только служебные `public.log[*].at` и `runtime.lastUpdatedAt`; эти отметки остаются в сохранённом аудите, а одноимённые поля вне точных путей продолжают влиять на результат.
- **ADR-039 (Player-web Plugin Bundle Handoff):** local preview baseline is implemented: `editor-web` builds a session-scoped browser file for project `player-web` plugins, `runtime-api` carries only references through the preview content-source boundary, and `player-web` loads that file only in preview mode. The production model is implemented through immutable content-hash published bundles exposed through `PlayerFacingContent.pluginBundles`; the bundle builder and browser loader accept the exact current plugin `apiVersion: "2.0"`, while other versions fail closed. `TSK-20260531-player-web-published-plugin-bundle-handoff` records that slice.
- **ADR-040 (Runtime-api Extension Policy And Declarative Mechanics First, amended by ADR-083/084):** новая серверная механика сначала проверяется на выразимость через универсальный декларативный Mechanics IR или общую платформенную возможность. Функционал конкретной игры в общем `runtime-api` запрещён, но исполняемый код игры разрешён как крайний полноценный путь, если декларативного языка и общих модулей недостаточно. Он остаётся в namespace/bundle игры, запускается вне процесса платформы в изолированном runner, получает только schema-defined вход, ограничивается по правам и ресурсам и возвращает проверяемое значение либо change set, который атомарно применяет платформа. Выбор этой уже принятой границы является реализационным решением; новое архитектурное согласование нужно только для дополнительных прав, ослабления изоляции, нового источника истины или существенно иной эксплуатационной модели. Для marketplace целевой путь — контейнерная песочница или WebAssembly/WASI; полноценный runner ещё не реализован и зафиксирован как `LEGACY-0014`.
- **ADR-041 (Gameplay Object State Model, amended by ADR-084):** вводит общий контракт для игрового состояния объектов: authoring-манифест описывает типы объектов и фасеты состояния, runtime state хранит session-scoped object instances, типизированные Mechanics commands создают и изменяют динамические ресурсы, а Presenter строит player-facing projection для View. Состояние остаётся хранилищем, а общие selectors/query/assert/command узлы выполняют переходы, массовые изменения и сценарии без предметных операций. Старые object effects/guards удалены. Общий путь и proof на `simple-choice` реализованы без постоянного legacy fallback.
- **ADR-042 (Editor Session Versioning And Lifecycle):** accepted editor hardening step: editor sessions are versioned resources with platform release and plugin API metadata, active-session reuse, upgrade dry-run diagnostics and automatic cleanup for expired/orphaned worktrees.
- **ADR-043 (CopilotKit And AG-UI Agent UI Foundation):** accepted platform foundation for user-facing AI assistants. CopilotKit is the default React/Next.js assistant UI layer, AG-UI is the default external agent event protocol, and all durable changes still pass through Cubica contracts and validation boundaries.
- **ADR-044 (Переносимость Agent UI и границы протоколов):** фиксирует CopilotKit как заменяемый UI/runtime-адаптер, AG-UI как заменяемый адаптер внешнего протокола и Cubica agent contracts как долговременную границу для инструментов, контекста, валидации, аудита и production LLM backend.
- **ADR-045 (Cubica-Owned Generative UI And MVP CopilotKit Adapter):** фиксирует CopilotKit как MVP-адаптер первого этапа, целевой собственный compatible Cubica Agent UI и Cubica Surface Protocol for declarative Generative UI.
- **ADR-046 (AI-Driven Game Runtime Mode, amended by ADR-084):** фиксирует `ai-driven` и `hybrid` execution modes, где Agent Runtime может быть обязательной частью игрового исполнения. Narration, diagnostics и Cubica Surface остаются валидируемым агентным результатом; долговременная мутация не принимает произвольные agent-produced state effects и сходится в actor-scoped Game Intent → Mechanics IR.
- **ADR-047 (AI Agent Safety Remediation Gates, amended by ADR-084):** accepted safety gates перед production Agent Runtime: human approval хранится как Cubica approval envelope, rejected Agent Turn не выбирает и не исполняет Game Intent, `allowedCapabilities` является исполняемым runtime allowlist, Surface actions проверяются channel policy, а non-Web projections fail closed. Игровой ИИ может выбрать только одно опубликованное и доступное намерение; trigger самого Agent Turn исключается из его каталога, а низкоуровневые IR-операции и прямые изменения состояния в результат агента не входят.
- **ADR-048 (Element Authoring Prompt Contract):** принятый контракт `_prompt` для authoring-экземпляров и `_promptTemplate` для прототипов; фиксирует отличие элементного промта от `generation.prompt`, `_semantics` и временных editor prompts. Обратная синхронизация и механизм синхронизации промта со структурой остаются отдельной последующей проработкой.
- **ADR-049 (Dynamic Element Prompt Projection And Sync Strategy):** accepted направление: `_prompt` хранит только невосстановимый static residue, dynamic YAML projection строится из текущего JSON-узла с русскими field labels из schema annotations/field dictionary, но показывает только значимые игровые, методические или визуальные свойства; reverse direction идет через agent-produced `EditorChangeSet`, optional Markdown refs разрешены для длинного static residue, covered pointers/hash дают `prompt-stale`, а canonical manifests остаются JSON.
- **ADR-050 (Authoring Prototype Extraction And Promotion):** accepted модель локального извлечения прототипов, ручного повышения game-level prototype до platform-level prototype, AI-assisted designer, регулярного аудита кандидатов и editor notifications о пропущенных weekly audits; runtime/player не резолвят эти прототипы, platform-level catalog остается authoring-only слоем с явными gates против game-specific drift, а недельный LLM-семантический аудит создает только candidate records.
- **ADR-051 (API First Contract For Modular Monolith):** API First применяется уже к current `services/runtime-api`: внешний OpenAPI-контракт описывает реализованные монолитные HTTP endpoints, future service boundaries выражаются тегами/компонентами, а старые Router/Engine/Repository OpenAPI specs считаются target/extraction references до реального выделения сервисов.
- **ADR-052 (In-Memory Editor Entity Projection And Optional Hints Sidecar):** accepted решение вводит project-level `EditorEntityProjection` как in-memory индекс редактора, который связывает logic, content, state, view, design and plugin source pointers across multiple authoring files. Первый срез не создает persisted `editor.entities.json`; optional hints sidecar допустим только позднее и только для невосстановимых editor-only подсказок. Runtime-api, player-web and compiler не читают projection или hints sidecar, а все player-visible изменения записываются через `EditorChangeSet` в исходные authoring manifests.
- **ADR-053 (Game-Defined UI Panels):** accepted решение отделяет primary screens от UI panels. Платформа владеет общим lifecycle and manifest renderer for panels, а game-specific panel UI, включая журнал ходов `Antarctica`, описывается в UI-манифесте игры. UI-only actions вроде открытия журнала не должны жить в game manifest как deterministic game actions, а миграция выполняется без постоянного legacy fallback.
- **ADR-054 (Game/UI Manifest Boundary And Metric Projection, clarified by ADR-084):** accepted правило смысловой границы: game manifest owns gameplay meaning, UI manifest owns channel presentation, Presenter owns player-facing projection. На write path Presenter использует опубликованную UI-привязку `actionId + parameter bindings`, добавляет стабильный `commandId` и не владеет lowering или авторитетными правилами. Для `Antarctica` `time` означает прошедшие дни, `remainingDays` является вычисляемой метрикой остатка дней, а `score` не должен использоваться как скрытый alias for remaining days.
- **ADR-055 (Player Renderer Purity And Declarative UI Action Binding):** accepted правило чистоты рендерера: generic `player-web` renderer не содержит game-specific идентификаторов кнопок, CSS-классов и карт подписей; привязка действий и ролей блоков объявляется декларативно в UI-манифесте и его схеме, а рендерер только исполняет декларацию.
- **ADR-056 (Manifest Contract ↔ JSON Schema Parity And Contract Testing):** accepted направление синхронизации контрактов: JSON Schema остаётся SSOT (ADR-025), TypeScript-контракты `packages/contracts/*` генерируются из схемы, дрейф ловится CI drift check и контрактными тестами и становится ошибкой сборки, а не тихой рассинхронизацией.
- **ADR-057 (Preview-First Editor UX Architecture):** accepted UX-архитектура редактора: целостность игровой сущности достигается проекцией (чтение) и `EditorChangeSet` (запись) без материализованного editor-manifest; текстовый режим промт-проекции интерпретирует возвращённое намерение с детерминированным быстрым путём, построчным отчётом и шкалой риска через approval envelope; дерево — одна структура с группировками «По экранам/По типам» и вхождениями; оси времени документа и прохождения разделены; закреплённые фикстуры состояния — проверяемый authoring-артефакт; кэш редактора трёхуровневый и никогда не источник истины. Детальный источник — `docs/architecture/editor-preview-first-ux.md`.
- **ADR-058 (Turn-Based Board Game Platform Capabilities, legacy effect forms superseded by ADR-084):** accepted предметные требования пошаговых игр — воспроизводимая случайность, колоды, player-scoped state, turn flow, экономика, итог и ограниченное ветвление — сохраняются. Их прежние формы `effects[]`, строковые подстановки и отдельные handlers заменены Game Intents, типизированными IR-командами, чистыми алгоритмами и структурными узлами. Хотсит использует стабильный principal контроллера сессии и отдельно разрешаемый actor, чтобы retry после смены активного игрока не выполнил ход повторно. Повторно используемый authoring-шаблон не содержит конкретных участников и `state.public.turn`: runtime материализует обязательные `order`, `activePlayerId`, `turnNumber` и `phase` до сохранения сессии, а проверка публикации доказывает совместимость их строгих типов со всем разрешённым диапазоном игроков и отклоняет второй источник этих значений в шаблоне. GSR-034 и полный mock-срез остаются доказательствами поведения. Единый stateModel-driven проектор удаляет весь `state.secret`, серверные и необъявленные поля, изолирует actor-данные; нужный интерфейсу сигнал Antarctica перенесён в объявленную public-проекцию.
- **ADR-059 (Multiplayer Realization In Modular Monolith; amended by ADR-084):** accepted посадка принятой модели мультиплеера ADR-011 на текущий модульный монолит: PostgreSQL-хранилище сессий (ADR-005) как предусловие, модель участников (seats) с join-токенами и `kind: human|agent`, хотсит как вырожденный случай той же модели, последовательная обработка под блокировкой сессии, WebSocket endpoint в `runtime-api` и единственный строитель персональной проекции (playerView) для REST/WebSocket/агентов. `session_events` хранит только подтверждённые неизменяемые факты; будущая очередь необработанных команд, если понадобится, называется `session_command_queue` и имеет отдельный контракт. Каждый `POST /actions` передаёт `commandId` и `expectedStateVersion`: прежняя квитанция по тому же стабильному principal и `commandId` возвращается до проверки версии, а HTTP 409 применяется только к новой команде на устаревшей версии. Router/Web Gateway не выделяются до подтверждённых operational boundaries (ADR-017).
- **ADR-060 (Agent-Controlled Players; amended by ADR-084):** accepted модель ИИ-оппонентов для пошаговых игр: агент — участник сессии, а не execution mode; платформа даёт actor-scoped проекцию опубликованных и доступных Game Intents, системный Agent Turn на ходе агентского места, а выбранные `actionId + params` проходят тот же входной допуск и Mechanics IR, что и ход человека. Прямые эффекты и IR от агента запрещены; честность гарантируется той же персональной проекцией без чужих секретов, а отказ ведёт к объявленному fallback по failure policy ADR-046.
- **ADR-061 (Action Parameters, extended by ADR-074 and amended by ADR-084):** действие объявляет `paramsSchema` (плоский объект со скалярными свойствами и явным `additionalProperties: false`), клиент передаёт `params` в `POST /actions`, а runtime валидирует их Ajv на этапе входного допуска до Mechanics. Провалидированные значения доступны только типизированным Mechanics expressions/assertions/commands и попадают в состояние через явную команду. Параметры по умолчанию являются инертными данными и не интерпретируются как пути, ключи или выражения. Узкое исключение ADR-074 — строка с заранее объявленной `x-cubica-ref`: runtime разрешает её только в заданной манифестом коллекции/сети и повторно проверяет живой ресурс; даже такая ссылка никогда не становится путём или кодом. Серверный JsonLogic/effects executor удалён.
- **ADR-062 (Realtime Client Simulation And Phaser Channel; amended by ADR-084):** accepted класс игр «клиентская симуляция реального времени» и Phaser-канал доставки: реальное время живёт в клиентской сцене плагина игры, runtime владеет границами раундов через Game Intents. Старт раунда получает зерно из закреплённого модуля случайности и записывает его типизированной командой; фиксация итогов передаёт параметры по ADR-061 и проверяет их Mechanics assertions. Вся случайность сцены выводится из зерна платформенной утилитой seeded PRNG; Phaser остаётся лениво загружаемой зависимостью платформы с `phaserSceneFactory` и `simulationSurface`. Граница честности MVP — сервер проверяет итоги схемой и инвариантами, но не пересимулирует; соревновательные режимы требуют нового ADR. Старые `random.seed` effect, guards и effects executor удалены.
- **ADR-063 (Game Asset Channel):** accepted канал игровых ассетов: файлы изображений живут в `games/<id>/assets/` с декларативным реестром `assets.json` (новая JSON Schema, обязательное происхождение/лицензия), runtime-api раздаёт их контент-адресуемо (индекс id→URL + файлы `/game-assets/{gameId}/{assetId}/{sha256}.{ext}` с иммутабельным кэшем) по образцу published-бандлов плагинов; Phaser-сцена получает резолвер `assets` в контексте (по id, не по пути), UI-манифест использует форму `asset:<id>`, SVG — основной агенто-писаемый формат с нормативной CI-санитизацией и защитными заголовками. Существующее размещение картинок «Антарктиды» в `apps/player-web/public/` оформлено долгом LEGACY-0023. Детальный источник — `docs/architecture/game-asset-channel-design.md`. Amended 2026-07-19: жёсткие лимиты (512 КБ/файл, 4 МБ/игра, 64 ассета) заменены не валящими рекомендациями (объективной границы нет), паттерн имён остаётся нормативным, канал расширен типом `css` (ADR-091).
- **ADR-064 (Headless Core And Channel Adapters, supersedes ADR-014):** accepted стратегия клиентского ядра: универсальная интеграционная поверхность платформы — контракты (OpenAPI ADR-051 + JSON Schema ADR-025/056), а не код-библиотека; каталог `SDK/` упразднён, живой код бывшего `SDK/core` (Abstract View Protocol ADR-002 + JSON Merge Patch/JSON Patch) перенесён в `packages/view-protocol`; шов будущего `player-core` (`presenter/`+`lib/` в `player-web` без React/Next-импортов) охраняется CI-проверкой `verify:player-core-seam`; извлечение `packages/player-core` выполняется по именованным триггерам (первый не-React потребитель: headless-клиент агентного игрока ADR-060, Phaser-канал ADR-062, встраивание вне Next.js); внешний публикуемый SDK — отложенная планируемая работа (LEGACY-0037); транспортные клиенты при росте API генерируются из OpenAPI, а не пишутся вручную.
- **ADR-065 (Editor As Product — Hosted Authoring Studio):** accepted целевая модель хостируемой студии — раздельные authoring/delivery-контуры (инвариант ADR-042 §8.1 как граница развёртывания); контент внешних авторов выносится из монорепозитория в Git-репозитории проектов (промежуточная ступень — контентный репозиторий арендатора) с привязкой к `platformReleaseId` (ADR-042), канонические игры остаются CI-фикстурами платформы, хранилищем проектов и опубликованных версий владеет Game Repository; единый вход портала и студии (IdP фиксируется отдельным ADR о доступе), мультиарендность — общий пул с логической изоляцией (`tenant_id`, префиксы путей) и квотами; публикация — неизменяемые контент-адресуемые версии игры по образцу ADR-039/063, публичный runtime читает только опубликованное; LLM-операции — через шлюз с бюджетами арендатора и журналом аудита (ADR-046/047); производительность сессий — только кэш ADR-057 §10 (без отдельного prebuild-механизма).
- **ADR-075 (Durable Author Version History, Accepted):** временная ветка рабочей сессии отделена от долговечной линии авторских версий проекта. «Сохранить» атомарно продвигает подтверждённое состояние в проектную линию, доступную после закрытия/GC и в новой сессии; возврат создаёт новую версию, а публикация остаётся отдельной операцией ADR-065. В локальном монорепозитории новая сессия берёт текущий platform checkout и восстанавливает из истории только allowlisted project paths, поэтому авторская версия не откатывает код платформы. Session-only история отклонена.
- **ADR-076 (Unified Cross-Scope Economic Transfers; amended by ADR-084):** accepted экономические инварианты перенесены в `core.resource.transfer`: дебет, кредит и политика недостатка средств выполняются атомарно над заранее объявленными типизированными конечными точками. Actor разрешается сервером; клиент не передаёт доверенный `playerId` и не конструирует JSON Pointer. Исторические `metric.transfer`, `scope/path/playerRef` и отдельный обработчик удалены вместе со старым effects-контуром.
- **ADR-077 (Serialized Editor Session Mutations, Accepted):** любой пользовательский Save требует действующую editor-сессию, а Save, Restore, Close и разрушительный GC одной сессии выполняются под общей межпроцессной арендой. GC под арендой повторно проверяет Git status, удаление ограничено подтверждённым `.tmp/editor-worktrees` root, а успешное продвижение долговечной версии считается точкой фиксации даже при последующей ошибке вспомогательных метаданных.
- **ADR-078 (Canonical Replay State Fingerprint, Accepted):** replay сравнивает версионированный SHA-256-отпечаток полного серверного `state`, включая скрытые колоды и состояние случайности. Ключи объектов сортируются, порядок массивов сохраняется, а исключаются только `public.log[*].at` и `runtime.lastUpdatedAt`; реальные отметки времени остаются в сессии, одноимённые поля вне этих путей остаются значимыми, изменение политики требует новой версии алгоритма.
- **ADR-066 (Renderer Core And UI Capability Packs, separated by ADR-084):** accepted ось декомпозиции рендерера: ядро (layout, экраны, привязка намерений, выражения, базовые компоненты) + жанровые UI capability packs. Они являются только клиентским представлением и вводом, загружаются по UI-манифесту и не совпадают с build-time mechanics capability packs или trusted runtime algorithm modules. Совместимость версий можно объявить явно, но UI pack не содержит авторитетных правил. Реестр типов компонентов заменяет статический switch вместе с первым реальным пакетом; каналы — ленивые острова, каждый пакет приносит schema module и conformance fixture; триггеры декомпозиции и CI-бюджет `TSK-20260707-player-web-bundle-budget` остаются прежними.
- **ADR-067 (Minimal Service Recovery — single-host, interim):** accepted промежуточный механизм восстановления: однохостовый супервайзер процессов с рестартом по health-проверке (`scripts/ops/service-supervisor.mjs`, без внешних зависимостей) как временная замена оркестратору до реализации задекларированной модели k8s/serverless (PROJECT_OVERVIEW §5, LEGACY-0026). Профиль по умолчанию — прод-контур доставки (runtime-api + player-web; редактор — локальный инструмент, LEGACY-0036); сигналы — существующие `/health`+`/readiness` (runtime-api) и `/` (Next). Восстанавливает процесс; состояние одиночных сессий переживает рестарт благодаря PostgreSQL-контуру ADR-005 (LEGACY-0009 закрыт), разрыв до orchestrator-grade восстановления — LEGACY-0042. Границы (нет HA/rolling-restart/распределённой готовности/восстановления состояния) и триггеры обязательной замены на orchestrator-grade восстановление зафиксированы; разрыв — долг LEGACY-0042. Исполнительная документация — `docs/processes/service-recovery-runbook.md`.
- **ADR-068 (Human-Approved Autonomous Agent Workflow):** accepted опциональный процесс `$cubica`: после его явного выбора человек утверждает общий план корневой `TSK-*` и архитектурные решения, после чего оркестратор автономно владеет неархитектурными решениями, декомпозицией, субагентами, проверкой и итоговой приемкой. Самостоятельные результаты могут получать дочерние `TSK-*` с `Parent` при глубине не более одного уровня; отдельный `PLN-*` не вводится, а временные пакеты оркестрации живут в `.tmp/agent-workflow/`. В обычном игровом режиме общий план не требует отдельного утверждения: PM согласует только существенные архитектурные решения. Старые общие `$cubica` и `wf-*` архивированы и заменены проектным `skills/C_cubica/SKILL.md`. Только `$cubica` требует явного указания пользователя; остальные навыки сохраняют свои обычные правила включения.
- **ADR-069 (Managed External Skill Adaptation):** accepted внешние навыки подключаются только через управляемую проектную адаптацию. Неизменённые предварительно отобранные снимки хранятся в отдельном `skill-candidates/` только как референсы или вход будущей адаптации: их запрещено применять, регистрировать или подключать мостами. Отдельный реестр фиксирует commit, лицензию и контрольные суммы и проверяется CI. Компактный декларативный профиль совместимости с устойчивыми rule IDs, тегами и отпечатками источников дополняется пополняемой памятью ошибок и практик; скрипты лишь собирают механические сигналы и минимальный пакет, а семантическое решение принимает агент с правом точечно читать проект. Upstream проверяется пакетно не чаще раза в 30 дней, кроме ручного запроса или события безопасности, и никогда не заменяет активную версию автоматически. Активной становится материализованная версия без конкурирующих инструкций; `$cubica` остается явно включаемым, остальные допущенные навыки могут включаться автоматически.
- **ADR-070 (Shared Project Skill Catalog):** accepted только активные проектные навыки имеют единое каноническое расположение `skills/`, одинаковое для всех агентов; `skill-candidates/` явно исключён из этой области. Внутренние навыки Cubica именуются `C_*`; адаптированные внешние навыки сохраняют предметные имена и происхождение в реестре. Среды агентов используют мосты обнаружения к активному каталогу, а не собственные копии: для Codex это символические ссылки из `.codex/skills/`. Реестр также декларативно фиксирует обязательные зависимости через `requires`; CI запрещает отсутствующие ссылки, самозависимости, циклы и неожиданные мосты. `$cubica` по-прежнему требует явного включения, остальные допущенные специализированные навыки могут включаться автоматически.
- **ADR-071 (Facilitated Training Sessions):** accepted общая модель учебной сессии отделяет игровые субъекты от персонала. Роли и полномочия объявляются манифестом; переходы фаз, методические точки и интервенции выполняются только ограниченными сервером действиями и журналируются. Отдельный предметный «движок обучения» не создаётся: недостающие селекторы, сценарии и события расширяют общий язык ADR-083 уже с первого доказанного случая. Конкретный состав персонала, рабочие места, вопросы рефлексии и допустимые вмешательства принадлежат игре; хранение персональных наблюдений требует отдельной политики цели, доступа и срока хранения.
- **ADR-072 (Declarative Dynamic Transport Network Board):** accepted общая модель изменяемой транспортной сети. Game-манифест владеет начальным графом, канонической геометрией, выбранной стратегией планирования, состояниями и правилами игры; runtime владеет связностью, создаваемыми узлами/связями, допустимостью операций и атомарными изменениями; UI владеет представлением и вводом намерения. Отдельный старый эффект активации строительства удалён: активация выражается общим запросом объектов, сравнением времени и атомарной командой Mechanics IR, а transport authoring-макрос компилируется в этот язык. Направленность, конкретная задержка, стоимость, финансирование и границы фаз остаются декларациями игры.
- **ADR-073 (Phaser Interactive Board Surface, Accepted):** общий лениво загружаемый Phaser-host расширяется отдельной поверхностью `interactiveBoardSurface` для пошаговых 2D-полей. Объявленный read-only расчёт получает лишь безопасное подмножество параметров, показывает результат в DOM и временной ломаной поверх карты, а подтверждение блокируется, пока расчёт не соответствует текущему выбору. Метод API 2.0 необязателен: новый пакет проверяет его наличие и не падает на старом хосте с API 2.0. Старый обработчик сцены не удовлетворяет инварианту аварийной доступности и сохранён только как помеченный устаревшим резерв до следующей несовместимой версии. Гибридная граница подтверждена PM 2026-07-11, независимый доступный путь — 2026-07-13.
- **ADR-074 (Schema-Declared Resource References, Accepted):** динамический выбор объектов сессии разрешен только через строковые параметры, чья семантика заранее объявлена расширением `x-cubica-ref` в JSON Schema действия. - **ADR-081 (Server-Planned Region-Minimal Roads, Accepted):** для явно включённой транспортной модели runtime получает только два конца и параметры объявленного действия, сам использует версионированный навигационный граф опубликованных областей и минимизирует число непрерывных региональных сегментов. Политика разрешения равных маршрутов является частью версии планировщика; первая версия использует воспроизводимый `session-random`. Read-only расчёт привязан к точной версии состояния и моделирует равный выбор на копии генератора: клиент видит конкретную ломаную и число сегментов, но состояние случайности не меняется и не раскрывается; подтверждение на неизменной версии повторяет тот же путь и атомарно сохраняет результат. Геометрический модуль не вычисляет цену, оплату или жизненный цикл: игровой макрос использует нейтральный результат в общей транзакции. Дорога сохраняет каноническую ломаную, упорядоченные сегменты и версии геометрии/алгоритма; прямые старые связи читаются как ломаная из двух точек. `region-segment-minimum-v1` поддерживает простые неперекрывающиеся полигоны без отверстий, точные общие границы и необязательное исключение областей через объявленный путь состояния; непроверенная или неподдерживаемая геометрия отклоняется без автоматической починки.
- **ADR-082 (Project Knowledge System, Accepted):** `docs/knowledge/` становится каноном проектного смыслового синтеза, но не заменяет полномочия ADR, JSON Schema/OpenAPI, кода, игровых манифестов, задач и `PROJECT_STRUCTURE.yaml`. Страницы, конфигурация и поисковые случаи имеют JSON Schema; Ajv проверяет разобранный YAML-frontmatter. Локальная SQLite-проекция полностью пересобирается и содержит явный wiki-граф, FTS5/BM25 и готовый векторный контур; явный hybrid-запрос и сравнительный `kb eval` доступны сразу, а режим по умолчанию выбирается при приёмке первой версии по измеримому качеству с FTS как начальным резервом. Производная карта модулей, символов, ссылок, реализаций, тестов и происхождения контрактов имеет отдельные пространства идентификаторов, не создаёт смысловые `depends_on`, не редактирует Markdown и соединяется с wiki только через `covers`. Иерархия вызовов вычисляется для выбранного символа по запросу; полная копия AST, постоянный универсальный граф вызовов и глобальный граф потоков данных отклонены. Целевые проверки потоков данных допускаются отдельными запросами специализированного анализатора для конкретных границ доверия. `contradicts` хранит постоянное `open/resolved/deferred` состояние в каноне, а дубли разрешаются через `supersedes` или `distinct_from`; отдельной очереди и акторных Git-трейлеров нет. `generate-structure.js` остаётся единственным производителем карты проекта.
- **ADR-083 (Universal Composable Gameplay Mechanisms, Accepted):** публичный язык игрового движка проектируется универсальным уже с первого подтверждённого игрового сценария. Он может включать нейтральные селекторы, ограниченные структурные узлы, преобразования, агрегации, планирование и атомарные массовые изменения; предметные названия остаются в игре или компилируемых шаблонах. Повторный межигровой случай не является обязательным gate. Если уникальную предметную функциональность нельзя разумно выразить языком или общим модулем, она остаётся в изолированном runtime-расширении конкретной игры, а не попадает game-specific веткой в core. После прямого уточнения PM поэтапная разработка завершается полным переключением без постоянного старого runtime-адаптера.
- **ADR-084 (Game Intents, Capability Catalog And Typed Mechanics IR, Accepted):** внешний изменяющий контракт — Game Intent в терминах игры, но его каноническим полем остаётся существующий `actionId`; отдельный `intentId` не вводится. Определение явно имеет профиль вызова `external` либо `system`; Presenter и публичный dispatcher видят только внешний профиль. Клиент передаёт `actionId + params + cli_ commandId + expectedStateVersion`, principal приходит только из аутентификации, его роль является авторитетной, actor разрешается сервером, а доверенный `playerId` в теле отсутствует. До канонизации текущий конверт ограничен 16 плоскими скалярными параметрами. Command ledger индексируется стабильным principal, поэтому retry безопасен и в хотсите после смены активного игрока. До Mechanics новая команда проходит входной допуск: аутентификацию, структуру, опубликованное действие, роль principal, версию, параметры, живые ссылки, частоту и квоту; такие отказы не получают квитанцию и не могут раздувать ledger. Входной Intent Agent Turn проходит ту же границу и начальные `core.assert` до обращения к модели; выбранное намерение затем исполняется общим dispatcher. После допуска стабильный отказ игрового правила получает отклонённую квитанцию. Presenter использует опубликованные UI-bindings и не выбирает действие из предметного состояния. Неизменяемые action definitions и actor/state-scoped availability являются разными проекциями. ИИ-разработчик и editor используют широкий operation catalog через build-time mechanics packs; UI packs независимы, runtime modules могут быть общими платформенными либо явно разрешёнными изолированными extensions игры. Cubica Mechanics IR `cubica.dev/mechanics/v1alpha1` разделяет чистые query/assert/algorithm, изменяющие command и structural control nodes; состояние, случайность и события меняют только команды. Checker выводит тип, аудиторию и целостность, переносит метку условия на запись и вместе с runtime ограничивает память, строки, литералы, результаты, state, events и audit. Графовые bindings проверяются до публикации; все прочитанные моделью коллекции/endpoints входят в информационный поток, а перекрывающиеся родитель/потомок пути разных аудиторий запрещены. `moduleLock` отдельно закрепляет `moduleId`, точную версию, hash полного исполняемого корпуса и версии значимых алгоритмов; независимые подсистемы используют доменно-разделённые именованные потоки случайности. Session pins immutable `cubica-bundle-v1:sha256:...`: точные канонические байты версионированного конверта хранятся отдельно от JSONB и проверяются без пересериализации текущим канонизатором; новая форма до появления потребует явного version-dispatch. State/random, защищённый ограниченный Mechanics audit, internal receipt, защищённые изменения расписания и полные события с монотонной последовательностью фиксируются одним commit. Курсор полного журнала изменяет только command transaction вместе с соответствующими событиями; preview restore сохраняет его. Публичный игровой журнал является отдельно объявленной проекцией событий в состоянии, а не источником защищённого аудита. Долговечный `command_result` ограничен 64 КиБ, версионирован и не дублирует state/events/secret; Agent Turn отдельно фиксирует trigger и выбранный intent. `system.schedule.register/cancel` закрепляют system intent, параметры, bounded trigger, политику `defer|skip` и не более 64 срабатываний; доверенные identity-поля добавляет исполнитель. Внутренний планировщик выводит `sys_ commandId` из schedule/occurrence, не использует неизвестную заранее версию и повторно проверяет trigger под session lock; префикс не даёт полномочий. Сохранённый trigger и целевой system intent составляют одно Mechanics-исполнение с общими candidate state, runtime budget и защищённым audit; checker применяет тот же единый предел к их суммарной статической стоимости. Проверка живых ссылок цели проходит после истинного trigger и до первого целевого шага, поэтому `defer` остаётся нетерминальным, а `skip` не исполняет цель и получает терминальную квитанцию. `config.runtimeReady:false` блокирует создание новых партий для неполного пакета и сопровождается ограниченными `runtimeBlockers`. Раскрытие `planHash` для учебных данных не считается критичным и не блокирует выпуск, но публичная receipt остаётся отдельной проекцией. После lockstep migration удалены `deterministic.effects[]`, 29 вариантов, old executor/templates, доверенный клиентский `playerId`, `params/payload` duality, client command maps, direct AI effects и постоянные adapters.
- **ADR-085 (Universal Deck, Ordering And Geometry Mechanics, Accepted):** общий язык получает безопасный полный жизненный цикл членов колоды через взаимоисключающие `order/discard/held` и операции `deck.extract/return/insert`; существующий `deck.draw` остаётся совместимым. Транзакционная `core.entities.order` принадлежит отдельному модулю упорядочивания, лексикографически обрабатывает ограниченную выборку по типизированным полям, связанным объектам и ограниченным агрегатам с явной политикой отсутствующих значений; `seeded-random` применяется только внутри группы полного равенства и атомарно изменяет именованный поток. `graph.edge.position.inspect` возвращает проверяемое серверное доказательство геометрии и областей, замкнутая принадлежность включает все области общей границы, `predicate.set.disjoint` сравнивает множества, а `graph.edge.split` сверяет доказательство до изменения. Предметные правила строятся композиционно в игре, а схемы, semantic checker, runtime и нейтральные проверки образуют единую версионированную возможность.
- **ADR-086 (Versioned Mechanics Execution, Accepted):** runtime разрешает закреплённый модуль только по точной тройке `moduleId + moduleVersion + artifactHash`, сохраняет доверенные неизменяемые реализации на период поддержки и хеширует поведение по границам модулей с явными версиями общих зависимостей. Гарантия относится к версиям внутри действующей архитектуры Mechanics и не требует возвращать удалённый старый runtime. Минимальная стабильная схема сначала проверяет конверт, байты, hash и lock, после чего диспетчер выбирает согласованные исторические полную схему, checker и executor; текущий валидатор не имеет права отклонить старую форму до выбора версии. Полный набор проверяется до создания недиспозируемой сессии. PostgreSQL и in-memory store предоставляют одинаковые archive/read-archive операции: архив не принимает команды, но остаётся доступен авторизованному просмотру и аудиту. Автоматическая миграция состояния запрещена.
- **ADR-087 (Finite Number and Read-only Nested Projections, Accepted):** Mechanics различает точный конечный binary64-тип с числовыми границами и `decimal` с фиксированной десятичной шкалой. Логическое поле коллекции может только читать путь длиной до шестнадцати ключей из одного непосредственно сохранённого bounded JSON или record-поля, не хранится и не изменяется отдельно, наследует аудиторию коллекции и проверяется вместе с полным состоянием. Все общие потребители используют один reader и учитывают стоимость пути. Упорядочивание не округляет finite-number и не разрешает его неассоциативное суммирование.

Рендерер подключает объявленные стили game-agnostic способом (`<link>` при монтировании, снятие при размонтировании, порядок после платформенных стилей). Возврат игровых стилей в `globals.css` запрещён автоматической охраной (`findGlobalStyleGameLeaks` в `scripts/ci/validate-game-agnostic.js`: baked-in `url()`, эмодзи, `#id`-селекторы игровых манифестов), включаемой флагом готовности `ENFORCE_GLOBAL_STYLE_BOUNDARY` после переноса стилей (блок R3). Отклонены: общий stylesheet всех игр (PM), CSS-in-JS в плагине, runtime-переписывание URL на сервере, единый массив `assets` с `kind`-объединением (ломает типизацию существующих потребителей). Контракт — блок R2 задачи `TSK-20260719-antarctica-remediation`; механизм и перенос — R3.
- **ADR-092 (Public Metric Deltas On Events, Accepted):** исполнитель транзакции снимает значения всех объявленных публичных метрик до и после транзакции действия и прикладывает к каждому публичному событию (`audience: public`) верхнеуровневый необязательный блок `metricChanges: [{ metricId, before, after }]` в порядке каталога метрик манифеста (`content.data.metrics`, только `kind: state` с `statePath` внутри `public.*`). Прикладывается всегда, когда у игры есть непустой каталог публичных метрик; для `audience != public` — никогда. Несколько публичных событий одной транзакции получают одинаковый блок (дельты всей транзакции, не пошаговые — атомарность ADR-084 §8). Правду об условных эффектах (карточка 61 «Антарктиды»: `time +7`, плюс `+5` если карточка 57 не разыграна) знает только исполнитель, поэтому статичные дельты в authoring отвергнуты; также отвергнуты клиентский пересчёт по снапшотам (нет промежуточных значений, гонки) и отдельный запрос истории метрик (усложняет replay/оффлайн-журнал). Платформа game-agnostic: читает `statePath` из каталога, имён метрик не знает. Размещение — верхнеуровневое поле (не внутри `data`, чей `payloadType` закрыт): аддитивно расширены `core.event-journal-entry` (запись журнала в состоянии, тип `core.bounded-json`), `MechanicsEvent` и `SessionEventRecord`. Существующий авторский флаг `auditMetrics` поглощён контрактом и становится no-op (долг на уборку). Контракт и реализация — блок W2-A задачи `TSK-20260719-antarctica-remediation`.
- **ADR-093 (Design-time объявление раскладки экрана, Accepted):** раскладка страницы (макет канала — `topbar` / `leftsidebar` вокруг одних и тех же игровых данных) — свойство представления, выбираемое разработчиком игры при проектировании UI (design-time), а не runtime-состояние. UI-манифест получает необязательное верхнеуровневое поле `default_layout_mode` (проекция `defaultLayoutMode`, тот же enum, что у `layout_mode` экрана; при отсутствии — `topbar`). Экранный роутер `player-web` сопоставляет условие маршрута `screen_routing[].conditions.layoutMode` с этим design-time значением, а не с серверным `state.public.ui.activeScreen`; итоговую раскладку по-прежнему задаёт `layout_mode` выбранного экрана. Поле `state.public.ui.activeScreen` признано аномалией границы ADR-054 (сервер не должен знать, как клиент верстает те же данные) и снимается у переведённых игр (в срезе — у «Антарктиды»), вместе с проекцией `public.ui`. Форсирование альтернативной раскладки для измерения — подмена одного значения `default_layout_mode` в preview-копии. Отклонены: переключение игроком в runtime (отложено PM, не закрыто навсегда), хранение раскладки в серверном состоянии (нарушает границу, мешает мультиплееру с разными раскладками у игроков), мёртвое поле `activeScreen` (дрейф). Экран `S1_LEFT` остаётся как design-вариант. Контракт и реализация — блок W2-B задачи `TSK-20260719-antarctica-remediation`.

- **ADR-095 (Product Knowledge System Boundaries, Draft):** предлагаются две системы верхнего уровня: принятая инженерная вики и отдельная продуктовая способность. В продуктовой системе диалоги и подтверждаемые операции хранятся в PostgreSQL, а текущая ветка управляемого Git является каноном подтверждённого Markdown. Универсальная область знаний заменяет специальные типы пакетов и задаёт владельца, доверенную зону, доступ и срок; опубликованная игра читается из своего неизменяемого пакета без второго канона. Текущий разговорный агент формирует точную правку, а сервер назначает область, различает реплику агента как формулировку и реплику пользователя как подтверждение и не даёт модели прямой Git-записи. Существующая страница меняется только точными операциями над одним фрагментом; остальные байты и страницы сохраняются, `index.md` генерируется программой, а Git ref переключается сравнением ожидаемой ревизии. Обычный контекст не читает историю; логическое забывание сохраняет её, а физическое стирание переписывает или заменяет репозиторий и очищает контролируемые копии. OKF является целью совместимости, не блокирующим контрактом. В первой версии нет FTS, векторов, графовой базы, фонового смыслового агента или отдельного сервиса. Решение не принято: открыты Git-граница и сроки удаления, автоматическое применение, перенос между ролями, внешняя обработка и единая модель личности.

## 3. Хранилища данных и кэши
 
Архитектура Cubica использует сочетание реляционной базы данных и высокопроизводительного in‑memory‑хранилища для балансировки надёжности и производительности.
 
- **PostgreSQL** — основной источник истины:
  - хранит игровые сессии (`game_sessions`), манифесты и служебные данные;
  - обеспечивает транзакционность (ACID) и гибкую работу с JSONB.
- **Redis** — отложенный резерв только для кэша чтения (Phase 2+; решение PM 2026-07-17):
  - возможный будущий write-through кэш манифестов/проекций; PostgreSQL остаётся единственным источником истины, потеря Redis не должна приводить к потере прогресса игр, режим write-behind отклонён;
  - роль владельца блокировок отменена: блокировки сессий выполняет только PostgreSQL (`SELECT FOR UPDATE NOWAIT`, ADR-005/059), ключи `lock:session:*` из ранних документов не являются целевой архитектурой;
  - внедрение кэша требует именованного триггера (например, измеренная нагрузка чтения хостируемой студии ADR-065) и актуализации `docs/architecture/redis-keys.md` и `docs/architecture/backend/redis-usage.md`, которые до этого остаются отложенными design references.

## 5. Тестирование, наблюдаемость и эксплуатация

Текущая стратегия тестирования задаёт целевой уровень качества для развёртывания сервисов, SDK и игровых пакетов. Детальная политика зафиксирована в `docs/architecture/testing-strategy.md`, а архитектурный выбор — в ADR-038.

Основной подход: сохранить текущие runners (запускатели тестов) по зонам ответственности и добавить единый policy layer (слой правил, который определяет обязательные проверки) поверх них.

- **Static/governance checks** — проверка типов, generated drift, legacy/stub registers, game-agnostic invariants и JSON Schema как source of truth.
- **Unit‑тесты** — проверка бизнес‑логики, валидации данных и вспомогательных утилит.
- **Contract‑тесты** — проверка DTO (Data Transfer Object, объект передачи данных между слоями), manifests, schemas и compiler output между слоями.
- **API contract checks** — OpenAPI validation, endpoint inventory drift checks and representative request/response fixture validation for current `runtime-api` по ADR-051.
- **Интеграционные тесты** — проверка согласованности публичных границ между сервисами, SDK и adapters.
- **Component‑тесты** — проверка поведения React‑компонентов через пользовательские роли, текст и доступные состояния.
- **End‑to‑End‑тесты** — сценарии уровня пользователя через Playwright, runtime-api, player-web и editor-web.
- **Replay/eval‑тесты** — будущий контур для gameplay/LLM behavior, где replay означает повтор записанного сценария, а eval — оценочный тест качества ответа или состояния.

Live LLM, реальные платежи и внешние сети не входят в быстрый PR-гейт; они проверяются через replay, моки, test VPS или отдельные release/nightly checks.

**Наблюдаемость и эксплуатация:**

- Все сервисы должны писать структурированные логи в JSON‑формате с correlation/trace ID для трассировки запросов.
- Ключевые метрики (latency, error rate, использование токенов LLM, попадания в rate limit) экспортируются в формате Prometheus.
- Распределённый трейсинг строится на базе OpenTelemetry и охватывает путь запроса через Router, Engine и View Adapters.
- Стандарты наблюдаемости и лимитирования должны фиксироваться в отдельном ADR и активной рабочей задаче в `docs/tasks/active/`.

## 7. Связь с планированием и ADR

Архитектурные решения и планы реализации синхронизированы с легковесной системой задач ADR-031 и границей полномочий ADR-068:

- `NEXT_STEPS.md` — текущая доска проекта (`Now / Next / Later / Blocked`).
- `docs/tasks/active/` — корневые `TSK-*` с общим планом и необязательные дочерние `TSK-*` для самостоятельных результатов. План утверждается человеком в явно выбранном `$cubica`; в обычном игровом режиме агент сам его формирует и реализует по вводным PM.
- `.tmp/agent-workflow/<task-id>/` — временные задания, отчеты и пакеты проверки субагентов; не источник общего плана.
- `docs/tasks/artifacts/` — постоянные артефакты рабочих задач.
- `docs/tasks/archive/` — архив старой системы `milestones/`, `epics/`, `features/`, `content-packs/`.

ADR‑файлы в `docs/architecture/adrs/` фиксируют архитектурные решения. `PROJECT_ARCHITECTURE.md` является обязательной краткой выжимкой этих решений и должен оставаться согласованным с актуальными ADR, `NEXT_STEPS.md` и активными задачами.

Правило синхронизации:

- любое добавление или изменение ADR должно в том же изменении обновлять `PROJECT_ARCHITECTURE.md`;
- в `PROJECT_ARCHITECTURE.md` должны быть отражены все текущие активные и перспективные ADR, включая `Accepted`, `Proposed` и `Draft`, если они описывают живое или возможное направление архитектуры;
- описание решения должно быть минимально необходимым и достаточным: суть решения, ключевые ограничения, инварианты и последствия, без планов исполнения и без дублирования всего ADR;
- агенты должны проверять актуальность `PROJECT_ARCHITECTURE.md` при изменении статуса, области действия, ограничений или последствий ADR;
- для общего понимания архитектуры достаточно `PROJECT_ARCHITECTURE.md`; отдельные ADR читаются только для дополнительного контекста, альтернатив или глубокого разбора.
