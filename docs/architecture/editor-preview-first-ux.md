## 1. Термины

- **Фасет** — грань единой игровой сущности из одного источника: смысл и содержание из game authoring manifest, вид — из UI authoring manifest конкретного канала (см. ADR-052).
- **CQRS** (Command Query Responsibility Segregation) — разделение модели чтения и модели записи: читаем данные через удобную проекцию, а изменяем через отдельные адресные команды.
- **Проекция** — автоматически построенное представление исходных данных, которое не хранится как самостоятельный источник (например, `EditorEntityProjection` из ADR-052).
- **Промт-проекция** — YAML-представление значимых полей выбранной сущности с русскими подписями по ADR-049; показывается автору как «промт элемента».
- **Возвращённое намерение** — текст, который автор вернул после свободного редактирования промт-проекции; не обязан быть корректным YAML.
- **Интент** — оформленное задание для агента или детерминированного интерпретатора: промт/правка плюс захваченный контекст (цели, указатели чтения и записи, хеши источников).
- **Source map фасетов** — карта соответствия секций проекции исходным файлам и JSON Pointer-ам; скрыта от автора и передаётся интерпретатору вместе с проекцией.
- **Фикстура состояния** — именованный зафиксированный снимок игрового состояния (этап, метрики, объекты), в контексте которого редактор показывает экран в режиме «Дизайн» и от которого можно запустить прохождение.
- **Невизуальная сущность** — игровая сущность без собственного отображения в UI: правило, метрика-механика, таймер, системное действие, скрытое состояние.
- **Декоративный элемент** — UI-элемент без игрового смысла (фон, разделитель, рамка), объявленный таковым в UI authoring манифесте.
- **Аутлайнер** (outliner) — дерево экземпляров, вложенность которого повторяет вложенность отображения (аналог панели Layers в Figma или Hierarchy в Unity).
- **Вхождение** (occurrence) — появление одной и той же сущности в нескольких местах дерева; вхождения ссылаются на одну сущность, а не создают копии.
- **Палитра команд** — быстрый ввод по горячей клавише (обычно Ctrl+K) для перехода к сущности и запуска команд редактора.
- **Approval envelope** — принятый в ADR-047 механизм подтверждения человеком: решение пользователя хранится как отдельный проверяемый объект Cubica, а не как аргумент модели.
- **Worktree** — отдельная рабочая папка Git-репозитория, связанная с общей историей коммитов; изолирует editor-сессию (ADR-036).
- **Dry-run** — пробное применение изменений без записи: проверка, что патч применим и результат проходит валидацию.
- **Debounce** — отложенный запуск действия после паузы во вводе, чтобы тяжёлая операция не выполнялась на каждое нажатие.
- **Hit-test** — определение объекта под координатой курсора или областью выделения в предпросмотре (ADR-036).
- **Инкрементальная пересборка** — обновление производных данных только для изменённых входов вместо полного перестроения (подход инкрементальных компиляторов и query-based систем вроде rust-analyzer).
- **Канальный просмотрщик** — облегчённый отрисовщик канальной проекции (например, Telegram) внутри редактора; показывает структуру и данные проекции, не эмулируя реальный клиент.

## 2. Принятая модель целостной игровой сущности

Игровая сущность (карточка, экран, метрика, действие) физически разложена на
фасеты в разных authoring-манифестах, но автор работает с ней как с целым.
Принято: **целостность — свойство представления, а не хранения**.

1. Единый объект автора — это `EditorEntityProjection` (ADR-052): модель
   чтения, пересобираемая из authoring-источников. Модель записи —
   `EditorChangeSet`, который сам маршрутизирует каждое изменение в правильный
   манифест. Это применение принципа CQRS к редактору.
2. **Дисциплина идентичности.** UI-элемент обязан явно ссылаться на `id`
   игровой сущности; направление ссылки — только UI → game (game manifest не
   знает о каналах). Проекция валидирует связи в обе стороны и превращает
   разрывы в диагностики: «вид без смысла» — для UI-элементов, **не
   объявленных декоративными**, без ссылки на игровую сущность; «смысл без
   вида в активном канале» — **только для типов, требующих отображения**
   (см. [раздел 2.1](#21-невизуальные-и-декоративные-сущности)).
3. **Рефакторинг-операции.** Переименование `id`, удаление сущности,
   извлечение в прототип — атомарные операции `EditorChangeSet`, которые
   обновляют все фасеты и ссылки сразу (аналог Rename Symbol в IDE). Ручная
   синхронизация ссылок между манифестами не является пользовательским путём.
4. **Прозрачность происхождения.** Панель свойств группирует поля по смыслу, а
   не по файлам, но каждое поле несёт ненавязчивый признак источника
   («игра» / «UI · web»). Автор не выбирает файл; маршрутизацию выполняет
   линза проекции.
5. **Канальность вида.** Фасет «вид» существует по каналам (web, telegram,
   ...). Панель сущности содержит переключатель канала; диагностика «нет вида»
   считается по каналу.

### 2.1. Невизуальные и декоративные сущности

Фасет «вид» **необязателен**: сущность может быть чисто логической (правило,
таймер, скрытая метрика, системное действие). Симметрично, UI-элемент может не
иметь игрового смысла, если он декоративный.

- Тип/прототип сущности декларативно объявляет признак «требует отображения»
  (в целом или по каналам) в authoring-схеме или метаданных прототипа.
  Диагностика «смысл без вида» выдаётся только для типов, которые отображение
  требуют; для невизуальных типов её нет.
- UI-элемент может быть объявлен **декоративным** в UI authoring манифесте:
  он не требует ссылки на игровую сущность и не участвует в диагностике «вид
  без смысла». Необъявленный элемент без ссылки остаётся диагностикой.
- Панель сущности для невизуальной сущности просто не содержит фасет «Вид»;
  остальные механизмы (форма, промт-строка, текстовый режим, рефакторинг)
  работают одинаково.
- Каждая сущность, включая невизуальные, достижима через группировку
  «По типам», поиск и палитру команд — это первичные пути навигации для
  невизуальных сущностей.
- Аутлайнер («По экранам») показывает визуальную вложенность, но может
  показывать невизуальные сущности, привязанные к экрану (действия, правила
  области экрана), в свёрнутой подгруппе «Логика экрана» с отличительной
  иконкой — по образцу невизуальных узлов сцены в Godot (Timer,
  AudioStreamPlayer) и компонентов Unity.
- Выделение в предпросмотре достигает только визуальных сущностей; выбор
  невизуальной сущности в дереве может подсвечивать в предпросмотре её
  косвенные визуальные проявления (элементы, на которые ссылается действие),
  если проекция знает связь; иначе подсветки нет.

## 3. Решение по промежуточному editor-manifest

Материализация editor-проекции в отдельный файл, компилируемый из
authoring-манифестов, **отклонена** (подтверждение ADR-052): такой файл
де-факто становится третьим источником истины, отстаёт от источников,
провоцирует ручные и агентские правки и требует собственных drift-проверок.

Файл на диске допустим только в трёх ролях:

- **одноразовый кэш** проекции и производных артефактов — необходимость
  измерена (медленная загрузка), архитектура принята в
  [разделе 10](#10-кэширование-и-производительность-загрузки);
- **диагностический снимок**: команда «экспортировать проекцию» для отладки и
  снапшот-тесты билдера проекции;
- **hints-sidecar** по ADR-052 — только невосстановимые editor-only подсказки.

Отклонена и обратная альтернатива — сделать единый editor-manifest источником,
а game/UI манифесты генерировать из него: это ломает канальную заменяемость UI,
независимое ревью логики и вида, укрупняет диффы и переворачивает цепочку
ADR-030 → ADR-052. Целостность достигается на уровне проекции и UX без платы
за хранение.

## 4. Принятая композиция рабочего пространства

**Состав рабочего пространства** (эталонный макет —
`docs/tasks/artifacts/TSK-20260704-editor-preview-first-ux-implementation/editor-workspace-mockup.html`):

- центр — предпросмотр (почти весь экран), сверху узкая строка меню с
  переключателем режимов, снизу timeline и статусная строка;
- **левый сайдбар**: дерево сущностей с переключателем группировки
  «По экранам» / «По типам» (см. [раздел 7](#7-принятая-структура-дерева-сущностей));
  ниже — секции «Сценарий», «Правила» и «Ассеты». Отдельной секции
  «Библиотека» нет: инвентарь прототипов — это режим «По типам»;
- **правый сайдбар**: вкладки «Чат» (сессионный чат агента), «Журнал»
  (журнал правок сессии), «Проверки» (диагностика);
- плавающая панель сущности — поверх предпросмотра, рядом с выделением.

**Панель сущности (плавающий инспектор).**

- Тело панели — форма параметров, сгруппированная по фасетам
  (Смысл / Содержание / Вид · канал), а не по файлам.
- Строка промта прикреплена к низу панели и растёт в короткий тред диалога с
  агентом; автор видит, как ответ агента меняет поля формы (изменённые поля
  подсвечиваются).
- Текстовый режим «источник» (промт-проекция) доступен за иконкой; его
  семантика принята в [разделе 6](#6-принятый-режим-текстовое-редактирование-через-промт-проекцию).
- Правила плавающести: панель не перекрывает выделенный элемент; закрепление в
  док; Esc закрывает; панель сохраняется при взаимодействии с предпросмотром;
  на узких экранах превращается в нижнюю панель (ADR-034).
- Мультивыделение редактирует общие поля выбранных сущностей. Промт-строка при
  мультивыделении работает по контракту региона: промт + список выбранных
  сущностей, без снимка области; текстовый режим «источник» при
  мультивыделении недоступен (он определён для одиночной сущности).

**Режимы предпросмотра.**

- Два режима: «Дизайн» (редактирование UI) и «Превью» (тестовое прохождение);
  индикация режима визуально громкая (цветная плашка/рамка).
- Политика правок в «Превью»: выделять и инспектировать можно; правка либо
  переключает в «Дизайн» с сохранением контекста состояния, либо применяется в
  authoring с пометкой «отразится после применения к предпросмотру»
  (см. [раздел 9.2](#92-две-оси-времени)). Молчаливая потеря правок
  (проблема Play Mode в Unity) недопустима.
- «Дизайн» всегда работает в явном контексте состояния: выбор этапа timeline
  задаёт экран и фикстуру состояния («проектировать на шаге i12, день 4»).
- Timeline — нижняя сворачиваемая полоса в «Превью»; в «Дизайне» — переключатель
  этапов/экранов (ADR-036).

**Выделение региона.**

- Рамка выделения с растущей строкой промта (ADR-036 §9). Контракт входа
  агента при регионном промте: текст промта, **список сущностей, захваченных
  выделением** (id, указатели, метки, диагностики), и **снимок экрана
  выделенной области**, если renderer adapter поддерживает необязательную
  возможность «снимок области»; при её отсутствии агент получает только список
  сущностей. Регион **не формирует YAML-проекцию**; текстовый режим
  промт-проекции остаётся режимом одиночной сущности.
- Снимок области и контекст сущностей, уходящие внешнему LLM-провайдеру,
  проходят те же гейты redaction/audit, что и остальной agent context
  (ADR-044).
- Чипы контекста у рамки показывают захваченные сущности с возможностью
  исключить лишние; после применения — резюме изменений с кнопкой «Отменить»
  прямо у рамки.

**Чаты агента.**

- Два уровня с явной областью действия: элементный тред (короткоживущий,
  автоконтекст выбранной сущности/региона) и сессионный чат в правом сайдбаре;
  кнопка эскалации переносит элементный тред в сайдбар; история одна, с
  пометками области действия.
- Инвариант: все чаты производят один и тот же конвейер
  `EditorChangeSet → dry-run → validation → undo journal`; независимые
  агентские стеки запрещены.

**Прочее.**

- Палитра команд (Ctrl+K): переход к сущности по `_label`, запуск команд.
- Журнал AI-правок сессии на виду (правый сайдбар): промт → резюме изменений →
  «Отменить» / «Показать diff».
- Статусная строка: worktree-сессия, число несохранённых изменений, счётчики
  диагностик, свежесть предпросмотра
  (см. [раздел 9.6](#96-диагностический-поток-и-сломанная-компиляция)),
  состояние weekly audit прототипов (требование ADR-050).

## 5. Принятые элементы дерева сущностей

В дополнение к структуре из [раздела 7](#7-принятая-структура-дерева-сущностей) приняты:

- поиск и фильтр по `_label`, типу и диагностикам;
- бейджи диагностик на узлах дерева;
- секция «Сценарий»: проекция manifest chronology; этап привязан к игровому
  объекту; развилки показываются бейджами-ссылками (переход, а не вложение),
  текущий этап синхронизирован с «Превью» и timeline; тупики и недостижимые
  этапы подсвечиваются диагностикой;
- секция «Правила»: группировка по системам/триггерам (`root.logic.systems`
  из ADR-036), а не плоский список;
- «Сценарий» и «Правила» не взаимоисключающие: гибридная игра показывает обе
  секции; состав секций определяется содержимым манифеста;
- свободное раскрытие ветвей с запоминанием состояния и auto-reveal выбранной
  в предпросмотре сущности; допустимы эксклюзивно-раскрываемые секции верхнего
  уровня при свободном раскрытии внутри секции;
- выбор узла в дереве синхронизирует выделение в предпросмотре (для визуальных
  сущностей) и открывает панель сущности.

## 6. Принятый режим: текстовое редактирование через промт-проекцию

Принято 2026-07-04. Базовая модель: автор редактирует не YAML как формат, а
«текст, похожий на YAML», созданный из исходной промт-проекции. Возвращённое
намерение никогда не применяется механическим парсингом как данные — это
команда для интерпретации. Режим универсален: один и тот же механизм
редактирует логику, содержание и вид (фасет-агностичность); агент получает
вместе с проекцией source map фасетов и потому знает, какая секция какому
манифесту принадлежит. Модель согласуется с ADR-049 (reverse direction через
агента, который возвращает только `EditorChangeSet`).

Режим определён для **одиночной сущности**. Выделение региона не формирует
YAML-проекцию и идёт по отдельному контракту (см.
[раздел 4](#4-принятая-композиция-рабочего-пространства), «Выделение региона»).

Принятые правила режима:

1. **Вход интерпретатора**: исходная YAML-проекция, возвращённое намерение,
   системная подсказка («пользователь не знает YAML и мог написать что
   угодно»), source map фасетов и хеш источников проекции.
2. **Детерминированный быстрый путь**: если возвращённое намерение парсится
   как YAML и отличается от исходной проекции только значениями известных
   ключей, `EditorChangeSet` строится детерминированно без вызова агента.
   Конвейер безопасности (dry-run → validation → undo journal) одинаков для
   быстрого и агентского пути. Доли путей измеряются телеметрией.
3. **Семантика удалений**: удаление элемента коллекции трактуется как
   намерение удалить; удаление скалярного свойства по умолчанию трактуется
   как «не менять», если контекст не говорит об обратном; все удаления
   обязательно отражаются в резюме.
4. **Построчная отчётность**: каждый изменённый/добавленный/удалённый фрагмент
   попадает в одну из трёх корзин — применено (с указанием поля), распознано
   без изменений, не распознано (показывается автору явно). Молчаливое
   игнорирование ввода запрещено. Формат корзин используется как основа
   eval-фикстур интерпретатора (ADR-038).
5. **Устаревание проекции**: возвращённое намерение несёт хеш источников; при
   несовпадении срабатывает `prompt-stale` (ADR-049) — интерпретатор
   сверяется со свежим состоянием или переспрашивает.
6. **Шкала риска операций**: безопасные (тексты, подписи, стили, значения) —
   стандартный конвейер; структурные (добавление/удаление элементов коллекций,
   переупорядочивание) — акцентированное резюме; опасные (смена `id`,
   перенацеливание ссылок, удаление сущности со входящими ссылками) —
   подтверждение через approval envelope (ADR-047). Смена `id` предлагается
   как рефакторинг-операция «переименовать с обновлением всех ссылок», а не
   как голый запрет. **Шкала применяется к любому агентскому
   `EditorChangeSet` независимо от канала ввода** (текстовый режим, чат
   панели, сессионный чат, регионный промт).

## 7. Принятая структура дерева сущностей

Принято 2026-07-04: одно дерево с переключаемой группировкой (рассмотренная
альтернатива — постоянное дублирование сущностей под группами прототипов —
отклонена из-за конфликта повторяющихся узлов с интуицией «место = вещь» у
нетехнических пользователей).

- **«По экранам»** — аутлайнер: верхний уровень — список всех экранов
  активного канала; активный экран автоматически раскрыт и синхронизирован с
  предпросмотром; вложенность внутри экрана повторяет вложенность отображения;
  невизуальные сущности экрана — в подгруппе «Логика экрана»
  (см. [раздел 2.1](#21-невизуальные-и-декоративные-сущности)).
- **«По типам»** — инвентарь: прототипы → экземпляры с хлебными крошками
  расположения («Экран Маршрут › Карточка»); экземпляр раскрывается в свою
  UI-структуру. Невизуальные типы присутствуют наравне с визуальными.
- Вхождения одной сущности маркируются как «тот же объект»: выбор любого
  вхождения даёт одно выделение, одну панель и мягкую подсветку остальных
  видимых вхождений.
- Правка и удаление всегда действуют на сущность; область действия называется
  явно. Строки прототипов визуально отличаются от строк экземпляров
  (различие «прототип против экземпляра» не должно быть скрытым).
- Выбранный режим группировки запоминается на пользователя.
- Юзабилити-проверка на нетехнических авторах (задачи вида «поменяй текст
  второй карточки», «удали кнопку из карточки», «сделай все карточки крупнее»
  с подсчётом ошибок области действия) сохраняется как валидация реализации,
  но не блокирует решение.

## 8. Следствия для editor-engine

- `TreeViewModel` должен разделить идентичность узла и идентичность сущности:
  одна сущность может иметь несколько вхождений-узлов; связь
  «сущность → указатели» уже даёт `EditorEntityProjection` (ADR-052).
- Нужен конвейер интерпретации возвращённого намерения: вход (исходная
  проекция, текст пользователя, source map фасетов, хеш источников), выход
  (`EditorChangeSet` + построчный отчёт интерпретации), с детерминированным
  быстрым путём перед агентским.
- Нужен слой политики риска операций `EditorChangeSet` (безопасные /
  структурные / опасные) с подключением approval envelope для опасных;
  политика применяется ко всем агентским каналам ввода.
- Нужна очередь интентов с захватом базового состояния и правилом наложения
  (см. [раздел 9.5](#95-параллельность-правок)).
- Renderer adapter contract (ADR-036) расширяется **необязательной**
  возможностью «снимок области» (region snapshot) для регионных промтов;
  отсутствие возможности деградирует контракт до списка сущностей.
- Проекции и производные артефакты должны строиться инкрементально и
  кэшироваться (см. [раздел 10](#10-кэширование-и-производительность-загрузки)):
  линза объявляет читаемые поддеревья указателей, изменения из DocumentStore
  инвалидируют только затронутые сущности.
- Фикстуры состояния получают собственную схему и валидацию
  (см. [раздел 9.3](#93-фикстуры-состояния)).
- Канальные просмотрщики реализуют существующий renderer adapter contract
  (ADR-036), подтверждая его renderer-agnostic статус вторым адаптером.
- Предпосылка всей реализации — модуляризация `editor-engine`
  (`TSK-20260630-editor-engine-modularization`, LEGACY-0018/0019): новый UX не
  должен наращиваться на монолитный `index.ts`.

## 9. Проектные решения по пробелам

### 9.1. Создание и удаление сущностей

Ориентиры: меню «/» в Notion, диалог Add Node в Godot (поиск + недавние),
перетаскивание компонентов из Assets в Figma.

**Создание из прототипа (основной путь).**

- Перетаскивание прототипа из «По типам» на канвас в режиме «Дизайн»: точка
  сброса определяет контейнер (hit-test контейнеров), создаётся экземпляр.
  Альтернатива — пункт «Создать экземпляр» в контекстном меню прототипа.
- Для невизуальных прототипов перетаскивание на канвас недоступно; создание —
  кнопкой «+» в соответствующей секции («Сценарий», «Правила», «Логика
  экрана») или через палитру команд.
- Атомарность: одна операция `EditorChangeSet` создаёт сразу все фасеты —
  запись содержания в game manifest и (для визуальных типов) UI-узел с
  ссылкой на `id` в UI manifest активного канала. Undo убирает всё вместе.
- После создания панель сущности открывается с подсвеченными обязательными
  полями (постепенное раскрытие: сначала только обязательное).

**Создание с нуля.**

- Точки входа: «+» в секциях дерева (контекстно: в «По экранам» — визуальный
  компонент в выбранном контейнере, на верхнем уровне списка экранов — новый
  экран; в «Сценарии» — этап с выбором привязанного объекта), палитра команд
  («Создать: …»), промт агенту.
- «+» в режиме «По типам» создаёт **новый прототип**: пустой по выбранному
  базовому типу или извлечением из выбранной сущности (ADR-050); экземпляры
  создаются перетаскиванием или контекстным меню прототипа.
- Выбор типа — компактное меню с поиском по типам и прототипам (game-level и
  platform-level каталог ADR-050), недавние/частые сверху, невизуальные типы
  помечены. Сущность без прототипа допустима; повторяющиеся структуры позже
  поймает prototype-candidate аудит ADR-050.
- `id` генерируется автоматически: ASCII-слаг из `_label` (транслитерация) с
  суффиксом уникальности; далее `id` стабилен, смена — только
  рефакторинг-операцией.

**Создание агентом.**

- Промт на пустом месте экрана или в чате («добавь панель ресурсов со
  счётчиками еды и топлива») → агент возвращает `EditorChangeSet`, создающий
  несколько связанных сущностей; стандартный конвейер; резюме перечисляет
  созданное; созданные сущности автоматически выделяются в дереве.

**Новая игра.**

- Галерея шаблонов (Game Templates из целевой архитектуры): шаблон — полный
  игровой проект (манифесты, прототипы, ассеты, фикстуры), клонируемый в новый
  проектный репозиторий; дальнейшая адаптация — обычными средствами редактора,
  включая массовые агентские правки («замени тему на ресторанный бизнес»).
- «Пустая игра» — не пустые файлы, а минимальный каркас (meta, один экран,
  один шаг сценария), чтобы предпросмотр всегда что-то показывал.
- Пустые секции дерева показывают приглашение с действием («Нет этапов —
  добавьте первый»).

**Удаление.**

- Всегда на уровне сущности, с диалогом области действия: перечисление
  фасетов и входящих ссылок. При входящих ссылках варианты: отменить /
  удалить и вычистить ссылки (рефакторинг-операция) / перенацелить ссылки на
  другую сущность. До Save потери необратимой нет (журнал сессии).

### 9.2. Две оси времени

У редактора две независимые оси: **история документа** (правки манифестов,
undo/redo) и **состояние предпросмотра** (позиция в прохождении, time travel
по трассе). Ориентиры: урок Unity Play Mode, разделение истории правок и
позиции воспроизведения (playhead) в видеоредакторах, Redux DevTools.

- Разные дома и словарь: «Отменить правку» (Ctrl+Z, меню, журнал в правом
  сайдбаре) против «Вернуться к шагу i12» (только timeline). Слово «откат» не
  используется для обеих осей одновременно.
- **Политика применения правок к предпросмотру:**
  - в «Дизайне» — автоматически: успешная компиляция по debounce обновляет
    предпросмотр в текущей фикстуре;
  - в «Превью» — вручную по умолчанию: идущее прохождение не выдёргивается
    из-под автора; на предпросмотре появляется плашка «Предпросмотр отстаёт от
    правок — Применить» (авто-режим доступен настройкой).
- **Лестница восстановления** при применении правок к идущему прохождению:
  restore на текущий шаг (если состояние совместимо) → ближайший snapshot +
  replay событий → начало текущего шага → перезапуск прохождения (с начала
  или с последней валидной фикстуры). Каждая ступень сообщается простыми
  словами («Состояние несовместимо, вернулись к началу шага»).
- Undo правки — это тоже правка: идёт по той же политике применения; undo
  никогда не трогает трассу напрямую.
- Трасса помечает события хешем контента манифеста на момент записи; события,
  записанные на старой версии, визуально приглушаются на timeline. Линейность
  истории прохождения (ADR-036) сохраняется.

### 9.3. Фикстуры состояния

Фикстура — ответ на открытый вопрос ADR-036 о долгоживущих именованных
сценариях: закреплённые фикстуры становятся **проверяемым проектным
артефактом**, а не tooling-only данными. Ориентир: stories в Storybook
(история = зафиксированное состояние), save-state в игровой разработке.

- **Модель**: `id`, `_label`, привязка к шагу/экрану, снимок состояния,
  хеш контента манифестов на момент снятия, ссылка на исходную трассу,
  заметка автора.
- **Хранение**: закреплённые — `games/<id>/authoring/fixtures/<fixtureId>.json`
  (ревью, версионирование вместе с игрой; собственная JSON Schema);
  незакреплённые авто-снимки — по-прежнему `.tmp/editor-playthroughs/`.
- **Потребители**: runtime-api и player-web **не читают файлы фикстур**;
  состояние закреплённой фикстуры передаёт редактор в preview-only restore
  endpoint (ADR-036, только сессии с временным `contentSourceId`). Второй
  потребитель — будущий replay/eval-контур ADR-038.
- **Создание**: кнопка «Закрепить как фикстуру» на текущем состоянии
  предпросмотра; авто-чекпоинты на границах шагов при любом прохождении, из
  которых можно закрепить нужный.
- **Удержание авто-чекпоинтов**: политика «последние N на сессию» (лимит
  конфигурируем); очистка подключена к GC editor-сессий ADR-042 вместе с
  `.tmp/editor-playthroughs/`.
- **Устаревание**: несовпадение хеша манифеста → диагностика `fixture-stale` и
  бейдж в селекторе состояния; путь починки — переигровка исходной трассы на
  новом манифесте с ревалидацией, иначе ручное повторное снятие.
- **Выбор по умолчанию** в «Дизайне»: закреплённая фикстура экрана →
  авто-чекпоинт → синтетическое начальное состояние. Фикстура хранит полное
  состояние, поэтому одинаково обслуживает визуальные и невизуальные сущности.

### 9.4. Ассеты и медиа

Ориентиры: панель Assets в Figma, Asset Manager в Webflow, выбор вариантов в
генеративных инструментах (Firefly/Midjourney).

- **Библиотека ассетов** — отдельная секция левого сайдбара: сетка с
  превью, поиск, фильтры (тип, использование), загрузка перетаскиванием,
  счётчик использований из проекции («используется в 3 местах»), диагностика
  ассетов-сирот. Ассеты — файлы проекта (`games/<id>/assets/`, ADR-009), а не
  сущности; сущности ссылаются на них полями типа asset-reference.
- **Поле-ссылка на ассет** в панели сущности получает виджет: выбрать из
  библиотеки / загрузить / сгенерировать.
- **Генерация**: промт предзаполняется из контекста сущности (`_label`,
  `_prompt`, дизайн-токены design artifacts ADR-016); результат — полоса из
  нескольких вариантов, автор выбирает; выбранный ассет сохраняется с
  метаданными происхождения (`generation.prompt`), доступны «перегенерировать
  с тем же промтом» и «изменить промт». Генерация — всегда явное действие
  пользователя и проходит операционную политику (стоимость); агент не
  генерирует медиа по собственной инициативе.
- **Интеграция с ChangeSet**: операции с ассетами — это `fileCreates` /
  `fileDeletes` из `EditorChangeSet` (форма уже поддержана ADR-036 §10);
  привязка/отвязка попадает в журнал и undo; бинарники живут в
  session worktree и коммитятся на Save.
- Методические markdown-материалы (`content.methodology`, ADR-012) относятся к
  фасету содержания и редактируются простым markdown-редактором из панели
  сущности игры; библиотека ассетов показывает их как документы.

### 9.5. Параллельность правок

Ориентиры: оптимистичные интерфейсы с согласованием на сервере, семантика
rebase, режим Copilot Edits (просмотр и применение пакета правок).

- **Один писатель**: все изменения сериализуются через один конвейер
  применения (DocumentStore). Ручные правки формы применяются немедленно
  (детерминированно и дёшево); агентские интенты — задания в очереди со
  статусами: ожидает → выполняется → применяется → готово / ошибка / отменён /
  устарел.
- **Оптимистичная конкуренция**: интент захватывает номер журнала и
  **множества указателей чтения и записи** (что агент читал как контекст и
  что собирается менять). При применении — dry-run против актуального
  состояния; конфликт определяется пересечением журнала с момента захвата с
  **объединением множеств чтения и записи**: непересекающиеся правки
  применяются; пересекающиеся помечают интент устаревшим, автор выбирает:
  применить всё равно / показать конфликт / отменить. Это обобщение
  `prompt-stale` на все агентские операции.
- **Без блокировок**: сущность не блокируется на время работы агента (бейдж
  «агент работает с этой сущностью» на панели и в дереве); конфликт решается
  при применении, а не запретом ввода.
- **Отмена в полёте**: у выполняющегося интента есть кнопка отмены; адаптер
  провайдера обязан поддерживать прерывание.
- **MVP-упрощение**: один агентский запуск на сессию одновременно, остальные в
  очереди; параллельные запуски по непересекающимся сущностям — позднейшая
  оптимизация.

### 9.6. Диагностический поток и сломанная компиляция

Ориентиры: панель Problems и code actions в IDE, подход Storybook
«показывать последнее рабочее состояние», оверлеи ошибок dev-серверов.

- **Путь «увидел → перешёл → исправил»**: счётчики в статус-баре открывают
  вкладку «Проверки» правого сайдбара; строка диагностики: серьёзность,
  простое сообщение, метка сущности, бейдж фасета; клик — выбор сущности
  (reveal в дереве, подсветка в предпросмотре для визуальных, панель
  открывается на проблемном поле). Для невизуальных сущностей навигация идёт
  в дерево и панель.
- **Быстрые исправления**: подготовленные ChangeSet по образцу code actions —
  добавить обязательное поле со значением по умолчанию, удалить неизвестный
  ключ, создать недостающий фасет вида, починить висячую ссылку выбором цели;
  плюс «Исправить с помощью агента» (диагностика уходит интентом).
- **Сломанная компиляция не гасит предпросмотр**: показывается последний
  валидный скомпилированный снимок с плашкой «Показана последняя рабочая
  версия — N правок назад; M ошибок блокируют обновление» и переходом к первой
  ошибке. Пустой экран запрещён: preview-first редактор не должен «выключаться»
  от одной ошибки.
- **Свежесть предпросмотра** — индикатор в статус-баре: актуален / отстаёт
  (валидные правки ещё не применены) / заблокирован ошибками.
- Валидация синтаксиса и схем — по debounce при вводе; компиляция — только на
  валидном состоянии; ошибки блокируют компиляцию, предупреждения — нет.

### 9.7. История версий для автора (концепт)

Ориентиры: история версий Google Docs/Figma/Notion; Git-словарь в UI не
используется.

- Представление «История»: точки Save с относительным временем, автором и
  авто-резюме, собранным из журнала сессии («Изменены 3 карточки, добавлен
  экран»). При Save автор видит авто-резюме и может отдельно добавить
  необязательную подпись «Что изменилось?».
- Действия над версией: посмотреть (изменения простыми словами по данным
  журнала), вернуть (новая версия возврата по ADR-036/075, никаких reset),
  позже — создать копию игры от версии.
- Несохранённые правки живут в session worktree и переживают сбой браузера;
  при повторном открытии — плашка восстановления «Есть несохранённые изменения
  из прошлой сессии». Статус-бар: «Сохранено 12 мин назад · 3 несохранённых
  изменения».
- Состояние реализации 2026-07-12: Save продвигает разрешённое содержимое в
  долговечную проектную линию; история переживает закрытие/GC сессии, новая
  сессия использует актуальный код платформы, а панель и безопасный возврат
  реализованы в `TSK-20260710-editor-author-version-history`. ADR-077 принят:
  пользовательский Save требует сессию, а Save/Restore/Close/GC работают под
  общей межпроцессной арендой. Защита и корневая production E2E-приёмка
  завершены 2026-07-12.

### 9.8. Предпросмотр не-Web каналов и размеров экрана (концепт)

Состояние реализации 2026-07-11: Web orientation и structural Telegram viewer
реализованы. Viewer строит сообщения/inline actions/unknown fallback из
Telegram UI authoring document, сохраняет file/pointer и через общий renderer
adapter поддерживает bounds, point/rect hit-test, highlight и переход из
диагностики `entity-missing-view`. Production E2E завершён 2026-07-12.

- Переключатель канала на панели инструментов предпросмотра. Web — текущий
  рендерер. Telegram — **канальный просмотрщик**: облегчённый отрисовщик
  channel-specific Telegram UI authoring/compiled projection в виде ленты
  сообщений и inline-клавиатур, с явной пометкой «структурный просмотр, не
  эмуляция клиента». Обычный источник вида — `ui/telegram.authoring.json`, а
  не генеративная AI-поверхность.
- Framework-neutral `CubicaSurface → Telegram` projection из
  `packages/contracts/ai` используется только для явно выбранной agent surface
  AI-driven игры. Она не является fallback и не заменяет Telegram UI manifest
  deterministic/hybrid игры.
- Просмотрщик реализует renderer adapter contract ADR-036: выделение и
  hit-test работают на элементах сообщений, панель сущности и правки — тем же
  циклом. Это станет вторым адаптером и первым практическим подтверждением
  renderer-agnostic границы.
- Диагностика «нет вида для Telegram» ведёт прямо в этот режим — автор видит
  пробел, а не абстрактное предупреждение.
- Для Web добавляются пресеты viewport (телефон/планшет/десктоп, ориентация)
  по образцу responsive-инструментов браузеров.
- Явно позже: реальное устройство/эмулятор Telegram, Phaser-адаптер (открытый
  вопрос ADR-036).

## 10. Кэширование и производительность загрузки

Проблема измерена: каждая загрузка редактора строит проекции заново независимо
от наличия изменений. Принята трёхуровневая схема; ориентиры — инкрементальные
компиляторы и query-based memoization (rust-analyzer/Salsa), кэши сборщиков.

**Уровень 1 — инкрементальная пересборка в памяти (главный).**

- Все правки идут через DocumentStore/JSON Patch, поэтому редактор точно знает
  изменённые указатели. Линза проекции объявляет читаемые поддеревья
  указателей; обратный индекс «указатель → сущности» инвалидирует только
  затронутые сущности и связанные производные (граф, дерево, YAML-проекция).
- Полная пересборка — только при изменении схем, словаря полей или версии линз.

**Уровень 2 — постоянный кэш на диске (тёплый старт).**

- Расположение: `.tmp/editor-cache/<projectId>/`; вне Git; удаление всегда
  безопасно.
- Ключи: по-файловые артефакты (parse tree, location map, результаты
  валидации) — хеш содержимого файла; проектные артефакты
  (`EditorEntityProjection`, межфайловый индекс ссылок, граф) — сводный хеш
  **всех входов проекции по ADR-052**: authoring-файлы, схемы и их аннотации,
  словарь полей, compiler source maps, метаданные design artifacts, метаданные
  проектных плагинов — плюс версия editor-engine/линз и `cacheFormatVersion`.
  Вход, не учтённый в ключе, — это ошибка проектирования кэша: он обязан
  инвалидировать проекцию.
- Открытие сессии: быстрый отсев по mtime/размеру, затем сверка хешей;
  попадание — загрузка артефактов, промах — пересборка только промахнувшихся
  уровней. Запись — отложенная (после простоя).
- Кэш не зависит от worktree: ключ по содержимому, поэтому сессии и ветки
  делят общие записи.

**Уровень 3 — кэш компиляции и валидаторов.**

- Результаты manifest compiler (generated runtime manifests + source maps)
  кэшируются по хешу authoring-входов — повторная компиляция без изменений не
  выполняется.
- Скомпилированные Ajv-валидаторы переиспользуются в памяти по хешу схемы
  (перекомпиляция схем на каждый прогон запрещена).

**Инварианты и гигиена.**

- Кэш одноразовый: никогда не источник истины, никогда не вход компиляции или
  runtime; любое несовпадение/повреждение → молчаливая пересборка.
- Версионирование `cacheFormatVersion` + версия editor-engine; сборка мусора —
  лимит размера/LRU, подключена к GC сессий ADR-042.
- Телеметрия: hit/miss, длительности пересборок — до и после внедрения.
- **Предостережение**: кэш не должен маскировать алгоритмические проблемы
  монолитного `index.ts`; профилирование входит в задачу модуляризации
  (`TSK-20260630-editor-engine-modularization`), и уровень 1 проектируется на
  модульных границах, а не поверх монолита.
- UX загрузки: оболочка редактора открывается сразу (скелет), проекция
  подгружается по готовности; блокирующий белый экран запрещён.
