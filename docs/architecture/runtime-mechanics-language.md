# Runtime Mechanics Language

Этот документ описывает минимальный псевдоязык механик: JSON-формат для игровых правил, который должен покрывать типовые квестовые, стратегические, карточные и обучающие сценарии без добавления кода конкретной игры в `runtime-api`.

Псевдоязык механик не является полноценным языком программирования. Это небольшая договоренность о том, как в манифесте описывать действие, условие, эффект и запись в журнал.

## 1. Назначение

Цель языка - дать авторам игр и редактору общий способ описывать серверные игровые правила как данные.

Он нужен для таких задач:

- разрешить или запретить действие игрока;
- изменить метрику;
- поставить флаг на карточке, персонаже, клетке, задании или ресурсе;
- перейти к другой сцене;
- добавить запись в журнал;
- открыть следующий элемент после выполнения условия;
- применить небольшой набор проверяемых изменений состояния.

Главная граница: если правило можно выразить этим языком, нельзя добавлять для него код, привязанный к конкретной игре, в `runtime-api`.

## 2. Основные правила

1. JSON Schema является единственным источником истины для структуры правила.
2. Правило читает только переданный снимок состояния и payload действия (данные, пришедшие вместе с действием игрока).
3. Правило возвращает эффекты, а не меняет состояние напрямую.
4. `runtime-api` валидирует эффекты перед применением.
5. Все изменения состояния должны быть понятны из манифеста.
6. Действие должно быть детерминированным: одинаковый вход дает одинаковый результат.
7. Нет доступа к файловой системе, сети, переменным окружения, текущему времени и случайности без отдельной общей возможности платформы.
8. Названия операций должны быть общими, а не привязанными к конкретной игре.

## 3. Модель выполнения

Обработка действия идет так:

1. `runtime-api` находит action definition в game manifest.
2. Если действие использует template, параметры подставляются в template.
3. Payload проверяется по schema, когда schema задана.
4. `guard` или `when` проверяет, можно ли выполнить действие.
5. Если guard не прошел, действие отклоняется с диагностикой.
6. Runtime вычисляет список эффектов.
7. Эффекты валидируются по JSON Schema и по разрешенным write-prefixes.
8. Эффекты применяются атомарно.
9. В журнал пишется результат с `actionId`, типами эффектов и диагностикой.

Атомарно означает: либо применяются все разрешенные изменения, либо состояние не меняется.

## 4. Минимальная форма правила

Целевая форма должна быть близка к такой структуре:

```json
{
  "mechanicsVersion": "1.0",
  "action": "game.team.member.select",
  "when": {
    "jsonLogic": {
      "<": [{ "var": "public.teamSelection.pickCount" }, 3]
    }
  },
  "effects": [
    {
      "op": "flag.set",
      "path": "/public/flags/team/member-1/selected",
      "value": true
    },
    {
      "op": "counter.add",
      "path": "/public/teamSelection/pickCount",
      "delta": 1
    },
    {
      "op": "collection.append",
      "path": "/public/teamSelection/selectedMemberIds",
      "value": "member-1"
    },
    {
      "op": "log.append",
      "kind": "choice",
      "summary": "Selected team member"
    }
  ]
}
```

На 2026-05-31 этот каркас является действующим контрактом для deterministic-действий: изменения состояния, метрик, timeline и журнала описываются через `effects[]`, а JSON Schema остается единственным источником правил валидации.

## 5. Доступ к данным

Для ссылок на состояние используется JSON Pointer-совместимый путь, например:

- `/public/metrics/trust`;
- `/public/timeline/stepIndex`;
- `/public/flags/cards/card-12/resolved`;
- `/secret/opening/selectedCardId`.

Права на чтение и запись должны различаться:

- `public` можно читать и писать только через разрешенные эффекты;
- `secret` можно читать только если это явно разрешено конкретной механикой;
- `runtime` служит для служебной диагностики и не должен использоваться как игровая модель;
- payload действия читается как отдельный источник, например `payload.cardId`.

Если новое правило требует писать в новый раздел состояния, этот раздел должен быть описан в JSON Schema и документации.

## 6. Условия

Условия должны быть выражены как данные. Базовый вариант - JsonLogic, потому что он уже используется в runtime-коде и хорошо подходит для простых проверок:

```json
{
  "and": [
    { "==": [{ "var": "public.flags.cards.card-12.available" }, true] },
    { "<": [{ "var": "public.metrics.time" }, 60] }
  ]
}
```

Допускаются короткие доменные формы, если они остаются общими:

```json
{
  "collection": {
    "path": "/public/flags/cards",
    "where": { "field": "resolved", "equals": true },
    "countAtLeast": 3
  }
}
```

Такая форма должна быть синтаксическим удобством, а не скрытым обработчиком под конкретную игру.

## 7. Эффекты

Эффект - это проверяемая операция, которую runtime может применить к состоянию или журналу.

Начальный набор кандидатов:

| Операция | Смысл | Пример применения |
| --- | --- | --- |
| `state.patch` | Применить ограниченный JSON Patch-подобный набор операций | точечное изменение состояния |
| `metric.add` | Изменить числовую метрику | доверие +1, время -2 |
| `counter.add` | Изменить счетчик | количество выбранных участников +1 |
| `flag.set` | Установить булев или небольшой структурный флаг | карточка решена, участник выбран |
| `collection.append` | Добавить значение в список | выбранный участник добавлен в список |
| `collection.remove` | Удалить значение из списка | ресурс потрачен |
| `timeline.set` | Изменить текущую сцену, экран или шаг | перейти к следующему инфо-экрану |
| `log.append` | Добавить запись в журнал | выбор игрока или итог действия |
| `event.emit` | Создать доменное событие для последующей обработки | future integration/event flow |

Не все операции нужно реализовывать сразу. На 2026-05-31 реализованы проверяемые эффекты `runtime.server.request`, `ui.panel.open`, `ui.screen.open`, `log.append`, `timeline.set`, `state.patch`, `flag.set`, `counter.add`, `collection.append` и `metric.add`.

Для условного эффекта используется поле `when`. Условие может смотреть на метрику, состояние или количество элементов коллекции. По умолчанию условие читает текущее состояние после предыдущих effects. Если нужно сохранить старую семантику "проверить до действия", используется `readFrom: "preAction"`.

## 8. Добавление новой возможности

Новая возможность платформы добавляется только после короткой классификации.

Возможность считается общей, если она подходит нескольким классам игр. Примеры:

- "открыть элемент после N завершенных элементов";
- "перейти на следующую сцену";
- "увеличить ресурс";
- "выбрать до N участников";
- "поставить флаг на объекте".

Возможность считается привязанной к конкретной игре, если она зависит от конкретного мира, персонажа, сюжетного шага или имени игры. Такая логика не должна попадать в `runtime-api`.

Для общей возможности обязательны:

- JSON Schema;
- краткая документация;
- runtime handler без проверки `gameId`;
- тесты с нейтральными fixture-данными;
- запись в журнале;
- явный список разрешенных путей записи.

## 9. Как это ложится на Antarctica

Текущий manifest `games/antarctica/game.manifest.json` уже близок к этому подходу:

- 145 actions;
- 145 actions с `handlerType: "manifest-data"`;
- 4 action templates;
- neutral capability families: `game.card.resolve`, `game.timeline.advance`, `game.info.advance`, `game.team.select`, `game.team.confirm`, `game.collection.threshold`, `runtime.server`, `ui.panel`, `ui.screen`;
- UI/runtime commands use `effects[]` for `runtime.server.request`, `ui.panel.open`, `ui.screen.open` and `log.append`;
- timeline, flags, team selection, board thresholds and selected card state use `effects[]`;
- `content.scripts` does not point to a runtime script;
- guards для карточек, команды, timeline и board thresholds;
- metrics, условные изменения метрик and journal entries use `effects[]`.

Проблема не в том, что `Antarctica` использует манифест. После cleanup 2026-05-31 runtime-script markers закрыты: schema/runtime принимают только `effects[]` для deterministic-изменений.

Целевое направление для `Antarctica`:

1. Сохранить action IDs, если они уже используются UI manifest.
2. Сохранять нейтральные capability families, например `game.card.resolve`, `game.timeline.advance`, `game.team.select`, `game.collection.threshold`.
3. Оставить `content.data.infos`, `boards`, `teamSelections`, `cards` в игровом манифесте, потому что это данные конкретной игры, а не runtime-код.
4. Все deterministic-изменения состояния описывать общими эффектами.
5. Не переносить ни одно правило `Antarctica` в общий `runtime-api` как отдельную ветку.

Подробная миграционная документация: `docs/tasks/artifacts/TSK-20260527-editor-engine-preview-timeline-editor/antarctica-plugin-migration.md`.

## 10. Чего не делаем сейчас

- Не строим полноценный workflow engine.
- Не добавляем произвольный JavaScript в manifest.
- Не разрешаем npm-зависимости плагинов до отдельной проверки зависимостей.
- Не делаем marketplace runtime-плагины без песочницы.
- Не вводим внешние сетевые вызовы в обработку действия игрока.
- Не переписываем весь `Antarctica` manifest ради красивой абстракции.
