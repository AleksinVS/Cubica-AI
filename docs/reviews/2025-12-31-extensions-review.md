Давай разберем, почему стоит оставить сборку и пакеты расширений (Extension Packs) для Game Engine, даже если мы возвращаем Runtime Sandbox для пользовательских скриптов.

Почему Engine Extensions нужны при наличии User Scripts?
Главная идея: Разделение ответственности.

User Scripts (Runtime Sandbox): Отвечают за контент и сценарий конкретной игры.
Engine Extensions (Build-time): Отвечают за возможности (Capabilities) и системы движка.
Если мы откажемся от расширений движка и оставим только "Монолитный движок + Скрипты", мы получим "God Object" (супер-движок), в который придется запихнуть вообще всё: и RPG-механики, и интеграцию с Google Sheets, и физику, и сложные AI-агенты. Это сделает движок тяжелым и сложным в поддержке.

Предлагаемая архитектура: "Слоеный пирог"
Я предлагаю следующую модель, которая объединяет твое желание использовать Runtime Scripts и концепцию Extension Packs:

1. Слой Контента (Runtime / Sandbox)
Что это: game.manifest.json + .js файлы скриптов.
Где живет: В репозитории игры (games/<id>/).
Как работает: Загружается движком динамически при старте сессии.
Исполнение: Внутри isolated-vm (Sandbox).
Ограничения: Строгие лимиты по памяти/CPU, доступ только к разрешенным API (state, log, math).
Пример: "Если игрок нажал кнопку, увеличить score на 10 и проверить условие победы".
2. Слой Возможностей (Build-time / Extensions)
Что это: NPM-пакеты (@cubica/ext-rpg-system, @cubica/ext-telegram-adapter).
Где живет: В package.json самого экземпляра движка.
Как работает: Компилируется вместе с ядром движка (Engine Core) в Docker-образ.
Исполнение: Нативный Node.js (без Sandbox).
Возможности:
Доступ к БД, Redis, внешним API.
Тяжелые вычисления (pathfinding, physics).
Регистрация новых нативных функций, которые становятся доступны в Sandbox.
Пример: Расширение "Inventory" добавляет в Sandbox функцию inventory.addItem(), которая под капотом делает сложные проверки веса, слотов и пишет в БД.
3. Слой Ядра (Engine Core)
Что это: Базовый код (services/game-engine).
Задачи: Загрузка расширений, управление Sandbox, цикл игры (Game Loop), маршрутизация событий.
Сравнительная таблица
Характеристика	User Scripts (Runtime)	Engine Extensions (Build-time)
Роль	Логика конкретной игры (квесты, диалоги)	Переиспользуемые механики (инвентарь, боевка)
Изменение	Мгновенное (обновил файл -> работает)	Медленное (CI/CD, пересборка Docker-образа)
Безопасность	Высокая (Sandbox, isolated-vm)	Доверенная среда (Code Review, SAST)
Производительность	Средняя (оверхед на изоляцию)	Максимальная (Native Node.js)
Доступ к API	Ограниченный (Whitelist)	Полный (FS, Network, DB)
Как это меняет ADR-015?
В текущем 015-extension-packs-architecture.md предполагалось, что вся игра собирается в Build-time. Твое предложение делает архитектуру гибче.

Нам нужно скорректировать ADR-015 следующим образом:

Viewers: Остаются полностью на Build-time Composition. (Тут мы согласны).
Engine: Переходит на Hybrid Composition:
Engine Instance собирается из Core + Extensions (Build-time). Это создает "Специализированный Движок" (например, "RPG Engine" или "Quest Engine").
Game Logic загружается в этот движок как данные (Runtime).
Пример сценария (User Story)
Разработчик хочет сделать RPG.
Он выбирает Base Engine и подключает расширение @cubica/ext-rpg-mechanics (содержит классы, формулы урона, инвентарь).
CI собирает Docker-образ my-rpg-engine:v1.
Разработчик пишет скрипт в манифесте:
javascript
// User Script (Runtime)
// Функция attack() пришла из расширения ext-rpg-mechanics!
const damage = rpg.attack(player, enemy);
if (damage > 0) {
   game.log("Удар!");
}
Движок выполняет скрипт. Когда скрипт вызывает rpg.attack, управление переходит из Sandbox в нативный код расширения, где быстро считается математика.
Резюме
Стоит ли оставлять пакеты расширения для Game Engine? Да, однозначно.

Они станут мостом между безопасным, но ограниченным миром пользовательских скриптов и мощным, но опасным миром Node.js. Это позволит разработчикам собирать свои "идеальные движки" из кубиков, а поверх них писать легкую скриптовую логику.
