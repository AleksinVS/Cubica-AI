## 1. Контекст и цель

Цель этого документа:

- зафиксировать фактическое состояние архитектуры репозитория Cubica;
- выделить слабые места текущего проекта;
- предложить практичную структуру графа знаний (knowledge graph — граф сущностей и связей между ними) для проекта;
- предложить локальный MCP-сервер (Model Context Protocol server — локальный сервер, который предоставляет LLM доступ к структурированным ресурсам, инструментам и контексту проекта).

Ревью основано на анализе:

- фактической структуры репозитория;
- корневых и пакетных `package.json`;
- архитектурных документов;
- ADR;
- текущего кода `services/router`, `SDK/core`, `SDK/react-sdk`;
- существующих тестов и признаков заглушек / черновиков / архивных артефактов.

## 2. Краткий вывод

Проект хорошо проработан на уровне целевой архитектуры и архитектурной документации, но находится в состоянии сильного разрыва между:

- **target architecture** — тем, как система должна выглядеть;
- **actual implementation** — тем, что реально реализовано;
- **draft / archive** — тем, что сохранено как исторический или переходный слой.

Для Cubica локальный MCP-сервер действительно имеет высокий потенциал, но только если он будет строиться не как простой полнотекстовый поиск по репозиторию, а как **knowledge graph с явной маркировкой статуса артефактов**:

- `actual`
- `target`
- `draft`
- `archive`
- `placeholder`

Иначе MCP-сервер будет регулярно возвращать смешанный и противоречивый контекст.

## 3. Ключевые проблемы проекта

### 3.1. Разрыв между документацией и фактическим репозиторием

Документация системно описывает более зрелый монорепозиторий, чем реально существует в коде.

Примеры:

- корневой workspace указывает на `games/antarctica-nextjs-player`, но фактический живой плеер расположен в `draft/antarctica-nextjs-player`;
- `PROJECT_STRUCTURE.md` и `PROJECT_OVERVIEW.md` местами описывают состояние, которого в кодовой базе уже нет или ещё нет;
- обзорная архитектура говорит о рабочем пайплайне сервисов, но по факту большинство сервисов находятся на уровне каркаса.

Риск:

- разработчики и ИИ-агенты получают ложную картину зрелости системы;
- knowledge graph без дополнительной нормализации будет считать intent и implementation равноправными источниками истины.

### 3.2. Слабая исполнимость монорепозитория

На уровне npm/workspaces репозиторий не выглядит как целостно исполнимая система:

- в корне нет `test`-скрипта;
- пакетные `build`, `test`, `lint` во многих пакетах являются `TODO`;
- отсутствует единый build/test/lint pipeline;
- отсутствует единая конфигурация TypeScript / test runner / lint runner для активных пакетов.

Риск:

- архитектура формально описана, но не может быть непрерывно проверена;
- MCP-серверу трудно определять, что считается реально валидным и проверенным кодом.

### 3.3. Backend в основном декларативный

Почти все backend-сервисы в `services/*` пока представлены `DEV_GUIDE.md` и пустыми каталогами `src/`/`tests/`.

Исключение:

- `services/router/src/sessionEvents.ts`
- `services/router/src/sessionRecovery.ts`

Но даже там пока зафиксированы в основном интерфейсы, а не production-реализация.

Риск:

- проект сейчас больше напоминает архитектурный каркас, чем работающую платформу;
- многие знания о системе находятся только в docs, а не в code/runtime.

### 3.4. Нестабильность контракта SDK

В `SDK/core` и `SDK/react-sdk` видны важные архитектурные заготовки, но не законченные контракты:

- `createSession.connect()` всегда выбрасывает placeholder error;
- `routerClient` жёстко завязан на `/submit`;
- `fetchState()` инициирует `StartGame`, то есть чтение и мутация смешаны;
- `useCubicaSession()` зависит от объекта `options` целиком и легко вызывает лишние переподключения.

Риск:

- SDK пока не является надёжным основанием для других клиентов;
- знание-граф должен маркировать эти узлы как `placeholder` / `partial`.

### 3.5. Иллюзия тестового покрытия

Часть файлов в `tests/` фактически не являются нормальными unit/integration tests:

- используются self-executing smoke-примеры;
- нет явного тестового раннера для Router;
- в React SDK есть placeholder test `expect(true).toBe(true)`.

Риск:

- наличие каталога `tests/` создаёт видимость качества без реального regression shield;
- graph-based assistant будет переоценивать надёжность частей системы.

### 3.6. Слишком много слоёв `draft` / `legacy` / `placeholder`

В проекте большое количество:

- архивных материалов;
- черновых клиентов;
- исторических сценариев;
- placeholder-реализаций;
- зарегистрированных и незарегистрированных заглушек.

Это само по себе не ошибка, но без строгой классификации артефактов становится проблемой.

Риск:

- любой интеллектуальный индексатор без модели жизненного цикла артефактов начнёт возвращать шум;
- ответы MCP-сервера будут путать референсную историю проекта с актуальным состоянием.

## 4. Варианты улучшения проекта

### 4.1. Ввести машиночитаемый слой истины по репозиторию

Нужен единый файл уровня репозитория, например:

- `repo-manifest.json`
- или `docs/architecture/repo-index.yaml`

В нём для каждого крупного артефакта должны быть поля:

- `id`
- `kind`
- `path`
- `status`
- `owner`
- `source_of_truth`
- `depends_on`
- `adr_refs`
- `task_refs`
- `last_verified_at`

Это даст фундамент для knowledge graph и сильно упростит работу локального MCP.

### 4.2. Отделить actual architecture от target architecture

Сейчас эти уровни смешаны в документации.

Нужно явно разделить:

- **Actual**: что реализовано и проверено сейчас;
- **Target**: что запланировано в ADR / roadmap;
- **Draft**: переходные или экспериментальные компоненты;
- **Archive**: исторические материалы;
- **Deprecated**: то, что не должно использоваться дальше.

### 4.3. Сделать один минимальный рабочий vertical slice

До масштабного роста архитектуры лучше собрать один вертикальный срез:

- одна игра;
- один Router API;
- один SDK client;
- один manifest validator;
- один сценарий запуска;
- один test pipeline.

Это даст настоящую опорную систему координат для графа знаний.

### 4.4. Нормализовать API-контракты

Нужно разделить команды и запросы:

- `startSession`
- `getSessionState`
- `dispatchAction`
- `subscribeSessionEvents`

Сейчас `fetchState -> StartGame` размывает границы и усложняет индексацию семантики API.

### 4.5. Заменить декларативные тесты на исполнимые

Минимальный уровень:

- единый тестовый раннер;
- реальные unit tests для `SDK/core`, `SDK/react-sdk`, `services/router`;
- smoke/integration tests как отдельный слой, а не как псевдо-unit tests;
- CI-команда, которая реально падает при нарушении контракта.

Для Cubica нужен не один плоский граф, а **многослойный knowledge graph**.

- способ извлечения (`static`, `schema`, `manual`, `git`, `inferred`).

## 7. Как организовать локальный MCP-сервер

### 7.1. Общий подход

Локальный MCP-сервер должен работать поверх нескольких источников:

- файловая система репозитория;
- git metadata;
- парсинг кода;
- парсинг docs / ADR / roadmap;
- парсинг manifest/schema/design artifacts;
- локальный графовый индекс;
- при необходимости отдельный векторный индекс для семантического поиска.

### 7.2. Практичный стек

Для Cubica на первом этапе я бы рекомендовал:

- **MCP server runtime**: Python `FastMCP` или TypeScript SDK;
- **code parsing**: `tree-sitter` + для TS/JS по возможности `SCIP`;
- **graph storage**: начать с SQLite/PostgreSQL, а не сразу с отдельной graph DB;
- **vector layer**: отдельный индекс только для docs / scenario / methodology / review docs;
- **incremental indexing**: по изменённым файлам через `git diff` и file hash.

### 7.3. Почему не начинать сразу с “большой graph DB”

Для текущего состояния Cubica основная проблема не масштаб графа, а качество и классификация знаний.

Сначала полезнее:

- ввести устойчивую схему;
- научиться извлекать узлы и связи;
- маркировать статус артефактов;
- обеспечить provenance.

После этого уже можно решать, нужен ли Neo4j / AGE / другой graph engine.

### 7.4. Базовые MCP resources

- `resource://repo/overview`
- `resource://repo/index`
- `resource://architecture/current`
- `resource://architecture/target`
- `resource://game/{game_id}/manifest`
- `resource://doc/{doc_id}`
- `resource://adr/{adr_id}`
- `resource://file/{path}`
- `resource://graph/node/{id}`

### 7.5. Базовые MCP tools

- `search_code`
- `search_docs`
- `get_dependencies`
- `get_source_of_truth`
- `trace_feature_to_code`
- `trace_game_action`
- `find_drift`
- `list_unverified_artifacts`
- `explain_component`
- `rebuild_index`

### 7.6. Самый полезный инструмент для Cubica

Для этого проекта особенно важен tool `find_drift`.

Он должен находить расхождения между:

- `PROJECT_STRUCTURE.md`
- `PROJECT_OVERVIEW.md`
- `docs/architecture/PROJECT_ARCHITECTURE.md`
- workspace config
- реальной файловой системой
- task / roadmap документами

Именно эта возможность даст реальную ценность команде.

## 8. Best practices для Cubica Knowledge Graph

### 8.1. Не смешивать intent и implementation

Если документ что-то планирует, а код этого не реализует, в графе это должны быть разные узлы или разные статусы, а не одна “истина”.

### 8.2. Не полагаться только на embeddings

Для архитектурных вопросов в Cubica важнее:

- точные связи;
- provenance;
- source-of-truth semantics;
- связь docs <-> code <-> tasks.

Векторный поиск полезен как вспомогательный слой, но не как основной источник знания.

### 8.3. Индексировать не только код

Для Cubica критично индексировать:

- ADR;
- roadmap;
- review docs;
- debt log;
- game manifests;
- UI manifests;
- design artifacts;
- schema files.

Именно в этих файлах содержится значимая часть архитектурной истины проекта.

### 8.4. Инкрементальная индексация обязательна

Полный rebuild графа на каждом изменении быстро станет дорогим и замедлит рабочий цикл.

### 8.5. У каждого ответа MCP должен быть provenance

Ответы MCP-сервера должны содержать:

- откуда взято знание;
- какой статус у артефакта;
- когда это знание было проверено;
- является ли это прямым фактом или выводом.

## 9. Практический roadmap внедрения

### Этап 0. Нормализация источников истины

- исправить workspace / путь к активному плееру;
- ввести машиночитаемый индекс репозитория;
- размечать `actual / target / draft / archive / placeholder`.

### Этап 1. Индексация структуры проекта

- репозиторий;
- каталоги;
- package manifests;
- docs;
- ADR;
- tasks;
- игры и манифесты.

### Этап 2. Индексация кода

- файлы;
- импорты / экспорты;
- символы;
- интерфейсы;
- API-контракты;
- тесты.

### Этап 3. Drift detection

- сравнение документации и файловой системы;
- сравнение roadmap и факта;
- сравнение manifest/schema/code integration.

### Этап 4. Traceability

- `task -> ADR -> doc -> code -> test`
- `game manifest -> action -> UI binding -> router contract -> sdk adapter`

### Этап 5. Семантический слой

- embeddings только для длинных текстов и поисковых сценариев;
- hybrid retrieval: graph + lexical + semantic.

## 10. Сравнение вариантов реализации

Ниже приведено сравнение трёх практических вариантов для Cubica:

1. собственный MCP-сервер + SQLite/PostgreSQL как graph store;
2. собственный MCP-сервер + Neo4j;
3. использование RAGex как основы или как опорного решения.

### 10.1. Собственный MCP + SQLite/PostgreSQL

Это наиболее прагматичный стартовый вариант для Cubica.

**Плюсы:**

- полный контроль над моделью знаний;
- удобно моделировать проектные статусы `actual / target / draft / archive / placeholder`;
- просто хранить provenance и evidence;
- легко индексировать не только код, но и ADR, roadmap, debt-log, manifests, design artifacts;
- невысокая инфраструктурная сложность;
- проще локальная эксплуатация и отладка.

**Минусы:**

- больше собственной разработки;
- сложные graph traversal-запросы придётся проектировать самостоятельно;
- визуализация графа не будет “из коробки”.

**Вывод:**

Для текущего состояния Cubica это лучший базовый вариант.

### 10.2. Собственный MCP + Neo4j

Этот вариант имеет смысл, если графовые запросы и обходы связей станут центральной частью инженерного процесса.

**Плюсы:**

- удобная нативная модель графа;
- сильные возможности для traversal и exploratory analysis;
- удобно строить длинные цепочки связей вроде `task -> ADR -> doc -> code -> test -> manifest`;
- хорошие возможности визуализации и анализа связности.

**Минусы:**

- выше инфраструктурная сложность;
- на раннем этапе это может быть преждевременным усложнением;
- основная текущая проблема Cubica не в storage engine, а в качестве и нормализации источников истины.

**Вывод:**

Хороший следующий шаг после стабилизации схемы и индексатора, но не лучший первый шаг.

### 10.3. RAGex

RAGex интересен как code-centric MCP/RAG слой для поиска по кодовой базе.

**Сильные стороны:**

- гибридный поиск: semantic + symbolic + regex;
- хорошая ориентация на code understanding;
- MCP-совместимая модель;
- полезен как быстрый кодовый retrieval layer для ИИ-ассистентов.

**Ограничения для Cubica:**

- по публичному описанию сильнее ориентирован на код, чем на проектный knowledge graph;
- Cubica требует покрытия не только кода, но и документов, ADR, roadmap, debt-log, manifests, UI manifests, design artifacts;
- нужен слой source-of-truth semantics, который различает intent и implementation;
- дополнительный риск: ранняя зрелость, Elixir-стек и GPL-3.0.

**Вывод:**

Для Cubica RAGex интересен как:

- референс по архитектурным идеям;
- возможный sidecar для code search;
- прототипный ускоритель для code retrieval.

Но не выглядит оптимальной основой для полного графа знаний проекта.

### 10.4. Итоговая рекомендация

Для Cubica оптимальная стратегия выглядит так:

- **сейчас**: собственный MCP + SQLite/PostgreSQL;
- **позже**: добавить отдельный vector layer;
- **ещё позже**: при необходимости мигрировать на Neo4j или добавить его как аналитический слой;
- **RAGex**: использовать как источник идей или как дополнительный code-search слой, но не как основу общего knowledge graph проекта.

## 11. Проверка репозитория

Во время ревью была проведена быстрая практическая проверка:

- `npm test` в корне не запускается, так как корневого `test`-скрипта нет;
- `npm run bootstrap` отработал успешно;
- в репозитории много `TODO`, `placeholder`, `draft`, `stub`, `mock` артефактов;
- фактический код сосредоточен главным образом в `SDK/*`, `services/router/*`, `games/antarctica/*`, `draft/antarctica-nextjs-player/*`.

Это подтверждает, что в текущем состоянии Cubica knowledge graph должен уметь различать:

- что реально исполнимо сейчас;
- что является контрактом;
- что является черновиком;
- что является архивом.

## Заключение

Для Cubica локальный MCP-сервер стоит строить.

Но строить его нужно не как “чат над репозиторием”, а как **слой структурированной инженерной правды**:

- с графом сущностей и связей;
- с provenance;
- с выделением source of truth;
- с обязательным различением `actual`, `target`, `draft`, `archive`, `placeholder`.

Именно в такой конфигурации MCP действительно сможет давать актуальные знания о проекте и кодовой базе, а не просто пересказывать смешанный контекст из документов и черновиков.
