## 1. Контекст и цель

- Проверена текущая реализация Next.js-плеера, который рендерит игру Antarctica из JSON-файлов в каталоге `draft/game-player-nextjs`.
- Сравнение выполнено с принципами MVP и LLM-first архитектуры из `PROJECT_OVERVIEW.md` и `docs/architecture/PROJECT_ARCHITECTURE.md`. Особое внимание уделено требованиям к Presenter, Router и контрактам JSON-манифестов.

## 2. Основные находки

1. **Жёстко зашитые сетевые параметры и обход Router.**  
   `fetchServerData` и `requestServer` используют статический IP/токен, ходят напрямую в движок и вставляют `<link>` в DOM (`src/app/utils/serverDataLoader.js`, `src/app/utils/actions.js`). Это нарушает требования API-first и делает клиент привязанным к одному окружению.

2. **Отсутствие Presenter и смешение ролей View/Model.**  
   Компоненты `GameButton`, `GameCard`, `GameVariable` напрямую импортируют `actionHandlers`, которые меняют состояние и вызывают сеть. View знает про все источники данных, что противоречит MVP-архитектуре.

3. **Состояние теряет структуру манифеста.**  
   `loadLocalData` и `GameScreenRenderer` сохраняют только первый дочерний узел `application`, поэтому метаданные (`meta`, `assets`, `application.cssFile`) недоступны и невозможно строить LLM-промпты или методические материалы.

4. **Жёсткая привязка к локальным JSON-файлам.**  
   Все действия подменяют состояние импортированными фикстурами (`screen_*`). Это увеличивает bundle и не позволяет загружать новые экраны без пересборки. Dev API (`src/app/api/route.js`) дублирует эту логику и светит данные наружу.

5. **Документация и тесты отсутствуют.**  
   README написан на английском, что нарушает правило репозитория. Нет автотестов для рендера и редьюсера, что повышает риск регрессий.

## 3. Вопросы к команде

1. Планируется ли миграция на `SDK/react-sdk` с использованием стандартизованного `useCubicaSession`, или текущий плеер должен эволюционировать самостоятельно?  
2. Требуется ли поддерживать устаревшие схемы `cssInline`, или можно нормализовать манифест и упростить рендер?

## 4. Рекомендации по доработке

1. **Вынести сетевую логику в Presenter/хуки и использовать Router API.**  
   Добавить конфигурацию через env, хранить sessionId, применять PATCH/полные состояния через `applyStateUpdates` и прекратить прямой доступ к движку.

2. **Разделить ответственность View и Presenter.**  
   Пусть компоненты генерируют только `ViewCommand`, а Presenter (или контекст) решает, какую фикстуру или сетевое действие выполнять. Это позволит удалить импорты JSON из UI и соблюсти MVP.

3. **Хранить полный манифест в состоянии.**  
   Сохранять `application`, `meta`, `assets` и `training`, чтобы зрело работать с CSS, подсказками, методичками и целями обучения.

4. **Абстрагировать источник данных.**  
   Создать адаптер, который умеет работать с локальными файлами и Router через единый интерфейс, вместо импорта JSON в bundle.

5. **Привести документацию и тесты к стандарту.**  
   Переписать README на русском, добавить smoke-тесты рендера (Jest + React Testing Library) и юнит-тесты для `applyStateUpdates`/редьюсера, как требует `docs/tasks/README.md`.

## 5. Соответствие целевой архитектуре

- Текущий плеер реализует только простую View-часть и не использует Router, Presenter или JSON-манифест целиком.  
- Отставание от целевой архитектуры оценивается как **уровень прототипа**: хватает для демо, но не для MVP.  
- Для выравнивания необходимо подключить слои SDK (`SDK/core`, `SDK/react-sdk`), интегрироваться с Router, хранить полный манифест и реализовать Presenter, который управляет состоянием и вызовами LLM через движок.
