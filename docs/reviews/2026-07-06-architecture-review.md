# Архитектурное ревью 2026-07-06: белые пятна и несоответствия

Ревью архитектурной документации платформы (не кода) по трём осям: белые пятна
без ADR/дизайна/исполнительной программы, решения, требующие согласования, и
направления улучшения. Документ — источник записей долга `LEGACY-0024..0036`
в `docs/legacy/debt-log.csv` (по образцу ревью 2026-06-27 → LEGACY-0018..0022).

Терминология: «белое пятно» — блок целевой архитектуры, у которого нет ни
архитектурного решения (ADR/дизайн-документа), ни исполнительной программы
(`TSK-*`); «дрейф» — расхождение документации с фактическим состоянием.

## 1. Итог

Ядро платформы (runtime, манифесты, редактор, Agent UI, плагины) проработано
плотно: 60+ ADR с работающей дисциплиной синхронизации
(`ADR ↔ PROJECT_ARCHITECTURE ↔ NEXT_STEPS`). Белые пятна концентрируются в трёх
зонах: production-обвязка (аутентификация, наблюдаемость, развёртывание),
бизнес-слой (лицензии, платежи, каталог, публикация) и реальный LLM-слой
(провайдер-адаптер, replay/eval). Главный структурный дрейф: слоистая
SDK-стратегия из `PROJECT_OVERVIEW.md` фактически заменена plugin-архитектурой
(ADR-037/039/055), но документально это не закрыто.

## 2. Исправлено в этом же изменении

Дешёвые несоответствия, устранённые вместе с этим ревью:

1. `docs/architecture/router-sequence.md` и `docs/architecture/router-ws-protocol.md`
   получили статус-пометку «future extraction reference»: они описывают
   отдельный Router, тогда как ADR-051 фиксирует API First на текущем
   монолите, а ADR-059 (Proposed) сажает WebSocket прямо в `runtime-api`.
2. Раздел «Клиентские каналы» в `PROJECT_OVERVIEW.md` приведён к реальности:
   канонический канал — `apps/player-web` + плагины, SDK-слой помечен как
   историческое направление со ссылкой на долг LEGACY-0031.
3. `LEGACY-0023` (active в `debt-log.csv`) отсутствовал в таблице «Текущие
   заглушки» `stubs-register.md`, что ломало `validate-legacy` — строка
   добавлена.
4. Черновики `game-ui-mockup-flow.md` и `hybrid-knowledge-architecture-v0.5.md`
   закоммичены с явными статус-пометками и привязаны к долгам
   LEGACY-0034/LEGACY-0033 (до этого существовали только в рабочем дереве и
   были невидимы для системы синхронизации).
5. Закоммичено уточнение правил происхождения бинарных ассетов в
   `rail-tycoon-mini-design.md` §8.0, на которое уже ссылался закоммиченный
   ADR-063.

## 3. Белые пятна (зарегистрированы как долги)

### 3.1. Production-обвязка

- **LEGACY-0024 — наблюдаемость и лимитирование.** `PROJECT_ARCHITECTURE.md` §5
  прямо требует отдельного ADR и активной задачи для стандартов логов,
  метрик, трейсинга и rate limiting — ни того, ни другого нет.
- **LEGACY-0025 — аутентификация и авторизация.** OAuth/IDP и RBAC упомянуты
  как принципы; единственное реальное решение — auth-gate внешнего agent
  backend (ADR-047). Платформенной модели (роли автор/консультант/игрок/
  ведущий/админ, auth endpoints `runtime-api`, связь с join-токенами ADR-059)
  нет. Блокер публичного test VPS launch.
- **LEGACY-0026 — развёртывание.** Build pipeline, Docker-образы на игру,
  Kubernetes/Serverless (`PROJECT_OVERVIEW.md` §5) — декларация без ADR;
  единственная исполнительная точка — test VPS задача портала.

### 3.2. Бизнес-слой

- **LEGACY-0027 — лицензирование и платежи.** Типы лицензий существуют только
  внутри ADR-033 о session binding; модели лицензирования, ценообразования и
  платёжной интеграции нет (payment stub — LEGACY-0013).
- **LEGACY-0028 — каталог и публикация.** Жизненный цикл публикации игры
  (валидация → версия → каталог → покупка) не спроектирован; проработана
  только нижняя часть (published plugin bundles, ADR-039) и поиск
  (`docs/architecture/search/qdrant.md`). Сервисы — scaffold (LEGACY-0004);
  туда же примыкает аналитика Metadata DB (модель событий не описана).

### 3.3. Реальный AI-слой

- **LEGACY-0029 — LLM provider adapter.** Весь AI-driven контур (ADR-046/047)
  принят и реализован против mock-адаптера (LEGACY-0001). Нет архитектуры
  реального адаптера: выбор провайдера, ретраи, стоимость/токены, persisted
  audit store.
- **LEGACY-0030 — replay/eval runner.** Контракты и фикстуры реализованы, но
  сам контур — «будущий» без исполнительной программы, при том что ADR-046
  делает replay/eval обязательным гейтом production AI-driven rollout.

### 3.4. Каналы и продукты

- **LEGACY-0035 — Telegram/Mobile.** Для Telegram есть только Surface
  projection contracts, архитектуры бота/view adapter нет; Mobile не
  проработан вовсе (в отличие от Phaser, получившего ADR-062).
- **LEGACY-0036 — хостируемый редактор.** `apps/editor-web` спроектирован как
  локальный инструмент (Git worktree-сессии, локальные пути); архитектуры
  многопользовательского хостируемого редактора нет, при том что «редактор с
  LLM-помощником» — витринный сценарий №1.

### 3.5. Статусные и документационные разрывы

- **LEGACY-0031 — судьба SDK-слоя.** `SDK/react-sdk`+`shared` мертвы
  (LEGACY-0021), `viewers` не реализован (LEGACY-0005, ADR-014 Proposed),
  роль занята plugin-архитектурой. Нужно явное решение: Superseded или
  возрождение.
- **LEGACY-0032 — ADR-030 в статусе Draft** при частично реализованном
  authoring-слое (v2-схемы, компилятор, миграция Antarctica): статус решения
  отстаёт от реальности.
- **LEGACY-0033 — hybrid-knowledge-architecture-v0.5.md.** Зрелый черновик
  системы управления знаниями лежит в `docs/architecture/`, но не связан с
  ADR-системой Cubica и ссылается на чужую нумерацию решений; нужна
  классификация (связать с треком либо вынести из канонического каталога).
- **LEGACY-0034 — game-ui-mockup-flow.md.** Черновик потока «текст → макет →
  прототип UI» по собственному тексту требует оформления в ADR — решения и
  задачи на это нет.

## 4. Решения, ожидающие согласования

Не долги, а очередь согласования владельцем проекта (видна в `NEXT_STEPS.md`):

> Обновление 2026-07-06 (после ревью): пункты 1 и 2 закрыты — владелец принял
> ADR-058/059/060 и ADR-063 (Proposed → Accepted 2026-07-06).

1. **ADR-058/059/060 (Proposed)** — блокируют треки настольных игр,
   мультиплеера, ИИ-оппонентов и «Монополию»; Phase 1 ADR-058 (PRNG) — ещё и
   flow-simulation Phase 2 и rail-tycoon. Критический путь.
2. **ADR-063 (Proposed)** — канал ассетов; блокирует ассеты rail-tycoon и
   миграцию LEGACY-0023.
3. **Межтрековая координация** — четыре параллельных трека сходятся на
   `PhaserSceneContext`, `httpServer.ts` и PRNG-модуле; сейчас координация
   держится только на конвенции Handoff Log. Рекомендация: короткая карта
   очерёдности срезов (добавлена в `NEXT_STEPS.md` → Later).
4. **Хвосты миграций Antarctica** — object-state migration
   (reopened-closeout) и LEGACY-0023: по правилу 9 CLAUDE.md закрытие
   задокументированных разрывов имеет высокий приоритет.

## 5. Рекомендованные приоритеты проработки

1. Принять пакет ADR-058/059/060 + ADR-063 (или явно отклонить части) и
   зафиксировать межтрековую карту очерёдности.
2. Минимальный production-пакет под test VPS: ADR authn/authz (LEGACY-0025),
   ADR наблюдаемости (LEGACY-0024), модель развёртывания test VPS
   (LEGACY-0026).
3. Закрыть статусный дрейф: решение по SDK/viewers (LEGACY-0031), статус
   ADR-030 (LEGACY-0032), классификация черновиков (LEGACY-0033/0034).
4. Спроектировать «второй этаж» AI-контура: провайдер-адаптер + audit store
   (LEGACY-0029), исполнительная программа replay/eval (LEGACY-0030) — без
   них гейты ADR-046/047 непроходимы в принципе.
5. Бизнес-слой по мере приближения к каталогу: лицензии/платежи (LEGACY-0027,
   нужен уже порталу), затем жизненный цикл публикации (LEGACY-0028).

## 6. Сильные стороны (сохранить как есть)

- Дисциплина «ADR ↔ PROJECT_ARCHITECTURE ↔ NEXT_STEPS» реально работает
  (ADR-063 синхронизирован во все три места в одном изменении).
- Реестр долга живой и проверяется CI (`validate-legacy`).
- Нормативные дизайн-документы для агентов-исполнителей (например,
  `game-asset-channel-design.md` §3) — лучшая защита от дрейфа при агентной
  разработке.
