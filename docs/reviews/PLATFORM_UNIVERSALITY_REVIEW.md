# Ревью платформы Cubica-AI: Универсальность, Game-Agnostic и AI-First

## 1. Исполнительное резюме (Executive Summary)

Платформа Cubica-AI представляет собой современную высокоуровневую систему для разработки и проведения интерактивных бизнес-игр. Архитектура построена на принципах **декларативности** и **разделения ответственности**, что является отличным фундаментом для AI-driven разработки. 

Текущая реализация (на примере игры "Antarctica") демонстрирует работающий вертикальный срез, где логика игры полностью отделена от движка и описывается через JSON-манифесты.

---

## 2. Оценка по ключевым критериям

### 2.1. Универсальность (Universality) — **Оценка: Высокая**
*   **Плюсы:**
    *   **Hybrid SDUI (Server-Driven UI):** Использование UI-манифестов позволяет описывать интерфейс в декларативном виде, что обеспечивает кросс-платформенность (Web, Telegram, Mobile) без изменения логики.
    *   **Многоканальность:** Архитектура изначально закладывает поддержку разных фронтендов через SDK-адаптеры.
    *   **Разделение Logic/UI (ADR-013):** Позволяет менять визуальное представление игры независимо от её механики.
*   **Минусы:**
    *   Сложность описания специфических игровых механик (например, сложные 2D/3D взаимодействия) в рамках текущих UI-компонентов.

### 2.2. Game-Agnostic (Независимость от игры) — **Оценка: Средняя**
*   **Плюсы:**
    *   **Deterministic Runtime:** Движок работает как конечный автомат (State Machine), обрабатывающий действия из манифеста, не зная деталей сюжета.
    *   **Пакеты контрактов:** Общие типы данных в `packages/contracts` обеспечивают единство протоколов.
*   **Минусы:**
    *   **Утечка контекста (Context Leakage):** В текущих тестах и структуре манифеста ("Antarctica") наблюдается жесткая привязка к ключу `content.antarctica`. Это нарушает принцип "Purity over game-specific hacks" (Rule 9 в `AGENTS.md`).
    *   **Хардкод метрик:** Хотя метрики динамические, их семантика никак не описана метаданными, что затрудняет создание игры с принципиально другим набором параметров (например, инвентарь вместо числовых метрик).

### 2.3. AI-First (Ориентированность на ИИ) — **Оценка: Высокая**
*   **Плюсы:**
    *   **Семантические артефакты (ADR-016):** Использование `design.json` с описанием "визуального намерения" позволяет LLM-агентам "видеть" и понимать дизайн без анализа пикселей.
    *   **JSON-First:** Манифесты являются идеальной формой ввода-вывода для LLM.
    *   **System Prompts в манифесте:** Возможность задавать роль и правила "движка" прямо в описании игры.
*   **Минусы:**
    *   **Избыточность манифеста:** Манифест "Antarctica" занимает более 9000 строк. Для LLM это слишком большой контекст, что ведет к деградации внимания и ошибкам при генерации обновлений.

## 3. Рекомендации по повышению универсальности

### Шаг 1: Стандартизация структуры данных (Agnostic Content)
*   **Что сделать:** Уйти от ключа `content.game_id` к универсальному `content.collections` или `content.data`.
*   **Зачем:** Движок не должен знать имя игры для доступа к её данным. Это позволит использовать одни и те же запросы к API для разных игр.

### Шаг 2: Внедрение Action Templates (Macros)
*   **Что сделать:** Вместо описания каждого `ранний формат изменения состояния` и `изменение метрики` для каждой кнопки, внедрить шаблоны действий.
    *   *Было:* 50 строк JSON для перехода на следующий шаг.
    *   *Стало:* `handler: "templates.advance_step"`.
*   **Зачем:** Сократит размер манифеста в 5-10 раз, сделает его читаемым для ИИ-агентов и снизит риск ошибок в логике переходов.

### Шаг 3: Мета-описание метрик и состояния
*   **Что сделать:** Добавить в манифест секцию `schema.state`, где описываются типы данных (число, строка, список, объект) и их ограничения.
*   **Зачем:** Агенты смогут автоматически генерировать UI для новых метрик (например, прогресс-бары для чисел или списки для инвентаря) на основе их типа, а не хардкода.

### Шаг 4: Дифференциальные манифесты (Patching)
*   **Что сделать:** Разработать механизм наложения "слоев" на базовый шаблон игры.
*   **Зачем:** Позволит агентам создавать новую игру "Mars Corp" путем отправки небольшого JSON-патча к игре "Antarctica Corp", заменяя только тексты и ассеты, не переписывая всю логику.

### Шаг 5: Протокол обратной связи (Agent Feedback Loop)
*   **Что сделать:** Добавить в `runtime-api` режим валидации манифеста, который возвращает не просто "error", а семантический отчет для ИИ (например: "Действие X ссылается на несуществующий screenId Y").
*   **Зачем:** Позволит агентам самообучаться и исправлять свои ошибки в логике игры в автономном режиме.

## 4. Конкретные шаги по реализации (Roadmap)

1.  **[Refactoring]** Переименовать `content.antarctica` в `content.data` во всех манифестах и тестах `services/runtime-api`.
2.  **[Architecture]** Создать ADR-026: "Action Templates for Compact Manifests".
3.  **[Tooling]** Написать скрипт-валидатор, который проверяет логическую целостность манифеста (ссылки на ID, достижимость экранов).
4.  **[SDK]** Добавить в `SDK/core` поддержку "Generic Collection Renderer", который может отобразить любой список данных из `content.data` без написания нового кода.
