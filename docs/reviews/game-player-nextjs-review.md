## 1. Краткое резюме

**Текущее состояние:** `game-player-nextjs` — это экспериментальный Next.js-плеер игр, реализованный как standalone-приложение без использования платформенных SDK (`SDK/core`, `SDK/react-sdk`). Проект демонстрирует базовую функциональность рендеринга игровых экранов из JSON-манифестов, но не соответствует архитектурным принципам платформы.

**Оценка соответствия:** ⚠️ **Частичное соответствие (30-40%)**

**Основные проблемы:**
- Отсутствие интеграции с SDK (`SDK/core`, `SDK/react-sdk`)
- Нарушение паттерна MVP (нет разделения Model/View/Presenter)
- Прямое обращение к серверу без использования Router API
- Отсутствие поддержки протокола ViewCommand/ViewResponse
- Хардкод конфигурации (URL, токены)
- Отсутствие тестов и документации

## 2. Оценка соответствия целевой архитектуре

### 2.1. Соответствие архитектурным принципам

| Принцип | Статус | Комментарий |
|---------|--------|-------------|
| **MVP паттерн** | ❌ Не соблюдается | Нет разделения Model/View/Presenter. Вся логика смешана в компонентах и утилитах. |
| **LLM-first архитектура** | ⚠️ Частично | Используются JSON-манифесты, но нет интеграции с Game Engine через Router. |
| **SDK-слой** | ❌ Не используется | Проект полностью обходит `SDK/core` и `SDK/react-sdk`, реализуя собственную логику. |
| **Протокол ViewCommand/ViewResponse** | ❌ Не реализован | Взаимодействие с сервером происходит через прямые fetch-запросы, а не через абстрактный протокол. |
| **Модульность** | ⚠️ Частично | Код разделён на компоненты и утилиты, но нет переиспользования через SDK/shared. |
| **Версионирование контрактов** | ❌ Отсутствует | Нет проверки версий схем манифестов (`schema_version`, `min_engine_version`). |

### 2.2. Соответствие структуре SDK

**Ожидаемая архитектура (согласно `PROJECT_ARCHITECTURE.md`):**

```
SDK/
├── core/          # Контракты, транспорт, управление сессиями
├── shared/        # UI-компоненты и утилиты
└── react-sdk/     # React-адаптер с хуками useCubicaSession
```

**Текущая реализация:**

```
game-player-nextjs/
├── src/app/
│   ├── components/        # Собственные компоненты (дублируют SDK/shared)
│   ├── utils/             # Собственная логика сессий и действий
│   └── data/              # Локальные фикстуры
```

**Проблемы:**
- Компоненты (`GameButton`, `GameCard`, `GameVariable` и др.) должны быть в `SDK/shared` для переиспользования
- Логика сессий (`GameScreenRenderer`, `serverDataLoader`) должна использовать `SDK/react-sdk` и `SDK/core`
- Утилиты рендеринга и действий должны быть частью Presenter-слоя, а не View

### 2.3. Соответствие протоколам взаимодействия

**Ожидаемый протокол (ADR-002, `docs/architecture/protocols/mvp-interaction.md`):**

- Presenter → View: через `ViewCommand` (абстрактные команды)
- View → Presenter: через `ViewResponse` (результаты выполнения)
- Транспорт: через Router API с поддержкой HTTP/WebSocket

**Текущая реализация:**

- Прямые fetch-запросы к хардкод URL (`http://188.120.246.243:3000/submit`)
- Собственная система действий (`actionTypes`, `actionHandlers`) без связи с протоколом
- Нет поддержки WebSocket для real-time обновлений

## 3. Детальный анализ компонентов

### 3.1. Компоненты рендеринга

**Файлы:** `src/app/utils/renderer.js`, `src/app/components/gameComponents/*`

**Проблемы:**
- ❌ Компоненты (`GameButton`, `GameCard`, `GameVariable`) дублируют функциональность, которая должна быть в `SDK/shared`
- ❌ Рендерер (`renderer.js`) напрямую маппит типы компонентов, вместо использования абстрактного протокола
- ⚠️ Поддержка legacy (`cssInline`) и нового (`css`) формата — хорошо, но должна быть частью SDK
- ❌ Избыточное логирование (`console.log`) в production-коде

**Рекомендации:**
- Перенести компоненты в `SDK/shared` с возможностью кастомизации через темы
- Использовать `ViewCommand` для управления рендерингом через Presenter
- Убрать логирование или использовать структурированный логгер

### 3.2. Управление состоянием

**Файлы:** `src/app/components/GameScreenRenderer.js`, `src/app/utils/applyStateUpdates.js`

**Проблемы:**
- ❌ Собственный reducer вместо использования `SDK/react-sdk` (`useCubicaSession`)
- ❌ Логика загрузки данных смешана с компонентом (нарушение MVP)
- ⚠️ `applyStateUpdates` — хорошая реализация deep merge, но должна быть частью SDK/core
- ❌ Нет обработки ошибок и retry-логики (должна быть в SDK/core)

**Рекомендации:**
- Использовать `useCubicaSession` из `SDK/react-sdk` для управления сессиями
- Вынести логику загрузки в Presenter-слой
- Интегрировать `applyStateUpdates` в SDK/core как утилиту для патчей состояния

### 3.3. Загрузка данных

**Файлы:** `src/app/utils/serverDataLoader.js`, `src/app/utils/localDataLoader.js`

**Проблемы:**
- ❌ Хардкод URL и токена (`http://188.120.246.243:3000/submit`, `'hiiamhardcodedtoken'`)
- ❌ Прямой fetch вместо использования Router Client из `SDK/core`
- ❌ Нет поддержки WebSocket для real-time обновлений
- ❌ Динамическая инъекция CSS через `document.head.appendChild` — небезопасно и не соответствует Next.js best practices
- ⚠️ Локальные фикстуры — хороший подход для разработки, но должен быть частью SDK

**Рекомендации:**
- Использовать `createRouterClient` из `SDK/react-sdk/adapters/routerClient`
- Вынести конфигурацию в environment variables (`NEXT_PUBLIC_ROUTER_URL`, `NEXT_PUBLIC_SESSION_TOKEN`)
- Реализовать поддержку WebSocket через `SDK/core` transport layer
- Использовать Next.js `<Head>` или CSS-in-JS для стилей

### 3.4. Система действий

**Файлы:** `src/app/utils/actions.js`

**Проблемы:**
- ❌ Собственная система действий вместо использования `ViewCommand`/`ViewResponse`
- ❌ Хардкод фикстур внутри обработчиков (`screen2`, `screen_s1`, `journal`)
- ❌ Нет типизации и валидации действий
- ❌ Обработчики напрямую манипулируют состоянием через `dispatchAppState`, нарушая MVP

**Рекомендации:**
- Перейти на протокол `ViewCommand`/`ViewResponse` из `SDK/core`
- Вынести логику действий в Presenter-слой
- Использовать типизированные контракты для действий

### 3.5. API Route (заглушка)

**Файлы:** `src/app/api/route.js`

**Проблемы:**
- ⚠️ Заглушка для разработки — нормально для MVP, но должна быть помечена как legacy
- ❌ Нет документации о том, что это временное решение
- ❌ Не соответствует реальному API Router

**Рекомендации:**
- Зарегистрировать как заглушку в `docs/legacy/stubs-register.md`
- Добавить комментарии о временном характере
- Подготовить план миграции на реальный Router API

## 4. Критические проблемы

### 4.1. Архитектурные

1. **Отсутствие интеграции с SDK**
   - **Влияние:** Высокое. Проект не может переиспользовать общие компоненты и контракты.
   - **Приоритет:** Критический
   - **Решение:** Миграция на `SDK/react-sdk` и `SDK/core`

2. **Нарушение паттерна MVP**
   - **Влияние:** Высокое. Смешение логики усложняет тестирование и поддержку.
   - **Приоритет:** Критический
   - **Решение:** Разделение на Model/View/Presenter слои

3. **Прямое обращение к серверу**
   - **Влияние:** Среднее. Нарушает принцип единой точки входа через Router.
   - **Приоритет:** Высокий
   - **Решение:** Использование Router Client из SDK

### 4.2. Безопасность и конфигурация

1. **Хардкод секретов**
   - **Влияние:** Критическое. Токены и URL в коде — уязвимость безопасности.
   - **Приоритет:** Критический
   - **Решение:** Environment variables и секреты CI/CD

2. **Небезопасная инъекция CSS**
   - **Влияние:** Среднее. Риск XSS при загрузке внешних стилей.
   - **Приоритет:** Высокий
   - **Решение:** Валидация и санитизация CSS, использование Next.js механизмов

### 4.3. Качество кода

1. **Отсутствие тестов**
   - **Влияние:** Среднее. Нет гарантий работоспособности при изменениях.
   - **Приоритет:** Высокий
   - **Решение:** Unit-тесты для компонентов и интеграционные тесты для сессий

2. **Избыточное логирование**
   - **Влияние:** Низкое. Замедляет работу и засоряет консоль.
   - **Приоритет:** Средний
   - **Решение:** Структурированное логирование с уровнями (dev/prod)

3. **Отсутствие TypeScript**
   - **Влияние:** Среднее. Нет типобезопасности и автодополнения.
   - **Приоритет:** Средний
   - **Решение:** Постепенная миграция на TypeScript

## 5. Рекомендации по доработке

### 5.1. Краткосрочные (MVP)

1. **Интеграция с SDK/core**
   - Установить зависимости `@cubica/sdk-core` и `@cubica/react-sdk`
   - Заменить `serverDataLoader` на `useCubicaSession` из `SDK/react-sdk`
   - Использовать `createRouterClient` для запросов к Router

2. **Вынос конфигурации**
   - Создать `.env.local` с переменными `NEXT_PUBLIC_ROUTER_URL`, `NEXT_PUBLIC_SESSION_TOKEN`
   - Удалить хардкод из `serverDataLoader.js` и `actions.js`

3. **Регистрация заглушек**
   - Добавить `src/app/api/route.js` в `docs/legacy/stubs-register.md`
   - Создать задачу на удаление заглушки

4. **Улучшение безопасности**
   - Убрать динамическую инъекцию CSS или добавить валидацию
   - Использовать Next.js `<Head>` для стилей

### 5.2. Среднесрочные (Фаза 1)

1. **Миграция на MVP паттерн**
   - Создать Presenter-слой для обработки действий и состояния
   - Разделить Model (данные сессии) и View (компоненты)
   - Использовать `ViewCommand`/`ViewResponse` для взаимодействия

2. **Перенос компонентов в SDK/shared**
   - Вынести `GameButton`, `GameCard`, `GameVariable` в `SDK/shared`
   - Настроить темизацию и кастомизацию
   - Обновить `game-player-nextjs` для использования компонентов из SDK

3. **Добавление тестов**
   - Unit-тесты для компонентов (Jest + React Testing Library)
   - Интеграционные тесты для сессий (msw для моков Router)
   - E2E тесты для основных сценариев игры

4. **Миграция на TypeScript**
   - Постепенная миграция файлов на `.ts`/`.tsx`
   - Типизация контрактов манифестов
   - Интеграция с типами из `SDK/core`

### 5.3. Долгосрочные (Фаза 2+)

1. **Поддержка WebSocket**
   - Интеграция с WebSocket transport из `SDK/core`
   - Real-time обновления состояния через подписки
   - Обработка переподключений и ошибок сети

2. **Версионирование манифестов**
   - Проверка `schema_version` и `min_engine_version` при загрузке игры
   - Обработка несовместимых версий
   - Миграция состояния при обновлении схемы

3. **Наблюдаемость**
   - Интеграция с OpenTelemetry для трассировки
   - Метрики производительности и ошибок
   - Структурированное логирование

4. **Оптимизация производительности**
   - Code splitting для компонентов игры
   - Lazy loading экранов и ассетов
   - Мемоизация рендеринга через React.memo и useMemo

## 6. План миграции на SDK

### Этап 1: Подготовка (1-2 недели)

1. Установить зависимости SDK в `game-player-nextjs`:
   ```json
   {
     "dependencies": {
       "@cubica/sdk-core": "workspace:*",
       "@cubica/react-sdk": "workspace:*",
       "@cubica/sdk-shared": "workspace:*"
     }
   }
   ```

2. Создать конфигурационный файл для Router:
   ```javascript
   // src/config/router.js
   export const routerConfig = {
     routerBaseUrl: process.env.NEXT_PUBLIC_ROUTER_URL || 'http://localhost:3001',
     transport: 'http', // позже 'ws'
   };
   ```

3. Зарегистрировать текущие заглушки в `docs/legacy/stubs-register.md`

### Этап 2: Замена загрузки данных (1 неделя)

1. Заменить `GameScreenRenderer` на использование `useCubicaSession`:
   ```typescript
   import { useCubicaSession } from '@cubica/react-sdk';
   
   const { status, data, error } = useCubicaSession(sessionId, routerConfig);
   ```

2. Удалить `serverDataLoader.js` и `localDataLoader.js` (или оставить как fallback)

3. Обновить `page.js` для передачи `sessionId` из query params или контекста

### Этап 3: Миграция компонентов (2-3 недели)

1. Проверить наличие компонентов в `SDK/shared`
2. Если компонентов нет — перенести из `game-player-nextjs` в `SDK/shared`
3. Обновить импорты в `game-player-nextjs`:
   ```typescript
   import { GameButton, GameCard, GameVariable } from '@cubica/sdk-shared';
   ```

4. Адаптировать пропсы под контракты SDK

### Этап 4: Внедрение MVP паттерна (3-4 недели)

1. Создать Presenter-слой:
   ```typescript
   // src/presenters/GamePresenter.ts
   import { ViewCommand, ViewResponse } from '@cubica/sdk-core';
   
   export class GamePresenter {
     async handleAction(action: ViewCommand): Promise<ViewResponse> {
       // Логика обработки действий
     }
   }
   ```

2. Разделить Model (данные сессии) и View (компоненты)
3. Обновить компоненты для использования Presenter вместо прямых действий

### Этап 5: Тестирование и документация (2 недели)

1. Написать unit-тесты для компонентов
2. Интеграционные тесты для сессий
3. Обновить README с примерами использования SDK
4. Создать ADR о миграции (если требуется)

## 8. Заключение

Проект `game-player-nextjs` демонстрирует рабочую концепцию рендеринга игр из JSON-манифестов, но требует значительной доработки для соответствия целевой архитектуре платформы Cubica. Основные направления улучшения:

1. **Интеграция с SDK** — критически важно для переиспользования компонентов и контрактов
2. **Соблюдение MVP паттерна** — необходимо для масштабируемости и тестируемости
3. **Безопасность** — устранение хардкода и небезопасных практик
4. **Качество кода** — добавление тестов и миграция на TypeScript

Рекомендуется начать с краткосрочных задач (интеграция SDK, вынос конфигурации), затем перейти к архитектурным изменениям (MVP паттерн, перенос компонентов).
