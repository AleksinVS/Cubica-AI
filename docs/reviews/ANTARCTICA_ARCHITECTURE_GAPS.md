# Анализ разрывов: Архитектура игры «Антарктика» vs Целевая архитектура платформы Cubica

Этот документ описывает расхождения между текущей реализацией игры «Antarctica» и целевым состоянием архитектуры платформы Cubica, зафиксированным в `PROJECT_ARCHITECTURE.md`, `PROJECT_OVERVIEW.md` и соответствующих ADR.

## 1. Организация сервисов (Service Maturity)

| Целевая архитектура | Текущее состояние (Antarctica) | Описание разрыва |
| :--- | :--- | :--- |
| Раздельные микросервисы: Router, Game Engine, Repository, Catalog. | **Runtime API (Modular Monolith)**. | Платформа находится в фазе модульного монолита (ADR-017). Функции роутинга, выполнения логики и загрузки контента объединены в `services/runtime-api`. |
| Полноценный **Game Repository** с API. | Прямое чтение файлов из `games/antarctica/` через `runtime-api`. | Отсутствует слой абстракции репозитория; runtime напрямую зависит от структуры файловой системы в папке `games/`. |
| **Game Editor** для авторинга. | Ручное редактирование JSON-манифестов и извлечение из legacy HTML-прототипа. | Инструменты авторинга являются «заглушками» (scaffolds); процесс миграции сценария детерминирован и ручной. |

## 2. Игровой движок и логика (Engine & Runtime)

| Целевая архитектура | Текущее состояние (Antarctica) | Описание разрыва |
| :--- | :--- | :--- |
| **LLM-first Engine**: LLM как основной движок. | **Deterministic Runtime**: Логика жестко прописана в манифесте. | Целевой LLM-слой пока является «capability layer» будущего; текущая реализация полностью детерминирована (ADR-024). |
| **Hybrid Execution**: JS-скрипты в `isolated-vm` (ADR-010). | Скрипты `actions.js` являются заглушками; логика вшита в метаданные действий в JSON. | Песочница для пользовательских скриптов не используется активно; расширение платформы идет через добавление новых типов метаданных в манифест. |
| **Platform Purity**: Отсутствие специфичного для игры кода в ядре. | Секция `content.antarctica` в манифесте и специфичные контракты. | В манифесте и контрактах (`packages/contracts/manifest`) присутствуют специфичные для Антарктики структуры. Это разрешено как «local manifest mechanics», но требует генерализации. |

## 3. Управление состоянием (State Management)

| Целевая архитектура | Текущее состояние (Antarctica) | Описание разрыва |
| :--- | :--- | :--- |
| **Persistence**: PostgreSQL (JSONB) + Redis (Locks/Cache). | **In-memory store**: Сессии хранятся в оперативной памяти. | Отсутствует персистентность сессий, механизмы блокировок и восстановления после сбоев (ADR-005). |
| **Multiplayer**: Очередь событий и синхронизация (ADR-011). | **Single-player**: Прямая обработка действий. | Многопользовательский режим не реализован; архитектура `runtime-api` ориентирована на синхронную обработку одного игрока. |

## 4. Клиент и SDK

| Целевая архитектура | Текущее состояние (Antarctica) | Описание разрыва |
| :--- | :--- | :--- |
| **Unified SDK**: `SDK/core`, `SDK/shared` как основа. | `apps/player-web` использует собственные компоненты и типы. | Использование общих пакетов SDK минимально; `player-web` развивается как независимый канонический слой, не опираясь на старую структуру `SDK/`. |
| **Abstract View Protocol**: Взаимодействие через `ViewCommand`. | Прямая проекция состояния манифеста в UI. | Протокол команд Presenter -> View (ADR-002) реализован частично через проекцию контента, а не через поток императивных команд. |

## 5. Инфраструктура и Наблюдаемость

| Целевая архитектура | Текущее состояние (Antarctica) | Описание разрыва |
| :--- | :--- | :--- |
| **Observability**: OpenTelemetry, Prometheus, ELK. | Базовое логирование в `runtime-api`. | Отсутствует распределенная трассировка, сбор метрик и централизованное управление логами. |
| **Validation**: Строгая валидация всех манифестов через AJV (ADR-025). | Валидация внедрена в `runtime-api`. | **Разрыв минимален**: Валидация по схеме уже является каноническим требованием и реализована для `Antarctica`. |

## Итог
Игра «Antarctica» на текущий момент является **каноническим примером миграции** на новую платформу, но сама платформа находится в состоянии «Minimal Viable Runtime». Основные разрывы связаны с отсутствием персистентности, недоразвитостью сервисной инфраструктуры (Repository/Editor) и временным отказом от LLM/JS-скриптов в пользу детерминированного JSON-описания для ускорения доставки первого игрового сценария.
