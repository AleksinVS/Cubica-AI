# Анализ разрывов: Архитектура игры «Антарктика» vs Целевая архитектура платформы Cubica

Этот документ описывает расхождения между текущей реализацией игры «Antarctica» и целевым состоянием архитектуры платформы Cubica с учетом текущего этапа развития проекта.

## 1. Организация сервисов (Service Maturity)

| Целевая архитектура | Текущее состояние (Antarctica) | Статус разрыва |
| :--- | :--- | :--- |
| Раздельные микросервисы. | **Runtime API (Modular Monolith)**. | **Легальный этап (Planned):** Платформа находится в фазе модульного монолита (ADR-017). |
| Полноценный **Game Repository** с API. | Прямое чтение файлов из `games/antarctica/` через `runtime-api`. | **Критичный разрыв (Gap):** Отсутствует абстракция репозитория; рантайм жестко привязан к локальной файловой системе, что противоречит масштабируемости. |
| **Game Editor** для авторинга. | Ручное редактирование JSON. | **Не приоритетно (Non-priority):** На текущем этапе реализация редактора не является приоритетом. |

## 2. Игровой движок и логика (Engine & Runtime)

| Целевая архитектура | Текущее состояние (Antarctica) | Статус разрыва |
| :--- | :--- | :--- |
| **LLM-first Engine**: LLM как основной движок. | **Deterministic Runtime**: Логика жестко прописана в манифесте. | **Легальный вариант (Legal):** Для игры "Антарктика" LLM избыточен. Детерминированный рантайм на основе JSON-манифеста полностью легален для этой игры. |
| **Hybrid Execution**: JS-скрипты в изолированной песочнице (Sandbox). | Скрипты являются заглушками; вся логика в JSON-декларациях. | **Отложено (Delayed):** Безопасность кода и песочница будут прорабатываться на более поздних этапах. |
| **Platform Purity**: Отсутствие специфичного для игры кода в ядре. | Секция `content.antarctica` в общих контрактах и манифестах. | **Разрыв (Gap):** Хардкод специфичных для игры структур данных в общих пакетах (например, в `packages/contracts/manifest`) противоречит концепции универсальной платформы. |

## 3. Управление состоянием (State Management)

| Целевая архитектура | Текущее состояние (Antarctica) | Статус разрыва |
| :--- | :--- | :--- |
| **Persistence**: PostgreSQL (JSONB) + Redis (Locks/Cache). | **In-memory store**: Сессии хранятся в оперативной памяти. | **Легаси (Legacy):** На данном этапе оставляем как есть, но это явное легаси, которое требует замены в будущем. |
| **Multiplayer**: Очередь событий и синхронизация. | **Single-player**: Синхронная обработка одного игрока. | **Не приоритетно (Non-priority):** "Антарктика" - одиночная игра, мультиплеер пока не требуется. |

## 4. Клиент и SDK

| Целевая архитектура | Текущее состояние (Antarctica) | Статус разрыва |
| :--- | :--- | :--- |
| **Unified SDK**: Базовые пакеты в `SDK/` для любых клиентов. | `apps/player-web` использует собственные компоненты. | **Разрыв (Gap):** Плеер сильно завязан на конкретный интерфейс Антарктики. Отсутствует переиспользование. |

## 5. Противоречия концепции универсальной платформы (Universal Platform Contradictions)

После глубокого анализа текущей реализации (в частности, пакета `packages/contracts/manifest`, схемы `game-manifest.schema.json` и `apps/player-web`) выявлены серьезные архитектурные дрейфы и противоречия:

1. **Доменно-связанные контракты (Domain-Coupled Contracts):**
   - В общих контрактах (`packages/contracts/manifest/src/index.ts`) и корневой JSON-схеме (`game-manifest.schema.json`) жестко прописаны типы, специфичные исключительно для механик Антарктики: `GamePlayerBoard`, `GamePlayerBoardCard`, `GamePlayerTeamSelectionScene`. 
   - Структура `GamePlayerContent` имеет поля `boards`, `infos`, `teamSelections`, `cards`. Это означает, что **любая** игра на платформе теперь обязана мыслить терминами "досок" и "карточек", что неприемлемо для универсальной платформы (где могут быть текстовые квесты, 2D-игры, симуляторы чатов и т.д.).

2. **Захардкоженные механики в ядре (Hardcoded Mechanics in Engine):**
   - Метаданные детерминированного рантайма (`GameManifestDeterministicStateUpdate`) включают поля `boardCardUnlock`, `boardEntryAltCardSwap`, `teamSelection.pickCountDelta`. Это означает, что правила конкретной игры (Антарктики) просочились на уровень схемы движка платформы.
   - В соответствии с ADR-024 предполагалось моделировать механики явными (explicit) переходами, но вместо обобщенных операций изменения состояния (например, JSON Patch или setVariable), в ядро платформы захардкодили `teamSelection`.

3. **Player Web маскируется под универсальный (Player is Antarctica-Specific):**
   - Приложение `apps/player-web` переименовало файлы в `game-player.tsx`, но внутри жестко привязано к `ANTARCTICA_GAME_CONFIG` и `AntarcticaGameState`.
   - В компоненте `fallback-renderer.tsx` жестко зашиты фоновые изображения (`/images/arctic-background.png`), названия кнопок ("журнал ходов", "подсказка") и пути к ассетам (`/images/jurnal-hodov.png`). Это делает плеер специализированным клиентом Антарктики, а не платформенным UI-рендерером.

### Решения без опоры на ADR:
- Нет ни одного ADR, который оправдывал бы вынос предметной области конкретной игры (`teamSelection`, `boardThreshold`) в `game-manifest.schema.json`. Прежний план разрешал явные переходы, но не засорение базовой схемы доменными примитивами конкретной игры.
- Нет ADR, оправдывающего хардкод `ANTARCTICA_GAME_CONFIG` в корневом универсальном плеере (`apps/player-web`).

## Детальный план улучшений (Universal Platform Alignment)

1. **Абстракция Game Repository:** 
   - Вынести логику загрузки игр (манифестов и ресурсов) из `runtime-api` за интерфейс репозитория, чтобы рантайм ничего не знал о `games/antarctica/` на диске.

2. **Очистка JSON Schema и контрактов (Schema Purity):**
   - Удалить хардкод `antarctica`, `boardCardUnlock`, `teamSelection`, `GamePlayerBoard` и прочие сущности Антарктики из общих типов `packages/contracts/manifest` и схемы `game-manifest.schema.json`.
   - Внедрить обобщенную модель `CustomGameContent` или `ExtensionPayload`, структуру которых платформа не проверяет строго (либо проверяет по schema-ref, поставляемой самой игрой).
   - Внедрить обобщенную модель модификации состояния для детерминированного рантайма (например, JSON Patch операции).

3. **Декаплинг Player Web (Player Decoupling):**
   - Вынести `ANTARCTICA_GAME_CONFIG` и `AntarcticaGameState` за пределы ядра плеера `apps/player-web`, передавая их как конфигурацию (Game Plugin) при инициализации.
   - Удалить хардкод фоновых изображений (`/images/arctic-background.png`) из `fallback-renderer.tsx`. Данные о фоне и стилях должны приходить из `UI Manifest` игры.
   - Элементы UI (layout-ы, компоненты карточек) должны определяться либо через SDUI-примитивы, либо через изолированные плагины компонентов для конкретной игры, а не зашиваться в платформенный fallback-рендерер.

4. **Оформление плагинов механик (Game Mechanics Plugins):**
   - Если детерминированные механики (как `boardCardUnlock`) нужны, они должны реализовываться как "Engine Extensions" (согласно ADR-015), либо как JS-скрипты `actions.js`, а не как хардкод в базовых метаданных платформы.
