# Цель: разделение ответственности между моделями разного уровня, разделение работ по субагентам в зависимости от их сложности, сужение контекста для исполнителей и уменьшение числа запросов к моделям. Очевидно, что сужение контекста и уменьшение числа запросов - конфликтующие задачи, поэтому нужно найти оптимальное компромиссное решение.

## Repo-local storage note

- `.codex/workflow/` хранит **canonical latest** workflow-файлы (то есть последние канонические файлы-указатели для быстрого входа в текущее состояние workflow).
- `.tmp/agent-workflow/{block_id}/` хранит рабочее пространство конкретного блока: `BLOCK_BRIEF.md`, `task-packets/`, run log, health/retrospective и другие временные артефакты.
- Термин `SLICE_TASK` ниже — это legacy-термин. Текущее предпочтительное имя runtime-артефакта: `task-packets/{task_id}.json`.

# Роли агентов: 
1. Оркестратор (GLM-4.7 medium с fallback/эскалацией до GPT-5.4 medium) - самая дешевая и простая модель не прерывает свою сессию между задачами, имеет доступ ко всем файлам проекта, запускает субагентов, следит за тем, чтобы субагенты точно следовали своим инструкциям, маршрутизирует, взаимодействует с пользователем. Может эскалировать до любого из субагентов или пользователя. Главная задача - получить от Архитектора точный json-план работ, с инструкциями для всех субагентов включая себя, запустить субагентов с этими инструкциями и проконтролировать выполнение.
2. Архитектор (GPT-5.4 high) - самая сильная и дорогая модель, определяет цели, планирует работы по проекту, следит за тем, чтобы не было дрейфа целей, предлагает улучшения в работе команды, определяет архитектуру проекта, декомпозирует архитектуру до конкретных срезов и задач, описывает необходимые тесты, проверяет текущее состояние на соответствие целевой архитектуре. Может взаимодействовать с пользователем по текущей задаче. Не прерывает сессию между задачами, имеет доступ ко всем файлам проекта. 
3. Исполнитель (opencode minimax high / GLM-4.7 medium) - код; - тесты; - локальная документация; self-check.
4. PM (Gemini high) - тестирование результатов, итоговое ревью по блокам разработки, приемка работ.
5. Аналитик данных (opencode minimax high)- составляет скрипты для анализа и обработки данных, проводит анализ данных.
6. web-аналитик (opencode GLM-4.7 medium) - выполняет поиск в интернете, предварительный анализ поисковой выборки и подготовку результатов в соответствии с контрактами.

# Общий план
ARCHITECT_PLAN.json → PM plan_review → Architect decision → SLICE_EXPORT_SPEC.json → materialize task packets → `task-packets/{task_id}.json` → исполнители → PM_DECISION_PACKET.

Примечание по методике `Contract-first / schema-first`:

- до финального `PM_DECISION_PACKET` по блоку обычно требуется `CONFORMANCE_REPORT.md` (проверка соответствия реализации контракту).
- Архитектор готовит один большой json-план для всех исполнителей;
- Оркестратор запускает для каждой части плана нужных исполнителей, при этом делает это "механически" ничего в плане не дописывая и не меняя; 
- PM проверяет результат, делает ревью тестов (при необходимости дописывает недостающие тесты) и сам их запускает, PM либо принимает выполненный блок, либо отдает на доработку исполнителю;
- После приемки блока PM, Архитектор проверяет результаты и переходит либо к следующему блоку, либо к новой итерации по текущему блоку. 

# Общие принципы:
Общую координацию осуществляет Архитектор через оркестратора.
**У всех субагентов четко определенные входные и выходные артефакты по строгому контракту.**
Контекст субагентов должен быть минимально необходимым и достаточным для полного решения задачи.
Если у кого-то из агентов возникает необходимость в поиске данных в интернете или анализе/обработке данных - они делегируют эту работу агентам-аналитикам.
**Даже если `L1` или `L2` нашёл проблему, он не должен лично вести цикл правок. Если исполнитель не смог реализовать задачу, `L1` принимает решение либо уточнить задачу для исполнителя (по умолчанию), либо может внести правки сам.**
Архитектор и оркестратор, не прерывают сессию между блоками разработки, а висят в постоянных сессиях, суммаризация происходит только при переполнении контекста и перехода к новой большой архитектутной части слабо связанной с предыдущей.

# Воркфлоу:
1. Архитектор (L1)- прорабатывает сначала высокоуровневую архитектуру (при необходимости), потом определяет следующий блок работ и передает оркестратору детальный план работ по выбранному блоку с разбиением на отдельные срезы и задачи. Блок должен выбираться так, чтобы объединять не только срезы близкие по архитектуре, но и по методике разработки. Значимые архитектурные решения оформляются отдельными ADR. В выходном документе Архитектор описывает:
- максимально точную инструкцию для Оркестратора,
- цели, 
- критерии оценки/принятия, 
- пользовательский опыт, 
- архитектурные решения,
- архитектурные ограничения и запреты,
- риски,
- применяемые технологии,
- общие правила,
- формулирует acceptance criteria для тестов,
- задаёт product-level сценарии тестов
- указывает, какие отклонения в тестах недопустимы,
- детальные правила, 
- тесты и чек-листы,
- определяет required checks,
- фиксирует, какие инварианты нельзя ломать
- определяет, какие тесты обязательны для slice
- контрольные точки для всех участников, 
- разделение задач между junior и senior исполнителями (junior получает только тривиальные задачи из `R0`),
- правила эскалации,
- входные и выходные артефакты,
- `allowed files`,
- `forbidden moves`,
- `required docs`,
- класс риска.
 
Нужны три класса рисков:
- `R0` — механический: переименование, локальный тест, небольшая правка документации, узкий refactor без изменения behavior;
- `R1` — bounded delivery: локальная механика, контракт в пределах существующей модели, новый rendering path внутри текущего canonical slice;
- `R2` — архитектурно опасный: новый public contract, новый reusable abstraction, изменение ownership boundary, новый ADR, cross-module semantics.

В итоге получается подробная инструкция-шаблон, по которой оркестратор может точно **без дополнительного анализа и рассуждений** в правильном порядке "нарезать" задачи исполнителю, запустить субагентов, проконтролировать их работу, маршрутизировать ход работ и при необходимости эскалировать к нужным субагентам. В инструкции Архитектор описывает не только план работ, но и возможные ошибки/сбои и правила их обработки (эскалации). Задачи должны быть описаны так, чтобы исполнитель четко понимал цели и требования в ограниченном контексте без дополнительных уточнений.
 **Структура выходного документа должна быть такой, чтобы при его декомпозиции до конкретных задач исполнителям, оркестратору не нужно было читать весь документ, при этом оркестратор мог точно понять какие части документа нужно читать для каждого среза.**
 
2. Оркестратор в соответствии с инструкциями из плана, создает (простым копированием из плана) отдельные файлы срезов для исполнителей и запускает субагентов. Текущий процесс оркестратора логируется в отдельном файле, чтобы PM всегда мог понять на какой стадии идет работа. При возникновении сбоев оркестратор эскалирует до Архитектора или PM (по умолчанию) в зависимости от инструкций.

3. Исполнители (L3) - отдельные срезы передаются субагнетам исполнителям (senior/junior). Исполнитель реализует описаные срезы, пишет и запускает тесты и доводит работу до соответствия критериям приемки. На выходе, кроме непосредственно кода и локальной документации, исполнитель создает один общий отчет включающий: 
- принятые решения;
- находки и сомнения;
- изменённые файлы;
- verification result;
- remaining risks;
- краткий self-check по шаблону. 
- next slice.

Self-check должен быть стандартным:
- не вышел ли за write scope;
- не ввёл ли новую абстракцию;
- не затронул ли канонический источник истины неправильно;
- `docs updated`
- `checks passed`
- `open questions`
- есть ли несогласованность между кодом и документацией.

**Описывать находки нужно так, чтобы для точного понимания их сути и контекста не требовался дополнительный анализ**. Исполнителю запрещено самовольно расширять архитектуру или изменять тесты. Если исполнитель находит ошибки в тестах или фактических данных - он отмечает это в специальном разделе своего отчета, этот раздел обязательно читает Архитектор при проверке результатов блока. Если junior исполнитель не может выполнить работу в определенных рамках он через оркестратора эскалирует ее до senior. Исполнитель обязан вести компактный rolling checkpoint после каждого slice.
У `L3` должен быть явный allowed files list;
Любое расширение write scope требует возврата к `L2`;
В self-check обязательно поле `scope changed: yes/no`.
Исполнитель пишет и выполняет тесты.
Исполнитель прогоняет тесты и доводит их до зеленого уровня, либо эскалирует к PM. Результаты тестов дописываются отдельным блоком в отчет исполнителя.
Если все тесты успешно пройдены текущий срез коммитится в отдельную ветку, исполнитель получает от оркестратора следующий срез из блока и так для каждого среза в блоке.

6. PM (L2)
- проверяет тесты на корректность и полноту, при необходимости исправляет или пишет новые;
- выполняет тесты;
- принимает результат, делает общее ревью блока с учетом инструкций архитектора и отчета исполнителя, обязательно проверяет работу в контрольных точках указанных архитектором.
- возвращает исполнителю на доработку (отдельным доп. планом), либо принимает блок и передает Архитектору.

`L2` должен передавать `L1` не весь контекст, а decision packet:
 - что хотели сделать;
 - что реально изменили;
 - какие файлы затронуты;
 - что проверили;
 - какие риски остались;
 - есть ли отклонение от charter;
 - требуется ли решение `L1`, и если да, то по какому узкому вопросу.

## Жёсткий формат handoff
 `L2` не должен писать длинные отчёты. Нужен единый формат:
- `goal`
- `status`
- `files changed`
- `checks`
- `deviations`
- `risks`
- `docs updated: yes/no`
- `decision needed`.

Обязательно проверяет следующие тесты:
      - contract tests
      - regression tests
      - negative tests
      - snapshot/fixture expectations.

Если отклонений нет, `L1` должен иметь право принять блок по этому пакету и по выборочной проверке 1-2 ключевых файлов, а не читать всё заново.

7. Ревью вместе с отчетом исполнителя передаются Архитектору, Архитектор сливает ветки принятых срезов в основную ветку и либо отправляет текущий блок на следующую итерацию (частично или полностью), либо переходит к следующему блоку работ. Если исполнитель не смог реализовать задачу, `L1` уточняет методику решения задачи для исполнителя.

Документирование должно быть частью definition of done;
`L1` не принимает slice без обновления нужных документов;

## Правило по классам срезов:
- `R0`: `L1` не делает полноценное ревью, достаточно утверждённого шаблона отчёта от `L2`;
- `R1`: `L1` смотрит только decision packet и при необходимости выборочно critical diff;
- `R2`: `L1` делает полноценное финальное ревью на основе decision packet.
