## 1. Главный принцип

> Никогда не верстай «прямо с картинки». Путь всегда:
> **изображение → письменная инвентаризация → структура → стиль → измеренная
> проверка.**

Почему это не бюрократия:

- При верстке напрямую с изображения модель **пропускает элементы** (особенно
  мелкие и периферийные), **выдумывает значения** отступов и цветов (13px
  вместо 16px, «примерно такой синий») и затем **рационализирует** расхождения
  («визуально совпадает»). Письменный инвентарь проверяем: пропуск виден как
  дыра в таблице, а не как незамеченный пиксель.
- Индустриальный опыт (зафиксирован в
  `docs/architecture/game-ui-mockup-flow.md`, раздел «Уроки индустрии»):
  image-first генерация даёт «слишком случайную» структуру, которую
  приходится пересобирать. Рабочий принцип проекта: **структура — источник
  истины, изображение — концепт стиля и источник ассетов.**

Инструменты для всех операций с изображениями — из навыка `ui-compare`
(соседний каталог навыков; в этом репозитории —
`.claude/skills/ui-compare/`): `node <каталог-навыков>/ui-compare/scripts/ui-visual-tool.mjs ...`
(команды `to-png`, `crop`, `sample`, `capture`, `compare`; требования к
окружению и описание команд — в SKILL.md навыка ui-compare).

## 2. Шаг 0 — режим строгости и рамки

Определи по происхождению образца и подтверди у пользователя, если неочевидно:

- **pixel-parity** — образец является скриншотом работающего UI (прежняя
  версия, draft-приложение). Цель — пиксельное совпадение, ворота < 5% diff.
- **style-parity** — образец сгенерирован (Midjourney и т.п.) или нарисован
  от руки. Пиксельное совпадение недостижимо и не требуется («картинка —
  концепт, а не пиксельный эталон»); цель — совпадение структуры зон,
  дизайн-токенов (палитра, типографика, скругления) и наполнения.

Зафиксируй выбранный режим в задаче/отчёте — от него зависят ворота приёмки.

## 3. Шаг 1 — инвентаризация макета

Обязательный письменный артефакт **до любого кода**.

1. **Приведи макет к PNG и узнай размеры:**
   ```bash
   node <каталог-навыков>/ui-compare/scripts/ui-visual-tool.mjs to-png <макет> --out .tmp/ui-compare/mockup.png
   ```
2. **Пройди макет сеткой кропов 3×3** (`crop` по девяти прямоугольникам) и
   перечисли все видимые элементы каждого кропа. Почему по кропам: на полном
   кадре, уменьшенном до размеров контекста, мелкие элементы теряются — именно
   так возникают «агент не увидел очевидное».
3. **Для каждой зоны и элемента запиши:** `id`, тип (container / button /
   indicator / text / canvas / decor...), границы в пикселях (`bounds`),
   текстовое содержимое, состояние (default/active/disabled), вложенность.
   **Размечай на уровне элементов** (кнопка, карточка, индикатор), а не только
   крупных областей: эта же разметка потом служит спецификацией регионов для
   приёмки (`compare --regions`), и в крупном регионе локальный дефект
   разбавляется процентом так же, как в полном кадре.
   **Перекрытия фиксируй явно:** если элементы накладываются (модальное окно,
   декоративный оверлей, тень поверх соседа), запиши порядок наложения
   («что поверх чего») — из плоской картинки z-порядок восстанавливается
   только в момент разметки, а при реализации он превращается в проверяемые
   требования (z-index, `pointer-events`) и точки проверки кликабельности.
   **Селектор для приёмки:** как только элемент реализован, допиши в его
   запись инвентаря поле `selector` (CSS-селектор DOM-элемента; схема ADR-016
   допускает дополнительные поля) — это включает двухуровневую приёмку
   `compare-elements`: геометрия, внешний вид, перекрытие и явный `MISSING`
   по каждому элементу.
4. **Цвета — только через `sample`, на глаз запрещено.** Для фонов бери
   `--rect` (среднее по зоне сглаживает JPEG-артефакты), для тонких элементов —
   `--points`. Модельная оценка оттенка «на глаз» систематически ошибается.
5. **Типографика и отступы:** семейство (или ближайший доступный аналог),
   размеры относительно высоты макета, шкала отступов (обычно кратна 4 или 8 —
   проверь измерением по кропам, а не предположением).
6. **Самопроверка полноты:** пройди инвентарём обратно по макету — каждый
   пиксель макета должен принадлежать какой-то зоне (хотя бы «декор/фон»).
   Незакрытые области = пропущенные элементы.

**Куда класть инвентарь:**

- Работа над игрой в `games/<id>/` — оформи как `*.design.json` по ADR-016:
  файл рядом с изображением в `games/<id>/design/mockups/`, схема
  `docs/architecture/schemas/design-artifact.schema.json`, примеры в
  `games/antarctica/design/mockups/` и `docs/architecture/schemas/examples/`.
  Не забудь запись в `design-history.json` и реестр `design_artifacts`
  UI-манифеста.
- Разовая задача вне структуры игры — JSON-массив регионов
  `[{id,x,y,w,h}]` или облегчённый design.json в `.tmp/` либо в файле задачи.

Двойная роль инвентаря: тот же `*.design.json` затем передаётся на приёмке в
`compare --regions` — инвентарь и есть спецификация сравнения по элементам.

## 4. Шаг 2 — классификация и рельсы проекта

Прежде чем писать код или манифест:

1. **Прототип UI в Cubica — это authoring-манифест, а не код** (ADR-050,
   ADR-057). Если задача касается экранов игры — структура собирается из
   каталога прототипов, а не свободной версткой; правки идут через конвейер
   `EditorChangeSet`, отдельные «генеративные» пути записи запрещены.
2. **Дисциплина идентичности** (ADR-057): каждый содержательный UI-элемент
   инвентаря ссылается на `id` игровой сущности (`maps_to_component`,
   `maps_to_state`, `maps_to_action` в design.json); декоративные элементы
   явно помечаются декоративными.
3. **Платформенная чистота** (CLAUDE.md п.10): для каждой механики из макета
   реши — общая она (платформенный прототип/контракт) или игровая (бандл
   игры). Никаких game-specific веток в общих слоях.
4. **Parity-работа в рендерере** (перенос внешнего вида в `player-web` и т.п.)
   — следуй `docs/processes/UI_MIGRATION_METHODOLOGY.md`:
   CSS-override-блоки, без изменения архитектуры target.

## 5. Шаг 3 — реализация слоями

Реализуй в четыре слоя, **после каждого слоя — быстрый прогон `compare`**
(достаточно сетки 4×4 без глубокой локализации):

1. **Каркас**: контейнеры, сетка, границы зон из инвентаря. Проверка: крупные
   зоны стоят на своих местах (ячейки сетки не показывают 100% diff из-за
   сдвига layout).
2. **Размеры и отступы**: точные bounds, gap, padding из инвентаря.
3. **Цвета и типографика**: значения из `sample`, шрифты, веса.
4. **Ассеты и декор**: иконки, фоны, иллюстрации (для игры — через принятый
   поток ассетов с метаданными происхождения).

Почему слоями с проверкой: стилизация **маскирует** структурные сдвиги — если
каркасная ошибка доживает до слоя 4, её поиск среди раскрашенных элементов
стоит в разы дороже. Ошибки каждого слоя чинятся, пока слой «голый».

Данные предпросмотра должны совпадать с содержимым макета (фикстуры с теми же
текстами/количеством карточек) — иначе `compare` измеряет разницу данных, а не
верстки. Подбор фикстур — часть задачи, а не опция.

## 6. Шаг 4 — приёмка

Полный проход навыка `ui-compare` (шаги 0–5 там):

- сравнение по регионам инвентаря: передай `*.design.json` из шага 1 в
  `compare --regions` — отчёт и автокропы будут в терминах конкретных
  элементов макета;
- для элементов с заполненным `selector` — двухуровневая проверка
  `compare-elements <ref.png> <url> --regions <design.json>`: геометрия
  (допуск ±2px), внешний вид (< 10% по кропу), перекрытие интерактивных
  элементов, явный `MISSING` для ненайденных;
- режим pixel-parity: ворота < 5% diff, ни один регион/ячейка не выше порога
  ячейки, все зоны классифицированы;
- режим style-parity: таблица токенов «образец (sample) → реализация
  (computed style)» — палитра, типографика, структура зон;
- кликабельность интерактивных элементов проверена отдельно
  (`elementFromPoint`, методология §7);
- намеренные отклонения от макета (недоступный шрифт, добавленные элементы)
  задокументированы в задаче;
- отчёт пользователю — по шаблону из ui-compare.

Не сообщай о завершении задачи без свежего прогона ворот: «сверстал по
инвентарю, должно совпадать» — не результат.

## 7. Ловушки

| Ловушка | Что делать |
|---|---|
| Шрифт макета недоступен/неопределим | Выбрать ближайший доступный, зафиксировать как намеренное отклонение; помнить, что он даст остаточный diff-шум |
| Сгенерированный макет содержит нерегулярности (кривые дубли, артефакты) | Это режим style-parity: воспроизводить смысл зоны, не артефакты; спорные зоны согласовать с пользователем |
| Мок-данные не совпадают с макетом | Сделать фикстуру с данными макета до этапа сравнения |
| Соблазн «подогнать» инвентарь под уже написанный код | Инвентарь пишется до кода и правится только по измерениям макета |
| Макет одного размера, целевой viewport другого | Масштабировать сравнение через `to-png --width`; пропорции и шкалу отступов пересчитать явно |
| Элемент есть в макете, но не имеет игровой сущности | Не выдумывать сущность молча: либо декоративный элемент, либо вопрос пользователю (дисциплина идентичности) |
