## 1. Главное правило и почему оно такое

> Вывод «совпадает / не совпадает» делается **только по числам** (процент
> pixelmatch, замеренные цвета, computed styles), никогда — по визуальному
> впечатлению от двух скриншотов.

Причины, почему глазам модели верить нельзя (это не перестраховка, а
подтверждённое ограничение):

- **Vision-модель видит смысл, а не геометрию.** Она надёжно читает «синяя
  кнопка с текстом "Продолжить"», но не отличает отступ 13px от 16px, оттенок
  `#0f4c75` от `#14507a` и сдвиг блока на 6px. Задачи типа «найди отличия»
  между двумя изображениями — документированная слабость мультимодальных LLM;
  индустрия использует их только для *классификации* различий, найденных
  детерминированным пиксельным сравнением, а не для их *поиска*.
- **Скриншоты в контексте уменьшены.** Полный кадр 1920×1080 при попадании в
  контекст модели сжимается — мелкие различия исчезают физически, ещё до
  всякого анализа. Поэтому детали смотрят только на кропах (вырезках) 1:1 или
  с увеличением.
- **Подтверждающее смещение.** Агент, который сам писал код, ожидает увидеть
  правильный результат — и «видит» его, рационализируя различия («визуально
  совпадает», «различие несущественно»). Числа не рационализируются.

Разделение ролей: **скрипт находит и локализует различия, модель объясняет их
причину и чинит**.

## 2. Два режима строгости

Перед сравнением определи происхождение образца — от этого зависит критерий:

| Режим | Когда | Критерий |
|---|---|---|
| **pixel-parity** | Образец — скриншот работающего UI (draft-приложение, прежняя версия, другой рендерер) | Пиксельные ворота: diff < 5% (см. [раздел 5](#5-ворота-приёмки)) |
| **style-parity** | Образец — сгенерированный концепт (Midjourney и т.п.) или рисованный макет | Пиксельный процент **не применим**: сгенерированные картинки нерегулярны по природе. Сравниваются дизайн-токены: цвета зон (через `sample`), структура и порядок зон, типографика, скругления. Принцип проекта: «картинка — концепт, а не пиксельный эталон» (`docs/architecture/game-ui-mockup-flow.md`) |

Если режим неочевиден из задачи — спроси пользователя или явно зафиксируй
принятое допущение в отчёте.

## 3. Инструмент

Все операции выполняет `scripts/ui-visual-tool.mjs` из каталога этого навыка.
Запускай из корня проекта, подставив фактический путь к каталогу навыка
(в этом репозитории — `.claude/skills/ui-compare/`):

```bash
node <каталог-навыка>/scripts/ui-visual-tool.mjs <команда> ...
```

**Требования и переносимость:** Node.js 18+; пакеты `playwright`,
`pixelmatch`, `pngjs` в `node_modules` проекта (в этом репозитории объявлены в
корневом `package.json`; если не установлены —
`npm install --no-save pngjs pixelmatch playwright`). Если нет браузера:
`npx playwright install chromium`. Инструмент ищет зависимости сначала рядом
со своим файлом, затем в `node_modules` каталога запуска, поэтому работает и
при установке навыка вне дерева проекта. Способность агента «видеть»
изображения нужна только для диагностики по кропам; сам вердикт
PASS/FAIL — числовой и доступен любому агенту с доступом к терминалу.

| Команда | Назначение |
|---|---|
| `to-png <image> --out <png> [--width N]` | Привести образец (jpg/webp/png) к PNG, при необходимости отмасштабировать к ширине viewport |
| `capture <url> --out <png> [--viewport WxH] [--dpr 1] [--wait-selector <css>] [--element <css>]` | Скриншот работающего UI с фиксированными viewport и плотностью пикселей |
| `compare <ref.png> <impl.png> [--out-dir D] [--grid 4x4] [--gate 5] [--cell-gate 10] [--regions <json>] [--no-crops]` | Пиксельное сравнение: общий %, разбивка по сетке и/или именованным регионам, diff.png, side-by-side.png, report.json; для проблемных зон автоматически пишутся кропы «образец\|реализация\|diff». `--regions` принимает простой массив `[{id,x,y,w,h}]` **или `*.design.json` (ADR-016) напрямую** — зоны и вложенные элементы с `bounds` |
| `compare-elements <ref.png> <url> --regions <файл> [--selectors <map.json>] [--tolerance 2] [--gate 10]` | **Двухуровневая проверка по элементам с селекторами** (см. шаг 2б): геометрия — bounds разметки против фактического `getBoundingClientRect` («не там»); внешний вид — кропы с нормализацией размера («не такой»); перекрытие — `elementFromPoint` («виден, но не кликается»); отсутствующий элемент — явный `MISSING` |
| `crop <img.png> --rect x,y,w,h --out <png> [--scale 2]` | Вырезка зоны для детального просмотра (смотреть кропы, не полный кадр) |
| `sample <img.png> --points "x,y;x,y"` или `--rect x,y,w,h` | Точные цвета: по точкам или среднее + доминирующие по зоне |

Артефакты пишутся в `.tmp/ui-compare/` (правило проекта: временные файлы — в
`.tmp/`, в репозиторий не коммитятся, после задачи подчищаются).

## 4. Процесс сравнения

### Шаг 0 — предусловия честного сравнения

Без них diff измеряет не CSS, а мусор:

1. **Одинаковое состояние данных.** Если образец показывает экран с 6
   карточками, реализация должна показывать те же 6 карточек (фикстуры или
   продвижение через API до нужного состояния). Diff 80%+ почти всегда значит
   «разные экраны/данные», а не «плохой CSS».
2. **Целевой экран подтверждён проверкой DOM** (наличие ожидаемых селекторов),
   а не таймаутом.
3. **Ровно один процесс dev-сервера, свежий bundle.** Устаревший кэш dev-сервера —
   классическая причина «вижу совсем другой UI» (методология, §9.6).
4. **Viewport = размеру образца, DPR = 1** — иначе постоянный пиксельный шум.

### Шаг 1 — получить сравнимую пару PNG

```bash
node <каталог-навыка>/scripts/ui-visual-tool.mjs to-png games/<id>/design/mockups/screen.jpg --out .tmp/ui-compare/ref-src.png
node <каталог-навыка>/scripts/ui-visual-tool.mjs capture http://localhost:3000/... --out .tmp/ui-compare/impl-src.png --viewport 1920x1080
```

Если до целевого экрана нужно дойти кликами/действиями — напиши одноразовый
Playwright-сценарий (шаблон продвижения по состояниям: методология §0.3, §8) и
сними скриншот из него с теми же viewport/DPR.

### Шаг 2 — измерить

```bash
node <каталог-навыка>/scripts/ui-visual-tool.mjs compare .tmp/ui-compare/ref-src.png .tmp/ui-compare/impl-src.png --regions games/<id>/design/mockups/screen.design.json
```

Выход: общий %, таблица ячеек сетки и именованных регионов (худшие сверху),
`diff.png`, `side-by-side.png`, `report.json` и автокропы проблемных зон.

**Семантические регионы приоритетнее сетки.** Если у образца есть разметка
зон (`*.design.json` по ADR-016) — передай её через `--regions` напрямую:
отчёт будет в терминах элементов («кнопка btn-move: 34%»), а не абстрактных
ячеек. Сетка остаётся страховкой для площади, не покрытой разметкой, и
работает всегда. Чем мельче размечены элементы (кнопки, карточки, индикаторы,
а не «весь контент»), тем лучше: в крупном регионе локальный дефект
разбавляется точно так же, как в полном кадре.

### Шаг 2б — сравнение по элементам (когда есть селекторы и живой URL)

Единый пиксельный процент смешивает два разных дефекта: «элемент сдвинут» и
«элемент выглядит иначе». Если реализация доступна как живая страница, а у
зон разметки есть CSS-селекторы — запусти двухуровневую проверку:

```bash
node <каталог-навыка>/scripts/ui-visual-tool.mjs compare-elements .tmp/ui-compare/ref-src.png http://localhost:3000/... --regions <разметка>
```

Для каждого элемента она выносит независимые вердикты:

1. **Геометрия («не там»)** — bounds из разметки против фактического
   `getBoundingClientRect`, допуск ±2px (`--tolerance`);
2. **Внешний вид («не такой»)** — кроп образца по bounds разметки против
   кропа реализации по *фактическим* bounds с нормализацией размера — сдвиг
   не загрязняет оценку внешнего вида;
3. **Перекрытие** — `elementFromPoint` в центре: интерактивный элемент под
   чужим слоем = FAIL, даже если пиксели идеальны (случай «кнопка видна, но
   не кликается»);
4. **Отсутствие** — ненайденный селектор даёт явный `MISSING`, а не
   растворяется в общем проценте.

**Откуда берутся селекторы:** необязательное поле `selector` у зоны/элемента
в `*.design.json` (схема ADR-016 допускает дополнительные поля, дрейфа нет)
или отдельная карта `--selectors map.json` вида `{"id": "css",
"region/element": "css"}`, когда разметку менять нельзя.

### Шаг 3 — локализовать

Для каждой ячейки/региона выше порога:

1. **Смотри автокропы** `crop-<зона>.png` (полоса «образец | реализация |
   diff» с увеличением), которые `compare` уже сохранил в out-dir, — а не
   полные кадры. Дополнительные вырезки — командой `crop --scale 2`.
2. Цвета проверяй только через `sample` — сравни hex-значения двух изображений
   в одинаковых координатах.
3. Метрики элементов (размеры, отступы, шрифты, z-index) снимай через
   `getComputedStyle` в Playwright, а не чтением CSS-файлов: файл говорит, что
   *написано*, computed style — что *применилось* (методология §9.1).

### Шаг 4 — классифицировать каждое различие

Каждая зона с различиями попадает ровно в одну корзину (молча пропускать зоны
нельзя — принцип «трёх корзин» из ADR-057):

- **Дефект** — реальное расхождение (layout, цвет, размер, отсутствующий
  элемент) → чинить;
- **Шум рендеринга** — сглаживание шрифтов, хинтинг, JPEG-артефакты → принять,
  зафиксировать в отчёте;
- **Намеренное отличие** — задокументированное решение (ADR/задача) → указать,
  где задокументировано; исключить зону из ворот.

### Шаг 5 — итерация

Фикс → повторный `compare` → снова шаг 3, пока не пройдены ворота. Успех
объявляется **только по свежему прогону** — «я поправил CSS, теперь должно
совпадать» без прогона не является результатом.

## 5. Ворота приёмки

- Общий diff **< 5%** (для итераций без структурных изменений — **< 3%**);
  ~1% остаточного шума от шрифтов — норма.
- **Ни одна ячейка/регион не превышает порог ячейки (10%).** Это отдельный
  критерий: локальный дефект (пропавшая кнопка — 25% в своей ячейке)
  «разбавляется» площадью кадра до 2–3% и проходит общий порог; порог на
  ячейку ловит именно такие случаи.
- Все ячейки/регионы выше порога классифицированы (шаг 4), намеренные отличия
  имеют ссылку на документ.
- **Для элементов с селекторами** (`compare-elements`): геометрия в допуске
  ±2px, внешний вид < 10%, ни один интерактивный элемент не перекрыт, ни один
  элемент не `MISSING`.
- **Кликабельность проверена отдельно** — визуальное совпадение не гарантирует
  работающие кнопки: элемент может быть перекрыт (stacking context, оверлей).
  Проверку перекрытия по центру элемента `compare-elements` делает
  автоматически; для сложных случаев — `elementFromPoint` вручную;
  `page.click()` даёт ложный успех. Детали: методология §7.
- Для режима style-parity вместо пиксельного процента: таблица «токен → в
  образце (sample) → в реализации (computed style) → совпадает?».
- **Перекрытия элементов не меняют вердикт:** пиксельный diff всегда
  считается по плоскому (итоговому) изображению — именно его видит игрок.
  Слои и z-порядок — предмет диагностики (чей пиксель неправильный) и
  проверки кликабельности через DOM, а не отдельного пиксельного критерия.

## 6. Формат отчёта

ВСЕГДА заверши сравнение отчётом такой структуры:

```markdown
## Сравнение UI с образцом
- Режим: pixel-parity | style-parity
- Образец: <путь>, реализация: <url/состояние>, viewport, DPR
- Общий diff: N% (порог M%) — PASS/FAIL

| Зона | Diff % | Корзина | Причина / действие |
|---|---|---|---|

- Исправлено в этой итерации: ...
- Осталось и почему допустимо: ...
- Кликабельность: проверена / не применимо
```

## 7. Типичные ловушки

| Симптом | Вероятная причина | Что делать |
|---|---|---|
| «Совсем другой UI», фиксы не применяются | Устаревший кэш dev-сервера, несколько процессов | Убить все процессы порта, запустить один, жёсткая перезагрузка (методология §9.6) |
| Diff 60–90% | Разные состояния/данные, не CSS | Выровнять фикстуры/состояние (шаг 0) |
| Стабильный шум 1–2% по всему кадру | Шрифты (subsets, хинтинг), DPR ≠ 1, антиалиасинг | Зафиксировать DPR=1, не включать `--include-aa` |
| Кнопки видны, но не работают | Stacking context: оверлей с `z-index: 0` поверх кнопок | `pointer-events: none` оверлею или `z-index: 1` кнопкам; не двигать grid-строки (методология §3.8, §7.5) |
| Ячейка сетки 100% diff | Отсутствующий/лишний элемент, сдвиг layout | Кропы + сравнение DOM-структуры |
| Разные размеры изображений | Viewport ≠ размер образца | `capture --viewport` по размеру образца или `to-png --width` |

## 8. Когда читать полную методологию

`docs/processes/UI_MIGRATION_METHODOLOGY.md` (в корне этого репозитория)
— обязательна к прочтению целиком, когда: переносится целая визуальная
парадигма между приложениями; нужны CSS-override-фиксы поверх чужого
рендерера; порог не достигается после 3+ итераций; есть подозрение на
проблемы кликабельности. Там же — шаблоны диагностических скриптов
(продвижение по состояниям, computed styles, проверка перекрытий).

Для разработки нового UI по изображению-макету используй навык
`ui-from-mockup` (соседний каталог навыков) — этот навык (ui-compare) служит
там воротами приёмки.
