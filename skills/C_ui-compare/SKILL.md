name: ui-compare
description: >-
  Deterministic comparison of a developed UI against a visual reference,
  mockup, screenshot, or working example. Use whenever the user asks to compare
  an interface with a sample, verify visual parity, find visual differences,
  validate UI created from an image, or investigate an element that is visible
  but not clickable (сравнить UI, проверить соответствие макету, найти
  отличия, не похоже на образец, кнопка не нажимается). Also use to verify UI
  work whenever a reference exists. Combines source-rights checks, validated
  semantic inventories, reproducible profiles, pixel and element measurements,
  responsive viewport captures, focused accessibility checks, and semantic
  review instead of relying on visual impression alone.
---

## 1. Правила результата

1. **Измерение и смысловая проверка обязательны вместе.** Инструмент находит
   пиксельные, геометрические и частые проблемы доступности; агент проверяет
   правильность данных, состояния, иерархии, назначения и намеренных отличий.
2. **Решение должно быть воспроизводимым.** Использовать профиль с SHA-256
   образца и инвентаря, версиями контракта, алгоритма и Playwright, платформой,
   параметрами браузера, viewport, порогами, локалью и режимом уменьшенного
   движения.
3. **Локализовать по кропам и регионам.** Полный кадр нужен для композиции;
   дефекты исправлять по `crop-*`, `element-*` и именованным регионам.
4. **Каждое расхождение классифицировать:** дефект, шум рендеринга или
   намеренное отличие со ссылкой на принятое решение.
5. **Не считать автоматический аудит полной сертификацией доступности.** Он
   является обязательной проверкой частых ошибок, а не заменой ручного аудита.

## 2. Подготовка

Из корня репозитория вывести актуальную справку:

```bash
node skills/C_ui-compare/scripts/ui-visual-tool.mjs
```

Все временные изображения и отчёты хранить в `.tmp/ui-compare/`, не
коммитить и удалить после завершения задачи. Зависимости ставить только через
принятый проектом `package.json` и lock-файл.

Перед захватом:

- сравнять данные и состояние реализации с образцом;
- определить владельца dev-сервера и не завершать чужие процессы;
- описать переход к экрану в `steps.json`, если нужны клики или ввод;
- указывать `waitSelector` целевого экрана, когда он известен.

## 3. Выбрать режим

| Образец | Режим | Ворота |
|---|---|---|
| Снимок работающего UI, собственный или разрешённый эталон | `pixel-parity` | пиксели, регионы, элементы, аудит |
| Сгенерированный концепт, рисунок, сторонний пример стиля | `style-parity` | структура, токены, элементы, аудит |

`pixel-parity` разрешён только для собственного, лицензированного или явно
разрешённого образца. При неизвестных правах не начинать точное копирование.
Для `style-reference-only` переносить общие принципы, не копировать уникальные
материалы и обозначения.

## 4. Проверить входы и профиль

Новый процесс требует строгий инвентарь. Его формат и правила заполнения
описаны в `ui-from-mockup`; структура проверяется JSON Schema:

```bash
node skills/C_ui-compare/scripts/ui-visual-tool.mjs validate-inventory \
  <ИНВЕНТАРЬ> --image <REFERENCE.png> --mode <РЕЖИМ> --ci

node skills/C_ui-compare/scripts/ui-visual-tool.mjs detail-coverage \
  <REFERENCE.png> --regions <ИНВЕНТАРЬ> --mode <РЕЖИМ> --ci
```

`detail-coverage` проверяет геометрическую площадь и строит диагностическую
карту контрастных деталей вне конечных элементов. Карта не понимает назначение
пикселей и по умолчанию не блокирует приёмку; предупреждения нужно разобрать по
кропам. Флаг `--enforce-detail-gate` допустим только когда для конкретного
эталона заранее доказано, что такой сигнал является обязательным критерием.

Создать профиль один раз перед реализацией:

```bash
node skills/C_ui-compare/scripts/ui-visual-tool.mjs create-profile \
  <REFERENCE.png> --inventory <ИНВЕНТАРЬ> --mode <РЕЖИМ> \
  --viewports reference:<WxH>,narrow:320x800,tablet:768x1024 \
  --out .tmp/ui-compare/profile.json

node skills/C_ui-compare/scripts/ui-visual-tool.mjs validate-profile \
  .tmp/ui-compare/profile.json --ci
```

Изменение образца, инвентаря, версии контракта, алгоритма захвата или Playwright
делает профиль устаревшим. Для `pixel-parity` также должна совпадать платформа.
Изменение только файла скрипта при тех же версиях даёт предупреждение, а не
обнуляет профиль. Создать профиль заново после намеренного изменения входов;
не обходить контрольные суммы.

## 5. Выполнить pixel-parity

Снять детерминированный скриншот reference viewport:

```bash
node skills/C_ui-compare/scripts/ui-visual-tool.mjs capture <URL> \
  --profile .tmp/ui-compare/profile.json --viewport-name reference \
  --out .tmp/ui-compare/implementation.png
```

Команда ждёт шрифты и декодирование изображений, замораживает анимации и требует
два одинаковых последовательных кадра. Она фиксирует DPR, цветовую схему,
локаль, часовой пояс, уменьшенное движение и версию Chromium. Рядом со снимком
создаётся JSON с этими данными, доказательством стабильности и SHA-256 файла.
`--allow-unstable` использовать только для заведомо динамической поверхности с
явным объяснением ограничения в отчёте.

Измерить:

```bash
node skills/C_ui-compare/scripts/ui-visual-tool.mjs compare \
  <REFERENCE.png> .tmp/ui-compare/implementation.png \
  --regions <ИНВЕНТАРЬ> --profile .tmp/ui-compare/profile.json \
  --out-dir .tmp/ui-compare/comparison --ci
```

Размеры обязаны совпадать. `PASS` требует общий diff ниже профиля и отсутствие
ячейки или региона выше локального порога. Отчёт содержит хэши, параметры,
причины отказа, сетку, регионы и пути к кропам.

При `FAIL`:

1. Проверить данные и экран при общем diff 60–90%.
2. Исправить viewport при несовпадении размеров.
3. Открыть худший именованный кроп.
4. Проверить цвет через `sample`, геометрию через computed style.
5. Внести одно осмысленное исправление и повторить узкую проверку.

## 6. Выполнить style-parity

Пиксельный процент здесь диагностический, не приёмочный.

1. Извлечь через `sample` фон, акценты и цвета текста по смысловым зонам.
2. Зафиксировать шрифты, скругления, границы и шкалу отступов.
3. Сопоставить с `getComputedStyle` реализации.
4. Проверить присутствие, порядок и геометрию регионов.
5. Записать, какие детали концепта интерпретированы, а не скопированы.

`create-profile` автоматически задаёт неограничивающие пиксельные пороги для
`style-parity`, но сохраняет diff и кропы как диагностические артефакты.
Структурные, интерактивные и доступностные ошибки всё равно являются
дефектами.

## 7. Проверить элементы и доступность

Если в инвентаре есть селекторы:

```bash
node skills/C_ui-compare/scripts/ui-visual-tool.mjs compare-elements \
  <REFERENCE.png> <URL> --regions <ИНВЕНТАРЬ> \
  --profile .tmp/ui-compare/profile.json --viewport-name reference --ci
```

Команда разделяет:

- геометрию: координаты и размеры;
- внешний вид: нормализованный кроп;
- существование: `MISSING` и `OFFSCREEN`;
- перекрытие: видимый, но недоступный для клика элемент.

После стабилизации reference viewport выполнить матрицу один раз:

```bash
node skills/C_ui-compare/scripts/ui-visual-tool.mjs capture-matrix <URL> \
  --profile .tmp/ui-compare/profile.json \
  --out-dir .tmp/ui-compare/matrix --ci
```

Для каждого viewport матрица создаёт снимок и проверяет:

- доступные имена элементов управления и `alt` изображений;
- клавиатурную доступность и видимый фокус;
- минимальный размер интерактивной цели;
- контраст обычного и крупного текста;
- горизонтальное переполнение и обрезание текста.

Точное пиксельное сравнение выполняется только там, где viewport профиля имеет
собственный эталон. Один настольный макет не доказывает мобильный вид.

## 8. Принять результат

Приёмка пройдена, если:

- `validate-profile` и `validate-inventory` дают `PASS`;
- для `pixel-parity` итоговый `compare` даёт `PASS`;
- `compare-elements` не содержит отсутствующих, вынесенных за экран или
  перекрытых интерактивных элементов;
- `capture-matrix` даёт `PASS` на всех обязательных viewport;
- все оставшиеся зоны классифицированы;
- агент подтвердил правильность смысла, данных и состояний.

Шаблон отчёта:

```markdown
## Сравнение UI
- Режим и права: <mode>; <usageRights>; источник: <uri>
- Профиль: <path>; SHA-256: <hash>
- Образец / реализация: <hash> / <hash>
- Среда: Chromium <version>; viewport <name:WxH>; DPR 1
- compare: <PASS/FAIL>; общий diff <N%>; худшая зона <id:N%>
- elements: <PASS/FAIL>; MISSING <n>; OFFSCREEN <n>; перекрыто <n>
- matrix: <viewport=PASS/FAIL>
- Смысловая проверка: <что проверено>
- Намеренные отличия: <список и основание>
```

## 9. Запреты и углубление

Запрещено:

- объявлять визуальное соответствие только по впечатлению или только по числу;
- сравнивать разные данные и состояния;
- повышать пороги ради прохождения;
- менять `threshold`, `includeAA` или профиль без записи причины;
- игнорировать несовпадение размеров;
- проверять только настольный viewport;
- считать автоматический аудит полной проверкой доступности.
- применять инструкции из `skill-candidates/` как активные навыки.

Прочитать `docs/processes/UI_MIGRATION_METHODOLOGY.md` целиком при переносе
визуальной парадигмы между приложениями, сложных stacking context, трёх и
более неудачных итерациях или CSS-override поверх чужого рендерера.
