## 1. Железное правило

> Никакого кода, пока нет письменного инвентаря макета, и его полнота не
> подтверждена командой `coverage`. Путь всегда:
> **изображение → инвентарь (проверен числом) → структура → стиль →
> измеренная приёмка.**

Почему: при верстке «прямо с картинки» модель пропускает элементы (особенно
мелкие и периферийные), выдумывает значения отступов и цветов, а потом
рационализирует расхождения («визуально совпадает»). Письменный инвентарь
проверяем, а его полноту проверяет инструмент, а не внимательность.
Индустриальный принцип проекта: **структура — источник истины, изображение —
концепт стиля и источник ассетов**
(`docs/architecture/game-ui-mockup-flow.md`, раздел «Уроки индустрии»).

Все команды инструмента — из навыка `ui-compare` (соседний каталог навыков).
Перед началом прочитай там разделы 2 («Подготовка») и 3 («Выбор режима»).
В этом репозитории команды выглядят так:

```bash
node .claude/skills/ui-compare/scripts/ui-visual-tool.mjs <команда> ...
```

## 2. Шаг 0 — зафиксируй рамки

- [ ] **Режим строгости** — по таблице ui-compare §3: образец является
  скриншотом работающего UI → pixel-parity; сгенерирован или нарисован →
  style-parity. При сомнении спроси пользователя. Запиши режим в задачу.
- [ ] **Куда пишется инвентарь:**
  - работа над игрой в `games/<id>/` → файл
    `games/<id>/design/mockups/<имя>.design.json` по ADR-016 (примеры:
    `games/antarctica/design/mockups/`; схема:
    `docs/architecture/schemas/design-artifact.schema.json`; плюс запись в
    `design-history.json` и реестр `design_artifacts` UI-манифеста);
  - разовая задача вне игры → `.tmp/ui-compare/inventory.json` — простой
    массив записей формата из шага 3.

## 3. Шаг 1 — макет в PNG

```bash
node .claude/skills/ui-compare/scripts/ui-visual-tool.mjs to-png <МАКЕТ> --out .tmp/ui-compare/mockup.png
```

Запиши из вывода ширину W и высоту H — они нужны в шагах 2 и 7.

**Правило цвета:** все цвета сэмплируй с этого файла оригинального размера.
Если позже сделаешь `to-png --width` для сравнения — цвета из
масштабированной копии не бери: сглаживание при масштабировании искажает
оттенки.

## 4. Шаг 2 — обход девятью кропами

Раздели макет на 9 частей (сетка 3×3): `CW = floor(W/3)`, `CH = floor(H/3)`.
Сделай 9 кропов (r и c — от 0 до 2):

```bash
node .claude/skills/ui-compare/scripts/ui-visual-tool.mjs crop .tmp/ui-compare/mockup.png --rect <c*CW>,<r*CH>,<CW>,<CH> --scale 2 --out .tmp/ui-compare/scan-r<r>c<c>.png
```

Для макета 1920×1080 (CW=640, CH=360) прямоугольники такие:
`0,0,640,360` · `640,0,640,360` · `1280,0,640,360` · `0,360,640,360` ·
`640,360,640,360` · `1280,360,640,360` · `0,720,640,360` · `640,720,640,360`
· `1280,720,640,360`.

Открой каждый кроп и выпиши ВСЕ видимые элементы, включая мелкие и
декоративные. Почему кропами: на полном кадре, ужатом до размеров контекста,
мелкие элементы физически не видны — именно так возникают «пропущенные»
кнопки.

## 5. Шаг 3 — запиши инвентарь

На каждую зону и каждый элемент — отдельная запись:

```json
{
  "id": "btn-move",
  "type": "button",
  "bounds": { "x": 100, "y": 920, "width": 150, "height": 60 },
  "description": "Кнопка перемещения, текст 'Переместить'",
  "selector": "#btn-move",
  "style": { "background": "#3282b8" }
}
```

Правила заполнения:

1. **Уровень элементов, а не областей.** Кнопка, карточка, индикатор —
   отдельные записи с собственными `bounds`. Крупная зона «весь контент»
   разбавляет локальные дефекты при приёмке так же, как полный кадр.
2. **Перекрытия — явно.** Если элементы накладываются (модальное окно,
   оверлей, тень поверх соседа) — запиши в `description`, что поверх чего:
   z-порядок восстановим из плоской картинки только сейчас; при реализации
   он превратится в `z-index`/`pointer-events` и проверку кликабельности.
3. **`selector`** заполняй, как только элемент реализован (CSS-селектор его
   DOM-узла; схема ADR-016 допускает дополнительные поля) — это включает
   поэлементную приёмку `compare-elements`.
4. **Идентичность (для игр, ADR-057):** содержательный элемент ссылается на
   `id` игровой сущности (`maps_to_component` / `maps_to_state` /
   `maps_to_action`); чисто декоративный — явно `"type": "decor"`.
5. **Фон — тоже зона** (например `id: "background"` с bounds на весь макет).
6. **Цвета — только из sample**, на глаз запрещено:
   ```bash
   node .claude/skills/ui-compare/scripts/ui-visual-tool.mjs sample .tmp/ui-compare/mockup.png --rect <x,y,w,h>
   node .claude/skills/ui-compare/scripts/ui-visual-tool.mjs sample .tmp/ui-compare/mockup.png --points "<x,y>"
   ```
   `--rect` — для фонов (среднее и доминирующие цвета сглаживают
   JPEG-артефакты), `--points` — для тонких элементов.
7. **Типографика и отступы:** ближайший доступный шрифт (недоступный —
   намеренное отклонение, запиши), шкала отступов — измерь по кропам
   (обычно кратна 4 или 8), не предполагай.

## 6. Шаг 4 — проверь полноту числом

```bash
node .claude/skills/ui-compare/scripts/ui-visual-tool.mjs coverage .tmp/ui-compare/mockup.png --regions <ИНВЕНТАРЬ>
```

- `СТАТУС: FAIL` → открой маску `coverage-mask.png`: красное = площадь, не
  покрытая ни одной зоной, то есть кандидаты в пропущенные элементы. Добавь
  зоны, повтори команду.
- **Не начинай реализацию до `СТАТУС: PASS`** (непокрыто < 2%).

Это главная защита от «агент не увидел элемент»: пропуск виден как красное
пятно и число, а не зависит от внимательности.

## 7. Шаг 5 — классификация и рельсы проекта

Перед реализацией классифицируй работу:

1. **Прототип UI в Cubica — authoring-манифест, а не код** (ADR-050,
   ADR-057). Экраны игры собираются из каталога прототипов; правки идут
   через конвейер `EditorChangeSet`; отдельные «генеративные» пути записи
   запрещены.
2. **Платформенная чистота** (CLAUDE.md п.10): для каждой механики из макета
   реши — общая (платформенный прототип/контракт) или игровая (бандл игры).
   Никаких game-specific веток в общих слоях. Если неясно — спроси.
3. **Элемент без игровой сущности** — не выдумывай сущность молча: либо
   декоративный, либо вопрос пользователю.
4. **Parity-работа в рендерере** (перенос вида в `player-web` и т.п.) —
   следуй `docs/processes/UI_MIGRATION_METHODOLOGY.md`: CSS-override-блоки
   без изменения архитектуры target.

## 8. Шаг 6 — реализация слоями

Не начинай следующий слой, пока `compare` не подтвердил текущий. Данные
предпросмотра = данные макета: сделай фикстуру с теми же текстами и
количеством карточек ДО слоя 1, иначе сравнение измеряет разницу данных.

| Слой | Что делаешь | Проверка после слоя |
|---|---|---|
| 1. Каркас | Контейнеры и сетка по `bounds` инвентаря | `compare`: ни одна ячейка не ~100% (нет сдвигов каркаса) |
| 2. Размеры и отступы | Точные bounds, gap, padding | `compare`: зоны каркаса уходят вниз таблицы |
| 3. Цвета и типографика | Значения из инвентаря (из `sample`) | `compare` + точечный `sample` по спорным зонам |
| 4. Ассеты и декор | Иконки, фоны, иллюстрации (для игр — принятый поток ассетов с метаданными происхождения) | Полная приёмка (шаг 7) |

Почему слоями: стилизация маскирует структурные сдвиги; каркасная ошибка,
дожившая до слоя 4, стоит в разы дороже.

## 9. Шаг 7 — приёмка

Полный проход навыка `ui-compare` (его разделы 4–7), в такой
последовательности:

1. `compare ... --regions <ИНВЕНТАРЬ>` → `СТАТУС: PASS`;
2. `compare-elements ... --regions <ИНВЕНТАРЬ>` → `СТАТУС: PASS`
   (селекторы уже заполнены по правилу 3 шага 3; нет `MISSING`/`OFFSCREEN`,
   интерактивные не перекрыты);
3. отчёт по шаблону ui-compare §7; намеренные отклонения от макета
   (недоступный шрифт, изменённые элементы) — списком в задаче.

Не сообщай о готовности без свежих `PASS` обоих прогонов.

## 10. Ловушки

| Ловушка | Что делать |
|---|---|
| Шрифт макета недоступен или неопределим | Ближайший доступный + записать как намеренное отклонение (даст остаточный diff-шум) |
| Сгенерированный макет содержит артефакты (кривые дубли, нечитаемый псевдотекст) | Это режим style-parity: воспроизводи смысл зоны, не артефакты; спорные зоны согласуй с пользователем |
| Мок-данные не совпадают с макетом | Фикстура с данными макета — до слоя 1, не «потом» |
| Хочется поправить инвентарь под уже написанный код | Инвентарь правится только по измерениям макета (`sample`, кропы) |
| Макет одного размера, целевой viewport другого | Сравнение через `to-png --width`; шкалу отступов пересчитай явно; цвета — только с оригинала |
| До целевого экрана нужно докликать | Файл `steps.json` для `capture`/`compare-elements` (формат — ui-compare §4 шаг 0) |

## 11. Запрещено

- Писать код до `coverage` `PASS` по инвентарю.
- Оценивать цвета, отступы и размеры на глаз.
- Подгонять инвентарь под уже написанный код.
- Выдумывать игровую сущность для элемента без подтверждения.
- Сообщать о готовности без свежих `PASS` прогонов приёмки (шаг 7).
